# 07-插件测试与项目内运行验证

## 文档元数据

- 文档目的：告诉开发者怎么验证插件不是“看起来写完了”，而是真的能在项目里跑通。
- 当前版本：v1.0
- 关联文档：`docs/开发者文档/插件开发/zh-CN/02-插件开发环境与本地调试.md`、`docs/开发者文档/插件开发/zh-CN/05-插件对接方式说明.md`、`docs/开发者文档/插件开发/zh-CN/06-从零开发一个可运行插件.md`、`apps/api-server/tests/test_plugin_manifest.py`、`apps/api-server/tests/test_plugin_runs.py`
- 修改记录：
  - `2026-03-13`：创建首版插件测试与项目内运行验证文档。

这篇文档只回答一个问题：

- 你开发的插件，怎么证明它在 runner 模式下可用，并且能被主项目接住

别把“代码能 import”当成“插件能交付”。那差得远。

## 1. 最小验证分 4 层

按这个顺序验，最稳：

1. `manifest` 校验
2. runner 内入口函数校验
3. 主服务协议链路校验
4. 项目内调用链校验

一层没过，就不要急着跳下一层。

## 2. 第一层：`manifest` 校验

这是最基本的一层。

你至少要确认：

1. `id`、`name`、`version`、`types`、`permissions`、`risk_level`、`triggers`、`entrypoints` 齐全
2. `id` 字符规则正确
3. `types` 没有写仓库不支持的新类型
4. 每个声明的类型都有对应入口

现成参考：

- `apps/api-server/tests/test_plugin_manifest.py:22`
- `apps/api-server/tests/test_plugin_manifest.py:31`

如果这一步不过，后面一切都不用谈。

## 3. 第二层：入口函数校验

这一步不看数据库，只看插件在自己的 venv / runner 里能不能工作。

### 3.1 `connector`

至少确认：

- 返回的是 `dict`
- 里面有 `records`
- `records` 是 `list`

### 3.2 `memory-ingestor`

至少确认：

- 返回的是 `list`
- 每一项都是 `dict`
- 每一项至少包含 `category`、`value`、`source_record_ref`

### 3.3 `action`

至少确认：

- 返回的是 JSON 可序列化 `dict`
- 调用方能看懂动作是否执行成功

### 3.4 `agent-skill`

至少确认：

- 返回的是 JSON 可序列化 `dict`
- 没有偷偷依赖额外隐藏上下文

## 4. 第三层：服务层链路校验

这一步开始验证“主服务会不会真的认这个插件返回结果”。

现在 runner 和后台任务主链都已经站住了，所以测试不要再停留在“同步跑一下函数就算完”。

### 4.1 只测插件执行入口

用 `execute_plugin()` 验证：

- 插件能被注册中心识别
- 插件能被统一入口调用
- 返回结果结构没有跑偏

参考：

- `apps/api-server/tests/test_plugin_manifest.py:95`

### 4.2 测完整数据同步链路

如果你的插件是 `connector + memory-ingestor`，必须再测：

- `run_plugin_sync_pipeline()`

这条链路会覆盖：

1. 调 `connector`
2. 保存原始记录
3. 调 `memory-ingestor`
4. 写入标准记忆
5. 写审计日志

参考：

- `apps/api-server/tests/test_plugin_runs.py:48`
- `apps/api-server/tests/test_plugin_runs.py:116`
- `apps/api-server/tests/test_plugin_runs.py:192`

但这里也要分清：

- `run_plugin_sync_pipeline()` 更偏内部编排验证
- 当前正式主链路还要再补后台任务验证

至少再测：

- `POST /api/v1/plugin-jobs`
- worker 执行后任务能进入终态
- `GET /api/v1/plugin-jobs/{job_id}` 能查到尝试和通知

### 4.3 测 Agent 调用桥接

如果你的插件要给 Agent 直接调用，还得测：

- `invoke_agent_plugin()`

当前只允许：

- `connector`
- `agent-skill`

参考：

- `apps/api-server/tests/test_agent_plugin_bridge.py:48`

### 4.4 测动作权限和高风险确认

如果你的插件是 `action`，至少还要测两件事：

1. 没权限时会被拒绝
2. 高风险动作会进入人工确认，而不是直接执行

参考：

- `apps/api-server/tests/test_action_plugin_permissions.py:50`
- `apps/api-server/tests/test_action_plugin_permissions.py:107`
- `apps/api-server/tests/test_action_plugin_permissions.py:164`

## 5. 推荐测试命令

在 runner 还没落地前，`apps/api-server/` 目录下最少可以直接跑这些测试：

```bash
python -m unittest tests.test_plugin_manifest
python -m unittest tests.test_plugin_runs
python -m unittest tests.test_agent_plugin_bridge
python -m unittest tests.test_action_plugin_permissions
```

如果你现在只是在做读数据插件，最先跑前两个就够了。

## 6. 如何判断“项目内已经能运行”

不是只看“测试绿了”就完事。

第三方插件后续还应该补一层 runner 自检：

- 在插件自己的 venv 里直接执行入口
- 确认依赖只装在插件环境，不装进主 API 环境

你至少要看到和你的插件类型对应的这些结果。

### 6.1 `connector + memory-ingestor`

至少要看到：

- 插件执行成功
- 原始记录条数大于 0
- 标准记忆写入条数大于 0
- 审计日志有记录

### 6.2 `connector` 给 Agent 直接调用

至少要看到：

- `invoke_agent_plugin()` 返回成功
- 返回结果里有预期输出
- 有对应审计日志

### 6.3 `action`

至少要看到：

- 没权限时被拒绝
- 有权限时执行成功
- 高风险时先返回 `confirmation_required`

## 7. 如果要做 HTTP 联调，看哪些入口

当前项目里对应入口是：

- `POST /api/v1/plugin-jobs`
- `GET /api/v1/plugin-jobs/{job_id}`
- `GET /api/v1/plugin-jobs`
- `POST /api/v1/plugin-jobs/{job_id}/responses`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/plugin-invocations`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/plugin-memory-checkpoint`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/action-plugin-invocations`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/action-plugin-confirmations/{confirmation_request_id}/confirm`

对应代码在：

- `apps/api-server/app/api/v1/endpoints/ai_config.py:505`
- `apps/api-server/app/api/v1/endpoints/ai_config.py:550`
- `apps/api-server/app/api/v1/endpoints/ai_config.py:576`
- `apps/api-server/app/api/v1/endpoints/ai_config.py:603`

但还是那句话：

- 先过服务层测试
- 再做 HTTP 联调

不然你出问题时根本分不清是接口、权限、数据库、还是插件本身的问题。

## 8. 最小交付前自检清单

提交注册 PR 之前，至少过完这 8 条：

1. `manifest` 能通过校验
2. 所有入口都能 import
3. 入口返回结构符合类型要求
4. 如果是数据插件，内部链路至少测过 `run_plugin_sync_pipeline()`
5. 至少走过一次“创建任务 -> worker 执行 -> 查询结果”
6. 如果要给 Agent 用，确认返回的是任务信息而不是旧的同步成功心智
7. 如果是动作插件，权限拒绝、高风险确认、响应入口都测过
8. README 已写清最小验证方式

## 9. 一句话总结

当前插件测试别搞虚的。

最小可交付标准就是：

- `manifest` 合法
- 入口能调
- 项目服务链路能跑
- 该有的权限、审计、确认逻辑没有被绕过

做到这一步，才叫“这个插件真的能在项目里运行”。
