# 11-插件配置接入说明

## 文档元数据

- 文档目的：说明插件配置为什么要走正式协议、哪些规则是稳定边界，以及高频变化细节应该去哪里看。
- 当前版本：v1.0
- 关联文档：`docs/开发者文档/插件开发/zh-CN/03-manifest字段规范.md`、`docs/开发者文档/插件开发/zh-CN/05-插件对接方式说明.md`、`specs/004.2.3-插件配置协议与动态表单/README.md`
- 修改记录：
  - `2026-03-16`：创建首版插件配置接入说明。

这份文档不重复贴 API 大样例，只讲稳定规则。

## 1. 先说结论

以后插件只要需要“让用户在管理端配置参数”，就应该走正式配置协议。

不要再用这些老办法：

- 宿主页自己写一份字段常量
- 某个业务表里偷偷塞一坨 JSON 当唯一配置来源
- 把 secret 当普通字符串读出来

那种写法短期快，长期就是垃圾。

## 2. 正式配置协议是什么

插件配置协议的正式入口在 `manifest.json`：

- `config_specs`

每个配置作用域至少要说明这些信息：

- `scope_type`
- `schema_version`
- `config_schema`
- `ui_schema`

稳定语义只有两条：

1. 配置字段定义只保留一份来源，也就是插件 manifest
2. 宿主页只负责读协议、渲染表单、提交配置

## 3. 第一版正式支持的作用域

这轮只支持两个作用域：

- `plugin`
- `channel_account`

怎么选很简单：

- 同一个家庭里只有一份配置，用 `plugin`
- 每个通道账号各有一份配置，用 `channel_account`

不要现在就发明第三层、第四层作用域。

## 4. secret 字段的死规矩

`secret` 字段不是普通字符串，必须守这两条：

1. 读取时绝不回显明文
2. 保存时不提交该字段，就表示保留旧值

如果要清空旧值，必须显式走：

- `clear_secret_fields`

这个语义是稳定规则，不应该随着页面实现来回变。

## 5. 配置读写和运行时配置是两回事

现在要分清：

- 正式配置实例：走统一配置协议和配置实例表
- 兼容运行时副本：某些旧链路为了不破坏现有能力，可能还保留旧字段副本

比如通道插件现在会兼容旧 `config_json`，这是为了不把现有运行链路砸坏。

但真正的字段来源已经不是旧 `config_json`，而是正式协议。

## 6. 哪些内容不要写死在这份文档里

下面这些东西变化快，不应该复制到这里：

- 当前支持的全部字段类型列表
- 当前支持的全部 widget 列表
- 当前 API 请求 / 响应样例
- 当前页面组件名

这些去看下面这些事实来源：

- `specs/004.2.3-插件配置协议与动态表单/docs/20260316-manifest-示例.md`
- `specs/004.2.3-插件配置协议与动态表单/docs/20260316-api-示例.md`
- `apps/api-server/app/modules/plugin/schemas.py`
- `apps/user-web/src/lib/types.ts`

## 7. 插件作者最少要记住什么

如果你是插件作者，记住这 5 条就够了：

1. 要配置就写 `config_specs`
2. 字段定义只保留在插件 manifest
3. `plugin` 和 `channel_account` 先选对
4. secret 不回显，清空走 `clear_secret_fields`
5. 页面不应该再按插件 id 硬编码字段
