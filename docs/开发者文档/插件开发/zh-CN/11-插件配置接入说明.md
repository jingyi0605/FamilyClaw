# 11-插件配置接入说明

这份文档只讲稳定边界。

## 1. 配置入口只有一个

插件需要让用户填参数时，只能走：

- `manifest.config_specs`

不要再用：

- 宿主页硬编码字段
- 某个业务表偷偷塞 JSON 当唯一事实来源

## 2. 当前正式作用域

- `plugin`
- `integration_instance`
- `device`
- `channel_account`

怎么选很简单：

- 整个插件共用一份，用 `plugin`
- 每个集成实例一份，用 `integration_instance`
- 每台设备一份，用 `device`
- 每个通道账号一份，用 `channel_account`

## 3. secret 字段的死规矩

- 不回显明文
- 不提交代表保留旧值
- 要清空必须显式走 `clear_secret_fields`

## 4. 宿主和插件分别做什么

插件负责：

- 定义字段
- 定义校验
- 定义 UI 提示
- 同时提供可读原文兜底，不要只给词典 key

宿主负责：

- 渲染表单
- 保存值
- 保护 secret
- 注入运行时配置

补一条当前已经落地的规则：

- 保存配置时，宿主先按 schema 做字段级校验
- 某些“只有切到特定 provider 才必填”的 secret，可以在插件真正执行时再报 `not_configured`
- 不要把和当前实例无关的 key 保存校验，硬拧成表单提交阻塞

## 5. 作用域边界别再搞混

当前最容易写烂的是这一条：

不要把实例级配置继续塞到 `plugin` 作用域。

天气、多网关、多地区、多账号这类能力，只要配置属于某个具体实例，就应该写到 `integration_instance`。

一个现成例子：

- 天气源类型、刷新间隔、OpenWeather / WeatherAPI key 放 `plugin`
- 当前天气设备绑定家庭默认坐标还是指定地区节点，放 `integration_instance`
