# 04-插件目录结构规范

## 文档元数据

- 文档目的：规定插件目录怎么摆、每类入口代码该放哪里，以及最小可维护的插件目录长什么样。
- 当前版本：v1.2
- 关联文档：`docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`、`docs/开发者文档/插件开发/zh-CN/02-插件开发环境与本地调试.md`、`docs/开发者文档/插件开发/zh-CN/03-manifest字段规范.md`
- 修改记录：

  - `2026-03-13`：调整为按阅读顺序编号，并补充文档元数据。
  - `2026-03-14`：补充 `locale-pack` 语言包插件目录示例。

这份文档只解决一个问题：插件目录到底该怎么摆。

第一版不要搞复杂。目录结构先贴着现有仓库能跑的方式来，不要另造抽象。

这里也别搞混：

- 目录结构管的是插件代码怎么摆、依赖放哪里、runner 怎么稳定找到入口
- 它不改变对外执行语义
- 对外执行现在默认还是先创建后台任务，再由 worker 和 runner 去处理

## 1. 最小目录模板

```text
your_plugin/
  manifest.json
  requirements.txt
  README.md
  __init__.py
  integration.py
  action.py
  channel.py
  agent_skill.py
  ai_provider.py
  memory_engine.py
  memory_provider.py
  context_engine.py
  locales/
  tests/
    test_manifest.md
```

上面不是说每个文件都必须存在，而是说：

- `manifest.json`：必须有
- `requirements.txt`：第三方插件强烈建议有，用来声明自己的 Python 依赖
- `README.md`：强烈建议有，给审核和后续维护者看
- `__init__.py`：强烈建议有，避免 Python 包路径不清楚
- `integration.py` / `action.py` / `channel.py` / `agent_skill.py` / `ai_provider.py` / `memory_*.py`：按你声明的正式类型和槽位决定要不要有
- `locales/`：给 `locale-pack` 或普通插件自带词典放资源文件
- `tests/`：强烈建议有，哪怕先放最小自检说明

## 2. 一插件一目录

这是硬规则：

- 一个插件目录只放一个 `manifest.json`
- 一个 `manifest.json` 只描述一个插件
- 不要在同一个目录里塞多个插件能力集合，再靠脚本分流

原因很简单：

- 插件注册中心就是按 `manifest.json` 找插件
- 一个目录一个插件，后续排错、审核、注册都最省事

## 3. 目录名怎么取

第一版建议直接和 `manifest.json` 的 `id` 保持接近，最好一致或可一眼对应。

合格示例：

- 目录：`health_basic`
- `id`：`health-basic-reader`

- 目录：`homeassistant_device_action`
- `id`：`homeassistant-device-action`

不用强制完全相同，因为 Python 模块目录通常会用下划线；但至少要让人一眼对得上。

## 4. 推荐文件职责

### `manifest.json`

- 放插件声明
- 不放运行时代码
- 不塞一堆未来才会用的字段

### `requirements.txt`

- 放插件自己的 Python 依赖
- 不要假设平台会自动安装
- 版本范围写清楚，别写飘
- 默认由插件自己的环境消费，不是让主 API 环境替你兜底

### `integration.py`

- 放 `integration` 正式入口
- 推荐导出 manifest 对应的刷新、发现或同步函数
- 只产出标准设备、标准实体、标准卡片和标准动作结果

### `action.py`

- 放 `action` 正式入口
- 只负责动作执行本身，不负责绕开系统权限

### `agent_skill.py`

- 放 `agent-skill` 正式入口
- 只暴露受控能力，不直接改 Agent 主流程

### `ai_provider.py`

- 放 `ai-provider` driver 入口
- 提供 `invoke`、`ainvoke` 或 `stream` 之类正式 driver 能力

### `memory_engine.py` / `memory_provider.py` / `context_engine.py`

- 放独占槽位插件入口
- 严格按宿主定义的槽位契约实现，不再使用旧 `memory-ingestor` 语义

### `locales/`

- 给声明了 `locales` 的插件放语言资源文件
- 推荐一个语言一个文件，比如 `locales/zh-TW.json`
- 文件内容就是 key-value 文案，不要塞执行代码
- `manifest.locales[].resource` 必须指向这里的真实文件

这里别再理解成“只有语言包插件才准有 `locales/`”。

当前规则是：

- `locale-pack` 用它承载整包语言资源
- 普通插件也可以用它承载自己的插件词典
- 同一个 `locale_id` 会在宿主侧按消息 key 合并，不是整包替换

### `README.md`

建议至少写清楚：

1. 插件做什么
2. 用到哪些权限
3. 需要什么外部配置
4. 怎么做最小验证
5. 明确不做什么

### `tests/`

第一版不强制你交一整套复杂自动化，但至少要能说明：

- `manifest` 是否自检过
- 入口是否能 import
- 核心样例 payload 是否跑过

## 5. 多能力插件怎么摆

如果一个插件同时有 `integration` 和 `action`，不要拆成两个插件目录。

最小可用方式就是：

```text
health_basic/
  manifest.json
  __init__.py
  integration.py
  action.py
```

一个插件目录对应一个插件 id，但可以同时暴露多个正式入口。

这样做的好处很实际：

- 一个插件 id 对应一组能力
- 一次开发、一次审核、一次注册
- 不会把同一来源的数据链路拆散

## 6. 动作插件怎么摆

动作插件目录通常更简单：

```text
homeassistant_device_action/
  manifest.json
  __init__.py
  executor.py
  README.md
```

如果是高风险动作，比如门锁，目录结构也不用特殊化。风险体现在：

- `manifest.json` 里的 `risk_level`
- 权限声明
- 后续注册表风险说明

不要为了高风险动作再发明一套特殊目录层级。那是给自己挖坑。

## 7. 语言包插件怎么摆

`locale-pack` 跟可执行插件不是一回事。它不需要 `plugin/connector.py` 这些入口文件。

最小可用方式就是：

```text
locale_zh_tw_pack/
  manifest.json
  README.md
  locales/
    zh-TW.json
```

这里真正重要的是 3 件事：

- `manifest.json` 里把 `types` 写成 `locale-pack`
- `locales/zh-TW.json` 这种资源文件真实存在
- `manifest.locales[].resource` 路径和文件一一对上

如果你想同时提供多种语言，也还是一个插件一个目录，只是在 `locales/` 下多放几个文件。

如果是普通插件自带词典，目录更简单：

```text
official_weather/
  manifest.json
  service.py
  integration.py
  locales/
    zh-CN.json
    en-US.json
```

这种场景里：

- `types` 还是原来的正式插件类型，比如 `integration`
- `locales/` 只负责这个插件自己的文案
- 不会把它自动变成 `locale-pack`

## 8. 现在不建议加的目录内容

这些内容第一版先别加，除非后面 Spec 明确扩范围：

- 自动安装脚本
- 远程下载器
- 沙箱配置目录
- 全量签名证书目录
- 前端市场展示资源包

原因不是这些东西永远不要，而是现在加了也没地方真正落地，还会误导第三方开发者。

## 9. 官方插件与镜像打包边界

这条规则现在要写死：

- `builtin` 插件放仓库里的 `apps/api-server/app/plugins/builtin/`
- `official` 插件放 `apps/api-server/data/plugins/official/`
- `third_party` 插件放宿主数据目录或市场安装目录

对应的打包规则也必须一起成立：

- Docker 最终镜像只内置宿主和 `builtin` 插件
- `official` 与 `third_party` 插件不应打进最终镜像，应通过挂载、安装或同步落到宿主数据目录
- `.dockerignore` 应排除 `apps/api-server/data/plugins/official/`
- 宿主核心代码不能在 import 阶段静态依赖 `official` 或 `third_party` 插件模块，否则插件文件没挂进来时宿主会直接起不来

一句话总结就是：

官方插件可以是受信任插件，但不能被写成宿主编译依赖。

## 10. 对照现有仓库的参考目录

直接可以看的现成例子：

- `apps/api-server/app/plugins/builtin/health_basic/`
- `apps/api-server/app/plugins/builtin/homeassistant_device_sync/`
- `apps/api-server/app/plugins/builtin/homeassistant_device_action/`
- `apps/api-server/app/plugins/builtin/homeassistant_door_lock_action/`

如果你写的目录结构和这些差太远，就先解释清楚为什么。解释不清楚，多半就是设计跑偏了。
