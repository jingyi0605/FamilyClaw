# 04-插件目录结构规范

## 文档元数据

- 文档目的：规定插件目录怎么摆、每类入口代码该放哪里，以及最小可维护的插件目录长什么样。

- 关联文档：`docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`、`docs/开发者文档/插件开发/zh-CN/02-插件开发环境与本地调试.md`、`docs/开发者文档/插件开发/zh-CN/03-manifest字段规范.md`
- 修改记录：

  - `2026-03-13`：调整为按阅读顺序编号，并补充文档元数据。

这份文档只解决一个问题：插件目录到底该怎么摆。

第一版不要搞复杂。目录结构先贴着现有仓库能跑的方式来，不要另造抽象。

## 1. 最小目录模板

```text
your_plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    connector.py
    ingestor.py
    executor.py
    skill.py
  tests/
    test_manifest.md
```

上面不是说每个文件都必须存在，而是说：

- `manifest.json`：必须有
- `requirements.txt`：第三方插件强烈建议有，用来声明自己的 Python 依赖
- `README.md`：强烈建议有，给审核和后续维护者看
- `plugin/`：建议作为真正代码目录，避免仓库根目录塞满代码文件
- `plugin/__init__.py`：强烈建议有，避免 Python 包路径不清楚
- `plugin/connector.py` / `plugin/ingestor.py` / `plugin/executor.py` / `plugin/skill.py`：按你声明的能力决定要不要有
- `tests/`：强烈建议有，哪怕先放最小自检说明

## 2. 一插件一目录

这是硬规则：

- 一个插件目录只放一个 `manifest.json`
- 一个 `manifest.json` 只描述一个插件
- 不要在同一个目录里塞多个插件能力集合，再靠脚本分流

原因很简单：

- 插件注册中心就是按 `manifest.json` 找插件
- 一个目录一个插件，后续排错、审核、注册都最省事

## 3. 目录名怎么取

第一版建议直接和 `manifest.json` 的 `id` 保持接近，最好一致或可一眼对应。

合格示例：

- 目录：`health_basic`
- `id`：`health-basic-reader`

- 目录：`homeassistant_device_action`
- `id`：`homeassistant-device-action`

不用强制完全相同，因为 Python 模块目录通常会用下划线；但至少要让人一眼对得上。

## 4. 推荐文件职责

### `manifest.json`

- 放插件声明
- 不放运行时代码
- 不塞一堆未来才会用的字段

### `requirements.txt`

- 放插件自己的 Python 依赖
- 不要假设平台会自动安装
- 版本范围写清楚，别写飘

### `plugin/connector.py`

- 放读数据入口
- 推荐入口函数名：`sync`
- 典型返回：包含 `records` 的字典

### `plugin/ingestor.py`

- 放原始记录转标准记忆的入口
- 推荐入口函数名：`transform`
- 典型返回：记忆候选对象列表

### `plugin/executor.py`

- 放动作执行入口
- 推荐入口函数名：`run`
- 只负责动作执行本身，不负责绕开系统权限

### `plugin/skill.py`

- 放 Agent 技能入口
- 推荐入口函数名：`run`
- 只暴露受控能力，不直接改 Agent 主流程

### `README.md`

建议至少写清楚：

1. 插件做什么
2. 用到哪些权限
3. 需要什么外部配置
4. 怎么做最小验证
5. 明确不做什么

### `tests/`

第一版不强制你交一整套复杂自动化，但至少要能说明：

- `manifest` 是否自检过
- 入口是否能 import
- 核心样例 payload 是否跑过

## 5. 多能力插件怎么摆

如果一个插件同时有 `connector` 和 `memory-ingestor`，不要拆成两个插件目录。

最小可用方式就是：

```text
health_basic/
  manifest.json
  __init__.py
  connector.py
  ingestor.py
```

这也是当前 `apps/api-server/app/plugins/builtin/health_basic/` 的做法。

这样做的好处很实际：

- 一个插件 id 对应一组能力
- 一次开发、一次审核、一次注册
- 不会把同一来源的数据链路拆散

## 6. 动作插件怎么摆

动作插件目录通常更简单：

```text
homeassistant_device_action/
  manifest.json
  __init__.py
  executor.py
  README.md
```

如果是高风险动作，比如门锁，目录结构也不用特殊化。风险体现在：

- `manifest.json` 里的 `risk_level`
- 权限声明
- 后续注册表风险说明

不要为了高风险动作再发明一套特殊目录层级。那是给自己挖坑。

## 7. 现在不建议加的目录内容

这些内容第一版先别加，除非后面 Spec 明确扩范围：

- 自动安装脚本
- 远程下载器
- 沙箱配置目录
- 全量签名证书目录
- 前端市场展示资源包

原因不是这些东西永远不要，而是现在加了也没地方真正落地，还会误导第三方开发者。

## 8. 对照现有仓库的参考目录

直接可以看的现成例子：

- `apps/api-server/app/plugins/builtin/health_basic/`
- `apps/api-server/app/plugins/builtin/homeassistant_device_sync/`
- `apps/api-server/app/plugins/builtin/homeassistant_device_action/`
- `apps/api-server/app/plugins/builtin/homeassistant_door_lock_action/`

如果你写的目录结构和这些差太远，就先解释清楚为什么。解释不清楚，多半就是设计跑偏了。
