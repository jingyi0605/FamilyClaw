# 00-文档使用与维护原则

## 文档元数据

- 文档目的：说明插件开发者文档应该怎么读、怎么维护，避免每次插件系统小改动都把整套手册拖着一起改。
- 当前版本：v1.0
- 关联文档：`docs/开发者文档/插件开发/zh-CN/README.md`、`docs/开发者文档/插件开发/zh-CN/03-manifest字段规范.md`、`docs/开发者文档/插件开发/zh-CN/05-插件对接方式说明.md`、`docs/开发者文档/插件开发/zh-CN/11-插件配置接入说明.md`
- 修改记录：
  - `2026-03-16`：创建首版文档使用与维护原则。

这份文档只说一件事：

- 哪些文档应该稳定
- 哪些内容应该引用，不该到处复制

## 1. 先把两类文档分开

插件开发文档以后默认分两层：

### A. 稳定规范文档

这类文档尽量少改。

它们只负责讲这些内容：

- 概念边界
- 数据结构原则
- 不该做什么
- 接入顺序
- 哪份文档才是事实来源

典型例子：

- `01-插件开发总览.md`
- `03-manifest字段规范.md`
- `04-插件目录结构规范.md`
- `08-插件注册PR提交流程.md`
- `11-插件配置接入说明.md`

### B. 高频变化引用文档

这类文档可以改，但不要把同样的字段表复制进 5 份文档。

它们负责承载这些内容：

- 当前 API 路径
- 当前请求 / 响应样例
- 当前 UI 组件名
- 当前支持的字段类型、widget、页面路由
- 当前代码文件位置

典型例子：

- `05-插件对接方式说明.md`
- `10-计划任务接口与OpenAPI说明.md`
- `specs/004.2.3-插件配置协议与动态表单/docs/`
- `apps/api-server/app/modules/plugin/schemas.py`
- `apps/api-server/app/api/v1/endpoints/ai_config.py`

## 2. 一条死规矩：变化快的细节不要复制

以后文档维护按下面这条规则来：

- 稳定文档只解释“为什么”和“该遵守什么”
- 变化快的接口和样例只保留一份事实来源

比如插件配置协议：

- `manifest` 里为什么要写 `config_specs`，这是稳定规则
- 当前支持哪些字段类型、widget、读取 / 保存接口长什么样，这是高频变化实现

前者写在开发者手册里。

后者引用：

- `specs/004.2.3-插件配置协议与动态表单/docs/20260316-manifest-示例.md`
- `specs/004.2.3-插件配置协议与动态表单/docs/20260316-api-示例.md`
- `apps/api-server/app/modules/plugin/schemas.py`

不要再把这三份内容抄到别的手册里。

## 3. 更新顺序

以后插件系统有变动，按这个顺序改：

1. 先改代码里的事实来源
2. 再改对应 Spec 或样例文档
3. 最后只在稳定手册里补一句“规则变了什么”

如果只是接口路径、字段样例、支持范围变了：

- 不要先去大面积改总览文档
- 先改事实来源和引用文档

如果只是页面组件名、文件路径、API 示例变了：

- 只改高频变化引用文档
- 稳定规范文档尽量不动

## 4. 事实来源优先级

插件开发相关内容，优先级按这个顺序看：

1. 代码里的 schema / service / endpoint
2. 对应 Spec 和实现示例
3. 开发者手册

如果开发者手册和代码冲突：

- 以代码和当前 Spec 为准
- 然后回头修手册

不要反过来拿旧手册要求现有实现倒退。

## 5. 这套规则想解决什么

想解决的不是“文档写得漂不漂亮”，而是两个很现实的问题：

1. 插件系统一变，就有一堆手册一起过时
2. 开发者照着旧文档写，继续把字段定义散落到宿主页和业务代码里

所以以后文档结构的目标很简单：

- 稳定规则收紧
- 变化细节集中
- 事实来源唯一
