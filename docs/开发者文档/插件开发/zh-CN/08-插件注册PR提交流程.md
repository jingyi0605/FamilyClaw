# 08-插件注册PR提交流程

## 文档元数据

- 文档目的：告诉第三方开发者怎么把自己的插件提交到项目注册表，不用猜该交什么、怎么交、审核会看什么。
- 当前版本：v1.0
- 关联文档：`docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`、`docs/开发者文档/插件开发/zh-CN/06-从零开发一个可运行插件.md`、`docs/开发者文档/插件开发/zh-CN/07-插件测试与项目内运行验证.md`、`specs/004.3-插件开发规范与注册表/design.md`
- 修改记录：
  - `2026-03-13`：创建首版插件注册 PR 提交流程文档。

这份文档只回答一件事：

- 你的插件已经按规则做完了，下一步怎么提交注册 PR

你不需要理解项目后端怎么解析注册表，也不需要自己设计注册表格式。

你要做的是：按项目提供的模板准备材料，然后提 PR。

再补一句，免得有人继续理解错：

- 注册 PR 通过，不代表你的插件会被 HTTP 直接同步执行
- 当前正式运行语义仍然是“先创建后台任务，再由 worker 触发 runner 或内置执行层”

## 1. 先确认你是不是已经到“能提 PR”的阶段

别插件还没做完，就急着提注册。

提 PR 之前，至少先满足这几条：

1. 插件目录和 `manifest.json` 已经按规范写完。
2. `manifest.json` 里的入口路径能对应到真实代码。
3. 你声明的插件类型、权限、风险等级都已经写清楚。
4. 你有一份能给人看的插件说明文档，至少告诉别人这个插件做什么、怎么验证。
5. 你已经给插件准备好自己的依赖清单，比如 `requirements.txt`。
6. 你没有依赖当前明确不支持的能力，比如自动安装、沙箱执行。

上面过不去，就别提 PR。那不是审核严格，是你东西还没准备好。

## 2. 这一步到底要提交什么

你提交注册 PR，最少要准备 4 类材料：

### 2.1 插件源码仓库地址

- 必须能公开访问，或者至少能让审核者访问
- 仓库里要能看到真实插件代码
- 仓库里要能看到 `manifest.json`

### 2.2 插件说明文档地址

建议直接放仓库里的 `README.md`。

至少说清楚：

1. 插件做什么
2. 支持哪些能力类型
3. 申请了哪些权限
4. 风险等级是什么
5. 怎么做最小验证
6. 需要哪些 Python 依赖
7. 明确不做什么

### 2.3 注册表条目文件

项目会提供注册表目录和条目模板。

你需要做的是：

- 按模板新增一个插件条目文件
- 文件名通常和 `plugin_id` 一致
- 条目里的 `plugin_id`、权限、风险等级、类型，要和你的 `manifest.json` 对得上

一句话说：别自己发明格式，直接照模板填。

### 2.4 PR 说明

PR 描述里至少写清楚：

1. 这个插件解决什么问题
2. 插件仓库地址
3. 文档地址
4. 风险等级和原因
5. 你自己怎么验证过

## 3. 推荐提交流程

按这个顺序提，最省事：

1. 先把插件源码仓库整理好。
2. 检查仓库里的 `manifest.json`、README、最小测试说明是否齐全。
3. 检查 `requirements.txt` 和最小 runner 自检说明是否齐全。
   - 这里的 runner 自检是为了证明入口和依赖没问题，不是让你绕过后台任务主链。
4. fork 官方注册表仓库，或者在项目指定分支上提交修改。
5. 按模板新增注册表条目文件。
6. 提交 PR，并把插件仓库地址、文档地址、风险说明写清楚。
7. 等待审核；如果维护者要求补材料，就按意见补，不要在评论区打太极。

## 4. PR 描述建议模板

你提 PR 时，正文建议至少写成这样：

```md
## 插件信息

- plugin_id: `your-plugin-id`
- 插件名称: `你的插件名`
- 插件仓库: `https://github.com/your-org/your-plugin-repo`
- 插件文档: `https://github.com/your-org/your-plugin-repo/blob/main/README.md`

## 能力说明

- 插件类型: `connector` / `memory-ingestor` / `action` / `agent-skill`
- 权限声明: `...`
- 风险等级: `low` / `medium` / `high`

## 自检结果

- [ ] `manifest.json` 已自检
- [ ] 入口可 import
- [ ] README 已补齐
- [ ] 已说明最小验证步骤

## 补充说明

- 这个插件明确不做什么：`...`
- 审核时需要重点关注什么：`...`
```

## 5. 审核方会重点看什么

你不需要替审核方做决定，但你得知道对方会卡你哪里。

第一版最小审核清单主要看这几件事：

1. 注册表条目和 `manifest.json` 是否一致。
2. 插件仓库是否可访问，代码是否真实存在。
3. README 是否能让人看懂插件用途、权限和验证方式。
4. 风险等级是否和插件真实能力匹配。
5. 如果是动作插件，是否把风险和权限边界写清楚。
6. 是否偷偷依赖远程安装、远程执行、沙箱、签名平台这类当前没落地的东西。

## 6. 什么情况会被直接打回

下面这些情况，第一版就别指望“先合了再说”：

- `plugin_id` 和 `manifest.json` 对不上
- 仓库打不开，或者只看到空壳说明，看不到真实插件代码
- README 写得像空气，看不出插件到底干什么
- 风险等级明显报低
- 动作插件没把权限声明写清楚
- 条目里字段乱填，和模板不一致
- 明显越过当前边界，比如要求平台自动下载安装或远程执行

## 7. 你真正要记住的事

第三方开发者这一步只需要做两件事：

1. 把自己的插件做完整
2. 按项目模板把注册 PR 材料交齐

注册表内部怎么解析、后端以后怎么读取，那是项目维护者要管的，不是你现在要猜的东西。
