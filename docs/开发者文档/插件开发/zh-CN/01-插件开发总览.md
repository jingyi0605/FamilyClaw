# 01-插件开发总览

这份文档只回答 4 个问题：

1. 宿主和插件到底怎么分工
2. 现在正式支持哪些插件类型
3. `ai-provider` 现在该怎么理解
4. 第一次开发插件该怎么入手

## 1. 先认清边界

宿主只保留这些东西：

- 家庭、成员、房间、权限、审计
- 插件注册、启停、配置、调度、运行时
- 统一设备模型、实体模型和动作路由
- 统一仪表盘卡片规范
- 统一记忆对象标准、可见性、修订和 fallback
- 统一 AI 网关、路由、密钥、插件状态校验和降级

插件负责这些事：

- 接第三方 API、设备或平台
- 把第三方字段映射成宿主标准 DTO
- 做资源发现、状态刷新、动作执行
- 产出标准设备、实体、卡片快照和动作
- 对 `ai-provider` 来说，负责供应商声明、字段 schema、driver entrypoint、协议适配、流式输出、模型自动发现和厂商特例

一句话：

宿主负责标准和主权，插件负责接入和映射。

先把三条硬规则记住：

- `official` 和 `third_party` 插件不进宿主最终镜像，宿主也不能在 import 阶段静态依赖它们。
- 插件私有表、插件私有缓存和领域绑定关系只属于插件自己，不能注册进宿主核心 ORM。
- 插件交给宿主的 `entities` 必须已经是最终标准实体，宿主不再按天气、电费、健康等领域做读时修补。

## 2. 现在正式支持什么插件

### 普通插件类型

- `integration`
- `action`
- `agent-skill`
- `channel`
- `region-provider`
- `ai-provider`
- `locale-pack`
- `theme-pack`

### 独占槽位插件

- `memory_engine`
- `memory_provider`
- `context_engine`

## 3. `ai-provider` 现在该怎么理解

从 `004.8.1` 开始，`ai-provider` 不是“只有元数据、没有执行边界”的特殊类型。

统一口径是：

- 宿主保留统一 AI Gateway、路由、权限、审计、密钥、插件状态校验和降级
- `ai-provider` 插件负责供应商声明、字段 schema、driver entrypoint、协议适配、流式输出、模型自动发现、厂商特例

如果你在旧 spec 或旧报告里看到旧说法，那是历史背景，不是当前规则。

## 4. 第一次开发插件怎么做

最小顺序如下：

1. 先判断你的能力属于哪一种正式类型
2. 先写 `manifest.json`，不要先写业务代码
3. 声明正式 `types`、`permissions`、`entrypoints`、`capabilities`
4. 如果需要配置，走 `config_specs`，不要让宿主页硬编码字段
5. 按正式入口写实现
6. 用统一测试验证 manifest、执行链路和输出 DTO
