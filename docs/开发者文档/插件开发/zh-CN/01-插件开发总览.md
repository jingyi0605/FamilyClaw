## 1. 先说边界

`004.2` 已经把插件运行底座做好了，但这里要分清两层：

- 当前已经实现：内置插件同进程运行
- 后续第三方推荐：同容器子进程 runner 运行

所以现在最准确的说法不是“所有插件都按一个方式跑”，而是：

- 官方内置插件继续走当前同进程底座
- 第三方插件文档从现在开始按“独立插件目录 + 自己的依赖 + runner 调用协议”来写
- 注册表负责发现和审核，主服务不再默认直接 import 第三方插件代码
- 不管是内置还是第三方，对外执行主路径都先创建后台任务，不保证同步直接返回最终结果

现在明确不做这些事：

- 自动下载安装第三方代码
- 远程执行外部仓库插件代码
- 完整签名体系
- 运行时代码沙箱
- 插件市场前端页面

如果你的方案依赖上面这些东西，先停。那不是这份 Spec 现在要解决的问题。

## 2. 现在支持的插件类型

当前 manifest 正式支持的类型已经不只 4 类，但别自己再发明第 8 类。

### `connector`

- 用途：从外部系统读原始数据
- 典型场景：读健康数据、读设备状态
- 常见输出：`records` 列表，供系统保存原始记录

### `memory-ingestor`

- 用途：把原始记录转成标准记忆
- 典型场景：把步数、心率、温湿度转成 Observation
- 常见输出：标准记忆候选列表

### `action`

- 用途：对外执行动作
- 典型场景：设备控制
- 额外限制：要过权限检查；高风险动作还要人工确认

### `agent-skill`

- 用途：给 Agent 提供额外工具或规则
- 典型场景：让 Agent 统一调用某个受控工具
- 额外限制：插件只能提供能力，不能直接改 Agent 主流程

### `locale-pack`

- 用途：注册界面语言和文案资源
- 典型场景：补一个新的 locale 包
- 边界：不走 worker，不创建执行任务

### `channel`

- 用途：接入外部通讯平台
- 典型场景：Telegram、Discord、飞书、企业微信机器人
- 边界：平台账号配置和成员绑定走正式配置协议与通道链路，不要再让宿主页硬编码平台字段

### `region-provider`

- 用途：把地区目录或地区解析规则挂进系统
- 典型场景：新增中国大陆以外的地区提供方
- 边界：当前走受控 provider 协议和 runner，不直接进主进程

## 3. 第一版最小开发流程

按这个顺序做，别一上来就写花活：

1. 先确认你的能力属于上面 4 类中的哪一类，或者哪几类。
2. 创建一个独立插件目录，先放 `manifest.json`。
3. 按 `manifest` 声明入口、权限、风险等级和触发方式。
4. 再补入口代码文件，让入口能被 Python 模块路径定位到。
5. 用现有内置插件对照检查字段和目录有没有跑偏。
6. 用后台任务主链做一次真实联调，确认创建任务、worker 执行、查询结果都能对上。
7. 补最小测试和自检说明，再准备后续注册表提交材料。

如果插件需要管理端配置，再补一步：

8. 在 `manifest.json` 里正式声明 `config_specs`，不要把字段定义散落到宿主页或业务表。

一句话说，这一版先把“声明清楚、入口能跑、边界不越线”做好。

## 4. 最小插件长什么样

按第三方插件推荐形态，一个最小插件目录至少要有这些东西：

```text
your_plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    connector.py 或 ingestor.py 或 executor.py 或 skill.py
  tests/
```

不是每个文件都必须有，但下面这两个文件对第三方插件已经很重要：

- `requirements.txt`：声明插件自己的 Python 依赖，后续由 runner 环境安装
- `README.md`：给审核、排错和接入方看

你声明了什么类型，就必须有能对应到入口路径的代码文件。

比如你声明的是：

- `connector`：通常会有 `connector.py`
- `memory-ingestor`：通常会有 `ingestor.py`
- `action`：通常会有 `executor.py`
- `agent-skill`：通常会有 `skill.py` 或其他你自己命名的模块

文件名不是硬编码规则，但入口路径必须能被真实 import 到。这个是死规则，不是建议。

## 5. 开发时先看哪些现成样例

当前仓库里已经有几类内置插件，可以直接拿来对照：

- `apps/api-server/app/plugins/builtin/health_basic/`：读健康数据，再写标准记忆
- `apps/api-server/app/plugins/builtin/homeassistant_device_sync/`：读设备和环境数据
- `apps/api-server/app/plugins/builtin/homeassistant_device_action/`：普通设备动作插件
- `apps/api-server/app/plugins/builtin/homeassistant_door_lock_action/`：高风险动作插件

如果你看完规范还是不确定，就先按这些目录结构和 `manifest.json` 方式靠齐。

但要注意，它们现在还是“内置同进程插件”样例，不是第三方 runner 样例。

## 6. 现有运行时边界

这部分不是建议，是按现有代码来的：

- Agent 统一调用入口当前只开放 `connector` 和 `agent-skill`
- `action` 插件必须先过权限检查
- `risk_level=high` 的动作插件会进入人工确认流程，不会直接执行
- `manifest.json` 的 `id` 只能用小写字母、数字和中划线
- `entrypoints` 必须是“模块路径 + 函数名”格式
- 对外接口和 Agent 入口默认返回 `job_id`、`job_status` 这类任务信息，不承诺同步执行结果

第三方目标模式会多一条边界：

- 主服务只认协议和返回结果，不直接把第三方插件模块 import 进主进程

如果规范和代码打架，先以不破坏现有实现为准。别为了看起来完整，写出一套仓库根本不支持的规则。

## 7. 第一版先别做什么

下面这些做法现在都不合格：

- 在文档里引导开发者依赖远程安装
- 默认让插件自动执行高风险动作
- 要求开发者接一套还没存在的签名平台
- 写一套和当前 4 类插件模型不兼容的新类型
- 让插件直接改 Agent 主流程或越过权限控制

说白了，别把第一版做成一个没边界的开放执行系统。

## 8. 自检清单

你准备继续写细则之前，先拿这 8 条过一遍：

1. 我声明的插件类型，是否只用了当前支持的正式类型？
2. `manifest.json` 里有没有漏掉入口、权限、风险等级？
3. 入口路径能不能被真实 import 到？
4. 触发方式是不是只写了当前可控场景？
5. 如果是动作插件，权限和风险是不是写清楚了？
6. 如果要给 Agent 用，是否仍然走统一入口，而不是绕过系统？
7. 有没有偷偷依赖远程执行、自动安装、沙箱这些还没做的东西？
8. 别人不看源码，只看文档和样例，能不能知道你这个插件怎么起步？

这 8 条过不去，就别急着提 PR。
