# 02-插件开发环境与本地调试

## 文档元数据

- 文档目的：告诉第三方开发者怎么把插件开发环境准备好、代码该放哪里、最小怎么做本地调试，不用一上来就翻项目源码。
- 当前版本：v1.0
- 关联文档：`docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`、`docs/开发者文档/插件开发/zh-CN/05-插件对接方式说明.md`、`docs/开发者文档/插件开发/zh-CN/06-从零开发一个可运行插件.md`、`docs/开发者文档/插件开发/zh-CN/07-插件测试与项目内运行验证.md`、`apps/api-server/pyproject.toml`
- 修改记录：
  - `2026-03-13`：创建首版开发环境与本地调试文档。

这篇文档只解决 4 个问题：

1. 开发插件前，本地环境怎么准备
2. 插件代码现在应该放到哪里
3. 不启动一堆服务的前提下，最小怎么调试
4. 如果插件有自己的依赖，应该装到哪里

## 1. 先把现实边界说清楚

当前这里也要分两件事看：

- 当前已经实现：内置插件在主进程里 import 执行
- 第三方推荐目标：插件在同容器子进程 runner 里执行

所以第三方插件开发时，应该按 runner 模式准备代码和依赖。

按这个目标模式，插件代码要满足两个前提：

- 代码能被 Python 模块路径 import 到
- `manifest.json` 里的入口路径能指向真实函数

所以第三方插件开发时，最稳的方式是：

- 在独立插件目录里开发插件代码
- 给插件准备自己的 `requirements.txt`
- 在插件自己的 venv 或 runner 环境里先把入口跑通
- 再用项目提供的 runner 协议做联调

官方内置插件怎么做，是项目维护者的事；第三方开发者不用再把代码塞进主项目源码树里。

## 2. 本地环境最小要求

先满足这几个条件：

1. Python `3.11+`
2. 能用 `venv` 创建虚拟环境
3. 能在本地运行 `pip`
4. 有基本的 Git 使用能力

后端当前项目依赖可以直接从 `apps/api-server/pyproject.toml` 看到，核心依赖包括：

- `fastapi`
- `uvicorn`
- `sqlalchemy`
- `alembic`
- `websockets`

## 3. 推荐环境准备步骤

以下步骤都在 `apps/api-server/` 目录完成。

### 3.1 创建虚拟环境

```bash
python -m venv .venv
```

Windows 激活：

```bash
.venv\Scripts\activate
```

macOS / Linux 激活：

```bash
source .venv/bin/activate
```

### 3.2 安装当前后端项目依赖

```bash
pip install -e .
```

这样装的好处很直接：

- `app.*` 模块路径能直接 import
- 你改后端代码或插件代码时，不用每次重装包

### 3.3 给插件自己建独立环境

第三方插件不要默认把依赖装进主后端环境。

更合理的方式是：

```bash
python -m venv .plugin-venv
```

然后安装插件自己的依赖：

```bash
.plugin-venv\Scripts\pip install -r requirements.txt
```

或者在 macOS / Linux：

```bash
source .plugin-venv/bin/activate
pip install -r requirements.txt
```

这套环境后面就是 runner 要用的环境，不应该去污染主 API 进程环境。

## 4. 插件代码现在该放哪里

如果你是第三方插件开发者，当前推荐目录就是你自己的插件仓库。

一个最小目录建议长这样：

```text
your-plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    connector.py
    ingestor.py
    executor.py
    skill.py
  tests/
```

如果你是项目维护者，要继续维护当前内置插件，现有位置仍然是：

```text
apps/api-server/app/plugins/builtin/
```

比如你要做一个新插件，可以先放成：

```text
apps/api-server/app/plugins/builtin/your_plugin/
  __init__.py
  manifest.json
  connector.py
  ingestor.py
```

原因不是“以后只能这样”，而是现在运行时就是按这个可加载路径工作的。

如果你把代码放到仓库外面，或者放到一个没进 Python 路径的位置，`manifest.json` 里那串入口模块路径就会直接失效。

## 5. 开发时最小需要改哪些东西

一个最小可调试插件，通常先准备这几个文件：

1. `manifest.json`
2. `__init__.py`
3. 对应能力的入口代码文件，比如 `connector.py`
4. 如果要走完整数据链路，再补 `ingestor.py`

第一轮先别急着做复杂逻辑。

先把这三件事跑通：

- `manifest.json` 能通过校验
- 入口模块能 import
- 入口函数直接调用时能返回对的结构

## 6. 本地调试先从哪一层开始

别一开始就启动整套系统查半天日志。

按这个顺序调最稳：

### 第一层：静态检查

看这几件事：

1. `manifest.json` 字段是不是齐全
2. `entrypoints` 路径和真实文件是不是一致
3. 函数名是不是写对
4. `types` 和入口是不是一一对应

### 第二层：直接 import 调函数

先在 Python 解释器或测试里直接调插件入口函数。

比如：

```python
from app.plugins.builtin.health_basic.connector import sync

result = sync({"member_id": "demo-member"})
print(result)
```

这一步的目标不是跑完整系统，而是先确认：

- 模块路径没写错
- 函数能调用
- 返回结构基本正确

### 第三层：走项目里的真实服务函数

如果是 `connector + memory-ingestor` 插件，后面再去测：

- `execute_plugin()`
- `run_plugin_sync_pipeline()`
- Agent 相关桥接入口

这些都已经在现有测试里有样例。

## 8. 如果你真要联调 HTTP 接口，先知道边界

当前确实有 HTTP 入口能验证插件链路，比如：

- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/plugin-invocations`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/plugin-memory-checkpoint`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/action-plugin-invocations`

但这篇文档不要求你现在就启动开发服务器。

更稳的顺序是：

1. 先过静态检查
2. 再过服务层测试
3. 最后如果需要，再做 HTTP 联调

这样排查最省时间。

## 9. 本地调试时最常见的坑

### 模块路径写错

症状：入口加载失败。

常见原因：

- 目录名和模块名对不上
- `manifest.json` 里的模块路径没跟着改
- 函数名改了，但 entrypoint 还指旧名字

### 插件代码放错位置

症状：`import_module()` 找不到模块。

常见原因：

- 插件没放在当前可加载路径里
- 少了 `__init__.py`

### 一上来就联调整套链路

症状：报错一堆，不知道是 manifest 错、入口错、返回值错，还是数据库没准备好。

解决办法：

- 先拆成静态检查 -> 入口函数 -> 服务层链路 -> HTTP 联调

## 10. 一句话记住当前开发方式

对第三方开发者来说，当前最正确的插件开发姿势应该改成：

- 先在独立插件目录里把入口和依赖做通
- 再在插件自己的 venv 里做静态检查和最小测试
- 后续通过 runner 协议接到主项目里验证

当前仓库真正已经支持的仍然是内置插件同进程模式；第三方 runner 模式是现在开始要对齐的目标模式。
