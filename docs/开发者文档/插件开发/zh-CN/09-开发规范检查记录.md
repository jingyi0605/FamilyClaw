## 2. 这一步确认了什么

### 开发入口已经清楚

现在新开发者至少能知道：

- 第一步先看哪份文档
- 当前只支持哪 4 类插件
- 最小开发流程怎么走
- 哪些内容明确不做

### `manifest` 规则已经够用

现在已经写清楚：

- 哪些字段是第一版最小必填
- `id` 的字符规则
- `types` 的允许值
- `entrypoints` 的真实格式要求
- 动作插件和 Agent 插件的现有边界

### 目录结构已经够照着搭

现在已经写清楚：

- 一插件一目录
- `manifest.json` 放哪里
- 不同能力类型推荐放哪个模块文件
- 多能力插件怎么摆
- 动作插件怎么摆

### 对接方式已经补清楚

现在已经写清楚：

- 系统怎么通过 `entrypoints` 调插件
- 插件函数收到什么输入
- `connector` / `memory-ingestor` / `action` 应该返回什么
- 数据怎么进入原始记录、标准记忆和 Agent
- 当前哪些 HTTP 入口可用，哪些能力还没开放成平台接口

## 3. 用现有内置插件对照结果

### `health_basic`

- 对得上“多能力插件”规范
- 对得上 `connector + memory-ingestor` 目录模板
- 对得上推荐入口 `connector.sync`、`ingestor.transform`

### `homeassistant_device_sync`

- 对得上“读数据 + 写标准记忆”链路
- 对得上低风险、可手动/定时触发的说明

### `homeassistant_device_action`

- 对得上动作插件目录模板
- 对得上 `device.control` 权限声明
- 对得上 `medium` 风险动作边界

### `homeassistant_door_lock_action`

- 对得上高风险动作说明
- 对得上人工确认入口边界

## 4. 这一步明确还没做什么

这些东西还是故意没做，不是漏了：

- 注册表 schema 细则
- GitHub PR 提交流程
- 官方注册表 / 第三方注册表并存机制细节
- 自动审核脚本
- 市场前端页面

这些属于后续阶段，不能在阶段 1 偷偷塞进来。

## 5. 阶段 1 是否可以视为完成

可以。

原因很实际：

1. 第三方开发者已经能知道先看什么、先做什么。
2. `manifest` 和目录结构已经能直接对照现有插件实现。
3. 文档明确写出了第一版边界，没有把远程安装、沙箱、签名平台偷偷塞回范围。

阶段 2 可以开始做注册表 schema 和 GitHub 提交流程。
