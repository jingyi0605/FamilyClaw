这份文档回答 4 个问题：

1. 系统怎么调用插件
2. 插件能拿到什么输入
3. 插件该返回什么输出
4. 正式配置该怎么接

## 1. 先把大实话说清楚

现在对外主路径已经不是“调接口就同步跑完插件”。

现在更准确的说法是：

1. 调用方提交插件请求
2. 系统创建 `plugin_job`
3. worker 在后台执行插件
4. 调用方通过任务状态、通知或后续查询继续跟结果

`execute_plugin()`、`run_plugin_sync_pipeline()` 还在，但它们更偏内部执行层和测试工具，不应该再被写成对外默认语义。

## 2. 当前主链路

### 2.1 数据同步链路

这是最完整、也最稳定的一条链路：

1. 创建 `plugin_job`
2. worker 调用 `connector`
3. 插件返回 `records`
4. 系统把原始记录保存到 `plugin_raw_records`
5. `memory-ingestor` 把原始记录转成 Observation
6. 系统把 Observation 写进家庭记忆

这条链路适合：

- 健康数据
- 智能家居状态
- 其他“外部数据先入库、再沉淀成记忆”的场景

### 2.2 Agent 直接调用链路

当前只开放两类：

- `connector`
- `agent-skill`

重点不是“Agent 能不能调”，而是“Agent 只能调受控能力，不能直接随便调所有插件”。

### 2.3 动作执行链路

动作插件也是统一入口，但多了权限和风险控制：

1. 校验插件声明了 `action`
2. 校验插件声明了所需权限
3. 校验当前成员有执行权限
4. 如果风险等级是 `high`，先走人工确认
5. 通过后再创建后台任务

### 2.4 计划任务触发链路

计划任务现在也是正式链路，但顺序别写反：

1. 先在计划任务系统里建任务定义
2. 到点后创建 `scheduled_task_run`
3. 调度层把这次运行转成标准 `plugin_job`
4. 还是由现有插件 worker 执行

`manifest.triggers` 里写了 `schedule`，意思只是“允许被计划任务系统调用”，不是“插件自己注册定时器”。

## 3. 插件怎么被系统调用

系统是按 `manifest.entrypoints` 调用插件，不是按文件名硬编码。

当前正式会被执行的类型：

- `connector`
- `memory-ingestor`
- `action`
- `agent-skill`
- `channel`
- `region-provider`

不会走执行链路的类型：

- `locale-pack`

也就是说，`locale-pack` 负责注册资源，`channel` 和 `region-provider` 是正式扩展类型，不要继续把它们当成“页面特例”或者“临时旁路”。

## 4. 插件函数能拿到什么输入

当前统一入口传给插件的就是一个 `dict payload`。

插件函数推荐形态还是这类：

```python
def sync(payload: dict | None = None) -> dict:
    ...


def transform(payload: dict | None = None) -> list[dict]:
    ...


def run(payload: dict | None = None) -> dict:
    ...
```

这里的边界很死：

- 插件函数不应该直接拿数据库会话
- 插件函数不应该依赖请求对象
- 插件函数不应该返回不可序列化对象

如果插件声明了地区上下文读取，系统会把受控上下文注入到：

- `_system_context.region.household_context`

如果调用来自计划任务，还会附带调度来源字段，但那是给追踪和幂等用的，不是让插件反向改调度状态。

## 5. 插件该返回什么输出

### `connector`

- 返回 `dict`
- 至少有 `records: list[dict]`

### `memory-ingestor`

- 返回 `list[dict]`
- 每一项至少有：
  - `category`
  - `value`
  - `source_record_ref`

### `action`

- 返回普通 `dict`
- 重点是让调用方看得懂动作结果

### `agent-skill`

- 返回 JSON 可序列化结果
- 它是给 Agent 暴露能力，不是接管 Agent 主流程

### `channel`

- 按正式通道协议收发标准化消息
- 平台专属细节写在通道插件实现和通道模块里，不再靠页面常量做事实来源

### `region-provider`

- 按正式地区 provider 协议返回 JSON 结果
- 现在已经是正式扩展点，不是临时试验字段

## 6. 正式插件配置怎么接

这轮最重要的新边界就在这里。

以后插件只要需要让用户配置参数，就应该走正式配置协议，不要再走这些老路：

- 前端页面自己维护一份字段常量
- 某个业务表偷偷塞一坨 JSON 当唯一事实来源
- 把 secret 当普通字符串读出来

正式入口在插件 manifest：

- `config_specs`

每个配置作用域最少要有：

- `scope_type`
- `schema_version`
- `config_schema`
- `ui_schema`

这轮只支持两个正式作用域：

- `plugin`
- `channel_account`

稳定规则只有 5 条：

1. 字段定义只保留在插件 manifest。
2. 宿主页只负责读协议、渲染表单、提交配置。
3. secret 字段读取时绝不回显明文。
4. 保存时没提交某个 secret 字段，表示保留旧值。
5. 如果要清空 secret，必须显式走 `clear_secret_fields`。

更多细节不要复制到这份手册里，直接看：

- `specs/004.2.3-插件配置协议与动态表单/docs/20260316-manifest-示例.md`
- `specs/004.2.3-插件配置协议与动态表单/docs/20260316-api-示例.md`

## 7. 通道插件为什么要特别说明

因为通道插件是这轮第一批迁移对象，最容易把旧事实来源带回来。

现在要明确：

- 正式字段来源是通道插件自己的 `manifest.config_specs`
- `channel_account` 作用域通过统一插件配置 API 读取和保存
- `channel_plugin_accounts.config_json` 只是兼容运行链路的副本，不再是唯一事实来源
- `SettingsChannelAccessPage` 现在应该按协议驱动动态表单
- `PluginDetailDrawer` 已经有统一的插件级配置入口

如果后面文档还把“channel 配置字段列表”写死在页面层，那就是把同一个坑重新挖一遍。

## 9. 一句话总结

当前插件接入 FamilyClaw，不是“挂一个外部 API 就完事”，而是：

- 先按 `manifest` 声明正式能力和正式配置协议
- 再通过统一任务链路被系统调用
- 数据先落原始记录，再转标准记忆
- 页面按协议驱动配置，不再按插件 id 写死字段
