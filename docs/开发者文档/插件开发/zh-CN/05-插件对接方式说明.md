# 05-插件对接方式说明

这份文档只讲稳定的调用规则。

精确 DTO 看这些事实来源：

- `apps/api-server/app/modules/plugin/schemas.py`
- `apps/api-server/app/modules/integration/schemas.py`
- `apps/api-server/app/modules/plugin/service.py`
- `apps/api-server/app/modules/integration/service.py`
- `apps/api-server/app/modules/ai_gateway/provider_driver.py`

## 1. 现在的主链路是什么

### `integration` 主链路

1. 宿主读取 manifest
2. 校验插件类型、入口和能力声明
3. 宿主注入运行时上下文和正式配置
4. 调用 `integration.refresh`
5. 插件返回标准刷新结果
6. 宿主统一校验、去重、落库、更新状态
7. 宿主写设备、实体、动作、卡片和错误状态

### `action` 主链路

1. 宿主校验插件已启用
2. 校验动作权限和风险级别
3. 必要时走人工确认
4. 调用动作入口

### `ai-provider` 主链路

1. 宿主根据 provider profile 找到对应 `ai-provider` 插件
2. 宿主读取 `entrypoints.ai_provider`
3. 宿主加载 provider driver
4. 宿主调用 `invoke`、`ainvoke` 或 `stream`
5. 宿主统一处理权限、审计、密钥、插件状态和结果收口

### 记忆槽位主链路

1. 宿主激活槽位插件
2. 宿主调用标准槽位契约
3. 宿主保留可见性、修订、审计和 fallback

## 2. 插件函数收到什么

普通执行入口统一吃 JSON 可序列化 `payload`。

不要依赖：

- 请求对象
- ORM Session
- 不可序列化对象

如果插件需要坐标上下文，正式入口是：

- `_system_context.region.household_context.coordinate`

不要从 `city` 文本重新猜点。

### plugin-managed integration 的运行时补充

普通第三方插件不要假设宿主一定会给你 ORM Session。

但当前正式链路里，宿主已经为“集成实例创建后立即同步”开放了这两个运行时入口：

- `_system_context.device_integration.database_url`
- `_system_context.device_control.database_url`
- `_system_context.integration_runtime.db_session`（仅内置插件同进程链路可用）

怎么理解：

- `device_integration.database_url` / `device_control.database_url` 是按执行入口注入的宿主数据库连接串，插件可以在需要时自己开短生命周期连接
- `db_session` 是宿主给内置插件的同进程优化，不是第三方插件通用特权

## 3. `integration.refresh` 应该返回什么

至少应该返回一个 JSON 对象。

第一版重点字段：

- `instance_status`
- `devices`
- `entities`
- `discoveries`
- `actions`
- `dashboard_snapshots`
- `diagnostics`
- `records`
- `observations`
- `message`
- `summary`
- `items`
- `sync_scope`

这里的 `records` 用于原始记录落库。

这里的 `observations` 用于让宿主统一写标准观察类记忆。

这里的 `dashboard_snapshots` 用于让宿主统一写插件卡片快照。

这里的 `instance_status` 用于让宿主更新实例级成功、降级和错误状态。

补一句很关键的话：

- 面向插件作者的正式输出字段现在用 `dashboard_snapshots`
- 宿主内部如果还有 `card_snapshots` 这种旧命名，那是内部遗留，不要继续照着写

### 设备、实体、卡片谁说了算

插件只能产出标准 DTO。

宿主负责：

- 校验
- 去重
- 落库
- 状态合并
- 错误语义

插件不能自己发明一套设备协议、实体协议或卡片协议。

再补三条硬规则：

- 插件返回的 `entities` 必须已经是最终标准实体，宿主不再按天气、电费、健康等领域做读时归一化。
- 插件私有表、插件私有缓存和插件私有绑定关系只属于插件自己；宿主不导入、不注册、不直接查询插件私有 ORM 模型，宿主核心 Alembic 也不直接管理这些表。
- 运行态标准实体进入宿主后，正式事实源就是宿主公共标准实体承载层；当前正式实现是宿主公共表 `device_entities`。
- `DeviceBinding.capabilities` 现在只保留绑定摘要、主实体标识、实体 ID 列表、能力标签和轻量元数据，不再承载完整运行态实体快照。
- 如果你在兼容旧数据时还能看到 `capabilities.entities`，把它当旧数据兜底，而不是给新插件照着实现。

私有迁移怎么落地，也按下面这套执行：

- `builtin` / `official` 插件如果需要私有表，目录约定就是插件根目录下的 `migrations/`，并且必须包含 `env.py` 与版本脚本。
- 宿主当前会在插件执行准备阶段先补齐这类插件的私有 Alembic，再进入正式执行链路。
- 插件私有 ORM 模型可以复用宿主共享 `Base`，但不得再把模型挂回宿主核心聚合入口；尤其不要再把插件模型注册到 `app.db.models`。
- 如果你自己搞独立 metadata，就自己承担跨宿主公共表外键、flush 顺序和 metadata 组织问题；宿主不会为单个领域再加 ORM 特判。

### 卡片快照当前开放到什么程度

插件现在不只是能“塞一坨 payload”。

宿主已经正式支持：

- manifest 里声明固定 `card_key`
- manifest 里声明动态 `card_key_prefix`
- snapshot 自带 `title`
- snapshot 自带 `subtitle`
- `payload.card_kind`
- `payload.card_state`

像天气这种按设备出卡片的插件，还可以在 payload 里带：

- `device_id`
- `binding_type`

这些字段是给前端和页面交互用的业务透传字段，不是让插件自己发明一套新卡片协议。

### 卡片 payload 里的用户可见字段怎么写

这里补的是硬规则，不是建议。

- 宿主前端只认标准字段，不认某个插件的私有字段名，更不会按 `official_weather.*` 这种插件 key 写专用分支。
- 只要是用户能看到的卡片明细、摘要、页脚、状态标签，都必须放进标准字段里。
- 最小可用字段是：`label`、`label_key`、`value`、`value_display`、`value_type`。
- 需要单位时再带 `unit`；需要多项列表时统一用 `items`，或者宿主已经开放的 `detail_items`、`footer_items`。
- `label` 是直接给人看的文本，`label_key` 是插件自己的词典 key；推荐两者同时提供，这样词典丢失时还能回退到原文。
- `value` 是原始值，`value_display` 是最终展示值；插件要自己把风向、摘要、格式化时间、带单位文本整理好，宿主不会再做天气专用、电费专用之类的二次加工。
- `value_type` 只允许表达标准展示语义，比如 `text`、`number`、`percentage`、`datetime`、`direction`；不要再发明一套插件私有渲染协议。
- 宿主统一按这个顺序渲染：优先 `label_key`，命不中就退回 `label`，最后才用安全默认值；宿主不会把 key 原样展示给用户。
- 如果你的卡片和标准实体展示的是同一组信息，二者必须来自同一套词典和同一套最终展示值，不能一半靠插件，一半靠宿主猜。

## 4. AI 供应商驱动谁说了算

`ai-provider` 插件自己负责：

- 供应商声明
- branding 资源
- 说明文案资源
- 字段 schema
- config_ui
- 请求适配
- 流式输出
- 模型自动发现
- 厂商特例

宿主只负责：

- 统一网关入口
- 路由和 fallback
- 密钥保护
- 插件启停和 household 校验
- 审计和统一错误语义

如果你要做“填完 URL / API Key 后自动刷新模型列表”，也按这个边界走。

当前正式做法只有一句话：

- 宿主只提供一个通用的模型发现入口
- 宿主按 `adapter_code` 找到对应 `ai-provider` 插件
- 真正访问哪个 endpoint、要不要带鉴权、怎么解析返回，全部由插件 driver 自己负责

再补三条现在必须遵守的硬规则：

- 前端 Logo、说明文案、字段分组、字段隐藏规则、动作按钮和模型发现提示语都不能再写回核心前端目录，必须从 `ai-provider` manifest 和插件资源里来。
- `ai-provider` manifest 现在至少要覆盖三类前端契约：`branding`、`config_ui`、`model_discovery`。
- 如果某段逻辑只有某个供应商自己才知道，那它就属于插件；宿主只能保留通用调用契约、统一入口、路由、fallback、状态、审计和错误收口。

别再把下面这些东西写回宿主核心：

- `Ollama` 用 `/api/tags`
- `LM Studio` 用 `/v1/models`
- `LocalAI` 的 `Authorization` 处理
- 某家返回字段叫 `models` 还是 `data`
- 某家供应商的 Logo SVG
- 某家供应商的说明文案 key
- `field.key === 'model_name'` 这种页面分支

## 5. 自动链路和错误语义

任何真正执行插件的入口，都必须走统一可用性检查。

禁用时统一语义：

- HTTP `409`
- `error_code=plugin_disabled`

外部依赖失败时：

- 插件返回失败或降级信息
- 宿主更新实例或设备错误状态
- 不允许把主链路直接打死

部署边界也别写错：

- `official` 和 `third_party` 插件不进宿主最终镜像
- 宿主不得在 import 阶段静态依赖它们
- 插件目录缺失时，应该是插件不可用，不应该是宿主起不来
- `official` 目录是运行时托管目录，不属于宿主源码组成部分
