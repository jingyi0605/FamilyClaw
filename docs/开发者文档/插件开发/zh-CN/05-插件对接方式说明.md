这份文档只解决一个核心问题：插件到底怎么和 FamilyClaw 对接。

前面那些目录结构、`manifest` 字段都只是壳。真正能让开发者动手的，是这几件事：

1. 系统怎么调用插件
2. 插件能拿到什么输入
3. 插件该返回什么输出
4. 数据怎么从插件进系统，再进记忆，再给 Agent 用

先把一句大实话写在前面：

- 现在对外主路径已经不是“接口直接同步执行插件”
- 现在更准确的说法是“接口或 Agent 入口先创建后台任务，再由 worker 执行插件”
- `execute_plugin()`、`run_plugin_sync_pipeline()` 这些函数还会继续存在，但它们更偏内部执行层和测试层工具

## 1. 先把大实话说清楚

这里也先说清两个状态：

- 当前已实现：内置插件同进程运行
- 第三方目标：同容器子进程 runner 运行

所以第三方插件对接文档，后面默认按 runner 模式讲。

主服务和第三方插件的职责分开如下：

- 主服务负责读取注册表、校验 manifest、做权限、审计、记忆写入和 Agent 桥接
- runner 负责拉起插件进程、准备 Python 环境、执行入口函数
- 插件只负责处理 payload 并返回 JSON 可序列化结果

说白了，第三方插件以后不是进主进程，而是进 runner。

## 2. 当前支持的 3 条对接主链路

### 2.1 数据同步链路

这是最完整的一条链路，也是现在最像“插件接入”的主线。

链路顺序：

1. 调用方先创建 `plugin_job`
2. worker 领取任务并调用 `connector`
3. 插件返回原始数据 `records`
4. 系统把这些 `records` 保存到 `plugin_raw_records`
5. `memory-ingestor` 插件把原始记录转成标准 Observation
6. 系统把 Observation 写入家庭记忆
7. 系统更新任务状态、尝试记录、通知记录
8. Agent 后续从记忆里读取这些内容

对应代码：

- 调用和总编排：`apps/api-server/app/modules/plugin/service.py:303`
- 原始记录保存：`apps/api-server/app/modules/plugin/service.py:190`
- 原始记录转记忆：`apps/api-server/app/modules/plugin/service.py:247`
- Observation 写入：`apps/api-server/app/modules/memory/service.py:756`

### 2.2 Agent 直接调用链路

这条链路用于 Agent 直接调用插件能力。

当前只开放两类：

- `connector`
- `agent-skill`

对应代码：

- `apps/api-server/app/modules/plugin/agent_bridge.py:29`
- 限制条件：`apps/api-server/app/modules/plugin/agent_bridge.py:103`

一句话说，Agent 不能直接随便调所有插件；现在只允许调受控的读能力和技能能力，而且当前返回的主语义已经是“已入队任务”。

### 2.3 动作执行链路

动作插件也是统一入口，但限制更严。

流程是：

1. 请求动作插件
2. 系统检查插件是否声明了 `action`
3. 系统检查插件是否声明了所需权限
4. 系统检查当前成员有没有执行权限
5. 如果风险等级是 `high`，先走人工确认
6. 确认通过后创建后台任务
7. worker 再真正执行插件

对应代码：

- 动作入口：`apps/api-server/app/modules/plugin/agent_bridge.py:130`
- 权限检查：`apps/api-server/app/modules/plugin/agent_bridge.py:223`
- 高风险确认：`apps/api-server/app/modules/plugin/agent_bridge.py:282`

## 3. 插件是怎么被系统调用的

先补一句例外：不是所有插件都会被执行。

- `connector` / `memory-ingestor` / `action` / `agent-skill` 这 4 类，才有运行时入口
- `locale-pack` 不会进 worker，也不会 import Python 入口函数
- 它只会被注册中心读出来，给前端或配置接口提供可用语言列表和资源位置

系统不是靠约定文件名硬编码调用，而是靠 `manifest.json` 里的 `entrypoints`。

例如：

```json
{
  "entrypoints": {
    "connector": "app.plugins.builtin.health_basic.connector.sync",
    "memory_ingestor": "app.plugins.builtin.health_basic.ingestor.transform"
  }
}
```

当前内置插件模式下，系统会做两件事：

1. 按字符串拆出模块路径和函数名
2. `import_module()` 加载模块，再调用目标函数

对应代码：

- 入口解析：`apps/api-server/app/modules/plugin/service.py:431`
- 类型到入口 key 的映射：`apps/api-server/app/modules/plugin/service.py:416`

第三方 runner 模式下，runner 也要做同样两件事，只是 import 发生在子进程里。

所以插件作者真正要保证的是：

- `entrypoints` 写对
- 对应 Python 模块真实存在
- 目标函数可调用

但别再误会一件事：

- 入口函数可调用，不等于系统会同步把它跑完再回给你
- 现在系统主链路会先把请求落成 `plugin_job`

如果你做的是语言包插件，那上面这一整段“调用执行”都可以跳过。语言包插件关注的是：

1. `manifest.locales` 写清楚
2. 文案资源文件存在
3. 前端语言注册表能识别并加载它

## 4. 插件函数能拿到什么输入

当前统一入口传给插件的就是一个 `dict payload`。

比如底层运行请求结构：

- `plugin_id`
- `plugin_type`
- `payload`
- `trigger`

对应结构：`apps/api-server/app/modules/plugin/schemas.py:154`

但插件函数本身实际收到的是：

```python
def sync(payload: dict | None = None) -> dict:
    ...
```

也就是说，不管是当前内置模式，还是后续 runner 模式，插件函数都不应该直接拿到数据库会话、请求对象、用户上下文这些重对象。

当前你能依赖的就是：

- 调用方传进来的 `payload`
- 你自己插件内部的配置和逻辑

如果你的插件在 `manifest.json` 里声明了：

```json
{
  "capabilities": {
    "context_reads": {
      "household_region_context": true
    }
  }
}
```

系统还会额外注入一段受控上下文：

```json
{
  "_system_context": {
    "region": {
      "entry": "region.resolve_household_context",
      "household_context": {
        "status": "configured",
        "provider_code": "builtin.cn-mainland",
        "country_code": "CN",
        "region_code": "110105",
        "admin_level": "district",
        "province": {"code": "110000", "name": "北京市"},
        "city": {"code": "110100", "name": "北京市"},
        "district": {"code": "110105", "name": "朝阳区"},
        "display_name": "北京市 朝阳区",
        "timezone": "Asia/Shanghai"
      }
    }
  }
}
```

这就是这轮正式给插件开放的地区入口。

别再去读 `households.city` 猜地区，那是兼容展示字段，不是正式上下文。

这很朴素，但边界清楚。

系统返回给调用方的主信息则要按两层理解：

- 第一层：任务信息，比如 `job_id`、`job_status`、`queued`
- 第二层：真正执行结果，要通过任务查询、通知、响应继续跟进

## 4.1 插件函数签名规范

当前插件入口函数没有复杂协议，直接按 Python 可调用函数来写。

推荐签名：

```python
def sync(payload: dict | None = None) -> dict:
    ...


def transform(payload: dict | None = None) -> list[dict]:
    ...


def run(payload: dict | None = None) -> dict:
    ...
```

按类型拆开看：

| 插件类型 | 推荐函数名 | 推荐签名 | 返回类型 |
| --- | --- | --- | --- |
| `connector` | `sync` | `def sync(payload: dict | None = None) -> dict` | `dict` |
| `memory-ingestor` | `transform` | `def transform(payload: dict | None = None) -> list[dict]` | `list[dict]` |
| `action` | `run` | `def run(payload: dict | None = None) -> dict` | `dict` |
| `agent-skill` | `run` | `def run(payload: dict | None = None) -> dict` | `dict` |

这里有 4 条死规则：

1. 入口必须是可调用函数。
2. 参数位置至少要能接受一个 `payload`。
3. 返回值必须是 JSON 可序列化的基础结构，别返回数据库对象、Session、复杂类实例。
4. `memory-ingestor` 必须返回列表，不是字典，不是生成器。
5. 读取家庭地区时，只能用 `_system_context.region.household_context`，不要自己猜 `city`。

别搞这些花活：

- 必须依赖额外上下文对象才能跑的签名
- 强依赖 `*args` / `**kwargs` 才能分辨行为
- 返回自定义类实例
- 在入口函数里偷偷操作数据库事务

## 5. 插件该返回什么输出

### 5.0 如果要读家庭地区，该怎么写

最小示例：

```python
def sync(payload=None):
    data = payload or {}
    system_context = data.get("_system_context", {})
    region = system_context.get("region", {})
    household_region = region.get("household_context", {})
    return {
        "display_name": household_region.get("display_name"),
        "region_code": household_region.get("region_code"),
        "records": [],
    }
```

这段代码只做一件事：从正式入口拿家庭地区上下文。

### 5.0.1 当前可用边界

- 这轮正式可用：插件读取家庭地区上下文
- 这轮正式可用：第三方地区 provider 通过 runner 子进程挂进主服务

原因很简单：

- 读取上下文只需要受控注入，不会破坏现有地区目录主链
- 第三方地区 provider 现在已经补了生命周期装配、注册表接入和 runner 隔离

### 5.0.2 如果插件要作为地区 provider 运行，该怎么写

manifest 最少要写成这样：

```json
{
  "types": ["region-provider"],
  "entrypoints": {
    "region_provider": "plugin.region_provider.handle"
  },
  "capabilities": {
    "region_provider": {
      "provider_code": "plugin.jp-sample",
      "country_codes": ["JP"],
      "entrypoint": "plugin.region_provider.handle",
      "reserved": false
    }
  }
}
```

入口函数会收到一个简单 payload：

```json
{
  "operation": "list_children",
  "provider_code": "plugin.jp-sample",
  "country_code": "JP",
  "parent_region_code": null,
  "admin_level": "province"
}
```

当前主服务会调用 4 个操作：

- `list_children`
- `search`
- `resolve`
- `build_snapshot`

你别搞花活，按这个协议返回 JSON 就行。

最小示例：

```python
def handle(payload=None):
    data = payload or {}
    operation = data.get("operation")
    if operation == "list_children":
        return []
    if operation == "search":
        return []
    if operation == "resolve":
        return None
    if operation == "build_snapshot":
        return {}
    raise ValueError(f"unsupported operation: {operation}")
```

这套 provider 插件现在真的会被主服务加载，但仍然走 runner 子进程隔离，不会直接 import 进主进程。

推荐返回一个 `dict`，里面至少包含：

- `records`: `list[dict]`

```python
return {
```

#### `connector records` 原始记录字段表

系统真正保存原始记录时，不是把 `connector` 返回项原样整条塞进库里，而是会拆成统一结构。

可以从两个层面理解：

##### A. `connector` 返回的单条原始数据项建议字段

| 字段 | 类型 | 必填 | 说明 | 进入系统后怎么用 |
| --- | --- | --- | --- | --- |
| `record_type` | string | 推荐 | 原始记录类别 | 优先作为系统里的 `record_type` |
| `type` | string | 可选 | 备用类别字段 | 如果没写 `record_type`，系统会用它 |
| `category` | string | 可选 | 备用类别字段 | 前两个都没有时再看它 |
| `source_ref` | string | 可选 | 外部对象引用 | 优先作为系统里的 `source_ref` |
| `external_member_id` | string | 可选 | 外部成员 id | `source_ref` 的候选来源 |
| `external_device_id` | string | 可选 | 外部设备 id | 方便后续追踪设备来源 |
| `device` | string | 可选 | 设备业务引用 | `source_ref` 的候选来源 |
| `member_id` | string | 可选 | 成员业务引用 | `source_ref` 的候选来源 |
| `captured_at` | string | 可选 | 原始采集时间 | 优先作为记录时间 |
| `observed_at` | string | 可选 | 备用时间字段 | `captured_at` 缺失时使用 |
| `occurred_at` | string | 可选 | 备用时间字段 | 再次降级使用 |
| `date` | string | 可选 | 最后兜底时间字段 | 再次降级使用 |
| `value` | any | 推荐 | 原始值 | 进入 `payload`，供 ingestor 转换 |
| `unit` | string | 推荐 | 单位 | 进入 `payload`，供 ingestor 转换 |
| 其他扩展字段 | any | 可选 | 比如 `room_id`、`status`、`vendor_name` | 原样进入 `payload` |

##### B. 系统保存后的标准原始记录结构

系统会把每条原始数据项转成下面这种统一结构：

| 字段 | 类型 | 来源 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 系统生成 | 原始记录 id |
| `household_id` | string | 调用上下文 | 家庭 id |
| `plugin_id` | string | 调用上下文 | 来源插件 id |
| `run_id` | string | 系统生成 | 同一次插件运行 id |
| `trigger` | string | 调用上下文 | 触发方式 |
| `record_type` | string | 从 `record_type` / `type` / `category` 推断 | 原始记录类别 |
| `source_ref` | string \| null | 从 `source_ref` / `external_member_id` / `external_device_id` / `device` / `member_id` 推断 | 观察对象引用 |
| `payload` | object | 原始返回项本体 | 后续 `memory-ingestor` 主要读取这里 |
| `captured_at` | string | 从 `captured_at` / `observed_at` / `occurred_at` / `date` 推断 | 记录时间 |
| `created_at` | string | 系统生成 | 入库时间 |

对应实现：

- 原始记录保存：`apps/api-server/app/modules/plugin/service.py:190`
- `record_type` 推断：`apps/api-server/app/modules/plugin/service.py:476`
- `source_ref` 推断：`apps/api-server/app/modules/plugin/service.py:484`
- `captured_at` 推断：`apps/api-server/app/modules/plugin/service.py:492`

### 5.2 `memory-ingestor` 应该返回什么

必须返回 `list[dict]`。

每一项代表一条标准 Observation 候选。

硬要求：

- 必须有 `source_record_ref`
- 必须有 `category`
- 必须有 `value`

现有示例：`apps/api-server/app/plugins/builtin/health_basic/ingestor.py:4`

系统要求见：

- `apps/api-server/app/modules/plugin/service.py:280`
- `apps/api-server/app/modules/plugin/service.py:287`
- `apps/api-server/app/modules/memory/service.py:772`

最小 Observation 示例：

```python
{
    "type": "Observation",
    "subject_type": "Person",
    "subject_id": "member-1",
    "category": "daily_steps",
    "value": 8421,
    "unit": "count",
    "observed_at": "2026-03-12T07:00:00Z",
    "source_plugin_id": "health-basic-reader",
    "source_record_ref": "raw-record-id"
}
```

#### Observation 完整字段表

`memory-ingestor` 返回的每一项，建议按下表组织：

| 字段 | 类型 | 必填 | 说明 | 当前用途 |
| --- | --- | --- | --- | --- |
| `type` | string | 推荐 | 固定写 `Observation` | 便于人看和后续扩展 |
| `subject_type` | string | 推荐 | 观察对象类型，如 `Person`、`Device` | 参与后续记忆内容组织 |
| `subject_id` | string | 推荐 | 观察对象 id | 用于定位观察对象 |
| `category` | string | 是 | 观察类别，如 `daily_steps`、`room_temperature` | 系统写记忆时必需 |
| `value` | any | 是 | 观察值 | 系统写记忆时必需 |
| `unit` | string | 推荐 | 单位，如 `count`、`celsius` | 用于摘要和展示 |
| `observed_at` | string | 推荐 | 观察时间 | 进入记忆的时间字段 |
| `source_plugin_id` | string | 推荐 | 来源插件 id | 便于追踪来源 |
| `source_record_ref` | string | 是 | 对应原始记录 id | 系统去重和追踪依赖它 |

当前真正的硬要求只有 3 个：

- `category`
- `value`
- `source_record_ref`

但如果你把其他字段都省了，虽然能勉强跑，后面追踪、展示、摘要都会很难看。

一个更完整的真实样例：

```python
{
    "type": "Observation",
    "subject_type": "Device",
    "subject_id": "living-room-sensor",
    "category": "room_temperature",
    "value": 23.5,
    "unit": "celsius",
    "observed_at": "2026-03-12T08:30:00Z",
    "source_plugin_id": "homeassistant-device-sync",
    "source_record_ref": "raw-record-123"
}
```

对应现有实现：`apps/api-server/app/plugins/builtin/homeassistant_device_sync/ingestor.py:4`

### 5.3 `action` 应该返回什么

当前动作插件返回普通 `dict` 即可。

现有示例：`apps/api-server/app/plugins/builtin/homeassistant_device_action/executor.py:4`

常见字段：

- `source`
- `mode`
- `target_ref`
- `action_name`
- `executed`
- `received_payload`

这里重点不是字段多漂亮，而是动作结果要能被调用方看懂。

### 5.4 `agent-skill` 应该返回什么

当前仓库还没有现成内置样例，但调用模型和 `connector` 一样，返回 JSON 可序列化的结果即可。

注意边界：

- 它是给 Agent 暴露能力
- 不是让插件直接接管 Agent 主流程

## 7. 当前能拿到哪些系统信息

插件当前能直接拿到的，不是“整个系统上下文”，而是受控输入和后续转换链路。

### 插件调用时直接拿到

- 调用方传入的 `payload`
- 当前触发语义对应的插件类型

### 系统替你补做的事

- 保存原始记录
- 生成插件运行记录
- 生成标准 Observation 记忆
- 记录审计日志
- 做 Agent 动作权限检查
- 对高风险动作挂人工确认

### 现在还没有直接开放给插件的东西

- 通用数据库读写 SDK
- 插件专用 HTTP SDK
- 自动配置注入协议
- 远程插件回调机制
- 沙箱隔离运行环境

所以别按“第三方云插件平台”的思路来写。当前不是那个模型。

## 8. 数据到底通过什么方式与项目同步

现在最稳、最正确的同步方式就一种：

### 方式 A：通过 `connector + memory-ingestor` 同步

步骤：

1. `connector` 拉外部数据
2. 返回 `records`
3. 系统保存到 `plugin_raw_records`
4. `memory-ingestor` 把原始记录转为 Observation
5. 系统写入 `memory_cards`
6. Agent 和业务模块读取这些记忆

这条链路是当前后端已经做好的主路。

### 方式 B：通过 Agent 触发同步检查点

如果你要从公开接口触发同步，当前最现成的是：

- `plugin-memory-checkpoint`

它本质上还是方式 A，只是入口包了一层 Agent 场景。

### 现在不支持的同步方式

- 插件自己直写数据库表
- 远程仓库插件主动回调本系统执行代码
- 自动下载后在服务器执行第三方插件
- 让插件绕过原始记录，直接随便写任意记忆类型

## 9. 开发插件时，最少要对接到什么程度才算合格

最小合格线很简单：

1. `manifest.json` 能被系统识别
2. 入口函数能被 import 到
3. `connector` 能返回有效 `records`
4. 如果声明了 `memory-ingestor`，就能把原始记录转成有效 Observation
5. 如果声明了 `action`，权限和风险等级就得说清楚
6. 不依赖自动安装、远程执行、沙箱这些当前没做的机制

## 10. 一句话总结

当前插件接入 FamilyClaw 的方式不是“挂一个外部 API 就完事”，而是：

- 先按 `manifest` 注册成可加载插件
- 再通过统一入口被系统调用
- 数据先落原始记录，再转标准记忆
- Agent 和动作能力再走受控桥接

这才是当前真实可用的对接方式。
