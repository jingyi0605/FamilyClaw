# 家庭版 OpenClaw MVP 与完整功能开发计划 v0.1

## 0. 文档目的

本文档基于 `docs/家庭版OpenClaw-详细设计方案-v0.1.md` 整理，目标是把“产品设计方案”进一步转化为一套可执行的开发计划，覆盖：

- MVP 的目标、边界、模块优先级
- 12~16 周的首期开发路线
- 从 `v0.1` 到 `v2.0` 的版本演进路径
- 技术落地建议、团队分工与验收标准

本文档适合用于：

- 项目立项与排期评估
- MVP 范围冻结
- 技术选型与模块拆解
- 团队协作与阶段验收

## 1. 规划原则

### 1.1 产品原则

家庭版 OpenClaw 的核心不是“控制更多设备”，而是先建立一个真正以家庭成员为主语的家庭 AI 中枢。

因此首期开发必须坚持以下原则：

1. **先做家庭 AI 中枢，不做全屋智能平台替代品**
2. **先打通 3 个高价值场景，不追求大而全**
3. **底层借力成熟生态，核心能力自研**
4. **先做隐私与权限底座，再扩展高级能力**
5. **语音体验优先级极高，简单控制必须走快路径**

### 1.2 技术原则

- 设备控制由 `Home Assistant` 负责
- 小米生态作为首批设备入口与家庭场景入口
- 人脸、声纹、相册底座优先集成现有能力
- 自研重点聚焦：
  - 家庭成员中心
  - 家庭长期记忆系统
  - 多成员上下文决策引擎
  - 家庭事务编排层
  - 家庭隐私权限治理层

### 1.3 版本原则

- `v0.1`：先验证“这东西值不值得继续做”
- `v0.5`：把差异化能力做成型
- `v1.0`：完成完整家庭服务闭环
- `v2.0`：进入家庭 AI OS 阶段

## 2. 项目总目标

## 2.1 MVP 总目标

在首个版本中，交付一个**可演示、可试用、可验证价值**的家庭 AI 中枢原型，满足以下目标：

- 能管理家庭成员档案、关系、偏好和权限
- 能感知“谁在家、谁在说话、谁需要服务”
- 能回答基础家庭问答
- 能完成提醒、广播和基础长期记忆写回
- 能联动 Home Assistant 与小米设备
- 能完整跑通 3 个代表性家庭场景

## 2.2 首期验证问题

MVP 的核心不是功能数量，而是验证以下 5 个问题：

1. 用户是否愿意把系统当作“家庭中枢”而不是“语音助手”
2. 多成员识别与上下文服务是否能明显优于普通智能家居联动
3. 家庭记忆是否能提升问答、提醒和陪伴体验
4. 家居联动是否足够自然、低打扰、可解释
5. 本地优先与权限机制是否能建立家庭信任

## 3. MVP 范围定义

## 3.1 MVP 功能范围

### P0：必须上线

#### 1）家庭成员中心
- 家庭管理员创建家庭
- 成员档案录入
- 角色建模：管理员、成人、儿童、老人、访客
- 基础关系建模：配偶、监护、父母子女
- 偏好录入：称呼、灯光、空调、提醒方式

#### 2）身份与在家状态识别
- 基础人脸识别接入
- 基础声纹识别接入
- 手机/蓝牙/门锁等在家状态辅助判断
- 多源融合输出当前活跃成员与置信度
- 低置信度回退到匿名模式

#### 3）家庭问答 v1
- 今日安排类问答
- 家庭成员状态类问答
- 设备状态类问答
- 简单记忆问答
- 按权限返回结果

#### 4）提醒与家庭广播
- 个人提醒
- 家庭共享提醒
- 儿童课程提醒
- 老人吃药提醒
- 家庭每日公告
- 未响应升级策略

#### 5）家庭记忆 v1
- 成员事实记忆
- 事件记录
- 偏好记忆
- 关键联动结果写回
- 管理员可查看、纠错、删除

#### 6）智能家居联动
- Home Assistant 接入
- 小米设备基础接入
- 房间、设备、场景映射
- 快路径控制：开灯、关灯、空调、窗帘、场景模式

#### 7）3 个关键演示场景
- 智能回家
- 儿童睡前陪伴
- 老人关怀提醒

### P1：建议在 MVP 尾声补上

- 基础照片归档
- 照片人物聚类
- 欢迎词与家庭摘要播报模板
- 审计日志后台
- 常见 TTS 短语缓存
- 热缓存与热索引

### P2：明确延后

- 复杂情绪分析
- 高级医疗建议
- 完整相册系统替代
- 多智能体自治编排
- 跨品牌全生态覆盖
- 高精度成长评估与教育画像推荐

## 3.2 MVP 非目标

MVP 阶段明确不追求：

- 替代 Home Assistant 全部设备管理能力
- 替代现有照片云盘或完整家庭相册平台
- 建立大而全的知识图谱系统
- 一次性覆盖所有家庭成员角色和所有场景
- 追求“最智能”，而忽视低延迟和稳定性

## 4. MVP 推荐架构拆分

建议首期不要一开始就做成大量微服务，而是采用**模块化单体 + 明确边界 + 可拆分接口**的方式推进。

## 4.1 核心模块

### A. 家庭成员中心
职责：

- 成员档案管理
- 关系管理
- 权限与可见范围
- 偏好配置

核心数据：

- `FamilyMember`
- `RelationshipGraph`
- `BiometricProfile`

### B. 家庭上下文中心
职责：

- 当前在家成员推断
- 当前房间占用推断
- 当前家庭模式管理
- 活跃发声者识别

核心输出：

- 当前成员上下文
- 房间上下文
- 家庭模式上下文

### C. 家庭记忆中心
职责：

- 事实记忆存储
- 事件记忆写回
- 偏好记忆更新
- 常见问答热索引

核心数据：

- `MemoryCard`
- `EventRecord`
- `PhotoAsset`

### D. 家庭事务中心
职责：

- 家庭问答
- 日程与提醒
- 家庭广播
- 基础健康提醒
- 睡前陪伴流程

### E. 家居联动网关
职责：

- 调用 Home Assistant
- 同步设备状态
- 调用小米场景或设备接口
- 执行动作回执与错误处理

### F. 编排与策略引擎
职责：

- 快路径与慢路径分流
- 场景判定
- 冲突处理
- 敏感操作拦截

### G. 管理台
职责：

- 成员录入与权限管理
- 场景配置
- 提醒任务配置
- 记忆查看与删除
- 审计日志查看

## 4.2 首期建议技术栈

仅作为第一版建议，可按团队熟悉度调整。

- 后端服务：`Python + FastAPI`
- 编排与策略：`Python`
- 管理台：`Next.js` 或 `React`
- 数据库：`SQLite`（MVP，本地优先；后续如并发和部署规模提升，再迁移到 `PostgreSQL`）
- 缓存：`Redis`
- 队列/任务：`Celery` 或 `RQ`
- 设备接入：`Home Assistant API`
- 人脸识别：`CompreFace`
- 相册底座：`Immich`
- 语音栈：本地唤醒 + 流式 ASR + TTS 组合
- 监控：`Prometheus + Grafana`

## 5. MVP 开发阶段计划

以下按一个 `3~5 人小团队、12~16 周` 节奏规划；若为单人开发，建议整体工期按 `1.8~2.5 倍` 估算。

## 5.1 阶段 0：项目定义与技术预研（第 1~2 周）

### 目标

冻结 MVP 范围，完成架构草案与关键依赖验证。

### 关键产出

- PRD v0.1
- MVP 用例清单
- 3 个演示场景流程图
- 领域模型初稿
- 接口清单初稿
- 技术选型结论
- 开发环境与基础仓库

### 重点任务

- 明确首期支持的家庭角色
- 明确首期支持的设备范围
- 明确 Home Assistant 与米家接入方式
- 确定识别组件与语音栈候选
- 定义核心数据模型
- 定义事件总线与日志规范

### 阶段验收

- 能清楚回答“首期不做什么”
- 能展示 3 个场景的事件流
- 能跑起本地开发环境

## 5.2 阶段 1：家庭成员中心与数据底座（第 3~4 周）

### 目标

先把“家庭是谁”这件事建起来。

### 关键产出

- 成员档案 API
- 关系管理 API
- 权限模型 v1
- 偏好模型 v1
- 管理台成员录入页面
- 生物信息占位与绑定机制

### 重点任务

- 实现 `FamilyMember`
- 实现 `RelationshipGraph`
- 实现 `BiometricProfile`
- 设计权限分级：公共、个人、敏感、高风险
- 支持成员导入、编辑、停用

### 阶段验收

- 管理员可创建完整家庭结构
- 不同角色有不同可见范围
- 后续模块都能以 `member_id` 为主键关联

## 5.3 阶段 2：设备接入与家庭上下文（第 5~6 周）

### 目标

建立“谁在家、谁在哪、设备状态如何”的上下文底座。

### 关键产出

- Home Assistant 网关 v1
- 小米设备基础接入 v1
- 房间模型与设备映射
- 在家状态推断
- 当前活跃成员推断
- 家庭模式缓存

### 重点任务

- 接入门锁、门磁、人体传感器、灯、空调、音箱
- 订阅设备状态变化
- 建立房间与设备映射表
- 建立成员在家判断规则
- 建立当前房间占用判断规则
- 建立热缓存

### 阶段验收

- 系统可查询当前谁在家
- 系统可查询当前各房间占用情况
- 系统可执行基础灯光与空调控制

## 5.4 阶段 3：家庭记忆与问答 v1（第 7~8 周）

### 目标

让系统不止“知道状态”，还要“记住这个家”。

### 关键产出

- `MemoryCard` 模型
- `EventRecord` 模型
- 家庭问答 API v1
- 家庭事件写回机制
- 管理台记忆列表页

### 重点任务

- 建立事实记忆、事件记忆、偏好记忆结构
- 建立问答检索链路
- 建立高频问题热索引
- 建立记忆的可纠错、可删除能力
- 将重要联动结果写回记忆

### 首期支持问答

- 谁在家
- 今天谁有什么安排
- 某成员是否到家
- 某设备是否开启
- 某老人是否完成今日提醒
- 某孩子今天是否有课程安排

### 阶段验收

- 至少支持 10 个高频家庭问答
- 结果遵守权限控制
- 联动场景可生成事件记录

## 5.5 阶段 4：提醒、广播与语音快路径（第 9~10 周）

### 目标

建立首批能被真实感知到的主动服务能力。

### 关键产出

- `ReminderTask` 模型
- 提醒调度器
- 广播模板系统
- 语音快路径控制链路
- 常见确认语缓存

### 重点任务

- 支持个人提醒、家庭提醒、服药提醒、课程提醒
- 支持按成员、按房间播报
- 支持未响应升级
- 高频语音控制不走重链路
- 简单命令“先执行，后解释”

### 阶段验收

- 简单控制首响应接近 1 秒内
- 提醒可按角色、房间和时机差异化触达
- 广播能区分公共信息与私密信息

## 5.6 阶段 5：3 个代表性场景闭环（第 11~12 周）

### 目标

把 MVP 从“模块集合”打磨成“用户可理解的完整体验”。

### 场景 A：智能回家

输入：

- 门锁开门
- 摄像头识别
- 手机在场状态

输出：

- 识别成员
- 打开玄关灯
- 调整空调
- 播报欢迎词与提醒
- 写回“已到家”事件

### 场景 B：儿童睡前陪伴

输入：

- 到达睡前时间
- 儿童房检测有人
- 孩子说“我要睡觉啦”

输出：

- 切睡前灯光
- 播放睡前故事
- 关闭高亮设备
- 记录入睡时间与互动结果

### 场景 C：老人关怀提醒

输入：

- 到达吃药时间
- 摄像头或在场状态确认老人位置

输出：

- 就近播报
- 未响应二次提醒
- 再未响应则通知家属
- 写回提醒完成状态

### 阶段验收

- 三个场景均可稳定重复演示
- 每个场景均有状态可视化
- 每个场景均能回写事件或记忆

## 5.7 阶段 6：治理、性能与试点验证（第 13~16 周）

### 目标

让 MVP 具备“可试用”与“可迭代”的基础。

### 关键产出

- 审计日志 v1
- 敏感操作确认机制
- 数据删除与纠错能力
- 性能监控大盘
- 灰度试点报告

### 重点任务

- 生物模板加密存储
- 隐私房间与隐私时段初版
- 超时降级处理
- 关键 API 监控
- 用户反馈收集
- 问题清单与下阶段计划

### 阶段验收

- 敏感问答先做权限校验
- 高风险动作需要确认
- 核心场景具备监控与日志
- 有至少一轮真实家庭试点反馈

## 6. MVP 模块优先级 Backlog

## 6.1 按业务价值排序

| 优先级 | 模块 | 是否 MVP 必做 | 原因 |
|---|---|---:|---|
| P0 | 家庭成员中心 | 是 | 所有能力的主数据底座 |
| P0 | 权限与关系模型 | 是 | 家庭场景天然需要分级可见 |
| P0 | 在家状态与身份识别 | 是 | 决定“服务谁” |
| P0 | Home Assistant 接入 | 是 | 决定是否能真正控制设备 |
| P0 | 家庭记忆 v1 | 是 | 决定是否有差异化 |
| P0 | 家庭问答 v1 | 是 | 最直接体现价值 |
| P0 | 提醒与广播 | 是 | 最容易形成高频使用 |
| P0 | 3 个场景闭环 | 是 | 用来验证产品方向 |
| P1 | 基础照片归档 | 否 | 能增强记忆，但不是首要闭环 |
| P1 | 相册人物聚类 | 否 | 对回忆能力有帮助，首期可弱化 |
| P1 | 管理台审计页 | 否 | 重要但可在 MVP 尾声补齐 |
| P2 | 教育成长深度画像 | 否 | 易膨胀，适合后续版本 |
| P2 | 健康趋势分析 | 否 | 首期只保留提醒与事件记录 |
| P2 | 情绪理解与主动陪伴 | 否 | 复杂度高，延后 |

## 6.2 按研发依赖排序

1. 成员中心
2. 权限模型
3. 设备接入
4. 上下文推断
5. 记忆模型
6. 问答与提醒
7. 场景编排
8. 管理台与治理

## 7. MVP 验收标准

## 7.1 功能验收

上线前至少满足：

- 支持 `5` 类角色：管理员、成人、儿童、老人、访客
- 支持 `3` 个完整端到端演示场景
- 支持 `10` 个高频家庭问答
- 支持 `4` 类提醒：个人、家庭、课程、服药
- 支持 `1` 套管理员管理台
- 支持设备控制、状态查询、广播与记忆写回

## 7.2 体验验收

- 唤醒到反馈：目标 `< 300ms`
- 简单控制首响应：目标 `< 800ms`，上限 `1.2s`
- 普通问答首响应：目标 `< 1.5s`
- 复杂检索：`1s` 内先确认，`3~5s` 内给结果
- 快路径控制不得依赖完整 LLM 推理

## 7.3 安全验收

- 敏感问答必须做权限校验
- 高风险动作必须二次确认
- 人脸与声纹模板本地加密
- 支持审计日志
- 支持记忆删除与纠错

## 7.4 稳定性验收

- 关键场景连续演示 `20` 次不出现阻断性错误
- 设备控制失败时有降级与回执
- 识别低置信度时不执行敏感动作

## 11. 风险与控制策略

## 11.1 最大风险

### 风险 1：范围失控

表现：

- 每个模块都想做成完整产品
- 很快膨胀到健康、教育、相册、语音、家居全自研

控制策略：

- 所有需求强制分为 `P0/P1/P2`
- 每两周进行一次范围复盘
- 新增需求必须说明对 3 个演示场景的直接价值

### 风险 2：识别不稳定

表现：

- 人脸误识别
- 声纹不稳定
- 孩子和老人识别效果波动大

控制策略：

- 用多模态融合，不依赖单一识别
- 敏感动作不依赖单次识别
- 置信度低时回退到匿名模式或追问

### 风险 3：语音体验慢

表现：

- 开灯还要等待几秒
- 问答没有及时反馈

控制策略：

- 简单控制走快路径
- 先执行再确认
- 常见短语缓存
- 超时降级

### 风险 4：隐私接受度不足

表现：

- 用户担心持续感知
- 不信任摄像头与生物信息存储

控制策略：

- 本地优先
- 默认最小上传
- 生物模板加密
- 提供删除、纠错、停用与隐私时段

### 风险 5：联动过于主动

表现：

- 系统太爱播报
- 频繁打断家庭成员

控制策略：

- 只优先做高价值、低打扰场景
- 广播支持关闭与降噪
- 主动服务可解释、可回退、可关闭

## 12. 首个版本发布建议

## 12.1 内测范围

建议先做非常小规模试点：

- `1~3` 个真实家庭
- 每个家庭 `3~6` 名成员
- 固定少量房间与设备
- 重点观察回家、睡前、老人提醒场景

## 12.2 数据收集重点

- 哪些问答最常见
- 哪些播报最容易引起反感
- 哪些提醒容易被忽略
- 哪些联动最有价值
- 哪类识别错误最影响信任

## 12.3 发布后第一轮迭代重点

优先修正：

1. 识别误判
2. 权限误放大
3. 播报打扰感
4. 控制链路延迟
5. 场景触发不稳定

## 14. 结论

这个项目最重要的，不是首期支持多少设备，也不是首期覆盖多少能力，而是先把以下闭环做扎实：

- 知道这个家里有哪些人
- 知道当前该服务谁
- 知道哪些信息可以说、哪些不能说
- 知道如何把一次交互沉淀成长期记忆
- 知道如何把理解结果落到真实设备和真实场景上

如果 MVP 能把这条链路跑通，家庭版 OpenClaw 就已经具备了继续扩展为完整家庭 AI 系统的基础。
