# 插件开发

这页只做一件事：告诉你插件开发现在该先看什么，不要在仓库里盲找。

## 先看哪份规则

如果你是在做通用插件开发，先按这个顺序看：

1. `docs/开发者文档/插件开发/README.md`
2. `docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`
3. `docs/开发者文档/插件开发/zh-CN/03-manifest字段规范.md`
4. `docs/开发者文档/插件开发/zh-CN/12-V1插件类型与接口总表.md`

## 如果你做的是 AI 供应商插件

请不要只看旧文档。

先按这个顺序看：

1. `specs/004.8.1-AI供应商彻底插件化迁移/`
2. `docs/开发设计规范/20260318-插件能力与接口规范-v1.md`
3. `docs/开发者文档/插件开发/zh-CN/README.md`

当前统一口径是：

- 宿主保留统一 AI 网关、路由、权限、审计、密钥、降级。
- `ai-provider` 插件负责供应商声明、字段 schema、协议适配、流式输出、厂商特例。
- 如果某个供应商在通用 OpenAI 兼容层上有明显性能或兼容问题，优先把修正收进该供应商插件 driver，不要把厂商特例继续堆回宿主公共 runtime。
- 例如 `GLM` / `GLM Coding Plan` 这类带思考能力的模型，可以在插件 driver 里针对短结构化任务关闭 `thinking`、收紧 `max_tokens`，先解决首轮超时，再决定要不要动公共链路。

注意：

- 这是当前文档主口径。
- 代码迁移还没有全部完成，仓库里仍可能看到兼容层。
- 新增工作必须按新模型走，不要继续把新供应商信息写回核心。

## 项目特有约束

- 插件启停规则先看：`docs/开发设计规范/20260317-插件启用禁用统一规则.md`
- 后端事件循环和周期任务先看：`docs/开发者文档/后端/20260316-后端事件循环与周期任务开发规范.md`
- 正式插件类型以开发者手册和插件规范 v1 为准，不再继续使用 `connector`、`memory-ingestor`

## 运行时安装边界

- 运行时安装插件包括两类：手动 ZIP 安装，以及插件市场安装、启用、升级。
- 这两类插件现在的统一口径是：安装或升级成功后，无需重启后端服务，插件列表和后续执行都会按当前挂载识别新版本。
- 这条语义只表示“刷新列表可见、下一次执行生效”，不是 `in_process` Python 模块热替换。
- 手动 ZIP 安装当前后端会固定写成 `third_party + subprocess_runner`，托管目录在 `/data/plugins/third_party/manual/<household_id>/<plugin_id>/<version>--<timestamp>--<id>/`。
- 启动同步会兼容旧目录和新的 release 目录布局，并优先恢复最近一次成功安装的版本。
- 市场安装也会直接更新当前挂载；安装后默认不自动启用，启用前仍要满足配置完成约束。

## 提交前至少检查

- manifest 是否使用正式类型
- 配置字段是否走正式 schema / config 入口
- 是否把宿主主权边界写回了插件
- 是否把插件边界误写回了宿主
- 文档是否同步更新
