# 文档发布

## 这页要解决什么

- 说明仓库里的正式文档怎么自动发布到 GitHub Pages
- 说明第一次启用这条流水线时，仓库管理员还要手工点哪些设置
- 说明默认仓库地址和自定义域名下，文档基础路径该怎么处理

## 目标读者

- 负责仓库设置的维护者
- 负责正式文档发布的开发者

## 这套发布链路做了什么

- 工作流文件在 `.github/workflows/docs-pages.yml`
- 只要 `main` 分支里和文档相关的内容有变更，就会自动触发构建和发布
- 手工从 GitHub Actions 页面点 `Run workflow` 也能强制发布一次
- 发布命令固定为仓库根目录的 `npm run docs:build`
- 发布产物固定为 `docs/Documentation/.vitepress/dist`
- 工作流已经显式设置 `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true`，提前按 GitHub 的 Node 24 运行时执行官方 action，避免 Node 20 弃用警告拖到最后再处理

## 首次启用步骤

1. 把工作流文件提交到默认分支。
2. 打开仓库 `Settings > Pages`。
3. 在 `Build and deployment > Source` 里选择 `GitHub Actions`。
4. 回到仓库主页，确认 `Actions` 页面里的 `发布文档到 GitHub Pages` 能正常运行。
5. 等第一次部署完成后，再打开 GitHub Pages 分配的地址验收。

## 基础路径怎么定

### 默认仓库地址

- 如果站点最终访问地址是 `https://<用户名>.github.io/<仓库名>/`
- 这套工作流会自动把 `DOCS_BASE` 算成 `/<仓库名>/`
- 这种情况下不需要再改 VitePress 配置

### 用户站点或组织站点

- 如果仓库名本身就是 `<用户名>.github.io` 或 `<组织>.github.io`
- 站点根路径就是 `/`
- 这套工作流也会自动处理，不需要额外设置

### 自定义域名

- 如果你给 GitHub Pages 绑了独立域名，例如 `https://docs.example.com/`
- 站点根路径也应该是 `/`
- 这时要到仓库 `Settings > Secrets and variables > Actions > Variables` 里新增仓库变量 `DOCS_BASE`
- 变量值写成 `/`

不要偷懒。自定义域名下如果还沿用 `/<仓库名>/`，CSS、脚本和图片路径都会错。

## 本地验证

先在仓库根目录跑构建：

```bash
cmd /c npm run docs:build
```

如果你想在本地看最终静态站效果，再跑：

```bash
cmd /c npm run docs:preview
```

## 发布触发规则

下面这些改动会触发自动发布：

- `docs/Documentation/**`
- `package.json`
- `package-lock.json`
- `.github/workflows/docs-pages.yml`

这样做的目的很简单：文档没变就别重新发站点，省掉无意义的构建。

## 常见坑

- 工作流已经合并，但 `Pages` 里还没把发布源切到 `GitHub Actions`
- 自定义域名已经绑定，但没有把 `DOCS_BASE` 改成 `/`
- 文档本地能跑，线上却 404，通常是 `base` 路径配错了
- 直接改了 `dist` 目录并提交，结果下一次 Actions 构建把这些手工改动全部覆盖

## 完成标准

- 维护者知道第一次启用要去哪里点设置
- 开发者知道这条流水线什么时候触发、用什么命令构建
- 默认仓库地址和自定义域名两条路径都能说清楚
