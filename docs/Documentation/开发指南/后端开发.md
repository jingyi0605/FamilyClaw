# 后端开发

> 这页只讲真正会踩坑的后端边界，不讲空话。

## 这页要解决什么

- 说明后端模块边界、数据库迁移规则、异步入口和后台任务约束
- 说明哪些改动必须同步补测试和正式文档
- 说明长期记忆召回这类 PostgreSQL 能力该怎么落地，避免再写成 Python 层临时补丁

## 后端代码放在哪里

- `apps/api-server/app/modules/`：业务模块源码，按领域拆分
- `apps/api-server/app/api/`：HTTP 接口与依赖注入
- `apps/api-server/app/db/`：数据库基座、模型注册、连接与通用工具
- `apps/api-server/migrations/`：Alembic migration，只能在这里改表结构
- `apps/api-server/tests/`：后端测试，数据库集成测试默认走 PostgreSQL schema 隔离

## 关键开发规则

- 只要改数据库表、字段、索引、约束，就必须写 Alembic migration。
- 不允许在业务代码里偷偷 `create_all()`、补列、补索引，也不允许用启动逻辑碰运气修库。
- 只要改异步入口、周期任务、worker 或插件执行链路，先读事件循环与周期任务规范，再动手。
- 只要改插件启用/禁用、自动触发和执行入口，先读插件启停统一规则，再动手。
- 只要改安装、配置、用户可见行为、接口语义或排障方式，正式文档必须同一变更同步更新。

## 长期记忆召回开发要点

- 第一阶段确实是从 `memory_cards` 起步，但当前代码已经进入统一 recall 形态，不要再按“只改 `L3` 就够了”的旧口径理解主链路。
- `memory_cards` 当前召回相关字段包括：
  - `search_text`
  - `search_tsv`
  - `embedding`
  - `projection_version`
  - `projection_updated_at`
- 结构创建必须经过 migration：
  - `search_tsv` 走 PostgreSQL `tsvector`
  - `idx_memory_cards_search_tsv` 走 `GIN`
  - `embedding` 在 `pgvector` 扩展可用时建成 `vector(16)`，否则退化成 `TEXT`
- 业务代码只能刷新投影内容，不能负责建扩展、建列、建索引。
- 召回查询顺序必须先过权限过滤，再做关键词召回、向量召回和 hybrid rerank。
- `pgvector`、embedding 或 SQL 查询出问题时，主链路必须降级，不能把 `free_chat` 和问答主路由一起拖死。
- 当前宿主的分层真相源已经是：
  - `conversation_session_summaries` = `L1`
  - `episodic_memory_entries` = `L2`
  - `memory_cards` = `L3`
  - `knowledge_documents` = `L4`
- 当前统一召回投影已经是 `memory_recall_documents`：
  - 只负责召回，不替代任何真相源
  - 当前会把 `L1/L2/L3/L4` 同步成统一的 `search_text / search_tsv / embedding` 投影
  - 统一 recall 失败且结果为空时，仍会退回宿主兼容链路兜底
- `L2/L4` 现在也必须像 `L3` 一样保留宿主修订历史：
  - `episodic_memory_entry_revisions`
  - `knowledge_document_revisions`
- 插件 observation 导入当前是兼容优先的双写：
  - 继续写旧的 `L3 observation memory_cards`
  - 同时写 `L4 knowledge_documents`
  这条线别乱删，除非你同时收掉旧接口和旧测试。
- 第二阶段补了 `conversation_session_summaries` 作为 `L1` 真相源：
  - 只存当前会话摘要、未完成话题和最近确认事项，不跟 `memory_cards` 混表
  - 目前通过 turn 后处理触发刷新，不额外起常驻 worker
  - 当前默认阈值是新增 4 条消息后刷新
  - 状态流转是 `fresh / stale / rebuilding / failed`
- `conversation_memory_reads.memory_id` 已不再绑定到 `memory_cards` 外键，后续允许记录 `L1/L2/L3/L4` 来源 trace。
- 第三阶段把长期记忆正式收口成分层真相源：
  - `conversation_session_summaries` 是 `L1`
  - `episodic_memory_entries` 是 `L2`
  - `memory_cards` 是 `L3`
  - `knowledge_documents` 是 `L4`
- 第三阶段新增统一投影 `memory_recall_documents`：
  - 只负责召回，不替代任何真相源
  - 当前会把 `L1/L2/L3/L4` 同步成同一套 `search_text / search_tsv / embedding` 投影
  - `L1 session_summary` 只有当前 `session_id` 命中时才允许进入统一召回
  - unified recall 失败且结果为空时，会退回旧的 `memory_cards + session_summary` 兼容链路
- `knowledge_documents` 当前支持三类来源：
  - `plugin_raw_record`
  - `doc`
  - `rule`
- 规则文档和说明文档当前只提供显式导入入口，不做文件系统自动扫描，避免跨期把范围炸开。

## free_chat 注入边界

- `free_chat` 不再只吃 `hot_summary`。
- 注入内容必须来自真实 recall hits，当前分成四组：
  - `session_summary`
  - `stable_facts`
  - `recent_events`
  - `external_knowledge`
- 不允许把别组内容混进来凑 prompt。
- 每轮注入都要留下 trace，至少记：
  - `memory_id`
  - `source_id`
  - `source_kind`
  - `group`
  - `layer`
  - `score`
  - `rank`
- 当前 trace 落在 `conversation_memory_reads`，用于排查“这轮为什么会这么答”。
- `session_summary` 的注入内容必须来自 `conversation_session_summaries` 的真实命中结果，不能拿最近几条消息硬拼装顶替。
- 当前会话如果已经有可用 `session_summary`，统一 recall 会把它作为会话级保底候选加入，避免因为问句分词没命中而整组消失。
- 只有 `context_engine` 槽位已经真接进主链路。
- `memory_engine / memory_provider` 目前只有槽位定义和 manifest/schema 约束，还没有切进主写入、主召回链路，不要按“已插件化”写实现。

## 开发流程

1. 先读对应 Spec、设计文档和项目规则，确认范围别跨期。
2. 改表先写 migration，再改模型、repository、service。
3. 先补最小可运行链路，再补集成测试和降级测试。
4. 用 PostgreSQL 测试库跑回归，不要只靠 `py_compile` 自我安慰。
5. 同步更新 `docs/Documentation/` 和对应 Spec 的 `tasks.md`。

## 回归检查

- 数据库结构检查：字段、索引、扩展兼容逻辑都要在 PostgreSQL 上验证。
- 主链路检查：接口兼容，旧读写路径不能被新字段破坏。
- 降级检查：`pgvector` 不可用、召回查询失败、LLM 超时这些路径都要能返回，而不是直接 500。
- trace 检查：`conversation_memory_reads` 和调试日志里能回放本轮注入来源。
- 会话摘要检查：确认 `conversation_session_summaries` 能刷新，`session_summary` 能进入 recall，失败时有 `session_summary.failed` 调试日志。
- 统一召回检查：确认 `memory_recall_documents` 已覆盖 `L1/L2/L3/L4`，并能按 `session_summary / stable_facts / recent_events / external_knowledge` 四组输出。
- 外部知识检查：确认 `knowledge_documents` 能接住 `plugin_raw_record / doc / rule`，且统一走主链路召回，而不是旁路偷读。
- 修订历史检查：确认 `memory_card_revisions`、`episodic_memory_entry_revisions`、`knowledge_document_revisions` 都在写，而且 action / actor / reason 能对上。
- 槽位边界检查：确认当前只有 `context_engine` 允许通过槽位改写主链路；如果有人开始让 `memory_engine / memory_provider` 直接接主写入或主查询，必须先补宿主 fallback 和回归测试。
- 文档检查：实现改了，正式文档也得一起改。
