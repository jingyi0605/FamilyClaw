# 文档治理规范

## 这页要解决什么

- 明确 `FamilyClaw` 文档由谁维护、放在哪里、什么时候必须跟代码一起改
- 让“文档要不要改”从口头约定变成可执行规则

## 核心原则

### 文档不是装饰品

- 只要代码改动会影响别人安装、使用、开发、运维或对接，相关正式文档就必须同步更新
- 文档和代码描述不一致时，以实际代码行为为准，但提交者必须修正文档，不允许长期漂移

### 正式文档与过程文档分开

- `docs/Documentation/`：正式官方文档，面向长期使用
- `docs/开发设计规范/`：项目内部开发规则和设计边界
- `docs/开发者文档/`：专题型、补充型、历史沉淀资料
- `specs/`：需求、设计、任务、验收过程文档

### 路径稳定，信息可变

- 正式文档文件名只保留语义，不带日期，不带树编号
- 编号、版本、状态通过 frontmatter 维护
- 不要通过复制旧文档再加“新版”“最终版”“v2”制造垃圾分叉

## 文档责任划分

### 功能提交者负责第一责任

- 谁改代码，谁先判断文档要不要改
- 不允许把文档更新默认甩给“后面再补”

### 模块负责人负责兜底

- 前端相关文档由前端负责人兜底
- 后端相关文档由后端负责人兜底
- 插件相关文档由插件负责人兜底
- 跨模块变更由主负责人统一收口

### Reviewer 负责查漏

- Code review 不只看代码，也要看对应文档是否同步
- 如果改动明显影响外部使用而没有文档更新，review 应直接指出

## 文档分级

### P0：必须和代码同提交

- 安装部署文档
- 使用指南中涉及用户可见行为的页面
- 开发指南中涉及目录、入口、命令、规则、约束的页面
- 运维与排障中涉及日志位置、诊断顺序、升级回滚的页面
- 配置参考、接口参考、术语表中被这次改动影响的条目
- `docs/开发设计规范/` 下被代码依赖执行的规则文档

### P1：可以同迭代补齐，但不能长期拖

- 新增能力的完整示例
- 截图、流程图、说明性补图
- 背景介绍、设计动机、补充 FAQ

### P2：按需维护

- 历史材料索引
- 非关键介绍页
- 不影响理解和使用的美化性内容

## 哪些代码改动必须同步改文档

### 必改场景一：安装或启动方式变化

- 新增或移除依赖
- 修改环境要求
- 修改启动命令、初始化步骤、配置来源

必须同步更新：

- `docs/Documentation/安装部署/环境要求.md`
- `docs/Documentation/安装部署/本地安装.md`
- `docs/Documentation/安装部署/配置说明.md`
- 必要时更新 `docs/Documentation/开始这里/快速开始.md`

### 必改场景二：用户可见功能变化

- 页面入口变化
- 主要操作路径变化
- 配置入口变化
- 默认行为变化

必须同步更新：

- 对应 `docs/Documentation/使用指南/` 页面
- 必要时更新 `docs/Documentation/开始这里/文档总览.md`

### 必改场景三：仓库结构或开发入口变化

- 目录迁移
- 脚本变化
- 构建方式变化
- 开发流程变化

必须同步更新：

- `docs/Documentation/开发指南/仓库结构说明.md`
- `docs/Documentation/开发指南/开发环境准备.md`
- 对应前端、后端、插件、测试页面

### 必改场景四：后端规则或运行边界变化

- 数据库迁移方式变化
- 异步入口变化
- worker、周期任务、插件执行边界变化

必须同步更新：

- 对应 `docs/Documentation/开发指南/后端开发.md`
- 对应 `docs/开发设计规范/` 规则文档

### 必改场景五：配置项、接口、错误语义变化

- 新增配置
- 删除配置
- 默认值变化
- 接口字段变化
- 错误码或错误语义变化

必须同步更新：

- `docs/Documentation/安装部署/配置说明.md`
- `docs/Documentation/参考资料/配置参考.md`
- `docs/Documentation/参考资料/接口参考.md`
- 必要时更新使用或开发指南中相关页面

### 必改场景六：运维手册失效

- 日志位置变化
- 监控信号变化
- 升级回滚步骤变化
- 高频故障排查路径变化

必须同步更新：

- `docs/Documentation/运维与排障/日志与诊断.md`
- `docs/Documentation/运维与排障/常见故障排查.md`
- `docs/Documentation/运维与排障/升级与回滚.md`

## 哪些改动通常不需要改正式文档

- 纯内部重构，且不改变外部行为
- 注释调整、命名微调、代码清理
- 不影响入口、命令、配置、接口、页面行为的性能优化
- 临时调试代码的删除

注意：

- 如果提交者声称“只是重构”，但目录、接口、配置、行为已经变了，那就不是纯重构，文档还是得改

## 提交前检查清单

提交代码前，至少问自己这几个问题：

1. 这次改动会不会让新同事按旧文档无法跑通？
2. 这次改动会不会让用户按旧文档点不到入口？
3. 这次改动会不会让开发者按旧目录说明找错地方？
4. 这次改动会不会让运维按旧手册排障失败？
5. 这次改动会不会让配置、接口、错误语义说明失真？

只要有一个答案是“会”，就必须补文档。

## Code Review 要求

Reviewer 至少检查下面几件事：

- 变更是否影响正式文档
- 文档改动是否和代码改动一致
- 新增页面或新入口是否有对应文档落点
- 删除旧行为时，旧文档是否同步清理

如果存在下面情况，review 应要求修改后再合并：

- 明显影响安装、使用、开发、运维，但没有文档更新
- 文档更新了，但内容和代码行为对不上
- 新能力没有任何落点，后续别人根本找不到

## 阻塞合并的红线

- 启动方式已变，但安装文档没改
- 用户主路径已变，但使用指南没改
- 目录或开发入口已变，但开发指南没改
- 迁移或运行边界已变，但规则文档没改
- 配置、接口、错误语义已变，但参考文档没改

## 推荐工作流

1. 先改代码
2. 立刻判断受影响文档
3. 在同一个分支内同步改文档
4. 本地至少跑一遍文档构建或链接检查
5. 提交时让代码和文档一起进入 review

## 与 Spec 的关系

- `specs/` 记录“为什么做、怎么拆、怎么验收”
- `docs/Documentation/` 记录“现在系统应该怎么被安装、使用、开发和维护”
- 一个需求做完后，如果行为已经成为长期事实，就应该沉淀到正式文档，而不是永远埋在 spec 里

## 完成标准

- 团队能明确判断哪些改动必须同步改文档
- Reviewer 能据此判断是否应该阻塞合并
- 正式文档、规则文档、spec 文档的边界不再混乱
