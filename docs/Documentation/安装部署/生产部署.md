# 生产部署

这页讲单容器生产部署的最短路径。目标不是把 Docker 玩出花，而是让普通用户只管一个镜像、一个 `/data` 目录和一组环境变量。

## 部署前提

- 宿主机已安装 Docker 24 或更新版本
- 宿主机至少预留 4 CPU、8 GB 内存、20 GB 可写磁盘
- 已准备一个持久化目录，例如 `/srv/familyclaw-data`
- 如需语音网关，宿主机到外部设备的 `4399` 端口已放通

## 单容器里实际跑什么

- PostgreSQL：唯一运行时数据库，数据固定写入 `/data/postgres`
- `api-server`：业务 API、Alembic 迁移、版本信息接口
- `open-xiaoai-gateway`：可选语音网关，默认跟主容器一起启动
- Nginx：统一对外入口，提供前端静态资源并反代 `/api/`
- `user-app` H5 构建产物：构建阶段打进镜像，运行时不再需要前端开发服务

## 部署步骤

1. 登录镜像仓库。

```bash
docker login ghcr.io
```

2. 构建并推送双架构镜像。

```bash
export IMAGE_NAME="ghcr.io/<你的组织或用户名>/familyclaw"
export RELEASE_URL_BASE="https://github.com/<你的组织或用户名>/FamilyClaw/releases/tag"

IMAGE_NAME="$IMAGE_NAME" \
RELEASE_URL_BASE="$RELEASE_URL_BASE" \
BUILD_CHANNEL=stable \
PUSH=1 \
./docker/build-multiarch.sh
```

如果你只想在 Ubuntu 本机先验证单架构镜像，不推远端，可以这样跑：

```bash
IMAGE_NAME="familyclaw" \
PLATFORMS="linux/amd64" \
BUILD_CHANNEL=development \
PUSH=0 \
./docker/build-multiarch.sh
```

3. 准备数据目录。

```bash
mkdir -p /srv/familyclaw-data
```

4. 启动容器。

```bash
docker run -d \
  --name familyclaw \
  -p 8080:8080 \
  -p 4399:4399 \
  -v /srv/familyclaw-data:/data \
  -e FAMILYCLAW_DB_PASSWORD='change-me' \
  -e FAMILYCLAW_VOICE_GATEWAY_TOKEN='replace-me' \
  ghcr.io/<你的组织或用户名>/familyclaw:0.1.0
```

5. 打开 `http://<宿主机地址>:8080` 检查首页是否可访问。

## 启动顺序

容器内部不是并发乱起，而是按下面顺序收口：

1. PostgreSQL 初始化或启动
2. 等待 PostgreSQL ready
3. 如数据库版本需要变化，先备份再执行 `alembic upgrade head`
4. 启动 `api-server`
5. 启动 `open-xiaoai-gateway`
6. 启动 Nginx

这条顺序不能打乱。迁移失败时容器应直接失败，不能带病对外提供服务。

## 部署后验证

先看容器状态：

```bash
docker ps --filter name=familyclaw
```

再看三类健康点：

```bash
docker exec familyclaw familyclawctl doctor
docker exec familyclaw familyclawctl version
curl -fsS http://127.0.0.1:8080/api/v1/system/version
```

你应该能确认这几件事：

- PostgreSQL、`api-server`、Nginx 都是 `ok`
- 如果启用了语音网关，`gateway` 也是 `ok`
- `/api/v1/system/version` 返回的版本信息和镜像打包参数一致
- `/data/runtime/installed-version.json` 已经写入当前安装状态

## 回滚准备

- `/data` 必须是宿主持久化目录，不能用匿名卷
- 升级前备份由容器里的 `familyclawctl backup` 执行，备份文件在 `/data/backups`
- 如果是有 schema 变更的版本升级，回滚前必须先恢复备份，再切回旧镜像

## 常见部署失误

- 只挂载了数据库目录，没有挂整个 `/data`
- 直接把 `change-me` 带到生产环境
- 迁移失败后继续复用异常容器
- 想当然认为 `latest` 能代表精确可审计版本
