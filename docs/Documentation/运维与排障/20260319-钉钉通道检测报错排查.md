# 钉钉通道检测报错与接收链路排查
## 这次实际修了什么

- 修掉了钉钉插件 `probe` 阶段缺少 `_load_account_config` 导致的 `NameError`。
- 把钉钉接收链路从“依赖 webhook 才能进消息”改成“账号配置为 `polling` 时，由插件内部自己维持钉钉 Stream 长连接，再把消息交给主项目的轮询入口”。
- 发送链路不再依赖入站消息里的 `sessionWebhook`，改为优先走钉钉 OpenAPI；旧 webhook 账号仍保留 `sessionWebhook` 兜底，避免马上把已有账号打坏。

## 先说结论

- `openclaw-china` 的钉钉实现核心不是 webhook，也不是普通 HTTP 轮询，而是钉钉 Stream 长连接。
- FamilyClaw 主项目核心目前没有给 channel 插件提供常驻 Stream 生命周期入口，所以这次没有把钉钉逻辑塞进主项目核心。
- 最终落地方式是：主项目仍然只认识 `poll`，但钉钉插件在自己模块里维护后台 Stream 线程，`poll` 只负责把已经收到的消息取出来。

## 如果“检测连接”已经成功，但发消息没反应

优先检查下面这几件事：

1. 当前钉钉账号的 `connection_mode` 必须是 `polling`，不要再用旧的 `webhook`。
2. 配置里必须同时有 `app_key` 和 `app_secret`。只有 `app_key` 的旧 webhook 兼容模式只能过检测，不能承担新的收发链路。
3. 钉钉开放平台里的机器人接收方式要开 Stream 模式，否则插件线程能启动，但永远不会收到消息。

## 你会看到什么现象

### 1. 旧问题：检测时报 `name '_load_account_config' is not defined`

- 这是插件代码缺函数，不是你参数填错。
- 修复后，这个报错应该直接消失。

### 2. 新链路下，检测成功但没有接收消息

- 如果账号还是 `webhook`，这是预期现象。
- 因为主项目不会主动帮钉钉账号拉起 Stream；只有 `polling` 账号才会被轮询 worker 选中。

## 怎么确认现在是不是走对了

### webhook 兼容账号

- 只填 `app_key`
- `connection_mode = webhook`
- 检测结果应该是：

`dingtalk legacy webhook mode does not require active probe`

这只说明“旧兼容模式没报错”，不代表新的消息接收链路已经可用。

### 新推荐账号

- 同时填写 `app_key` 和 `app_secret`
- `connection_mode = polling`
- 检测结果应该是：

`dingtalk stream credentials validated`

这才是现在应该使用的接法。

## 这次代码改动边界

- 只改了钉钉插件自己的代码和 manifest。
- 没有把钉钉 Stream 生命周期、OpenAPI 专用逻辑或额外路由写进主项目核心模块。
- 主项目核心继续只处理通用的 `poll / webhook / send / probe` 约定。

## 如果还不通，下一步看哪里

1. 看账号是不是 `polling`，而不是旧的 `webhook`。
2. 看 `app_secret` 是否真的填了，而且是当前企业内部应用的 App Secret。
3. 看钉钉开放平台里机器人是否开启了 Stream 模式。
4. 再看后端状态页里的最近接收时间、最近发送时间是否开始变化；如果都还是空，先怀疑接收链路没真正切到 `polling`。
