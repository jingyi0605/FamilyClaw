# 升级与回滚

单容器升级的核心规则只有一句话：先备份，再迁移，再启动；如果迁移已经改了 schema，就不能只回退镜像。

## 升级前检查

升级前先确认下面四件事：

1. 宿主机还有足够磁盘空间放一份数据库备份
2. 当前容器健康，`docker exec familyclaw familyclawctl doctor` 能通过
3. 你知道现在正在跑哪个版本，`docker exec familyclaw familyclawctl version` 能读到 manifest
4. 新镜像对应的版本号和变更说明已经确认清楚

## 推荐升级步骤

1. 拉取或构建新镜像。
2. 在旧容器还在运行时，先手工做一次备份：

```bash
docker exec familyclaw familyclawctl backup pre-upgrade
```

3. 停止并删除旧容器，但保留同一个 `/data` 目录。

```bash
docker stop familyclaw
docker rm familyclaw
```

4. 用新镜像和原来的 `/data` 目录启动新容器。
5. 新容器启动时会自动检查当前 schema 是否需要迁移：
   - 需要迁移：先再做一份升级前备份，再执行 `alembic upgrade head`
   - 不需要迁移：直接启动业务服务
6. 启动后重新执行：

```bash
docker exec familyclaw familyclawctl doctor
curl -fsS http://127.0.0.1:8080/api/v1/system/version
```

## 什么时候可以直接回滚

满足任一条件才可以直接切回旧镜像：

- 新旧版本之间没有 schema 变化
- 发布说明明确写了“允许直接回滚”

如果你不确定，就按“不能直接回滚”处理。

## 什么时候不能只回滚镜像

只要新容器已经执行过数据库迁移，就不能直接把镜像切回去了。因为这时数据库结构已经变了，旧应用未必认识新 schema。

## 回滚步骤

### 可直接回滚

1. 停止当前容器
2. 用旧镜像重新创建容器
3. 挂载原 `/data` 目录
4. 再跑一次 `familyclawctl doctor`

### 不能直接回滚

1. 停止当前容器
2. 找到升级前的备份文件，通常在 `/data/backups`
3. 先用同版本或兼容工具恢复数据库
4. 再用旧镜像启动容器

示例：

```bash
docker exec familyclaw familyclawctl restore /data/backups/familyclaw-20260318T120000Z-pre-upgrade.sql.gz
```

## 失败时先看哪里

- `docker logs familyclaw`：先看是 PostgreSQL 启动失败、迁移失败还是 API 没起来
- `/data/backups`：确认升级前备份还在不在
- `/data/runtime/installed-version.json`：确认当前容器已经写入了哪一版安装状态

## 排障原则

- 不要在迁移失败后继续让坏容器对外提供服务
- 不要为了“快一点”跳过备份
- 不要只凭 `latest` 判断你要回到哪个版本
