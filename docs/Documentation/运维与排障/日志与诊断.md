# 日志与诊断

> 这页只讲排障入口和顺序，重点覆盖对话与长期记忆召回链路。

## 这页要解决什么

- 让排障时先知道去哪里看日志、先看哪个组件、再看哪个链路

## 日志来源

- 应用日志：后端标准输出和部署平台收集到的应用日志。
- 对话调试日志：`conversation_debug_logs` 表，按 `session_id + request_id` 看一轮对话发生了什么。
- 记忆注入 trace：`conversation_memory_reads` 表，专门回答“这一轮到底注入了哪些长期记忆”。
- 会话记忆真相源：`conversation_session_summaries` 表，用来确认当前会话摘要有没有生成、覆盖到哪条消息、现在是 `fresh` 还是 `failed`。
- 情节记忆真相源：`episodic_memory_entries` 表，用来确认事件或会话摘要有没有先沉淀到 `L2`。
- 情节记忆修订：`episodic_memory_entry_revisions` 表，用来确认 `L2` 是谁写的、为什么改的。
- 外部知识真相源：`knowledge_documents` 表，用来确认插件原始记录、说明文档、规则文档有没有进入 `L4`。
- 外部知识修订：`knowledge_document_revisions` 表，用来确认 `L4` 的 create/update 有没有留下痕迹。
- 统一召回投影：`memory_recall_documents` 表，用来确认 `L1/L2/L3/L4` 是否都已经进入统一召回。
- 第一阶段兼容投影：`memory_cards` 表里的 `search_text / search_tsv / embedding / projection_version / projection_updated_at`，用于确认 `L3` 自身投影有没有真正刷新。

## 诊断顺序

1. 先拿到 `session_id` 和 `request_id`。
2. 查 `conversation_debug_logs`，按时间看是否已经走到 `memory_context.read`、`orchestrator.completed`、`assistant.completed`。
3. 如果怀疑“答非所问”，再查 `conversation_memory_reads`：
   - 看有没有记录
   - 看每条的 `group_name / layer / score / rank / source_id`
   - 看是不是把 `session_summary`、`stable_facts`、`recent_events` 和 `external_knowledge` 混了
4. 如果怀疑会话上下文丢了，先查 `conversation_session_summaries`：
   - 看 `status` 是不是 `fresh` 或 `stale`
   - 看 `covered_message_seq` 有没有覆盖到当前追问前一轮
   - 看 `summary / open_topics_json / recent_confirmations_json` 是否为空
5. 如果怀疑统一召回没把数据接住，先查 `memory_recall_documents`：
   - 看对应来源有没有 `status=ready`
   - 看 `layer / group_hint / source_kind / source_id` 是否正确
   - 看 `search_text` 是否为空、`search_tsv` 是否生成、`embedding` 是否可用
6. 如果 trace 为空，再看真相源数据本身：
   - `episodic_memory_entries` 是否已经写入 `L2`
   - `episodic_memory_entry_revisions` 是否至少有一条 `create`
   - `knowledge_documents` 是否已经写入 `plugin_raw_record / doc / rule`
   - `knowledge_document_revisions` 是否至少有一条 `create`，更新后是否继续累加 `update`
   - `memory_cards` 是否已经从 `L2` 提升出 `L3`
7. 如果还只剩 `L3` 问题，再看 `memory_cards` 投影字段：
   - `search_text` 是否为空
   - `search_tsv` 是否生成
   - `projection_version` 是否正确
   - `projection_updated_at` 是否更新
8. 如果 recall 降级，检查调试日志或 prompt 注入里是否出现这些原因：
   - `pgvector_unavailable`
   - `embedding_unavailable`
   - `vector_query_failed`
   - `memory_recall_query_failed`
   - `memory_recall_document_query_failed`
9. 最后再判断是数据问题、权限问题、召回问题还是 LLM 回答问题，不要一上来就甩锅模型。

## 常见信号

- `memory_context.read` 存在，但 `conversation_memory_reads` 为空：
  - 先看 recall 分组是不是本来就没命中
  - 再看这轮是不是走了非 free_chat 路径
- `conversation_memory_reads` 只有 `recent_events` 没有 `stable_facts`：
  - 先核对用户问题是否包含稳定事实相关词
  - 再看对应 `memory_cards.search_text` 是否缺词、`search_tsv` 是否为空
- `conversation_memory_reads` 没有 `session_summary`，但本轮其实是同一会话追问：
  - 先查 `conversation_session_summaries.status` 是否仍是 `fresh / stale`
  - 再查 `memory_recall_documents` 里当前 `session_id` 对应的 `L1` 投影是否存在、是否 `status=ready`
- `conversation_memory_reads` 没有 `external_knowledge`：
  - 先查 `knowledge_documents.source_kind` 是否已经写入 `plugin_raw_record / doc / rule`
  - 再查 `memory_recall_documents.group_hint` 是否已经生成 `external_knowledge`
- `knowledge_documents` 有数据，但 revision 是空的：
  - 这不是小问题，说明宿主修订历史断了
  - 先查写入是不是绕过了 `upsert_knowledge_document(...)` 或相关 service
- `episodic_memory_entries` 有数据，但 revision 是空的：
  - 说明 `L2` 写入链路漏了宿主审计
  - 先查事件沉淀和会话摘要沉淀是不是绕过了 revision helper
- 调试日志出现 `session_summary.failed`：
  - 说明摘要后处理失败了，但主链路应该仍然完成回复
  - 继续查异常信息、会话消息序列和 `conversation_session_summaries.status`
- 调试日志出现 `session_summary.updated`，但下一轮还是像失忆：
  - 先看 `covered_message_seq` 是否落后
  - 再看这一轮 trace 里有没有 `group_name=session_summary`、`layer=L1`
- 调试日志出现 `pgvector_unavailable`：
  - 这是允许的降级，不是故障
  - 主链路应该退回 FTS-only，而不是直接失败
- 调试日志出现 `memory_recall_query_failed`：
  - 说明 recall SQL 整体异常，系统已经退回可见卡片兜底
  - 继续查 PostgreSQL 执行错误和 migration 是否完整
- 插件 observation 导入报 400 或命中跨家庭成员：
  - 先查 `subject_member_id` 是否属于当前 `household`
  - 这是宿主强校验，不允许插件替你越权写人
- `knowledge_documents` 已有数据，但还是召回不到：
  - 先查 `memory_recall_documents` 是否同步了对应 `L4` 投影
  - 再看查询词是否真的落在 `search_text`，不要把“没写进投影”误判成“排序不好”
- 用户反馈“为什么这么答”：
  - 先查 `conversation_memory_reads`
  - 再对照 `conversation_debug_logs.payload.trace_items`
  - 不要直接猜 prompt
