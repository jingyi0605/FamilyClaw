# FamilyClaw 完整文档目录与写作计划

## 目标

这份计划只解决一件事：把 `FamilyClaw` 的项目文档拆成稳定栏目、稳定文件和稳定写作顺序。  
先把骨架定死，后面填正文时就不用反复搬目录、改链接、改导航。

## 文档边界

- `docs/`：放项目级、跨模块、跨阶段的正式文档
- `specs/`：放需求拆解、设计过程、任务跟踪和验收过程
- 不做的事：不把安装说明继续分散到某个 `specs/<id>/docs/` 里，不把长期有效的开发规范只写在聊天记录里

## 命名规则

- 路径只保留语义名称，不带日期，不带树编号
- 树编号、版本、状态放在文档 frontmatter 中维护
- 展示顺序由 `VitePress` 配置和 `order` 元数据控制，不靠文件名字典序硬排

## 每一栏到底写什么

### 开始这里

- 解决“新来的人先看什么”
- 控制页数，不超过 3 页
- 不写大量细节，只给阅读路径和入口

### 安装部署

- 写清环境要求、依赖、初始化步骤、配置来源、部署方式
- 做完后，新同事应该能在不翻聊天记录的前提下完成安装
- 不写业务功能介绍

### 使用指南

- 面向普通使用者和测试同学
- 写用户能看到的功能路径、操作步骤、注意事项
- 不写实现细节，不讲源码

### 开发指南

- 面向接手代码的人
- 讲仓库结构、启动路径、前后端边界、插件扩展、测试提交流程
- 只写稳定规则，不把临时排障过程当长期文档

### 运维与排障

- 讲日志位置、常见故障、升级、回滚、排查顺序
- 让值班和联调时能快速定位问题
- 不重复安装步骤，避免两边内容打架

### 参考资料

- 放配置项、接口项、术语解释这类查询型内容
- 允许以后自动生成部分内容，但先保留人工模板

### 文档规范

- 约束写法、图片、文件命名、排版组件和链接方式
- 目标是降低后续维护成本，而不是写作文比赛

### 存量资料

- 只做入口和归类
- 不迁移、不重写当前已有规范和历史文档

## 写作顺序

### 第一阶段

1. 快速开始
2. 环境要求
3. 本地安装
4. 配置说明

结果：新同事能把项目拉起来，不再靠口口相传。

### 第二阶段

1. 用户快速上手
2. 家庭与成员管理
3. AI 配置与插件使用
4. 常见操作

结果：测试、产品和普通体验者知道怎么把主要功能跑通。

### 第三阶段

1. 开发环境准备
2. 仓库结构说明
3. 前端开发
4. 后端开发
5. 插件开发
6. 测试与提交流程

结果：开发者接手时，先看文档就能定位代码入口和开发边界。

### 第四阶段

1. 日志与诊断
2. 常见故障排查
3. 升级与回滚
4. 配置参考
5. 接口参考
6. 术语表

结果：运维和联调时不用到处翻聊天记录。

## 每页模板最低要求

- 这页解决什么问题
- 目标读者是谁
- 前置条件是什么
- 操作步骤或说明结构是什么
- 需要哪些截图或示意图
- 做完后怎么验证这页真的可用
- 这页明确不覆盖什么，防止范围失控

## 图片策略

- 单页图片：放在同目录 `assets/`
- 多页共用图片：放在 `docs/Documentation/.vitepress/public/`
- 默认优先使用 `webp` 或压缩过的 `png`
- 需要控宽度、标题、边框时，统一使用 `<DocImage />`
- 不接受把大截图直接裸贴在正文里撑坏布局

## 责任划分建议

- 产品/设计：补使用指南中的操作目标、界面变化和术语
- 前端开发：补前端开发、主要页面截图、交互说明
- 后端开发：补安装部署、后端开发、配置和运维诊断
- 插件负责人：补插件开发和插件相关参考资料

## 验收标准

- 文档站能构建成功
- 站内导航和侧边栏不死链
- 每个大栏目至少有入口页或模板页
- 图片插入方式只有一套主路径，不各写各的
- 新人第一次进入仓库，能在 5 分钟内找到自己该看的页面
