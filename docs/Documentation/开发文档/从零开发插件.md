# 从零开发插件

这是一个最小可执行 walkthrough，只讲现在正确的开发路径。

## 第一步：把目录建对

假设我们要开发 `health-demo-sync`，源码目录应该建在：

- `apps/api-server/plugins-dev/health-demo-sync/`

不要建到：

- `apps/api-server/data/plugins/official/`

这是历史目录，已经废弃。

## 第二步：准备最小目录

```text
apps/api-server/plugins-dev/health-demo-sync/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    integration.py
```

上面这套是“主仓库内最小联调模板”。

如果你做的是独立维护的插件仓库，或者后面准备进插件市场，当前也允许直接使用 `official_weather` 那种仓库布局：

```text
health-demo-sync-repo/
  README.md
  requirements.txt
  health_demo_sync/
    __init__.py
    manifest.json
    integration.py
    resources/
```

然后把整个仓库放到：

- `apps/api-server/plugins-dev/health-demo-sync-repo/`

这里要记住一条硬口径：

- `plugins-dev/<plugin>/` 规定的是开发目录位置
- 外部插件仓库内部怎么排，可以保留它自己的正式仓库结构

## 第三步：先写 `manifest.json`

最少要写清楚：

- `id`
- `name`
- `version`
- `types`
- `permissions`
- `risk_level`
- `entrypoints`

如果插件需要配置，再补 `config_specs`。

## 第四步：把入口写通

入口模块路径必须和 `manifest.json` 里的声明一致。

比如：

```json
{
  "entrypoints": {
    "integration": "plugin.integration.sync"
  }
}
```

那你就必须提供：

```python
def sync(payload=None):
    return {"ok": True, "payload": payload or {}}
```

如果你采用独立仓库 + 子目录插件包的布局，也一样，原则只有一个：

- 入口模块路径必须和最终运行目录里的真实文件一致
- 不要写一个市场看起来对、运行时却导不进来的假入口

## 第五步：先按开发版联调

这个阶段不用先安装：

- 把源码放进 `plugins-dev`
- 直接确认插件注册表里能看到它
- 验证执行入口和配置行为是否正确

如果当前家庭里已经装了同 `plugin_id` 的第三方插件，现在的正确行为是：

- 插件管理页会同时显示开发版和已安装版
- 默认优先开发版启用
- 当前实际跑的是开发版时，页面会显示“开发插件生效中”
- 想测试正式安装版，直接切换启用，不需要手动移动 `plugins-dev` 目录

## 第六步：只有验证安装链路时才安装

### 本地安装

- 目标目录是 `data/plugins/third_party/local/`

### 市场安装

- 目标目录是 `data/plugins/third_party/marketplace/`
- 市场安装会保留能同时覆盖 `manifest.json` 和运行包目录的最小目录。像“仓库根目录放 `manifest.json`，代码放 `plugin/` 子目录”这种官方市场结构，现在可以直接安装，不需要为了迁就宿主去硬改仓库布局。

无论哪种安装方式，都不要把开发源码直接塞进安装目录。

## 第七步：记住冲突规则

- `builtin` 仍然最高优先级
- `plugins-dev` 不能覆盖 `builtin`
- `plugins-dev` 和已安装 third_party 同名时，两个变体都要保留可见
- 同一时刻只允许启用一个变体
- 默认仍优先开发版，方便联调

## 最后检查

至少确认这几件事：

1. `manifest.json` 能通过校验。
2. 入口模块能正常导入。
3. 插件注册表里能看到插件。
4. 如果存在同名安装版，页面能同时看到开发版和已安装版。
5. 能直接在页面里切换启用已安装版，不需要手动挪目录。
6. 删除安装产物不会删除 `plugins-dev` 源码。
