# 本地调试

这页只讲第三方插件的本地开发和联调，不讲旧 `official` 目录。

## 开发源码应该放哪里

统一放到：

- `apps/api-server/plugins-dev/`

不要再放到：

- `apps/api-server/data/plugins/official/`
- `apps/api-server/data/plugins/third_party/local/`
- `apps/api-server/data/plugins/third_party/marketplace/`

原因很简单：

- `plugins-dev` 是开发源码目录，也是开发覆盖源。
- `third_party/local` 和 `third_party/marketplace` 是运行时安装目录。
- `data/plugins/official` 是历史目录，已经废弃。

## 现在怎么处理“开发插件和已安装插件同名”

### 规则

- `builtin > plugins-dev > 已安装 third_party`
- `plugins-dev` 只覆盖第三方插件，不覆盖 `builtin`
- 同名开发插件会直接覆盖家庭注册表中第三方插件的 manifest、entrypoint 和 runner 配置
- 市场插件被覆盖时，市场安装状态和版本治理信息继续保留

### 这意味着什么

- 你在 `plugins-dev/foo/` 里改源码，当前家庭里如果已经装了 `plugin_id=foo` 的第三方插件，界面和执行都会优先看到开发版。
- 你删除本地安装或市场安装产物时，不会把 `plugins-dev/foo/` 删掉。
- 你想验证“纯安装态行为”时，必须临时移走或重命名 `plugins-dev` 中的同名插件。

### builtin 冲突怎么处理

- 直接失败。
- `plugins-dev` 里的同名插件不会覆盖内置插件。

## 推荐调试顺序

1. 在 `plugins-dev/<plugin_dir>/` 下改源码。
2. 先直接检查 `manifest.json` 和入口模块能否 import。
3. 再通过宿主已有测试或服务函数验证注册表、配置和执行行为。
4. 只有在需要验证安装链路时，才把插件打包走“本地安装”或“市场安装”流程。

## 什么时候需要真正安装

以下场景才需要落到运行时安装目录：

- 验证 ZIP 本地安装链路。
- 验证 marketplace 安装、升级、回滚、卸载。
- 验证启动同步只从安装态目录恢复挂载。

这时才会用到：

- `apps/api-server/data/plugins/third_party/local/`
- `apps/api-server/data/plugins/third_party/marketplace/`

## 最小自测建议

先做这三层，不要一上来就全链路乱跑：

1. 静态层：`manifest.json`、`entrypoints`、资源路径、依赖声明。
2. 模块层：直接 import 入口函数，验证返回结构。
3. 服务层：验证插件注册表、配置读写、执行入口和启停状态。

## 常见坑

- 还把 `plugins-dev` 当成“不会加载的纯示例目录”。
- 误以为开发覆盖也会覆盖 `builtin`。
- 在 `data/plugins/third_party/...` 里直接改文件，结果下次安装或卸载又被覆盖。
- 调试市场插件时忘了 `plugins-dev` 同名覆盖仍然生效，误以为市场安装没有更新。

## 一句话结论

开发时改 `plugins-dev`，安装时写 `third_party/local` 或 `third_party/marketplace`，冲突时由 `plugins-dev` 覆盖第三方运行态，但永远不盖过 `builtin`。
