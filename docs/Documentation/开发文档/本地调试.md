# 本地调试

这页只解决 4 个问题：

1. 开发插件前，本地环境怎么准备。
2. 插件代码应该放哪里。
3. 不启动整套服务的前提下，最小怎么调试。
4. 如果插件有自己的依赖，应该装到哪里。

## 先把现实边界说清楚

当前要分两件事看：

- 现有仓库已经稳定支持：内置插件在主进程里 import 执行。
- 第三方推荐目标模式：插件在同容器子进程 runner 里执行。

所以第三方插件开发时，别再假设“把代码塞进主仓库源码树里”才算对。

最稳的方式是：

- 在独立插件目录里开发代码。
- 给插件准备自己的 `requirements.txt`。
- 在插件自己的虚拟环境里先把入口跑通。
- 最后再用项目提供的 runner 协议或后台任务链路联调。

## 本地环境最小要求

先满足这几个条件：

1. Python `3.11+`
2. 能用 `venv` 创建虚拟环境
3. 能在本地运行 `pip`
4. 有基本的 Git 使用能力

后端项目依赖可以直接看 `apps/api-server/pyproject.toml`。

## 推荐环境准备步骤

以下命令都在 `apps/api-server/` 目录执行。

### 1. 创建主项目虚拟环境

```bash
python -m venv .venv
```

Windows 激活：

```bash
.venv\Scripts\activate
```

macOS / Linux 激活：

```bash
source .venv/bin/activate
```

### 2. 安装后端项目依赖

```bash
pip install -e .
```

这样装的好处很直接：

- `app.*` 模块路径能直接 import。
- 改后端代码或内置插件代码时，不用每次重装包。

### 3. 给插件自己建独立环境

第三方插件不要默认把依赖装进主后端环境。

更合理的方式是：

```bash
python -m venv .plugin-venv
```

然后安装插件自己的依赖：

```bash
.plugin-venv\Scripts\pip install -r requirements.txt
```

或在 macOS / Linux：

```bash
source .plugin-venv/bin/activate
pip install -r requirements.txt
```

这套环境后面就是 runner 要消费的环境，不应该去污染主 API 进程环境。

## 插件代码现在该放哪里

如果你是第三方插件开发者，当前推荐目录就是你自己的插件仓库。

最小目录建议长这样：

```text
your-plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    integration.py
    action.py
    driver.py
  tests/
```

如果你是项目维护者，要继续维护当前内置插件，现有位置仍然是：

- `apps/api-server/app/plugins/builtin/`

如果你把代码放到仓库外面，或者放到一个没进 Python 路径的位置，`manifest.json` 里的入口模块路径就会直接失效。

## 开发时最小需要改哪些东西

一个最小可调试插件，通常先准备这几个文件：

1. `manifest.json`
2. `__init__.py`
3. 对应能力的入口代码文件，比如 `integration.py`、`action.py` 或 `driver.py`
4. 如果要走完整验证，再补 `tests/`

第一轮先别急着做复杂逻辑，先把这三件事跑通：

- `manifest.json` 能通过校验
- 入口模块能 import
- 入口函数直接调用时能返回对的结构

## 本地调试先从哪一层开始

别一上来就启动整套系统查半天日志。按这个顺序调最稳。

### 第一层：静态检查

先看这几件事：

1. `manifest.json` 字段是不是齐全
2. `entrypoints` 路径和真实文件是不是一致
3. 函数名是不是写对
4. `types` 和入口是不是一一对应

### 第二层：直接 import 调函数

先在 Python 解释器或测试里直接调插件入口函数。

例如：

```python
from app.plugins.builtin.health_basic.integration import sync

result = sync({"member_id": "demo-member"})
print(result)
```

这一步的目标不是跑完整系统，而是先确认：

- 模块路径没写错
- 函数能调用
- 返回结构基本正确

### 第三层：走项目里的真实服务函数

如果是内置 `integration` 插件，后面再去测：

- `execute_plugin()`
- `integration.refresh`
- 后台任务与 worker 执行链路

如果你接手的是存量 reader 链路，再去对照：

- `run_plugin_sync_pipeline()`
- Agent 相关桥接入口

### 第四层：走后台任务接口和联调入口

更贴近当前系统主路径的验证，至少要看这些入口：

- `POST /api/v1/plugin-jobs`
- `GET /api/v1/plugin-jobs/{job_id}`
- `GET /api/v1/plugin-jobs`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/plugin-invocations`
- `POST /api/v1/ai-config/{household_id}/agents/{agent_id}/action-plugin-invocations`

顺序建议是：

1. 先过静态检查
2. 再过内部执行层测试
3. 再走后台任务接口和 worker 联调
4. 最后如果需要，再做 Agent / 动作入口联调

## 本地调试时最常见的坑

### 模块路径写错

症状：入口加载失败。

常见原因：

- 目录名和模块名对不上
- `manifest.json` 里的模块路径没跟着改
- 函数名改了，但 entrypoint 还指旧名字

### 插件代码放错位置

症状：`import_module()` 找不到模块。

常见原因：

- 插件没放在当前可加载路径里
- 少了 `__init__.py`

### 一上来就联调整套链路

症状：报错一堆，分不清是 manifest 错、入口错、返回值错，还是数据库没准备好。

解决办法：

- 先拆成静态检查 -> 入口函数 -> 服务层链路 -> HTTP 联调

## 一句话记住当前开发方式

对第三方开发者来说，当前最正确的姿势是：

- 先在独立插件目录里把入口和依赖做通。
- 再在插件自己的 venv 里做静态检查和最小测试。
- 后续通过 runner 协议或后台任务链路接到主项目里验证。
