# 本地调试

这页只讲第三方插件的本地开发和联调，不再讲旧 `official` 目录。

## 开发源码应该放哪里

统一放到：

- `apps/api-server/plugins-dev/`

不要再放到：

- `apps/api-server/data/plugins/official/`
- `apps/api-server/data/plugins/third_party/local/`
- `apps/api-server/data/plugins/third_party/marketplace/`

原因很简单：

- `plugins-dev` 是开发源码目录，而且会参与加载。
- `third_party/local` 和 `third_party/marketplace` 是运行时安装目录。
- `data/plugins/official` 是历史目录，已经废弃。

## 现在怎么处理“开发插件和已安装插件同名”

### 规则

- `builtin` 仍然最高优先级。
- `plugins-dev` 不能覆盖 `builtin`。
- 当 `plugins-dev` 和已安装 third_party 的 `plugin_id` 相同时，插件管理页会同时显示两个变体。
- 同一时刻只允许启用一个变体。
- 默认优先开发版，这样改完源码就能直接联调。

### 这意味着什么

- 你在 `plugins-dev/foo/` 里改源码时，如果当前家庭已经装了 `plugin_id=foo` 的第三方插件，页面上会同时看到“开发版”和“已安装版”。
- 当前实际跑的是开发版时，插件列表和插件详情都会显示“开发插件生效中”。
- 你想验证正式安装版时，直接切换启用到“已安装版”就行，不需要临时移动、重命名或删除 `plugins-dev/foo/`。
- 删除本地安装包或市场安装包时，不会把 `plugins-dev/foo/` 源码删掉。
- marketplace 插件的安装状态、配置状态、版本治理信息不会因为存在开发版而消失。

### builtin 冲突怎么处理

- 直接以 `builtin` 为准。
- `plugins-dev` 里的同名插件不会覆盖内置插件。

## 推荐调试顺序

1. 在 `plugins-dev/<plugin_dir>/` 下改源码。
2. 先检查 `manifest.json`、依赖声明和入口模块能否正常导入。
3. 通过已有测试或服务接口验证插件注册表、配置、执行和页面展示。
4. 如果同 `plugin_id` 已经存在安装版，到插件管理页确认“开发版”和“已安装版”都能看见，再按需要切换启用。
5. 只有在验证安装链路时，才去跑“本地安装”或“市场安装”流程。

## 什么场景下才需要真正安装

以下场景才需要落到运行时安装目录：

- 验证 ZIP 本地安装链路。
- 验证 marketplace 安装、升级、回滚、卸载。
- 验证启动同步只从安装目录恢复挂载。

这时才会用到：

- `apps/api-server/data/plugins/third_party/local/`
- `apps/api-server/data/plugins/third_party/marketplace/`

## 最小自测建议

先做这三层，不要一上来就全链路乱跑：

1. 静态层：`manifest.json`、`entrypoints`、资源路径、依赖声明。
2. 模块层：直接导入入口函数，验证返回结构。
3. 服务层：验证插件注册表、配置读写、执行入口和启停状态。

## 常见坑

- 还把 `plugins-dev` 当成“不会加载的示例目录”。
- 误以为开发版出现后，已安装版就应该在页面上消失。
- 为了测试安装版，手动去挪 `plugins-dev` 目录。
- 误以为开发版也能覆盖 `builtin`。
- 在 `data/plugins/third_party/...` 里直接改文件，结果下次安装或卸载又被覆盖。

## 一句话结论

开发时改 `plugins-dev`，安装时看 `third_party/local` 或 `third_party/marketplace`；同 `plugin_id` 冲突时保留两个变体可见，默认开发版启用，想测安装版直接切换，不要再挪目录。
