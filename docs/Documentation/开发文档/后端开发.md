# 后端开发

FamilyClaw 后端不是一个“大而全工具箱”，而是一个很明确的宿主内核。你改后端前先记住一句话：

宿主负责统一规则、统一数据、统一权限和统一调度；领域特例优先放插件，不要继续往宿主里塞。

## 当前技术栈

- Web 框架：FastAPI
- ORM：SQLAlchemy 2.x
- 迁移：Alembic
- 数据库：PostgreSQL
- 实时链路：WebSocket
- 运行入口：`apps/api-server/app/main.py`

## 真实启动链路

应用启动时会在 `lifespan` 里做这些事：

1. 初始化日志
2. 确保引导管理员账号和待完成 bootstrap 账号存在
3. 同步持久化插件状态
4. 启动 `PluginJobWorker`
5. 启动 `ScheduledTaskWorker`
6. 启动 `ChannelPollingWorker`

也就是说，后端不是单纯的 HTTP 服务，它本身就带后台 worker。你随便往主事件循环里塞阻塞逻辑，就是在同时拖垮 API、WebSocket 和 worker。

## 不能碰的硬规则

### 1. 数据库结构只能通过 Alembic 改

硬要求：

- 新增表、字段、索引、约束，必须带 migration。
- 不允许在应用启动、脚本或测试里偷用 `create_all()`、`drop_all()`。
- 遇到脏库，先核对结构，再 `alembic stamp`，不要碰运气。

### 2. 阻塞逻辑不能塞进主事件循环

实话实说，这条规则不是“建议”，是红线：

- 同步 DB
- 同步 HTTP / SDK
- 同步插件执行
- 长时间 CPU 计算
- `while True + sleep` 轮询

这些都必须下沉，不能直接挂在 `async` 入口里赌运气。

高频事实源：

- `apps/api-server/app/core/blocking.py`
- `apps/api-server/app/core/worker_runtime.py`

可以直接这样理解：

- `async` 入口负责收请求、做轻量编排、返回结果。
- 任何会卡线程、会卡事件循环、会卡 WebSocket 的活，都要显式下沉。
- 后台常驻逻辑不要自己手搓 `while True + sleep`，统一走现有 worker 体系。

### 3. 插件可用性语义必须统一

关键点：

- 查看/配置类场景用 `get_household_plugin(...)`
- 真正会执行插件的场景必须用 `require_available_household_plugin(...)`
- 禁用语义统一返回 `409` + `plugin_disabled`

如果你在某个接口里绕开这层统一判断，结果通常很丑：

- 页面看起来已经禁用了插件
- 但后台任务还在继续跑
- 或者会话链路还能继续调这个插件

这不是小 bug，这是边界失守。

### 4. 宿主不要继续吸收领域特例

你如果发现自己正在写：

- 专门给天气、电费、健康、步数再开一个宿主专用聚合服务
- 给某个供应商单独加一套宿主内特殊处理
- 让插件直接写宿主核心表

那就是方向错了。

FamilyClaw 当前的正式做法很明确：

- 宿主维护统一模型、统一权限、统一调度、统一审计。
- 插件吸收第三方差异、厂商特例和接入细节。
- 宿主消费的是标准结果，不是第三方原始形态。

## 主要接口组

下面这些是当前开发最常碰到的接口组：

| 分组 | 路由前缀 | 主要用途 |
| --- | --- | --- |
| 系统健康 | `/api/v1/healthz` | 健康检查 |
| 认证 | `/api/v1/auth` | 登录、登出、当前身份 |
| 家庭 | `/api/v1/households` | 家庭详情、初始化状态 |
| 首页仪表盘 | `/api/v1/dashboard/home` | 首页卡片与布局 |
| 对话 | `/api/v1/conversations` | 会话、回合、提案确认 |
| 记忆 | `/api/v1/memories` | 记忆卡片、修订、查询、纠错 |
| AI 配置 | `/api/v1/ai-config` | 模型供应商、路由、Agent、主题、插件挂载 |
| 集成 | `/api/v1/integrations` | 集成实例、资源、动作 |
| 通讯账号 | `/api/v1/ai-config/{household_id}/channel-accounts` | 渠道账号、绑定、探测 |
| 插件市场 | `/api/v1/plugin-marketplace` | 市场源、目录、安装任务 |
| 计划任务 | `/api/v1/scheduled-tasks` | 任务定义、启停、草稿确认 |
| 声纹 | `/api/v1/voiceprints` | 建档、汇总、删除 |

声纹链路的当前约定：

- 默认建档 6 轮，不再是 3 轮
- 前 4 轮使用同一句基础短句，后 2 轮切到不同短句
- 实时识别在送入声纹模型前会先做静音裁剪、基础 VAD 和响度归一化

## 推荐改动顺序

### 新增或修改后端功能

1. 先看现有模块和数据结构是不是已经能承载
2. 再判断这件事应该进宿主还是进插件
3. 改 model 前先确认 migration 方案
4. 改接口前先看 `packages/user-core` 里有没有现成类型
5. 涉及阻塞调用时，先想清楚下沉边界
6. 最后补文档和定向验证

### 新增接口

1. 先加 endpoint
2. 再补 service / repository
3. 再补 `packages/user-core/src/domain/types.ts`
4. 再补 `packages/user-core/src/api/create-api-client.ts`
5. 最后补使用手册或开发手册里受影响的文档

## 最后一句难听但有用的话

如果一个改动需要靠一堆 `if/else` 去兜插件状态、线程边界、迁移版本和接口兼容，那多半不是代码还不够多，而是数据结构或者执行边界设计错了。先把结构理顺，再写代码。
