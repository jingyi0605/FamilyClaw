# 环境准备

这页只解决一件事：让新开发者第一次接手仓库时，知道机器上要装什么、先跑什么、哪些命令才是当前仓库真的在用的。

## 本机依赖

- 操作系统：macOS、Linux、Windows 都可以。
- Python：`3.11+`。后端最低要求是 `3.11`，启动脚本会优先选择机器上可用的 `3.11+` 解释器。
- Node.js：`20+`，根目录文档站和 `apps/user-app` 都依赖它。
- PostgreSQL：`15+`，开发和测试基线都是 PostgreSQL，不支持 SQLite 兜底。
- Git + Bash：仓库里的后端和网关启动脚本都是 Bash，Windows 建议直接用 Git Bash。

## 仓库里真正需要关注的目录

```text
FamilyClaw/
├── apps/
│   ├── api-server/               # FastAPI + SQLAlchemy + Alembic
│   ├── user-app/                 # Taro + React + TypeScript
│   ├── open-xiaoai-gateway/      # 语音网关
│   ├── start-api-server.sh       # 后端开发启动脚本
│   └── start-open-xiaoai-gateway.sh
├── packages/
│   ├── user-core/                # 前端接口与共享类型
│   ├── user-platform/            # 平台适配
│   └── user-ui/                  # UI 组件与主题能力
├── docs/Documentation/           # 官方文档站源码（VitePress）
├── docker/                       # 镜像运行脚本与 nginx/supervisor 配置
└── specs/                        # 规格与设计资料
```

## 第一次拉起仓库的正确顺序

### 1. 克隆仓库

```bash
git clone <你的仓库地址>
cd FamilyClaw
```

### 2. 安装根目录依赖

根目录依赖主要服务文档站和工作区包链接：

```bash
npm install --legacy-peer-deps
```

### 3. 安装前端依赖

`apps/user-app` 不是根 workspace 自动装完就能跑的那种目录，它有自己的一套依赖：

```bash
npm --prefix ./apps/user-app install --legacy-peer-deps
```

### 4. 准备 PostgreSQL

至少要有一个本地 PostgreSQL 数据库，默认示例配置如下：

- 数据库：`familyclaw`
- 用户：`familyclaw`
- 密码：`change-me`

如果你自己改了用户名、密码或端口，就同步改 `.env`，不要指望代码自动猜对。

### 5. 准备后端环境变量

```bash
cp apps/api-server/.env.example apps/api-server/.env
```

最关键的几个配置项：

- `FAMILYCLAW_DATABASE_URL`
- `FAMILYCLAW_TEST_DATABASE_URL`
- `FAMILYCLAW_BASE_URL`：可选。插件配置预览或认证回调需要生成对外可访问地址时，宿主优先用它；没配时退回当前请求的 `base_url`
- `FAMILYCLAW_BOOTSTRAP_ADMIN_USERNAME`
- `FAMILYCLAW_BOOTSTRAP_ADMIN_PASSWORD`
- `FAMILYCLAW_BOOTSTRAP_HOUSEHOLD_USERNAME`
- `FAMILYCLAW_BOOTSTRAP_HOUSEHOLD_PASSWORD`
- `FAMILYCLAW_AI_PROVIDER_CONFIGS`
- `OPENAI_API_KEY`（如果你启用 OpenAI 兼容供应商）

### 5.1 同步项目版本元数据

版本现在按两条规则处理，不要再手工各改各的：

- 后端开发环境默认直接读取仓库根目录 `VERSION`，不再在后端配置里手写正式版本号。
- `pyproject.toml`、`package.json`、`package-lock.json` 和 `Dockerfile` 这类不会自己读取 `VERSION` 的文件，统一用脚本同步。

常用命令：

```bash
# Windows
py -3.11 apps/scripts/sync_versions.py
py -3.11 apps/scripts/sync_versions.py --check

# macOS / Linux
python3 apps/scripts/sync_versions.py
python3 apps/scripts/sync_versions.py --check
```

建议习惯：

- 改完根目录 `VERSION` 后，先跑一次同步脚本。
- 提交发版相关改动前，再跑一次 `--check`，别把漂移版本带进提交里。

### 6. 启动后端

仓库已经给了正式开发脚本，别自己拼散装命令：

```bash
bash apps/start-api-server.sh
```

如果你机器上同时装了多个 Python，脚本会自动挑一个可用的 `3.11+` 解释器；需要强制指定时，再用 `PYTHON_BIN=<python路径>`。

这个脚本会做几件事：

- 自动创建 `apps/api-server/.venv`
- 自动安装 Python 依赖
- 自动检查 Alembic 版本并执行 `alembic upgrade head`
- 以热重载方式启动 `uvicorn app.main:app`

默认监听：

- API：`http://127.0.0.1:8000`
- 健康检查：`http://127.0.0.1:8000/api/v1/healthz`
- OpenAPI：`http://127.0.0.1:8000/docs`

### 7. 启动前端 H5

```bash
npm run dev:user-app:h5
```

H5 开发环境会把 `/api` 代理到 `http://127.0.0.1:8000`。实际访问地址以终端输出为准，不要自己脑补端口。

如果你要调试手机上的页面，先跑这个命令，再做下面几件事：

- 真机调 H5：让手机和电脑在同一个局域网，打开终端里打印出来的局域网地址，不要只用 `localhost`。
- 浏览器模拟移动端：直接在桌面浏览器 DevTools 里切到设备模式，优先查布局、滚动、点击热区和横向溢出。
- 这条命令只负责 H5，不是原生 RN 调试。原生 RN 要走 `apps/user-app` 里的 `build:ios` / `build:android` 链路，`npm run dev:user-app:h5` 代替不了它。

### 8. 启动文档站

```bash
npm run docs:dev
```

文档站地址同样以终端输出为准。

## 最小自检清单

第一次接手项目，至少确认这几件事：

- `bash apps/start-api-server.sh` 能正常跑完，终端里没有迁移失败。
- `GET /api/v1/healthz` 返回成功。
- H5 页面能打开，并且 `/api` 请求不是 404。
- 登录页能出现，初始化向导能正常进入。
- `npm run docs:build` 可以成功构建文档站。

## 必读规范

开发时以当前官方文档站里的开发文档为准，不要再把规则理解建立在 `docs/Documentation` 之外的资料上。

推荐阅读顺序：

1. 先看 [目录结构](./目录结构.md)，确认仓库哪些目录才是主工作区。
2. 动后端前先看 [后端开发](./后端开发.md)，里面已经把数据库迁移、事件循环、后台 worker、接口入口和改动顺序写全了。
3. 动插件前先看 [插件开发](./插件开发.md)、[插件规范](./插件规范.md)、[字段规范](./字段规范.md)、[对接方式](./对接方式.md)。
4. 要交付插件时再看 [实例插件](./实例插件.md) 和 [插件提交](./插件提交.md)。

你至少要记住下面这些硬规则：

- 数据库结构只能通过 Alembic migration 改，不能在业务代码里偷用 `create_all()` 或 `drop_all()`。
- 任何同步阻塞逻辑都不能直接塞进 `async` 入口，长耗时逻辑要下沉到 worker 或阻塞适配层。
- 插件启用、禁用、执行边界必须统一，不能某个页面自己拦，后台任务又偷偷跑。
- 插件负责第三方适配，宿主负责统一权限、统一标准、统一调度，不要反着来。

## Windows 开发注意事项

- 推荐直接用 Git Bash 跑 `apps/start-api-server.sh`。
- 如果你强行用 PowerShell，就自己承担路径、激活脚本和换行差异的问题。
- `apps/user-app` 的 H5 开发仍然可以在 PowerShell 里跑，但前后端混用时，统一用 Git Bash 最省事。

## 不要做的事

- 不要在业务代码里用 `create_all()` 或 `drop_all()` 硬改数据库。
- 不要自己手写另一套后端启动命令替代 `apps/start-api-server.sh`。
- 不要把前端显示文案写死到组件里，新增页面必须走统一 i18n。
- 不要去改 `docs/Documentation/.vitepress/dist` 里的构建产物。
