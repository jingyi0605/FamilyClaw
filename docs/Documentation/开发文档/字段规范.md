# 字段规范

这页只讲当前插件 manifest 和配置 schema 里最容易写错的字段。

事实源：

- `apps/api-server/app/modules/plugin/schemas.py`
- 仓库内置插件的 `manifest.json` 实例
- 当前文档站内的 [插件规范](./插件规范.md) 与 [实例插件](./实例插件.md)

## manifest 核心字段

| 字段 | 是否必需 | 说明 |
| --- | --- | --- |
| `id` | 是 | 插件唯一标识，后端和计划任务都靠它找插件 |
| `name` | 是 | 展示名称 |
| `version` | 是 | 插件版本 |
| `api_version` | 大多数类型是 | 插件协议版本 |
| `types` | 是 | 插件类型数组 |
| `permissions` | 是 | 权限声明 |
| `risk_level` | 是 | `low` / `medium` / `high` |
| `triggers` | 是 | 允许的触发方式，如 `manual`、`schedule` |
| `entrypoints` | 视类型而定 | 入口函数路径 |
| `capabilities` | 是 | 类型对应的能力声明 |

## `entrypoints` 怎么写

入口必须是：

- Python 模块路径 + 函数名

例如：

```json
"entrypoints": {
  "integration": "app.plugins.builtin.health_basic.integration.sync"
}
```

错的写法包括：

- 只有模块名，没有函数名
- 空字符串
- 路径和真实代码对不上

## `types` 和 `entrypoints` 必须一致

例如：

- `types` 包含 `integration`，就应该有 `entrypoints.integration`
- `types` 包含 `channel`，就应该有 `entrypoints.channel`
- `theme-pack`、`locale-pack`、`ai-provider` 这类类型不一定需要可执行入口，但能力声明必须完整

## `config_specs` 的作用

如果插件需要用户填写配置，就用 `config_specs`，不要把配置协议散落到 README 或代码注释里。

典型结构：

```json
{
  "scope_type": "channel_account",
  "title": "飞书账号配置",
  "schema_version": 1,
  "config_schema": {
    "fields": []
  },
  "ui_schema": {
    "sections": [],
    "field_order": [],
    "widgets": {}
  }
}
```

## 配置字段类型

当前常见字段类型包括：

- `string`
- `text`
- `integer`
- `number`
- `boolean`
- `enum`
- `multi_enum`
- `secret`
- `json`

当前常见 widget 包括：

- `input`
- `password`
- `textarea`
- `switch`
- `select`
- `multi_select`
- `json_editor`

## 动态选项字段

如果某个字段的选项不是写死的，就用：

- `option_source`

当前正式支持的动态选项源：

- `region_provider_list`
- `region_catalog_children`

注意：

- `enum_options` 和 `option_source` 不要同时乱写
- 依赖联动时要配好 `depends_on`
- 依赖变化需要清空旧值时要写 `clear_on_dependency_change`

## 不同插件类型的额外重点

### `integration`

重点看：

- `capabilities.integration.domains`
- `instance_model`
- `refresh_mode`
- `supports_discovery`
- `supports_actions`
- `supports_cards`

### `channel`

重点看：

- `capabilities.channel.platform_code`
- `inbound_modes`
- `delivery_modes`
- `supports_member_binding`
- `config_specs.scope_type = channel_account`

### `ai-provider`

重点看：

- `capabilities.ai_provider.adapter_code`
- `field_schema`
- `supported_model_types`
- `runtime_capability`

### `theme-pack`

重点看：

- `theme_id`
- `display_name`
- `tokens_resource` / `entry_resource`
- `resource_source`
- `resource_version`
- `platform_targets`

## 最容易犯的错

- `id` 改了，但 README、计划任务引用、配置记录没同步
- 声明了 `schedule` trigger，却没有实现对应能力
- `field_schema` 和运行时真实读取字段不一致
- 把用户文案硬编码进代码，不走配置和 i18n 资源

如果字段设计本身就糊，后面调试起来会比写代码更恶心。
