# 字段规范

这页只讲当前插件 `manifest.json` 和配置 schema 里最容易写错的字段。

事实源：

- `apps/api-server/app/modules/plugin/schemas.py`
- 仓库内置插件的 `manifest.json`
- 当前官方文档站内的 [插件规范](./插件规范.md) 和 [配置接入](./配置接入.md)

## 最小 manifest 该有什么

每个插件至少要声明：

- `id`
- `name`
- `version`
- `api_version`
- `types`
- `permissions`
- `entrypoints`
- `capabilities`

如果插件需要配置，再加：

- `config_specs`
- `locales`

如果插件准备进入插件市场，还必须再补一条：

- `compatibility.min_app_version`

最小示例：

```json
{
  "id": "demo-plugin",
  "name": "示例插件",
  "version": "0.1.0",
  "api_version": 1,
  "types": ["integration"],
  "permissions": ["device.read"],
  "entrypoints": {
    "integration": "demo_plugin.integration.sync"
  },
  "capabilities": {
    "integration": {
      "domains": ["demo"]
    }
  },
  "compatibility": {
    "min_app_version": "0.1.0"
  }
}
```

## `compatibility.min_app_version` 应该怎么写

用途很直接：

- 告诉 FamilyClaw 这个插件最低支持哪个宿主版本
- 让插件市场在安装和升级前就能做兼容性判断

当前推荐写法：

```json
{
  "compatibility": {
    "min_app_version": "0.1.0"
  }
}
```

死规矩：

- 这是宿主最低版本，不是插件自己的版本号说明栏
- 字段缺失时，官方市场收录流程会直接失败
- 市场条目里的每个版本都必须带这个值，不再接受“兼容性未知但先收录”

## `entrypoints` 怎么写

入口必须是：

- Python 模块路径 + 函数名

例如：

```json
"entrypoints": {
  "integration": "app.plugins.builtin.health_basic.integration.sync"
}
```

错误写法包括：

- 只有模块名，没有函数名
- 空字符串
- 路径和真实代码对不上

## `types` 和 `entrypoints` 必须一致

例如：

- `types` 包含 `integration`，就应该有 `entrypoints.integration`
- `types` 包含 `action`，就应该有 `entrypoints.action`
- `types` 包含 `channel`，就应该有 `entrypoints.channel`
- `ai-provider` 必须声明 `entrypoints.ai_provider`

## `channel.send` 的通用附件字段

`channel` 插件现在要认清宿主统一的出站投递结构：

- `delivery.text`：可选，纯文本回复
- `delivery.attachments`：可选，平台无关的附件列表
- `delivery.metadata`：可选，插件自己的补充上下文

附件项当前统一字段是：

- `kind`
- `file_name`
- `content_type`
- `source_path` 或 `source_url`
- `size_bytes`
- `metadata`

死规矩：

- 附件是宿主通用能力，不是某个平台私有字段。
- 宿主不会替插件做厂商上传、转码或鉴权；这些脏活还是插件自己处理。
- 如果插件只支持文本，就忽略 `attachments`；如果声明支持媒体，就自己明确处理支持范围。

## `speaker_adapter` 现在怎么声明

如果插件要把音箱文本轮询、设备发现或者命令执行桥接进宿主，就不要再偷塞私有字段了，直接按正式字段声明：

```json
"entrypoints": {
  "integration": "migpt_xiaoai_speaker.integration.sync",
  "action": "migpt_xiaoai_speaker.action.execute",
  "speaker_adapter": "migpt_xiaoai_speaker.speaker_adapter.create_adapter"
},
"capabilities": {
  "speaker_adapter": {
    "adapter_code": "migpt_xiaoai_speaker",
    "supported_modes": ["text_turn"],
    "supported_domains": ["speaker"],
    "requires_runtime_worker": true,
    "supports_discovery": true,
    "supports_commands": true
  }
}
```

这几个字段的意思别搞混：

- `entrypoints.speaker_adapter`
  宿主用来加载 speaker 适配器入口的 Python 路径。
- `capabilities.speaker_adapter.adapter_code`
  适配器自己的稳定标识，不是 `plugin_id` 的替代品。
- `capabilities.speaker_adapter.supported_modes`
  当前适配器支持的桥接模式。首版文本轮询就写 `text_turn`，不要冒充 `audio_session`。
- `capabilities.speaker_adapter.supported_domains`
  这个列表必须包含 `speaker`，因为宿主需要明确知道它服务的是音箱域。
- `capabilities.speaker_adapter.requires_runtime_worker`
  用来说明这个适配器是否需要常驻 runtime/worker。
- `capabilities.speaker_adapter.supports_discovery`
  用来说明插件是否负责候选设备发现。
- `capabilities.speaker_adapter.supports_commands`
  只表示“这个适配器还能接统一动作执行”，不是说它可以绕过宿主直接控制设备。

宿主当前的硬校验也很直接：

- 声明 `capabilities.speaker_adapter` 时，必须同时声明 `integration` 类型。
- 必须同时有 `entrypoints.integration` 和 `entrypoints.speaker_adapter`。
- `supported_modes` 不能为空。
- `supported_domains` 不能为空，而且必须包含 `speaker`。
- 如果 `supports_commands=true`，还必须同时声明 `action` 类型和 `entrypoints.action`。
- 如果只写了 `entrypoints.speaker_adapter`，却没写 `capabilities.speaker_adapter`，加载阶段就会直接失败。

## `speaker text_turn` 回复现在怎么回音频 URL

宿主给 `speaker_adapter` 的回复现在不只支持纯文本，还支持正式 `audio_url` 结果。

当前约定很简单：

- 插件提交 `text_turn` 后，宿主仍然返回 `SpeakerTextTurnResult`
- 如果宿主只给文本，就走 `result_type="text" + reply_text`
- 如果宿主要让插件直接播远程音频，就走 `result_type="audio_url" + audio_url`

宿主当前从 `assistant_message.facts` 里认下面这种结构：

```json
{
  "type": "speaker_audio_reply",
  "audio_url": "https://media.example.com/reply.mp3",
  "content_type": "audio/mpeg",
  "fallback_text": "这段文字用于不支持 audio_url 的设备回退"
}
```

死规矩：

- `audio_url` 目前必须是绝对 `http/https` 地址，不能给相对路径。
- `fallback_text` 不是必填，但建议带上，设备不支持直链播放时还能退回文本播报。
- 首版 `text_turn` 仍然不是原始音频实时会话；这只是“宿主回复阶段可以回音频 URL”。

## `config_specs` 现在能声明什么

配置字段仍然分成两层：

- `config_schema`：数据语义
- `ui_schema`：展示提示

别小看 `ui_schema.sections`。

现在它不只是“把字段分组显示一下”。

对于设备与集成里的多 section 配置表单，前端已经会把这些 section 当成分阶段步骤来展示。也就是说：

- section 顺序，就是用户看到的步骤顺序
- section 标题和描述，会直接变成步骤标题和步骤说明
- 如果 `section.mode = "advanced"`，前端会把它当成“默认折叠的高级步骤”，普通用户可以先走基础步骤，最后按需再展开
- 如果你把所有字段都塞进一个 section，前端就只能继续给你画成长表单
- 如果你认真拆成“账号”“基础行为”“区域参数”“高级排障”这种结构，前端就能自然做成向导

如果你的插件需要在配置过程中做真实动作，比如：

- 真实登录第三方账号
- 拉取设备候选列表
- 提示二次认证或风控验证

不要把这些副作用硬塞进 `config/resolve`。

现在推荐的做法是：

- 在 manifest 里声明 `entrypoints.config_preview`
- 前端显式调用 `POST /plugins/{plugin_id}/config/preview`
- 宿主继续返回 `PluginConfigFormRead`
- 额外状态统一放在 `view.runtime_state`

如果预检流程里还有第三方认证回调，不要再自建插件私有 callback 接口。宿主现在会通过 `view.runtime_state.auth_session` 提供统一认证会话对象。

插件侧要按这个约定接：

- 读取 `auth_session.callback_url`，把它作为第三方回调目标
- 把继续登录需要的厂商私有恢复数据写进 `auth_session.payload`
- 等宿主把会话推进到 `callback_received`
- 下一次 `config_preview` 再读取 `auth_session.callback_payload`，继续跑自己的认证或设备加载
- 完成后把会话标成 `completed` 或 `failed`

这样表单字段还是标准字段，登录态、验证链接、设备候选列表这些临时运行态则放进 `runtime_state`，不会污染正式配置结构。

当前正式作用域：

- `plugin`
- `integration_instance`
- `device`
- `channel_account`

死规矩补一句：

- `channel_account` 表单现在只认插件自己的 `config_specs` 和 `ui_schema`。
- 不要再指望宿主前端按平台名硬编码字段、占位符和帮助文案。
- 如果通道插件需要配置，却没把 `channel_account` 的 `config_specs` 写完整，那就是插件自己的 bug，不是宿主替你猜。

## 设备与集成表单现在支持哪些字段类型

以前后端 schema 虽然早就允许更多字段类型，但前端集成表单只把最基础那几种画出来，这很容易把插件配置面做成半残。

现在设备与集成里的通用插件表单，至少已经正式支持下面这些类型：

- `string`
- `text`
- `integer`
- `number`
- `boolean`
- `enum`
- `multi_enum`
- `secret`
- `json`

对应的常用 widget 也已经能跑通：

- `input`
- `password`
- `textarea`
- `switch`
- `select`
- `multi_select`
- `json_editor`

人话解释：

- `multi_enum + multi_select` 现在不再只是 schema 里能写，前端也能真的多选。
- `json + json_editor` 现在不再只是给后端留口子，前端也能编辑 JSON 对象和数组。
- 如果插件要暴露高级筛选规则、profile override、调试参数，不需要再偷偷塞字符串协议。

## `ui_schema.actions` 和 `runtime_sections` 现在怎么配

如果插件的配置流程里有“先点一下，再看结果，再继续填”的阶段式步骤，不要再让核心前端猜。直接在 manifest 里声明：

- `ui_schema.actions`：这个 section 里有哪些可点击的预检动作
- `ui_schema.runtime_sections`：动作执行后，要在当前 section 里展示哪些运行时内容

最常见的接法就是：

1. 动作通过 `depends_on_fields` 约束“哪些字段先填完才能点”
2. 前端点击动作后调用 `config_preview`，并把 `action_key` 一起带过去
3. 插件把登录结果、验证链接、候选设备之类内容放进 `runtime_state`
4. `runtime_sections.items` 再用 `status_badge`、`text`、`link`、`candidate_select` 把这些内容渲染出来
5. 如果用户在 `candidate_select` 里选了一个候选项，宿主只负责把结果写回 `target_field`，不会理解你背后的平台语义

人话边界：

- 宿主只认动作 key、展示文案、依赖字段和 runtime item 结构
- 宿主不理解“小米登录”“微信扫码”“二次认证”这种平台细节
- 真正要保存的东西还是正式配置字段，比如 `device_selector`，不要只塞在临时 `runtime_state` 里

## `integration` 能力补了什么展示字段

以前实例名称输入框的建议文案只能宿主自己猜，这很蠢。

现在 `capabilities.integration` 额外支持：

- `instance_display_name_placeholder`
- `instance_display_name_placeholder_key`

用途只有一个：

- 告诉宿主“创建这个集成实例时，实例名称输入框该显示什么建议占位”

边界也很明确：

- 这是占位提示，不是默认值
- 宿主不应该再把插件名直接写进输入框
- 真正保存什么实例名称，还是用户最后输入什么就保存什么

## 枚举字段不再只有静态选项

以前 `enum` / `multi_enum` 只能写死 `enum_options`。

现在正式支持两种写法，二选一：

### 静态选项

```json
{
  "key": "mode",
  "type": "enum",
  "enum_options": [
    { "label": "严格", "value": "strict" },
    { "label": "宽松", "value": "loose" }
  ]
}
```

### 动态选项源

```json
{
  "key": "provider_code",
  "type": "enum",
  "option_source": {
    "source": "region_provider_list",
    "country_code": "CN"
  }
}
```

死规矩：

- `enum_options` 和 `option_source` 不能同时写
- 只有 `enum` / `multi_enum` 能写 `option_source`

## 动态选项源规范

### `region_provider_list`

用途：列出当前家庭可用的地区 provider。

字段：

- `source`: 固定为 `region_provider_list`
- `country_code`: 目标国家，例如 `CN`

### `region_catalog_children`

用途：按 provider 和父级地区拉子级目录。

字段：

- `source`: 固定为 `region_catalog_children`
- `country_code`: 目标国家
- `provider_code` 或 `provider_field`: 二选一
- `parent_field`: 当级联到 `city` / `district` 时必填
- `admin_level`: `province` / `city` / `district`

示例：

```json
{
  "key": "city_code",
  "type": "enum",
  "depends_on": ["provider_code", "province_code"],
  "clear_on_dependency_change": true,
  "option_source": {
    "source": "region_catalog_children",
    "country_code": "CN",
    "provider_field": "provider_code",
    "parent_field": "province_code",
    "admin_level": "city"
  }
}
```

## 字段联动怎么声明

如果一个字段依赖别的字段，正式用下面两个字段表达：

- `depends_on`
- `clear_on_dependency_change`

示例：

```json
{
  "key": "district_code",
  "type": "enum",
  "depends_on": ["provider_code", "province_code", "city_code"],
  "clear_on_dependency_change": true,
  "option_source": {
    "source": "region_catalog_children",
    "country_code": "CN",
    "provider_field": "provider_code",
    "parent_field": "city_code",
    "admin_level": "district"
  }
}
```

含义很直接：

- 上游字段一变，宿主要重新解析这个字段的可选项
- 如果当前值已经不在新候选里，宿主应该把它清掉

## 文案字段怎么写

插件配置表单允许“原文 + 词典 key”并存。

当前正式支持的 key 字段包括：

- `config_specs[].title_key`
- `config_specs[].description_key`
- `config_specs[].config_schema.fields[].label_key`
- `config_specs[].config_schema.fields[].description_key`
- `config_specs[].config_schema.fields[].enum_options[].label_key`
- `config_specs[].ui_schema.sections[].title_key`
- `config_specs[].ui_schema.sections[].description_key`
- `config_specs[].ui_schema.widgets[].placeholder_key`
- `config_specs[].ui_schema.widgets[].help_text_key`
- `config_specs[].ui_schema.submit_text_key`

推荐做法：

- 给用户看的原文一定要有，别只塞 key
- 有多语言时再补 key
- 宿主先走 key，没命中再退回原文

## `ai-provider` 现在的必填契约

`ai-provider` 现在不是只给一个 `field_schema` 就算完了。

至少必须完整声明下面三类前端契约：

- `branding`
- `config_ui`
- `model_discovery`

如果缺任何一类，插件加载阶段就应该失败。

### `branding`

至少包含：

- `logo_resource`
- `description_resource`

可选：

- `logo_resource_dark`

资源路径必须是插件目录内的相对路径，不能越界。

### `config_ui`

至少应该覆盖：

- `field_order`
- `sections`
- `actions`

如果有字段隐藏规则，还要声明：

- `field_ui`
- `hidden_when`

### `model_discovery`

至少应该覆盖：

- `enabled`
- `action_key`
- `depends_on_fields`
- `target_field`
- `debounce_ms`
- `empty_state_text`

如果启用模型发现，`action_key` 必须绑定到 `config_ui.actions` 中一个 `kind=model_discovery` 的动作。

## `ai-provider` 运行时字段重点

重点看：

- `capabilities.ai_provider.adapter_code`
- `capabilities.ai_provider.field_schema`
- `capabilities.ai_provider.branding`
- `capabilities.ai_provider.config_ui`
- `capabilities.ai_provider.model_discovery`
- `capabilities.ai_provider.supported_model_types`
- `capabilities.ai_provider.runtime_capability`

## `ai-provider` 入口字段重点

`entrypoints.ai_provider` 现在应该指向插件自己的 driver 入口。

推荐写法：

```json
"entrypoints": {
  "ai_provider": "app.plugins.builtin.ai_provider_chatgpt.driver.build_driver"
}
```

不推荐写法：

- 指回核心 `app.modules.ai_gateway.provider_driver.*`
- 使用裸模块名 `driver.build_driver` 让多个插件共享不稳定导入名

## 天气插件该怎么写

天气插件现在的正确写法不是把省市区整包塞进 manifest。

正确路线是：

1. `provider_code` 走 `region_provider_list`
2. `province_code` 走 `region_catalog_children(province)`
3. `city_code` 依赖 `provider_code + province_code`
4. `district_code` 依赖 `provider_code + city_code`

这样后续装了新的地区 provider，宿主就能直接把它列出来，天气插件不用再改代码。

## 最容易犯的错

- `manifest.version` 更新了，但 `compatibility.min_app_version` 没跟着改
- `enum_options` 和 `option_source` 同时写
- 字段联动只在前端隐藏，不声明 `depends_on` / `clear_on_dependency_change`
- `ai-provider` 只写 `field_schema`，把品牌、表单分组、模型发现继续留给核心前端硬编码
- `entrypoints.ai_provider` 指回核心 builder
- `model_discovery` 开了，但 `config_ui.actions` 没绑动作
- 资源路径越界，或者 `description.json` 结构不合法
- 供应商明明有特殊协议，却还强行要求核心帮它补分支
- 字段设计本身如果就是糊的，后面调试起来会比写代码更恶心。
