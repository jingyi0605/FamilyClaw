# 插件提交

这页讲的是插件写完以后，怎么把它交出来，而不是把一堆“以后再补”的借口塞进 PR。

## 提交前先看哪几页

提交插件前，至少把这几页过一遍：

- [插件规范](./插件规范.md)
- [字段规范](./字段规范.md)
- [对接方式](./对接方式.md)
- [实例插件](./实例插件.md)

这几页已经把“类型怎么定、字段怎么写、怎么联调、拿什么做参考、交付时审什么”串起来了，不需要再跳出当前文档站找说明。

## 提交前最低要求

至少满足下面这些：

1. `manifest.json` 能通过当前规范校验
2. 入口路径和真实代码一致
3. 如果准备进入插件市场，`manifest.json` 已声明 `compatibility.min_app_version`
4. README 讲清楚用途、配置项、权限、风险和验证方式
5. 插件类型、权限、风险等级不是瞎填的
6. 如果涉及配置，`config_specs` 和实际读取字段一致
7. 如果涉及数据库或私有迁移，迁移方式清楚可重复
8. 如果影响官方文档或用户可见行为，对应文档已同步

## 提交材料应该包含什么

- 插件源码
- `manifest.json`
- README
- 最小测试或验证说明
- 版本信息
- 最低宿主版本信息
- 如果接入外部平台，还要写清楚前置依赖和所需账号类型

## 市场多版本规则

如果你准备进官方市场，版本这块现在别再靠猜：

1. 单个插件只会有一个 `plugins/<plugin_id>/entry.json`
2. 所有可安装版本都保存在这个文件的 `versions[]` 里
3. `latest_version` 必须指向 `versions[]` 里的最高版本
4. 正式多版本发布必须来自 tag，`git_ref` 统一写成 `refs/tags/<tag>`
5. 没有 tag / release 时，市场只允许生成一个引用 branch 的单版本开发态条目
6. 每个版本都要有自己独立的 `min_app_version`

想保留旧版本回滚，就保留旧 tag；Issue 文本不会替你造版本历史。

## 发版本前检查

每次准备打新 tag 前，至少确认：

1. `manifest.version` 已更新
2. `compatibility.min_app_version` 已按这个版本真实要求更新
3. `requirements.txt`、README 里和安装、依赖、配置有关的内容已同步
4. tag 名和 `manifest.version` 一致

现在机器人会按每个 tag 分别读取当时的 `manifest.json`，不会拿最新分支的兼容性字段去替历史版本补洞。

## 已收录插件后续怎么进新版本

插件第一次进入官方市场，走的是收录 Issue。

但插件已经收录以后，后面的版本更新不需要你每次都重新提 Issue。现在官方市场仓库会每 2 小时做一次轻量扫描：

1. 读取所有已收录条目的 `source_repo`
2. 只检查 release / tag 列表有没有新版本
3. 只有发现新 tag 时，才读取那个 tag 对应的 `manifest.json`
4. 自动更新 `plugins/<plugin_id>/entry.json`，并创建或更新机器人 PR

这条同步链路只做保守更新：

- 追加新版本
- 把同版本的 branch 记录收口成 tag 记录
- 一旦开始进入正式 tag 模式，旧的 branch 兜底记录会被移除
- 不会因为上游删了 tag，就自动删市场里的旧版本

所以开发者真正要做的事很简单：

1. 把仓库里的版本事实改对
2. 打正确的 tag / release
3. 等市场仓库自动扫描并生成 PR
4. 审核通过后，新版本才正式进入市场

## 审核时别人最先看什么

### 1. 类型是不是选对了

如果一个本该是 `integration` 的东西被你硬写成 `action`，后面全错。

### 2. 权限和风险等级是不是诚实

高风险插件报成 `low`，这不是“方便测试”，这是在埋雷。

### 3. README 能不能让接手的人活下来

README 如果只写“这是一个很强大的插件”，那等于没写。

至少要包括：

- 这个插件干什么
- 需要什么配置
- 怎么安装
- 怎么验证
- 已知限制是什么

### 4. 是否破坏宿主边界

如果插件实现里开始直接依赖宿主私有内部细节、直接写宿主核心表，审核就该停。

## 内置插件和第三方插件的区别

### 内置插件

- 代码通常放在 `apps/api-server/app/plugins/builtin/`
- 跟随主仓库版本一起维护
- 文档、配置、前端入口往往需要同步更新

### 第三方插件

- 更强调 README、自描述和隔离性
- 不能假设别人知道你的私有上下文
- 不能要求宿主替你补一堆特判

## 提交前自检清单

- [ ] `manifest.id` 稳定且唯一
- [ ] `entrypoints` 和真实函数路径一致
- [ ] `compatibility.min_app_version` 已声明
- [ ] 配置字段与实际读取逻辑一致
- [ ] 禁用插件后，执行链路会被正确拦截
- [ ] 计划任务引用时，`triggers` 已正确声明
- [ ] README 里写明配置项、依赖、验证步骤
- [ ] 影响文档的改动已同步更新

## 一句实话

插件提交最怕的不是代码少，而是信息不完整。

别人如果看不懂你的插件到底要什么、会干什么、怎么验证，那它就不配进入正式体系。
