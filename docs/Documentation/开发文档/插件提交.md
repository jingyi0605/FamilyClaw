# 插件提交

这页讲的是插件写完以后，怎么把它交出来，而不是把一堆“以后再补”的借口塞进 PR。

## 提交前先看哪几页

提交插件前，至少把这几页过一遍：

- [插件规范](./插件规范.md)
- [字段规范](./字段规范.md)
- [对接方式](./对接方式.md)
- [测试验证](./测试验证.md)

## 先确认你是不是已经到“能提 PR”的阶段

提 PR 之前，至少先满足这几条：

1. 插件目录和 `manifest.json` 已经按规范写完
2. `manifest.json` 里的入口路径能对应到真实代码
3. 你声明的插件类型、权限、风险等级都已经写清楚
4. 如果要进入插件市场，`manifest.json` 已声明 `compatibility.min_app_version`
5. 你有一份能给人看的插件说明文档，至少告诉别人这个插件做什么、怎么验证
6. 你已经给插件准备好自己的依赖清单，比如 `requirements.txt`
7. 你没有依赖当前明确不支持的能力，比如自动安装、远程执行、沙箱安装

## 提交材料应该包含什么

- 插件源码仓库地址
- `manifest.json`
- README
- 最小测试或验证说明
- 版本信息
- 最低宿主版本信息
- 如果接入外部平台，还要写清楚前置依赖和所需账号类型

## 推荐提交流程

按这个顺序提，最省事：

1. 先把插件源码仓库整理好
2. 检查仓库里的 `manifest.json`、README、最小测试说明是否齐全
3. 检查 `requirements.txt` 和最小 runner / 宿主链路自检说明是否齐全
4. 如果你想让市场保留多个正式版本，先把对应 tag 打出来，推荐 `v1.0.0` 这种格式
5. fork 官方注册表仓库，或者在项目指定分支上提交修改
6. 按模板新增注册表条目文件
7. 提交 PR，并把插件仓库地址、文档地址、风险说明写清楚
8. 等待审核；如果维护者要求补材料，就按意见补，不要打太极

## 市场多版本规则

如果你准备进官方市场，版本这块别再靠猜：

1. 单个插件只会有一个 `plugins/<plugin_id>/entry.json`
2. 所有可安装版本都保存在这个文件的 `versions[]` 里
3. `latest_version` 必须指向 `versions[]` 里的最高版本
4. 正式多版本发布必须来自 tag，`git_ref` 统一写成 `refs/tags/<tag>`
5. 没有 tag / release 时，市场只允许生成一个引用 branch 的单版本开发态条目
6. 每个版本都要有自己独立的 `min_app_version`

想保留旧版本回滚，就保留旧 tag；Issue 文本不会替你造版本历史。

## 发版本前检查

每次准备打新 tag 前，至少确认：

1. `manifest.version` 已更新
2. `compatibility.min_app_version` 已按这个版本真实要求更新
3. `requirements.txt`、README 里和安装、依赖、配置有关的内容已同步
4. tag 名和 `manifest.version` 一致

现在机器人会按每个 tag 分别读取当时的 `manifest.json`，不会拿最新分支的兼容性字段去替历史版本补洞。

## 已收录插件后续怎么进新版本

插件第一次进入官方市场，走的是收录 Issue。

但插件已经收录以后，后面的版本更新不需要你每次都重新提 Issue。现在官方市场仓库会每 2 小时做一次轻量扫描：

1. 读取所有已收录条目的 `source_repo`
2. 只检查 release / tag 列表有没有新版本
3. 只有发现新 tag 时，才读取那个 tag 对应的 `manifest.json`
4. 自动更新 `plugins/<plugin_id>/entry.json`，并创建或更新机器人 PR

这条同步链路只做保守更新：

- 追加新版本
- 把同版本的 branch 记录收口成 tag 记录
- 一旦开始进入正式 tag 模式，旧的 branch 兜底记录会被移除
- 不会因为上游删了 tag，就自动删市场里的旧版本

## PR 描述建议模板

```md
## 插件信息

- plugin_id: `your-plugin-id`
- 插件名称: `你的插件名`
- 插件仓库: `https://github.com/your-org/your-plugin-repo`
- 插件文档: `https://github.com/your-org/your-plugin-repo/blob/main/README.md`

## 能力说明

- 插件类型: `integration` / `action` / `channel` / `ai-provider`
- 权限声明: `...`
- 风险等级: `low` / `medium` / `high`

## 自检结果

- [ ] `manifest.json` 已自检
- [ ] 入口可 import
- [ ] README 已补齐
- [ ] 已说明最小验证步骤

## 补充说明

- 这个插件明确不做什么：`...`
- 审核时需要重点关注什么：`...`
```

## 审核时别人最先看什么

### 1. 类型是不是选对了

如果一个本该是 `integration` 的东西被你硬写成别的类型，后面全错。

### 2. 权限和风险等级是不是诚实

高风险插件报成 `low`，这不是“方便测试”，这是在埋雷。

### 3. README 能不能让接手的人活下来

至少要包括：

- 这个插件干什么
- 需要什么配置
- 怎么安装
- 怎么验证
- 已知限制是什么

### 4. 是否破坏宿主边界

如果插件实现里开始直接依赖宿主私有内部细节、直接写宿主核心表，审核就该停。

## 什么情况会被直接打回

- `plugin_id` 和 `manifest.json` 对不上
- 仓库打不开，或者看不到真实插件代码
- README 写得像空气，看不出插件到底干什么
- 风险等级明显报低
- `manifest.json` 没声明 `compatibility.min_app_version`
- 条目里字段乱填，和模板不一致
- 明显越过当前边界，比如要求平台自动下载安装或远程执行

## 一句实话

插件提交最怕的不是代码少，而是信息不完整。

别人如果看不懂你的插件到底要什么、会干什么、怎么验证，那它就不配进入正式体系。
