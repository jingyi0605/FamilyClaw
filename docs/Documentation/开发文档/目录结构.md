# 目录结构

这页不讲套话，只讲现在仓库里真正影响插件开发和维护的目录。

## 项目级目录

```text
FamilyClaw/
├── apps/
│   ├── api-server/
│   ├── user-app/
│   ├── open-xiaoai-gateway/
│   ├── start-api-server.sh
│   └── start-open-xiaoai-gateway.sh
├── packages/
│   ├── user-core/
│   ├── user-platform/
│   └── user-ui/
├── docs/
│   └── Documentation/
├── docker/
├── specs/
├── package.json
└── Dockerfile
```

## 各目录到底干什么

### `apps/api-server`

后端主服务。

重点看：

- `app/main.py`
- `app/api/v1/`
- `app/modules/`
- `app/core/`
- `migrations/`

### `apps/user-app`

前端主应用，基于 Taro。

重点看：

- `src/app.config.ts`
- `src/pages/`
- `src/runtime/`
- `config/platform/h5.ts`

### `apps/open-xiaoai-gateway`

语音网关。

### `packages/user-core`

前端共享类型和 API client 事实源。

### `packages/user-platform`

平台能力适配层，比如存储、实时能力等。

### `packages/user-ui`

UI 组件与主题归一化层。

### `docs/Documentation`

官方文档站源码。改用户可见行为、安装方式、接口语义时，这里必须同步更新。

## 插件目录长什么样

这页只解决一个问题：插件目录到底该怎么摆。

目录结构先贴着现有仓库能跑的方式来，不要另造抽象。

当前内置插件都在：

- `apps/api-server/app/plugins/builtin/`

第三方插件推荐放在自己的插件仓库里，由 runner 或安装目录接入。

## 最小目录模板

```text
your_plugin/
  manifest.json
  requirements.txt
  README.md
  __init__.py
  integration.py
  action.py
  channel.py
  agent_skill.py
  ai_provider.py
  memory_engine.py
  memory_provider.py
  context_engine.py
  locales/
  tests/
```

上面不是说每个文件都必须存在，而是说：

- `manifest.json`：必须有
- `requirements.txt`：第三方插件强烈建议有
- `README.md`：强烈建议有
- `__init__.py`：强烈建议有，避免 Python 包路径不清楚
- `integration.py` / `action.py` / `channel.py` / `agent_skill.py` / `ai_provider.py` / `memory_*.py`：按你声明的正式类型决定
- `locales/`：给 `locale-pack` 或普通插件自带词典放资源文件
- `tests/`：强烈建议有，哪怕先放最小自检说明

## 一插件一目录

这是硬规则：

- 一个插件目录只放一个 `manifest.json`
- 一个 `manifest.json` 只描述一个插件
- 不要在同一个目录里塞多个插件能力集合，再靠脚本分流

## 目录名怎么取

第一版建议直接和 `manifest.json` 的 `id` 保持接近，最好一致或可一眼对应。

合格示例：

- 目录：`health_basic`
- `id`：`health-basic-reader`
- 目录：`homeassistant_device_action`
- `id`：`homeassistant-device-action`

不用强制完全相同，因为 Python 模块目录通常会用下划线；但至少要让人一眼对得上。

## 推荐文件职责

### `manifest.json`

- 放插件声明
- 不放运行时代码
- 不塞一堆未来才会用的字段

### `requirements.txt`

- 放插件自己的 Python 依赖
- 不要假设平台会自动安装
- 版本范围写清楚，别写飘
- 默认由插件自己的环境消费，不是让主 API 环境替你兜底

### `integration.py`

- 放 `integration` 正式入口
- 只产出标准设备、标准实体、标准卡片和标准动作结果

### `action.py`

- 放 `action` 正式入口
- 只负责动作执行本身，不负责绕开系统权限

### `agent_skill.py`

- 放 `agent-skill` 正式入口
- 只暴露受控能力，不直接改 Agent 主流程

### `ai_provider.py` 或 `driver.py`

- 放 `ai-provider` driver 入口
- 提供 `invoke`、`ainvoke` 或 `stream` 之类正式 driver 能力

### `memory_engine.py` / `memory_provider.py` / `context_engine.py`

- 放独占槽位插件入口
- 严格按宿主定义的槽位契约实现

### `locales/`

- 给声明了 `locales` 的插件放语言资源文件
- 推荐一个语言一个文件，比如 `locales/zh-TW.json`
- 文件内容就是 key-value 文案，不要塞执行代码
- `manifest.locales[].resource` 必须指向这里的真实文件

这里别再理解成“只有语言包插件才准有 `locales/`”。

当前规则是：

- `locale-pack` 用它承载整包语言资源
- 普通插件也可以用它承载自己的插件词典
- 同一个 `locale_id` 会在宿主侧按消息 key 合并，不是整包替换

### `README.md`

建议至少写清楚：

1. 插件做什么
2. 用到哪些权限
3. 需要什么外部配置
4. 怎么做最小验证
5. 明确不做什么

### `tests/`

第一版不强制你交一整套复杂自动化，但至少要能说明：

- `manifest` 是否自检过
- 入口是否能 import
- 核心样例 payload 是否跑过

## 典型目录示例

```text
apps/api-server/app/plugins/builtin/health_basic/
├── __init__.py
├── integration.py
└── manifest.json
```

另一个带配置的通道插件示例：

```text
apps/api-server/app/plugins/builtin/channel_feishu/
├── __init__.py
├── channel.py
└── manifest.json
```

主题插件示例：

```text
apps/api-server/app/plugins/builtin/theme_chun_he_jing_ming_pack/
├── manifest.json
└── themes/
```

## 多能力插件怎么摆

如果一个插件同时有 `integration` 和 `action`，不要拆成两个插件目录。

最小可用方式就是：

```text
health_basic/
  manifest.json
  __init__.py
  integration.py
  action.py
```

一个插件目录对应一个插件 id，但可以同时暴露多个正式入口。

## 语言包插件怎么摆

`locale-pack` 跟可执行插件不是一回事。它不需要 `integration.py`、`action.py` 这些入口文件。

最小可用方式就是：

```text
locale_zh_tw_pack/
  manifest.json
  README.md
  locales/
    zh-TW.json
```

如果是普通插件自带词典，目录更简单：

```text
official_weather/
  manifest.json
  integration.py
  locales/
    zh-CN.json
    en-US.json
```

## 现在不建议加的目录内容

这些内容第一版先别加，除非后面 spec 明确扩范围：

- 自动安装脚本
- 远程下载器
- 沙箱配置目录
- 全量签名证书目录
- 前端市场展示资源包

## 官方插件与镜像打包边界

这条规则现在要写死：

- `builtin` 插件放仓库里的 `apps/api-server/app/plugins/builtin/`
- `official` 插件放 `apps/api-server/data/plugins/official/`
- `third_party` 插件放宿主数据目录或市场安装目录

对应的打包规则也必须一起成立：

- Docker 最终镜像只内置宿主和 `builtin` 插件
- `official` 与 `third_party` 插件不应打进最终镜像，应通过挂载、安装或同步落到宿主数据目录
- `.dockerignore` 应排除 `apps/api-server/data/plugins/official/`
- 宿主核心代码不能在 `import` 阶段静态依赖 `official` 或 `third_party` 插件模块，否则插件文件没挂进来时宿主会直接起不来

## 目录设计的坏味道

看到下面这些情况，通常说明结构已经开始烂了：

- 一个插件目录里塞了多个 `manifest.json`
- 插件目录名和 `manifest.id` 完全对不上
- 把宿主业务逻辑写进插件目录
- 把插件私有模型塞回宿主总模型入口
- 为高风险动作单独发明一套目录层级
- 把 `official` 插件写成宿主编译依赖

结构不清楚，后面所有文档和排障都会一起烂。
