# 目录结构

这页只保留当前仍然有效的插件目录规则。

## 插件相关目录总表

```text
apps/api-server/
  app/
    plugins/
      builtin/
  plugins-dev/
  data/
    plugins/
      third_party/
        local/
        marketplace/
```

## 每个目录到底是干什么的

### `app/plugins/builtin/`

- 内置插件源码目录。
- 跟宿主一起发布。
- 优先级最高。

### `plugins-dev/`

- 仓库内第三方插件开发源码目录。
- 也是开发覆盖源。
- 会参与家庭插件注册表合并。
- 不写入挂载记录，不属于安装产物。

### `data/plugins/third_party/local/`

- 本地安装的第三方插件运行时目录。
- 对应 `install_method=local`。
- 会参与启动同步和卸载清理。

### `data/plugins/third_party/marketplace/`

- 市场安装的第三方插件运行时目录。
- 对应 `install_method=marketplace`。
- `is_system=true` 只表示这是系统内置市场源，不表示插件类型变成了别的东西。

## 已废弃目录

- `apps/api-server/data/plugins/official/`

这是历史目录，不再作为开发目录，也不再作为正式运行时目录。
看到它时，按“历史说明”理解，不要再继续往这里放源码或安装产物。

## 加载和覆盖顺序

当前固定规则：

1. `builtin`
2. `plugins-dev`
3. 已安装 third_party

具体解释：

- `plugins-dev` 里的同名插件可以覆盖第三方插件。
- `plugins-dev` 里的同名插件不能覆盖 `builtin`。
- 只有安装态目录会参与启动同步、卸载和清理。

## 开发和安装为什么必须拆开

如果把开发源码和运行时产物混在一个目录里，会立刻出现这些问题：

- 市场安装把开发目录覆盖掉。
- 本地开发改动被卸载逻辑误删。
- 你根本分不清当前跑的是源码还是安装包。

所以现在规则必须分开：

- 开发改 `plugins-dev`
- 安装落 `third_party/local` 或 `third_party/marketplace`

## 最小开发目录模板

```text
apps/api-server/plugins-dev/your_plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    integration.py
  tests/
```

## 最小运行时目录模板

```text
apps/api-server/data/plugins/third_party/local/<household_id>/<plugin_id>/<release_dir>/
  manifest.json
  requirements.txt
  plugin/
```

```text
apps/api-server/data/plugins/third_party/marketplace/<household_id>/<plugin_id>/<version>/
  manifest.json
  requirements.txt
  plugin/
```

## 看目录时的判断原则

- 看到 `builtin`，理解成“宿主内置”。
- 看到 `plugins-dev`，理解成“开发覆盖源”。
- 看到 `third_party/local`，理解成“本地安装产物”。
- 看到 `third_party/marketplace`，理解成“市场安装产物”。
- 看到 `official`，理解成“历史目录，已废弃”。
