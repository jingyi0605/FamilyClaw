# 目录结构

这页只保留当前仍然有效的插件目录规则。

## 插件相关目录总表

```text
apps/api-server/
  app/
    plugins/
      builtin/
  plugins-dev/
  data/
    plugins/
      third_party/
        local/
        marketplace/
```

## 每个目录到底是干什么的

### `app/plugins/builtin/`

- 内置插件源码目录。
- 跟宿主一起发布。
- 优先级最高。

### `plugins-dev/`

- 仓库内第三方插件开发源码目录。
- 会参与插件发现。
- 在插件管理页里会以“开发版”身份出现。
- 不属于安装产物目录。

### `data/plugins/third_party/local/`

- 本地安装的第三方插件运行时目录。
- 对应 `install_method=local`。
- 参与启动同步和卸载清理。

### `data/plugins/third_party/marketplace/`

- 市场安装的第三方插件运行时目录。
- 对应 `install_method=marketplace`。
- 参与启动同步和卸载清理。

## 已废弃目录

- `apps/api-server/data/plugins/official/`

这是历史目录，不再作为开发目录，也不再作为正式运行目录。看到它时，按“历史说明”理解，不要再往里面放源码或安装产物。

## 同 `plugin_id` 冲突时怎么落到目录口径

### 和 `builtin` 冲突

- `builtin` 仍然生效。
- `plugins-dev` 不覆盖内置插件。

### 和已安装 third_party 冲突

- `plugins-dev` 下的开发版和安装目录下的已安装版会同时保留。
- 插件管理页会同时显示两个变体。
- 同一时刻只允许启用一个。
- 默认优先开发版。
- 想验证安装版，直接在页面里切换启用，不需要移动目录。

### 前端会怎么显示

- 开发版和已安装版会被明确区分。
- 当前实际跑的是开发版时，会显示“开发插件生效中”。

## 为什么开发目录和安装目录必须拆开

如果把开发源码和运行时产物混在一个目录里，立刻就会出这些问题：

- 市场安装把开发目录覆盖掉。
- 本地开发改动被卸载逻辑误删。
- 你根本分不清当前跑的是源码还是安装包。

所以现在必须分开：

- 开发改 `plugins-dev`
- 安装看 `third_party/local` 或 `third_party/marketplace`

## 两套标准不要混

这件事现在必须说清楚，不然最容易把开发目录位置和外部插件仓库布局混为一谈。

### 1. 仓库内开发目录标准

这条规则只管“源码放在哪”：

```text
apps/api-server/plugins-dev/<plugin>/
```

也就是说，只要你是在 FamilyClaw 主仓库里联调第三方插件，源码就统一放在 `plugins-dev/<plugin>/` 下。

### 2. 外部插件仓库 / 市场提交标准

这条规则管的是“独立插件仓库自己怎么排目录”。

当前允许，而且已经实际通过市场更新和安装环境测试的布局，就是 `official_weather` 这种：

```text
your_plugin_repo/
  README.md
  requirements.txt
  <plugin_package>/
    __init__.py
    manifest.json
    integration.py / driver.py / channel.py
    resources/
```

一句话：

- `plugins-dev/<plugin>/` 解决的是“放哪开发”
- `official_weather` 这种布局解决的是“独立插件仓库内部怎么组织”

如果你把一个独立维护的插件仓库直接挂到 `plugins-dev/` 里，本项目认可保留它自己的仓库内部结构，不需要为了进 `plugins-dev` 再强行改成另一套假模板。

## 仓库内最小开发目录模板

```text
apps/api-server/plugins-dev/your_plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    integration.py
  tests/
```

这套模板适合：

- 在主仓库里快速做最小插件联调
- 还没有拆成独立插件仓库的时候

## 外部插件仓库 / 市场提交推荐模板

```text
your_plugin_repo/
  README.md
  requirements.txt
  your_plugin/
    __init__.py
    manifest.json
    integration.py / driver.py / channel.py
    resources/
```

这套模板适合：

- 独立维护的第三方插件仓库
- 未来准备进入插件市场的插件
- 需要在仓库根目录保留 README、依赖说明和其他仓库级文件的场景

像 `official_weather` 这种已通过市场链路验证的插件，就是这套结构。

## 最小运行时目录模板

```text
apps/api-server/data/plugins/third_party/local/<household_id>/<plugin_id>/<release_dir>/
  manifest.json
  requirements.txt
  plugin/
```

```text
apps/api-server/data/plugins/third_party/marketplace/<household_id>/<plugin_id>/<version>/
  manifest.json
  requirements.txt
  plugin/
```

## 看目录时的判断原则

- 看到 `builtin`，理解成“宿主内置”。
- 看到 `plugins-dev`，理解成“开发源码目录”。
- 看到 `third_party/local`，理解成“本地安装产物”。
- 看到 `third_party/marketplace`，理解成“市场安装产物”。
- 看到 `official`，理解成“历史目录，已废弃”。
