# 实例插件

第一次做插件，别一上来就搞高风险控制或复杂协议。先拿仓库里现成的内置插件看清套路。

## 示例 1：`health-basic-reader`

位置：

- `apps/api-server/app/plugins/builtin/health_basic/`

适合学什么：

- 最小 `integration` 插件长什么样
- 怎么声明 `schedule` trigger
- 怎么给首页提供 dashboard card

## 示例 2：`builtin.provider.chatgpt`

位置：

- `apps/api-server/app/plugins/builtin/ai_provider_chatgpt/manifest.json`
- `apps/api-server/app/plugins/builtin/ai_provider_chatgpt/driver.py`

适合学什么：

- `ai-provider` 的 `branding / config_ui / model_discovery` 怎么声明
- runtime capability 怎么声明
- 一个标准 OpenAI 兼容 provider 怎么接插件层 SDK

这个例子现在更适合学“插件入口怎么接”，而不是学业务特例。

## 示例 3：`builtin.provider.glm`

位置：

- `apps/api-server/app/plugins/builtin/ai_provider_glm/driver.py`

适合学什么：

- 在复用插件 SDK 的前提下，怎么做 provider 自己的参数修正
- 怎么在插件里禁用 thinking、调整 `max_tokens`、修温度等供应商特例

这类例子最能说明现在的正式边界：

- 协议基础能力可以复用 SDK
- 供应商特例应该留在插件里

## 示例 4：`builtin.provider.ollama`

位置：

- `apps/api-server/app/plugins/builtin/ai_provider_ollama/manifest.json`
- `apps/api-server/app/plugins/builtin/ai_provider_ollama/driver.py`

适合学什么：

- 本地 LLM provider 怎么声明品牌资源
- 怎么声明模型自动发现按钮和字段依赖
- 怎么通过插件驱动模型发现，而不是让前端写死“刷新模型”

## 示例 5：`channel-feishu`

位置：

- `apps/api-server/app/plugins/builtin/channel_feishu/manifest.json`

适合学什么：

- `config_specs` 怎么配
- `ui_schema` 怎么描述表单
- channel 绑定能力怎么声明

## 做第一个插件的推荐顺序

1. 先选简单类型，优先 `integration` 或 `theme-pack`
2. 先把 manifest 写对
3. 再写最小入口函数
4. 再把配置接上
5. 最后跑宿主内验证

## 做 AI provider 时的额外提醒

如果你做的是 `ai-provider`，现在要先决定一件事：

- 这个 provider 能不能直接复用插件 SDK
- 还是需要在插件目录里完全自己实现协议

这两个都是正式支持的。

但无论你选哪一个，规则都一样：

- 不要再去改核心 `ai_gateway` 里的协议分支
- 不要要求前端再给你补 `adapter_code -> logo/description` 这种硬编码
- 不要把“刷新模型”做成核心特判

## 不推荐的第一插件选择

- 高风险动作插件
- 需要复杂鉴权和回调的平台渠道
- 需要大量宿主上下文写权限的插件

第一个插件要先学结构，不是先秀复杂度。
