# 插件开发

插件开发不是“给宿主塞一个补丁口子”，而是 FamilyClaw 当前正式的能力扩展方式。

如果你记不住整套规则，先记住这四句：

1. 宿主只保留平台规则和统一标准。
2. 第三方接入、字段映射、厂商特例优先放插件。
3. 插件返回的结果必须已经是标准结果，不要让宿主再补丁式修正。
4. 插件是否可执行，只认统一启停状态，不认页面自己猜。

## 先看哪里

推荐阅读顺序：

1. 先看 [插件规范](./插件规范.md)，先把宿主和插件的边界搞清楚。
2. 再看 [字段规范](./字段规范.md)，把 `manifest.json` 和配置 schema 写对。
3. 再看 [对接方式](./对接方式.md)，确认执行链路和联调接口。
4. 然后看 [实例插件](./实例插件.md)，拿现成内置插件对照着做。
5. 最后看 [插件提交](./插件提交.md)，按正式交付要求自检。

这几页加起来，就是当前官方文档站里可直接落地的插件开发手册，不需要再跳到 `docs/Documentation` 之外去拼信息。

## 当前正式插件类型

基于 `apps/api-server/app/modules/plugin/schemas.py`，当前正式类型包括：

- `integration`
- `action`
- `agent-skill`
- `channel`
- `locale-pack`
- `region-provider`
- `theme-pack`
- `ai-provider`

独占槽位包括：

- `memory_engine`
- `memory_provider`
- `context_engine`

## 你该把什么做成插件

### 适合做插件的

- 第三方 API 接入
- 第三方设备协议适配
- 某个供应商的大模型接入
- 通讯平台通道
- 主题资源、语言资源
- 地区目录与地区解析

### 不该做成插件的

- 家庭、成员、房间、权限这些宿主主权数据
- 宿主统一审计
- 宿主统一调度
- 宿主统一实体、卡片、动作标准

## 最小开发流程

### 1. 先定类型，不要先写代码

先决定它到底是：

- `integration`
- `action`
- `channel`
- `ai-provider`
- 还是别的正式类型

类型定错了，后面 manifest、执行链路、配置范围全会跟着错。

### 2. 先写 `manifest.json`

宿主先看 manifest，再决定怎么装载你。

至少要写清楚：

- `id`
- `name`
- `version`
- `api_version`
- `types`
- `permissions`
- `risk_level`
- `entrypoints`
- `capabilities`

### 3. 再写入口代码

例如：

- `integration` 入口
- `channel` 入口
- `ai_provider` 入口

入口路径必须和 `manifest.json` 里声明的一致。

### 4. 再接配置

如果插件需要用户填写参数，就补：

- `config_specs`

不要把配置协议自己藏到代码里，让前端猜字段。

### 5. 最后做宿主内验证

至少要确认：

- 插件能被扫描到
- 插件状态能显示
- 配置能保存
- 执行入口能跑
- 禁用时能被统一拦住

## 当前宿主负责什么

- 插件注册与挂载
- 启用/禁用治理
- 配置值保存
- 权限、审计、调度
- 标准实体、标准卡片、标准动作语义
- 插件市场与版本治理

## 当前插件负责什么

- 对接第三方平台
- 把第三方字段映射成宿主标准数据
- 处理插件自己的缓存、去重、限流和厂商特例
- 产出最终标准 DTO

## 开发时最容易踩的坑

### 1. 把宿主主权数据写进插件

这是边界错了，不是实现快了。

### 2. 让宿主按插件类型去读时修补

如果宿主还要靠 `if plugin_type == ...` 才能读懂结果，说明插件输出根本没标准化。

### 3. 自己发明调度系统

计划任务、后台任务、重试策略都归宿主管。

插件要做的是声明能力，不是自己注册 cron。

### 4. 忽视禁用语义

插件禁用后：

- 不能新增使用
- 不能继续执行
- 但可以查看和改配置

这条边界必须和全系统一致。

## 什么时候该停下来先问一句

如果你发现自己正准备：

- 再给宿主加一个领域专用 service
- 再给某个插件类型开一套例外逻辑
- 再让前端或后端各自补一层兼容判断

那就先停。大概率不是逻辑不够，而是边界已经开始烂了。
