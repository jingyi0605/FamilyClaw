# 插件开发

这页只讲当前还有效的插件开发口径，不再保留旧 `official` 时代的说法。

## 先记住 7 句话

1. 插件来源只区分 `builtin` 和 `third_party`。
2. 第三方插件的安装方式只区分 `local` 和 `marketplace`。
3. 仓库内第三方插件开发源码统一放在 `apps/api-server/plugins-dev/`。
4. `plugins-dev` 会被加载，但它是开发源码目录，不是安装目录。
5. 当 `plugins-dev` 和已安装 third_party 的 `plugin_id` 相同时，插件管理页会同时显示“开发版”和“已安装版”两个变体。
6. 同一个 `plugin_id` 在同一时刻只允许启用一个变体，默认优先启用开发版；想验证安装版，直接切换启用，不需要手动挪目录。
7. `builtin` 永远优先，`plugins-dev` 不能覆盖内置插件；当前实际生效的是开发版时，前端会明确显示“开发插件生效中”。

## 现在到底有哪些目录

- `apps/api-server/app/plugins/builtin/`
  内置插件源码目录，跟宿主一起发布。
- `apps/api-server/plugins-dev/`
  第三方插件开发源码目录，也是开发联调时会被加载的目录；该目录不属于安装产物。
- `apps/api-server/data/plugins/third_party/local/`
  本地安装的第三方插件运行时目录。
- `apps/api-server/data/plugins/third_party/marketplace/`
  市场安装的第三方插件运行时目录。

不要再把开发中的插件放到 `apps/api-server/data/plugins/official/`。那个路径已经是历史目录。

## `plugins-dev` 和安装目录的区别

### `plugins-dev` 负责什么

- 放开发中的第三方插件源码。
- 参与家庭插件注册表构建。
- 在插件管理页里以“开发版”身份出现。
- 当存在同 `plugin_id` 的已安装 third_party 时，默认成为当前启用变体。

### `plugins-dev` 不负责什么

- 不写入安装目录。
- 不写入 `plugin_mounts`。
- 不参与卸载时的磁盘清理。
- 不替代 `local` / `marketplace` 的安装语义。

## 冲突规则

### 和 `builtin` 同名

- 始终以 `builtin` 为准。
- `plugins-dev` 中的同名插件不会覆盖内置插件。
- 这不是一个可切换场景。想做开发，请换 `plugin_id`，或者改内置实现本身。

### 和已安装 third_party 同名

- 插件管理页会同时显示两个变体：开发版和已安装版。
- 两个变体都可见，是为了让你明确知道现在系统里到底有哪些版本。
- 同一时刻只允许启用其中一个。
- 默认仍优先开发版，这样改完源码就能直接联调。
- 想测试正式安装版，直接在页面里切换启用即可，不需要临时移走 `plugins-dev` 里的源码。
- 如果已安装插件来自 marketplace，它的安装状态、配置状态、版本治理信息仍然保留，不会因为存在开发版而丢失。

### 只有 `plugins-dev` 里有这个插件

- 该插件会直接出现在家庭插件列表中，按第三方插件处理。
- 前端会把它标识为开发版。
- 它仍然走第三方插件的执行边界，不会获得 `builtin` 特权。

## 插件和宿主怎么分工

宿主负责：

- 家庭、成员、权限、审计、任务调度。
- 插件注册、启停、配置、执行、回收。
- 统一 DTO、统一错误语义、统一状态口径。

插件负责：

- 对接第三方 API、协议或平台。
- 把第三方数据映射成宿主需要的标准结构。
- 处理自己的供应商特例、缓存、限流、重试和资源文件。
- 插件自己的后端逻辑和运行时依赖。

后端边界现在要再写死一句：

- 插件后端逻辑和代码必须使用 Python 完成。
- 不要为了单个插件再往宿主运行时里塞 Node、JS sidecar 或其它第二套后端语言运行时。
- 真有第三方协议细节，就在 Python 插件里处理，不要把跨语言桥接伪装成“通用能力”。

第三方 Python 插件的依赖模型也别再误会：

- 安装态第三方插件会准备自己的 Python venv。
- 插件仓库里的 `requirements.txt` 会安装到这个插件自己的运行环境里。
- runner 执行第三方插件时，使用的是插件自己的 `python_path`，不是宿主 `sys.executable`。
- `plugins-dev` 下的开发版插件也应该走同一套思路，不再默认偷宿主环境。
- 第三方插件专属依赖不应该再回填进宿主 `pyproject.toml`；谁需要，谁装进自己的插件环境。

如果一段逻辑只有某个插件自己才知道，那它就该放在插件里，不要倒灌回宿主。

## `speaker/voice` 插件现在该怎么放

这类插件现在要分两层看，不要再把它们混成一坨：

1. 宿主先提供通用 `speaker_adapter` 契约。
2. 第三方厂商逻辑再作为独立插件接进来。

正确放法就是：

- 宿主通用桥接代码放在 `apps/api-server/app/modules/voice/` 和 `apps/api-server/app/modules/plugin/`。
- 第三方插件源码放在 `apps/api-server/plugins-dev/<plugin_dir>/`。
- 厂商自己的设备发现、轮询、播报、命令翻译，都留在第三方插件目录里。

拿一个中性的第三方音箱插件示例来说，应该放在：

- `apps/api-server/plugins-dev/speaker_gateway_adapter_demo/manifest.json`
- `apps/api-server/plugins-dev/speaker_gateway_adapter_demo/integration.py`
- `apps/api-server/plugins-dev/speaker_gateway_adapter_demo/action.py`
- `apps/api-server/plugins-dev/speaker_gateway_adapter_demo/speaker_adapter.py`

别再走错路：

- 不要直接修改 `app/plugins/builtin/open_xiaoai_speaker/` 来冒充“第三方插件接入”。
- 不要把厂商私有字段塞回宿主 `voice` 核心。
- 不要把文本轮询方案假装成原始音频会话方案；如果首版只支持文本轮询，就老老实实只声明 `supported_modes=["text_turn"]`。

如果插件配置流程里需要做真实登录、拉设备、提示二次认证，还要再加一层：

- 在 manifest 里声明 `entrypoints.config_preview`
- 由插件实现一个显式预检函数
- 前端只在用户点击“登录并加载设备”这类按钮时调用它
- 预检返回的临时状态可以放进 `view.runtime_state`
- 如果有二维码、临时链接、提示文本这类可直接展示的产物，统一放进 `view.preview_artifacts`

这层是给“配置过程里的副作用”用的，不是给正式保存字段用的。别再把真实登录塞进 `config/resolve`，那会把普通字段联动和真正的网络动作搅成一锅粥。

像这类浏览器 token / 厂商账号插件，还有一个很实际的坑要提前写清楚：

- `passToken` 主流程通常还需要同一份浏览器 Cookie 里的数值 `userId`
- 不要在表单里含糊写成“支持邮箱/手机号/账号”，那会把用户直接带进坑里
- 正确做法是把字段明确命名成“账号 ID”，并在帮助文案里写清楚去 Cookies 里找 `userId`

### `ui_schema.actions + runtime_sections` 的通用接法

如果你要做阶段式配置，比如“登录账号并加载设备”，现在不要再去改核心页面。正确做法是：

- 在 `manifest.json` 的 `ui_schema.actions` 里声明一个动作
- 给动作配 `section_id`、`depends_on_fields`、`action_key`
- 在 `ui_schema.runtime_sections` 里声明动作执行后要展示的内容
- 需要用户二次选择候选项时，用 `candidate_select`，并把结果写回一个正式字段，例如 `device_selector`
- 表单里的 `label / description / help_text / placeholder / runtime` 文案也由插件自己在 `manifest + locales` 里提供，宿主不再承载任何插件专属字典
- 如果同时写了原文和 `*_key`，就把原文写成真正能给用户看的内容。宿主会先尝试翻译 key，当前会话里拿不到翻译时，就回退到 `manifest` 里的原文，不要只留 key 名字当兜底

这样以后同一套宿主页面就能跑不同来源的音箱插件。核心只负责渲染和回填，不再知道任何厂商细节。

现在这套通用 staged form 不只跑集成实例创建弹窗，也跑 `channel_account` 作用域的账号创建/编辑弹窗。
如果你的渠道插件要做扫码登录、状态摘要、二维码预览，就在 `channel_account` 的 `manifest + locales` 里声明 `ui_schema.actions + runtime_sections + preview_artifacts`，不要再去改宿主页面的平台特判。

### `config_preview` 现在该返回什么

宿主现在正式识别下面这两类返回内容：

- `field_errors`
- `runtime_state`
- `preview_artifacts`

如果你的 staged login 流程里有第三方二次认证，不要把这套厂商细节塞进宿主。

现在推荐的做法是：

- 插件在 `config_preview` 里返回真实验证链接，让宿主只负责展示
- 用户在浏览器完成认证后，把最终跳转到的厂商 URL 粘贴回插件自己的表单字段
- 插件自己保存待恢复事务，自己解析这条 URL，再继续后半段登录或设备加载

边界别搞错：

- 宿主只负责通用 `config_preview`、`runtime_state`、`preview_artifacts` 渲染
- 插件自己负责厂商验证链接、待恢复事务、手工粘贴 URL 的解析和后续登录恢复
- 如果某条厂商认证链本身不支持稳定自动回调，就不要在宿主里硬造一套假的 callback 协议

`preview_artifacts` 是平台无关的展示产物列表。当前支持：

- `kind = "image_url"`：展示一张图片，比如登录二维码；可以是远程图片 URL，也可以是 `data:image/...` 这类可直接渲染的 data URL
- `kind = "external_url"`：展示一个外部链接
- `kind = "text"`：展示一段临时文本

最小例子：

```json
{
  "runtime_state": {
    "status": "waiting_scan",
    "message": "二维码已生成，请扫码。"
  },
  "preview_artifacts": [
    {
      "key": "login-qr",
      "kind": "image_url",
      "label": "登录二维码",
      "url": "https://example.com/login-qr.png"
    },
    {
      "key": "login-tip",
      "kind": "text",
      "label": "当前状态",
      "text": "二维码已生成，请扫码。"
    }
  ]
}
```

边界别搞错：

- `preview_artifacts` 是给宿主直接渲染的，不是给宿主理解厂商协议的
- `image_url` 必须真的是图片资源；如果上游给你的是二维码落地页、H5 页面或中转链接，应该由插件先转成可直接显示的图片，再交给宿主
- 不要再发 `qr_code_url`、`provider_login_status` 这种平台私有字段给宿主页面
- 真正要持久化的字段还是走正常配置保存；预览产物只是临时展示

#### staged login 现在该怎么正式调用

如果你的 `config_preview` 会触发真实第三方认证，宿主正式接口仍然只有这个：

- `POST /api/v1/ai-config/{household_id}/plugins/{plugin_id}/config/preview`

最小调用顺序：

1. 前端点某个 `ui_schema.actions` 声明的按钮
2. 调 `config/preview`，请求体里带 `action_key`
3. 插件返回 `runtime_state.verification_url` 或其他临时状态
4. 用户完成厂商认证后，把最终跳转 URL 粘贴回插件自己的字段
5. 前端再次调同一个 `config/preview`
6. 插件自己解析这条 URL，恢复登录并继续加载设备

这套流程里，宿主不再承担第三方 callback 会话管理。

### `speaker_adapter` 的 worker 和 heartbeat 边界

如果 manifest 里声明了 `capabilities.speaker_adapter.requires_runtime_worker=true`，插件就要自己把常驻 worker 做完整，不要指望宿主替你轮询第三方平台。

这个 worker 至少要负责四件事：

- 轮询、长连接消费，或者其它厂商侧事件接收
- 把拿到的 query 转成宿主 `speaker text_turn` 请求
- 把宿主返回的 `reply_text` 或 `audio_url` 真正播回设备
- 定时上报 `SpeakerRuntimeHeartbeat`

推荐把 heartbeat 状态用得老实一点：

- 空闲但健康：`idle`
- 正常处理：`running`
- 还能继续跑，但已经进入回退：`degraded`
- 最近一次关键处理失败：`error`
- 主动停止或无法继续：`stopped`

宿主只做状态收口，不接管你的厂商逻辑：

- `running` / `idle` 收口成实例 `active`
- `degraded` / `error` / `stopped` 收口成实例 `degraded`
- 插件本身被停用时，实例统一收口成 `disabled`

死规矩还是那句：

- 宿主不替插件维持账号会话
- 宿主不替插件轮询第三方平台
- 宿主只负责 `text_turn`、`heartbeat`、配置表单渲染和状态展示

### `text_turn` 结果怎么接音频 URL

如果你的音箱插件要支持“宿主给一段可直接播放的远程音频”，现在别再自己发明私有字段了，按宿主正式结果走：

1. 插件先把用户 query 提交给宿主 `text_turn`
2. 宿主如果只回文本，插件读 `reply_text`
3. 宿主如果回远程音频，插件读 `result_type="audio_url"` 和 `audio_url`
4. 如果设备不支持直接播 URL，就退回 `fallback_text` 或 `reply_text`

宿主识别的事实载荷约定是：

```json
{
  "type": "speaker_audio_reply",
  "audio_url": "https://media.example.com/reply.mp3",
  "content_type": "audio/mpeg",
  "fallback_text": "这段文字给不支持直链播放的设备回退"
}
```

别再搞错边界：

- 这仍然是文本轮询型对话，不是 `audio_session`
- `audio_url` 必须是插件端实际能访问到的绝对地址
- 如果你根本不能保证设备能访问这个 URL，就别宣称自己已经支持 `custom_audio_url`

## 最小开发流程

1. 先确认插件类型和能力边界。
2. 在 `apps/api-server/plugins-dev/<plugin_dir>/` 下建目录。
3. 先写对 `manifest.json`，再写入口模块。
4. 需要配置时补 `config_specs`，不要把配置协议偷偷写死在代码里。
5. 先做最小自测，再走宿主联调。
6. 如果同时存在已安装版，到插件管理页确认开发版和已安装版两个变体都能看见，再按需要切换启用。

补一句很实际的话：

- 如果你的插件配置里本来就是结构化数据，比如设备筛选器、profile override、批量枚举选项，不要再伪装成一大段字符串。
- 现在设备与集成的通用表单已经能承接 `multi_enum` 和 `json`，直接用正式字段类型声明。

推荐最小目录：

```text
apps/api-server/plugins-dev/your_plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    integration.py
  tests/
```

## 开发时最容易犯的错

- 还在用 `data/plugins/official/` 当开发目录。
- 把“开发源码目录”和“安装目录”混成一件事。
- 以为开发版一出现，已安装版就应该被隐藏。
- 为了测试安装版，手动移动、重命名或删除 `plugins-dev` 目录下的源码。
- 试图让 `plugins-dev` 覆盖 `builtin`。
- 直接改 `data/plugins/third_party/...` 里的运行时产物，而不是改源码目录。

## `channel.send` 现在的通用投递载荷

宿主现在给 `channel` 插件的出站投递，不再只剩下 `text + metadata`。

当前统一载荷至少包含：

- `delivery.text`
- `delivery.attachments`
- `delivery.metadata`

其中 `attachments` 是平台无关的通用列表。每一项至少表达：

- `kind`：`image` / `video` / `audio` / `file`
- `file_name`
- `content_type`
- `source_path` 或 `source_url`
- `size_bytes`
- `metadata`

边界要看清：

- 宿主只负责把统一附件信息交给插件，不负责理解厂商上传协议。
- 插件可以只支持自己声明的附件类型；不支持就明确报错，不要静默吞掉。
- 老插件如果只关心文本，继续读 `delivery.text` 就行；新增 `attachments` 不会要求它们改宿主逻辑。

最小示例：

```json
{
  "delivery": {
    "text": "请看附件",
    "attachments": [
      {
        "kind": "image",
        "file_name": "photo.png",
        "content_type": "image/png",
        "source_path": "C:/tmp/photo.png",
        "size_bytes": 2048,
        "metadata": {
          "caption": "示例图片"
        }
      }
    ],
    "metadata": {
      "context_token": "token-123"
    }
  }
}
```

正式约束：

- `text` 和 `attachments` 至少要有一个，不能都空
- 每个附件至少要有 `source_path` 或 `source_url`
- 宿主不替插件做上传、转码、鉴权
- 插件如果支持媒体，就应该只从 `delivery.attachments` 读取，不要再发明平台私有顶层字段

## `channel.poll` 里的通用媒体引用

如果 `channel` 插件把入站媒体下载到了自己的私有运行目录，就不要再把厂商下载参数直接塞回宿主。

当前推荐做法是：

- 下载后的文件保存在插件自己的 `working_dir/media/` 下面
- 把标准化附件列表放到 `normalized_payload.attachments`
- 为了兼容当前只读 metadata 的消费方，再镜像一份到 `normalized_payload.metadata.attachments`
- 如果有部分媒体下载失败，把错误摘要放到 `normalized_payload.metadata.media_download_errors`

入站附件也尽量沿用和出站附件一致的通用结构：

- `kind`
- `file_name`
- `content_type`
- `source_path`
- `size_bytes`
- `metadata`

边界还是那条老规矩：

- 宿主不理解厂商 CDN token、解密参数和上传下载协议
- 插件负责把这些脏活关在自己目录里，只把宿主能消费的标准引用和错误摘要交出来

## 通讯平台接入页怎么发现第三方 `channel` 插件

前端“通讯平台接入”页现在必须按已注册插件快照动态展示平台，不能再写死 Telegram、Discord 这类内置名单。
如果页面还是靠硬编码数组渲染平台选择弹窗，第三方 `channel` 插件就永远不会出现，这不是功能缺失，这是宿主把插件边界写坏了。

现在的通用规则是：

- 平台选择列表基于 `listRegisteredPlugins()` 返回的插件快照生成
- 只展示 `types` 包含 `channel`、并且声明了 `capabilities.channel.platform_code` 的启用插件
- 创建账号时直接提交真实 `plugin_id`，不要自己拼 `channel-${platformCode}`
- 已知平台可以复用宿主内置品牌名和 Logo；未知平台就回退到 `plugin.name` 和通用图标，不准默认冒充 Telegram 之类的内置平台

这条规则是宿主通用能力，不是某个平台的特判。今天接的是一个第三方通道，明天换成另一个，入口也应该自动吃进去。

## `channel` 插件怎么声明账号级动作和状态摘要

如果你的 `channel` 插件需要在账号详情页提供“开始登录”“刷新状态”“清理本地状态”这类动作，不要让前端自己猜按钮，也不要让宿主硬编码动作名。

现在要按 manifest 正式声明：

- `capabilities.channel.ui.account_actions`
- `capabilities.channel.ui.status_action_key`

最小例子：

```json
{
  "types": ["channel", "action"],
  "entrypoints": {
    "channel": "plugin.channel.handle",
    "action": "plugin.action.execute"
  },
  "capabilities": {
    "channel": {
      "platform_code": "third-party-chat",
      "inbound_modes": ["polling"],
      "delivery_modes": ["reply"],
      "ui": {
        "status_action_key": "refresh-login-status",
        "account_actions": [
          {
            "key": "start-login",
            "action_name": "start_login",
            "label": "开始登录",
            "variant": "primary"
          },
          {
            "key": "refresh-login-status",
            "action_name": "get_login_status",
            "label": "刷新登录状态"
          }
        ]
      }
    }
  }
}
```

边界继续守死：

- 只要声明了 `account_actions`，插件就必须同时声明 `action` 类型和 `entrypoints.action`
- 宿主只认 `key`、`action_name`、按钮文案、危险级别和确认提示，不理解二维码、微信登录阶段之类平台细节
- 账号详情页里展示什么状态，优先读插件动作结果里的 `status_summary`
- 如果动作有可展示产物，比如二维码或跳转链接，统一放进 `artifacts`，不要再给宿主发平台私有字段

插件 `action` 结果现在推荐至少返回：

- `message`
- `status_summary`
- `artifacts`

其中 `status_summary` 是平台无关的摘要对象，`artifacts` 是平台无关的展示产物列表。宿主只负责展示，不负责理解厂商协议。

## 配置表单怎么给普通用户减负

如果你的插件要暴露很多参数，不要把所有字段继续堆成一个长表单。

如果其中某个字段只是为了展示说明、状态或只读内容，不要再假装它是可编辑输入框。现在 `ui_schema.widgets` 里可以显式写：

```json
{
  "widgets": {
    "login_hint": {
      "widget": "display"
    }
  }
}
```

`display` 的含义很简单：

- 宿主按只读块渲染这个字段
- 它不会出现输入控件
- 适合展示静态说明、只读摘要或插件通过默认值回填的只读内容

别滥用：

- 二维码、链接、临时文本这类预览产物优先走 `preview_artifacts`
- `display` 适合字段级只读内容，不是拿来替代 `config_preview` 副作用入口的

如果只是“帮助说明很长”，也不要把整段教程直接摊在输入框底下。现在同一个 `widgets` 里还能声明折叠帮助：

```json
{
  "widgets": {
    "pass_token_secret_ref": {
      "widget": "password",
      "help_text": "1. 浏览器登录\n2. 打开开发者工具\n3. 到 Cookies 里找 passToken",
      "help_text_collapsible": true,
      "help_text_toggle_label": "展开查看如何获取 passToken"
    }
  }
}
```

这玩意的定位很明确：

- 说明必须留在插件自己的 `manifest + locales` 里
- 宿主只负责渲染“默认折叠、点击展开”的通用交互
- 适合登录教程、导入步骤、注意事项这种长帮助，不适合拿来塞运行时状态

现在集成创建弹窗会把 `ui_schema.sections` 当成阶段式向导，而且支持把某些 section 标成高级步骤：

```json
{
  "id": "advanced",
  "title": "高级排障",
  "mode": "advanced",
  "fields": ["request_timeout_ms", "enable_trace"]
}
```

直接按这个规则拆：

- 默认 section 不写 `mode`，就是普通步骤
- `mode = "advanced"` 的 section 会先折叠，普通用户走完基础步骤就可以直接保存
- 只有用户主动点“高级设置”时，这些步骤才会继续展开
- 所以账号、基础行为、必要绑定信息放前面；排障、覆盖、筛选参数放高级步骤
