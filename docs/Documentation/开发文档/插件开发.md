# 插件开发

这页只讲当前还有效的插件开发口径，不保留旧 `official` 时代的废话。

## 先记住 6 句话

1. 插件类型只剩两种：`builtin`、`third_party`。
2. 安装方式只区分两种：`local`、`marketplace`。
3. 仓库内第三方开发源码统一放 `apps/api-server/plugins-dev/`。
4. `plugins-dev` 会参与家庭插件注册表合并，是开发覆盖源，不是安装目录。
5. 同名优先级固定为 `builtin > plugins-dev > 已安装 third_party`。
6. `plugins-dev` 可以覆盖第三方插件，但绝不覆盖 `builtin`。
7. 当前家庭实际生效的是 `plugins-dev` 版本时，前端插件页会直接显示“开发插件生效中”标签。

## 现在到底有哪些目录

- `apps/api-server/app/plugins/builtin/`
  内置插件源码目录，跟宿主一起发布。
- `apps/api-server/plugins-dev/`
  第三方插件开发源码目录，也是开发覆盖源，默认已经 git ignore。
- `apps/api-server/data/plugins/third_party/local/`
  本地安装的第三方插件运行时目录。
- `apps/api-server/data/plugins/third_party/marketplace/`
  市场安装的第三方插件运行时目录。

不要再把开发中的插件放到 `apps/api-server/data/plugins/official/`。
那个路径已经是历史目录，不再是当前开发规范的一部分。

## `plugins-dev` 和安装目录的区别

### `plugins-dev` 负责什么

- 放开发中的第三方插件源码。
- 参与 `list_registered_plugins_for_household()` 的注册表合并。
- 在 `plugin_id` 相同且目标是第三方插件时，覆盖运行时看到的 manifest、entrypoint 和 runner 配置。
- 让开发者改源码后，不需要反复把插件重新装到市场目录或本地安装目录。

### `plugins-dev` 不负责什么

- 不写入 `plugin_mounts`。
- 不参与 marketplace 或 local 安装落盘。
- 不参与卸载时的磁盘清理。
- 不替代启动同步目录。

## 冲突规则

### 和 `builtin` 同名

- 直接以 `builtin` 为准。
- `plugins-dev` 中的同名插件会被跳过。
- 不允许用开发插件去覆盖内置插件。

### 和已安装第三方插件同名

- 开发态以 `plugins-dev` 为准。
- 如果已安装插件来自 marketplace，市场实例状态、安装状态、配置状态、版本治理信息仍然保留。
- 当前家庭插件列表和插件详情会显示“开发插件生效中”，明确告诉你现在跑的是开发版。
- 删除安装态时，只删除 `third_party/local` 或 `third_party/marketplace` 下的产物，不删除 `plugins-dev` 源码。

### 只有 `plugins-dev` 有该插件

- 该插件会直接出现在家庭插件注册表里，按第三方插件处理。
- 默认可见，便于开发联调。
- 仍然走 `subprocess_runner`，不会获得 `builtin` 的同进程特权。

## 插件和宿主怎么分工

宿主负责：

- 家庭、成员、权限、审计、任务调度。
- 插件注册、启停、配置、执行、回收。
- 统一 DTO、统一错误语义、统一状态口径。

插件负责：

- 对接第三方 API、协议或平台。
- 把第三方数据映射成宿主需要的标准结构。
- 自己的供应商特例、缓存、限流、重试和资源文件。

如果一段逻辑只有某个插件自己才知道，那它就属于插件，不属于宿主。

## 最小开发流程

1. 先确定插件类型和能力边界。
2. 在 `apps/api-server/plugins-dev/<plugin_dir>/` 下建目录。
3. 先写对 `manifest.json`，再写入口模块。
4. 需要配置时补 `config_specs`，不要把配置协议偷偷写死在代码里。
5. 先做最小自测，再走宿主联调。

推荐最小目录：

```text
apps/api-server/plugins-dev/your_plugin/
  manifest.json
  requirements.txt
  README.md
  plugin/
    __init__.py
    integration.py
  tests/
```

## 开发时最容易犯的错

- 还在用 `data/plugins/official/` 当开发目录。
- 把“开发覆盖源”和“安装目录”混成一件事。
- 试图让 `plugins-dev` 覆盖 `builtin`。
- 直接改 `data/plugins/third_party/...` 里的运行时产物，而不是改源码目录。
- 让宿主按插件名字写分支逻辑。
