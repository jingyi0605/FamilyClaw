# 配置接入

这页只讲当前稳定的接入边界。

## 配置入口只有一套

插件需要让用户填参数时，只能走：

- `manifest.config_specs`

不要再用：

- 宿主页硬编码字段
- 业务表里偷偷塞 JSON 再让前端自己猜

## 当前正式作用域

- `plugin`
- `integration_instance`
- `device`
- `channel_account`

怎么选：

- 整个插件共用一份，用 `plugin`
- 每个集成实例一份，用 `integration_instance`
- 每台设备一份，用 `device`
- 每个通道账号一份，用 `channel_account`

## 宿主现在提供三条配置接口

### 读取当前表单

`GET /api/v1/ai-config/{household_id}/plugins/{plugin_id}/config`

特点：

- 返回的 `config_spec` 已经按当前已保存值解析过
- 前端打开表单时先用它

### 解析当前草稿

`POST /api/v1/ai-config/{household_id}/plugins/{plugin_id}/config/resolve`

特点：

- 不落库
- 只根据当前草稿值重算可用选项
- 适合创建新实例和字段联动

### 保存配置

`PUT /api/v1/ai-config/{household_id}/plugins/{plugin_id}/config`

特点：

- 后端统一校验
- 后端统一处理默认值、动态选项、secret 和清空语义

## 清空语义别再乱来

### 普通字段

普通字段要删值，正式走：

- `clear_fields`

### secret 字段

secret 字段要删值，正式走：

- `clear_secret_fields`

死规矩：

- 不提交不代表清空，只代表“保持旧值”
- 同一字段不能一边提交新值一边要求清空

## 做联动表单时怎么写 manifest

需要联动的枚举字段，至少考虑四件事：

1. 它的选项是不是动态来的
2. 它依赖哪些上游字段
3. 上游变了以后旧值要不要清掉
4. 字段没准备好时是不是应该隐藏

一个完整例子：

```json
{
  "key": "district_code",
  "type": "enum",
  "depends_on": ["provider_code", "province_code", "city_code"],
  "clear_on_dependency_change": true,
  "option_source": {
    "source": "region_catalog_children",
    "country_code": "CN",
    "provider_field": "provider_code",
    "parent_field": "city_code",
    "admin_level": "district"
  }
}
```

对应的 `ui_schema.widgets` 可以再加 `visible_when`：

```json
{
  "district_code": {
    "widget": "select",
    "visible_when": [
      { "field": "binding_type", "operator": "equals", "value": "region_node" },
      { "field": "city_code", "operator": "truthy" }
    ]
  }
}
```

## 宿主和插件分别负责什么

插件负责：

- 定义字段
- 定义校验
- 定义动态选项来源
- 定义字段依赖和 UI 提示

宿主负责：

- 读取 schema
- 渲染表单
- 在字段变化后调用 resolve
- 清理失效值
- 保存值
- 保护 secret

## 天气插件接法已经定了

天气插件现在的标准接法：

- 默认天气实例也按普通 `integration_instance` 显式创建，不再依赖“启用插件时自动补一个默认实例”
- `provider_code`：动态拉可用地区 provider
- `province_code`：按 provider 拉省级目录
- `city_code`：按 provider + province 拉城市目录
- `district_code`：按 provider + city 拉区县目录

不要再写：

- 巨型静态省市区 `enum_options`
- `provider_selector` 这种只为某一个插件做的过渡字段

## 一个最容易踩的坑

如果你做的是“父级变化后隐藏子级字段”的表单，只在前端把字段藏起来还不够。

你必须同时：

1. 把子级值从当前草稿里删掉
2. 在保存时把它放进 `clear_fields`

不然数据库里还是会留着旧值，最后看起来像是联动了，实际一保存还是脏的。
