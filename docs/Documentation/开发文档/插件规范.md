# 插件规范

这页把宿主和插件的分工边界说死，避免大家继续拿插件当“哪里不顺手就塞一段代码”的垃圾桶。

## 这页的定位

这页就是当前官方文档站里的插件边界基线。

如果仓库里别的补充资料和这页写的不一致，先以这里和当前代码事实为准，再去核代码，而不是反过来让读者自己去外面拼文档。

当前代码里的直接事实源主要是：

- `apps/api-server/app/modules/plugin/schemas.py`
- `apps/api-server/app/modules/plugin/`
- `apps/api-server/app/plugins/builtin/`

## 宿主和插件各干什么

| 能力 | 宿主负责 | 插件负责 |
| --- | --- | --- |
| 家庭、成员、房间、权限 | 是 | 否 |
| 插件注册、挂载、启停 | 是 | 否 |
| 配置值保存、密钥保存 | 是 | 否 |
| 任务调度、重试、审计 | 是 | 否 |
| 第三方 API / 设备 / 平台接入 | 否 | 是 |
| 第三方字段映射 | 否 | 是 |
| 标准实体、标准动作、标准卡片语义 | 是 | 否 |
| 插件私有缓存和厂商特例处理 | 否 | 是 |

一句话：

宿主是平台内核，插件是领域接入层。

## 当前正式插件类型

- `integration`
- `action`
- `agent-skill`
- `channel`
- `locale-pack`
- `region-provider`
- `theme-pack`
- `ai-provider`

独占槽位：

- `memory_engine`
- `memory_provider`
- `context_engine`

## 插件私有状态边界

插件可以有自己的：

- 私有表
- 私有缓存
- 刷新游标
- 原始快照

但宿主核心不应该：

- 直接导入插件私有 ORM
- 把插件私有模型注册到宿主总模型入口
- 直接靠 SQL 去读插件私表

宿主只消费标准 DTO、标准状态表和公开接口。

## 启用 / 禁用语义

最终状态统一只认：

- `PluginRegistryItem.enabled`
- `PluginRegistryItem.disabled_reason`

禁用后的正确语义是：

- 不能新增使用
- 不能继续执行
- 不能继续自动触发
- 但可以看
- 但可以改配置

执行类入口必须统一校验，不允许某个页面拦了、另一个后台任务又偷偷跑。

如果你要改这块逻辑，最好顺手对照 [后端开发](./后端开发.md) 里的“插件可用性语义必须统一”章节一起看，别只在一个接口上修补。

## 配置联动规则

动态配置联动属于宿主能力，不属于插件自己乱发明协议的范围。

当前正式动态选项源包括：

- `region_provider_list`
- `region_catalog_children`

插件负责声明：

- 字段依赖
- `option_source`
- 依赖变化后需要清空哪些字段

宿主负责：

- 解析依赖
- 重算动态选项
- 保存配置草稿

## 主题和语言插件的边界

### `theme-pack`

必须声明：

- `theme_id`
- `display_name`
- `tokens_resource` 或 `entry_resource`
- `resource_source`
- `resource_version`
- `theme_schema_version`
- `platform_targets`

宿主和前端运行时现在只认插件资源，不再认宿主里偷偷藏的静态主题表。

### `locale-pack`

负责提供完整语言资源包。

普通插件也可以声明自己的 `locales`，但只能补自己这部分文案，不能假装自己是整套系统语言包。

## AI 供应商插件的边界

`ai-provider` 插件负责：

- 供应商字段 schema
- adapter 能力声明
- driver 入口
- 厂商协议适配

宿主负责：

- AI 路由
- 权限
- 密钥引用
- 审计
- 降级与回退

## 最重要的判断标准

如果一个能力上线后，宿主还要为了它单独写一堆领域分支，那这个插件就没有把复杂度吞掉，设计是不合格的。
