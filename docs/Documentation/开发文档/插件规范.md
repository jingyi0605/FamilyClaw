# 插件规范

这页把宿主和插件的分工边界说死，避免大家继续拿插件当“哪里不顺手就塞一段代码”的垃圾桶。

## 这页的定位

这页就是当前官方文档站里的插件边界基线。

如果仓库里别的补充资料和这页写的不一致，先以这里、相关 spec 和当前代码事实为准，再去核代码，而不是让读者自己去外面拼文档。

这次和 AI provider 插件化直接相关的事实源主要是：

- `apps/api-server/app/modules/plugin/schemas.py`
- `apps/api-server/app/modules/ai_gateway/provider_driver.py`
- `apps/api-server/app/modules/ai_gateway/provider_runtime.py`
- `apps/api-server/app/plugins/_sdk/`
- `apps/api-server/app/plugins/builtin/`
- `specs/004.8.1-AI供应商彻底插件化迁移/`

## 宿主和插件各干什么

| 能力 | 宿主负责 | 插件负责 |
| --- | --- | --- |
| 家庭、成员、房间、权限 | 是 | 否 |
| 插件注册、挂载、启停 | 是 | 否 |
| 配置值保存、密钥保存 | 是 | 否 |
| 任务调度、重试、审计 | 是 | 否 |
| 第三方 API / 设备 / 平台接入 | 否 | 是 |
| 第三方字段映射 | 否 | 是 |
| 标准实体、标准动作、标准卡片语义 | 是 | 否 |
| 插件私有缓存和厂商特例处理 | 否 | 是 |

一句话：
宿主是平台内核，插件是领域接入层。

## 插件私有状态边界

插件可以有自己的：

- 私有表
- 私有缓存
- 刷新游标
- 原始快照

但宿主核心不应该：

- 直接导入插件私有 ORM
- 把插件私有模型注册到宿主总模型入口
- 直接靠 SQL 去读插件私表

宿主只消费标准 DTO、标准状态表和公开接口。

## 启用 / 禁用语义

最终状态统一只认：

- `PluginRegistryItem.enabled`
- `PluginRegistryItem.disabled_reason`

禁用后的正确语义是：

- 不能新增使用
- 不能继续执行
- 不能继续自动触发
- 但可以看
- 但可以改配置

执行类入口必须统一校验，不允许某个页面拦了、另一个后台任务又偷偷跑。

## AI 供应商插件边界

这一块是本轮最重要的更新。

从 `004.8.1` 开始，AI 供应商插件化已经收紧到下面三条红线：

1. 核心不得再出现任何供应商协议枚举驱动的请求拼装分支。
2. 任一 provider 的请求编码、流解析、响应提取必须可在插件内独立实现。
3. 新增 provider 如需改核心，只允许改宿主通用能力，不允许改供应商协议逻辑。

这三条不是建议，是正式规范。

### 宿主现在只保留什么

宿主 `ai_gateway` 只应该保留：

- 统一调用入口
- provider profile 管理
- 路由、fallback、timeout、retry
- 插件启停和 household 可见性校验
- 密钥解析、权限、审计
- 统一错误语义和统一结果对象
- 模板降级输出

宿主不应该再保留：

- OpenAI 请求体拼装
- Anthropic 请求体拼装
- Gemini 请求体拼装
- 某家 provider 专属 stream parser
- 某家 provider 专属 response extractor
- 某家 provider 专属 header、messages 结构、参数编码规则

### `ai-provider` 插件必须负责什么

`ai-provider` 插件必须自己负责：

- 供应商字段 schema
- `branding`
- `config_ui`
- `model_discovery`
- provider driver 入口
- 请求编码
- messages 结构映射
- stream/event 解析
- 响应提取
- 供应商特例
- 模型级参数修正

### 共享 helper 可以有，但必须在插件层

允许有共享 helper，但位置必须在插件层，而不是核心目录。

当前正式位置：

- `apps/api-server/app/plugins/_sdk/ai_provider_messages.py`
- `apps/api-server/app/plugins/_sdk/ai_provider_drivers.py`

它们的定位是“插件 SDK”，不是“核心协议层”。

所以现在的正式边界是：

- 开发者可以复用插件 SDK
- 开发者也可以完全自己写协议实现
- 只要最终满足统一 driver 契约就行

## 最重要的判断标准

如果一个能力上线后，宿主还要为了它单独写一堆领域分支，那这个插件就没有把复杂度吞掉，设计就是不合格的。
