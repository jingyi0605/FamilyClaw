# 对接方式

这页讲的是宿主到底怎么调用插件，以及联调时应该盯哪些接口。

## 总原则

插件执行入口只接受可序列化输入，只返回标准结果。

不要把下面这些东西直接塞进插件调用：

- FastAPI 请求对象
- SQLAlchemy Session
- 不能序列化的 Python 对象

## 典型执行链路

### `integration`

1. 宿主读取插件 manifest
2. 校验插件类型、状态和能力声明
3. 注入配置与运行时上下文
4. 调用 `integration` 入口
5. 插件返回标准实体、卡片、动作描述
6. 宿主统一落库和更新状态

### `action`

1. 校验插件已启用
2. 校验调用权限与风险等级
3. 必要时走确认链路
4. 执行动作入口
5. 宿主统一收口执行结果

### `ai-provider`

1. 宿主按路由策略找到供应商插件
2. 读取 provider profile 和 secret 引用
3. 调 `entrypoints.ai_provider`
4. 宿主统一处理超时、重试、回退、审计

### `channel`

1. 宿主保存渠道账号与绑定关系
2. 渠道事件进入后由 channel 插件解析
3. 宿主决定会话归属、成员绑定与后续分发
4. 回复或通知再经 channel 插件发送

### `theme-pack` / `locale-pack`

这类资源插件不走重执行链路，主要是：

1. 宿主注册插件
2. 前端读取可用资源清单
3. 再按资源路径加载主题 token 或语言资源

## OpenAPI 怎么看

后端跑起来后，最直接的入口就是：

- `/docs`
- `/redoc`
- `/openapi.json`

别把 OpenAPI 当设计文档。它告诉你字段长什么样，不告诉你边界为什么这么定。边界还是要看规范和代码。

## WebSocket 联调

当前实时入口：

- `/api/v1/realtime/voice`
- `/api/v1/realtime/agent-bootstrap`
- `/api/v1/realtime/conversation`

联调时重点看：

- 鉴权是否通过
- `household_id` 和 `session_id` 是否匹配
- 事件类型是否符合后端定义

## 当前最靠谱的事实源

如果你只看一层就想改代码，至少看这里：

- `apps/api-server/app/api/v1/router.py`
- `apps/api-server/app/api/v1/endpoints/`
- `packages/user-core/src/api/create-api-client.ts`

前两者是后端真实入口，后者是前端当前正在调用的契约。

## 最后提醒

不要把“联调通过”理解成“设计没问题”。

如果一个新插件要联调成功，必须靠宿主和前端各自打三层补丁，那不是联调难，而是对接方式本身就已经偏了。
