# 对接方式

这页只讲当前稳定的插件接入边界，不再沿用 `official` 时代的旧说法。

## 插件来源模型

- 插件来源只区分 `builtin` 和 `third_party`。
- 第三方插件的安装方式只区分 `local` 和 `marketplace`。
- `plugins-dev` 不是新的来源类型，它是仓库内第三方插件的开发源码目录。

## 当前运行时口径

宿主现在处理第三方插件冲突，不再靠“把安装版藏起来”这类做法凑合，而是按下面的规则工作：

1. `builtin` 永远最高优先级。
2. `plugins-dev` 会被加载。
3. 当 `plugins-dev` 和已安装 third_party 的 `plugin_id` 相同时，插件管理页会同时显示两个变体。
4. 同一个 `plugin_id` 在同一时刻只允许启用一个变体。
5. 默认优先开发版，方便开发联调。
6. 想测试正式安装版，直接切换启用，不需要手动移动目录。

## `plugins-dev` 的正确定位

- 它是开发源码目录。
- 它会参与插件发现和注册表构建。
- 它不是安装目录。
- 它不写入 `plugin_mounts`。
- 它不参与卸载清理。

不要再把它理解成“纯示例目录”，也不要再把它理解成“开发版一出现就把安装版记录覆盖没了”。

## 前后端对齐规则

### 后端必须说清楚变体身份

当同一个 `plugin_id` 同时存在开发版和已安装版时，后端查询结果必须明确告诉前端：

- 这是开发版还是已安装版。
- 当前实际启用的是哪个变体。
- 当前是否是“开发插件生效中”。

前端不应该再靠 `manifest_path`、目录路径或安装方式去猜当前跑的是谁。

### 前端必须显式展示

插件管理页至少要让用户一眼看明白：

- 这是开发版还是已安装版。
- 当前是否正在使用开发版。
- 切换启用会不会把另一个同名变体自动停掉。

## 开发时怎么联调

### 只改业务逻辑

- 直接改 `apps/api-server/plugins-dev/<plugin_dir>/`
- 用插件注册表、配置接口和执行接口验证行为
- 如果同 `plugin_id` 已有安装版，到插件管理页直接切换开发版和已安装版

### 验证安装链路

- 本地安装写入 `data/plugins/third_party/local`
- 市场安装写入 `data/plugins/third_party/marketplace`
- 启动同步只恢复安装目录，不给 `plugins-dev` 创建挂载记录

## 冲突处理规则

### 和内置插件同名

- 直接保留 `builtin`
- 开发版不生效

### 和第三方插件同名

- 开发版和已安装版都保留可见
- 默认开发版启用
- 同时只允许启用一个
- marketplace 或 local 的安装语义继续保留
- 删除安装产物不会删除 `plugins-dev`

## 路径口径

- 当前开发源码目录：`apps/api-server/plugins-dev/`
- 当前本地安装目录：`apps/api-server/data/plugins/third_party/local/`
- 当前市场安装目录：`apps/api-server/data/plugins/third_party/marketplace/`
- 历史废弃目录：`apps/api-server/data/plugins/official/`

## 一句话结论

把“插件来源”“安装方式”“开发源码目录”“当前启用变体”这几件事拆开说清楚。再拿一个旧 `official` 词同时表示来源、目录、信任级别和安装方式，只会继续制造混乱。
