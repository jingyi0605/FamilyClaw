# 对接方式

这页只讲当前稳定的插件接入边界，不再沿用 `official` 时代的旧说法。

## 插件来源模型

- 插件类型只区分 `builtin` 和 `third_party`
- 安装方式只区分 `local` 和 `marketplace`
- `is_system` 只描述市场源是不是系统内置源，不描述插件类型

## 运行时解析顺序

家庭插件注册表按下面顺序组装：

1. `builtin`
2. `plugins-dev`
3. 已安装 third_party

因此：

- `plugins-dev` 会覆盖同名第三方插件的运行时 manifest、entrypoint 和 runner 配置
- `plugins-dev` 不会覆盖 `builtin`
- 已安装 marketplace 插件被开发版覆盖时，市场实例状态仍然保留

## `plugins-dev` 的正确定位

- 它是开发源码目录
- 它是开发覆盖源
- 它不是安装目录
- 它不写入 `plugin_mounts`
- 它不参与卸载清理

不要再把它理解成“纯示例目录”或“完全不参与加载的目录”。

## 开发时该怎么联调

### 只改业务逻辑

- 直接改 `apps/api-server/plugins-dev/<plugin_dir>/`
- 用家庭插件注册表、配置接口和执行接口验证行为

### 验证安装链路

- 本地安装写入 `data/plugins/third_party/local`
- 市场安装写入 `data/plugins/third_party/marketplace`
- 启动同步只恢复安装态目录，不给 `plugins-dev` 创建挂载记录

## 冲突处理规则

### 和内置插件同名

- 直接保留 `builtin`
- 开发版被跳过

### 和第三方插件同名

- 开发版覆盖第三方运行态
- marketplace/local 的安装语义继续保留
- 删除安装态不会删掉 `plugins-dev`

## 路径口径

- 当前开发源码目录：`apps/api-server/plugins-dev/`
- 当前本地安装目录：`apps/api-server/data/plugins/third_party/local/`
- 当前市场安装目录：`apps/api-server/data/plugins/third_party/marketplace/`
- 历史废弃目录：`apps/api-server/data/plugins/official/`

## 一句话结论

接入时把“插件类型”“安装方式”“开发覆盖源”这三件事拆开想。别再用一个旧 `official` 词同时表示来源、信任级别、目录位置和安装方式，那就是复杂度垃圾。
