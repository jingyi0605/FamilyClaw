# 对接方式

这页讲的是宿主到底怎么调用插件，以及联调时应该盯哪些接口。

## 总原则

插件执行入口只接收可序列化输入，只返回标准结果。

不要把下面这些东西直接塞进插件调用：

- FastAPI 请求对象
- SQLAlchemy Session
- 不可序列化的 Python 对象

## 典型执行链路

### `integration`

1. 宿主读取插件 manifest
2. 校验插件类型、状态和能力声明
3. 注入配置与运行时上下文
4. 调用 `integration` 入口
5. 插件返回标准实体、卡片、动作描述
6. 宿主统一落库和更新状态

### `action`

1. 校验插件已启用
2. 校验调用权限与风险等级
3. 必要时走确认链路
4. 执行动作入口
5. 宿主统一收口执行结果

### `ai-provider`

这是本轮更新的重点。

当前正式链路是：

1. 宿主按路由策略找到供应商插件
2. 宿主读取 provider profile 和 secret 引用
3. 宿主加载 `entrypoints.ai_provider`
4. 宿主调用统一 driver 契约：
   - `invoke(...)`
   - `ainvoke(...)`
   - `stream(...)`
5. 宿主统一处理 timeout、fallback、审计和错误收口

### `channel`

1. 宿主保存渠道账号与绑定关系
2. 渠道事件进入后由 channel 插件解析
3. 宿主决定会话归属、成员绑定与后续分发
4. 回复或通知再经 channel 插件发送

## `ai-provider` 的正式边界

### 宿主负责什么

宿主只负责：

- 统一调用入口
- provider profile 管理
- 路由、fallback、timeout、retry
- 插件启停和 household 可见性校验
- 密钥解析、权限、审计
- 统一错误语义和统一结果对象

### 插件负责什么

插件负责：

- 请求编码
- messages 结构映射
- stream/event 解析
- 响应提取
- 供应商特例
- 模型发现

### 现在允许怎么复用

当前正式做法是两种都允许：

1. 复用插件层 SDK
2. 在自己的插件目录里完全自己实现协议

当前插件 SDK 位置：

- `apps/api-server/app/plugins/_sdk/ai_provider_messages.py`
- `apps/api-server/app/plugins/_sdk/ai_provider_drivers.py`

它们只是工具箱，不是强制框架。

## 业务模块如何做流式调用

业务模块如果要做流式调用，必须：

1. 先解析 provider driver
2. 再调用 `driver.stream(...)`

不应该继续直接调用核心协议 runtime。

也就是说，下面这类老路径不该再新增：

- `stream_provider_invoke(...)`
- `ProviderAdapter(...)`

## 当前联调最常用的接口组

### AI 配置与插件挂载

- `GET /api/v1/ai-config/provider-adapters`
- `GET /api/v1/ai-config/{household_id}/provider-profiles`
- `POST /api/v1/ai-config/{household_id}/provider-profiles`
- `GET /api/v1/ai-config/{household_id}/plugins`
- `PUT /api/v1/ai-config/{household_id}/plugins/{plugin_id}/state`
- `GET /api/v1/ai-config/{household_id}/plugin-mounts`
- `POST /api/v1/ai-config/{household_id}/plugin-mounts`

### 模型发现

- `POST /api/v1/ai-config/provider-adapters/{adapter_code}/discover-models`

前端应该只消费 `/provider-adapters` 返回的展开后 adapter 数据，不要再从原始 registry 重建一份 provider adapter。

## 最后提醒

如果一个新 provider 要联调成功，必须靠宿主和前后端各自打三层补丁，那不是联调难，而是对接方式本身就已经烂了。
