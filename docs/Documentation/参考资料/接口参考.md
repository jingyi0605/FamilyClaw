# 接口参考

> 这页现在先承接项目里最容易被前后端一起改坏的关键接口约定，后续再继续补全。

## 这页要解决什么

- 统一接口参考入口，避免接口说明分散在多个历史文档里
- 把“家庭初始化状态”和“成员级导览状态”这种容易混淆的接口边界写清楚
- 把运行时插件安装这类容易被“是否需要重启”说乱的接口语义收口

## 运行时插件安装接口

这组接口当前已经有正式后端实现，`user-app` 也已经补上 ZIP 上传入口。

### 手动 ZIP 安装插件

- 方法：`POST /api/v1/ai-config/{household_id}/plugin-packages`
- 认证：管理员
- 请求格式：`multipart/form-data`
- 表单字段：
  - `file`：本地 ZIP 包
  - `overwrite`：是否允许覆盖同 `plugin_id` 的现有第三方手动安装
- 当前返回的关键字段：
  - `plugin_id`
  - `plugin_name`
  - `version`
  - `previous_version`
  - `install_action`：`installed` / `upgraded` / `reinstalled`
  - `overwritten`
  - `enabled`
  - `source_type`：当前固定为 `third_party`
  - `execution_backend`：当前固定为 `subprocess_runner`
  - `plugin_root`
  - `manifest_path`
  - `message`
- 运行时语义：
  - 安装成功后，无需重启后端服务，刷新插件列表就能看到当前挂载
  - 首次手动 ZIP 安装默认不自动启用
  - 覆盖升级也不会把“立即生效”解释成 Python 模块热替换

### 当前错误边界

- `plugin_package_invalid`：不是 ZIP、空包、包结构不合法，或者 manifest 校验失败
- `plugin_package_conflict`：同 `plugin_id` 已存在，但这次没有显式传 `overwrite=true`
- `plugin_id_conflict`：上传包的 `plugin_id` 和内置插件冲突
- `plugin_source_mismatch`：当前插件由插件市场管理，或现有来源不是第三方手动安装，不能直接用 ZIP 覆盖

## 插件市场运行时语义

- 市场安装成功后，插件会直接进入“已安装”状态，不需要重启后端服务才能被列表和后续链路识别。
- 市场安装默认不自动启用，且配置未完成时不能启用。
- 市场插件启用后，后续执行直接按当前挂载生效。
- 市场升级和回滚会直接更新当前挂载；已经在跑的旧任务不会被强行热切换。

## 成员级导览状态接口

这组接口服务 `Spec 001.9 - 初始化完成后跨端新手向导` 的阶段 1。

它只解决一件事：记录“当前成员已经完成了哪个版本的 user-app 新手导览”。

### 数据落点

- 表：`member_preferences`
- 字段：`user_app_guide_version`
- 语义：
  - `null`：这个成员还没完成任何一版导览
  - `>= 1`：这个成员已经完成对应版本号的导览
- 注意：
  - 这是成员级状态，不是家庭级状态
  - 不允许继续污染 `household.setup_status`

### 读取当前成员导览状态

- 方法：`GET /api/v1/member-preferences/{member_id}/guide-status`
- 认证：必须是已绑定成员账号；同家庭成员可以读取
- 返回示例：

```json
{
  "member_id": "member-001",
  "user_app_guide_version": 1,
  "updated_at": "2026-03-19T23:20:00Z"
}
```

### 回写当前成员导览状态

- 方法：`PUT /api/v1/member-preferences/{member_id}/guide-status`
- 认证：默认只允许当前成员本人回写自己的状态；系统账号可例外
- 请求体：

```json
{
  "user_app_guide_version": 1
}
```

- 规则：
  - 版本号必须是正整数
  - 版本号只能前进，不能回退
  - 成员不能回写别人的导览状态

## 启动摘要里的导览状态

`packages/user-core` 的启动快照现在会额外返回当前登录成员的 `guideStatus`。

这样做的目的很简单：

- 前端 runtime 启动时能低成本拿到成员导览版本
- 后续自动触发新手导览时，不用再把这件事塞回 `household.setup_status`

## 前端触发与恢复口径

这部分不是后端接口，但前后端联调时最容易搞混，所以一起写在这里。

- 初始化完成页点击“进入应用”前，前端会先写一个一次性本地触发标记
- 应用启动后，前端同时看两件事：
  - 本地是否存在“刚完成初始化”标记
  - 当前成员的 `guideStatus.user_app_guide_version` 是否低于当前导览版本
- 只有两件事都满足，才会自动拉起导览
- 导览进行中，前端会把最近一步写到本地 checkpoint
- 如果用户刷新页面或切页回来，前端会先尝试从 checkpoint 恢复最近一步
- 用户跳过或完成时，前端会：
  - 回写当前版本号到成员导览状态接口
  - 清理本地一次性触发标记
  - 清理本地 checkpoint

## 错误语义

- `401 authentication required`：未登录
- `403 cannot access another household`：跨家庭访问
- `403 cannot update another member guide status`：成员试图改别人的导览状态
- `409 guide version cannot move backward`：试图把版本号回退

## 完成标准

- 前后端能分清家庭初始化状态和成员导览状态不是一回事
- 新手导览相关接口的权限、字段和回退规则有明确口径
