# 20260318-现有插件系统迁移路线图

状态：Active

## 这份路线图要解决什么问题

插件系统不能靠一夜重写，但也不能继续拖着错误边界往前走。

FamilyClaw 当前还没有正式发布，所以这次迁移路线不再为“旧插件兼容”留长期过渡层。目标很明确：

- 直接把现有仓库里的插件全部迁到新架构
- 直接删除错误类型和错误边界
- 直接把天气和记忆都纳入新平台设计

这份路线图关注的是“怎么平稳改完整个仓库”，不是“怎么兼容一个已经对外发布的旧生态”。

## 当前迁移进度快照

到 2026-03-18 为止，这条迁移路线已经不是纯计划了，关键主链路已经落地：

- 插件类型体系已经按 V1 收口，`connector`、`memory-ingestor` 不再是正式类型语义
- `integration`、`integration_instance`、统一配置作用域和统一 DTO 已经落地
- 官方天气已经从宿主领域模块迁到正式插件链路
- 宿主天气专用 API 和核心天气模块已经移除
- 插件卡片、配置、实例同步和错误语义已经按统一宿主链路承接
- 宿主公共标准实体承载层已经落地，设备读取、控制和快捷指令开始统一消费标准实体

这份路线图后面的作用，不再是“提醒团队将来要做什么”，而是：

- 记录这次迁移到底收到了哪里
- 标出已经完成的边界
- 标出后续继续演进时不能退回去的地方

## 当前系统盘点

### 已经完成或接近最终形态的部分

- 插件 manifest 和注册表
- 插件启用/禁用统一规则
- 插件执行链路和 worker
- 配置 schema 和 secret 存储
- 插件卡片 snapshot
- `integration_instances` / `integration_discoveries` 主链路
- `integration_instance` 配置作用域
- `integration.refresh` 统一输出口径
- 官方天气插件的默认实例自动创建、多实例地区绑定和卡片接入

### 已经明确收口、不允许回退的部分

- 不再新增 `connector` 语义
- 不再新增 `memory-ingestor` 语义
- 不再回到“宿主里单独做天气、电费、步数”这种领域模块做法
- 不再允许插件主链路绕开 `integration_instance -> device -> entity -> dashboard card`
- 不再允许卡片按 `city` 文本绑天气，必须绑标准设备

### 还要继续盯住的剩余边界

- `health-basic-reader`、`homeassistant`、`open-xiaoai-speaker` 还要继续按同一口径做深度收口
- 记忆中心虽然已经定了槽位边界，但后续演进仍要防止重新把主权外包出去
- 文档与代码每次改动后都要同步，不允许再出现“代码已经走新链路，手册还在写旧字段”的情况
- 插件私有持久化 / 插件私有迁移机制已经收口到宿主统一入口 + 插件私有 Alembic runner；天气这类私有绑定表不再挂在宿主核心 Alembic 历史上
- `official` 目录是运行时托管目录，不属于宿主源码编译产物；宿主必须允许该目录缺失时照常导入、迁移和启动

### 当前内置插件分类

现有仓库里的内置插件，大致可以归到下面几类：

- `channel`
  - `channel-dingtalk`
  - `channel-discord`
  - `channel-feishu`
  - `channel-telegram`
  - `channel-wecom-app`
  - `channel-wecom-bot`
- 已迁到新状态型插件主链路
  - `official-weather`
- 继续按新口径收口的状态型插件
  - `health-basic-reader`
  - `homeassistant`
  - `open-xiaoai-speaker`
- 资源包
  - `locale-zh-tw`
- 宿主记忆实现
  - 当前边界已经定为“宿主内核 + 槽位型插件”，后续按这个方向继续收口

## 迁移目标

迁移完成后，平台应达到以下状态：

- 所有状态型插件统一使用 `integration`
- 宿主不再新增单领域业务模块
- 所有状态型能力都通过“插件 -> 实例 -> 设备 -> 实体 -> 卡片/动作”接入
- 记忆中心被拆成“宿主内核 + 独占槽位插件”
- 仓库内不再保留旧 `connector`、旧 `memory-ingestor` 作为主设计语义

## 迁移阶段与实际结果

### 阶段 0：冻结错误方向

目标：

- 先停掉继续把新领域能力写进宿主的做法

具体动作：

- 停止新增 `weather_service`、`power_service` 这类宿主领域模块
- 新能力默认先走插件评审
- 新插件设计必须对照《插件能力与接口规范 v1》

完成标志：

- 后续新 Spec 不再把第三方领域逻辑默认写进宿主

当前结果：

- 已完成，后续插件相关 Spec 以 V1 规范和槽位边界为准

### 阶段 1：补平台最小缺口

目标：

- 先把平台层缺的关键接口补齐

要做的事：

1. 新增一等插件类型 `integration`
2. 新增独占槽位 `memory_engine`、`memory_provider`、`context_engine`
3. 补齐配置作用域 `integration_instance`
4. 新增统一设备 DTO
5. 新增统一实体状态 DTO
6. 新增 `integration.refresh` 标准输出
7. 新增统一卡片设备绑定协议
8. 明确槽位型接口契约

这一阶段明确不做：

- 不立即细化所有业务插件实现
- 不先做市场化和远程仓库

完成标志：

- 平台能承接新的 `integration` 插件
- 平台能承接新的记忆槽位插件

当前结果：

- 已完成，类型体系、配置作用域、标准 DTO 和槽位边界已经进入正式文档与宿主代码

### 阶段 2：全量改造现有插件

目标：

- 直接把仓库里的现有插件全部改成新架构

要做的事：

1. 移除 `connector` 作为主插件类型
2. 移除 `memory-ingestor` 作为主插件类型
3. 逐个修改现有插件 manifest、入口和数据输出
4. 把所有状态型插件统一接到 `integration` 运行时

完成标志：

- 仓库内不再有“按旧语义设计”的内置插件

当前结果：

- 主体已完成，正式类型语义已经统一
- 后续剩下的是按同一口径继续打磨个别插件细节，而不是再保留旧架构并存

### 阶段 3：迁天气，作为第一个 integration 样板

为什么先迁天气：

- 纯状态型
- 低风险
- 不依赖复杂控制动作
- 最适合验证“实例、子项、设备、实体、卡片、缓存、降级”这整套设计

要做的事：

1. 去掉宿主对天气的专门分支
2. 改成标准 `integration`
3. 默认天气设备改成默认实例下的虚拟家庭设备
4. 附加地区天气改成额外实例绑定指定地区节点
5. 卡片只绑定设备，不绑定城市文本

完成标志：

- 天气主链路不再依赖宿主领域模块

当前结果：

- 已完成
- 宿主天气专用 API 已删除
- 宿主核心天气模块已移除主源码引用
- 宿主对 `official-weather` 的静态 import 依赖和设备读取链路里的天气专用修补已经移除
- 宿主插件执行准备阶段会先补齐 `builtin` / `official` 插件的私有 migration，再进入正式执行链路
- 官方天气插件当前走的是：
  - `plugin -> integration instance -> device -> entities -> dashboard card`
- 默认天气实例启用后自动创建
- 多地区天气通过多个 `integration_instance` 建模，而不是回到宿主特判
- 默认源是 MET Norway，OpenWeather / WeatherAPI 作为插件配置里的可选 provider
- 卡片通过设备和卡片快照绑定，不再按城市文本走主链路
- 天气私有绑定表已经迁出宿主核心 Alembic
- 运行态标准实体已经迁到宿主公共实体承载层，`DeviceBinding.capabilities` 只保留绑定摘要和轻量元数据

### 阶段 4：迁其他状态型插件

顺序建议：

1. `official-weather`
2. `health-basic-reader`
3. `homeassistant`
4. `open-xiaoai-speaker`

原因：

- 先简后难
- 先无动作后有动作
- 先纯状态后复杂同步

当前结果：

- 顺序仍然成立
- 但这已经不是“先证明架构可行”的阶段，而是“按已经成立的架构继续清理插件实现细节”的阶段

### 阶段 5：记忆中心拆核与槽位化

目标：

- 把当前宿主记忆模块拆成宿主内核和可替换槽位

要做的事：

1. 抽出 `context_engine`
2. 抽出 `memory_engine`
3. 抽出 `memory_provider`
4. 宿主保留记忆标准、可见性、修订、人工修正和 fallback

顺序必须固定：

1. 先 `context_engine`
2. 再 `memory_engine`
3. 最后 `memory_provider`

原因：

- 先动上下文组装，对现有表结构冲击最小
- 再动记忆提取逻辑
- 最后才动外部后端，风险最高

完成标志：

- 记忆中心的主权仍在宿主
- 记忆算法和后端已经可插件化替换

当前结果：

- 边界已经正式定稿到《记忆中心内核与插件边界规范》
- 后续实现必须继续遵守“宿主保留对象标准、权限、修订、审计和 fallback；插件只接引擎、provider、context 槽位”

### 阶段 6：清理旧债

要做的事：

- 清理宿主里的领域特判
- 删除旧 `connector` 分支
- 删除旧 `memory-ingestor` 语义
- 收口过时 manifest 字段
- 收口旧式状态存储

完成标志：

- 宿主只保留平台内核
- 新架构成为唯一架构

当前结果：

- 宿主天气专用分支已经作为第一批旧债清掉
- 后续凡是发现“领域专线”“页面特判”“旧字段名继续当正式契约”这类残留，都按旧债处理，不再留长期兼容层
- 插件私有表如果还挂在宿主核心 Alembic 上，也按旧债处理，不允许在新规范下继续扩散

## 这份路线图现在怎么用

从现在开始，这份文档主要回答三件事：

1. 哪些边界已经正式定了，不能退回去
2. 哪些插件已经迁完，哪些还要继续收口
3. 后续新能力应该按什么路径进系统，而不是重新写进宿主

如果后续代码实现和这份路线图冲突，以正式规范和已经落地的统一主链路为准，不再给旧思路开回头路。

## 风险与控制

### 风险 1：边迁移边继续长宿主

这是最大风险。

如果平台一边定规范，一边还在宿主里新增领域逻辑，那迁移永远收不回来。

控制办法：

- 新增领域能力必须先过边界评审

### 风险 2：迁移不彻底，旧语义继续留存

如果 `connector`、旧记忆语义、宿主特判还继续保留，最后只会形成两套系统并存。

控制办法：

- 直接全量改造现有内置插件
- 迁移完成后立即删旧分支

### 风险 3：把记忆主权一起外包出去

这是比天气更危险的坑。

如果把记忆权限、修订、fallback 一起交给插件，系统会失去可控性。

控制办法：

- 宿主保留记忆标准、修订、可见性和人工修正
- 插件只接记忆引擎、后端和上下文槽位

### 风险 4：一次性大爆炸改造

这是最蠢的做法。

控制办法：

- 先补平台能力
- 先迁天气样板
- 再迁更复杂插件
- 记忆中心按固定顺序拆槽位

## 验收标准

迁移路线推进过程中，每个阶段至少要回答这几个问题：

1. 这一步建出来的能力是什么？
2. 哪些旧设计已经被真正删除？
3. 哪些宿主特判已经可以删掉？
4. 哪些现有插件已经被迁到新架构？
5. 记忆主权是否仍然留在宿主？

只有这些问题答得清，这份路线图才不是空话。
