# 20260318-现有插件系统迁移路线图

状态：Draft

## 这份路线图要解决什么问题

插件系统不能靠一夜重写，但也不能继续拖着错误边界往前走。

FamilyClaw 当前还没有正式发布，所以这次迁移路线不再为“旧插件兼容”留长期过渡层。目标很明确：

- 直接把现有仓库里的插件全部迁到新架构
- 直接删除错误类型和错误边界
- 直接把天气和记忆都纳入新平台设计

这份路线图关注的是“怎么平稳改完整个仓库”，不是“怎么兼容一个已经对外发布的旧生态”。

## 当前系统盘点

### 已经接近最终形态的部分

- 插件 manifest 和注册表
- 插件启用/禁用统一规则
- 插件执行链路和 worker
- 配置 schema 和 secret 存储
- 插件卡片 snapshot
- `integration_instances` / `integration_discoveries` 骨架

### 明显需要重构的部分

- `connector` 语义太宽，既像集成又像执行器
- 缺少真正的一等 `integration` 类型
- 缺少 `integration_instance` 配置作用域
- 设备状态和实体状态没有统一标准 DTO
- `DeviceBinding.capabilities` 混了声明和运行态
- 宿主里仍有天气等领域专用分支
- 记忆中心还没有按“宿主内核 + 插件槽位”拆开

### 当前内置插件分类

现有仓库里的内置插件，大致可以归到下面几类：

- `channel`
  - `channel-dingtalk`
  - `channel-discord`
  - `channel-feishu`
  - `channel-telegram`
  - `channel-wecom-app`
  - `channel-wecom-bot`
- 待迁移为新状态型插件
  - `official-weather`
  - `health-basic-reader`
  - `homeassistant`
  - `open-xiaoai-speaker`
- 资源包
  - `locale-zh-tw`
- 宿主内存实现
  - 当前 `memory` 模块后续要拆成宿主内核 + 槽位型插件实现

## 迁移目标

迁移完成后，平台应达到以下状态：

- 所有状态型插件统一使用 `integration`
- 宿主不再新增单领域业务模块
- 所有状态型能力都通过“插件 -> 实例 -> 设备 -> 实体 -> 卡片/动作”接入
- 记忆中心被拆成“宿主内核 + 独占槽位插件”
- 仓库内不再保留旧 `connector`、旧 `memory-ingestor` 作为主设计语义

## 迁移阶段

### 阶段 0：冻结错误方向

目标：

- 先停掉继续把新领域能力写进宿主的做法

具体动作：

- 停止新增 `weather_service`、`power_service` 这类宿主领域模块
- 新能力默认先走插件评审
- 新插件设计必须对照《插件能力与接口规范 v1》

完成标志：

- 后续新 Spec 不再把第三方领域逻辑默认写进宿主

### 阶段 1：补平台最小缺口

目标：

- 先把平台层缺的关键接口补齐

要做的事：

1. 新增一等插件类型 `integration`
2. 新增独占槽位 `memory_engine`、`memory_provider`、`context_engine`
3. 补齐配置作用域 `integration_instance`
4. 新增统一设备 DTO
5. 新增统一实体状态 DTO
6. 新增 `integration.refresh` 标准输出
7. 新增统一卡片设备绑定协议
8. 明确槽位型接口契约

这一阶段明确不做：

- 不立即细化所有业务插件实现
- 不先做市场化和远程仓库

完成标志：

- 平台能承接新的 `integration` 插件
- 平台能承接新的记忆槽位插件

### 阶段 2：全量改造现有插件

目标：

- 直接把仓库里的现有插件全部改成新架构

要做的事：

1. 移除 `connector` 作为主插件类型
2. 移除 `memory-ingestor` 作为主插件类型
3. 逐个修改现有插件 manifest、入口和数据输出
4. 把所有状态型插件统一接到 `integration` 运行时

完成标志：

- 仓库内不再有“按旧语义设计”的内置插件

### 阶段 3：迁天气，作为第一个 integration 样板

为什么先迁天气：

- 纯状态型
- 低风险
- 不依赖复杂控制动作
- 最适合验证“实例、子项、设备、实体、卡片、缓存、降级”这整套设计

要做的事：

1. 去掉宿主对天气的专门分支
2. 改成标准 `integration`
3. 默认天气设备改成实例下的虚拟家庭设备
4. 多地区天气改成同一实例下的附加位置设备
5. 卡片只绑定设备，不绑定城市文本

完成标志：

- 天气主链路不再依赖宿主领域模块

### 阶段 4：迁其他状态型插件

顺序建议：

1. `official-weather`
2. `health-basic-reader`
3. `homeassistant`
4. `open-xiaoai-speaker`

原因：

- 先简后难
- 先无动作后有动作
- 先纯状态后复杂同步

### 阶段 5：记忆中心拆核与槽位化

目标：

- 把当前宿主记忆模块拆成宿主内核和可替换槽位

要做的事：

1. 抽出 `context_engine`
2. 抽出 `memory_engine`
3. 抽出 `memory_provider`
4. 宿主保留记忆标准、可见性、修订、人工修正和 fallback

顺序必须固定：

1. 先 `context_engine`
2. 再 `memory_engine`
3. 最后 `memory_provider`

原因：

- 先动上下文组装，对现有表结构冲击最小
- 再动记忆提取逻辑
- 最后才动外部后端，风险最高

完成标志：

- 记忆中心的主权仍在宿主
- 记忆算法和后端已经可插件化替换

### 阶段 6：清理旧债

要做的事：

- 清理宿主里的领域特判
- 删除旧 `connector` 分支
- 删除旧 `memory-ingestor` 语义
- 收口过时 manifest 字段
- 收口旧式状态存储

完成标志：

- 宿主只保留平台内核
- 新架构成为唯一架构

## 对现有插件的具体处理建议

### `official-weather`

现状：

- 逻辑已经部分插件化
- 但首页卡片和状态链路仍然有宿主专门分支

目标：

- 迁成标准 `integration`
- 作为新架构样板

### `health-basic-reader`

现状：

- 既像 connector，又像 memory ingestor

目标：

- 迁成 `integration`
- 通过宿主标准记忆写入接口输出观察类记忆，不再保留旧 `memory-ingestor` 语义

### `homeassistant`

现状：

- 同时承担同步、控制、记忆写入

目标：

- 迁成 `integration + action`
- 记忆相关输出改走宿主统一记忆接口

### `open-xiaoai-speaker`

现状：

- 已经有集成实例和设备级配置意识

目标：

- 迁成 `integration + action`

### `channel-*`

现状：

- 类型边界已经比较清楚

目标：

- 保持 `channel`
- 统一按新 manifest 和权限规范收口

### `locale-zh-tw`

现状：

- 已接近最终形态

目标：

- 继续保留为 `locale-pack`

### 当前宿主 `memory` 模块

现状：

- 混合了事件沉淀、记忆抽取、权限过滤、上下文组装

目标：

- 宿主保留 `EventRecord`、`MemoryCard`、`MemoryCardRevision`、可见性过滤、人工修正
- 把记忆提取、检索后端、上下文组装抽成独占槽位插件

## 风险与控制

### 风险 1：边迁移边继续长宿主

这是最大风险。

如果平台一边定规范，一边还在宿主里新增领域逻辑，那迁移永远收不回来。

控制办法：

- 新增领域能力必须先过边界评审

### 风险 2：迁移不彻底，旧语义继续留存

如果 `connector`、旧记忆语义、宿主特判还继续保留，最后只会形成两套系统并存。

控制办法：

- 直接全量改造现有内置插件
- 迁移完成后立即删旧分支

### 风险 3：把记忆主权一起外包出去

这是比天气更危险的坑。

如果把记忆权限、修订、fallback 一起交给插件，系统会失去可控性。

控制办法：

- 宿主保留记忆标准、修订、可见性和人工修正
- 插件只接记忆引擎、后端和上下文槽位

### 风险 4：一次性大爆炸改造

这是最蠢的做法。

控制办法：

- 先补平台能力
- 先迁天气样板
- 再迁更复杂插件
- 记忆中心按固定顺序拆槽位

## 验收标准

迁移路线推进过程中，每个阶段至少要回答这几个问题：

1. 这一步建出来的能力是什么？
2. 哪些旧设计已经被真正删除？
3. 哪些宿主特判已经可以删掉？
4. 哪些现有插件已经被迁到新架构？
5. 记忆主权是否仍然留在宿主？

只有这些问题答得清，这份路线图才不是空话。
