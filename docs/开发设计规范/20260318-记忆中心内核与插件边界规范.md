# 20260318-记忆中心内核与插件边界规范

状态：Active

## 这份规范解决什么问题

记忆中心比天气、电费、设备状态更敏感。

因为它不只是“接第三方数据”，它还直接影响：

- 对话上下文
- 家庭隐私
- 可见性控制
- 审计与修订历史
- 用户对系统是否可信的判断

所以记忆不能简单分成“全宿主”或者“全插件”。

这份规范的核心结论是：

宿主保留记忆内核，插件接管记忆引擎。

这不是一个“以后再讨论”的方向文档，而是当前插件系统 V1 的正式边界。

后续无论是内置记忆能力、第三方记忆后端，还是类似 lossless 的上下文插件，都必须按这条边界实现：

- 宿主保留主权
- 插件接算法和后端
- 失败时由宿主兜底

## 核心判断

### 宿主必须保留的部分

- 记忆对象标准
- 权限与可见性规则
- 审计与修订历史
- 最低可用的默认记忆实现
- 对话侧统一读取接口
- 插件失败时的降级逻辑

### 插件可以接管的部分

- 事件转记忆的提取策略
- 长期记忆索引
- 向量检索和召回排序
- 记忆压缩和摘要
- 外部记忆存储适配
- 替代式上下文引擎

一句话版本：

记忆主权在宿主，记忆算法和后端可以插件化。

## 为什么不能把整个记忆中心都交给插件

如果全部交出去，会立刻踩中这些坑：

- 权限判定不可信
- 插件一挂，对话主链路就失忆
- 修订历史不可追踪
- 用户无法确认某条记忆是谁写的、谁改的、为什么改

这不是理论问题，是一定会发生的问题。

## 记忆中心的三层结构

### 第一层：记忆内核

由宿主负责。

职责：

- 记忆对象标准
- 记忆 ID 和归属关系
- household/member/room 关联
- visibility 和权限判断
- revision 记录
- 人工修正、失效、删除
- 最低可用 fallback 检索

### 第二层：记忆引擎

可由插件替换。

职责：

- 从事件中抽取记忆候选
- 去重策略
- 合并策略
- 重要度和置信度估计
- 长短期记忆划分

### 第三层：上下文引擎

可由插件替换。

职责：

- 从记忆中心检索与当前对话相关的内容
- 选择哪些记忆进入 prompt
- 做压缩和摘要
- 输出最终上下文 bundle

## FamilyClaw 当前记忆模块的职责拆分建议

按当前仓库来看，记忆模块大致混了四块事：

- 事件事实沉淀
- 记忆抽取与归一
- 权限和可见性
- 对话上下文组装

建议这样拆：

### 保留在宿主

- `EventRecord`
- `ConversationSessionSummary`
- `EpisodicMemoryEntry`
- `MemoryCard`
- `MemoryCardRevision`
- `EpisodicMemoryEntryRevision`
- `KnowledgeDocument`
- `KnowledgeDocumentRevision`
- `MemoryRecallDocument`
- 可见性和权限过滤
- 手工创建、修正、失效、删除
- 默认上下文读取入口

### 抽成可替换槽位

- 事件转记忆策略
- 记忆检索策略
- 对话上下文组装策略

## 记忆插件的类型与槽位

### 插槽，而不是随意扩展点

记忆能力不能做成“谁都能随便接一脚”的普通插件入口。

它应该是平台级独占槽位。

规范 v1 定义三个槽位：

- `memory_engine`
- `memory_provider`
- `context_engine`

### `memory_engine`

负责写入和抽取。

典型能力：

- `ingest_event`
- `derive_memory_candidates`
- `merge_memory`
- `score_importance`

### `memory_provider`

负责存储、索引和检索。

典型能力：

- `upsert_memory`
- `query_memory`
- `delete_memory`
- `rebuild_index`

### `context_engine`

负责把记忆变成对话上下文。

典型能力：

- `build_context_bundle`
- `summarize_memory_hits`
- `compress_context`

## 当前代码状态

这部分不是“将来可能怎么做”，而是 2026-03-19 这版代码已经落地到哪里。

### 已经真正落地的部分

- 宿主已经正式拆出四层真相源：
  - `conversation_session_summaries` 对应 `L1`
  - `episodic_memory_entries` 对应 `L2`
  - `memory_cards` 对应 `L3`
  - `knowledge_documents` 对应 `L4`
- 宿主已经正式落地统一召回投影 `memory_recall_documents`，负责把 `L1/L2/L3/L4` 同步成统一 recall 文档。
- `free_chat` 已经不再只吃 `hot_summary`，当前按 `[session_summary] / [stable_facts] / [recent_events] / [external_knowledge]` 四组注入真实 recall hits。
- 注入 trace 已经统一落在 `conversation_memory_reads`，可以回放 `memory_id / source_id / source_kind / group / layer / score / rank`。
- `context_engine` 槽位已经真正接入 `build_memory_context_bundle(...)` 主链路。槽位插件执行失败、输出不合法或存在多个启用插件时，会直接回退宿主默认实现。
- `L2` 和 `L4` 现在也有宿主 revision 表，不再只有 `L3 memory_cards` 能追修订：
  - `episodic_memory_entry_revisions`
  - `knowledge_document_revisions`
- 插件 observation 导入当前是宿主双写：
  - 兼容旧链路，继续写 `L3 memory_cards`
  - 同时写 `L4 knowledge_documents`
  这不是漂亮设计，只是为兼容现有接口和测试保留的过渡策略。

### 还没有真正插件化的部分

- `memory_engine`
  - 现在已经有正式槽位类型和 manifest/schema 定义
  - 但主写入链路还没有切成“优先调槽位插件，失败回宿主实现”
- `memory_provider`
  - 现在也只有槽位定义
  - 主检索和主投影仍由宿主 PostgreSQL 实现承担
- 换句话说，当前真正落地到主链路的只有 `context_engine` 槽位；
  `memory_engine / memory_provider` 还是预留边界，不要在文档里吹成已经完成。

## 标准记忆接口

宿主对外只暴露统一接口，不允许插件自己定义一套完全不同的 memory API。

### 写入接口

```json
{
  "kind": "event",
  "household_id": "xxx",
  "subject_member_id": "yyy",
  "payload": {}
}
```

宿主统一入口：

- `write_memory_event`
- `write_memory_fact`
- `write_memory_observation`

### 查询接口

```json
{
  "household_id": "xxx",
  "query": "最近孩子睡眠怎么样",
  "member_scope": "child-1",
  "actor": {
    "type": "member",
    "id": "member-1"
  }
}
```

宿主统一入口：

- `query_memory`

### 上下文接口

```json
{
  "household_id": "xxx",
  "session_id": "sess-1",
  "actor": {
    "type": "member",
    "id": "member-1"
  },
  "message": "明天上学要带什么"
}
```

宿主统一入口：

- `build_memory_context`

### 修正接口

宿主统一保留：

- `correct_memory`
- `invalidate_memory`
- `delete_memory`
- `list_memory_revisions`

这些不能交给插件独占。

## 标准数据模型

### 记忆主记录

宿主必须保留以下最小主记录：

- `memory_id`
- `household_id`
- `memory_type`
- `visibility`
- `status`
- `subject_member_id`
- `source_plugin_id`
- `source_ref`
- `created_at`
- `updated_at`

### 修订记录

宿主必须保留：

- `revision_no`
- `action`
- `actor_type`
- `actor_id`
- `before`
- `after`
- `reason`

当前宿主已落地三类修订记录：

- `memory_card_revisions`
- `episodic_memory_entry_revisions`
- `knowledge_document_revisions`

### 可见性规则

可见性判定只能由宿主执行。

至少支持：

- `public`
- `family`
- `private`
- `sensitive`

插件可以建议可见性，但不能最终拍板。

## 外部记忆后端的边界

### 可以支持

可以支持：

- pgvector
- LanceDB
- lossless 一类外部记忆系统
- 其他专用索引服务

### 不能放权过度

即使使用外部后端，宿主也必须保留最小影子记录：

- 记忆归属
- 可见性
- 审计元信息
- 最近一次同步状态
- fallback 摘要

否则第三方服务挂掉，FamilyClaw 会直接失忆。

## 降级规则

### 记忆引擎失败

处理方式：

- 保留事件记录
- 标记“待提取”或“提取失败”
- 不影响宿主最小记忆读写能力

### 记忆 provider 失败

处理方式：

- 回退到宿主默认 provider
- 至少保留最近热记忆摘要
- 不让对话主链路完全空掉

### context engine 失败

处理方式：

- 回退到宿主默认 context engine
- 减少上下文质量，但不允许中断对话

## 权限和安全边界

### 插件无权直接决定谁能看哪条记忆

这是硬规则。

插件可以建议：

- 重要度
- 置信度
- 可见性默认值

但最终裁决权在宿主。

### 插件无权绕过修订历史

任何记忆变更都必须留下 revision。

不允许插件直接覆盖而不留痕。

### 插件无权删除宿主审计元数据

哪怕外部 provider 要求删除原文，宿主也必须保留最小审计影子。

## 平台默认实现

为了保证系统始终可用，宿主必须自带默认实现：

- `builtin_memory_engine`
- `builtin_memory_provider`
- `builtin_context_engine`

当前代码状态是：

- `builtin_context_engine` 已经真正在主链路兜底
- `builtin_memory_engine` 和 `builtin_memory_provider` 仍由宿主当前实现承担，但还没有被独立成“先调槽位、再回退”的正式切换层

第三方插件是增强，不是生命线。

这条规则和插件系统 V1 是绑定的：

- 可以拔掉第三方记忆插件
- 不能拔掉宿主最小记忆能力

## 演进建议

### 第一步

先插件化 `context_engine`。

原因：

- 对现有数据模型冲击最小
- 最适合先验证 lossless 一类插件

当前状态：这一步已经落地，而且已经接进 `build_memory_context_bundle(...)` 主链路。

### 第二步

再插件化 `memory_engine`。

把事件提取、归一、去重从宿主服务中抽出来。

当前状态：只完成了槽位定义，主链路还没切。

### 第三步

最后开放 `memory_provider`。

这一步风险最大，必须等前两步稳定后再做。

当前状态：只完成了槽位定义，统一 recall 仍然由宿主 PostgreSQL 路径承担。

## 验收标准

记忆插件化完成后，至少要满足：

1. 关闭第三方记忆插件后，系统仍有最小可用记忆能力
2. 任何记忆修正都能追溯
3. 任何记忆查询都要先过宿主权限过滤
4. 对话主链路不会因为第三方记忆插件失败而中断
5. `L1/L2/L3/L4` 真相源和 `memory_recall_documents` 投影边界清楚，不允许混表偷省事
6. `context_engine` 插件失败时，宿主默认上下文组装必须能继续回答
7. 插件 observation 不允许越权写入别的家庭成员，宿主必须校验 `subject_member_id` 归属

满足不了这四条，就说明边界切错了。
