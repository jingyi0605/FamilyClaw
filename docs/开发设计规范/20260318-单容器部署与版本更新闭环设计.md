# 单容器部署与版本更新闭环设计

状态：Draft

## 1. 这份文档解决什么问题

FamilyClaw 计划主要面向普通家庭用户运行在 NAS、迷你主机和家用服务器上。

这类用户最怕三件事：

1. 部署时要同时管理很多容器
2. 升级时搞不清该拉哪个镜像、会不会坏库
3. 出问题时不知道当前到底跑的是哪个版本

所以这里不讨论“理论上最优雅”的多服务拆分，而是确定一套更实用的方案：

- 对外交付一个单 Docker 镜像
- 运行时只要求用户挂载一个数据目录
- 发布依赖 GitHub Release 和 GHCR，不额外部署发布服务器
- 升级时先备份、再迁移、再启动，失败时能回退
- 版本信息在运行时、发布页和镜像内保持一致

一句话说清楚：让普通用户只需要理解“安装、升级、回退”三件事。

## 2. 核心判断

### 2.1 值得做

值得做。

这是一个真实问题，不是过度设计。当前项目包含数据库、后端、gateway、前端和反向代理。如果把这些东西按工程师习惯拆成多个服务，维护成本会立刻转嫁给最终用户。对于家用 NAS 场景，这就是坏设计。

### 2.2 不值得做的事情

下面这些事情不建议做：

- 不建议再发明一套独立的“数据库版本号 051、052、053”作为升级主依据
- 不建议让应用自己偷偷执行镜像拉取和容器自更新
- 不建议让数据库初始化、迁移和业务服务启动并发进行
- 不建议要求普通用户理解 compose 里多个服务之间的依赖关系

原因很简单：

- 数据库 schema 的真相已经由 Alembic revision 维护
- 镜像更新应该由 Docker 宿主机负责，不是容器里的业务程序负责
- 启动顺序混乱会直接把数据搞坏
- 用户只需要一个稳定入口，不需要一套“云原生课程”

## 3. 设计目标

本方案要达成以下目标：

1. 对外只发布一个正式运行镜像
2. 用户只需要挂载一个数据目录 `/data`
3. 镜像内包含 PostgreSQL、api-server、open-xiaoai-gateway、Nginx 和前端静态资源
4. 启动时固定执行“数据库就绪 -> Alembic 迁移 -> 后端启动 -> gateway 启动 -> Nginx 启动”
5. 运行中的应用能够明确报告当前版本、发布日期、更新说明链接和升级状态
6. 发布流程完全依赖 GitHub Release、GitHub Actions 和 GHCR
7. 升级前自动备份，升级失败时保留回退路径

## 4. 单容器总体方案

### 4.1 容器内包含什么

单容器镜像内包含以下组件：

- PostgreSQL
- FastAPI 后端 `api-server`
- `open-xiaoai-gateway`
- Nginx
- `user-app` 编译后的静态文件
- Alembic migrations
- 发布版本清单 `release-manifest.json`
- 启动与诊断脚本

这里说的“单容器”不是“单进程”，而是“一个镜像、一个容器、一个数据卷、一个对外入口”。

### 4.2 推荐的容器进程管理

不建议用一段很长的 bash 脚本硬扛所有进程生命周期。那种东西前期看起来省事，后面一定会变成垃圾。

推荐使用下面任一方式作为 PID 1：

- `s6-overlay`
- `supervisord`

推荐优先使用 `s6-overlay`，原因是：

- 多进程监管清晰
- 服务依赖关系更容易表达
- 容器退出语义更稳定
- 健康检查和日志转发更容易收口

### 4.3 对外端口

建议只保留尽量少的端口：

- `8080`：Nginx 对外入口，统一提供前端和 API
- `4399`：仅当语音网关需要被外部设备直接访问时暴露

普通 Web 使用场景下，用户只需要记住一个访问地址。

## 5. 容器内的职责拆分

### 5.1 PostgreSQL

职责：

- 提供唯一运行时数据库
- 使用挂载卷持久化数据
- 供 Alembic 和业务服务共用

要求：

- 数据目录固定在 `/data/postgres`
- 启动时先做健康检查再允许后续服务继续
- 不允许业务代码绕过 Alembic 直接建表

### 5.2 api-server

职责：

- 提供所有业务 API
- 暴露运行时版本信息接口
- 启动时连接 PostgreSQL
- 管理 worker、scheduler、插件运行链路

现有代码里已经有好的基础：

- [start-api-server.sh](C:/Code/FamilyClaw/apps/start-api-server.sh) 已经具备“检查数据库版本并执行 `alembic upgrade head`”的思路
- [config.py](C:/Code/FamilyClaw/apps/api-server/app/core/config.py) 已经收口为 PostgreSQL-only

后续容器化时，不应该重写这套规则，只应该把它收口到统一入口脚本。

### 5.3 open-xiaoai-gateway

职责：

- 提供对外语音网关能力
- 通过内部地址访问 `api-server`

建议：

- 默认跟随主容器一起运行
- 通过环境变量控制是否启用
- 内部访问地址固定为 `http://127.0.0.1:8000/api/v1`

### 5.4 Nginx

职责只保留两件事：

1. 提供前端静态文件
2. 把 `/api/` 反代到后端

不要在 Nginx 里堆复杂业务逻辑。它只是入口，不是业务层。

### 5.5 user-app

前端在构建阶段编译成静态文件，运行时由 Nginx 直接提供。

这意味着：

- 运行中的前端不需要额外进程
- 单容器结构更稳定
- 用户升级时只需要替换整个镜像

## 6. 启动顺序

启动顺序必须固定，不能自由发挥。

推荐顺序：

1. 启动 PostgreSQL
2. 等待 PostgreSQL ready
3. 执行数据库备份前检查
4. 执行 `alembic upgrade head`
5. 迁移成功后启动 `api-server`
6. 启动 `open-xiaoai-gateway`
7. 启动 Nginx
8. 健康检查全部通过后把容器标记为健康

### 6.1 为什么必须这样

原因很简单：

- 数据库没 ready，后端起来也没用
- 迁移没完成，业务服务先起来就会踩脏数据
- gateway 依赖后端，不能抢跑
- Nginx 最后起来，用户看到的才是完整可用系统

### 6.2 失败策略

任一关键步骤失败时，容器应直接失败退出。

尤其是下面两种情况，绝不能带病运行：

- PostgreSQL 无法启动或连接失败
- Alembic 迁移失败

如果迁移失败还继续拉起业务服务，那不是“尽量可用”，那是在破坏数据。

## 7. 数据目录设计

为了降低普通用户的理解成本，建议只暴露一个挂载卷：

- `/data`

容器内部再拆分成固定子目录：

- `/data/postgres`：PostgreSQL 数据
- `/data/plugins`：插件与相关持久化文件
- `/data/backups`：升级前备份
- `/data/runtime`：当前安装状态、锁文件、诊断信息

### 7.1 这样设计的好处

- 用户只需要备份一个目录
- 升级前备份和回退路径清楚
- 不会让用户在多个卷之间猜哪个最重要

### 7.2 用户侧说明应该怎么写

部署文案应该简单到这种程度：

1. 拉取镜像
2. 挂载本地目录到 `/data`
3. 配置环境变量
4. 启动容器

不要让普通用户读“数据卷 A 给数据库，数据卷 B 给插件，数据卷 C 给日志”这种工程师文案。

## 8. 版本治理与发布闭环

### 8.1 唯一应用版本源

应用版本应当只有一个事实源：

- 仓库根目录 `VERSION`

其他地方的版本号都应该由脚本同步，不允许手工各改各的。

建议同步目标包括：

- `apps/api-server/pyproject.toml`
- `apps/open-xiaoai-gateway/pyproject.toml`
- `apps/user-app/package.json`
- `packages/*/package.json`

### 8.2 发布入口

正式发布应由 Git tag 驱动：

- 稳定版：`vX.Y.Z`
- 预发布版：`vX.Y.Z-rc.N`

GitHub Actions 负责：

1. 校验 tag 与 `VERSION` 一致
2. 构建前端静态资源
3. 构建单容器镜像
4. 读取 Alembic head
5. 生成 `release-manifest.json`
6. 推送 GHCR
7. 创建 GitHub Release

### 8.3 发布清单是闭环的关键

每次正式发布都应生成一份 `release-manifest.json`，同时满足两种用途：

- 镜像内嵌，供运行时读取
- 作为 GitHub Release 附件，供发布记录追溯

建议字段至少包括：

```json
{
  "app_version": "0.1.0",
  "git_tag": "v0.1.0",
  "git_sha": "abc1234",
  "built_at": "2026-03-18T12:00:00Z",
  "release_url": "https://github.com/org/repo/releases/tag/v0.1.0",
  "docker_image": "ghcr.io/org/familyclaw:0.1.0",
  "schema_heads": ["20260318_0055"],
  "upgrade_policy": {
    "min_supported_app_version": "0.1.0",
    "allowed_source_schema_heads": ["20260317_0053"],
    "requires_manual_migration": false,
    "manual_migration_doc": null
  },
  "build_channel": "stable"
}
```

## 9. 应用版本与数据库版本关系

### 9.1 结论

不建议把“数据库版本 051”设计成真正的升级判断依据。

最佳实践应当是：

- 应用版本：用 SemVer，例如 `0.1.0`
- 数据库 schema 真相：用 Alembic `revision/head`
- 发布清单：记录“这个应用版本目标对应哪些 schema heads”

### 9.2 为什么不建议维护独立数据库编号

主要有四个原因：

1. Alembic 已经是数据库结构的事实源
2. 多个应用版本完全可能对应同一个 schema head
3. merge revision、分支收敛时，手工整数编号会很快失真
4. 人工维护第二套编号体系，只会增加漂移风险

### 9.3 如果一定要给人看懂

如果后续确实需要给文档或界面展示一个更顺手的代号，可以加一个派生展示字段，例如：

- `schema_label: "051"`

但必须明确：

- `schema_label` 只能是展示别名
- 真正参与升级判断和兼容校验的，仍然是 `schema_heads`

## 10. 运行时版本暴露

后端应继续提供统一接口：

- `GET /api/v1/system/version`

现有实现基础已经存在：

- [version_metadata.py](C:/Code/FamilyClaw/apps/api-server/app/core/version_metadata.py)
- [system.py](C:/Code/FamilyClaw/apps/api-server/app/api/v1/endpoints/system.py)

这个接口应返回：

- 当前应用版本
- 构建通道
- 构建时间
- 当前 release 链接
- 更新状态
- 最新版本
- 最新版本说明摘要

### 10.1 离线和在线两种信息来源

运行时需要区分两种真相：

1. 离线真相：镜像内的 `release-manifest.json`
2. 在线真相：GitHub Release API 返回的“最新发布”

离线真相用于回答“我现在是谁”。

在线真相用于回答“有没有更新”。

不要反过来。否则一旦离线运行，系统连自己是谁都说不清。

## 11. 普通用户视角的更新体验

版本管理页面应该继续坚持普通用户视角，不要退回工程师视图。

普通用户真正需要知道的只有这些：

- 当前是不是最新版本
- 如果不是，最新版本是什么
- 新版本更新了什么
- 去哪里看完整更新说明

普通用户默认不需要看到：

- Git SHA
- Alembic head
- 构建通道细节
- 镜像 tag
- 内部升级策略字段

这些信息可以放在诊断区，但默认收起。

## 12. 升级策略

### 12.1 推荐升级路径

升级过程建议固定为：

1. 用户在 NAS 或 Docker 宿主机拉取新镜像
2. 停止旧容器
3. 用同一个 `/data` 目录启动新容器
4. 容器启动后先执行升级前检查
5. 自动执行数据库备份
6. 执行 `alembic upgrade head`
7. 迁移成功后启动业务服务
8. 在 `/data/runtime` 记录当前已安装版本

### 12.2 升级前自动备份

建议镜像内提供统一命令，例如：

- `familyclawctl backup`
- `familyclawctl doctor`
- `familyclawctl restore`
- `familyclawctl version`

其中 `backup` 应在升级时自动执行，至少备份 PostgreSQL 数据。

### 12.3 升级失败时怎么办

升级失败时必须满足：

- 容器退出
- 旧数据不被覆盖清理
- 备份文件还在
- 日志明确指出失败阶段

不要在失败后继续让服务半活不活地挂着。

## 13. 回退策略

### 13.1 可直接回退的情况

如果两个版本之间没有 schema 变化，或者发布清单明确声明允许直接回退，那么可以：

1. 停止当前容器
2. 切回旧镜像
3. 用原 `/data` 目录启动

### 13.2 不可直接回退的情况

如果升级已经执行了数据库迁移，则不允许只回退应用镜像。

这时必须：

1. 停止当前容器
2. 恢复升级前备份
3. 使用旧镜像重新启动

这条规则必须写进发布说明，不能让用户靠猜。

## 14. GHCR 和 GitHub Release 约定

### 14.1 镜像标签

推荐标签策略：

- `ghcr.io/<org>/familyclaw:X.Y.Z`
- `ghcr.io/<org>/familyclaw:X.Y.Z-rc.N`
- `ghcr.io/<org>/familyclaw:X.Y`
- `ghcr.io/<org>/familyclaw:X`
- `ghcr.io/<org>/familyclaw:latest`
- `ghcr.io/<org>/familyclaw:preview`
- `ghcr.io/<org>/familyclaw:sha-<shortsha>`

其中：

- 排障和审计只认精确标签
- `latest` 只是用户方便入口，不能当审计依据

### 14.2 GitHub Release 内容

每个 GitHub Release 至少应包含：

- 版本标题
- 人话版更新说明
- `release-manifest.json`
- 如有需要，人工迁移说明

## 15. 健康检查与诊断

建议至少提供以下健康检查：

- PostgreSQL ready
- `api-server` 存活
- Nginx 可响应
- 如启用 gateway，则 gateway 存活

建议提供一个统一诊断入口，至少能输出：

- 当前应用版本
- 当前镜像标签
- 当前 schema heads
- 当前数据库连接状态
- 最近一次升级结果

## 16. 实施顺序建议

建议分三步走，不要一口吃胖：

### 第一步：先把单镜像跑通

- 前端编译进镜像
- Nginx 提供静态资源
- PostgreSQL、api-server、gateway、Nginx 收口到一个容器
- 固定启动顺序
- 固定 `/data` 卷布局

### 第二步：把发布流程拉直

- 引入根目录 `VERSION`
- 做版本同步脚本
- GitHub Actions 构建单镜像
- 生成并内嵌 `release-manifest.json`
- 发布到 GHCR 和 GitHub Release

### 第三步：把升级闭环补完

- 启动前自动备份
- 启动时记录安装状态
- 设置页显示更新摘要
- 补齐回退和诊断工具

## 17. 最终结论

这套方案的核心不是“把所有东西塞进一个容器”这么简单，而是把下面这几件事同时做对：

- 交付物只有一个镜像
- 用户只需要管理一个数据目录
- 数据库迁移始终由 Alembic 负责
- GitHub Release 是发布真相
- `release-manifest.json` 是离线真相
- `/api/v1/system/version` 是运行时真相
- 升级必须先备份，再迁移，再启动

只要这条线是直的，普通用户就能用，后续维护也不会失控。
