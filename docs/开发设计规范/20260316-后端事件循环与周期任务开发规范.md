# 20260316-后端事件循环与周期任务开发规范

状态：Draft

## 这份规范解决什么问题

后端最容易烂掉的地方，不是某个函数写得丑，而是执行边界乱了：

- `async` 入口里直接跑同步 DB、同步 HTTP、同步插件
- 后台周期任务自己写一套 `while True + sleep`
- 请求作用域 `Session` 被直接带进线程池
- worker 卡住以后把 API 和 WebSocket 一起拖慢

这份文档只解决一个问题：

新增或修改后端周期任务、后台 worker、异步入口时，必须遵守统一的执行边界和回归要求，别再把主事件循环拖死。

## 先问三个问题

开始动手前，先问：

1. 这是个真问题还是假问题？
2. 有没有更简单的方法？
3. 会破坏什么吗？

如果回答不清，就不要急着加新后台任务或新抽象。

## 主事件循环允许做什么

主事件循环只做轻量协调：

- 参数解析
- 鉴权和权限判断
- 事件分发
- 响应拼装
- 已确认不会阻塞的 async 调用

只要一段逻辑会长时间占住线程，就不允许继续留在主事件循环里。

## 哪些逻辑必须下沉

下面这些一律视为阻塞调用，必须显式下沉：

- 同步数据库查询、写入、事务提交
- 同步 HTTP / SDK 调用
- 同步插件执行
- 长时间 CPU 计算
- 任何会 `sleep`、重试等待、慢规则评估的同步逻辑

一句话：只要会卡线程，就下沉，不要争论。

## 阻塞调用规范

### 1. 同步 DB

- 一律走统一阻塞包装，例如 `run_blocking_db(...)`
- 不准自己复制 `to_thread + commit + rollback + close` 的散装逻辑

### 2. 同步 HTTP / SDK / 插件执行 / CPU 重活

- 一律走统一阻塞包装，例如 `run_blocking(...)`
- 必须显式写清楚：
  - 调用标签
  - 调用类型
  - 超时时间

### 3. 禁止的写法

- 先直接在 `async` 里跑，再赌不会卡
- 每条链路各写各的 `run_in_executor`
- 没有超时，没有 label，没有上下文

## Session 线程边界

这条是铁律。

1. 请求作用域 `Session` 不允许直接带进线程池。
2. WebSocket 持有的 `Session` 不允许直接带进线程池。
3. 生命周期协程里的 `Session` 不允许直接带进线程池。
4. 线程池里的数据库逻辑必须使用线程内新建的 `Session`。
5. 不允许把半提交状态的 ORM 对象直接跨线程传递。

要传就传普通数据。真要跨边界，先序列化，或者先提交事务再下沉。

## 周期任务和 worker 规范

### 1. 不要手写散装循环

新增周期任务、常驻 worker、后台轮询任务时：

- 不要手写散装 `while True + sleep`
- 必须优先复用统一 runtime

### 2. 每个 tick 必须有边界

至少要有：

- `tick_timeout_seconds`
- 连续失败阈值
- 降级策略
- 健康状态

### 3. busy / idle 语义要清楚

- 本轮实际处理了任务，可以直接进入下一轮
- 本轮没处理任务，再按间隔等待

不要把吞吐和等待策略写成一团垃圾条件判断。

### 4. worker 必须可观察

至少能读到这些健康信息：

- `state`
- `consecutive_failures`
- `last_succeeded_at`
- `last_failed_at`
- `last_duration_ms`
- `last_error_summary`

如果一个 worker 出了问题，别人连它是不是在降级都看不出来，那就是不合格实现。

## 新增后端周期任务的最低要求

任何新周期任务，至少满足下面这些：

1. 执行骨架走统一 runtime
2. 阻塞 DB 调用走统一 DB helper
3. 阻塞网络 / 插件 / CPU 调用走统一 blocking helper
4. 不跨线程乱传 `Session` 和 ORM 对象
5. 有明确 timeout、失败阈值、降级和健康状态
6. 不破坏现有 API / WebSocket / webhook 对外行为

## 回归要求

新增或修改周期任务时，必须补定向测试，至少覆盖：

1. 阻塞 helper 边界
2. worker 成功 / 失败 / 超时 / 降级
3. 慢任务不会把别的 HTTP / WebSocket 请求一起拖慢

如果这些都没有，代码评审默认不应该放行。

## 这份规范明确不做什么

- 不要求全项目一次性改成纯 async
- 不要求为了新任务重做整个调度系统
- 不允许拿“以后再治理”当理由继续把阻塞逻辑塞回主事件循环
