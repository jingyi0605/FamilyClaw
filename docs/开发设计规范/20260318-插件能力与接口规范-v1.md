# 20260318-插件能力与接口规范-v1

状态：Active

## 这份规范解决什么问题

FamilyClaw 现在已经有插件注册、启停、执行、配置、卡片快照这些能力，但还没有把“宿主内核做什么、插件做什么”彻底写死。

这会带来几个直接问题：

- 新能力很容易继续写进宿主，天气就是已经踩到的边界问题
- 同样叫“插件”，有的像通道，有的像任务执行器，有的又像持续状态集成，语义混在一起
- 记忆这种高敏感能力到底是不是普通插件，也没有在总规范里说死

这份规范的目标很明确：

- 把宿主收缩成平台内核
- 把领域能力交给插件
- 给插件定义统一的能力边界、输入输出和错误语义
- 把记忆相关能力纳入统一插件规范，但保留宿主主权

## 核心判断

宿主只做平台内核，不做领域功能。

插件只做领域接入和能力扩展，不碰宿主主权数据。

一句话版本：

宿主负责标准、权限和落库，插件负责接入、映射和执行。

## 当前决策边界

项目当前还没有正式发布，所以这次规范不再为旧插件结构保留兼容债。

这意味着：

- 不需要为了历史包袱长期保留 `connector` 语义
- 不需要保留旧 `memory-ingestor` 作为一等插件类型
- 现有仓库里的所有内置插件都应该直接迁到新架构

后续版本兼容要考虑的是“未来发布后的规范演进”，不是“当前仓库里的旧实现习惯”。

## 设计原则

### 1. 宿主是内核，不是全能业务模块

宿主必须只保留这些能力：

- 家庭、成员、房间、权限、审计
- 对话与自动化编排
- 插件运行时与版本治理
- 配置、密钥、启停和任务调度
- 统一设备模型
- 统一实体状态模型
- 统一动作路由模型
- 统一仪表盘卡片规范
- 统一记忆对象标准、修订历史和可见性规则
- 缓存、降级、错误语义

宿主不应该继续增加这些模块：

- `weather_service`
- `power_service`
- `steps_service`
- `health_service`
- 其他任何针对单一领域的宿主专用聚合模块

### 2. 插件负责磨平第三方复杂度

插件应该负责：

- 接第三方 API、设备协议或外部平台
- 把第三方字段映射成宿主标准对象
- 发现资源、建立绑定、同步刷新
- 做插件自己的缓存、去重、限流和字段降级
- 产出标准设备、实体、卡片快照、动作描述
- 在宿主授权范围内读取上下文或调用宿主保留接口

插件不应该负责：

- 直接写宿主核心业务表
- 自己发明设备协议
- 自己定义卡片协议
- 自己定义一套独立于宿主之外的记忆权限规则
- 绕过宿主权限系统直接暴露敏感数据

### 2.1 插件私有状态边界

这条规则现在也要写死：

- 插件私有表、插件私有缓存、provider 绑定关系、刷新游标和原始快照，只属于插件自己。
- 宿主核心不导入、不注册、不直接查询插件私有 ORM 模型。
- 宿主只通过标准 DTO、插件公开接口和通用插件状态表消费插件结果。
- 如果插件需要迁移自己的私有表，迁移责任也属于插件私有边界，不能倒挂回宿主核心模型入口。
- 宿主核心 Alembic migration 也不得直接创建、升级或删除插件私有领域表。

这里再补四条落地规则：

- `builtin` 插件以及所有 `third_party` 插件（原 `official` 归类在这里）如果需要私有迁移，目录约定就是插件根目录下的 `migrations/`，并且必须包含 `env.py` 与版本脚本。
- 宿主当前只为 `builtin` / `third_party` 开放同进程私有 migration runner；触发时机是插件执行准备阶段，宿主会先确保该插件私有 Alembic 已经到 `head`，再继续实例同步或其他插件执行。
- 插件私有 ORM 模型允许复用宿主共享 `Base`，但不得再注册进 `app.db.models` 这类宿主核心聚合入口；“复用 Base”不等于“回到宿主核心模型集合”。
- 如果插件坚持使用独立 metadata，那跨宿主公共表的外键约束、flush 顺序和 metadata 组织都由插件自己负责；宿主不为某个领域再加专用 ORM 特判。
- 运行态标准实体统一落到宿主公共实体承载层；当前正式实现是宿主公共表 `device_entities`，后续任何设备读取、控制和展示链路都只认这层标准实体。
- `DeviceBinding.capabilities` 只允许保留绑定摘要、主实体标识、实体 ID 列表、能力标签和轻量元数据，不允许再承载完整运行态 `entities` 快照。

一句话总结：

宿主只认“插件实例 + 虚拟设备 + 标准实体 + 标准卡片”，不认“这是天气绑定表”。

### 3. 数据标准比执行入口更重要

平台能不能长期稳定，不取决于“能不能执行插件”，而取决于“插件产出的数据是不是统一的”。

所以后续演进优先级必须是：

1. 统一数据模型
2. 统一生命周期
3. 统一执行模型

### 4. 记忆是插件化能力，但不是普通扩展点

记忆中心允许插件接入，但它不是普通 `integration`。

记忆相关插件必须建立在下面这条硬边界上：

- 宿主保留记忆对象标准
- 宿主保留可见性和权限裁决
- 宿主保留修订历史和人工修正入口
- 插件只接管记忆引擎、检索后端和上下文组装

## 插件配置联动与地区接口（v1.1 补充）

这部分是这次必须补上的正式边界，不然插件配置系统永远只能停留在“静态表单”。

### 1. 动态表单联动属于宿主能力

宿主负责：

- 解析插件声明的 `option_source`
- 提供配置草稿 `resolve` 接口
- 在字段变化后重算动态选项
- 在提交时支持 `clear_fields` 和 `clear_secret_fields`

插件负责：

- 声明字段依赖
- 声明动态选项源
- 决定哪些字段在依赖变化后需要清掉旧值

不要再做：

- 在插件前端里自己写一套独立联动协议
- 在宿主里给某个插件硬编码特判数据源

### 2. 地区 provider 和地区目录接口已经对插件开放

宿主对插件配置正式开放两类动态数据：

- `region_provider_list`
  - 返回当前家庭可用的地区 provider 列表
- `region_catalog_children`
  - 按 provider、国家、父级地区返回子级目录

这条边界的含义很明确：

- 插件可以用这些接口做地区选择
- 插件不应该直接依赖宿主内部 `RegionNode` 表结构
- 后续安装新的地区 provider 后，宿主应该直接把它暴露给表单，不需要再改天气插件

### 3. 天气插件是第一批正式案例

天气插件现在应该走的标准路径是：

1. 动态拉 `provider_code`
2. 选中 provider 后拉 `province_code`
3. 选中省份后拉 `city_code`
4. 选中城市后拉 `district_code`

如果宿主没有这套能力，天气插件就只能继续维护巨型静态省市区枚举。那不是“实现成本高”这么简单，而是设计边界烂了。

## 插件类型体系

普通插件类型统一定义为以下几类。

### `integration`

持续状态型插件。

适用场景：

- 天气
- 电费
- 健康步数
- Home Assistant
- 智能音箱
- 可穿戴设备

典型特点：

- 有实例
- 可发现资源
- 会持续刷新状态
- 产出设备和实体
- 可选产出动作和卡片

### `action`

纯动作型插件。

适用场景：

- 调用某个第三方服务执行动作
- 对现有设备发送命令
- 风险操作审批后执行

### `channel`

消息和会话通道插件。

适用场景：

- Telegram
- 飞书
- 企业微信
- Discord

### `agent-skill`

对话代理可调用的工具能力插件。

特点是短执行链路、明确输入输出、可编排。

### `ai-provider`

模型或推理服务供应商接入插件。

### `region-provider`

地区、行政区和坐标解析插件。

### `locale-pack`

语言包资源插件。

它负责提供一整套 locale 定义和大范围翻译资源，比如 `zh-TW` 这种完整语言包。

同时要注意一件事：

- `locale-pack` 不是唯一能声明 `locales` 的插件类型
- 普通插件也可以声明自己的 `locales`
- 普通插件声明 `locales` 的目的，只能是提供自己这部分插件文案，比如卡片标题、插件页说明、动作标签
- 普通插件不应该假装自己是整套语言包，更不应该覆盖宿主的大范围基础文案

### `theme-pack`

主题资源插件。

### `theme-pack` 资源字段（v1.1 补充）

为了让内置主题和远端主题完全走同一条插件链路，`theme-pack` 的 `capabilities.theme_pack` 在 v1.1 里统一补充以下字段：

- `theme_id`：主题 ID
- `display_name`：展示名称
- `entry_resource`：主题 token 入口资源路径
- `resource_source`：`builtin_bundle` 或 `managed_plugin_dir`
- `resource_version`：资源版本号
- `platform_targets`：可用端，至少包含 `h5` 或 `rn`
- `theme_schema_version`：token schema 版本

兼容说明：

- `tokens_resource` 作为历史字段仍可读，但新实现统一以 `entry_resource` 为准。
- 前端运行时只把 `user-core` 当作主题选择状态持久化层，不再把它当静态主题资源源。
- `user-ui` 只保留 token 归一化、CSS 变量映射、RN 映射工具，不再内置 8 套 canonical theme token 表。
- RN 端通过统一插件 token 派生语义 token，差异只允许出现在 RN 适配层。
- `user-ui` 允许保留一个“运行时占位主题”用于首帧渲染安全，但该占位主题不能映射真实 `theme_id`，也不能进入可选主题 canonical 列表。
 - RN 端在 `theme_id` 不可解析时必须返回 `missing` 状态（如 `theme_not_registered`），不允许静默回退到宿主默认主题。

### 主题插件注册表接口

后端不再生成任何宿主静态主题表，而是直接扫描 `apps/api-server/app/plugins/builtin/theme_*_pack` 中的内置插件和运行时复制到 `data/plugins` 的官方/第三方插件包，将它们的 `theme-pack` manifest、`entry_resource`、`resource_source`、`resource_version`、`platform_targets` 等交给前端。`list_registered_plugin_themes_for_household(db, household_id)` 会返回包含 `plugin_id`、`theme_id`、`state`（`ready`/`disabled`/`invalid`）、`resource_source`、`resource_version`、`preview`（若插件提供）等字段的注册表快照，前端以此判断是否继续展示当前主题插件，`state` 非 `ready` 时必须把该 selection 标记为失效并进入待重选状态，而不是偷偷 fallback 到宿主默认主题。

前端获取注册表后，在确认 `plugin_id + theme_id` 可用时再调用 `GET /api/v1/ai-config/{household_id}/plugin-themes/{plugin_id}/{theme_id}` 读取真正的 `tokens` 正文；后端通过 `_resolve_theme_resource_path` 校验 `entry_resource` 并在 `builtin_bundle`（随前端包内置）或 `managed_plugin_dir`（托管在 `data/plugins`）这两种资源源之间切换，同时返回 token 所在的 `resource_version`、`theme_schema_version` 与 `platform_targets`，H5/RN 共用这一套数据，只在适配层转成 CSS 变量或 RN semantic token。

登录页、未登录壳层和首屏必须只使用内置插件生成的 tokens，拿到家庭上下文后再叠加官方/第三方插件注册表；整个链路上只有插件 manifest + manifest 附属资源才是 canonical theme source，再也没有 `packages/user-core`、`packages/user-ui` 或宿主配置里隐藏的旧 token 表。

## 平台独占槽位

除了上面的普通插件类型，平台还保留三类独占槽位。

这些槽位也是插件，但不是开放式的普通扩展点，而是宿主控制的独占能力位。

### `memory_engine`

负责事件到记忆候选的抽取、去重、合并和重要度评估。

### `memory_provider`

负责记忆的索引、存储和检索。

### `context_engine`

负责把记忆结果组装成对话上下文 bundle。

### 槽位规则

- 同一宿主同一时间只能激活一个 `memory_engine`
- 同一宿主同一时间只能激活一个 `memory_provider`
- 同一宿主同一时间只能激活一个 `context_engine`
- 宿主必须始终保留内置 fallback 实现

## 现有插件的目标归位

当前仓库里的插件，后续目标类型应直接调整为：

- `official-weather` -> `integration`
- `health-basic-reader` -> `integration`
- `homeassistant` -> `integration + action`
- `open-xiaoai-speaker` -> `integration + action`
- `channel-*` -> `channel`
- `locale-zh-tw` -> `locale-pack`

如果现有插件还带有旧类型语义，应直接改 manifest 和实现，不保留旧路线。

## 插件能力声明

### Manifest 最低字段

每个插件 manifest 至少要声明：

- `id`
- `name`
- `version`
- `api_version`
- `types`
- `permissions`
- `entrypoints`
- `config_specs`
- `capabilities`

### 插件自带 locale 资源

如果插件有自己专属的文案键，比如：

- 仪表盘卡片标题
- 插件配置页说明
- 插件动作按钮文案

那么插件可以在 manifest 里直接声明 `locales`，把这些键跟着插件一起发布。

规则如下：

- `locale-pack` 仍然必须至少声明一个 `locale`
- 非 `locale-pack` 插件也允许声明 `locales`
- 非 `locale-pack` 插件声明的 `locales`，只应用在该插件自己的文案键上
- 这里的“插件自己的文案键”不只包括配置页和动作文案，也包括卡片 payload 里所有用户可见字段
- 同一个家庭下，如果多个启用插件同时为同一个 `locale_id` 提供消息，宿主必须按消息键逐条合并，不能整包二选一覆盖
- 合并冲突时，来源优先级仍然是 `builtin > third_party`（原 `official` 的内容也按照第三方插件处理）
- 如果同优先级下还有冲突，再按插件 id 稳定排序
- locale 元信息的归属优先给 `locale-pack`；如果没有 `locale-pack`，再从其他插件里选最高优先级来源

### `integration` 必须额外声明

- `domains`
- `instance_model`
- `refresh_mode`
- `supports_discovery`
- `supports_actions`
- `supports_cards`
- `entity_types`
- `default_cache_ttl_seconds`
- `max_stale_seconds`

如果插件要走“启用后自动建默认实例”的主链路，还应该一起声明：

- `auto_create_default_instance`
- `default_instance_display_name`
- `default_instance_config`

如果插件会按设备或实例动态产出多张卡片，`dashboard_cards` 里还应该声明：

- `card_key`
- `card_key_prefix`

### 槽位型插件必须额外声明

- `slot_name`
- `exclusive`
- `fallback_required`
- `input_contract_version`
- `output_contract_version`

```

  "api_version": 1,

  "capabilities": {
    "memory_provider": {
      "slot_name": "memory_provider",
      "exclusive": true,
      "fallback_required": true,
      "input_contract_version": 1,
      "output_contract_version": 1
    }
  }
}
```

## 生命周期

插件不是只有“执行”这一个动作。

### 普通插件生命周期

对于普通插件，宿主统一按以下生命周期调用：

1. `validate_manifest`
2. `create_instance`
3. `configure_instance`
4. `discover`
5. `refresh`
6. `invoke_action`
7. `render_card_snapshot`
8. `diagnostics`
9. `teardown`

### 槽位插件生命周期

对于独占槽位插件，宿主统一按以下生命周期调用：

1. `validate_manifest`
2. `activate_slot`
3. `execute_slot_contract`
4. `diagnostics`
5. `deactivate_slot`

### 实现要求

- `integration` 至少实现 `refresh`
- 有发现能力的插件实现 `discover`
- 有控制能力的插件实现 `invoke_action`
- 有仪表盘接入的插件实现 `render_card_snapshot`
- `memory_engine` 至少实现记忆抽取主接口
- `memory_provider` 至少实现写入和查询主接口
- `context_engine` 至少实现上下文构建主接口

## 配置作用域

规范 v1 正式定义以下配置作用域：

- `plugin`
- `integration_instance`
- `device`
- `channel_account`

规则如下：

- `plugin` 用于插件全局默认值
- `integration_instance` 用于某个实例自己的连接参数和策略
- `device` 用于设备级可见配置
- `channel_account` 用于通道账号配置

不允许继续拿 `plugin` scope 假扮 `integration_instance` scope。

典型用法：

- 像天气源类型、全局缓存时间、OpenWeather / WeatherAPI key 这种“整个插件共用一份”的配置，放 `plugin`
- 像天气地区绑定、多网关接入、多账号实例这种“某个实例自己独有”的配置，放 `integration_instance`

## 标准数据接口

### 原则

普通状态型插件不直接落库，只返回标准 DTO。

宿主负责：

- 校验
- 去重
- 落库
- 审计
- 回滚
- 状态聚合

### `integration.refresh` / plugin-managed sync 标准输出

对插件作者来说，正式口径不是“只会返回设备和实体”。

第一版至少要把下面这些字段当成正式边界来理解：

```json
{
  "instance_status": {
    "success": true,
    "degraded": false,
    "error_code": null,
    "error_message": null,
    "refreshed_at": "2026-03-18T12:00:00Z"
  },
  "devices": [],
  "entities": [],
  "discoveries": [],
  "actions": [],
  "diagnostics": [],
  "dashboard_snapshots": [],
  "records": [],
  "observations": [],
  "message": "同步完成",
  "sync_scope": "device_sync",
  "items": [],
  "summary": {}
}
```

说明：

- `instance_status` 用来告诉宿主这次实例级同步是成功、降级还是错误
- `dashboard_snapshots` 是当前插件执行链路的正式卡片输出字段，新插件不要再写旧名字 `card_snapshots`
- `records` 交给宿主统一落原始记录
- `observations` 交给宿主统一写标准观察类记忆
- `message`、`items`、`summary`、`sync_scope` 主要给实例动作返回和前端交互用，不是每次都必须返回

插件不是所有字段都要填，但字段名和语义必须统一。

### 标准设备 DTO

```json
{
  "unique_key": "weather:home-default",
  "device_kind": "virtual_household",
  "domain": "weather",
  "name": "家庭天气",
  "room_id": null,
  "metadata": {}
}
```

字段说明：

- `unique_key`：插件内唯一，不允许随便变
- `device_kind`：`physical` / `virtual_household` / `virtual_member`
- `domain`：领域名，例如 `weather`
- `metadata`：显示层和调试层可消费的补充信息

### 标准实体 DTO

```json
{
  "device_unique_key": "weather:home-default",
  "entity_key": "weather.temperature",
  "entity_type": "sensor",
  "value": 21.5,
  "unit": "C",
  "available": true,
  "degraded": false,
  "observed_at": "2026-03-18T12:00:00Z",
  "expires_at": "2026-03-18T12:30:00Z",
  "attributes": {}
}
```

补一条硬规则：

- 插件返回的 `entities` 必须已经是最终标准实体。
- 宿主只做通用校验、去重、持久化、状态聚合和标准控制路由，不做任何按插件或按领域的二次归一化。
- 运行态标准实体进入宿主后，正式事实源就是宿主公共实体承载层，不是 `DeviceBinding.capabilities.entities`。
- 如果插件仍往 `capabilities` 里塞完整实体快照，那只是兼容旧数据时的临时兜底，不是正式协议，也不允许继续扩张。

### 标准发现 DTO

```json
{
  "resource_type": "device",
  "discovery_key": "speaker:living-room",
  "title": "客厅小爱音箱",
  "subtitle": "等待认领",
  "capability_tags": ["speaker", "voice"],
  "metadata": {}
}
```

### 标准卡片快照 DTO

卡片结构仍由宿主卡片规范控制。

插件只能返回卡片数据，不允许定义新的卡片协议。

插件当前可以稳定依赖的字段包括：

- `card_key`
- `placement`
- `title`
- `subtitle`
- `payload`
- `actions`

匹配规则：

- 宿主先按 manifest 里的 `card_key` 精确匹配
- 如果声明了 `card_key_prefix`，宿主会把同前缀的动态卡片也收进来

`payload` 里的正式保留字段：

- `card_kind`
- `card_state`

领域插件也可以带上自己的透传字段，比如天气卡片里的 `device_id`、`binding_type`，但这类字段不能替代宿主卡片协议本身。

### 卡片 payload 文案硬规则

这条规则现在正式写死，不再靠默认理解。

- 卡片 payload 里所有用户可见字段，必须通过标准字段传递，宿主不得按插件 id、插件 key、领域类型写专用翻译或专用兜底分支。
- 面向列表项、明细项、页脚项这类结构，插件应统一产出可被宿主直接消费的标准对象；常用字段至少包括：`label`、`label_key`、`value`、`value_display`、`value_type`，需要单位时再带 `unit`。
- 如果卡片里有集合型内容，统一放进 `items`，或者在宿主已允许的固定结构里放进 `detail_items`、`footer_items`；集合内部每一项也要遵守同一套字段规则。
- `label` 是可直接展示的人类可读文本，`label_key` 是插件词典 key；两者可以并存，但不能只给一个宿主根本不认识的自造字段。
- `value` 是原始值，`value_display` 是最终展示值；插件需要自己决定保留几位小数、如何拼接方向词、如何显示摘要，宿主不再按天气、电费、健康等领域做二次加工。
- `value_type` 用来告诉宿主这是 `text`、`number`、`percentage`、`datetime`、`direction` 还是其他标准展示类型；宿主只按标准类型渲染，不按插件做 if/else。
- 宿主渲染顺序固定为：优先 `label_key`，未命中时退回 `label`，再退到安全默认值；禁止把 `label_key` 原样显示给用户。
- 如果插件已经在标准实体里产出了同一批展示字段，卡片 payload 和标准实体必须保持一致，不能一边走插件词典，一边又要求宿主写一套硬编码解释。

## plugin-managed integration 运行时上下文

默认规则还是那句老话：插件只能依赖 JSON 可序列化输入，不要假定宿主会把一堆运行时对象塞给你。

但为了让“宿主创建实例后立刻同步”这条主链路能跑通，当前宿主已经正式开放了一小块 integration 运行时上下文：

- `_system_context.device_integration.database_url`
- `_system_context.device_control.database_url`

这用于插件在拿不到 live Session 时，按当前执行入口拿到宿主数据库连接串，再自己重建一次只读/短生命周期访问。

对宿主信任并以同进程方式执行的插件，宿主还允许在 sync 链路里额外注入：

- `_system_context.integration_runtime.db_session`

当前默认规则是：

- `builtin` 插件放在 `apps/api-server/app/plugins/builtin/`，纳入仓库管理。
- `plugins-dev` 放在 `apps/api-server/plugins-dev/`，是第三方插件开发源码目录，也是开发覆盖源；它参与家庭插件注册表合并，但不写入安装态目录。
- 当前家庭如果实际加载的是 `plugins-dev` 版本，前端插件列表和插件详情必须明确显示“开发插件生效中”，不能再让用户自己猜现在跑的是开发版还是安装版。
- `third_party/local` 插件放在 `apps/api-server/data/plugins/third_party/local/`，对应本地安装的第三方插件（`install_method=local`）。
- `third_party/marketplace` 插件放在 `apps/api-server/data/plugins/third_party/marketplace/`，对应市场安装（`install_method=marketplace`），`is_system=true` 只表示系统内置市场源。
- `third_party/local` 与 `third_party/marketplace` 都是运行时托管目录，不属于宿主源码编译产物；这些目录缺失时，宿主导入、宿主 Alembic 和应用初始化仍必须成立。

这不是给所有第三方插件开的通用特权，而是宿主与受信任同进程插件之间的优化边界。

补充一条部署和打包规则：

- Docker 最终运行镜像不应内置任何 `third_party` 插件代码；这些内容应通过宿主数据目录挂载或运行时安装。
- `.dockerignore` 和 Dockerfile 都应避免把 `apps/api-server/data/plugins/third_party/` 打进最终镜像。
- 宿主核心代码不得在 import 阶段静态依赖 `third_party` 插件模块，否则插件缺失时会直接导致宿主启动失败。
- 宿主如果需要某类插件共享的展示兜底、标准 DTO 校验或持久化表，应把这些能力收回宿主公共层，插件只依赖宿主接口，不允许反过来把插件变成宿主编译依赖。

### 标准动作 DTO

动作必须声明：

- `action_key`
- `label`
- `risk_level`
- `requires_confirmation`
- `payload_schema`

宿主负责鉴权和确认，插件只负责实际执行。

## 槽位型接口契约

独占槽位插件不走 `integration.refresh` 返回 DTO 这一套，而是走宿主保留接口契约。

### `memory_engine` 契约

至少包括：

- `ingest_event`
- `derive_memory_candidates`
- `merge_memory`
- `score_importance`

### `memory_provider` 契约

至少包括：

- `upsert_memory`
- `query_memory`
- `delete_memory`
- `rebuild_index`

### `context_engine` 契约

至少包括：

- `build_context_bundle`
- `summarize_memory_hits`
- `compress_context`

### 槽位边界

- 槽位插件无权最终决定记忆可见性
- 槽位插件无权绕过 revision
- 槽位插件无权删除宿主最小审计影子

## 错误语义和降级规则

### 标准错误码

平台统一使用以下错误语义：

- `plugin_disabled`
- `not_configured`
- `instance_not_found`
- `upstream_unavailable`
- `auth_failed`
- `rate_limited`
- `invalid_response`
- `partial_success`
- `stale_snapshot`

### 记忆槽位补充错误码

- `memory_engine_failed`
- `memory_provider_failed`
- `context_engine_failed`

### 降级原则

- 临时失败时保留最后一次成功状态
- 超过 `expires_at` 后标记 `stale`
- 只有明确不可恢复或配置错误时才打 `error`
- `memory_engine` 失败时保留事件记录
- `memory_provider` 失败时回退宿主默认 provider
- `context_engine` 失败时回退宿主默认 context engine

## 安全和权限边界

### 插件权限必须显式声明

插件只能获得声明过的权限。

未声明就没有。

### 插件不能直接拥有这些能力

- 任意数据库写权限
- 任意成员隐私读取权限
- 绕过宿主审计写入核心数据
- 绕过宿主确认执行高风险动作
- 绕过宿主权限裁决写入高敏感记忆

### 推荐的运行边界

- 默认优先子进程隔离
- 高风险插件默认禁用自动执行
- 外部网络访问建议声明 allowlist
- 所有敏感配置统一走宿主 secret 存储

## 版本兼容策略

项目当前不为仓库里的旧插件结构保留兼容层，但要为将来发布后的规范演进留出口。

### 平台要做的事

- 给插件 manifest 增加 `api_version`
- 给宿主保留规范升级路径
- 对废弃字段给出明确迁移窗口

### 插件要遵守的事

- 不能随便修改 `id`
- 不能随便改变 `unique_key` 生成规则
- 不能无迁移说明就修改实体语义
- 不能无迁移说明就修改槽位接口契约

## 明确禁止的做法

- 在宿主里新增单领域业务模块承接插件逻辑
- 插件直接写宿主核心业务表
- 用 `capabilities` 保存运行态实体状态
- 卡片绑定 city 文本而不是标准设备
- 用“页面特判”代替统一平台协议
- 把记忆权限、记忆修订和记忆 fallback 一起外包给第三方插件

### 插件词典与卡片文案补充规则

这一条是为了避免天气插件这类问题再次出现。

- 插件自带的 `manifest.json`、`locales/*.json`、运行时生成的文案模块，统一按 UTF-8 保存，不能把乱码文件继续发布给宿主。
- 插件卡片里只要出现插件专属文案，优先提供插件自己的 i18n key，并由宿主按家庭 locale 合并后渲染；不要把这类字段硬编码到宿主页面字典里。
- 插件写入标准实体前，必须先把 `name`、`unit`、`state_display`、`metadata.condition_text`、`metadata.error_message` 这类展示字段整理成最终态。宿主只负责渲染标准实体，不负责按插件类型做乱码修复。
- 如果插件同时提供卡片 payload 和标准实体，卡片展示文案与实体展示文案必须来自同一套插件词典或同一套标准字段，不能一边走插件词典，一边走宿主硬编码兜底。

## 落地建议

平台后续所有新能力，先问这四个问题：

1. 这是不是宿主内核能力？
2. 这是不是某个外部领域接入？
3. 这是不是一个普通插件类型就够了？
4. 如果是记忆相关能力，它是不是应该进独占槽位而不是普通插件类型？

如果答案是“领域接入”，默认先做插件，不要先进宿主。
