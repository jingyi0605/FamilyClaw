# 20260318-插件能力与接口规范-v1

状态：Draft

## 这份规范解决什么问题

FamilyClaw 现在已经有插件注册、启停、执行、配置、卡片快照这些能力，但还没有把“宿主内核做什么、插件做什么”彻底写死。

这会带来几个直接问题：

- 新能力很容易继续写进宿主，天气就是已经踩到的边界问题
- 同样叫“插件”，有的像通道，有的像任务执行器，有的又像持续状态集成，语义混在一起
- 记忆这种高敏感能力到底是不是普通插件，也没有在总规范里说死

这份规范的目标很明确：

- 把宿主收缩成平台内核
- 把领域能力交给插件
- 给插件定义统一的能力边界、输入输出和错误语义
- 把记忆相关能力纳入统一插件规范，但保留宿主主权

## 核心判断

宿主只做平台内核，不做领域功能。

插件只做领域接入和能力扩展，不碰宿主主权数据。

一句话版本：

宿主负责标准、权限和落库，插件负责接入、映射和执行。

## 当前决策边界

项目当前还没有正式发布，所以这次规范不再为旧插件结构保留兼容债。

这意味着：

- 不需要为了历史包袱长期保留 `connector` 语义
- 不需要保留旧 `memory-ingestor` 作为一等插件类型
- 现有仓库里的所有内置插件都应该直接迁到新架构

后续版本兼容要考虑的是“未来发布后的规范演进”，不是“当前仓库里的旧实现习惯”。

## 设计原则

### 1. 宿主是内核，不是全能业务模块

宿主必须只保留这些能力：

- 家庭、成员、房间、权限、审计
- 对话与自动化编排
- 插件运行时与版本治理
- 配置、密钥、启停和任务调度
- 统一设备模型
- 统一实体状态模型
- 统一动作路由模型
- 统一仪表盘卡片规范
- 统一记忆对象标准、修订历史和可见性规则
- 缓存、降级、错误语义

宿主不应该继续增加这些模块：

- `weather_service`
- `power_service`
- `steps_service`
- `health_service`
- 其他任何针对单一领域的宿主专用聚合模块

### 2. 插件负责磨平第三方复杂度

插件应该负责：

- 接第三方 API、设备协议或外部平台
- 把第三方字段映射成宿主标准对象
- 发现资源、建立绑定、同步刷新
- 做插件自己的缓存、去重、限流和字段降级
- 产出标准设备、实体、卡片快照、动作描述
- 在宿主授权范围内读取上下文或调用宿主保留接口

插件不应该负责：

- 直接写宿主核心业务表
- 自己发明设备协议
- 自己定义卡片协议
- 自己定义一套独立于宿主之外的记忆权限规则
- 绕过宿主权限系统直接暴露敏感数据

### 3. 数据标准比执行入口更重要

平台能不能长期稳定，不取决于“能不能执行插件”，而取决于“插件产出的数据是不是统一的”。

所以后续演进优先级必须是：

1. 统一数据模型
2. 统一生命周期
3. 统一执行模型

### 4. 记忆是插件化能力，但不是普通扩展点

记忆中心允许插件接入，但它不是普通 `integration`。

记忆相关插件必须建立在下面这条硬边界上：

- 宿主保留记忆对象标准
- 宿主保留可见性和权限裁决
- 宿主保留修订历史和人工修正入口
- 插件只接管记忆引擎、检索后端和上下文组装

## 宿主与插件边界

| 能力项 | 宿主负责 | 插件负责 | 备注 |
| --- | --- | --- | --- |
| 家庭/成员/房间 | 是 | 否 | 宿主主权数据 |
| 权限与审计 | 是 | 否 | 不能外包 |
| 插件注册与启停 | 是 | 否 | 统一治理 |
| 插件配置与密钥保存 | 是 | 否 | 插件声明 schema，宿主保存值 |
| 调度、任务、重试 | 是 | 否 | 插件只声明能力 |
| 第三方接入 | 否 | 是 | 插件核心职责 |
| 第三方字段映射 | 否 | 是 | 插件核心职责 |
| 设备标准模型 | 是 | 否 | 插件只能产出标准 DTO |
| 实体标准模型 | 是 | 否 | 插件不能自造实体语义 |
| 设备发现 | 部分 | 是 | 宿主负责承接与落库 |
| 资源同步 | 部分 | 是 | 插件拉取，宿主写入 |
| 动作执行 | 部分 | 是 | 宿主做鉴权和确认，插件做实际执行 |
| 卡片规范 | 是 | 否 | 插件只能填数据 |
| 卡片快照生成 | 部分 | 是 | 插件可选产出 snapshot |
| 记忆对象标准 | 是 | 否 | 不能外包 |
| 记忆修订与人工修正 | 是 | 否 | 不能外包 |
| 记忆引擎/召回策略 | 部分 | 是 | 通过独占槽位接入 |
| 上下文读取 | 是 | 部分 | 插件只能读获授权上下文 |
| 对话最终编排 | 是 | 否 | 不能交给插件接管主链路 |

## 插件类型体系

普通插件类型统一定义为以下几类。

### `integration`

持续状态型插件。

适用场景：

- 天气
- 电费
- 健康步数
- Home Assistant
- 智能音箱
- 可穿戴设备

典型特点：

- 有实例
- 可发现资源
- 会持续刷新状态
- 产出设备和实体
- 可选产出动作和卡片

### `action`

纯动作型插件。

适用场景：

- 调用某个第三方服务执行动作
- 对现有设备发送命令
- 风险操作审批后执行

### `channel`

消息和会话通道插件。

适用场景：

- Telegram
- 飞书
- 企业微信
- Discord

### `agent-skill`

对话代理可调用的工具能力插件。

特点是短执行链路、明确输入输出、可编排。

### `ai-provider`

模型或推理服务供应商接入插件。

### `region-provider`

地区、行政区和坐标解析插件。

### `locale-pack`

语言包资源插件。

### `theme-pack`

主题资源插件。

## 平台独占槽位

除了上面的普通插件类型，平台还保留三类独占槽位。

这些槽位也是插件，但不是开放式的普通扩展点，而是宿主控制的独占能力位。

### `memory_engine`

负责事件到记忆候选的抽取、去重、合并和重要度评估。

### `memory_provider`

负责记忆的索引、存储和检索。

### `context_engine`

负责把记忆结果组装成对话上下文 bundle。

### 槽位规则

- 同一宿主同一时间只能激活一个 `memory_engine`
- 同一宿主同一时间只能激活一个 `memory_provider`
- 同一宿主同一时间只能激活一个 `context_engine`
- 宿主必须始终保留内置 fallback 实现

## 现有插件的目标归位

当前仓库里的插件，后续目标类型应直接调整为：

- `official-weather` -> `integration`
- `health-basic-reader` -> `integration`
- `homeassistant` -> `integration + action`
- `open-xiaoai-speaker` -> `integration + action`
- `channel-*` -> `channel`
- `locale-zh-tw` -> `locale-pack`

如果现有插件还带有旧类型语义，应直接改 manifest 和实现，不保留旧路线。

## 插件能力声明

### Manifest 最低字段

每个插件 manifest 至少要声明：

- `id`
- `name`
- `version`
- `api_version`
- `types`
- `permissions`
- `entrypoints`
- `config_specs`
- `capabilities`

### `integration` 必须额外声明

- `domains`
- `instance_model`
- `refresh_mode`
- `supports_discovery`
- `supports_actions`
- `supports_cards`
- `entity_types`
- `default_cache_ttl_seconds`
- `max_stale_seconds`

### 槽位型插件必须额外声明

- `slot_name`
- `exclusive`
- `fallback_required`
- `input_contract_version`
- `output_contract_version`

### 示例：天气插件

```json
{
  "id": "official-weather",
  "name": "官方天气插件",
  "version": "1.0.0",
  "api_version": 1,
  "types": ["integration"],
  "permissions": ["device.read", "entity.write", "dashboard.card.write"],
  "capabilities": {
    "integration": {
      "domains": ["weather"],
      "instance_model": "single_instance_with_subitems",
      "refresh_mode": "polling",
      "supports_discovery": false,
      "supports_actions": false,
      "supports_cards": true,
      "entity_types": [
        "weather.condition",
        "weather.temperature",
        "weather.humidity"
      ],
      "default_cache_ttl_seconds": 1800,
      "max_stale_seconds": 7200
    }
  }
}
```

### 示例：记忆 provider 插件

```json
{
  "id": "lossless-memory-provider",
  "name": "Lossless 记忆后端",
  "version": "1.0.0",
  "api_version": 1,
  "types": ["memory_provider"],
  "permissions": ["memory.read", "memory.write.index"],
  "capabilities": {
    "memory_provider": {
      "slot_name": "memory_provider",
      "exclusive": true,
      "fallback_required": true,
      "input_contract_version": 1,
      "output_contract_version": 1
    }
  }
}
```

## 生命周期

插件不是只有“执行”这一个动作。

### 普通插件生命周期

对于普通插件，宿主统一按以下生命周期调用：

1. `validate_manifest`
2. `create_instance`
3. `configure_instance`
4. `discover`
5. `refresh`
6. `invoke_action`
7. `render_card_snapshot`
8. `diagnostics`
9. `teardown`

### 槽位插件生命周期

对于独占槽位插件，宿主统一按以下生命周期调用：

1. `validate_manifest`
2. `activate_slot`
3. `execute_slot_contract`
4. `diagnostics`
5. `deactivate_slot`

### 实现要求

- `integration` 至少实现 `refresh`
- 有发现能力的插件实现 `discover`
- 有控制能力的插件实现 `invoke_action`
- 有仪表盘接入的插件实现 `render_card_snapshot`
- `memory_engine` 至少实现记忆抽取主接口
- `memory_provider` 至少实现写入和查询主接口
- `context_engine` 至少实现上下文构建主接口

## 配置作用域

规范 v1 正式定义以下配置作用域：

- `plugin`
- `integration_instance`
- `device`
- `channel_account`

规则如下：

- `plugin` 用于插件全局默认值
- `integration_instance` 用于某个实例自己的连接参数和策略
- `device` 用于设备级可见配置
- `channel_account` 用于通道账号配置

不允许继续拿 `plugin` scope 假扮 `integration_instance` scope。

## 标准数据接口

### 原则

普通状态型插件不直接落库，只返回标准 DTO。

宿主负责：

- 校验
- 去重
- 落库
- 审计
- 回滚
- 状态聚合

### `integration.refresh` 标准输出

```json
{
  "devices": [],
  "entities": [],
  "discoveries": [],
  "card_snapshots": [],
  "actions": [],
  "diagnostics": []
}
```

### 标准设备 DTO

```json
{
  "unique_key": "weather:home-default",
  "device_kind": "virtual_household",
  "domain": "weather",
  "name": "家庭天气",
  "room_id": null,
  "metadata": {}
}
```

字段说明：

- `unique_key`：插件内唯一，不允许随便变
- `device_kind`：`physical` / `virtual_household` / `virtual_member`
- `domain`：领域名，例如 `weather`
- `metadata`：显示层和调试层可消费的补充信息

### 标准实体 DTO

```json
{
  "device_unique_key": "weather:home-default",
  "entity_key": "weather.temperature",
  "entity_type": "sensor",
  "value": 21.5,
  "unit": "C",
  "available": true,
  "degraded": false,
  "observed_at": "2026-03-18T12:00:00Z",
  "expires_at": "2026-03-18T12:30:00Z",
  "attributes": {}
}
```

### 标准发现 DTO

```json
{
  "resource_type": "device",
  "discovery_key": "speaker:living-room",
  "title": "客厅小爱音箱",
  "subtitle": "等待认领",
  "capability_tags": ["speaker", "voice"],
  "metadata": {}
}
```

### 标准卡片快照 DTO

卡片结构仍由宿主卡片规范控制。

插件只能返回卡片数据，不允许定义新的卡片协议。

### 标准动作 DTO

动作必须声明：

- `action_key`
- `label`
- `risk_level`
- `requires_confirmation`
- `payload_schema`

宿主负责鉴权和确认，插件只负责实际执行。

## 槽位型接口契约

独占槽位插件不走 `integration.refresh` 返回 DTO 这一套，而是走宿主保留接口契约。

### `memory_engine` 契约

至少包括：

- `ingest_event`
- `derive_memory_candidates`
- `merge_memory`
- `score_importance`

### `memory_provider` 契约

至少包括：

- `upsert_memory`
- `query_memory`
- `delete_memory`
- `rebuild_index`

### `context_engine` 契约

至少包括：

- `build_context_bundle`
- `summarize_memory_hits`
- `compress_context`

### 槽位边界

- 槽位插件无权最终决定记忆可见性
- 槽位插件无权绕过 revision
- 槽位插件无权删除宿主最小审计影子

## 错误语义和降级规则

### 标准错误码

平台统一使用以下错误语义：

- `plugin_disabled`
- `not_configured`
- `instance_not_found`
- `upstream_unavailable`
- `auth_failed`
- `rate_limited`
- `invalid_response`
- `partial_success`
- `stale_snapshot`

### 记忆槽位补充错误码

- `memory_engine_failed`
- `memory_provider_failed`
- `context_engine_failed`

### 降级原则

- 临时失败时保留最后一次成功状态
- 超过 `expires_at` 后标记 `stale`
- 只有明确不可恢复或配置错误时才打 `error`
- `memory_engine` 失败时保留事件记录
- `memory_provider` 失败时回退宿主默认 provider
- `context_engine` 失败时回退宿主默认 context engine

## 安全和权限边界

### 插件权限必须显式声明

插件只能获得声明过的权限。

未声明就没有。

### 插件不能直接拥有这些能力

- 任意数据库写权限
- 任意成员隐私读取权限
- 绕过宿主审计写入核心数据
- 绕过宿主确认执行高风险动作
- 绕过宿主权限裁决写入高敏感记忆

### 推荐的运行边界

- 默认优先子进程隔离
- 高风险插件默认禁用自动执行
- 外部网络访问建议声明 allowlist
- 所有敏感配置统一走宿主 secret 存储

## 版本兼容策略

项目当前不为仓库里的旧插件结构保留兼容层，但要为将来发布后的规范演进留出口。

### 平台要做的事

- 给插件 manifest 增加 `api_version`
- 给宿主保留规范升级路径
- 对废弃字段给出明确迁移窗口

### 插件要遵守的事

- 不能随便修改 `id`
- 不能随便改变 `unique_key` 生成规则
- 不能无迁移说明就修改实体语义
- 不能无迁移说明就修改槽位接口契约

## 明确禁止的做法

- 在宿主里新增单领域业务模块承接插件逻辑
- 插件直接写宿主核心业务表
- 用 `capabilities` 保存运行态实体状态
- 卡片绑定 city 文本而不是标准设备
- 用“页面特判”代替统一平台协议
- 把记忆权限、记忆修订和记忆 fallback 一起外包给第三方插件

## 落地建议

平台后续所有新能力，先问这四个问题：

1. 这是不是宿主内核能力？
2. 这是不是某个外部领域接入？
3. 这是不是一个普通插件类型就够了？
4. 如果是记忆相关能力，它是不是应该进独占槽位而不是普通插件类型？

如果答案是“领域接入”，默认先做插件，不要先进宿主。
