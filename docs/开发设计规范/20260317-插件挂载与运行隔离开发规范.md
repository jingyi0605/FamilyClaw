# 20260317-插件挂载与运行隔离开发规范

状态：Draft

## 这份规范解决什么问题

插件系统最容易犯的错，不是“插件功能写坏了”，而是“一个插件写坏了，整个后端跟着陪葬”。

常见表现：

- 一个 `manifest.json` 写错，插件列表接口直接 500
- 一个语言包资源坏了，语言列表接口直接 500
- 一个挂载插件路径失效，地区 provider 同步链路直接炸
- 某个入口自己重新读一遍插件文件，没有做隔离，结果把前面已经修好的容错又绕开了

这不是插件坏了这么简单，这是系统边界没守住。

这份文档只规定一件事：

以后扩展插件系统时，单个插件的挂载错误、声明错误、运行错误、资源错误，都不能直接把查询链路、管理链路、执行链路一起拖死。

## 先问三个问题

开始写插件相关代码前，先问：

1. 这是在执行插件，还是在读取插件元数据？
2. 这里遇到坏插件时，应该“拒绝这一个插件”，还是“拒绝整个请求”？
3. 这个异常如果直接往上抛，会不会把无关用户请求打成 500？

如果第三个问题的答案是“会”，那这段设计大概率就是错的。

## 一句话总规则

坏插件只能影响它自己，不能成为系统控制流。

翻成人话：

- 坏插件可以被跳过
- 坏插件可以被禁用
- 坏插件可以被记录日志
- 坏插件可以让当前这一次执行失败
- 但坏插件不能让插件目录查询、挂载列表、语言列表、家庭信息、地区上下文这些公共读链路一起崩

## 故障分层

以后所有插件故障，只能落到下面四层里的某一层，不能乱窜。

### 第一层：发现层故障

典型问题：

- `manifest.json` 不存在
- `manifest.json` 不是合法 JSON
- manifest 字段校验失败
- 挂载记录里的 `plugin_id` 和 manifest 里的 `id` 不一致

处理规则：

- 记录错误日志
- 跳过当前插件
- 继续处理其他插件

不允许：

- 直接把异常抛到接口层
- 因为一个插件坏了，整个注册表构建失败

### 第二层：资源层故障

典型问题：

- locale 资源文件不存在
- locale JSON 损坏
- dashboard snapshot payload 结构不合法
- 插件配置持久化数据损坏

处理规则：

- 如果是可跳过资源，就跳过当前资源并记日志
- 如果是可降级展示，就返回空状态、错误状态或默认值
- 不要把资源文件损坏升级成整个接口失败

### 第三层：执行前校验故障

典型问题：

- 插件不存在
- 插件已禁用
- 插件类型不匹配
- trigger 不支持
- 家庭看不到这个插件

处理规则：

- 统一转成业务错误
- 返回明确的错误码和状态码
- 不进入实际执行

这类错误是“当前请求不能执行”，不是“系统崩了”。

### 第四层：执行期故障

典型问题：

- entrypoint 导入失败
- runner 启动失败
- 插件抛异常
- runner 超时
- 插件输出结构非法

处理规则：

- 收敛成执行失败结果
- 写运行日志
- 当前这次任务失败
- 不要把进程级异常继续往外炸成全局 500

## 发现层开发规则

### 1. 只有一个地方负责“发现插件”

内置插件和家庭挂载插件的发现逻辑必须统一收口。

不要在下面这些地方自己重新写一套：

- 插件列表接口
- 集成目录接口
- 通讯账号入口
- 地区 provider 同步
- dashboard 读卡片

正确做法：

- 统一先走插件注册表快照
- 其他模块只消费快照结果

### 2. 发现层允许跳过坏插件，不允许抛致命异常

读取插件目录、构建家庭插件快照时，如果发现坏插件：

- 打日志
- 带上 `household_id`、`plugin_id`、`manifest_path`、`operation`
- 跳过当前插件

不要因为“我想让问题暴露得明显一点”就直接抛异常。

那不是暴露问题，那是在扩大故障面。

### 3. 挂载记录和 manifest 必须双向校验

只要是挂载插件，就必须检查：

- 挂载记录里的 `plugin_id`
- manifest 里的 `id`

这两个不一致时：

- 认定为坏挂载
- 记录日志
- 跳过当前挂载

不要继续假装它是正常插件。

## 查询链路开发规则

### 1. 查询接口默认必须容错

下面这些接口默认都属于“公共读链路”：

- 插件注册表列表
- 家庭插件列表
- 挂载插件列表
- 语言包列表
- 集成目录
- 家庭地区上下文

公共读链路的原则：

- 单个插件坏了，只影响这一个插件的可见性
- 不能影响其他插件
- 不能影响家庭、地区、集成等外围功能继续读取

### 2. 查询链路里不要直接读文件后裸抛异常

危险写法：

```python
manifest = load_plugin_manifest(path)
return build_xxx(manifest)
```

如果这里是公共读链路，这种写法基本就是错的。

正确思路：

```python
manifest = safe_load(...)
if manifest is None:
    continue
```

### 3. 资源型查询要支持“跳过当前资源”

例如 locale 资源、dashboard 资源、配置 schema 附属资源：

- 当前资源坏了，只跳过当前资源
- 同一插件下其他资源继续处理
- 其他插件继续处理

### 4. 开发覆盖态必须显式暴露，不能让前端靠猜

如果 `plugins-dev` 里的插件已经实际进入家庭插件注册表，后端查询结果必须显式告诉前端“当前跑的是开发版”。

最低要求：

- 用明确字段表达，例如 `is_dev_active`
- 这个字段表示“当前运行态来自 `plugins-dev`”，不是“磁盘上存在开发源码”
- 前端插件列表和详情直接显示“开发插件生效中”

不要让前端再去猜：

- `manifest_path` 是不是落在 `plugins-dev`
- `install_method=local` 是不是就等于开发版
- 有没有同名安装插件

这些猜法全是脆弱补丁。真实运行态应该由后端一次说清楚。

## 执行链路开发规则

### 1. 执行前必须先做“可见性 + 可用性”检查

统一要求：

- 查询用 `get_household_plugin(...)`
- 执行用 `require_available_household_plugin(...)`

不要自己拼：

- “先查一遍 registry 再 if enabled”
- “先查 mount 表再判断类型”

规则统一，才能避免不同入口行为不一致。

### 2. 执行函数必须返回失败结果，不能随便把异常炸出去

执行链路至少要收敛下面这些异常：

- 插件业务错误
- manifest 校验错误
- 模块导入错误
- 类型错误
- runner 错误
- 其他未预期异常

返回值应该是：

- 成功结果
- 或失败结果

而不是直接把异常扔给上层 HTTP 路由。

### 3. 执行前构建运行时上下文时，也要遵守隔离规则

例如：

- 家庭地区上下文
- 账号配置
- 调度上下文
- 插件系统上下文

这些步骤本质上还是“执行前准备”。

不要因为它不是真正的 runner 执行，就把异常处理偷懒了。

## 地区 provider 和其他插件桥接规则

### 1. 桥接模块不要自己重复实现 manifest 解析

像地区 provider、channel、集成、scheduler 这类桥接模块，经常会想“我就读一下 manifest，不复杂”。

这是最容易把故障隔离绕开的地方。

规则很简单：

- 不要自己手写第二套 manifest 解析
- 不要绕开插件服务层的容错入口

如果桥接模块确实要按挂载记录逐个处理插件：

- 也必须复用统一的安全加载逻辑
- 遇到坏挂载时只跳过当前项

### 2. 同步类链路必须按“每个挂载一条隔离边界”处理

比如：

- 地区 provider 注册
- 周期轮询准备
- 通道账号派发
- 集成同步候选项构建

每处理一个挂载插件，都必须是独立边界：

- 当前挂载出错，记录日志
- 继续处理下一个挂载

不要把整批同步写成“一条异常全退出”。

## 日志规则

### 1. 日志必须带上下文

至少带这些字段：

- `operation`
- `household_id`（如果有）
- `plugin_id`
- `manifest_path` 或资源路径
- `error`

### 2. 同一问题要去重

坏插件经常会被重复读到。

如果不去重：

- 日志会刷爆
- 真正的新问题会被淹没

所以对同一类故障要做一次性告警或有限频率告警。

### 3. 日志是告警，不是控制流

不要写这种逻辑：

- 先记日志
- 然后还是把异常往上抛

如果当前链路本来就该容错，那日志写完就该继续走。

## 新增插件功能时的开发清单

以后只要新增下面任一能力，都必须过这份清单：

- 新插件入口
- 新插件查询接口
- 新插件挂载能力
- 新 runner
- 新资源文件类型
- 新插件桥接模块

检查项：

1. 这个入口是在“查询插件”还是“执行插件”？
2. 单个插件坏了以后，最小影响范围是什么？
3. 是否复用了统一的插件发现结果？
4. 是否重复手写了 manifest 解析？
5. 是否把资源错误限制在当前资源内？
6. 是否把执行错误限制在当前执行内？
7. 是否写了坏插件场景的测试？

如果第 4 条答案是“是”，基本就该停下来重构。

## 必测回归场景

每次改插件系统，至少回归下面这些场景：

1. 一个内置插件 manifest 非法，插件列表接口仍可用
2. 一个挂载插件 manifest 非法，挂载列表接口仍可用
3. 一个挂载插件 manifest 非法，家庭地区上下文仍可读取
4. 一个 locale 资源文件非法，locale 列表接口仍可用
5. 一个插件执行 entrypoint 导入失败，只影响当前执行结果
6. 一个被禁用插件在执行入口被正确拦截，返回统一错误语义
7. 一个坏挂载不会阻塞其他正常挂载继续注册、同步或展示

## 禁止事项

明确禁止下面这些写法：

- 在公共查询接口里直接 `load_plugin_manifest()` 然后裸抛异常
- 在桥接模块里自己写第二套 manifest 解析
- 用“一个异常退出整个循环”处理批量插件
- 把坏插件的文件错误当成 HTTP 500 的理由
- 让插件资源文件损坏影响家庭、地区、集成等外围基础功能

## 一句话结论

插件系统的隔离边界应该是“单插件、单资源、单次执行”。

谁把边界扩大到“整个请求”甚至“整个后端”，谁就在制造系统级 bug。
