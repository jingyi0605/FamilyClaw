# 20260317-插件启用禁用统一规则

状态：Draft

## 这份规范解决什么问题

现在插件系统最容易出错的地方，不是有没有启用/禁用状态，而是同一个状态在不同入口语义不一样：

- 插件管理里已经禁用了插件
- 某些新增入口里还是能继续选这个插件
- 某些执行链路会拦，某些链路又不拦
- 有的地方报 409，有的地方报 404，有的地方干脆不提示原因

这不是功能缺失，这是规则没统一。

这份文档只解决一个问题：

插件启用/禁用的语义必须全局一致，新增入口、执行入口、存量对象展示和错误返回都按同一套规则走。

## 先问三个问题

开始改插件状态相关逻辑前，先问：

1. 这是个真问题还是假问题？
2. 有没有更简单的方法？
3. 会破坏什么吗？

如果只是某个页面自己补一个 `if (!enabled)`，那不是解决问题，那是在复制下一个 bug。

## 统一状态源

插件是否可用，只认统一状态：

- `PluginRegistryItem.enabled`
- `PluginRegistryItem.disabled_reason`

不要在页面里自己猜，不要在业务模块里自己再拼一套“半启用”状态。

谁负责算最终状态：

- 基础状态
- 家庭级覆盖状态
- 合并后的最终可用状态

这些统一由插件模块维护，其他模块只消费结果，不重复发明规则。

## 禁用的准确定义

禁用的唯一准确定义是：

禁用 = 不允许任何新的业务使用和任何新的执行触发，但保留查看、排障、重新启用所需的管理能力。

翻成人话就是：

- 不能新增使用
- 不能继续执行
- 不能继续自动跑
- 但可以看
- 但可以改配置
- 但不能假装它还在正常服务

## 新增使用入口规则

任何“新增选择插件”的入口，都只能允许选择 `enabled=true` 的插件。

包括但不限于：

- 设备与集成里新增实例
- 通讯平台里新增账号
- 计划任务里选择插件目标
- 后续任何插件选择器、插件下拉框、插件目录弹窗

规则很简单：

- 禁用插件默认不应出现在可选列表里
- 如果业务上必须保留在列表里，也必须明确标记禁用，并且不可选
- 不允许前端只做展示过滤，后端不校验

## 存量对象规则

已存在的对象不删除、不隐藏：

- 集成实例
- 通讯账号
- 计划任务
- 历史记录
- 日志
- 配置

但它们必须满足下面几条：

- 明确显示“插件已禁用”
- 明确显示 `disabled_reason`
- 所有执行类按钮禁用
- 不要把禁用对象伪装成“只是暂时失败”

一句话：

对象可以保留，服务能力不能假装还活着。

## 执行类入口规则

任何会让插件真正跑起来的入口，必须统一走可用性检查。

统一要求：

- 统一调用 `require_available_household_plugin(...)`

适用范围：

- 手动执行
- Agent 调用
- 设备控制
- 同步动作
- 计划任务触发
- worker 重试
- 后台自动触发
- webhook / 回调 / 定时调度转成插件执行的链路

不要再出现下面这种垃圾状态：

- 页面上能点
- 后端里部分方法能跑
- 另一个入口又拦住

执行语义必须只有一套。

## 配置类入口规则

禁用后允许继续查看和修改配置。

原因很实际：

- 需要排障
- 需要补配置后重新启用
- 需要保留存量对象的管理能力

但配置类入口只能做配置类动作，不能偷偷混入执行动作。

禁用时明确不允许：

- 测试连接
- 立即同步
- 立即执行
- 设备控制
- 任何“点一下就会真正调用插件”的动作

## 自动链路规则

禁用不仅拦 UI，还要拦后端实际执行。

尤其要注意下面这个边界：

- 任务在禁用前已经入队
- 真正执行发生在禁用后

这种情况也必须在执行前再次校验，并拒绝运行。

不要把“入队时校验过一次”当成免死金牌。
真正执行前不复查，禁用就只是个摆设。

## 错误语义规则

已存在但被禁用，统一按下面的语义返回：

- HTTP 状态码：`409`
- 错误码：`plugin_disabled`

不要出现下面这些乱象：

- 有的地方报 404，像是插件不存在
- 有的地方报 400，像是参数错了
- 有的地方直接吞掉
- 有的地方前端自己拼一句模糊文案

禁用不是“没找到”，也不是“参数非法”，就是“当前不可用”。

## UI 反馈规则

前端只要遇到插件禁用，必须做到两件事：

1. 禁掉不能继续做的动作
2. 明确展示 `disabled_reason`

不允许只灰掉按钮不解释。

用户不是来猜谜的。

## 状态矩阵

### 插件管理页

- 可见
- 可查看详情
- 可切换启用/禁用

### 插件配置页

- 可见
- 可修改配置

### 新增实例 / 新增账号 / 新增绑定

- 禁用时不允许新增

### 同步 / 控制 / 执行 / 计划任务 / Agent 调用

- 禁用时不允许执行

### 已有实例和历史数据

- 保留显示
- 必须标记为禁用状态

### 后台 worker / 重试 / 自动触发

- 执行前再次校验
- 禁用时不运行

## 后端实现边界

后端实现时，统一按下面这条分工：

- `get_household_plugin(...)` 用于查看、读取、配置类场景
- `require_available_household_plugin(...)` 用于任何执行类场景

如果某条链路会让插件真正跑起来，却还在用 `get_household_plugin(...)`，那就是实现错了。

## 前端实现边界

前端实现时，统一按下面这条分工：

- 新增选择器只展示可新增使用的插件
- 存量对象页面展示禁用状态和禁用原因
- 执行按钮不要自己猜规则，优先消费后端返回的 `disabled` / `disabled_reason` / 插件状态字段

不要每个页面自己定义一套“什么时候能点”。

## 回归检查清单

改任何插件启用/禁用相关逻辑后，至少回归下面这些场景：

1. 在插件管理里禁用插件后，新增入口里不能再选到它
2. 直调后端新增接口时，禁用插件返回 `409/plugin_disabled`
3. 已存在的实例或账号仍然能看到
4. 已存在的实例或账号会显示禁用原因
5. 同步、控制、执行按钮被禁用
6. 配置页仍然可以打开和保存
7. 禁用前已入队的任务在真正执行时会被再次拦截
8. 错误码和状态码在不同入口保持一致

## 一句话规则

以后项目里统一按这句执行：

插件禁用 = 保留对象和配置，但禁止任何新的使用与任何新的执行。
