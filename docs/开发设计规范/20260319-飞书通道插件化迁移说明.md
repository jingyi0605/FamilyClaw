# 20260319-飞书通道插件化迁移说明

## 这次改动到底做了什么

这次不是单纯修一个飞书收不到回复的 bug，而是把飞书通道往“核心只保留抽象能力，平台细节都下沉到插件”这个方向推进了一步。

当前已经落地的边界是：

- 飞书长连接建立、保活、ACK、重连都在 `channel-feishu` 插件内实现
- 飞书 webhook 和长连接收到的消息，都先在插件里归一化
- 飞书的未绑定用户识别、未绑定私聊回复、候选绑定消息落库，都在插件里处理
- 已绑定消息由插件写入绑定决策，再交给核心桥接层继续走通用会话流程

换句话说，飞书特有逻辑现在已经前移到插件，核心不再直接负责飞书平台成员识别。

## 为什么这次还改了核心

理想终态当然是核心完全不碰平台绑定逻辑。

但这次迁移要先保证飞书能正常工作，而仓库里其他通讯渠道还没有全部迁完。如果现在把核心旧逻辑一刀砍掉，别的渠道会被直接打断。

所以这次只做了一个最小过渡改动：

- 核心桥接层先读取插件塞进来的 `normalized_payload.metadata.plugin_binding`
- 如果插件已经给出绑定决策，核心直接使用，不再重复做飞书自己的匹配
- 如果插件没有给出绑定决策，核心仍然走旧的 fallback，继续兼容还没迁移完成的渠道

这个过渡入口目前在：

- `apps/api-server/app/modules/channel/conversation_bridge.py`

文件里已经留了明确 TODO，说明这不是最终设计。

## 当前插件侧承担的飞书逻辑

飞书插件当前负责下面这些事：

- 建立账号级长连接
- 拉取飞书 WebSocket endpoint
- 处理 protobuf frame
- 收包后做 ACK
- 心跳和断线重连
- 统一解析 webhook / 长连接消息
- 识别私聊和群聊
- 判断当前发信人是否已经绑定家庭成员
- 未绑定私聊自动回复“请联系管理员先完成绑定”
- 未绑定消息写入候选绑定事件
- 已绑定消息在 metadata 中写入插件绑定结果

这些逻辑都应该继续留在飞书插件里，不要再回流到核心。

## 后续必须继续处理的事

这次只是迁移中的过渡，不是收工。

后续当其他通讯渠道也完成同样的插件化下沉后，必须做下面这一步：

- 删除 `conversation_bridge.py` 里的核心绑定 fallback

删掉以后，核心应只保留这些通用能力：

- 接收插件产出的标准化入站事件
- 根据插件给出的绑定决策进入会话桥接
- 管理通用的会话、消息、投递记录

核心不应该再承担任何具体平台的成员匹配实现。

## 现在排障先看哪些日志

这次补了一组统一前缀日志：

- 关键词：`feishu-debug`

排障时按下面顺序看：

1. 先看插件是否收到消息
   - `webhook_received`
   - `ws_event_queued`
   - `ws_poll_drain`
2. 再看插件是否把消息挡在前面
   - `binding_gate_enter`
   - `binding_gate_bound`
   - `binding_gate_unbound_recorded`
   - `binding_gate_direct_reply_sent`
3. 最后看核心桥接有没有继续处理
   - `bridge_enter`
   - `bridge_binding_resolved`
   - `bridge_command_dispatched`
   - `bridge_message_dispatched`

如果连 `webhook_received` / `ws_event_queued` 都没有，问题就在飞书入站之前。

如果有 `binding_gate_enter` 但没有 `bridge_enter`，说明消息被插件侧截住了，重点看未绑定判断、落库和自动回复。

如果已经有 `bridge_enter`，但没有后续 `bridge_message_dispatched` / `bridge_command_dispatched`，问题就在核心桥接或后续会话处理链路。
