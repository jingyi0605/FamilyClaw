# 数据库迁移规范

这份文档只解决一件事：FamilyClaw 以后怎么改数据库结构，谁来改，怎么验证改完没有把库搞脏。

## 当前基线

- 主库统一使用 PostgreSQL。
- 测试库也统一使用 PostgreSQL。
- SQLite 已退出运行时、测试和迁移基线，不再作为兜底方案。

## 硬规则

1. 只有 Alembic 可以修改表结构。
2. 禁止在应用启动、业务代码、测试初始化脚本里使用 `create_all()`、`drop_all()`、自动补表、自动补列。
3. 新增表、字段、索引、约束时，必须同时提交 Alembic migration。
4. model 只是当前结构定义，不负责把旧库自动变成新库。

如果有人绕开这几条，结果通常都一样：表已经偷偷变了，但 `alembic_version` 没跟上，后面的升级直接炸。

## 正常流程

### 1. 先改 model

- 先把 SQLAlchemy model 改对。
- 改之前先想清楚会不会破坏旧数据、旧接口和现有查询。

### 2. 再写 migration

- 每次 schema 变更都新建一个 revision。
- 文件名说人话，直接说明这次改了什么。
- 如果出现多个 head，单独补一个 merge revision，不要把业务变更混进去。

### 3. 自查 migration

重点检查：

- `upgrade()` 和 `downgrade()` 是否对称。
- 表名、列名、索引名、外键名有没有写错。
- PostgreSQL 下的默认值、约束名长度、部分索引条件是否正确。

### 4. 本地验证

至少做两类验证：

1. 对一个全新的 PostgreSQL 空库执行 `alembic upgrade head`。
2. 对一个旧版本 PostgreSQL 库执行增量升级，确认链路不断。

### 5. 测试只走 migration

- 测试初始化数据库时，先跑 migration 到 `head`。
- 不允许再用 `Base.metadata.create_all()` 假装真实建库流程。

## 脏库处理

脏库的典型症状是：表已经存在，但 `alembic_version` 不在正确 revision。

处理顺序：

1. 先核对现有表结构是否和目标 revision 一致。
2. 一致时，用 `alembic stamp <revision>` 校准版本号。
3. 再执行 `alembic upgrade head`。

不要在没核对结构前就直接 `stamp`，那只是把问题藏起来。

## 怎么算真的做完

- PostgreSQL 空库可以成功执行 `alembic upgrade head`。
- PostgreSQL 旧库可以成功升级到 `head`。
- 仓库里搜不到业务代码或测试代码里的 `create_all()`、`drop_all()`。
- 当前数据库结构和 `alembic_version` 对得上。

## 出问题时怎么查

按这个顺序查：

1. 当前 `alembic_version` 是多少。
2. 失败的 revision 想改什么。
3. 对应表、列、索引是不是已经被别人手工改过。
4. 有没有代码绕过 Alembic 偷偷改库。
5. revision 图是不是出现了多 head 但没合并。

先查事实，再改代码。不要靠猜。
