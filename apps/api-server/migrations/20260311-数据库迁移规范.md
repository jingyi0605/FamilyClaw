# 数据库迁移规范

这份文档只解决一件事：以后数据库表怎么增量更新，谁有权改，出了问题怎么排查。

## 先定死一条规则

- 只有 Alembic 可以改数据库表结构。
- 应用启动时不允许 `create_all()`、`drop_all()`、自动补表、自动补列。
- SQLAlchemy model 只是当前结构定义，不负责把旧库变成新库。

如果有人绕过这条规则，结果通常只有一个：表已经变了，但 `alembic_version` 没跟上，后面升级直接炸。

## 这一步到底做什么

- 新增表、字段、索引、约束时，必须新建一个 migration。
- 老数据库升级时，只跑 `alembic upgrade head`。
- 应用本身默认假设数据库已经迁移完成，不在启动阶段偷偷修库。

## 做完以后能看到什么结果

- 本地、测试、线上都走同一条升级路径。
- 新库和旧库都能通过 migration 到达同一个结构。
- `alembic_version` 能准确表示数据库现在处于哪个版本。

## 依赖什么

- `alembic.ini`
- `migrations/env.py`
- `migrations/versions/*.py`
- 启动脚本里的 `alembic upgrade head`

## 主要改哪些文件

- 表结构变更：`app/modules/**/models.py`
- 迁移脚本：`migrations/versions/*.py`
- 启动流程：`start-api-server.sh`
- 测试初始化：`tests/**`

## 明确不做什么

- 不在接口启动时自动建表。
- 不在业务代码里顺手补列、补索引。
- 不手工改线上库结构后假装没事。
- 不把 ORM model 变更当成数据库已经升级完成。

## 标准流程

### 1. 改 model

- 先改 SQLAlchemy model。
- 改之前先想清楚会不会破坏旧数据、旧接口、旧逻辑。

### 2. 新建 migration

- 每次 schema 变更都新建一个 revision。
- 文件名直接写人话，比如“新增账户基础表”“给 households 增加 city 字段”。
- 如果出现多分支 head，合并时单独建 merge revision，不要把业务改动塞进去。

### 3. 检查 migration 内容

- 重点看 `upgrade()` 和 `downgrade()`。
- 确认表名、列名、索引名、外键名没有写错。
- SQLite 下尤其注意存在性判断，别重复建表、重复建索引。

### 4. 本地升级验证

- 执行 `alembic upgrade head`。
- 用一个全新的空库跑一次。
- 再用一个旧版本库升级一次，确认增量路径没断。

### 5. 测试只走 migration

- 集成测试初始化数据库时，优先执行 migration 到 `head`。
- 不再使用 `Base.metadata.create_all()` 模拟真实数据库。

## 多数据库时怎么做

如果以后不止一个数据库，不要把所有库硬塞进同一条迁移链。

- 每个物理数据库单独维护一套 Alembic 配置和 `versions` 目录。
- 每个数据库各自维护自己的 `alembic_version`。
- 启动脚本按数据库逐个迁移，哪个失败就直接停。

简单说：一个数据库一套迁移历史，别混账。

## 历史脏库怎么处理

脏库的典型特征：表已经存在，但 `alembic_version` 还没到对应 revision。

处理顺序：

1. 先确认现有表结构和目标 migration 一致。
2. 一致时，用 `alembic stamp <revision>` 校准版本。
3. 再执行 `alembic upgrade head`。

不要在没核对结构前直接 stamp，不然只是把脏状态藏起来。

## 怎么验证是真的做完了

- 空数据库执行 `alembic upgrade head` 能成功。
- 一个旧版本数据库升级到 `head` 能成功。
- 应用启动时不再出现 `create_all()`。
- 全仓搜索不到业务代码里的 `create_all()`、`drop_all()`。
- 数据库当前结构和 `alembic_version` 对得上。

## 排查顺序

如果以后再出现迁移失败，按这个顺序查：

1. `alembic_version` 现在是多少。
2. 失败的 revision 想改什么。
3. 对应表、列、索引是不是已经被别人提前建过。
4. 有没有人在启动代码、测试代码、脚本里偷偷用了 `create_all()`。
5. 当前 revision 图是不是出现了多 head 但没合并。

先查事实，再改代码。不要靠猜。
