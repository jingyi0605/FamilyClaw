# 20260314-插件文档更新执行计划

这份文档只做一件事：把“哪些插件相关文档现在必须更新、每个文件到底改什么”说清楚，避免后面代码已经切到后台任务了，文档还停在旧世界。

## 先说结论

现在插件系统已经发生了三件大变化：

1. 插件执行默认先创建后台任务，不再把同步执行当对外主路径。
2. 第三方插件依赖按插件自己的环境处理，不再默认装进主 API 环境。
3. 第三方插件运行边界按同容器子进程 runner 收口，而不是继续混在主进程里。

所以后续文档更新的目标也很明确：

- 把“同步直接执行插件”的旧说法换掉。
- 把“后台任务 + worker + 查询 + 通知 + 响应”的新主链路写清楚。
- 把“独立依赖 + 子进程 runner”的开发者心智模型写稳。

## 按文件写清楚要改什么

### 1. `specs/004.2-插件系统与外部能力接入/README.md`

要改什么：

- 在“这次要做到什么”里明确写：插件执行对外主路径已经变成后台任务。
- 补一句：接口和 Agent 入口主要负责建任务，不再承诺同步返回最终插件执行结果。
- 在文档末尾加一个跳转，明确 004.2.1 是运行可靠性和任务机制的补充 Spec。

改完以后别人能看到什么：

- 一进插件系统总入口就知道，现在不是“调用接口 = 直接执行插件”，而是“调用接口 = 创建插件任务”。

### 2. `specs/004.2-插件系统与外部能力接入/design.md`

要改什么：

- 把“插件运行层”改写成两层：
  - 对外创建任务层
  - 后台 worker 执行层
- 在“数据进入系统的主链路”里补一个任务化版本：
  1. 创建 `plugin_job`
  2. worker claim 任务
  3. 调 `connector` / `action` / `agent-skill`
  4. 保存原始记录或动作结果
  5. 必要时写记忆
  6. 更新任务状态和通知
- 补运行失败、超时、重试、人工响应的边界说明。
- 明确区分：`execute_plugin()` 是内部执行入口，不是推荐对外 API 语义。

改完以后别人能看到什么：

- 插件运行模型从“统一同步调用”升级成“后台任务 + worker + 状态流转”的正式设计。

### 3. `specs/004.3-插件开发规范与注册表/README.md`

要改什么：

- 明确第三方插件的运行过程不是“HTTP 直接拉起 runner”，而是“先建任务，再由 worker 触发 runner”。
- 增加一句：注册表文档管的是插件来源、结构、审核和 runner 边界，不直接替代后台任务文档。

改完以后别人能看到什么：

- 知道注册表和后台任务是两层，不会把它们混成一件事。

### 4. `specs/004.3-插件开发规范与注册表/design.md`

要改什么：

- 在第三方 runner 设计章节补全这条链：
  - `plugin_job` 创建
  - worker 领取
  - 根据来源和后端选择 `in_process` 或 `subprocess_runner`
  - runner 在插件独立环境里执行
  - 回传结构化结果
  - 收口到任务状态和尝试记录
- 把“依赖独立安装”写得更死一点：
  - 插件依赖归插件环境
  - 主 API 环境只保留平台公共依赖
- 补“安装”和“执行”不是一回事：
  - 现在不做自动下载安装
  - 但文档必须说明人工准备 venv、`python_path`、`working_dir` 的要求

改完以后别人能看到什么：

- 子进程 runner、独立依赖、后台任务三件事会被写成一条完整运行时链，而不是散落几段说明。

### 5. `docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`

要改什么：

- 在“先说边界”里补一条：对外插件执行默认返回 job，不保证同步结果。
- 在“第一版最小开发流程”里加一步：开发者要学会看任务状态、通知和失败记录，不只盯接口返回体。
- 在“现有运行时边界”里明确 builtin 与 third-party 的共同点：对外都先走后台任务。

改完以后别人能看到什么：

- 一开始就形成正确心智：不是“写个插件然后接口立刻跑完”，而是“写个插件然后进入受控后台执行链”。

### 6. `docs/开发者文档/插件开发/zh-CN/02-插件开发环境与本地调试.md`

要改什么：

- 把“服务层真实函数”分成两类：
  - 内部执行调试：`execute_plugin()`、`run_plugin_sync_pipeline()`
  - 正式对外验证：`POST /api/v1/plugin-jobs`、任务查询、worker 执行结果
- 增加一个最小调试顺序：
  1. 直接 import 函数
  2. 在插件自己环境里验证依赖
  3. 创建后台任务
  4. 看任务状态、通知、失败记录
- 说明 `python_path`、`working_dir`、插件 venv 在挂载和 runner 调试时分别有什么用。

改完以后别人能看到什么：

- 知道哪些命令是内部调试工具，哪些才是模拟真实系统行为的主入口。

### 7. `docs/开发者文档/插件开发/zh-CN/05-插件对接方式说明.md`

要改什么：

- 这是最重要的一个，整篇要按新链路改。
- 把三条主链路都改成“先入队，再执行”：
  - 数据同步链路
  - Agent 调用链路
  - 动作执行链路
- 补接口说明：
  - `POST /api/v1/plugin-jobs`
  - `GET /api/v1/plugin-jobs/{job_id}`
  - `GET /api/v1/plugin-jobs`
  - `POST /api/v1/plugin-jobs/{job_id}/responses`
- 加一节专门解释：
  - 什么时候直接看 job
  - 什么时候看最新 attempt
  - 什么时候会收到 `plugin.job.updated`
- 把旧的“同步返回 output / pipeline_result”的示例改成“先返回 job，再查询终态”。

改完以后别人能看到什么：

- 这份文档能真正反映当前系统对接方式，而不是停在旧的同步执行时代。

### 8. `docs/开发者文档/插件开发/zh-CN/06-从零开发一个可运行插件.md`

要改什么：

- 把“走一次完整同步调用”改成“创建一次后台任务并跟到终态”。
- 原来 `plugin-memory-checkpoint` 直接返回 `pipeline_success`、`raw_record_count`、`memory_card_count` 的示例，要改成：
  - 先返回 `queued=true`、`job_id`
  - 再通过任务查询或通知看到终态
- 新增一个“怎么判断任务真正执行完成”的章节，教开发者看：
  - job.status
  - latest_attempt.status
  - 失败错误码
  - recent_notifications

改完以后别人能看到什么：

- walkthrough 不会再误导开发者以为一个接口调用就能立刻拿到完整同步结果。

### 9. `docs/开发者文档/插件开发/zh-CN/07-插件测试与项目内运行验证.md`

要改什么：

- 把“runner 还没实现前”的表述删掉或改掉，因为现在 runner 和后台任务链已经在系统里站住了。
- 重新划分测试层次：
  1. manifest / 入口静态检查
  2. 内部执行函数测试
  3. 后台任务接口测试
  4. worker / 重试 / 超时 / 恢复测试
  5. 查询 / 通知 / 响应测试
- 把推荐命令补充到 `tests.test_plugin_jobs`
- 把验收标准从“跑通 `run_plugin_sync_pipeline()`”升级成“能走完创建任务 -> 执行 -> 查询/通知/响应”。

改完以后别人能看到什么：

- 测试文档终于会和当前代码一致，不再把内部函数当成唯一项目内运行标准。

### 10. 英文镜像文档

要改什么：

- 对应中文文件逐一同步，不允许只改中文不改英文。
- 重点同步这些词：

  - worker claim

  - plugin dependencies live in plugin environment

改完以后别人能看到什么：

- 中英文规则一致，不会出现中文文档说后台任务，英文文档还在讲同步执行的烂情况。

### 11. `specs/004.4-用户前端插件管理与插件市场/*`

要改什么：

- 检查前端交互是不是还在默认“点击执行就立刻拿结果”。

  - 任务列表
  - 任务详情
  - 失败提示
  - 待确认动作
  - 任务状态刷新或订阅
- 如果暂时不做，也要在文档里明确“当前前端仍未接完整任务交互，不要把同步执行当默认模型”。

改完以后别人能看到什么：

- 前端 spec 不会再和后端任务模型打架。

## 改完后的验收标准

这批文档改完以后，至少要满足这 6 条：

1. 不再把同步执行写成对外主路径。
2. 明确写出后台任务创建、worker 执行、查询、通知、响应的完整链路。
3. 明确第三方插件依赖装在插件环境，不装进主 API 环境。
4. 明确第三方插件默认通过子进程 runner 执行，不直接 import 到主进程。
5. 中文和英文文档内容一致，不互相打架。
6. 前端、后端、Spec、开发者文档对“插件怎么跑”这件事讲的是同一套话。

## 这份计划明确不做什么

- 不在这里直接改完所有文档。
- 不在这里展开插件市场 UI 细节。
- 不在这里补新的运行时能力设计。

这份计划只是把后续文档补齐工作排成一张可执行清单，避免继续边写代码边让文档腐烂。
