# 需求文档 - 插件后台任务与可靠执行

状态：Draft

## 简介

插件已经能执行，但执行方式还停留在“接口进来，立刻跑插件，尽量当场给结果”的阶段。这种方式对短任务还能凑合，对长任务、失败任务、需要人工确认的任务就是灾难。

这份需求文档要把问题收口成一套正式能力：插件执行变成后台任务，接口只负责创建任务和查询任务；真正执行、失败通知、人工响应都走统一流程。

这套能力主要服务三类角色：

- 前端页面：需要马上拿到任务编号，再持续看状态
- 家庭成员或管理员：需要知道任务成功没有、失败了为什么、接下来能做什么
- 后端系统和 Agent：需要稳定调用插件，不因为一次慢任务把整个请求链拖死

## 术语表

- **System**：FamilyClaw API Server
- **插件任务**：一次插件执行请求对应的一条后台任务记录
- **任务状态**：任务当前处于排队、执行中、成功、失败、待响应等哪一种状态
- **任务响应**：用户或系统对失败任务、待确认任务做出的后续动作，比如重试、确认、取消、补参数
- **通知事件**：任务状态变化时发给前端或通知中心的结构化消息

## 范围说明

### In Scope

- 为插件执行建立正式的后台任务模型和状态流转
- 提供创建任务、查询任务、轮询任务、订阅任务状态变更的接口
- 为失败、待确认、待补参任务提供通知和响应入口
- 提供有限重试、幂等保护、超时处理和审计记录

### Out of Scope

- 不改造成独立消息中间件集群
- 不做跨进程分布式 worker 调度
- 不做复杂编排 DAG
- 不做新的前端设计稿和完整交互稿

## 需求

### 需求 1：插件执行必须先变成后台任务

**用户故事：** 作为系统调用方，我希望插件执行先创建后台任务，以便接口能快速返回，不被插件运行时间拖死。

#### 验收标准

1. WHEN 任意插件执行入口收到合法请求 THEN System SHALL 先写入任务记录并返回任务编号，而不是同步等待插件完成
2. WHEN 任务已经入队 THEN System SHALL 在任务记录中保存插件标识、触发来源、请求载荷摘要、创建时间和当前状态
3. WHEN 插件执行时间很长 THEN System SHALL 仍然允许其他接口继续正常响应，不因等待任务结果阻塞整个进程

### 需求 2：任务状态必须可查、可订阅、可审计

**用户故事：** 作为前端和运维人员，我希望能稳定看到任务状态变化，以便知道任务现在卡在哪、是否成功、失败原因是什么。

#### 验收标准

1. WHEN 客户端按任务编号查询 THEN System SHALL 返回任务当前状态、开始时间、结束时间、执行结果摘要和错误信息
2. WHEN 客户端按列表查询 THEN System SHALL 支持按家庭、插件、状态、时间范围查看任务记录
3. WHEN 任务状态发生变化 THEN System SHALL 产出可订阅的通知事件，供 WebSocket 或其他推送通道使用
4. WHEN 任务执行完成或失败 THEN System SHALL 保留审计信息和关键错误上下文，便于排查

### 需求 3：失败任务必须有通知和后续处理入口

**用户故事：** 作为家庭成员或管理员，我希望任务失败时系统能明确通知我，并告诉我下一步能重试、取消还是补充信息。

#### 验收标准

1. WHEN 任务执行失败 THEN System SHALL 记录失败原因、失败阶段、可重试标记和下一步建议动作
2. WHEN 任务失败且存在可见用户 THEN System SHALL 生成一条面向前端的失败通知事件
3. WHEN 任务需要人工确认或补充参数 THEN System SHALL 把任务状态标记为待响应，而不是直接标记成功或失败
4. WHEN 用户对任务作出响应 THEN System SHALL 记录响应内容、响应时间和响应结果，并驱动任务进入下一状态

### 需求 4：重试和幂等必须受控

**用户故事：** 作为系统维护者，我希望失败任务可以在明确规则下重试，以便提升成功率，同时避免同一动作被重复执行多次。

#### 验收标准

1. WHEN 任务属于可重试错误 THEN System SHALL 按预设上限执行有限次数重试，并记录每次尝试结果
2. WHEN 任务已经达到最大重试次数 THEN System SHALL 停止自动重试，并把任务标记为终态失败
3. WHEN 同一请求因前端重复提交或网络重放再次进入 THEN System SHALL 基于幂等键避免重复创建等价任务
4. WHEN 任务是高风险动作任务 THEN System SHALL 在没有明确确认的情况下拒绝自动重试

### 需求 5：任务执行必须能安全停机和恢复

**用户故事：** 作为系统维护者，我希望服务重启后任务状态仍然一致，以便不会出现任务丢失或永远卡在执行中。

#### 验收标准

1. WHEN 服务正常启动 THEN System SHALL 扫描异常中断的任务，并按规则恢复、重置或标记为失败
2. WHEN worker 在执行中崩溃 THEN System SHALL 保留最后一次心跳或更新时间，供恢复逻辑判断
3. WHEN 任务超过执行时限 THEN System SHALL 标记超时并触发失败处理，而不是无限挂起

## 非功能需求

### 非功能需求 1：性能

1. WHEN 客户端创建插件任务 THEN System SHALL 在短时间内返回任务编号，而不等待插件执行结果
2. WHEN 多个家庭同时提交插件任务 THEN System SHALL 保持接口可响应，不因为单个长任务拖慢全局请求处理

### 非功能需求 2：可靠性

1. WHEN 插件执行失败、超时或需要人工确认 THEN System SHALL 进入明确状态，并保留通知和后续响应入口
2. WHEN 服务重启或 worker 中断 THEN System SHALL 通过持久化状态恢复任务，而不是静默丢失

### 非功能需求 3：可维护性

1. WHEN 后续新增新的插件任务类型 THEN System SHALL 复用统一任务模型和状态机，而不是再造一套表和接口
2. WHEN 排查一条失败任务 THEN System SHALL 能通过任务记录、尝试记录、通知记录快速定位问题

## 成功定义

- 插件执行入口不再同步等待插件跑完
- 前端能通过轮询或订阅稳定拿到任务状态变化
- 失败任务会有通知，也有明确的重试、确认或取消入口
- 服务重启后不会出现一批没人管的“幽灵执行中任务"
