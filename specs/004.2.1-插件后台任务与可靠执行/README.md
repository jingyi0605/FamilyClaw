# Spec 004.2.1 - 插件后台任务与可靠执行

状态：Draft

## 这份 Spec 解决什么问题

现在插件调用虽然已经从主请求线程里隔离出去了，但还有一个根问题没收好：

- 请求发出去后，插件执行还是和这次接口调用强绑定
- 一旦插件跑得久、失败、超时、需要人工确认，前端和后端都不知道后面该怎么继续
- 失败以后缺少统一通知、重试、人工接手入口，最后只能靠人翻日志

这份 Spec 要解决的就是这件事：把插件执行改成真正的后台任务，有状态、有结果、有失败通知，也有后续响应机制。

## 这次要做到什么

- 插件执行请求先入队，再由后台 worker 执行，不再要求接口一直等结果
- 每个任务都有稳定状态、执行记录、错误信息和重试信息
- 前端可以轮询，也可以订阅任务状态变化
- 任务失败时，系统要能通知到人，并给出后续操作入口
- 对需要人工确认或补充信息的任务，要有“待响应”状态，而不是直接算失败

## 这次明确不做什么

- 不做通用消息队列平台替换
- 不做跨服务分布式调度系统
- 不做无限自动重试
- 不做短信、电话这类高成本通知通道
- 不做插件市场侧的任务面板设计

## 为什么现在做

因为插件已经不是“偶尔跑一下的小工具”了。它后面会接智能家居、健康同步、Agent 动作执行这些真正会失败、会超时、会需要人工确认的事情。

如果没有后台任务模型，系统只会出现三种垃圾结果：

- 请求长时间挂着不返回
- 失败了没人知道
- 需要后续处理时没有正式入口

## 主要文档

- `requirements.md`：说清楚后台任务到底要覆盖哪些场景，什么算成功
- `design.md`：说清楚任务模型、状态流转、接口、通知和响应机制怎么落地
- `tasks.md`：说清楚先搭数据模型，还是先接 worker、接口、通知、响应
- `docs/README.md`：给后续补接口样例、通知样例、联调说明留位置
