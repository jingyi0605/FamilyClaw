# Spec 005.9 - speaker/voice 通用插件契约

状态：Draft

## 这份 Spec 要解决什么问题

当前项目虽然已经把小爱音箱做成了内置插件的一部分，但真实边界还没有彻底收口。

- 设备发现、绑定、控制这条链路已经基本插件化。
- 但“语音对话怎么接进来”这件事，宿主还没有一份正式、通用、能给第三方插件复用的 `speaker/voice` 契约。
- 现有 `open-xiaoai-speaker` 更接近“刷机网关 + 实时事件上报”的接入模式。
- 像 `mi-gpt` 这种“米家账号登录 + 小米云会话轮询 + 文本下发 + TTS 播放”的方案，虽然能覆盖更多新型号小爱音箱，但现在没有正式入口，只能继续长特例。

这就是问题本质：不是“再接一个小爱插件”这么简单，而是宿主缺了一层正式的 `speaker/voice` 通用契约。

## 核心判断

✅ 值得做。

原因很直接：

1. 这是真问题，不是想象出来的抽象设计。
2. 如果不先把宿主契约立住，后面每接一个新音箱方案，核心里都会再长一坨厂商分支。
3. 这件事必须先做宿主通用边界，不能直接把 `mi-gpt` 逻辑塞进核心，否则以后一定烂掉。

## 本 Spec 的边界

### In Scope

- 定义宿主侧正式的 `speaker/voice` 插件契约。
- 定义第三方 `speaker` 适配插件需要声明的 manifest 能力、入口和运行时边界。
- 定义“文本轮询型实时对话”和“音频会话型实时对话”两条正式车道。
- 定义 discovery、binding、command、runtime health 的统一宿主语义。
- 定义插件禁用、错误语义、可观测性和回归要求。

### Out of Scope

- 不写任何小米专有 API 细节。
- 不直接规定 `mi-gpt` 插件内部怎么登录、怎么轮询、怎么做 TTS。
- 不重做现有设备控制协议和声纹业务本身。

## 和其他 Spec 的关系

- `005.7` 解决的是“小爱音箱全插件化迁移”。
- `005.9` 解决的是“宿主正式契约还不够，第三方 speaker/voice 插件没法按规范接进来”。
- `005.9.1` 会在本契约基础上定义 `migpt_xiaoai_speaker` 第三方插件怎么落地。

## 成功标准

- 宿主不再持有任何“小米文本轮询方案”的厂商特例。
- 第三方 `speaker` 插件可以只靠正式契约接入设备发现、绑定、对话和控制。
- 后续新增类似 `mi-gpt` 的音箱插件时，不需要再改宿主核心分支。
