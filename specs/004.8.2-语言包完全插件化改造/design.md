# 设计文档 - 语言包完全插件化改造

状态：Draft

## 1. 概览

### 1.1 目标

- 把 `zh-CN`、`en-US`、`zh-TW` 迁成真实 `locale-pack` 插件。
- 让 `apps/user-app` 的 H5 和 RN 都通过统一语言插件运行时消费资源。
- 把插件安装、升级、卸载后的语言资源同步链路补齐到可执行状态。

### 1.2 覆盖需求

- `requirements.md` 需求 1
- `requirements.md` 需求 2
- `requirements.md` 需求 3

### 1.3 技术约束

- 后端：FastAPI、SQLAlchemy、插件注册与挂载体系、插件市场安装链路。
- 前端：Taro `user-app`，同时覆盖 H5 和 RN。
- 数据存储：PostgreSQL + `apps/api-server/data/plugins` 托管插件目录。
- 认证授权：沿用当前家庭访问权限与插件启停权限体系。
- 外部依赖：插件市场仓库同步、插件挂载、前端 H5/RN 构建产物。

## 2. 架构

### 2.1 系统结构

这次改造后，语言资源链路固定为四层：

1. **语言插件目录层**
   内置语言插件放在仓库插件目录中，远端语言插件放在 `apps/api-server/data/plugins/...`。
2. **后端注册与分发层**
   后端负责扫描 manifest、维护语言资源注册表、按家庭权限和启停状态分发资源。
3. **前端统一运行时层**
   `user-app` 建立统一的语言插件运行时，屏蔽“内置资源”和“远端资源”的来源差异。
4. **页面消费层**
   H5 页面、RN 页面、登录页和未登录壳层统一通过 `useI18n` 或等价运行时接口取文案。

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 语言插件 manifest/schema | 声明 `locale_id`、资源入口、版本、适用端、回退链 | 插件目录 | 统一插件定义 |
| 后端语言注册表服务 | 汇总当前家庭可见语言插件 | 插件清单、启停状态、家庭上下文 | 语言资源注册表 |
| 后端语言资源接口 | 返回某个插件语言资源正文 | 插件 ID、`locale_id`、家庭上下文 | 语言资源 JSON |
| 前端语言插件运行时 | 统一选择、加载、缓存、失效处理 | 注册表、资源接口、内置资源映射 | 当前语言资源状态 |
| 页面层 | 按 key 取文案并渲染 | 当前语言运行时 | 页面文案 |

### 2.3 关键流程

#### 2.3.1 内置语言插件启动流程

1. 后端扫描内置 `locale-pack` 插件目录，注册 `zh-CN`、`en-US`、`zh-TW`。
2. `user-app` 在登录前和首屏启动阶段只装配内置语言插件清单，并从随前端包发布的内置资源映射中解析 manifest 对应资源。
3. 前端根据当前选择的 `plugin_id + locale_id` 加载消息资源；如果当前没有用户选择，则按 manifest 声明的内置默认语言启动，而不是回退到宿主硬编码文案。
4. 拿到家庭上下文后，前端再叠加当前家庭可见的远端语言插件注册表，统一交给语言插件运行时管理。

#### 2.3.2 远端语言插件安装与加载流程

1. 插件市场或手动挂载把语言插件安装到 `apps/api-server/data/plugins/...`。
2. 后端同步插件挂载信息，刷新语言资源注册表与资源索引。
3. 前端收到新的注册表快照后，可以拉取该插件的语言资源并应用到 H5 或 RN。

#### 2.3.3 语言插件失效流程

1. 当前语言插件被停用、卸载、升级或资源校验失败。
2. 后端注册表将该插件标记为不可用或版本失效。
3. 前端运行时把当前语言状态标记为“失效待重选”，提示用户重新选择可用语言插件。

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3

- **`PluginManifestLocalePackSpec` 扩展**：补充资源来源、适用端、资源版本和回退链。
- **语言资源注册表服务**：替代“直接内嵌全部 messages”的旧返回模式，统一输出元数据和资源索引。
- **语言资源分发接口**：为 H5 和 RN 按需返回插件语言资源正文。
- **前端语言插件运行时**：统一处理内置语言插件和远端语言插件。

### 3.2 数据结构

覆盖需求：1、2

#### 3.2.1 `LocalePackResourceDescriptor`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `locale_id` | string | 是 | 语言标识 | 与 manifest 中 locale 定义一致 |
| `entry_resource` | string | 是 | 资源入口相对路径 | 仅允许插件目录内路径 |
| `resource_source` | enum | 是 | `builtin_bundle` / `managed_plugin_dir` | 内置与远端统一枚举 |
| `resource_version` | string | 是 | 资源版本号 | 升级后必须变化 |
| `platform_targets` | string[] | 是 | 适用端 | 至少包含 `h5` 或 `rn` |
| `integrity_sha256` | string | 否 | 资源文件校验摘要 | 内置与远端资源都走同一套校验语义 |

#### 3.2.2 `LocaleRegistryItemV2`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `plugin_id` | string | 是 | 插件 ID | 当前家庭可见 |
| `locale_id` | string | 是 | 语言 ID | 唯一标识一个插件语言资源 |
| `display_name` | string | 是 | 显示名称 | 面向用户 |
| `source_type` | enum | 是 | `builtin` / `official` / `third_party` | 继承插件来源 |
| `enabled` | boolean | 是 | 是否可选 | 受启停规则控制 |
| `resource_source` | enum | 是 | 资源来源类型 | 与资源描述一致 |
| `resource_version` | string | 是 | 当前资源版本 | 用于前端刷新判断 |
| `state` | enum | 是 | `ready` / `disabled` / `invalid` / `stale` | 用于前端显示 |

#### 3.2.3 `BuiltinLocaleBundleIndex`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `plugin_id` | string | 是 | 内置插件 ID | 必须与 manifest 一致 |
| `locale_id` | string | 是 | 语言 ID | 必须与 manifest 一致 |
| `bundle_module` | string | 是 | 前端 bundle 内的资源模块标识 | 只用于解析内置资源，不新增宿主语言定义 |
| `resource_version` | string | 是 | 内置资源版本 | 必须与注册表版本一致 |

这个索引只是把“已经是插件目录里的资源”映射到前端构建产物里，不能再变成第二套宿主语言常量表。

### 3.3 接口契约

覆盖需求：2、3

#### 3.3.1 `GET /ai-config/{household_id}/locales`

- 类型：HTTP
- 路径或标识：`/ai-config/{household_id}/locales`
- 输入：家庭 ID、当前登录成员上下文
- 输出：语言资源注册表快照，不再直接内嵌整包 messages
- 校验：仅返回当前家庭可见、且符合启停规则的语言插件
- 错误：家庭不可见、插件注册异常、资源索引损坏

#### 3.3.2 `GET /ai-config/{household_id}/plugin-locales/{plugin_id}/{locale_id}`

- 类型：HTTP
- 路径或标识：`/ai-config/{household_id}/plugin-locales/{plugin_id}/{locale_id}`
- 输入：家庭 ID、插件 ID、语言 ID
- 输出：语言资源正文、资源版本、schema 版本、校验信息
- 校验：插件必须当前家庭可见、语言资源文件存在且校验通过
- 错误：插件不可见、资源不存在、资源无效、版本不匹配

## 4. 数据与状态模型

### 4.1 数据关系

- 一个 `locale-pack` 插件可以声明多个 `locale_id`。
- 一个家庭可以看到多个语言插件，但同一个前端会话同一时刻只会激活一个 `plugin_id + locale_id`。
- 内置语言插件和远端语言插件都进入同一个注册表，由前端统一选择。

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `ready` | 资源可正常加载 | manifest 有效，资源可用 | 被停用、升级、卸载或校验失败 |
| `disabled` | 插件被禁用 | 家庭或基础启停状态关闭 | 再次启用 |
| `invalid` | 资源内容无效 | JSON 损坏、schema 不符 | 修复资源并重新注册 |
| `stale` | 当前前端持有旧版本 | 插件升级但前端缓存未更新 | 重新拉取新版本 |
| `missing` | 当前选择的资源不存在 | 插件卸载或资源删除 | 用户重新选择可用语言 |

## 5. 错误处理

### 5.1 错误类型

- **资源索引错误**：manifest 声明存在，但资源入口缺失或越界。
- **资源校验错误**：JSON 无法解析或结构不符合约束。
- **前端激活错误**：当前选择的语言插件已经失效。
- **同步错误**：插件安装或升级后，注册表没有正确刷新。

### 5.2 错误响应格式

```json
{
  "detail": "Locale plugin resource is unavailable",
  "error_code": "plugin_locale_resource_unavailable",
  "field": "entry_resource",
  "timestamp": "2026-03-18T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：拒绝注册该语言插件，并记录明确字段路径。
2. 资源校验错误：注册表中标记为 `invalid`，前端不可激活。
3. 远端加载错误：前端显示加载失败，并保留最近一次成功加载的插件资源快照用于诊断。
4. 当前语言失效：前端进入“待重选”状态，不静默回退到宿主文案。

## 6. 正确性属性

### 6.1 属性 1：宿主不是语言资源源

*对于任何* `user-app` 取文案场景，系统都应该满足：宿主代码不再保存可直接充当正式资源源的默认消息表。

**验证需求：** `requirements.md` 需求 1、需求 2

### 6.2 属性 2：内置与远端语言插件同构

*对于任何* 语言插件，系统都应该满足：内置、官方、第三方都遵守同一份 manifest 规则、注册表结构和前端选择语义。

**验证需求：** `requirements.md` 需求 1、需求 3

## 8. 风险与已确认边界

### 8.1 风险

- H5 现有 `I18nProvider` 里宿主文案较多，迁移时容易出现 key 漏失。
- RN 当前主要消费共享状态，但没有完整的语言插件运行时，补齐时可能暴露更多平台差异。
- 插件升级后的前端缓存失效如果处理不清，容易出现“注册表是新版本、页面还是旧文案”的不一致。

### 8.2 已确认边界

- H5 和 RN 共用同一份插件语言正文，平台差异只允许出现在运行时适配层，不再维护两套 canonical source。
- 登录页、未登录壳层和首屏启动阶段只使用内置 `locale-pack` 插件；拿到家庭上下文后，再叠加官方和第三方语言插件注册表。
- 运行时不允许宿主 fallback；当前已选语言插件一旦失效，前端必须进入“待重选”状态，而不是静默切回宿主默认文案。
