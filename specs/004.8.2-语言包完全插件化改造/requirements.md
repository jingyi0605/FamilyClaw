# 需求文档 - 语言包完全插件化改造

状态：Draft

## 简介

现在 `apps/api-server` 已经能把 `locale-pack` 插件注册出来，`apps/user-app` 也能读到部分插件语言资源，但整条链路还没有真正收口成“完全插件化”：

- `zh-CN`、`en-US` 这类基础语言仍然主要活在宿主代码里，不是正式插件资源。
- H5 页面目前仍以宿主内置消息表为主，插件语言包更像覆盖层，不是唯一资源源。
- RN 页面还没有和 H5 一样走统一的语言插件运行时。
- 插件安装、升级、卸载后，前端怎么拿到最新语言资源，还没有一套完整规则。

这次 Spec 要解决的不是“再补一个语言包”，而是把语言资源的所有权彻底交给 `locale-pack` 插件：

- 内置语言包也必须是“真实插件目录 + manifest + 资源文件”。
- `user-app` 不再把宿主常量表当 canonical source。
- 未来官方插件和第三方插件统一从 `data/plugins` 加载资源，并同时支持 H5 和 RN。

## 术语表

- **System**：FamilyClaw 的插件平台、前后端运行时和插件市场安装链路。
- **locale-pack 插件**：声明一个或多个 `locale_id`，并提供消息资源文件的正式插件。
- **内置语言插件**：仓库内自带、随 `user-app` 一起发布的 `locale-pack` 插件。
- **远端语言插件**：安装到 `apps/api-server/data/plugins` 下，由后端托管并分发资源的官方或第三方 `locale-pack` 插件。
- **语言资源注册表**：后端输出给前端的当前家庭可用语言插件清单，包含插件、版本、状态和资源索引信息。

## 范围说明

### In Scope

- 把 `zh-CN`、`en-US`、`zh-TW` 收口成真实 `locale-pack` 插件目录。
- 定义内置语言插件和远端语言插件的统一 manifest 与资源目录规则。
- 定义 `apps/api-server` 的语言资源注册表、资源分发接口和安装同步规则。
- 定义 `apps/user-app` 在 H5 和 RN 上统一的语言插件运行时、切换逻辑和失效状态。
- 把登录页、未登录壳层、已登录业务页都纳入语言插件链路。

### Out of Scope

- `docs/Documentation` 文档站点的主题或语言体系。
- 离线资源平台化、签名校验、商业化分发。
- 其它插件类型的顺手重构。

## 需求

### 需求 1：所有语言资源都必须落成真实插件

**用户故事：** 作为平台维护者，我希望所有可选语言都是真实 `locale-pack` 插件，而不是宿主里藏着一份默认语言资源，这样语言包的注册、启停、升级和排障才能统一。

#### 验收标准

1. WHEN 系统检查内置语言资源 THEN System SHALL 让 `zh-CN`、`en-US`、`zh-TW` 都以真实插件目录、manifest 和资源文件存在。
2. WHEN 系统注册语言插件 THEN System SHALL 统一从插件 manifest 读取 `locale_id`、资源入口、版本和适用端信息，而不是从宿主常量表拼装。
3. WHEN 语言插件被停用、卸载或损坏 THEN System SHALL 明确标记该插件不可用，而不是静默回退到宿主默认文案。

### 需求 2：前端必须统一走语言插件运行时

**用户故事：** 作为前端维护者，我希望 H5 和 RN 都通过同一套语言插件运行时拿到文案，这样登录页、设置页和业务页不会各走一套来源。

#### 验收标准

1. WHEN `user-app` 的 H5 或 RN 页面请求文案 THEN System SHALL 先解析当前家庭可见的语言插件，再加载对应插件资源。
2. WHEN 当前语言是内置语言插件 THEN System SHALL 从随前端发布的插件资源映射里读取语言资源，而不是读取宿主硬编码消息表。
3. WHEN 当前语言是远端语言插件 THEN System SHALL 通过后端资源接口加载资源，并在 H5 和 RN 上使用同一套选择逻辑和状态语义。
4. WHEN 页面处于登录前、首屏启动中或尚未拿到家庭上下文 THEN System SHALL 只使用内置 `locale-pack` 插件 manifest 和随 `user-app` 发布的内置资源完成启动，而不是直接读取宿主硬编码文案。

### 需求 3：安装链路必须同步语言资源状态

**用户故事：** 作为插件平台维护者，我希望语言插件安装、升级、卸载后，前端能及时看到新的语言资源状态，这样插件市场链路和前端体验才是闭环。

#### 验收标准

1. WHEN 官方或第三方语言插件安装到 `data/plugins` THEN System SHALL 更新语言资源注册表，并允许前端读取对应资源。
2. WHEN 语言插件升级或卸载 THEN System SHALL 同步更新资源版本、可用状态和前端可见列表。
3. WHEN 前端持有的语言插件版本已经失效 THEN System SHALL 进入明确的“资源失效待重选”状态，而不是继续使用宿主 fallback。

## 非功能需求

### 非功能需求 1：一致性

1. WHEN H5 和 RN 加载同一个 `locale-pack` 插件 THEN System SHALL 使用同一份插件资源定义、同一套版本语义和同一套失效规则。
2. WHEN 内置、官方、第三方语言插件共存 THEN System SHALL 通过同一份注册表和同一套 manifest 规则管理，而不是为内置插件保留特例语义。

### 非功能需求 2：可靠性

1. WHEN 远端语言插件资源接口失败 THEN System SHALL 明确返回错误状态，并保留最近一次成功加载的插件资源快照用于诊断。
2. WHEN 当前选中的语言插件不可用 THEN System SHALL 阻止静默回退，并给出明确可恢复路径，例如重新选择可用语言包。

### 非功能需求 3：可维护性

1. WHEN 后续新增语言包 THEN System SHALL 只需要增加新的插件目录和资源文件，而不需要继续修改宿主默认消息表。
2. WHEN 排查语言问题 THEN System SHALL 让开发者能沿着“manifest -> 资源文件 -> 注册表 -> 前端运行时”一条链快速定位。

## 成功定义

- `zh-CN`、`en-US`、`zh-TW` 都以真实 `locale-pack` 插件存在。
- `apps/user-app` 的 H5 和 RN 页面都通过统一语言插件运行时取文案。
- 登录页、未登录壳层和首屏启动链路也通过内置 `locale-pack` 插件取文案。
- 官方和第三方语言插件可以从 `data/plugins` 安装、注册、分发并被前端加载。
- 宿主不再保留默认语言资源 fallback，也不再保留宿主消息表作为 canonical source。
