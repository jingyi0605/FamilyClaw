# Spec 010.1 - 非网页会话来源识别与控制命令

状态：Draft

## 这份 Spec 解决什么问题？

`Telegram` 已经打通了，说明渠道插件这条路是通的。但现在还卡着三个真正会影响产品可用性的硬问题：

- 非网页渠道来的消息，到了统一会话层以后，来源信息没有被正式落库，只能在渠道层零散看到
- 非网页端还没有一套正式控制命令，用户没法在 `Telegram`、小爱音响这类入口上新建会话或重置会话
- `Telegram` 的 `message_thread_id` 虽然已经能透传，但还只是“能跑”，不是正式的一等能力

这三个问题本质上是同一件事：渠道入口已经有了，会话主链却还在把“非网页来源”当旁路。结果就是能接消息，但系统没法稳定知道“这条消息从哪来、属于哪个外部会话、用户能不能在原入口控制会话边界”。

这份 Spec 不重做整套通讯平台，只做一件事：把非网页入口正式接入统一会话层。

## 为什么现在做？

因为这不是锦上添花，这是把现有渠道能力从“演示可用”变成“产品可用”的分水岭。

如果现在不补这层正式能力，后面每多接一个非网页渠道，就会继续复制同样的烂问题：

- 入口能识别来源，但统一会话层还是看不见
- 小爱这类语音终端继续靠内存状态硬撑，进程重启就容易丢上下文
- `/new`、`/reset` 这类基本控制能力在网页外不可用
- `Telegram` 话题线程只是被动兼容，不能稳定作为线程级会话使用

## 核心判断

✅ 值得做：这三个问题都是真问题，而且已经直接影响多渠道接入后的稳定性、可维护性和用户体验。

## 关键洞察

- 数据结构：来源信息应该作为统一会话层的正式附属数据存在，不能只停留在渠道入站表或 debug 日志里
- 复杂度：会话控制命令应该走统一解析层，而不是让每个渠道插件自己发明一套 `/new` `/reset`
- 风险点：任何改动都不能破坏现有网页会话主链，也不能让已经可用的 `Telegram` 私聊流程回退

## 这次要交付什么？

- 给统一会话层补齐“消息来源”正式落库能力
- 给语音/小爱这类非网页终端补齐持久化会话绑定，不再只靠内存注册表
- 增加非网页端统一控制命令层，先覆盖 `new` 和 `reset`
- 把 `Telegram` 线程会话正式化，明确线程键、绑定规则、命令行为和回归测试
- 每个任务都带上 Alembic migration 和测试范围，不允许只改 model 或只写设计

## 这次明确不做什么？

- 不重写现有 `conversation` 主链
- 不扩展成所有渠道的富媒体、卡片、按钮、投票能力
- 不在这份 Spec 里处理跨平台会话合并
- 不设计复杂的管理员远程会话编排功能，先把用户入口最基本的会话边界补齐
