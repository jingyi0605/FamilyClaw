# 需求文档 - 插件 ZIP 手动安装、覆盖升级与安装后立即生效

状态：In Progress

## 简介

当前项目已经有插件市场安装链路，也有后端挂载插件能力，但这两头都不顺手：

- 普通用户没有本地 ZIP 上传安装入口
- 手工安装还是工程师视角，要求你先知道服务器目录怎么挂
- “安装后自动重载”没有统一定义，讨论半天最后都在猜要不要重启服务

真正的问题不是“能不能解压一个 ZIP”。

真正的问题是：

1. 用户能不能在插件管理页直接上传插件包完成安装
2. 同一个插件能不能安全覆盖升级
3. 安装成功后系统能不能不用重启服务就进入新挂载
4. 现有市场插件安装、启用、升级到底是不是也应该遵守同一条运行时生效规则

这份需求服务三类人：

- **家庭管理员**：希望拿到 ZIP 包就能装，不想碰服务器目录
- **前后端开发者**：希望“立即生效”的边界稳定，不再靠猜
- **插件开发者**：希望知道运行时安装插件该遵守什么包结构和执行边界

## 当前实现回写

当前真实状态是：

- 后端 ZIP 安装接口、覆盖升级链路和运行时生效边界已经落地
- 插件市场安装、启用、升级都已经按“无需重启后端服务”工作
- `user-app` 还没有把 ZIP 上传入口接到插件管理页里

所以这份需求里的“页面上传入口”还没全部落地，但“安装后无需重启即可被列表和后续执行识别，且不是 Python 模块热替换”这条边界已经成了现状，不再只是目标

## 术语表

- **System**：FamilyClaw 当前的插件管理、插件市场和插件执行系统
- **ZIP 手动安装**：用户在插件管理页上传本地 ZIP 包，由系统完成解压、校验、托管和挂载
- **覆盖升级安装**：同一家庭下，对同一个 `plugin_id` 上传新版本 ZIP 包后，用新版本替换后续生效挂载
- **立即生效**：安装成功后，无需重启后端服务，后续查询和下一次执行都基于新挂载
- **热重载**：对当前进程里已经导入的 Python 模块做原位替换
- **运行时可安装插件**：可以通过页面上传 ZIP 或市场安装在服务运行期间接入的插件
- **隔离执行**：插件不和主进程共享导入缓存，按独立 runner 或子进程执行
- **旧任务**：安装或升级前已经开始执行的任务
- **待执行任务**：已经入队但还没真正开始执行的任务

## 范围说明

### In Scope

- 在插件管理页增加本地 ZIP 上传安装入口
- 后端接收 ZIP 包、解压、校验、托管安装
- 支持同 `plugin_id` 的覆盖升级安装
- 明确“立即生效”的运行时语义
- 明确 ZIP 安装插件与市场安装插件的一致运行边界
- 安装成功后的列表刷新、详情刷新和启用链路收口
- 失败回滚和错误提示

### Out of Scope

- 非 ZIP 格式上传
- 插件签名体系
- 手工 ZIP 的完整版本历史浏览和回滚 UI
- 安装成功后自动启用
- 启用后自动执行一次插件
- 对 `in_process` 模块做 `importlib.reload` 级别热替换
- 内置插件和随应用发布插件的运行时覆盖安装

## 需求

### 需求 1：插件管理页必须支持本地 ZIP 上传安装

**用户故事：** 作为家庭管理员，我希望在插件管理页直接选择本地 ZIP 包安装插件，这样我不用先懂服务器目录结构。

#### 验收标准

1. WHEN 管理员打开插件管理页 THEN System SHALL 提供明确的“上传 ZIP 安装插件”入口。
2. WHEN 管理员选择一个本地 ZIP 包并提交 THEN System SHALL 把文件上传给后端安装链路，而不是要求用户自己填写服务器路径。
3. WHEN 上传请求不是 ZIP 文件或缺少文件 THEN System SHALL 阻止提交并给出明确错误提示。

### 需求 2：后端必须在安装前校验 ZIP 包结构和 manifest

**用户故事：** 作为后端开发者，我希望系统先校验 ZIP 包结构和 manifest，再决定是否安装，这样坏包只影响自己，不会把运行链路拖死。

#### 验收标准

1. WHEN 后端接收到 ZIP 包 THEN System SHALL 先解压到临时目录并校验 `manifest.json` 是否存在且合法。
2. WHEN ZIP 包里的 `plugin_id`、`version`、入口声明或必要文件不合法 THEN System SHALL 拒绝安装，并返回明确错误码和字段定位。
3. WHEN ZIP 包校验失败 THEN System SHALL 清理临时产物，不留下半安装目录和半挂载记录。

### 需求 3：系统必须支持同插件 ID 的覆盖升级安装

**用户故事：** 作为家庭管理员，我希望给同一个插件重新上传新版本 ZIP 包时，系统能按升级处理，而不是逼我先删后装。

#### 验收标准

1. WHEN 当前家庭还没有该 `plugin_id` 的安装记录 THEN System SHALL 按首次安装处理。
2. WHEN 当前家庭已经存在该 `plugin_id`，且用户显式允许覆盖安装 THEN System SHALL 按覆盖升级处理，并把后续生效挂载切到新版本。
3. WHEN 当前家庭已经存在该 `plugin_id`，但用户没有显式允许覆盖安装 THEN System SHALL 阻止安装，并明确提示这是覆盖升级操作。

### 需求 4：安装成功后必须立即进入新挂载链路，无需重启后端

**用户故事：** 作为家庭管理员，我希望安装完成后系统马上认这个新插件或新版本，而不是还得重启服务碰碰运气。

#### 验收标准

1. WHEN ZIP 安装或覆盖升级成功 THEN System SHALL 让插件列表和插件详情在刷新后直接看到新的挂载结果，无需重启后端服务。
2. WHEN 插件在安装成功后被启用或后续被执行 THEN System SHALL 使用新的挂载路径和新版本代码，而不是继续要求等待服务重启。
3. WHEN 用户只是完成安装但还未启用 THEN System SHALL 保持“已安装但未启用”状态，不得把“立即生效”误解成自动启用。

### 需求 5：运行时可安装插件必须遵守统一的隔离执行边界

**用户故事：** 作为维护者，我希望运行时安装进来的插件都走一条统一执行边界，这样“立即生效”不会因为导入缓存搞得真假难辨。

#### 验收标准

1. WHEN 插件属于 ZIP 手动安装或市场安装这类运行时安装来源 THEN System SHALL 以隔离执行方式运行该插件，而不是依赖主进程已导入模块缓存。
2. WHEN 系统需要说明“立即生效”语义 THEN System SHALL 明确表示“下一次查询和下一次执行使用新挂载”，而不是宣称已经完成 Python 进程内热替换。
3. WHEN 插件属于内置插件或随应用发布插件 THEN System SHALL 不把这类能力错误地承诺为本次运行时覆盖安装范围。

### 需求 6：已在运行中的旧任务和待执行任务必须有清楚边界

**用户故事：** 作为家庭管理员，我希望知道覆盖升级时哪些任务会继续跑旧版本，哪些任务会切到新版本，而不是系统行为全靠猜。

#### 验收标准

1. WHEN 覆盖升级发生时已有插件任务正在执行 THEN System SHALL 允许该次运行继续用旧版本收口，不强行中断并切换代码。
2. WHEN 覆盖升级发生时存在尚未真正开始执行的排队任务 THEN System SHALL 在任务真正执行前重新解析当前插件快照，并使用新挂载。
3. WHEN 系统向用户展示升级结果或文档说明 THEN System SHALL 明确写出“运行中任务继续旧版本，后续任务走新版本”的边界。

### 需求 7：市场插件安装、启用和升级也必须明确为无需重启后端

**用户故事：** 作为家庭管理员，我希望市场插件和 ZIP 手动安装插件遵守同样的运行时规则，而不是一个要重启、一个不要重启。

#### 验收标准

1. WHEN 市场插件安装成功 THEN System SHALL 在不重启后端服务的前提下，让该插件进入已安装状态并可被后续链路识别。
2. WHEN 市场插件满足配置要求且用户点击启用 THEN System SHALL 让后续执行直接生效，而不是要求重启后端服务。
3. WHEN 市场插件升级成功 THEN System SHALL 与 ZIP 覆盖升级一样，让后续查询和下一次执行直接基于新挂载。

### 需求 8：错误、回滚和文档口径必须一致

**用户故事：** 作为后续接手的人，我希望接口、页面和文档对“安装成功”“覆盖升级”“立即生效”“不需要重启”的说法一致，不要 README 说一套，代码做一套。

#### 验收标准

1. WHEN ZIP 安装、覆盖升级或市场升级失败 THEN System SHALL 保留失败前稳定状态，并返回明确错误阶段、错误码和错误消息。
2. WHEN 前端展示安装结果和启用提示 THEN System SHALL 明确区分“已安装”“已启用”“下一次执行将使用新版本”。
3. WHEN 开发者查看 Spec 和正式文档 THEN System SHALL 能看到这轮支持的立即生效边界，以及明确不支持的热替换边界。

## 非功能需求

### 非功能需求 1：安全性

1. WHEN 系统处理 ZIP 包 THEN System SHALL 拒绝路径穿越、非法解压目标和 manifest 与包内容不一致的安装请求。
2. WHEN 系统执行覆盖升级 THEN System SHALL 只允许同一家庭范围内覆盖当前 `plugin_id`，不得跨家庭或跨来源误覆盖。

### 非功能需求 2：可靠性

1. WHEN 安装或覆盖升级失败 THEN System SHALL 保持旧挂载和旧启停状态不被破坏。
2. WHEN 页面刷新插件列表或任务列表 THEN System SHALL 能稳定看到最新安装结果，而不是依赖服务重启恢复。

### 非功能需求 3：可维护性

1. WHEN 后续要继续扩展手工 ZIP 回滚或更多包格式 THEN System SHALL 基于统一安装流水线扩展，而不是再做第二套上传安装逻辑。
2. WHEN 排查“为什么安装后没生效” THEN System SHALL 能从日志、安装结果和文档中明确区分是未启用、任务仍在跑旧版本，还是超出本次热重载边界。

## 成功定义

- 插件管理页已经有本地 ZIP 上传安装入口。
- ZIP 包安装和覆盖升级已经有正式后端链路，不再要求用户自己处理服务器目录。
- 覆盖升级后无需重启后端服务，后续查询和下一次执行直接基于新挂载。
- 市场插件安装、启用、升级也已经明确不需要重启后端服务。
- “立即生效”和“真正 Python 热重载”的边界已经写清楚，不再含糊。
