# Spec 004.6.1 - 插件 ZIP 手动安装、覆盖升级与安装后立即生效

状态：DONE

## 这份 Spec 解决什么问题

现在插件管理页有两个现实问题：

1. 用户端没有“上传 ZIP 直接安装插件”的入口，手工安装还停留在后端挂载目录这种工程师玩法。
2. “安装后自动重载”这句话现在没有统一定义。有人理解成页面刷新，有人理解成下一次执行生效，还有人理解成 Python 模块热替换。

这两件事不先写清楚，后面实现一定会变成垃圾：

- 前端加了个上传按钮
- 后端糊一个解压接口
- 最后谁都说不清到底要不要重启服务

这份 Spec 就干三件事：

- 给插件管理页补一个正式的 ZIP 手动安装入口
- 让手动安装支持同插件 ID 的覆盖升级安装
- 把“安装后立即生效”和“真正热重载”之间的边界写死

## 当前实现回写

这轮真实落地状态别再写虚的：

- 后端已经有正式 ZIP 安装接口：`POST /api/v1/ai-config/{household_id}/plugin-packages`
- 后端已经支持同 `plugin_id` 的覆盖升级、版本目录托管、挂载切换和无需重启的后续识别
- 手动 ZIP 安装目录已经固定为 `data/plugins/third_party/manual/<household_id>/<plugin_id>/<version>--<timestamp>--<id>/`
- 运行时安装插件现在按 `subprocess_runner` 边界处理，列表刷新和下一次执行都会认当前挂载
- 插件市场安装、启用、升级也已经按“无需重启后端服务”的语义落地
- `user-app` 里的 ZIP 上传入口、冲突覆盖确认和安装后刷新也已经接通，这份 Spec 可以收口

  - 已经解决市场来源、一键安装、手动启用

## 这次先把哪几件事写死

### 1. ZIP 手动安装只服务运行时可安装插件

- 这次支持的是用户在插件管理页上传本地 ZIP 包
- 后端负责解压、校验、托管、挂载
- 不再要求用户自己先把目录丢到服务器上

### 2. 覆盖升级安装必须有明确规则

- 同一个 `plugin_id` 再次上传时，默认不是静默覆盖
- 要明确区分：
  - 首次安装
  - 同插件覆盖升级
  - 版本相同重复上传
  - `plugin_id` 冲突但来源不合法

### 3. “安装后自动重载”只表示新挂载立即进入后续链路

这次必须把人话说清楚：

- 安装成功后，插件列表、详情、启用状态、下一次执行都应该基于新挂载立即生效
- 不要求用户重启后端服务
- 但这**不等于**对已经导入到当前 Python 进程里的 `in_process` 模块做热替换
- 首次 ZIP 手动安装和市场安装默认都不会自动启用
- 市场插件在安装后会立刻进入“已安装”状态；满足配置约束并手动启用后，后续执行直接生效

### 4. 运行时可安装插件统一走隔离执行

为了让“安装后立即生效”不是一句屁话，这次运行时安装得到的插件必须统一走隔离执行边界：

- ZIP 手动安装插件统一按 `third_party + subprocess_runner` 运行
- 市场安装插件只要属于运行时安装链路，也统一按可重载语义处理
- 已经在跑的任务继续跑旧版本
- 还没开始执行的任务，下一次取快照时看到新版本

### 5. 这次明确不做什么

- 不做任意目录上传
- 不做 tar.gz、7z 等多格式上传，先只支持 ZIP
- 不做“安装完自动启用并自动执行一次”
- 不做 `importlib.reload()` 级别的 Python 热替换
- 不做内置插件和随应用发布插件的运行时覆盖
- 不做完整手工 ZIP 回滚历史管理

## 一句话结论

这次不是加一个“上传 ZIP”按钮就算完事。

真正要做的是把这条规则钉死：

运行时安装插件 = 上传 ZIP 或市场安装后，马上进入新的挂载和执行链路；  
但不承诺替换已经在当前进程里跑着的旧模块。
