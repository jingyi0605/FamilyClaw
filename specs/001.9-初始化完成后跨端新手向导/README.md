# Spec 001.9 - 初始化完成后跨端新手向导

状态：Draft

## 这份 Spec 解决什么问题？

现在的初始化向导只解决了“用户能不能进系统”，没有解决“用户进来以后知不知道每个页面是干什么的”。

这会造成一个很典型的半成品体验：

- 新用户刚做完初始化，以为已经结束了
- 进入首页以后，看到一堆卡片、侧边导航、设置入口，不知道先用哪个
- 管家已经创建好了，但用户不知道去哪里对话、去哪里配 AI、去哪里看家庭和记忆
- H5 和 RN 页面都在用同一套产品能力，但如果新手引导只做 H5，RN 端体验会直接断层

这份 `001.9` 要做的事情很直接：

1. 在用户完成初始化向导后，第一次进入仪表盘时，自动启动一套完整的新手向导
2. 新手向导用“动态管家气泡 + 页面高亮”的方式，逐步介绍首页、家庭、助手、记忆、设置等核心页面
3. 前端负责整套导览编排、跨页面跳转、定位和恢复；后端只保存一个“这个成员是否完成当前版本导览”的状态
4. 方案从一开始就同时覆盖 H5 和 RN，不允许先做一个只在网页上能跑的残缺版本

## 先说结论

- 这个功能不应该继续污染 `household.setup_status`
- 导览完成状态应该是“成员级”，不是“家庭级”
- 后端保存的不是布尔值，而应该是“导览版本号”，这样以后改导览时不用补第二套字段
- 导览脚本必须前端统一建模，H5 和 RN 共享同一份脚本，只在“锚点定位”和“浮层渲染”上做平台适配

## 为什么不把它继续塞进初始化状态？

因为这不是一个问题。

初始化状态解决的是：

- 家庭资料齐没齐
- 首成员有没有
- AI 供应商能不能用
- 第一个管家有没有建好

新手向导解决的是：

- 用户是不是看过产品入口介绍
- 用户会不会找到核心页面
- 用户第一次进入时会不会迷路

这两个状态强行揉在一起，最后只会让路由守卫和体验守卫互相污染。

## 这次明确不做什么？

- 不做后台配置式 CMS 导览编辑器
- 不做语音播报、TTS、动画剧情这类花哨但没必要的东西
- 不做“每次版本升级都强制重新看一遍”的骚扰式导览
- 不做 Web 单端方案，RN 必须从第一版就在范围内
- 不把插件页面深度导览纳入第一版，只覆盖核心固定页面和首页关键区域
