# 需求文档 - 初始化完成后跨端新手向导

状态：Draft

## 简介

初始化向导完成以后，产品现在直接把用户扔到首页。

这看起来像“流程结束了”，实际上只是“系统能用了”。  
对新用户来说，这一步之后最容易发生的事不是继续顺畅使用，而是看着一堆页面发呆，不知道首页、家庭、助手、记忆、设置分别管什么。

这份需求文档定义的是：

- 新手向导什么时候触发
- 它要覆盖哪些页面和内容
- H5 与 RN 怎么共用一套导览逻辑
- 后端只保存什么最小状态
- 用户中断、跳过、重新打开时系统应该怎么处理

## 术语表

- **System**：FamilyClaw user-app 与其配套 API
- **新手向导**：初始化完成后首次进入仪表盘时启动的产品导览流程
- **导览版本**：当前导览脚本的版本号，用来判断成员是否已经完成当前版本导览
- **导览锚点**：页面里可被导览系统高亮和定位的稳定目标
- **跨端导览引擎**：共享同一套导览脚本，但分别适配 H5 和 RN 渲染与定位方式的运行时能力

## 范围说明

### In Scope

- 初始化完成后首次进入仪表盘时自动触发新手向导
- 首页、家庭、助手、记忆、设置等核心页面的跨页面导览
- H5 与 RN 共用导览脚本、共享完成状态
- 成员级导览完成状态的后端持久化
- 跳过、完成、刷新恢复、手动重新打开的规则

### Out of Scope

- 插件页面、第三方页面和所有动态卡片的深度逐项教学
- 语音播报、角色配音、剧情式互动
- 后台可视化导览编辑器

## 需求

### 需求 1：初始化完成后的首次首页进入必须能触发新手向导

**用户故事：** 作为刚完成初始化的新用户，我希望第一次进入首页时能有人带我快速认识产品，而不是让我自己猜每个页面是干什么的。

#### 验收标准

1. WHEN 用户完成初始化向导并首次进入仪表盘 THEN System SHALL 自动启动新手向导。
2. WHEN 用户已经完成当前版本的新手向导 THEN System SHALL 不再自动重复弹出。
3. WHEN 用户是历史老用户且没有“刚完成初始化”的触发条件 THEN System SHALL 不因为新增导览功能而自动打断其正常使用。

### 需求 2：新手向导必须覆盖核心固定页面，而不是只讲首页

**用户故事：** 作为新用户，我希望系统按顺序告诉我首页、家庭、助手、记忆、设置分别做什么，这样我第一次用时不会迷路。

#### 验收标准

1. WHEN 新手向导开始 THEN System SHALL 至少覆盖首页、家庭、助手、记忆、设置五个核心页面或入口。
2. WHEN 某一步进入目标页面 THEN System SHALL 用管家气泡说明该页面的主要用途，而不是只做高亮不解释。
3. WHEN 某页面存在多个重点区域 THEN System SHALL 只介绍与首次上手直接相关的核心区域，不把用户拖进冗长教学。

### 需求 3：H5 与 RN 必须共用一套导览逻辑

**用户故事：** 作为同时使用网页和移动端的用户，我希望两端都能看到一致的新手引导，而不是网页有、RN 没有，或者两边讲的不是一回事。

#### 验收标准

1. WHEN H5 或 RN 启动新手向导 THEN System SHALL 依据同一份导览脚本决定步骤顺序、文案和目标页面。
2. WHEN H5 与 RN 在同一导览步骤定位目标元素 THEN System SHALL 允许平台使用不同的渲染与定位实现，但导览语义必须一致。
3. WHEN 某个平台缺少某个目标区域的可视实现 THEN System SHALL 提供平台内可用的降级说明，而不是让导览流程卡死。

### 需求 4：后端只保存最小完成状态，但这个状态必须能支持版本演进

**用户故事：** 作为产品和研发团队，我希望后端只保存最小必要状态，保持简单，但以后导览版本升级时又不用推倒重来。

#### 验收标准

1. WHEN 成员尚未完成当前版本的新手向导 THEN System SHALL 能从后端状态判断该成员仍未完成当前版本。
2. WHEN 成员完成或主动跳过新手向导 THEN System SHALL 将该成员的导览状态持久化到后端。
3. WHEN 未来导览脚本升级到新版本 THEN System SHALL 能通过版本号判断成员是否需要重新参与新版本导览，而不是新增第二套字段。

### 需求 5：导览过程必须支持中断恢复、跳过和手动重新打开

**用户故事：** 作为用户，我可能在导览过程中刷新页面、切到别的页面，或者暂时不想看；我希望系统行为清楚，不要把我困住，也不要让我永远找不到这套引导。

#### 验收标准

1. WHEN 用户在导览过程中刷新当前端或切换页面 THEN System SHALL 在当前会话内恢复到最近一步，或明确降级到当前页继续说明。
2. WHEN 用户点击跳过 THEN System SHALL 结束本次导览，并将当前版本标记为已处理，避免反复骚扰。
3. WHEN 用户后续主动从设置或其他明确入口重新打开导览 THEN System SHALL 允许重新开始完整导览。

### 需求 6：导览必须兼容角色权限和页面真实状态

**用户故事：** 作为普通成员或管理员，我希望导览只带我看我真能看到、真能用的内容，而不是让我点进无权限页面或者高亮一个根本没渲染出来的区域。

#### 验收标准

1. WHEN 当前成员没有管理员权限 THEN System SHALL 跳过仅管理员可见的导览步骤，或改为只说明普通成员可见内容。
2. WHEN 某个目标区域因为当前数据为空或页面布局差异没有渲染 THEN System SHALL 自动降级为页面级说明，不中断整个流程。
3. WHEN 当前平台或页面状态不满足某一步条件 THEN System SHALL 继续后续可执行步骤，而不是让整个导览失败。

### 需求 7：导览必须以用户能理解的方式呈现，不得破坏正常操作

**用户故事：** 作为第一次上手的用户，我希望导览像一个可靠的管家在带路，而不是一层挡住所有内容、又长又烦的广告弹窗。

#### 验收标准

1. WHEN 导览展示某一步 THEN System SHALL 以管家气泡说明当前重点，并配合清晰的高亮或定位提示。
2. WHEN 用户需要继续、返回、结束或跳过 THEN System SHALL 提供明确的操作按钮。
3. WHEN 导览运行中 THEN System SHALL 不阻断用户关闭导览后继续正常使用产品。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 首页首次加载并触发导览判定 THEN System SHALL 不因为导览状态查询而明显延长首屏可用时间。
2. WHEN 导览跨页面跳转并等待目标锚点出现 THEN System SHALL 在合理超时后自动降级，而不是无限等待。

### 非功能需求 2：可靠性

1. WHEN 后端导览状态保存失败 THEN System SHALL 给出明确提示，并允许用户重试或继续本地结束当前会话。
2. WHEN 某一步锚点注册失败或页面结构变化 THEN System SHALL 记录可排查信息，并继续可执行步骤。

### 非功能需求 3：可维护性与一致性

1. WHEN 研发新增或调整导览步骤 THEN System SHALL 通过共享导览脚本扩展，而不是分别在 H5 和 RN 页面里复制一套流程。
2. WHEN 文案或目标页面发生变更 THEN System SHALL 能在统一配置处完成维护，并同步影响 H5 与 RN。

### 非功能需求 4：可访问性

1. WHEN 用户设备开启减少动画或类似无障碍偏好 THEN System SHALL 降低不必要的动效，保证导览仍然可理解。
2. WHEN 用户只使用键盘或等效操作方式 THEN System SHALL 仍可完成下一步、上一步、跳过和结束操作。

## 成功定义

- 新用户完成初始化后，第一次进入产品时能在 1 到 2 分钟内建立页面认知
- H5 和 RN 都能跑通同一套核心导览，不再出现“一端有引导、一端裸奔”
- 后端只多维护一个成员级导览版本状态，不引入第二套复杂流程状态机
- 历史用户不会因为新手向导上线被强制打断
