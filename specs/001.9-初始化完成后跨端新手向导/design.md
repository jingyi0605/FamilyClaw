# 设计文档 - 初始化完成后跨端新手向导

状态：Draft

## 1. 概述

### 1.1 目标

- 在初始化完成后的首次首页进入时，自动启动一套完整的新手导览
- 用一套共享脚本同时驱动 H5 与 RN，而不是做两个逻辑不同的分叉实现
- 让后端只承担最小持久化职责：保存成员是否完成当前版本导览
- 保证历史用户、普通成员、缺失页面元素等场景都不会把流程搞崩

### 1.2 覆盖需求

- `requirements.md` 需求 1：首次触发规则
- `requirements.md` 需求 2：核心页面覆盖
- `requirements.md` 需求 3：H5 / RN 共用逻辑
- `requirements.md` 需求 4：最小后端状态
- `requirements.md` 需求 5：恢复、跳过、重开
- `requirements.md` 需求 6：权限与缺失目标兼容
- `requirements.md` 需求 7：可理解的管家气泡交互

### 1.3 技术约束

- 后端：FastAPI + SQLAlchemy + Alembic
- 前端：`apps/user-app`，Taro 4，多端页面已有 H5 / RN 分支
- 数据存储：PostgreSQL，成员偏好走 `member_preferences`
- 认证授权：现有 `require_bound_member_actor` / `ensure_actor_can_access_household`
- 外部依赖：无新增第三方导览 SaaS，不引入外部 SDK

## 2. 架构

### 2.1 系统结构

整体结构分三层：

1. **后端最小状态层**
   - 只保存“成员已完成的导览版本号”
   - 提供读取当前状态和回写完成状态的接口
2. **共享导览编排层**
   - 放在 `user-app` 运行时里
   - 负责判断是否自动触发、当前走到哪一步、下一步跳哪一页、何时结束
   - 使用一份共享 `guideManifest`
3. **平台适配层**
   - H5：负责 DOM 锚点查找、浮层定位、遮罩和滚动对齐
   - RN：负责原生视图锚点注册、定位、滚动和浮层展示

核心思路不是“分别做一个 H5 导览和 RN 导览”，而是：

- **同一份导览脚本**
- **同一套运行时状态机**
- **两套薄适配器**

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 后端导览状态模块 | 读写成员已完成导览版本 | 当前成员身份、目标版本 | 当前版本状态、更新结果 |
| 导览运行时 Provider | 管理触发判定、步骤流转、恢复和结束 | 初始化完成来源、本地会话进度、后端状态 | 当前导览会话状态 |
| 导览脚本清单 | 定义步骤顺序、页面、文案、锚点和分支条件 | 导览版本定义 | 可执行步骤列表 |
| H5 平台适配器 | 定位 DOM 锚点、渲染浮层、滚动和遮罩 | 步骤目标、页面结构 | 可视高亮与气泡 |
| RN 平台适配器 | 定位 RN 视图锚点、滚动和浮层 | 步骤目标、页面结构 | 可视高亮与气泡 |
| 页面锚点注册层 | 暴露稳定导览目标，不让脚本依赖脆弱样式类名 | 逻辑锚点 ID | H5 / RN 统一锚点注册结果 |

### 2.3 关键流程

#### 2.3.1 初始化完成后的自动触发

1. 用户在初始化向导完成页点击进入应用。
2. 前端在本地写入一次性“刚完成初始化”的触发标记。
3. 首页壳层启动时读取当前成员的导览版本状态。
4. 如果本地存在“刚完成初始化”标记，且后端版本低于当前导览版本，则自动开启导览。
5. 导览启动后清掉一次性触发标记，避免重复拉起。

#### 2.3.2 跨页面导览执行

1. Provider 从共享导览脚本读取当前步骤。
2. 如果步骤目标页面不是当前页，先导航到目标页面。
3. 页面加载后注册锚点。
4. 平台适配器等待目标锚点出现。
5. 找到锚点则高亮并显示管家气泡；找不到则降级成页面级说明。
6. 用户点击下一步后进入下一条脚本。

#### 2.3.3 跳过、完成和恢复

1. 用户点击跳过时，前端结束当前会话并回写后端版本。
2. 用户完成最后一步时，前端回写后端版本并结束会话。
3. 用户刷新页面时，会话从本地恢复最近一步。
4. 如果本地恢复失败或目标页已经变化，系统降级到当前页最近可执行步骤继续。

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3、5、6、7

- `UserGuideProvider`：导览运行时入口，负责状态机、触发、导航协调和结束回写
- `guideManifest.v1`：共享导览步骤清单，定义页面、锚点、文案和平台可见规则
- `GuideAnchor`：页面锚点注册组件；H5 与 RN 都通过它暴露稳定目标
- `ButlerGuideBubble`：管家气泡 UI，负责展示文案和步骤控制
- `GuidePlatformAdapterH5` / `GuidePlatformAdapterRn`：分别处理浮层定位、滚动、等待锚点和降级策略

### 3.2 数据结构

覆盖需求：3、4、5、6

#### 3.2.1 成员导览状态

建议把该状态落在 `member_preferences`，新增一个明确字段，而不是埋进现有 JSON 里。

理由很简单：

- 这是成员级状态，不是家庭级状态
- 这是一个需要明确查询和升级判断的值，不适合塞到不透明 JSON
- 它虽然只是一项前端体验状态，但 `member_preferences` 本来就是成员偏好落点，比污染 `members` 更干净

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `member_id` | `string` | 是 | 成员 ID | 现有主键 |
| `user_app_guide_version` | `integer \| null` | 否 | 已完成的最新导览版本 | `null` 或 `>= 1` |
| `updated_at` | `string` | 否 | 更新时间 | ISO 时间 |

#### 3.2.2 导览脚本步骤

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `step_id` | `string` | 是 | 稳定步骤 ID | 全局唯一 |
| `route` | `string` | 是 | 目标页面路由 | 必须是 `user-app` 实际页面 |
| `anchor_id` | `string \| null` | 否 | 逻辑锚点 ID | 缺失时表示页面级说明 |
| `title_key` | `string` | 是 | 标题 i18n key | 统一走字典 |
| `content_key` | `string` | 是 | 正文 i18n key | 统一走字典 |
| `placement` | `string` | 否 | 气泡首选方位 | 平台可二次调整 |
| `required_role` | `string \| null` | 否 | 角色限制 | 为空表示所有成员 |
| `optional` | `boolean` | 是 | 找不到目标时能否跳过 | 默认 `false` |

#### 3.2.3 导览运行时状态

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `idle` | 未运行 | 默认状态 | 触发导览 |
| `pending_start` | 已决定启动，等待进入首步 | 满足自动触发条件 | 当前步骤就绪 |
| `navigating` | 正在切换目标页 | 当前步骤目标页不是当前页 | 路由稳定 |
| `waiting_anchor` | 已进目标页，等待锚点注册 | 当前步骤需要锚点 | 锚点可用或超时 |
| `showing` | 正在展示气泡 | 页面或锚点已就绪 | 下一步、上一步、结束 |
| `completing` | 正在写回完成状态 | 最后一步完成或用户跳过 | 保存成功或失败处理完成 |
| `finished` | 本次会话已结束 | 写回成功或本地结束 | 新一轮手动启动 |

### 3.3 接口契约

覆盖需求：1、4、5

#### 3.3.1 读取当前成员导览状态

- 类型：HTTP
- 路径或标识：`GET /member-preferences/{member_id}/guide-status`
- 输入：当前登录成员身份、`member_id`
- 输出：
  - `member_id`
  - `user_app_guide_version`
  - `updated_at`
- 校验：
  - 只能读取当前成员本人或同家庭内允许访问的成员
  - 默认返回 `null` 版本表示从未完成
- 错误：
  - 成员不存在
  - 当前 actor 无权访问该成员

说明：

- 如果实现上更适合把该字段并入现有 `GET /member-preferences/{member_id}` 响应，也可以不新增单独读取接口
- 但无论是否拆独立接口，前端都必须能低成本拿到这个状态

#### 3.3.2 回写当前成员导览状态

- 类型：HTTP
- 路径或标识：`PUT /member-preferences/{member_id}/guide-status`
- 输入：
  - `user_app_guide_version`
- 输出：
  - 最新状态对象
- 校验：
  - 只允许回写当前成员本人
  - 版本号只能前进，不能倒退
- 错误：
  - 输入版本非法
  - 权限不足
  - 数据库写入失败

#### 3.3.3 初始化完成后的触发标记

- 类型：前端本地状态
- 路径或标识：`sessionStorage` / RN 本地会话存储
- 输入：初始化完成页“进入应用”动作
- 输出：一次性本地触发标记
- 校验：
  - 仅表示“刚完成初始化”这一事实
  - 标记使用一次后清除
- 错误：
  - 本地写入失败时不阻断进入应用，但自动导览可能降级为不触发

## 4. 数据与状态模型

### 4.1 数据关系

关系很简单：

- `household.setup_status` 仍然只负责“家庭是否允许进入主链路”
- `member_preferences.user_app_guide_version` 负责“当前成员是否完成当前版本导览”
- 本地会话进度只负责“这一次导览走到哪一步”，不写回后端

这三个层次必须分开：

1. **准入状态**
2. **完成状态**
3. **会话进度**

如果把它们混在一起，后面只会出现一堆莫名其妙的路由判断分支。

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `not_started` | 当前成员未完成当前版本导览 | 后端版本为空或低于当前版本 | 自动启动或手动启动 |
| `in_session` | 当前端正在执行导览 | Provider 启动导览 | 完成、跳过、异常结束 |
| `completed_current_version` | 已完成当前版本导览 | 完成或跳过并成功写回 | 导览脚本升级到新版本 |

## 5. 错误处理

### 5.1 错误类型

- `guide_status_load_failed`：读取导览状态失败
- `guide_status_save_failed`：回写导览状态失败
- `guide_anchor_timeout`：目标锚点在超时内未出现
- `guide_route_unavailable`：目标页面不可进入或当前平台未实现
- `guide_manifest_invalid`：脚本配置错误

### 5.2 错误响应格式

```json
{
  "detail": "导览状态保存失败，请重试",
  "error_code": "guide_status_save_failed",
  "field": "user_app_guide_version",
  "timestamp": "2026-03-19T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：拒绝写入非法版本号，前端提示并保留当前会话。
2. 业务规则错误：版本回退、越权访问等直接返回明确错误。
3. 页面目标缺失：记录告警，降级为页面级说明，然后继续后续步骤。
4. 重试、降级或补偿：
   - 读取失败：本次不自动拉起导览，但允许用户手动重试
   - 保存失败：提示用户重试，并在当前会话结束前保留本地状态
   - 跳转失败：降级到当前页结束说明，不无限重试

## 6. 正确性属性

### 6.1 属性 1：导览完成状态不能阻断产品准入

*对于任何* 已经通过初始化向导的家庭，系统都应该满足：导览是否完成只影响是否自动展示引导，不影响用户进入首页和其他正常页面。

**验证需求：** 需求 1、需求 4

### 6.2 属性 2：同一导览步骤在 H5 与 RN 语义一致

*对于任何* 同一个 `step_id`，系统都应该满足：无论运行在 H5 还是 RN，目标页面、文案含义、前后顺序和结束条件都保持一致。

**验证需求：** 需求 2、需求 3

### 6.3 属性 3：导览遇到局部目标缺失时不能拖垮整条链路

*对于任何* 单一步骤锚点缺失、页面布局差异或角色不可见场景，系统都应该满足：该步骤最多降级或跳过，不影响整条导览完成。

**验证需求：** 需求 5、需求 6

## 8. 风险与待确认项

### 8.1 风险

- RN 端页面结构和 H5 不同，部分锚点可能不能一比一复用
- 如果把状态继续挂在现有 `member_preferences` 全量更新接口上，权限模型会不顺手
- 首页、设置页未来若大改布局，脆弱锚点会失效

### 8.2 待确认项

- RN 当前是否已经有统一可复用的浮层 Portal 能力；如果没有，需要在实现阶段补一个轻量方案
- 设置页里“重新打开导览”的入口放在总设置页还是放在首页帮助入口，产品要尽快定
- 第一版是否需要把“插件页”纳入管理员导览；当前设计默认不纳入
