# Spec 003 - 家庭记忆中心

状态：Draft

## 这份 Spec 到底要解决什么

现在仓库里已经有一套能看“当前状态”的骨架：

- `apps/api-server/app/modules/context/service.py` 能聚合家庭上下文总览
- `apps/api-server/app/modules/family_qa/fact_view_service.py` 能把上下文、提醒、场景拼成问答事实视图
- `apps/api-server/app/modules/family_qa/schemas.py` 里已经预留了 `QaMemorySummary`
- `apps/user-web/src/pages/MemoriesPage.tsx` 已经有“记忆页”外壳

但问题也很直接：

- 还没有真正落库的长期记忆表
- `QaMemorySummary` 现在明确还是“暂未接入”
- 记忆页还是 mock 数据，不是后端真实能力
- 提醒、场景、在家状态、问答还没有统一写回记忆

这就意味着系统现在“会看眼前”，但还谈不上“记住这个家”。

本 Spec 的目标，就是把这块空位补上：让系统能把家庭事实、偏好、事件和关系长期保存，并按权限取出来服务问答、提醒、陪伴和后续主动服务。

## 规划结论

### 【核心判断】

✅ 值得做：这是项目的核心差异化，不是锦上添花。没有长期记忆，这个系统永远只是“会调接口的家庭面板”。

### 【关键洞察】

- 数据结构：真正该稳定保存的不是一坨 prompt，而是两层数据——`event_records` 这种原始事件流水，以及 `memory_cards` 这种可服务的结构化记忆。
- 复杂度：第一版别急着上重型向量库。先把结构化记忆、关键词检索、热摘要、权限过滤做扎实。
- 风险点：最大的风险不是“查不到记忆”，而是“把不该看到的家庭隐私拼进上下文里”。

### 【Linus式方案】

1. 先把原始事件写成 append-only 流水，别一上来就只存摘要。
2. 再从事件里提炼出事实记忆、偏好记忆、事件记忆和关系记忆。
3. 做一个真正可控的 `Context Engine`，按请求目标和权限拼上下文，而不是把所有记忆粗暴塞给模型。
4. 纠错、失效、删除必须有 revision 和审计，不然这套系统迟早烂掉。

## 本 Spec 包含什么

- `requirements.md`：定义家庭记忆中心为什么做、做什么、怎么验收
- `design.md`：定义数据模型、写回链路、检索链路、上下文拼装和错误处理
- `tasks.md`：定义分阶段落地计划、依赖关系、主要改动文件和验证方式

## 本 Spec 明确不做什么

- 先不做照片向量检索和复杂多模态 embedding
- 先不做云端知识库同步
- 先不做无限制的“全量历史对话都喂模型”
- 先不把现有问答逻辑整套推翻重写

## 参考方向

这份规划吸收了两个方向，但没有照抄：

- **OpenClaw 长期记忆方案**：记忆要持久化，且要按能力动态组装上下文，不是把历史一股脑塞进 prompt。
- **OpenClaw Context Engine**：上下文要按能力、角色、请求目标裁剪。
- **lossless-claw 上下文管理思路**：原始上下文不能直接丢，只能在保留原始事件的前提下做摘要、压缩和重算。
