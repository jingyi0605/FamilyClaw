# 需求文档 - 家庭记忆中心

状态：Draft

## 简介

家庭版 OpenClaw 现在已经有成员、关系、权限、房间、设备、提醒、场景、上下文总览和家庭问答的基础骨架，但“长期记忆”还停留在概念层。

当前真实问题不是“模型不够聪明”，而是系统没有稳定的数据结构去记住这个家：

- 在家状态会变，但不会沉淀成可追溯事件
- 提醒完成了，但不会自然形成长期服务线索
- 问答能答一部分，但 `QaMemorySummary` 还没有真正接入
- 用户端记忆页已经有壳，但后面没有真实数据

这个 Spec 要解决的就是这个根问题：把家庭里的事实、偏好、事件、关系沉淀成长期可服务资产，同时保证权限、可纠错、可删除、可回放。

## 术语表

- **System**：家庭版 OpenClaw
- **事件流水（Event Record）**：append-only 的原始记录，表示某件事确实发生过
- **记忆卡（Memory Card）**：从原始事件提炼出的结构化长期记忆，用于检索、问答和服务
- **上下文引擎（Context Engine）**：按请求目标、权限和 token 预算拼装上下文的模块
- **热摘要（Hot Summary）**：为了加速问答和提醒而缓存的高频记忆摘要
- **修订记录（Revision）**：对记忆卡的更正、合并、失效、删除操作历史

## 范围说明

### In Scope

- 统一家庭长期记忆的数据模型
- 建立事件写回与记忆提炼链路
- 将长期记忆接入问答与上下文拼装
- 建立记忆查询、纠错、失效、删除能力
- 接入管理台和用户端记忆页面
- 保证权限过滤、审计、降级和幂等

### Out of Scope

- 第一版不做照片 embedding 和复杂向量检索
- 第一版不做跨家庭的知识共享
- 第一版不做云端长期训练或自动精调
- 第一版不重做全部前端视觉稿，只补齐主链路

## 需求

### 需求 1：统一记忆写回入口

**用户故事：** 作为系统维护者，我希望所有会影响长期记忆的家庭事件都走统一写回入口，以便后续检索、纠错和重算都基于同一份源数据。

#### 验收标准

1. WHEN 在家状态、提醒执行、场景执行、问答结果或人工录入产生可沉淀信息 THEN System SHALL 先写入 append-only 的事件流水，再决定是否生成记忆卡。
2. WHEN 同一来源事件因重试或重复上报再次进入 THEN System SHALL 基于 `dedupe_key` 或等价幂等键避免生成重复记忆。
3. WHEN 记忆提炼过程失败 THEN System SHALL 保留原始事件并标记待重试状态，而不是直接丢失这次输入。

### 需求 2：结构化长期记忆卡

**用户故事：** 作为家庭成员，我希望系统保存的是可理解、可管理的家庭记忆，而不是不可控的 prompt 碎片，以便问答、提醒和陪伴都能建立在真实数据上。

#### 验收标准

1. WHEN System 识别到稳定事实、偏好、关系或重要事件 THEN System SHALL 生成或更新结构化记忆卡，并记录来源事件、相关成员、可见性和置信度。
2. WHEN 同一事实被后续事件再次观察到 THEN System SHALL 更新已有记忆卡的 `last_observed_at`、置信度或内容，而不是无限新增重复卡片。
3. WHEN 记忆属于敏感或私密范围 THEN System SHALL 按可见性规则限制查询和上下文拼装结果。

### 需求 3：长期记忆接入 Context Engine

**用户故事：** 作为问答、提醒和陪伴服务的调用方，我希望系统能按目标动态取出最相关的记忆，而不是把所有历史都塞给模型，以便响应更快、结果更准、泄露更少。

#### 验收标准

1. WHEN 家庭问答、助手、提醒广播或陪伴流程请求上下文 THEN System SHALL 组装“实时上下文 + 热记忆 + 最近事件 + 权限信息”的上下文包。
2. WHEN 当前请求只需要实时状态 THEN System SHALL 优先返回上下文中心已有信息，不强行拉入不相关长期历史。
3. WHEN 实时上下文或热摘要不可用 THEN System SHALL 退回到持久化记忆与事件查询，并在结果中明确标记降级状态。

### 需求 4：长期记忆检索与问答接入

**用户故事：** 作为家庭成员，我希望系统能回答“谁喜欢什么”“谁最近做了什么”“这个家以前发生过什么”，以便长期记忆真的能服务日常生活。

#### 验收标准

1. WHEN 用户提问涉及成员事实、偏好、关系或历史事件 THEN System SHALL 从长期记忆中检索候选结果，并把可解释事实返回给问答模块。
2. WHEN 当前问题同时依赖实时状态和长期记忆 THEN System SHALL 合并上下文事实视图与记忆检索结果，而不是二选一。
3. WHEN 没有命中可用长期记忆 THEN System SHALL 明确返回“当前没有可用记忆”，并提供补录或纠错建议。

### 需求 5：记忆纠错、失效和删除治理

**用户故事：** 作为家庭管理员或被记录成员，我希望错误记忆能被纠正、过期记忆能失效、敏感记忆能删除，以便系统不会一直记着垃圾数据。

#### 验收标准

1. WHEN 管理员或有权限成员更正记忆内容 THEN System SHALL 生成修订记录并更新当前有效版本。
2. WHEN 记忆被标记失效或删除 THEN System SHALL 让后续检索和上下文拼装不再使用该记忆，同时保留必要审计信息。
3. WHEN 无权限成员尝试查看、纠错或删除敏感记忆 THEN System SHALL 拒绝操作并返回明确错误。

### 需求 6：管理台与用户端记忆中心接入

**用户故事：** 作为家庭管理员或普通成员，我希望能在前端查看、搜索、筛选、纠错和删除自己有权限看到的记忆，以便这套能力是可用的，不是后台表结构自嗨。

#### 验收标准

1. WHEN 管理员打开记忆中心 THEN System SHALL 提供记忆列表、筛选、详情、纠错、失效和删除入口。
2. WHEN 普通成员打开用户端记忆页 THEN System SHALL 只返回其有权限查看的记忆，并支持提交纠错建议。
3. WHEN 某条记忆仍在提炼中、待确认或已降级 THEN System SHALL 在前端明确展示状态，而不是伪装成正常完成。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 记忆列表或条件搜索在单家庭内执行 THEN System SHALL 在常规开发环境下于 2 秒内返回首屏结果。
2. WHEN 问答请求需要拼装长期记忆上下文 THEN System SHALL 优先走结构化筛选与热摘要，使额外上下文拼装时间控制在 500 毫秒级别。

### 非功能需求 2：可靠性

1. WHEN 事件写回成功但记忆提炼失败 THEN System SHALL 支持后续重试或回放，不因一次失败丢掉长期记忆来源。
2. WHEN 缓存不可用或部分模块未完成接入 THEN System SHALL 保持现有问答、提醒、上下文接口可降级运行，不破坏已有能力。

### 非功能需求 3：可维护性

1. WHEN 后续增加新的记忆来源（如照片、语音转写、健康设备） THEN System SHALL 通过统一事件写回接口接入，而不是给每个模块各写一套私有逻辑。
2. WHEN 需要追查错误记忆来源 THEN System SHALL 能从记忆卡追到来源事件、修订历史和审计记录。

## 成功定义

- `family_qa` 不再返回“记忆中心暂未接入”作为长期占位说明
- 管理台和用户端都能查询真实记忆数据，而不是 mock
- 提醒、场景、在家状态、人工录入至少四类来源能沉淀为长期记忆
- 纠错、失效、删除都能回溯来源并影响后续检索
- 问答主链路能同时利用实时上下文与长期记忆
