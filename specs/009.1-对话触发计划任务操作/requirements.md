# 需求文档 - 对话触发计划任务操作

状态：Draft

## 简介

现在系统已经能通过独立接口把一句话解析成计划任务草稿，再确认创建正式任务，但用户在真实聊天里还用不上这条能力。

这就很别扭：

- 计划任务底座已经有了
- 对话主链也已经有 proposal 机制
- 但计划任务没有成为 proposal 的正式一种

结果就是用户在聊天里说“每天晚上 9 点提醒我吃药”，系统还不能像处理记忆提案、配置提案那样，直接给出计划任务确认卡。

这份 Spec 要解决的就是这个缺口：把计划任务接进现有对话主链，而不是继续挂一条私有旁路。

目标用户主要有三类：

- 家庭成员：想直接用自然语言创建自己的计划任务
- 家庭管理员：想通过聊天快速创建家庭公共任务或代成员创建任务
- 后端维护者：想复用现有 proposal pipeline，不要把聊天入口拆成两套逻辑

## 术语表

- **System**：FamilyClaw API Server
- **计划任务提案**：用户在聊天里表达计划任务意图后，系统先生成的一条待确认提案
- **任务草稿**：还没正式落库的结构化计划任务候选
- **确认卡片**：聊天里展示给用户确认的一段结构化结果，可以确认也可以取消
- **proposal pipeline**：现有对话提案主链，负责提案识别、持久化、确认和执行

## 范围说明

### In Scope

- 在现有对话提案主链里新增“计划任务创建”这一类提案
- 让用户在正常聊天里表达计划任务意图时，系统能生成计划任务提案
- 提案确认后复用现有计划任务 service 创建正式任务
- 缺关键字段时，能明确告诉前端或聊天层还缺什么
- 保持家庭公共任务、成员私有任务、跨成员创建的权限边界不被绕过

### Out of Scope

- 不做完整多轮对话状态机
- 不做完全自由的 cron 自然语言解析
- 不做在聊天里直接编辑、启停、删除所有计划任务操作
- 不做高风险自动动作型任务的默认聊天放开

## 需求

### 需求 1：聊天主链必须能识别计划任务意图并产生提案

**用户故事：** 作为家庭成员，我希望在正常聊天里说一句计划任务需求时，系统能把它识别出来并生成确认提案，而不是要求我换一套入口。

#### 验收标准

1. WHEN 用户在对话中表达明确计划任务意图 THEN System SHALL 在 proposal pipeline 中生成 `scheduled_task_create` 提案，而不是只生成普通提醒提案。
2. WHEN 同一轮对话里已经能识别出计划任务字段 THEN System SHALL 把字段整理成结构化草稿，并保留提案标题、摘要和证据消息。
3. WHEN 当前消息不构成清晰计划任务意图 THEN System SHALL 不强行生成计划任务提案。

### 需求 2：计划任务提案确认后必须走正式创建入口

**用户故事：** 作为系统维护者，我希望聊天里的计划任务确认仍然复用统一 service，以便权限、校验和审计不会分叉。

#### 验收标准

1. WHEN 用户确认计划任务提案 THEN System SHALL 复用正式计划任务创建逻辑落库，而不是在聊天模块里偷偷写库。
2. WHEN 提案缺少关键字段 THEN System SHALL 不允许直接确认成功，并明确返回缺失字段。
3. WHEN 用户取消计划任务提案 THEN System SHALL 只更新提案状态，不创建正式任务。

### 需求 3：聊天入口必须保留已有权限和风险边界

**用户故事：** 作为家庭管理员，我希望聊天创建更方便，但不希望它绕过成员归属和高风险限制。

#### 验收标准

1. WHEN 普通成员尝试在聊天里创建家庭公共任务或代别人创建任务 THEN System SHALL 按现有权限规则拒绝或要求更高权限。
2. WHEN 计划任务目标缺依赖、插件不支持 `schedule`、Agent 不存在或目标已禁用 THEN System SHALL 返回明确失败结果，而不是静默创建。
3. WHEN 聊天提案涉及高风险执行目标 THEN System SHALL 不默认放行。

### 需求 4：前端和聊天展示必须能看懂这是一条计划任务提案

**用户故事：** 作为家庭成员，我希望确认卡片一眼就能看懂“什么时候提醒、提醒谁、这是长期任务还是一次性动作”，而不是看到一堆内部字段。

#### 验收标准

1. WHEN 计划任务提案返回给前端 THEN System SHALL 提供清晰的标题、摘要、归属和缺失字段信息。
2. WHEN 字段已经完整 THEN System SHALL 标记该提案可以确认。
3. WHEN 字段不完整 THEN System SHALL 明确指出缺什么，而不是只给一个笼统失败消息。

## 非功能需求

### 非功能需求 1：可维护性

1. WHEN 后续继续扩展聊天里的计划任务编辑、启停或删除 THEN System SHALL 复用同一套计划任务草稿和执行逻辑，而不是再开新旁路。
2. WHEN 排查聊天创建任务失败 THEN System SHALL 能通过提案记录、提案 payload 和正式任务结果定位问题。

### 非功能需求 2：可靠性

1. WHEN 聊天提案确认失败 THEN System SHALL 保留失败原因，不把提案直接吃掉。
2. WHEN 系统重试或用户重复确认 THEN System SHALL 避免同一提案重复创建等价任务。

### 非功能需求 3：兼容性

1. WHEN 现有记忆提案、配置提案、提醒提案继续运行 THEN System SHALL 不破坏它们当前行为。
2. WHEN 前端暂时还没专门渲染计划任务提案卡片 THEN System SHALL 至少返回兼容现有 proposal 展示结构的数据。

## 成功定义

- 用户在正常聊天里说一句计划任务需求时，系统能产出正式提案
- 用户确认后能创建正式计划任务，而不是停在草稿接口
- 缺字段、越权、目标依赖失效这些情况都有明确返回
- 现有聊天提案主链不被破坏
