# 设计文档 - 主题包完全插件化改造

状态：Draft

## 1. 概览

### 1.1 目标

- 把现有 8 套默认主题迁成真实 `theme-pack` 插件。
- 让 H5 和 RN 都通过统一主题插件运行时消费主题 token。
- 把插件安装、升级、卸载后的主题资源同步链路补齐。

### 1.2 覆盖需求

- `requirements.md` 需求 1
- `requirements.md` 需求 2
- `requirements.md` 需求 3

### 1.3 技术约束

- 后端：FastAPI、插件注册与挂载体系、插件市场安装链路。
- 前端：Taro `user-app`，同时覆盖 H5 和 RN。
- 数据存储：PostgreSQL + `apps/api-server/data/plugins` 托管插件目录。
- 认证授权：沿用当前家庭访问权限与插件启停权限体系。
- 外部依赖：前端 H5/RN 构建产物、插件市场仓库同步。

## 2. 架构

### 2.1 系统结构

这次改造后，主题资源链路固定为四层：

1. **主题插件目录层**
   内置主题插件放在仓库插件目录和前端 bundle 映射中，远端主题插件放在 `apps/api-server/data/plugins/...`。
2. **后端注册与分发层**
   后端负责扫描 manifest、维护主题资源注册表、按家庭权限和启停状态分发 token 资源。
3. **前端统一运行时层**
   `user-app` 建立统一的主题插件运行时，屏蔽“内置资源”和“远端资源”的来源差异。
4. **平台适配层**
   H5 负责把插件主题 token 映射到 CSS 变量，RN 负责把同一份 token 映射到 RN style token。

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 主题插件 manifest/schema | 声明 `theme_id`、token 资源、版本、预览和适用端 | 插件目录 | 统一插件定义 |
| 后端主题注册表服务 | 汇总当前家庭可见主题插件 | 插件清单、启停状态、家庭上下文 | 主题资源注册表 |
| 后端主题资源接口 | 返回某个插件主题 token 正文 | 插件 ID、`theme_id`、家庭上下文 | 主题 token JSON |
| 前端主题插件运行时 | 统一选择、加载、缓存、失效处理 | 注册表、资源接口、内置资源映射 | 当前主题资源状态 |
| H5/RN 适配层 | 把主题 token 落到具体平台 | 当前主题 token | CSS 变量或 RN token |

### 2.3 关键流程

#### 2.3.1 内置主题插件启动流程

1. 后端扫描内置 `theme-pack` 插件目录，注册现有 8 套主题。
2. `user-app` 在登录前和首屏启动阶段只装配内置主题插件清单，并从随前端包发布的内置资源映射中解析 manifest 对应 token。
3. 前端根据当前选择的 `plugin_id + theme_id` 加载 token；如果当前没有用户选择，则按 manifest 声明的内置默认主题启动，而不是回退到宿主静态主题表。
4. 拿到家庭上下文后，前端再叠加当前家庭可见的远端主题插件注册表，统一交给主题插件运行时管理。

#### 2.3.2 远端主题插件安装与加载流程

1. 插件市场或手动挂载把主题插件安装到 `apps/api-server/data/plugins/...`。
2. 后端同步插件挂载信息，刷新主题资源注册表和资源索引。
3. 前端收到新的注册表快照后，可以拉取该插件主题 token 并应用到 H5 或 RN。

#### 2.3.3 主题插件失效流程

1. 当前主题插件被停用、卸载、升级或 token 校验失败。
2. 后端注册表将该插件标记为不可用或版本失效。
3. 前端运行时把当前主题状态标记为“失效待重选”，提示用户重新选择可用主题插件。

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3

- **`PluginManifestThemePackSpec` 扩展**：补充资源来源、适用端、资源版本和主题 schema 版本。
- **主题资源注册表服务**：替代当前虚拟主题目录逻辑，统一输出元数据和资源索引。
- **主题资源分发接口**：为 H5 和 RN 按需返回插件主题 token 正文。
- **前端主题插件运行时**：统一处理内置主题插件和远端主题插件。

### 3.2 数据结构

覆盖需求：1、2

#### 3.2.1 `ThemePackResourceDescriptor`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `theme_id` | string | 是 | 主题标识 | 与 manifest 一致 |
| `entry_resource` | string | 是 | token 资源入口 | 仅允许插件目录内路径 |
| `resource_source` | enum | 是 | `builtin_bundle` / `managed_plugin_dir` | 内置与远端统一 |
| `resource_version` | string | 是 | 资源版本号 | 升级后必须变化 |
| `platform_targets` | string[] | 是 | 适用端 | 至少包含 `h5` 或 `rn` |
| `theme_schema_version` | integer | 是 | 主题 token schema 版本 | 前后端一致 |

#### 3.2.2 `ThemeRegistryItemV2`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `plugin_id` | string | 是 | 插件 ID | 当前家庭可见 |
| `theme_id` | string | 是 | 主题 ID | 唯一标识一个插件主题资源 |
| `display_name` | string | 是 | 显示名称 | 面向用户 |
| `source_type` | enum | 是 | `builtin` / `official` / `third_party` | 继承插件来源 |
| `enabled` | boolean | 是 | 是否可选 | 受启停规则控制 |
| `resource_source` | enum | 是 | 资源来源类型 | 与资源描述一致 |
| `resource_version` | string | 是 | 当前资源版本 | 用于前端刷新判断 |
| `state` | enum | 是 | `ready` / `disabled` / `invalid` / `stale` | 用于前端显示 |

#### 3.2.3 `BuiltinThemeBundleIndex`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `plugin_id` | string | 是 | 内置插件 ID | 必须与 manifest 一致 |
| `theme_id` | string | 是 | 主题 ID | 必须与 manifest 一致 |
| `bundle_module` | string | 是 | 前端 bundle 内的 token 模块标识 | 只用于解析内置资源，不新增宿主静态主题定义 |
| `resource_version` | string | 是 | 内置资源版本 | 必须与注册表版本一致 |

这个索引只是把“已经是插件目录里的资源”映射到前端构建产物里，不能再保留第二套宿主静态主题表。

### 3.3 接口契约

覆盖需求：2、3

#### 3.3.1 `GET /ai-config/{household_id}/themes`

- 类型：HTTP
- 路径或标识：`/ai-config/{household_id}/themes`
- 输入：家庭 ID、当前登录成员上下文
- 输出：主题资源注册表快照
- 校验：仅返回当前家庭可见、且符合启停规则的主题插件
- 错误：家庭不可见、插件注册异常、资源索引损坏

#### 3.3.2 `GET /ai-config/{household_id}/plugin-themes/{plugin_id}/{theme_id}`

- 类型：HTTP
- 路径或标识：`/ai-config/{household_id}/plugin-themes/{plugin_id}/{theme_id}`
- 输入：家庭 ID、插件 ID、主题 ID
- 输出：主题 token 正文、资源版本、schema 版本、校验信息
- 校验：插件必须当前家庭可见、token 文件存在且校验通过
- 错误：插件不可见、资源不存在、资源无效、版本不匹配

## 4. 数据与状态模型

### 4.1 数据关系

- 一个 `theme-pack` 插件可以声明一个或多个 `theme_id`。
- 一个家庭可以看到多个主题插件，但同一个前端会话同一时刻只会激活一个 `plugin_id + theme_id`。
- 内置主题插件和远端主题插件都进入同一个注册表，由前端统一选择。

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `ready` | 主题资源可正常加载 | manifest 有效，token 可用 | 被停用、升级、卸载或校验失败 |
| `disabled` | 插件被禁用 | 家庭或基础启停状态关闭 | 再次启用 |
| `invalid` | token 内容无效 | JSON 损坏、schema 不符 | 修复资源并重新注册 |
| `stale` | 当前前端持有旧版本 | 插件升级但前端缓存未更新 | 重新拉取新版本 |
| `missing` | 当前选择的资源不存在 | 插件卸载或资源删除 | 用户重新选择可用主题 |

## 5. 错误处理

### 5.1 错误类型

- **资源索引错误**：manifest 声明存在，但 token 入口缺失或越界。
- **资源校验错误**：token JSON 无法解析或结构不符合 schema。
- **前端激活错误**：当前选择的主题插件已经失效。
- **同步错误**：插件安装或升级后，注册表没有正确刷新。

### 5.2 错误响应格式

```json
{
  "detail": "Theme plugin resource is unavailable",
  "error_code": "plugin_theme_resource_unavailable",
  "field": "entry_resource",
  "timestamp": "2026-03-18T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：拒绝注册该主题插件，并记录明确字段路径。
2. 资源校验错误：注册表中标记为 `invalid`，前端不可激活。
3. 远端加载错误：前端显示加载失败，并保留最近一次成功加载的插件主题快照用于诊断。
4. 当前主题失效：前端进入“待重选”状态，不静默回退到宿主默认主题。

## 6. 正确性属性

### 6.1 属性 1：宿主不是主题资源源

*对于任何* `user-app` 取主题 token 场景，系统都应该满足：宿主代码不再保存可直接充当正式主题源的静态主题定义。

**验证需求：** `requirements.md` 需求 1、需求 2

### 6.2 属性 2：H5 和 RN 使用同一份插件主题资源

*对于任何* 主题插件，系统都应该满足：H5 与 RN 使用同一份插件主题 token，只允许在平台适配层做落地映射，不再分叉维护两套 canonical source。

**验证需求：** `requirements.md` 需求 2、需求 3

## 8. 风险与已确认边界

### 8.1 风险

- 当前 H5 主题链路使用静态主题列表和 CSS 变量映射，迁移时容易出现变量遗漏。
- RN 侧目前更多是消费共享 token，真正切到插件资源后可能暴露更多平台差异。
- 插件升级后的主题缓存和界面刷新如果处理不清，容易出现“注册表已更新但页面还是旧主题”的不一致。

### 8.2 已确认边界

- H5 和 RN 共用同一份插件主题 token，平台差异只允许出现在 H5 CSS 变量映射层和 RN style token 映射层。
- 登录页、未登录壳层和首屏启动阶段只使用内置 `theme-pack` 插件；拿到家庭上下文后，再叠加官方和第三方主题插件注册表。
- 运行时不允许宿主 fallback；当前已选主题插件一旦失效，前端必须进入“待重选”状态，而不是静默切回宿主默认主题。
