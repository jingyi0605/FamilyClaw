# 需求文档 - 主题包完全插件化改造

状态：Draft

## 简介

当前主题链路比语言链路更不彻底：

- 现有 8 套默认主题主要还在宿主代码里，以静态 ID、静态 token 和静态列表的方式存在。
- 后端虽然能把主题展示成 `theme-pack`，但本质上还是从宿主硬编码目录生成的虚拟插件。
- H5 的主题写入、RN 的主题 token 消费和插件启停之间还没有完全打通。
- 插件市场安装、升级、卸载后的主题资源同步也没有正式闭环。

这次 Spec 的目标是把主题资源的主导权也彻底交给 `theme-pack` 插件：

- 现有 8 套默认主题都必须变成真实插件目录、manifest 和 token 资源文件。
- `apps/user-app` 不再把 `packages/user-core`、`packages/user-ui`、H5 壳层里的静态主题列表当 canonical source。
- 内置主题插件可以随前端发版，未来官方和第三方主题插件统一从 `data/plugins` 加载，并同时覆盖 H5 和 RN。

## 术语表

- **System**：FamilyClaw 的插件平台、前后端运行时和插件市场安装链路。
- **theme-pack 插件**：声明 `theme_id`、主题 token 资源、预览信息和适用端的正式插件。
- **内置主题插件**：仓库自带、随 `user-app` 一起发布的 `theme-pack` 插件。
- **远端主题插件**：安装到 `apps/api-server/data/plugins` 下，由后端托管分发的官方或第三方 `theme-pack` 插件。
- **主题资源注册表**：后端输出给前端的当前家庭可用主题插件清单，包含插件、版本、状态和资源索引信息。

## 范围说明

### In Scope

- 把现有 8 套默认主题落成真实 `theme-pack` 插件目录。
- 定义主题插件的资源目录规则、manifest 规则和资源版本规则。
- 定义后端主题注册表、主题资源分发接口和安装同步规则。
- 定义 H5 与 RN 共用的主题插件运行时和渲染适配边界。
- 把登录页、未登录壳层和已登录业务页都纳入主题插件链路。

### Out of Scope

- `docs/Documentation` 文档站点主题体系。
- 整个设计系统的额外重做。
- 插件签名、离线资源平台化和商业化能力。

## 需求

### 需求 1：所有主题资源都必须落成真实插件

**用户故事：** 作为平台维护者，我希望所有默认主题和未来主题包都是真实 `theme-pack` 插件，这样主题的注册、启停、升级和排障才有统一边界。

#### 验收标准

1. WHEN 系统检查现有默认主题 THEN System SHALL 让全部 8 套主题都以真实插件目录、manifest 和 token 资源文件存在。
2. WHEN 系统注册主题插件 THEN System SHALL 统一从插件 manifest 读取 `theme_id`、资源入口、版本和适用端信息，而不是从宿主静态列表拼装。
3. WHEN 当前主题插件被停用、卸载或资源损坏 THEN System SHALL 进入明确的“主题失效待重选”状态，而不是静默回退到宿主默认主题。

### 需求 2：前端必须统一走主题插件运行时

**用户故事：** 作为前端维护者，我希望 H5 和 RN 都通过统一主题插件运行时拿到主题 token，这样登录页、业务页和设置页不会继续各自维护一套主题源。

#### 验收标准

1. WHEN `user-app` 的 H5 或 RN 页面请求主题 THEN System SHALL 先解析当前家庭可见的主题插件，再加载对应主题资源。
2. WHEN 当前主题是内置主题插件 THEN System SHALL 从随前端发布的内置插件资源映射中读取 token，而不是读取宿主静态主题定义。
3. WHEN 当前主题是远端主题插件 THEN System SHALL 通过后端资源接口加载主题 token，并用同一套选择语义应用到 H5 和 RN。
4. WHEN 页面处于登录前、首屏启动中或尚未拿到家庭上下文 THEN System SHALL 只使用内置 `theme-pack` 插件 manifest 和随 `user-app` 发布的内置 token 资源完成启动，而不是直接读取宿主静态主题表。

### 需求 3：安装链路必须同步主题资源状态

**用户故事：** 作为插件平台维护者，我希望主题插件安装、升级、卸载后，前端能及时看到新的主题资源状态，这样插件市场和主题切换体验才是一条完整链路。

#### 验收标准

1. WHEN 官方或第三方主题插件安装到 `data/plugins` THEN System SHALL 更新主题资源注册表，并允许 H5 与 RN 加载对应 token。
2. WHEN 主题插件升级或卸载 THEN System SHALL 同步更新资源版本、可用状态和前端可见列表。
3. WHEN 前端当前主题版本已经失效 THEN System SHALL 进入明确的“资源失效待重选”状态，而不是继续用宿主 fallback。

## 非功能需求

### 非功能需求 1：一致性

1. WHEN H5 和 RN 加载同一个 `theme-pack` 插件 THEN System SHALL 使用同一份主题插件资源定义和同一套失效规则。
2. WHEN 内置、官方、第三方主题插件共存 THEN System SHALL 通过同一份注册表和同一套 manifest 管理，而不是为内置主题保留宿主特例。

### 非功能需求 2：可靠性

1. WHEN 远端主题资源接口失败 THEN System SHALL 明确返回错误状态，并允许前端保留最近一次成功加载的插件主题快照用于诊断。
2. WHEN 当前主题插件不可用 THEN System SHALL 阻止静默回退，并给出明确可恢复路径，例如重新选择可用主题插件。

### 非功能需求 3：可维护性

1. WHEN 后续新增主题包 THEN System SHALL 只需要增加新的插件目录和资源文件，而不需要继续修改宿主静态主题列表。
2. WHEN 排查主题问题 THEN System SHALL 让开发者能沿着“manifest -> token 文件 -> 注册表 -> H5/RN 运行时”一条链快速定位。

## 成功定义

- 现有 8 套默认主题都以真实 `theme-pack` 插件存在。
- H5 和 RN 都通过统一主题插件运行时消费资源。
- 登录页、未登录壳层和首屏启动链路也通过内置 `theme-pack` 插件消费主题资源。
- 官方和第三方主题插件可以从 `data/plugins` 安装、注册、分发并被前端加载。
- 宿主不再保留默认主题 fallback，也不再保留宿主静态主题表作为 canonical source。
