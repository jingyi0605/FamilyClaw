# 需求文档 - 插件配置协议与动态表单

状态：Draft

## 简介

当前插件系统最大的问题不是“没有配置页面”，而是“没有统一配置协议”。

具体表现很直接：

- 插件 manifest 里没有正式的通用配置描述
- 后端没有一张正式的插件配置实例表
- 前端不能按协议动态渲染表单，只能硬编码
- `channel` 页面已经写死了各平台字段
- AI provider 虽然有动态字段方案，但没沉淀成插件系统通用能力

这份 Spec 面向三类人：

- **家庭管理员**：希望不同插件都能在一个统一入口里配置，而不是每个插件跳不同页面
- **插件开发者**：希望声明字段和说明就能得到配置界面，而不是顺手再写一套前端
- **前后端开发者**：希望配置协议、校验、保存、secret 处理只有一套，不再每个模块重复造轮子

成功后的核心收益只有一句话：

**插件配置从“页面特例”变成“系统能力”。**

## 术语表

- **System**：FamilyClaw 当前家庭范围内的插件配置系统
- **配置协议**：插件声明配置字段、校验规则、UI 展示提示的统一结构
- **数据 schema**：字段类型、必填、默认值、枚举、长度等数据规则
- **UI schema**：字段顺序、分组、控件类型、占位说明、显示条件等前端提示
- **配置作用域（scope）**：配置实例归属的层级。第一版只支持 `plugin` 和 `channel_account`
- **配置实例**：某个家庭、某个插件、某个作用域下真正保存的一份配置值
- **secret 字段**：像 token、密钥、签名串这类不能在读接口里明文回显的字段
- **动态表单 renderer**：前端根据协议自动渲染表单的通用组件，而不是按插件写死 JSX

## 范围说明

### In Scope

- 通用插件配置协议，包含数据 schema、UI schema、secret 规则
- 家庭范围内插件配置实例持久化
- 基于协议的前端动态表单渲染
- 统一的配置读取、保存、校验和错误返回
- `channel` 插件作为首批迁移对象
- 对无配置协议插件的兼容处理

### Out of Scope

- 低代码拖拽表单设计器
- 任意远程拉取 schema 并直接执行
- 全量历史插件一次性迁移
- AI provider 配置中心这轮直接并到插件系统
- 配置历史版本回滚和审计详情页

## 需求

### 需求 1：插件必须能声明统一的配置协议

**用户故事：** 作为插件开发者，我希望插件 manifest 能正式声明配置字段和展示提示，这样系统才能稳定生成配置页面，而不是每次再写一份特例代码。

#### 验收标准

1. WHEN 插件提供配置能力 THEN System SHALL 在 manifest 中支持稳定的 `config_schema` 和 `ui_schema` 结构，而不是只接受零散自定义字段。
2. WHEN 插件声明字段类型、必填、默认值、枚举或校验规则 THEN System SHALL 能按统一协议解析这些规则。
3. WHEN 插件声明了 System 当前不支持的字段类型、控件类型或显示规则 THEN System SHALL 明确拒绝该配置协议或给出清晰错误，而不是静默忽略。

### 需求 2：系统必须持久化插件配置实例，并按作用域读取与更新

**用户故事：** 作为家庭管理员，我希望我给插件填过的配置能被系统正式保存，并且刷新后还能读回来。

#### 验收标准

1. WHEN 用户保存插件配置 THEN System SHALL 按 `household_id + plugin_id + scope_type + scope_key` 唯一定位并保存配置实例。
2. WHEN 某个作用域下还没有保存记录 THEN System SHALL 返回默认值和空配置状态，而不是返回一堆不确定的临时值。
3. WHEN 插件 schema 版本发生变化 THEN System SHALL 在配置实例里保留对应 `schema_version`，避免后续排查时根本不知道这份配置按哪版协议保存的。

### 需求 3：前端必须按协议动态渲染配置表单

**用户故事：** 作为前端开发者，我希望插件配置页可以直接消费后端协议并渲染表单，这样新增插件字段时不需要再改一遍页面代码。

#### 验收标准

1. WHEN 前端打开某个插件的配置页 THEN System SHALL 根据 `config_schema` 和 `ui_schema` 动态渲染表单，而不是按插件 id 写死字段列表。
2. WHEN UI schema 定义了字段顺序、分组、占位说明或简单显示条件 THEN System SHALL 按这些规则渲染。
3. WHEN 某个字段或控件当前 renderer 不支持 THEN System SHALL 展示明确的降级提示或错误状态，而不是整页空白。

### 需求 4：校验、默认值和 secret 处理必须统一

**用户故事：** 作为系统维护者，我希望无论哪个插件保存配置，校验规则、默认值填充和敏感字段处理都走同一套逻辑，这样不会每个插件再搞一套安全坑。

#### 验收标准

1. WHEN 用户提交不合法配置 THEN System SHALL 在后端统一校验并返回字段级错误，而不是把非法值直接存进数据库。
2. WHEN 字段被标记为 `secret` THEN System SHALL 在读取配置时不返回明文，只返回 `has_value`、`masked` 之类的状态信息。
3. WHEN 用户保存配置时没有提交某个已有的 secret 字段 THEN System SHALL 保留原 secret；WHEN 用户显式清空 THEN System SHALL 能按统一语义删除该 secret。

### 需求 5：`channel` 和后续插件必须能迁到新协议，且不破坏现有能力

**用户故事：** 作为项目维护者，我希望先把 `channel` 迁到新协议验证通路，同时不影响当前插件启停、挂载和运行逻辑。

#### 验收标准

1. WHEN `channel` 插件迁移到新协议 THEN System SHALL 继续支持当前通道配置、账号配置和状态探测能力，不允许功能回退。
2. WHEN 某个旧插件暂时还没有提供配置协议 THEN System SHALL 继续允许它被列表展示、启停和执行，不因为没有配置表单就坏掉。
3. WHEN 后续新增一个有配置的插件 THEN System SHALL 在不新增专用前端页面的前提下，提供基础可用的配置表单能力。

## 非功能需求

### 非功能需求 1：兼容性

1. WHEN 旧插件没有 `config_schema` THEN System SHALL 把它视为“无可配置项”，而不是接口报错。
2. WHEN 当前家庭已经有旧的通道配置或插件配置数据 THEN System SHALL 提供迁移路径，不要求用户手工重填所有字段。

### 非功能需求 2：可扩展性

1. WHEN 后续新增新的插件字段类型或新的配置作用域 THEN System SHALL 在现有协议上扩展，而不是推翻第一版协议重来。
2. WHEN 后续需要新增新的配置页面入口 THEN System SHALL 继续复用同一套读取、保存和 renderer 机制，而不是复制一套新表单系统。

### 非功能需求 3：可观测性与可测试性

1. WHEN 配置保存失败 THEN System SHALL 返回可定位到字段和原因的结构化错误。
2. WHEN 排查“某个插件为什么没有生效” THEN System SHALL 能区分 schema 问题、配置缺失、secret 缺失和作用域不匹配等原因。

## 成功定义

- 插件 manifest 能正式声明配置协议，而不是继续塞零散字段
- 后端有正式的插件配置实例持久化模型和读写接口
- 前端可以按协议渲染插件配置表单，不再按插件硬编码字段
- secret 字段不会在读接口明文泄漏，保存语义统一
- `channel` 首批迁移后，现有能力不回退，后续插件也能复用这套协议
