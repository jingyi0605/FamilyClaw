# Spec 004.2.3 - 插件配置协议与动态表单

状态：Draft

## 这份 Spec 解决什么问题

当前插件系统已经能做三件事：

- 知道有哪些插件
- 管理插件启停
- 执行插件任务

但它还缺一层很关键的东西：**插件怎么配置**。

现在的坏味道已经很明显了：

- `PluginManifest` 没有正式的通用 `config_schema` / `ui_schema`
- 后端没有插件配置实例持久化模型
- 前端插件页没有通用动态表单 renderer
- `channel` 方向有一点零散的 UI 字段雏形，但没有打通
- AI provider 又单独长出了一套动态字段协议，和插件系统没有统一

结果就是一句人话：每来一个要配置的插件，就得再手写一套页面、接口和存储。

这不是功能不完整，这是数据结构没立住。

## 这次要做到什么

- 定义一套通用插件配置协议，明确数据 schema、UI schema、字段校验和敏感字段规则
- 让插件能声明“我需要哪些字段、怎么展示、哪些是 secret”
- 给后端补一层正式的插件配置实例持久化，而不是继续散落在各个业务模块里
- 给前端补一套动态表单渲染能力，让插件配置页不再按插件一份份硬编码
- 先把 `channel` 作为第一批迁移对象，验证这套协议真能落地

## 这次明确不做什么

- 不做插件市场改造
- 不做低代码表单设计器
- 不做任意远程 schema 拉取和执行
- 不做所有历史插件一次性迁移
- 不把 AI provider 现有配置页这轮强行并入插件系统

## 为什么现在做

因为这个问题已经不是“将来也许会乱”，而是现在就已经乱了：

- 一个方向在硬编码页面
- 一个方向在后端 schema 里留半套结构
- 另一个方向单独做了一套字段协议

继续这样搞，只会得到更多“这个插件特殊一点，所以先单独写”的垃圾代码。

更简单的路不是再补一个页面，而是把插件配置这层协议先定死。

## 当前的核心判断

- 这是真问题，不是臆想需求
- 更简单的方法不是继续给每类插件写专用表单，而是把配置模型收成一个协议
- 最大风险不是表单不好看，而是不同插件把配置、校验、secret 存储各搞一套，最后没人能维护

## 这次怎么定边界

这次把问题拆成三层，但按一件事一起设计：

1. **协议层**
   - 插件 manifest 如何声明配置字段
   - 哪些字段是数据规则，哪些是 UI 提示
2. **持久化层**
   - 当前家庭下某个插件的配置怎么存
   - secret 怎么存，怎么读，怎么清空
3. **渲染层**
   - 前端怎么按协议动态渲染表单
   - 错误、默认值、禁用状态、字段依赖怎么表现

少任何一层，这件事都会重新碎回“一个插件一个特例”。

## 与其他 Spec 的关系

  - 这份是插件系统底座。
  - `004.2.3` 负责把“插件能不能被配置”这件事补完整。

  - `004.4` 负责插件管理页和市场页。
  - `004.2.3` 负责插件配置表单怎么生成、怎么保存，给 `004.4` 提供稳定配置入口。
- `../010-通讯通道插件与多平台机器人接入/`
  - `010` 里大量通道插件都需要单独配置页面。
  - `004.2.3` 是它们后续从硬编码表单迁到统一协议的基础。

## 主要文档

- `requirements.md`：说清楚这次到底要交付哪些配置能力，什么算做完
- `design.md`：说清楚协议结构、存储模型、接口契约、动态渲染和迁移方案
- `tasks.md`：说清楚按什么顺序落地，先定协议，再做持久化，再接前端，再迁移 `channel`
- `docs/README.md`：说明后续补 API 示例、manifest 示例、迁移说明和联调清单放哪里
