# channel 首批迁移说明

这一轮只干一件事：

- 把 `SettingsChannelAccessPage` 从页面硬编码字段，切到插件协议驱动

别再让页面自己定义 Telegram、Discord、飞书这些字段。那是脏数据源。

## 这次迁了什么

1. 通道插件在 `manifest.json` 里正式声明 `config_specs`
2. 通道账号配置页按 `channel_account` 作用域去读表单
3. 保存时先保存账号元数据，再保存配置实例
4. 保存配置实例时同步回写旧 `channel_plugin_accounts.config_json`

## 为什么还要同步旧 `config_json`

因为现有通道探测、挂载、收发链路还在吃这份数据。

如果现在只写新表，不回写旧字段，就是典型的“代码看起来更干净，用户直接坏掉”。

这不叫重构，这叫破坏用户空间。

## 页面侧现在的事实来源

只有一份：

- 插件 manifest 里的 `config_specs`

页面只做这些事：

- 请求作用域列表
- 读取表单
- 按协议渲染
- 提交配置

页面不再做这些事：

- 维护平台字段常量
- 按插件 id 写死输入框
- 偷偷猜某个字段是不是 secret

## 迁移后要回归什么

至少看这几项：

1. 新建 channel 账号时，能按协议看到字段
2. 编辑 channel 账号时，旧 `config_json` 能被兼容读出来
3. secret 字段不会回显明文
4. 不提交 secret 字段时会保留原值
5. 显式清空 secret 时，旧运行时配置也会同步去掉
6. 账号创建、编辑、探测、成员绑定链路不能回退

## 已知边界

这一版只支持：

- `plugin`
- `channel_account`

这一版不支持：

- 更细粒度的多层作用域
- 远程异步选项
- 任意表达式可见性
- 自定义脚本校验

先把最小可用版本站稳，再谈后面的花活。
