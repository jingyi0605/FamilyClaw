# 插件仓库自动校验规则

## 这份文档是干什么的

这份文档回答的是：

机器人到底查什么，查不过时为什么不过。

不要把“自动校验”写成一句空话。真正落地时，必须能说清楚每条失败是怎么来的。

## 1. 仓库级检查

### 1.1 仓库地址必须合法

要求：

- 必须是 GitHub 仓库地址
- 必须能解析出 owner 和 repo
- 不接受 release asset 直链冒充仓库地址

失败示例：

- 填的是压缩包下载链接
- 填的是主页，不是仓库
- URL 结构不完整

### 1.2 仓库必须可访问

要求：

- 至少能访问默认分支或作者指定分支
- 仓库不是 404
- 仓库不是私有且机器人无权限

失败时应该返回：

- `plugin_repo_unreachable`

## 2. 文件级检查

### 2.1 `manifest.json` 必须存在

要求：

- Issue 指定路径存在
- 文件内容能被当前插件规范正常解析

失败时应该返回：

- `manifest_invalid`

### 2.2 README 必须存在

要求：

- Issue 指定 README 路径存在
- 不是空文件

README 不需要写成小说，但不能没有。

### 2.3 安装相关文件必须能对上

机器人至少要检查：

- `manifest.json`
- README
- `requirements.txt`
- 插件包根目录
- 版本信息引用的安装来源

如果市场条目准备走 `source_archive`，那至少要能定位源码仓库和目标版本。

## 3. `manifest.json` 内容检查

### 3.1 `plugin_id` 必须稳定

要求：

- `id` 存在
- 格式合法
- 后续生成目录时能安全落到 `plugins/<plugin_id>/entry.json`

### 3.2 插件名和描述必须可读

要求：

- `name` 存在
- 描述信息足够生成市场摘要

如果插件仓库完全没有可用描述，机器人可以允许作者用 `summary_override` 补，但不能无限放水。

### 3.3 权限和风险声明必须可消费

要求：

- 权限字段存在且格式正常
- 风险等级字段存在且值合法

这两项不是摆设，市场前端和审核都要靠它们。

### 3.4 安装版本信息必须自洽

要求：

- 至少有一个可安装版本
- `latest_version` 必须能在版本列表里找到
- 版本引用的 tag / git ref 不能明显对不上

## 4. README 检查

README 至少应该让维护者看懂这几个问题：

- 这个插件干什么
- 需要什么配置
- 依赖什么外部平台
- 怎么验证

机器人不需要理解整篇 README，但至少要挡掉这类垃圾：

- README 缺失
- README 空白
- README 只写一句“一个强大的插件”

## 5. 生成条目前的交叉检查

机器人在生成 `entry.json` 前，至少要再做一次交叉核对：

- Issue 里声明的仓库地址和真实拉取源一致
- 生成出的 `plugin_id` 和目标路径一致
- 权限、风险等级和 `manifest.json` 一致
- 版本信息和安装来源一致

只要这些信息打架，就不要生成正式条目草案。

## 6. 错误分类规则

### 6.1 作者可修复错误

这类错误应该标记为 `failed`，并明确要求作者补：

- Issue 字段缺失
- 仓库地址错误
- `manifest.json` 缺失
- README 缺失
- 版本信息不完整

### 6.2 系统异常

这类错误应该标记为 `system_error`，不要甩锅给作者：

- GitHub API 限流
- 网络超时
- 工作流权限异常
- 机器人脚本自身崩溃

## 7. 明确不做的自动判断

这些事情机器人不该装懂：

- 自动决定插件是否值得收录
- 自动决定是否官方推荐
- 自动给出最终分类结论
- 自动批准高风险插件

机器人负责查事实，不负责替维护者下最终判断。

## 一句实话

自动校验最重要的不是“尽量多做”，而是“把能稳定判断的事实查清楚，把不能稳定判断的东西留给人”。
