# Spec 004.6.2 - 插件市场 Issue 提交、自动校验与机器人收录

状态：Draft

## 这份 Spec 解决什么问题

现在插件市场的注册入口还是“提 PR 改 `entry.json`”。

这条路不是不能用，但对大多数第三方作者来说太硬了：

- 不知道市场仓库目录怎么放
- 不知道 `entry.json` 该填什么
- 不知道哪些字段可以从插件仓库自动读出来
- 提交材料容易漏，审核的人还得手工补

结果就是两边都烦：

- 提交方靠猜
- 审核方靠人工擦屁股

这份 Spec 要把提交流程收口成一条更像人话的主链路：

- 第三方作者先提 Issue
- 机器人自动检查插件仓库和 `manifest.json`
- 能自动生成的市场条目就自动生成
- 机器人发 PR
- 维护者只审核 PR，不手写条目

一句话说清楚：

这次不是做“Issue 一提交就自动进市场”，而是做“Issue 成为正式收录入口，机器人帮你把脏活做掉，但人工审核边界继续保留”。

## 为什么不能直接 Issue 写进 main

这个问题必须先钉死。

如果走：

- Issue
- 机器人直接改 `main`

那市场仓库很快就会烂掉。原因很简单：

- Issue 里的信息可能是错的
- 仓库地址可能是假的
- `manifest` 和条目可能不一致
- 风险等级和权限声明可能在撒谎
- 维护者会失去最后一道收口审核

所以这份 Spec 的底线是：

- Issue 不是正式收录结果
- PR 才是正式收录候选
- 合并后的仓库内容才是市场事实来源

## 这份 Spec 要把哪几件事写死

### 1. 提交入口

- 市场仓库提供正式 Issue Form
- 第三方作者通过 Issue 提交收录申请
- 不再要求作者先会写 `entry.json`

### 2. 自动校验

- 机器人自动检查插件源码仓库是否可访问
- 自动读取 `manifest.json`
- 自动检查 README、版本、权限、风险和安装信息

### 3. 自动生成条目

- 能从插件仓库和 Issue 推导出来的字段，机器人直接生成
- 生成结果落到 `plugins/<plugin_id>/entry.json`
- 机器人负责创建或更新 PR

### 4. 人工审核边界

- 维护者审核的是机器人生成的 PR
- 维护者不需要手写条目
- 没合并前，不进入正式市场

### 5. 明确不做什么

- 不做 Issue 直接写主分支
- 不做“提 Issue 就自动上架”
- 不做根据市场仓库 Issue 自动修改所有用户实例的市场源配置
- 不做 GitHub App 级别的复杂双向审批流
- 不做付费、签名、信誉分之类的扩展治理系统

## 一句话结论

`004.6.2` 要解决的不是“怎么多加一种提交方式”，而是把市场收录流程做成一条稳定、低摩擦、可自动化、但不失控的链路：

- 作者提 Issue
- 机器人做脏活
- 维护者做判断
- 合并结果才进入市场
