# 20260316-H5单位策略与designWidth边界

## 这份文档解决什么问题

最近几次 H5 页面尺寸异常，核心不是“设计稿突然不对了”，而是项目里同时混用了三套尺寸语义：

1. 会被 Taro H5 换算的设计稿尺寸
2. 手写 `rem`
3. JSX/TS 行内样式里的固定浏览器像素，比如 `'24px'`

这三套东西混在一起，就一定会有人看着页面说“怎么整体变大了”，然后再有人想去改 `designWidth` 救火。这个方向大概率会把事情越修越坏，所以这里把边界直接写死。

## 当前已确认事实

### 1. 全局设计稿基准现在是 `750`

- `apps/user-app/config/index.ts:38`
  - `designWidth: 750`

这说明项目当前默认假设：样式源码里的设计稿尺寸，是按 750 宽设计稿来换算的。

### 2. H5 配置里没有额外覆盖 `pxtransform`

- `apps/user-app/config/platform/h5.ts`
  - 当前只配置了路由、输出和 devServer
  - 没有看到 `baseFontSize`、`maxRootSize`、`minRootSize`、`targetUnit` 这类 H5 尺寸换算覆盖

结论很直接：现在基本是在吃 Taro H5 默认的尺寸换算行为。

### 3. 仓库里已经有人踩过这个坑，而且写了注释

- `apps/user-app/src/pages/assistant/index.h5.scss:28`
  - 明确写了：Taro H5 的 rem 根字号会放大旧页尺寸
- `apps/user-app/src/runtime/h5-shell/theme/tokens.ts:73`
  - 明确写了：Taro H5 会把 rem 挂到 40px 根字号上，所以壳层字体 token 必须固定成 px
- `apps/user-app/src/runtime/h5-shell/styles/index.h5.scss:5999`
  - 明确写了：这里不用 rem，避免 Taro H5 根字号放大后把加载态整体撑大

这说明问题不是猜测，是项目里已经反复出现过的真实问题。

### 4. 现在确实还在混用手写 `rem` 和固定像素

已确认例子：

- `apps/user-app/src/pages/home/index.h5.scss`
  - 多处 `clamp(rem, vw, rem)`
- `apps/user-app/src/pages/setup/WelcomeStep.css`
  - 多处 `clamp(rem, vw, rem)` 和直接手写 `rem`
- `apps/user-app/src/components/AppUi.tsx`
  - 多处 `'24px'`、`'26px'`
- `apps/user-app/src/components/AuthShellPage.tsx`
  - 多处 `'24px'`、`'26px'`
- `apps/user-app/src/components/MainShellPage.tsx`
  - 多处 `'24px'`

重点要记住一件事：

- SCSS 里的 `px`，语义上还是“设计稿尺寸源码单位”，会走 Taro H5 的换算链路
- JSX/TS 行内样式里的 `'24px'`，是浏览器固定像素，不会自动走同一套换算链路

这两个看起来都叫 `px`，其实根本不是一回事。

## 先回答结论

### 结论 1：长期继续把问题改成硬编码固定 `px`，不是好路子

可以救急，但不能当长期标准。

### 结论 2：`designWidth` 不是修所有尺寸问题的万能开关

它只能修“设计稿基准设错了”，修不了“单位体系已经混乱”。

### 结论 3：真正该收口的是单位语义，不是先改一个全局配置碰碰运气

## 为什么“到处把 rem 改成 px 硬编码”长期有问题

### 1. 它把“设计稿尺寸”和“浏览器固定像素”混成一锅

如果一个页面里：

- 一部分尺寸来自 SCSS 里的 `px`，会被 Taro 换算
- 一部分尺寸来自 JSX 里的 `'24px'`，不会被 Taro 换算

那页面就不再存在统一比例。后面谁来维护，谁都只能靠猜。

### 2. 它会让跨端 token 失去意义

我们这次做 UI 标准化，本来就是想让页面优先吃 token 和基础组件。

如果页面继续直接写大量固定 `'24px'`、`'32px'`、`'48px'`，那 token 只是摆设。后面想统一字号、卡片圆角、间距，只能继续全仓硬搜。

### 3. 它会抬高后续改版成本

今天为了救某个页面先钉死几个 `px`，短期能过。

但等你后面要：

- 换字号阶梯
- 换卡片密度
- 调整移动端和桌面的间距策略
- 做跨端一致性校准

这些硬编码都会变成债。

### 4. 它会制造“局部看着正常，全局比例越来越乱”的假稳定

最糟糕的不是页面炸了，而是页面没炸，但整个系统比例慢慢散掉了：

- 标题像一个系统
- 表单像另一个系统
- 壳层又像第三个系统

这种代码最难清理，因为每个局部都能找出一个“我这里必须例外”的理由。

## 那是不是只要把 `designWidth` 改合理就行

不是。

### `designWidth` 真正负责的只有一件事

回答这个问题：

“样式源码里的设计稿尺寸，是按多宽的设计稿来算的？”

### 它不负责下面这些事

1. 不负责修手写 `rem`
2. 不负责修 JSX/TS 行内固定 `'24px'`
3. 不负责修一个页面里同时混用两套尺寸语义
4. 不负责修历史页面按浏览器 16px 根字号心智写出来的旧样式

### 什么情况下才应该改 `designWidth`

只有一种正当理由：

项目真实设计稿基准从一开始就不是 750，而我们配错了。

比如设计师所有稿子本来都是按 375 基准出，但项目配置成了 750，那这才叫配置错了。

### 为什么随手改 `designWidth` 风险很大

因为它是全局换算基准。

一旦改了，所有依赖 Taro 换算链路的页面都会一起变。你以为你在修一个页面，实际上是在给整仓做全局缩放。

如果现有问题本质上是：

- 某些页面手写了 `rem`
- 某些组件写了 JSX 固定像素
- 某些地方又沿用了设计稿尺寸 `px`

那你动 `designWidth` 以后，只会让这三套东西错得更不整齐。

## 这次 Spec 里建议的正式规则

### 规则 1：H5 样式文件默认只写设计稿尺寸源码单位 `px`

也就是：

- 在 `.scss/.css` 里写 `24px`
- 语义上把它当成“设计稿尺寸”
- 交给 Taro H5 去换算

不要把它理解成浏览器固定 24 像素。

### 规则 2：默认禁止手写 `rem`

原因很简单：

- `rem` 直接绑定 H5 根字号
- 当前项目已经多次证明这会把页面整体放大
- 这套心智和 Taro 的设计稿换算不是一回事

允许例外，但必须写明原因，而且只能留在局部。

### 规则 3：JSX/TS 行内样式里禁止直接写固定 `'24px'`

优先级应该是：

1. 先用共享组件
2. 再用 token
3. 实在不行，再走显式转换函数
4. 只有明确需要固定浏览器像素时，才允许保留 `'24px'`，并写清原因

### 规则 4：`clamp()` 可以用，但默认不要拿 `rem` 当主单位

推荐：

- `clamp(24px, 2vw, 32px)`

不推荐作为默认写法：

- `clamp(1.5rem, 2vw, 2rem)`

前者至少语义还和当前设计稿尺寸体系一致，后者会继续绑定根字号。

## 对后续实现的直接影响

接下来真要开始改代码，优先顺序应该是：

1. 先审计手写 `rem`
2. 再审计 JSX/TS 里的固定像素
3. 先把共享 token 和基础组件收口
4. 再迁移 `AppUi.tsx`、壳层和高频页面
5. 最后再决定要不要动 `designWidth`

顺序不能反。

先动 `designWidth`，就是在没搞清楚病因前先下猛药，风险最大。

## 这份文档的最终判断

### 值得做

值得，而且应该先做规则收口，再做页面迁移。

### 核心原因

- 真问题不是“某几个页面尺寸不对”
- 真问题是 `designWidth`、手写 `rem`、样式文件 `px`、行内 `'24px'` 这些东西已经混出了多套语义

### 直接结论

- 不要继续把“统一改成固定 `px`”当长期方案
- 不要把“改 `designWidth`”当默认修复动作
- 先统一单位语义，再做组件和页面迁移
