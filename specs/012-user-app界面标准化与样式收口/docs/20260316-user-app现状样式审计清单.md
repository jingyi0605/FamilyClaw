## 补充审计结果（2026-03-16）

按 `.scss/.css` 文件里直接匹配 `rem` 统计，当前至少有下面这些集中点：

- `apps/user-app/src/runtime/h5-shell/styles/index.h5.scss`
  - 261 处
  - 最大历史包袱区，壳层和多块 H5 页面样式揉在一起，不能继续往这里堆新规则
- `apps/user-app/src/pages/setup/index.h5.scss`
  - 41 处
  - 主要是 setup H5 页面布局和卡片区域

  - 24 处
  - 主要是 `clamp(rem, vw, rem)` 和响应式间距写法
- `apps/user-app/src/pages/setup/WelcomeStep.css`
  - 11 处
  - 欢迎页动效、按钮和标题字号仍然直接绑 `rem`

- 最大风险不是某一处 `rem`，而是壳层和页面继续把 `rem` 当默认单位扩散

- `setup/index.h5.tsx` 和 `runtime/guard.tsx` 说明“运行时兜底页面”也还在绕过统一单位策略

### 8. token 来源不只是两套，实际上已经长到三套

当前已确认的来源与重复定义：

- `packages/user-ui/src/theme/tokens.ts`
  - 只有一层很薄的 `userAppTokens`
  - 只覆盖 14 个基础字段：颜色 9 个、间距 3 个、圆角 2 个
  - 缺字体阶梯、阴影、边框层级、状态变体、组件 token、主题列表
- `apps/user-app/src/runtime/h5-shell/theme/tokens.ts`
  - 维护完整的 `ThemeTokens` 结构、主题列表和实际主题值
  - 同时承担“标准源”和“H5 运行时主题数据”两种职责
- `apps/user-app/src/pages/login/theme-presets.ts`
  - 又复制了一套登录页专用 CSS 变量 preset
  - 里面重复了圆角、字号、间距、阴影和主题色

当前判断：

- 现在不是“共享层一套、H5 一套”这么简单，而是“共享层薄 token + H5 完整 token + 登录页 preset”三套并行
- 如果 1.2 不把 `login/theme-presets.ts` 一起纳入边界，后面还是会出现页面绕过共享层自己复制变量

### 9. `packages/user-ui` 和 H5 主题层的职责边界需要直接写死

建议从这一步开始按下面的边界执行：

| 层 | 应该负责什么 | 这一步明确不再负责什么 |
| --- | --- | --- |
| `packages/user-ui/src/theme/` | 唯一上游标准源：foundation token、semantic token、component token、跨端共享导出 | 不再只做一个薄壳映射，不再把完整标准留在 H5 私有目录 |
| `packages/user-ui/src/components/` | 共享基础组件和稳定变体出口 | 不继续只留两个示例组件，让页面自己补样式 |
| `apps/user-app/src/runtime/h5-shell/theme/` | H5 主题选择状态、共享 token 到 CSS 变量的映射、兼容旧变量名 | 不再维护另一套完整 canonical token 值 |
| `apps/user-app/src/components/AppUi.tsx` | 短期兼容层，帮助旧页面过渡 | 不再继续新增固定字号、圆角、间距当正式标准 |
| `apps/user-app/src/pages/**` | 消费共享 token 和基础组件 | 不再复制主题 preset，不再新增手写 `rem` 和固定 `'24px'` 一类尺寸 |

### 10. 1.1 阶段的直接结论

- 唯一 UI 标准源必须上移到 `packages/user-ui`，否则 H5 壳层永远会继续“先有一套私货，再映射一层共享壳”
- `apps/user-app/src/runtime/h5-shell/theme` 应该降级成适配层，而不是标准源
- `login/theme-presets.ts` 已经属于重复 token 定义，不能继续当例外长期保留
- 后续进入 1.2 时，重点不是先改页面，而是先把共享 token 分层、H5 映射链路和登录页 preset 的来源关系收口

## 当前最应该优先处理的对象

### P1：`AppUi.tsx`

原因：

- 它是公共组件入口
- 里面已经有多处固定字号和间距
- 不先收口这里，后面页面迁移会继续复制旧写法

### P1：壳层主题与壳层样式

原因：

- `tokens.ts` 和 `index.h5.scss` 已经承担了全局语义
- 如果共享层不上移，这里会长期和 `packages/user-ui` 并行长两套标准

### P1：`assistant` / `setup-guard` 这类已经出现尺寸异常的区域

原因：

- 这些位置已经有明确的“避坑注释”
- 适合反向提炼正式规则，而不是继续靠局部补丁

## 当前不应该急着做的事

### 1. 不要先全局改 `designWidth`

原因：

- 还没有证据表明设计稿基准本身配错
- 现有问题更像是单位语义混用

### 2. 不要先把所有页面一起重写

原因：

- 真正的瓶颈是标准源没收口
- 标准层没立住，页面改得越多，返工越多

### 3. 不要继续让行内 `'24px'` 扩散

原因：

- 这是最容易复制、最难统一回收的写法

## 下一步建议

1. 继续补完整的单位审计，至少分清手写 `rem`、样式文件 `px`、行内固定像素三类来源
2. 先把 `packages/user-ui` 里的 token 分层和导出入口补齐
3. 设计 `AppUi.tsx` 到共享层的归并方案
4. 给新增代码补最小可执行检查，先拦住新的手写 `rem` 和新的行内固定尺寸
