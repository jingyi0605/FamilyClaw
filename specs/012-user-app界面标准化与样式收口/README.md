# Spec 012 - user-app界面标准化与样式收口

状态：Draft

## 这份 Spec 解决什么问题？

`apps/user-app` 现在最大的前端问题，不是某一个页面丑，而是整套界面规则没有真正收口。

仓库里其实已经有两套半成品：

- `apps/user-app/src/runtime/h5-shell/theme/tokens.ts` 里有一套比较完整的 H5 主题 token
- `packages/user-ui/src/theme/tokens.ts` 里有一套很薄的跨端 token 映射
- 页面和组件层又大量直接写 `style={{ ... }}`、字号、圆角、间距和颜色

结果就是：

1. 字体、字号、圆角、阴影、卡片和按钮没有统一标准
2. 同一种 UI 元素在不同页面长得不一样
3. 主题变量虽然存在，但很多组件根本没真正使用统一变量
4. 想改样式时，得去翻页面和组件里的硬编码，改一次漏一次
5. H5 里的尺寸体系已经混了三套语义：Taro 会换算的设计稿尺寸、手写 `rem`、以及 JSX 行内的固定像素

这不是审美问题，这是结构问题。继续边修页面边补样式，只会把代码越补越烂。

## 这次拍板要做什么？

- 给 `user-app` 先立一套真正可执行的 UI 标准
- 明确唯一的样式标准来源，结束“页面自己写一套、`user-ui` 再写一套”的状态
- 在 `packages/user-ui` 里沉淀语义 token、基础组件和常用变体
- 让 `apps/user-app` 的页面和业务组件优先消费变量和基础组件，而不是继续写散装样式
- 明确 `designWidth`、`pxtransform`、`px/rem` 和行内样式的单位使用规则，结束现在这种靠猜的写法
- 给后续新增页面立规矩，避免刚统一完又重新长歪

## 这次明确不做什么？

- 不在这一轮重做整套视觉品牌
- 不为追求“完美设计系统”去发明一堆没人会用的抽象
- 不一次性重写 `user-app` 所有页面
- 不改 `user-web`，项目规则已经写明前端新增和修改要落在 `user-app`
- 不把 H5 专用视觉效果强行塞进 RN 和其他端，跨端优先保证结构一致、层级一致、语义一致
- 不把“全局改 `designWidth`”当成修所有尺寸问题的万能开关

## 这份 Spec 的核心判断

- 这是个真问题，而且已经开始拖慢开发效率
- 最简单的做法不是“逐页修好看一点”，而是先收口样式标准，再批量替换
- `designWidth` 只是设计稿基准，不是拿来修补旧页面 `rem` 心智错乱的止痛药
- 最大风险不是工作量大，而是又搞出第二套标准，最后比现在更乱

## 主要文档

  - 预留后续补充的审计表、token 对照表、单位策略文档和迁移清单

## 当前阶段策略

- 先定唯一标准源，再改组件，不反过来
- 先做高复用基础件，再迁移页面，不从最长最乱的页面硬啃
- 先把 H5 单位规则说清楚，再谈是否该动 `designWidth`
- 先收口新增入口，再回头清历史债，不允许一边标准化一边继续新增硬编码
- H5、RN、Harmony 都吃同一套语义 token，但允许平台层各自保留必要的实现差异
