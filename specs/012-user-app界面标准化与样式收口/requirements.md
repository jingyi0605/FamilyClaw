# 需求文档 - user-app界面标准化与样式收口

状态：Draft

## 简介

`user-app` 现在已经不是“零散页面风格不一致”这么简单，而是样式来源和尺寸语义一起分裂了。

- H5 主题 token 已经存在，但主要停留在壳层和部分页面
- `packages/user-ui` 已经存在，但能力太薄，只覆盖了少量颜色、间距和卡片
- 业务组件和页面里还混着大量硬编码字号、圆角、边距和内联样式
- H5 下同时存在 Taro 会换算的设计稿尺寸、手写 `rem`、以及 JSX/TS 行内固定像素

这会直接带来四个坏结果：

1. 新页面每做一次，都要重新决定字体、间距、按钮和卡片长什么样
2. 旧页面改一点样式，容易把别的页面拉出新的差异
3. 多端迁移会越来越痛，因为规则不在共享层，而是散在页面里
4. H5 的尺寸问题会一直被误判成“设计稿不对”或“页面写错了”，但其实是单位体系本身就没收口

这份 Spec 的目标很直接：先把 `user-app` 的 UI 标准立起来，再让组件和页面按统一标准消费变量和基础组件，并把 H5 单位策略写死。

## 术语表

- **System**：`apps/user-app`、`packages/user-ui` 及其运行时主题层
- **设计 token**：颜色、字号、圆角、阴影、间距这类最底层的原始变量
- **语义 token**：例如“正文主文字色”“主卡片背景”“危险按钮边框”这类带用途含义的变量
- **基础组件**：`Text`、`Button`、`Card`、`Input`、`Tag`、`Section` 这种不带具体业务的通用 UI 组件
- **页面迁移**：把业务页面从硬编码样式改成消费统一 token 和基础组件
- **设计稿尺寸源码单位**：写在样式文件里的 `px`，语义上表示设计稿尺寸，交给 Taro H5 去换算
- **固定浏览器像素**：明确不走 Taro 设计稿换算，只在浏览器里按固定像素显示的尺寸

## 范围说明

### In Scope

- 定义 `user-app` 的字体、字号、颜色、间距、圆角、阴影、边框等基础标准
- 明确唯一的 token 来源和导出方式
- 在 `packages/user-ui` 里补齐语义 token 和高频基础组件变体
- 迁移 `user-app` 的公共壳层、高频表单和高频卡片类组件
- 明确 H5 下 `designWidth`、`pxtransform`、样式文件单位和 JSX 行内样式单位的使用规则
- 制定新增页面和组件的样式约束，避免继续新增散装样式

### Out of Scope

- 大规模品牌升级、重新定视觉风格方向
- 全量重做所有历史页面的布局和交互
- 为了这次标准化新增独立设计平台或外部设计工具流程
- 修改 `user-web` 的现有代码

## 需求

### 需求 1：系统必须先建立一套唯一的 UI 标准来源

**用户故事：** 作为前端开发者，我希望界面标准只有一个来源，这样我在改页面时不用猜到底该跟哪套 token、哪套样式走。

#### 验收标准

1. WHEN 团队查看 `user-app` 的样式标准 THEN System SHALL 明确标出唯一的 token 来源、导出入口和消费方式。
2. WHEN H5 主题层和 `packages/user-ui` 都需要读取样式变量 THEN System SHALL 复用同一套基础定义，而不是手抄两份不同的值。
3. WHEN 发现旧的样式变量命名和新标准冲突 THEN System SHALL 给出兼容映射或迁移策略，而不是直接破坏现有页面。

### 需求 2：系统必须提供可复用的语义 token 和基础组件变体

**用户故事：** 作为前端开发者，我希望常见的卡片、按钮、输入框、说明文字和标签都有统一的公共写法，而不是每个页面自己拼一套样式。

#### 验收标准

1. WHEN 开发者需要实现常见卡片、按钮、输入框、表单字段、提示文本和状态标签 THEN System SHALL 在 `packages/user-ui` 中提供可直接消费的基础组件或样式变体。
2. WHEN 开发者需要字体、字号、颜色、间距、圆角、阴影或边框 THEN System SHALL 提供语义化 token，而不是要求开发者手写具体像素值和颜色值。
3. WHEN 基础组件在 H5 和 RN 端渲染 THEN System SHALL 保持语义和层级一致，即使底层样式实现存在平台差异。

### 需求 3：系统必须让 `user-app` 的公共页面和公共组件优先改为消费统一标准

**用户故事：** 作为产品和开发共同维护的项目负责人，我希望高频公共页面先统一，这样后续新功能至少站在同一套底座上，而不是继续复制旧问题。

#### 验收标准

1. WHEN 团队完成第一轮迁移 THEN System SHALL 至少统一公共壳层、常用表单组件、常用卡片组件和主要导航区块。
2. WHEN 页面已经迁移到统一标准 THEN System SHALL 不再保留同一类元素的重复硬编码样式实现。
3. WHEN 业务页面仍未完全迁移 THEN System SHALL 明确记录剩余页面和迁移顺序，而不是留成“以后再说”。

### 需求 4：系统必须阻止新增散装样式继续进入仓库

**用户故事：** 作为维护者，我希望标准一旦建立，后续新增代码不要再继续写大量硬编码字号、颜色和圆角，不然这次整理很快就白做了。

#### 验收标准

1. WHEN 开发者新增常见 UI 组件 THEN System SHALL 优先通过公共组件或 token 完成，而不是新增同类硬编码样式。
2. WHEN 仓库中出现新增的典型硬编码样式模式 THEN System SHALL 有检查脚本、约定或代码审查准则去阻止它继续扩散。
3. WHEN 某些平台差异确实无法避免 THEN System SHALL 明确允许的例外边界，而不是放任页面层随意兜底。

### 需求 5：系统必须明确 H5 单位体系的使用规则，而不是继续混用 `rem`、可转换 `px` 和固定像素

**用户故事：** 作为前端开发者，我希望知道在 `user-app` 的 H5 页面里什么时候该写 `px`，什么时候绝对不能写 `rem`，什么时候需要显式转换行内样式，这样我不会继续把尺寸体系越写越乱。

#### 验收标准

1. WHEN 开发者在 H5 样式文件中编写页面尺寸 THEN System SHALL 明确规定源码设计单位、是否允许手写 `rem`、以及这些单位最终如何经过 Taro 转换。
2. WHEN 开发者在 JSX/TS 中编写行内样式尺寸 THEN System SHALL 明确禁止直接混写浏览器像素和设计稿像素，并给出共享组件、token 或显式转换的正式做法。
3. WHEN 团队讨论是否需要修改 `designWidth` THEN System SHALL 先核对真实设计稿基准和现有换算依赖，而不是把它当成修复单页尺寸异常的默认手段。

### 需求 6：系统必须保证标准化过程不破坏现有主题和多端能力

**用户故事：** 作为用户和维护者，我希望样式收口不会把现有 H5 主题切换、多端适配和已上线页面直接改坏。

#### 验收标准

1. WHEN 统一 token 来源和组件层 THEN System SHALL 保持现有主题切换能力仍然可用。
2. WHEN 同一组件在 H5、RN、Harmony 使用 THEN System SHALL 保持核心结构和语义一致，不得因为 H5 的实现细节破坏其他端。
3. WHEN 第一轮迁移完成 THEN System SHALL 通过类型检查、样式检查或人工验收，确认主要页面没有明显回归。

### 需求 7：系统必须让后续接手的人一眼看懂规则

**用户故事：** 作为后来接手这个项目的人，我希望打开文档和共享层代码后，能迅速知道“该用什么”“不要用什么”“先看哪里”。

#### 验收标准

1. WHEN 新人查看 Spec 和共享 UI 目录 THEN System SHALL 明确说明 token 分层、基础组件入口、单位策略和迁移顺序。
2. WHEN 团队需要新增新的基础组件变体 THEN System SHALL 能清楚判断应该加在共享层还是业务页面里。
3. WHEN 团队排查样式问题 THEN System SHALL 能快速定位问题是 token、组件变体、单位选型，还是页面私有实现导致。

## 非功能需求

### 非功能需求 1：可维护性

1. WHEN 需要新增主题、调整字号阶梯或修改卡片风格 THEN System SHALL 优先在共享层改动，并让页面自动吃到变化。
2. WHEN 需要迁移旧页面 THEN System SHALL 提供足够稳定的基础件，避免每个页面都重新造轮子。

### 非功能需求 2：兼容性

1. WHEN 标准化改动上线 THEN System SHALL 不破坏现有 H5 主题变量注入机制。
2. WHEN 旧组件尚未迁移完成 THEN System SHALL 允许短期兼容，但必须提供清楚的迁移边界。

### 非功能需求 3：单位一致性

1. WHEN H5 页面使用样式文件、共享 token 和 JSX 行内样式 THEN System SHALL 保持这些入口的单位语义一致，不允许各走各的换算规则。
2. WHEN 团队确实需要固定浏览器像素或规避 Taro 换算 THEN System SHALL 明确写出允许场景和实现方式。

### 非功能需求 4：可验证性

1. WHEN 完成每一阶段迁移 THEN System SHALL 能通过类型检查、组件检查或人工验收验证结果。
2. WHEN 规则需要长期执行 THEN System SHALL 有明确的检查点，防止仓库重新回到散装样式状态。

## 成功定义

- `user-app` 的字体、字号、颜色、间距、圆角和阴影不再分裂成多套来源
- `packages/user-ui` 成为 `user-app` 公共样式和基础组件的正式出口
- 高复用组件和公共壳层不再写重复的硬编码样式
- H5 下 `designWidth`、`pxtransform`、样式文件单位和行内样式单位的规则明确且可执行
- 后续新增页面能优先通过 token 和基础组件实现，而不是继续写一堆内联样式
- 团队能清楚区分哪些地方还没迁移，哪些地方已经进入统一标准
