# 20260315-普通对话声纹识别联调与验收手册

> 迁移说明：自 `005.3.2` 完成后，本文里原先提到的 `voice-runtime` 已全部收回 `api-server` 内嵌 runtime。独立 `apps/voice-runtime` 目录已经删除，联调时请直接看 `api-server` 内的 embedded runtime 实现。

## 这份文档解决什么问题

这份手册只讲三件事：

1. 普通对话怎么确认已经变成“先声纹识别，再进 LLM”
2. 声纹识别失败时怎么确认系统真的走了降级，而不是把主链搞死
3. 样本、档案、识别结果和隐私数据该去哪里看、哪里删

别把它当宣传稿。它是联调和验收清单。

## 当前首版边界

- 本地 ONNX
- 工具链：`sherpa-onnx`
- 模型：`weSpeaker/ResNet34`
- 普通对话识别窗口：`1s ~ 2s`
- 识别范围：仅当前 `household`
- 识别失败策略：回退到现有上下文身份推断或匿名身份，不能中断现有语音主链

## 联调前提

- 终端已经完成 binding，且属于目标家庭
- 目标成员已经完成至少一份有效声纹档案
- 不要求现在启动开发服务器；自动化回归优先用单元测试跑
- 如果要做人工联调，先确认 `voice_runtime_client` 在 commit 后返回里能看到音频产物元数据

## 建档联调怎么验

### 1. 创建建档任务

调用：

`POST /api/v1/voiceprints/enrollments`

重点看返回：

- `status=pending`
- `sample_goal=3`
- `terminal_id / member_id / household_id` 正确

### 2. 终端开始采样

确认 gateway 上行事件：

- `session.start.payload.session_purpose == "voiceprint_enrollment"`
- `session.start.payload.enrollment_id` 存在
- `audio.commit.payload.session_purpose == "voiceprint_enrollment"`

### 3. runtime 落音频产物

确认 commit 返回里有：

- `audio_artifact_id`
- `audio_file_path`
- `sample_rate`
- `channels`
- `sample_width`
- `duration_ms`
- `audio_sha256`

样本文件落点应位于：

`artifacts_root/<household_id>/<terminal_id>/voiceprint_enrollment/...`

### 4. api-server 完成建档

建档完成后确认：

- `voiceprint_enrollments.status == completed`
- `voiceprint_enrollments.sample_count == 3`
- `member_voiceprint_profiles.status == active`
- `member_voiceprint_samples.status == accepted`

## 普通对话怎么验“先声纹，再进 LLM”

### 验收目标

一句话说清楚：

普通 `conversation` commit 后，系统先用本次音频产物做 household 范围声纹识别，再把结果写进统一 `VoiceIdentityResolution`，最后快路径和慢路径都只读这一份身份结果。

### 成功路径

看这几件事：

1. gateway 上行还是普通会话
   - `session.start.payload.session_purpose == "conversation"`
   - `audio.commit.payload.session_purpose == "conversation"`
2. runtime commit 返回里有 `.wav` 产物元数据
3. `voice_session_registry` 对应会话里有
   - `audio_file_path`
   - `requester_member_id`
   - `identity_summary.voiceprint_hint.status == "matched"`
4. 如果走慢路径，`conversation` 会话创建时的 `requester_member_id` 应该等于声纹识别出的成员

### 低置信度 / 冲突 / provider 不可用路径

看这几件事：

1. 主链不能中断，仍然能继续走快路径或慢路径
2. `identity_summary.voiceprint_hint.status` 会记录真实状态
   - `low_confidence`
   - `conflict`
   - `unavailable`
   - `no_profile`
3. `requester_member_id` 应该来自上下文兜底结果，或者为空匿名
4. 不允许因为声纹失败直接返回 `agent.error`

## 快路径怎么验没被搞坏

重点不是“有没有声纹”，而是“快路径是不是仍然只吃统一身份结果”。

验收方式：

1. 构造一个会命中设备控制的普通对话
2. 确认 `voice_fast_action_service.resolve(...)` 收到的 `identity.primary_member_id` 与最终会话记录一致
3. 确认快路径执行成功后
   - `lane == "fast_action"`
   - `route_type == "device_action"` 或 `scene`
   - `requester_member_id` 仍是同一份身份结果

## 慢路径怎么验没串人

重点看两个字段：

- `VoiceIdentityResolution.primary_member_id`
- `ConversationSession.requester_member_id`

验收方式：

1. 用已有声纹档案成员发起普通对话
2. 让这轮对话明确走 `conversation` 慢路径
3. 确认创建会话时传入的 `requester_member_id` 等于声纹识别出的成员
4. 确认后续会话详情里 `requester_member_id` 没被丢掉

## 出问题先查哪里

### 1. 看本次会话有没有音频产物

如果 `audio_file_path` 都没有，别往声纹算法上甩锅，先查 embedded runtime commit。

### 2. 看声纹提示字段

优先看：

- `identity_summary.voiceprint_hint.status`
- `identity_summary.voiceprint_hint.reason`
- `identity_summary.voiceprint_hint.member_id`
- `identity_summary.voiceprint_hint.score`

### 3. 看是不是 household 隔离问题

声纹搜索只允许读当前家庭的 `active` profile。  
如果你怀疑串家庭，先查 profile 的 `household_id`。

### 4. 看是不是档案没建出来

如果状态是 `no_profile`，别再看识别链路，直接回去查：

- `voiceprint_enrollments`
- `member_voiceprint_profiles`
- `member_voiceprint_samples`

### 5. 看是不是 provider 本身挂了

如果状态是 `unavailable`，重点查：

- 模型文件路径
- `sherpa-onnx / soundfile / numpy` 依赖
- provider 开关和阈值配置

## 隐私与清理

### 哪些是隐私数据

- 原始样本音频：`.wav`
- 样本元数据：`member_voiceprint_samples`
- 声纹档案：`member_voiceprint_profiles`

### 怎么删

当前正式入口：

`DELETE /api/v1/voiceprints/members/{member_id}`

这会做两件事：

1. 把该成员现有声纹档案标记为 `deleted`
2. 取消该成员仍在进行中的建档任务

### 这次明确不做的事

- 不在 gateway 删除样本
- 不在 embedded runtime 做身份决策
- 不做跨家庭共享档案

## 验收结论怎么写才算合格

至少要写清楚这四项：

1. 建档成功路径是否成立
2. 普通对话是否已经变成“先声纹，再进 LLM”
3. provider 不可用或低置信度时是否稳定降级
4. 快路径和慢路径是否读取同一份身份结果
