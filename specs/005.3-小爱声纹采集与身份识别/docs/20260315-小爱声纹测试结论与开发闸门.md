## 测试 1：open-xiaoai 能不能把音频送到网关

### 看了什么

- `open-xiaoai` Rust client README
- `client-rust/src/bin/client.rs`
- `client-rust/src/services/audio/record.rs`
- `client-rust/src/services/connect/data.rs`
- FamilyClaw 的 `open_xiaoai_gateway/translator.py`

### 看到的事实

- Rust client 用 `send_stream("record", bytes, None)` 发音频流
- 录音来自 `arecord -t raw`
- `Stream` 结构里有 `tag` 和 `bytes`
- FamilyClaw gateway 会把 `tag="record"` 翻译成 `audio.append`
- `audio.append` 里已经带了：
  - `chunk_base64`
  - `chunk_bytes`
  - `codec=pcm_s16le`
  - `sample_rate=16000`

### 实测结果

- 已完成一次无损恢复测试
- `recovered.wav` 可正常读出
- 格式信息：
  - 单声道
  - `16kHz`
  - `16-bit`
  - `500ms`

### 结论

这条链路拿到的不是“音频文件”，而是“原始音频流分片”。但这不构成问题，因为它已经足够无损恢复成声纹可用的源文件。

## 测试 2：多轮录音能不能做成可识别声纹

### 测试方法

- 采用本地 ONNX 声纹方案，不碰项目正式代码
- 用 `SpeakerEmbeddingExtractor` 提取 embedding
- 用 `SpeakerEmbeddingManager` 做建档、搜索和验证
- 用 `3` 轮样本做一次建档
- 每轮样本长度按 `2s` 构造
- 查询样本也按 `2s` 构造

### 为什么这样测

用户现在要先确认“值不值得做”和“怎么做”，不是要先把生产级识别准确率吹出来。

所以这轮测试只回答：

- 多轮样本能不能喂进统一方案
- 建档和识别闭环能不能跑通
- 时延大概落在哪个数量级

### 实测结果

- 建档轮数：`3`
- 单次查询长度：`2000ms`
- 建档 embedding 平均耗时：`67.961ms`
- 查询 embedding 平均耗时：`67.780ms`
- 查询 embedding P95：`71.284ms`
- `search()` 平均耗时：`0.0278ms`
- `verify()` 平均耗时：`0.0125ms`
- 同一说话人变体测试：`5/5` 成功命中 `speaker_a`
- 同源反转负样本：
  - `search()` 返回空
  - `verify()` 返回 `False`
  - 负样本得分：`0.4212`
- 同说话人查询平均得分：`0.9868`

### 结论

- 多轮建档是通的
- 这类方案天然适合“多轮录音 -> 多个 embedding -> 平均或聚合后入档”
- 第一版不该押注单条录音

## 测试 3：100ms 能不能做到

### 测试方法

- 同一模型
- 本地 CPU
- 直接测“接受 waveform + 计算 embedding”的总耗时
- 每个时长跑 `20` 次

### 结果

| 查询音频长度 | 平均耗时 | P95 | 结论 |
| --- | --- | --- | --- |
| `1000ms` | `35.912ms` | `39.857ms` | 明显可进 `100ms` |
| `2000ms` | `68.207ms` | `73.365ms` | 可进 `100ms` |
| `3000ms` | `102.865ms` | `109.817ms` | 已经贴边并开始超 |
| `4000ms` | `134.923ms` | `142.033ms` | 明显超出 |

### 结论

- 如果第一版查询窗口控制在 `1s ~ 2s`，本地 CPU 上有机会把声纹识别压进 `100ms`
- 如果查询窗口要求 `3s` 以上，当前基线不该直接承诺 `100ms`
- 如果后面业务坚持更长窗口还要 `100ms`，就得补：
  - 流式预计算
  - 更轻模型
  - 或更强硬件

## 这轮测试后给出的首版方案建议

### 录入方案

- 通过小爱音响走现有 `record` 流
- 网关继续转成 `audio.append`
- runtime backend 必须先补出正式 `.wav` 落盘；当前实现已经收口到 `api-server` 内嵌 runtime
- 建档默认采用多轮采样
- 第一版建议：
  - `3` 轮样本
  - 每轮有效语音目标 `2s ~ 5s`

### 解析方案

- 第一版先用本地 ONNX 方案
- 具体基线：
  - `sherpa-onnx`
  - `wespeaker_en_voxceleb_resnet34.onnx`
- 建档时保存：
  - 原始 `.wav`
  - 样本元数据
  - 聚合后的声纹向量或 provider 引用
- 识别时先做 embedding，再在当前家庭成员范围内 search / verify

### 为什么先选这个

- 本地可跑
- 不依赖云端往返
- 测得出真实 CPU 时延
- 方案简单，适合先把主链立住

## 补充测试：`CAM++` 与 `ResNet34` 公开样本对比

### 测试方法

- 公开样本：`LibriSpeech test-clean`
- 选取 `20` 个 speaker
- 每个 speaker：
  - `5` 条语音做 enrollment
  - `10` 条语音做 query
- 总 query 数：`200`
- 识别逻辑：
  - enrollment embedding 求 speaker centroid
  - query embedding 与所有 centroid 做余弦相似度
  - top1 作为预测结果

### 结果摘要

| 模型 | 条件 | 准确率 | 平均总识别时间 |
| --- | --- | --- | --- |
| `CAM++` | `crop_2s` | `30.0%` | `28.2ms` |
| `CAM++` | `crop_3s` | `10.5%` | `37.2ms` |
| `ResNet34` | `crop_2s` | `98.0%` | `67.6ms` |
| `ResNet34` | `crop_3s` | `98.5%` | `99.4ms` |

### 结论

- 本次公开样本对比里，`CAM++` 虽然更快，但准确率明显异常，不适合作为当前项目第一版方案
- `ResNet34` 在相同规则下准确率和稳定性都明显更好
- 最终决定：`005.3` 第一版正式选用 `weSpeaker/ResNet34`

## 开发闸门

### 已通过

- `Gate A`：小爱音频流可恢复为标准源文件
- `Gate B`：多轮建档和识别闭环可跑通
- `Gate C`：在 `1s ~ 2s` 查询窗口下，`100ms` 目标可达
- `Gate D`：已完成 `CAM++` 与 `ResNet34` 对比，并锁定首版模型为 `ResNet34`

### 还没通过的

- 真实小爱实机环境下，不同家庭成员的准确率测试
- 真实噪声环境下的阈值校准
- 长查询窗口下仍严格压进 `100ms`

## 这份结论文档允许后面怎么做

允许：

- 继续推进 `005.3` 的正式实现设计
- 先按本地 ONNX 路线做第一版
- 先把 `.wav` 落盘、建档任务、识别前置接上

不允许：

- 跳过 `.wav` 落盘，直接在主链里拍脑袋接声纹
- 在没有真实时延数据的情况下承诺“任何查询都 100ms”
- 还没做真实家庭成员验收，就宣称准确率已经足够上线
