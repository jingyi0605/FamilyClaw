# Spec 005.3 - 小爱声纹采集与身份识别

状态：Draft

> 迁移说明：自 `005.3.2` 完成后，音频缓存、commit 落盘和 artifact 生成已经完全收回 `api-server` 内嵌 runtime；独立 `voice-runtime` 已从仓库移除，不再是任何正式运行路径。

## 这份 Spec 现在解决什么问题

这份 Spec 不再假设“声纹能力肯定能做”，而是先把该验证的东西验证掉，再决定怎么入库实现。

当前已经明确的前提是：

- 小爱音响已经接入主链
- 项目里还没有正式的语音获取落盘和声纹识别能力
- 这轮工作的重点不是先写代码，而是先确认三件事到底站不站得住

这三件事是：

1. 小爱这条链路，能不能稳定拿到适合做声纹的原始音频
2. 如果要做声纹，第一版到底该选什么方案录入和解析
3. 识别时延有没有机会压到 `100ms` 以内，边界条件是什么

一句话说清楚：

> `005.3` 先做测试闸门，再做实现设计。先证明这事值得做、做得通，再把代码正式放进仓库。

## 【核心判断】

✅ 值得做，但必须先测完再开发。

理由很简单：家庭场景下一个音响会被多人共用，没有可靠身份，后面的权限、记忆、个性化回答都在瞎猜。问题是真的，但如果声纹方案本身不稳定、太慢，硬接进主链就是制造新故障。

## 【2026-03-15 已完成的前置测试结论】

### 1. 小爱链路能拿到什么

- `open-xiaoai` Rust client 发送的是 `record` 原始音频流分片，不是“只有文本”
- FamilyClaw gateway 当前已经能把它翻译成 `audio.append`
- 这些分片可以无损还原为 `.pcm` / `.wav`
- 已有恢复产物：`recovered.wav`，格式是 `16kHz / 单声道 / 16-bit / 500ms`

### 2. 多轮声纹建档能不能做

- 能做，但应该定义成“多轮采样 + 平均 embedding”，不要赌单条样本
- 首版方案已经定为本地 ONNX `weSpeaker/ResNet34`
- 这轮临时测试采用本地 ONNX 方案：
  - `sherpa-onnx 1.12.29`
  - `wespeaker_en_voxceleb_resnet34.onnx`
- 用 `3` 轮样本做建档、再用查询样本做匹配，已经跑通“建档 -> 搜索 -> 验证”最小闭环

### 4. `CAM++` 与 `ResNet34` 对比后的最终选择

- 已使用公开样本 `LibriSpeech test-clean` 做对比测试
- `ResNet34` 在统一闭集识别小基准下表现稳定：
  - `2s` 查询准确率约 `98.0%`
  - 平均识别时间约 `67.6ms`
- 本次拿到的公开 `CAM++ ONNX` 模型虽然更快，但在相同测试下准确率明显异常，不适合作为当前项目第一版基线
- 结论：`005.3` 第一版正式选用 `weSpeaker/ResNet34`

### 3. `100ms` 能不能做到

- 对 `1s` 查询音频，平均约 `35.9ms`
- 对 `2s` 查询音频，平均约 `68.2ms`
- 对 `3s` 查询音频，平均约 `102.9ms`
- 对 `4s` 查询音频，平均约 `134.9ms`

结论不是“绝对能”也不是“绝对不能”，而是：

- 如果第一版把查询窗口控制在 `1s ~ 2s`，本地 CPU 上有机会把识别压进 `100ms`
- 如果查询窗口拉到 `3s` 以上，当前基线基本会超过 `100ms`
- 所以第一版不能瞎承诺“任何情况下都 100ms”，要么限制窗口，要么后续补流式预计算或更轻模型

## 【关键洞察】

- 数据结构：这次核心对象不是“聊天请求”，而是“录音样本、声纹档案、识别结果”
- 复杂度：gateway 只管采集和转发，别把声纹算法塞进网关
- 风险点：声纹失败不能打断现有主链，否则就是新功能破坏旧用户空间

## 【Linus式方案】

1. 先把测试闸门写清楚，没过就不要入库实现
2. 第一版直接按已经测过的 `weSpeaker/ResNet34` 落地，不再在当前 Spec 内摇摆
3. 录音采样、声纹建档、对话识别三条链路分开
4. 识别成功时增强身份，失败时回退到现有身份推断

## 和已有 Spec 的关系

- `005`：解决“小爱音响怎样接成正式语音终端”
- `005.2`：解决“什么时候接管、什么时候让小爱原生继续说”
- `005.3`：解决“接管之后怎么知道到底是谁在说话”
- `005.3.2`：解决“运行时到底放哪儿，以及本地默认怎样启动”

它们是同一条语音链路上的三个连续层次：

1. 先接进来
2. 再决定何时接管
3. 最后决定说话人是谁

## 本次范围

这次要先做这些事：

- 固化小爱音频链路可恢复为声纹源文件的测试结论
- 固化第一版声纹录入与解析方案的选择依据
- 固化多轮建档和时延测试结果
- 把“测试先过闸门，再开发”写进需求、设计和任务
- 在此基础上保留后续实现方案

这次明确不做：

- 不先把声纹能力直接写进正式业务代码
- 不先接多家 provider 做抽象层秀操作
- 不做说话人分离、降噪算法研发、跨家庭共享声纹库
- 不把声纹识别结果当成唯一真相
