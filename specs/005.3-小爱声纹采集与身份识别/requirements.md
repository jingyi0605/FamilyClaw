# 需求文档 - 小爱声纹采集与身份识别

状态：Draft

## 简介

当前小爱音响已经能把语音送进 FamilyClaw 主链，但“谁在说话”这件事还没有正式能力。现有身份判断更多是上下文猜测：

- 终端在哪个房间
- 当前谁被标记为活跃成员
- 文本里有没有显式提到某个成员

这套东西能先跑起来，但它不是声纹识别。真正的问题在于：

- 同一房间里可能有多个家庭成员
- 音响是共享设备，不是账号私有设备
- 后续 LLM 对话、权限控制、个人记忆写入都依赖身份

这次不能再走“先写代码再看行不行”的路子。要先把三件事测清楚：

- 小爱链路能不能给出靠谱的声纹源文件
- 第一版到底选什么声纹录入和解析方案
- 识别时延到底能不能压到 `100ms`

只有这些前置问题过了闸门，后面的实现才允许正式入库。

## 术语表

- **System**：FamilyClaw 整体语音系统，包含 `open-xiaoai-gateway`、`voice-runtime` 和 `api-server`
- **声纹建档**：把某个成员的录音样本送入声纹提供方或本地适配器，生成可用于后续识别的声纹档案
- **录音样本**：一次建档或补样时保存下来的原始音频源文件，至少可恢复为 `.pcm` 或 `.wav`
- **声纹档案**：某个家庭成员当前生效的声纹信息，可能是 provider 返回的 profile id、模板或 embedding
- **建档任务**：一次面向指定成员的录音采集和建档流程，包含目标成员、目标终端、采集状态和样本结果
- **对话识别**：用户正常通过小爱音响说话时，在进入 LLM 前做一次声纹匹配
- **身份回填**：把声纹识别出的成员写回语音会话上下文，后续路由、记忆和对话都沿用这个成员身份
- **多轮采样**：为同一成员连续采集多条独立录音样本，再聚合成一个可用的声纹档案

## 开发前必须先过的测试闸门

### 闸门 A：确认小爱链路能提供声纹源文件

1. WHEN 项目准备接入声纹能力 THEN System SHALL 先证明 `open-xiaoai -> gateway` 拿到的是可恢复的原始音频分片，而不是只有文本。
2. WHEN 做完一次音频恢复测试 THEN System SHALL 产出至少一个可读的 `.wav` 或 `.pcm` 样本，并记录采样率、声道数、位深和时长。

### 闸门 B：确认多轮建档方案跑得通

1. WHEN 项目准备选第一版声纹方案 THEN System SHALL 先跑通“多轮样本 -> embedding 聚合 -> 搜索/验证”的最小闭环。
2. WHEN 第一版方案确定 THEN Spec SHALL 明确写清是本地方案还是云端方案，不能含糊写成“后续再定”。

### 闸门 C：确认 `100ms` 目标的真实边界

1. WHEN 项目准备承诺识别时延 THEN System SHALL 先给出基线机器上的实测数据，而不是靠估计。
2. WHEN 基线测试表明某个查询窗口超过 `100ms` THEN Spec SHALL 明确写出限制条件或后续优化项，不能继续对外承诺“默认 100ms”。

## 范围说明

### In Scope

- 先完成开发前测试闸门和结论文档
- 为指定家庭成员创建声纹建档任务
- 从小爱音响录音链路获取可保存的音频样本
- 把录音样本转成可快速解析的源文件并保存
- 先落一个可本地跑通的首版声纹方案
- 调用声纹 provider 或本地适配器生成/更新成员声纹档案
- 在小爱音响对话进入 LLM 前先做声纹识别
- 识别成功时按对应成员身份继续处理对话
- 识别失败、冲突或 provider 不可用时回退到现有身份推断链路

### Out of Scope

- 还没过测试闸门前就直接把声纹能力写入正式业务代码
- 多说话人混音分离
- 完整前端管理页面
- 跨家庭声纹共享和全局黑名单
- 端侧本地离线声纹识别算法研发
- 用声纹直接替代全部权限确认和高风险二次校验

## 需求

### 需求 1：系统必须支持为指定家庭成员发起多轮声纹建档任务

**用户故事：** 作为家庭管理员，我希望先指定一个家庭成员和一个小爱终端，再连续采集多轮录音样本，这样系统保存下来的声纹档案不会混成一团，也不会只靠一条录音瞎赌。

#### 验收标准

1. WHEN 管理员为某个家庭成员发起建档任务 THEN System SHALL 保存目标家庭、目标成员、目标终端、任务状态和建档批次信息。
2. WHEN 一次建档任务被创建 THEN System SHALL 支持配置目标样本数，第一版默认按多轮采样处理，而不是只收一条录音。
3. WHEN 同一成员已有历史声纹档案 THEN System SHALL 支持追加样本更新，而不是强制只能全量重建。
4. WHEN 管理员指定的成员和终端不属于同一个家庭 THEN System SHALL 拒绝创建建档任务。
5. WHEN 同一终端已有进行中的建档任务 THEN System SHALL 阻止再创建新的冲突任务，避免录音上下文混乱。

### 需求 2：gateway 与后端必须能拿到可保存为声纹源文件的录音样本

**用户故事：** 作为后端维护者，我希望从小爱音响录音链路里拿到稳定、可还原的音频数据，并最终保存为标准源文件，这样后续声纹识别才有靠谱输入。

#### 验收标准

1. WHEN 小爱音响处于建档采样流程 THEN System SHALL 从现有录音链路接收原始音频分片，而不是只拿到文本。
2. WHEN 一次采样提交完成 THEN System SHALL 产出至少一种可快速解析的源文件格式，至少包含 `.wav` 或 `.pcm`。
3. WHEN 保存样本文件 THEN System SHALL 记录样本的采样率、声道数、位深、时长、终端和所属成员信息。
4. WHEN 录音数据损坏、为空或不满足最小采样要求 THEN System SHALL 拒绝把它当作有效声纹样本入库。

### 需求 3：系统必须先落一个已选定的首版声纹方案

**用户故事：** 作为项目负责人，我希望第一版先有一个能本地验证、能测出真实时延的声纹方案，而不是一开始就堆多家 provider 和一堆抽象层。

#### 验收标准

1. WHEN 第一版声纹方案进入正式实现 THEN System SHALL 使用已经通过前置测试并已在 Spec 定稿的 `weSpeaker/ResNet34` 基线方案。
2. WHEN 第一版方案被写进 Spec THEN Spec SHALL 明确写清录入方式、解析方式、档案保存方式和时延边界。
3. WHEN 基线方案是本地 ONNX `weSpeaker/ResNet34` THEN System SHALL 支持“多轮样本 -> embedding 聚合 -> 搜索/验证”闭环。
4. WHEN 后续要替换为其他 provider 或模型 THEN System SHALL 在保留首版可运行基线的前提下再扩展，而不是推翻已有链路。

### 需求 4：系统必须支持把录音样本转成成员声纹并保存

**用户故事：** 作为系统维护者，我希望录音样本能被送到统一的声纹适配层生成可复用的声纹档案，并且结果能和成员绑定保存下来，这样后续识别才有正式依据。

#### 验收标准

1. WHEN 一次录音样本通过校验 THEN System SHALL 调用统一的声纹适配层生成或更新目标成员的声纹档案。
2. WHEN provider 或本地适配器返回成功结果 THEN System SHALL 保存成员声纹档案，并把本次样本与对应档案版本关联起来。
3. WHEN provider 或本地适配器返回失败或超时 THEN System SHALL 把建档任务标记为失败或待重试，而不是悄悄吞掉。
4. WHEN 同一成员已有生效中的声纹档案 THEN System SHALL 支持版本化更新或追加样本，不允许把旧档案直接丢失而没有追踪记录。

### 需求 5：小爱音响对话在进入 LLM 前必须先做声纹识别

**用户故事：** 作为家庭成员，我希望我通过小爱音响发起对话时，系统在真正进入 LLM 之前先识别出是谁在说话，这样回答、记忆和个性化上下文才不会串人。

#### 验收标准

1. WHEN 小爱音响对话完成音频提交并拿到转写文本 THEN System SHALL 在进入 LLM 慢路径前先执行声纹识别。
2. WHEN 声纹识别结果置信度满足阈值 THEN System SHALL 把识别出的成员写入语音会话身份结果中。
3. WHEN 对话后续进入 `conversation` 慢路径 THEN System SHALL 使用声纹识别出的成员身份作为 `requester_member_id` 或等价身份字段。
4. WHEN 对话命中设备控制等快路径 THEN System SHALL 也能读取同一份身份结果，而不是再走另一套不一致的身份逻辑。

### 需求 6：声纹识别失败时不能破坏现有语音主链

**用户故事：** 作为项目负责人，我希望声纹能力是增强项，不是新的单点故障。识别失败时系统可以降级，但不能把现有语音对话直接搞死。

#### 验收标准

1. WHEN 声纹 provider 不可用、超时或返回异常 THEN System SHALL 回退到现有身份推断逻辑或匿名身份，而不是中断整个对话主链。
2. WHEN 声纹识别结果低置信度或多个成员冲突 THEN System SHALL 把这次身份结果标记为冲突或降级，而不是强行认人。
3. WHEN 声纹识别失败 THEN System SHALL 仍允许后续对话继续进入现有快路径或慢路径，除非当前业务本身要求强身份。
4. WHEN 新能力关闭或没有已建档成员 THEN System SHALL 保持当前 `Spec 005 / 005.2` 已有行为不变。

### 需求 7：系统必须保证声纹数据的家庭隔离、可追踪和可清理

**用户故事：** 作为家庭管理员，我希望成员声纹样本和档案只在本家庭内使用，并且后续能查到它从哪来的、必要时可以删除，这样这项能力才不会变成隐私炸弹。

#### 验收标准

1. WHEN 系统做声纹匹配 THEN System SHALL 只在当前家庭的有效成员声纹档案范围内识别。
2. WHEN 保存录音样本和声纹档案 THEN System SHALL 记录创建时间、来源终端、目标成员、任务批次和处理状态。
3. WHEN 管理员删除某个成员的声纹档案 THEN System SHALL 支持停用或删除对应档案及其样本引用，并保证后续不再参与识别。
4. WHEN 系统达到样本保留策略上限或管理员主动清理 THEN System SHALL 支持按规则清理旧样本，而不影响仍生效的识别结果追踪。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 第一版本地声纹方案进入实现 THEN System SHALL 先以本地 CPU 基线验证 `1s ~ 2s` 查询窗口下的识别延迟，目标是把 embedding 提取和匹配控制在 `100ms` 内。
2. WHEN 查询窗口、模型或硬件条件导致基线超过 `100ms` THEN System SHALL 明确写出限制条件或优化计划，而不是继续承诺默认 `100ms`。
3. WHEN 一次建档样本提交完成 THEN System SHALL 在合理时间内完成文件保存和声纹建档状态回写，而不是让任务长期卡在处理中。

### 非功能需求 2：可靠性

1. WHEN gateway、voice-runtime 或声纹 provider 任一环节短暂失败 THEN System SHALL 提供明确失败状态和重试入口，而不是留下脏状态。
2. WHEN 重复提交同一个建档任务或同一段样本 THEN System SHALL 有幂等或去重策略，避免重复建档。

### 非功能需求 3：可维护性

1. WHEN 后续替换声纹 provider THEN System SHALL 通过统一适配层完成切换，而不是把 provider 私有逻辑散到 gateway 和语音主链里。
2. WHEN 排查识别问题 THEN System SHALL 能查到任务、样本、档案、识别结果和回退原因。

### 非功能需求 4：隐私与安全

1. WHEN 保存声纹样本或档案 THEN System SHALL 明确区分“源文件”“档案”“识别结果”三类数据，并支持独立清理。
2. WHEN 非管理员访问声纹建档和样本管理能力 THEN System SHALL 拒绝访问。

## 成功定义

- 已有一份开发前测试结论文档，能说明音频链路、首版方案和时延边界
- 管理员能为指定成员创建建档任务，并通过小爱音响采到有效录音样本
- 系统能保存可解析的 `.wav` 或 `.pcm` 样本文件，并与成员、终端、任务关联
- 系统能把样本转成可用声纹档案并保存
- 小爱音响对话在进入 LLM 前先跑声纹识别，并把结果写回身份上下文
- 识别失败时仍能平稳回退到现有身份逻辑，不破坏现有语音链路
