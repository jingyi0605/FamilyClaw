# 需求文档 - 插件禁用能力统一与停用收口

状态：Draft

## 简介

当前插件系统最尴尬的问题不是“没有禁用能力”，而是“禁用能力到处都有一点，但不统一”。

具体表现很直接：

- 内置插件后端底层能禁，但前端说不支持
- 前端有时拿挂载状态推导启用状态，没有直接信后端最终结果
- 某些运行链路会拦禁用插件，某些链路没有统一拦截

这份 Spec 面向三个角色：

- **家庭管理员**：希望能真正关掉某个插件，而不是只关掉一个假开关
- **前后端开发者**：希望插件状态只有一个可信来源，不再每层自己猜
- **系统维护者**：希望禁用后插件功能真的停下来，不再出现绕过路径

成功后的核心收益是：插件禁用从“页面上的一个按钮”变成“系统里一致生效的一条规则”。

## 术语表

- **System**：FamilyClaw 当前家庭范围内的插件管理与运行系统
- **基础启用状态**：插件原本的启用状态来源。比如内置插件来自注册中心基础状态，挂载插件来自挂载记录。
- **家庭覆盖状态**：当前家庭对某个插件额外设置的启用或禁用结果。
- **最终启用状态**：前端展示和后端运行都必须共同使用的最终 `enabled` 结果。
- **停用收口**：插件被禁用后，相关执行、展示和可配置入口一起收住，不再继续生效。

## 范围说明

### In Scope

- 家庭级插件禁用能力，覆盖内置、官方、第三方插件
- 统一最终 `enabled` 状态计算规则
- 插件管理页和插件详情页的启停交互调整
- 后端插件列表接口和状态更新接口调整
- 运行链路里的禁用拦截，包括任务、Agent、通道、地区 provider、语言包
- 旧插件系统 Spec 的同步补充说明

### Out of Scope

- 插件卸载
- 插件市场安装流程
- 全局平台级紧急熔断开关设计
- 自动删除已禁用插件的历史配置数据

## 需求

### 需求 1：系统必须给每个家庭返回统一的插件最终启用状态

**用户故事：** 作为前后端开发者，我希望系统只给出一个可信的插件启用结果，这样页面显示和运行行为才不会互相打架。

#### 验收标准

1. WHEN 前端请求当前家庭的插件列表 THEN System SHALL 返回每个插件统一的最终 `enabled` 状态，而不是让前端自己拼接推断。
2. WHEN 插件来源是 `builtin`、`official` 或 `third_party` THEN System SHALL 都使用同一套最终状态语义。
3. WHEN 基础状态和家庭覆盖状态同时存在 THEN System SHALL 按固定规则合并，并返回可追踪的结果。

### 需求 2：家庭管理员必须可以禁用内置插件

**用户故事：** 作为家庭管理员，我希望内置插件也能在家庭插件管理页里被禁用，这样我能真正控制当前家庭不想启用哪些能力。

#### 验收标准

1. WHEN 家庭管理员在插件管理页看到内置插件 THEN System SHALL 提供和其他来源插件一致的禁用操作入口。
2. WHEN 家庭管理员禁用内置插件 THEN System SHALL 保存当前家庭的禁用状态，并在刷新后保持一致。
3. WHEN 禁用操作失败 THEN System SHALL 返回明确错误，不允许前端假装更新成功。

### 需求 3：插件被禁用后，相关运行链路必须停止对应功能

**用户故事：** 作为系统维护者，我希望插件一旦被禁用，相关功能都真的停下来，而不是只在页面上变灰。

#### 验收标准

1. WHEN 已禁用插件收到任务执行请求 THEN System SHALL 拒绝创建或执行该插件任务，并给出明确原因。
2. WHEN Agent、通道投递、地区 provider、语言包等链路使用到该插件 THEN System SHALL 统一尊重禁用状态，不继续执行或暴露该插件能力。
3. WHEN 某条链路无法执行已禁用插件 THEN System SHALL 返回结构化错误或跳过结果，不允许静默绕过。

### 需求 4：插件被禁用后，前端配置和展示必须同步更新

**用户故事：** 作为家庭管理员，我希望禁用插件后，页面上的状态、操作入口和配置展示能一起更新，这样我不会误以为插件还在生效。

#### 验收标准

1. WHEN 插件被禁用 THEN System SHALL 在插件列表和详情页同步展示禁用状态。
2. WHEN 插件被禁用 THEN System SHALL 不再把该插件当成“当前可用能力”展示给依赖它的页面区域。
3. WHEN 插件仍保留历史配置 THEN System SHALL 清楚区分“配置仍存在”和“插件当前已停用”，不把旧配置伪装成仍在运行。

### 需求 5：本次变更必须明确更新哪些旧 Spec

**用户故事：** 作为项目维护者，我希望本次设计能明确哪些旧插件 Spec 需要同步补充，这样后续接手的人不会只看到旧文档然后继续做错。

#### 验收标准

1. WHEN `004.2.2` 建立后 THEN System SHALL 明确列出需要同步更新的旧 Spec 和更新原因。
2. WHEN 旧 Spec 仍涉及插件启停语义 THEN System SHALL 补上指向 `004.2.2` 的说明，而不是继续保留模糊表述。
3. WHEN 后续开发者按 Spec 落地 THEN System SHALL 能从文档里直接看出“插件禁用统一规则应该去哪份 Spec 看”。

## 非功能需求

### 非功能需求 1：一致性

1. WHEN 任一前端页面展示插件状态 THEN System SHALL 使用后端返回的最终 `enabled` 结果，不允许页面本地硬编码推断。
2. WHEN 任一后端运行入口判断插件可用性 THEN System SHALL 复用统一状态解析，不允许各模块自己拼条件。

### 非功能需求 2：兼容性

1. WHEN 当前家庭没有任何插件覆盖记录 THEN System SHALL 保持现有默认行为，不因为这次改造把所有插件状态都打乱。
2. WHEN 已有第三方挂载插件继续运行 THEN System SHALL 保持现有挂载和配置模型可用，不要求用户重新安装插件。

### 非功能需求 3：可维护性

1. WHEN 后续新增新的插件执行入口 THEN System SHALL 能直接复用最终状态判断，而不是再抄一份禁用逻辑。
2. WHEN 排查“为什么这个插件没生效” THEN System SHALL 能区分基础状态、家庭覆盖状态和最终状态。

## 成功定义

- 当前家庭可以禁用内置插件，且刷新后状态一致
- 前端不再把内置插件写死成永远启用
- 已禁用插件不会再在任务、Agent、通道、地区 provider、语言包这些链路里继续生效
- 相关旧 Spec 已补充到位，后续不会再被旧文档带偏
