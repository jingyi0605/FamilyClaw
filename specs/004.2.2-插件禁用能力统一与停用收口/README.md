# Spec 004.2.2 - 插件禁用能力统一与停用收口

状态：Draft

## 这份 Spec 解决什么问题

当前插件系统已经有“启用 / 禁用”这个词，但实际行为不统一：

- 第三方挂载插件有家庭级 `enabled`
- 内置插件后端底层有自己的禁用状态
- 前端管理页却把内置插件写成“不支持禁用”
- 有些运行链路会检查插件是否禁用，有些链路可能直接绕过去继续执行

这会造成一个很糟糕的结果：页面显示和真实运行状态可能不是一回事。

这份 Spec 就是把这件事收好：让“插件是否启用”只剩一个清楚答案，并且禁用后前端、后端、运行时都一起停下来。

## 这次要做到什么

- 支持在当前家庭范围内禁用插件，包括内置插件
- 前端插件管理页不再把内置插件硬编码成永远启用
- 后端对外返回统一的最终 `enabled` 状态
- 插件被禁用后，任务执行、Agent 调用、通道投递、地区 provider、语言包这些链路都要尊重禁用状态
- 说清楚哪些旧 Spec 要一起更新，避免只写一份新文档，旧文档继续误导人

## 这次明确不做什么

- 不做插件卸载
- 不做插件配置历史版本管理
- 不做全局插件市场安装流程改造
- 不做“自动清理所有旧配置”这类高风险操作
- 不把平台级总开关和家庭级禁用混成一个概念

## 为什么现在做

因为当前问题已经不是“以后也许会不一致”，而是代码里已经不一致了：

- 后端底层能禁内置插件
- 前端却不让用户禁
- 部分运行路径看 `enabled`，部分路径不看

这类问题继续拖，只会把插件系统越做越脏。现在不收口，后面每接一个插件业务线都得再补一次判断。

## 这次怎么定

这次先把语义定死：

1. 用户在家庭插件管理页里做的禁用，是“当前家庭禁用”，不是全局禁用。
2. 不管插件来源是 `builtin`、`official` 还是 `third_party`，家庭视角下都要能得到一个统一的最终启用状态。
3. 前端展示、后端接口、运行时执行都只认这个最终状态，不允许各写各的推断逻辑。

## 这次同步更新哪些旧 Spec

这次不是只新建 `004.2.2` 就算完。下面这些旧 Spec 也要同步补说明：

- `../004.2-插件系统与外部能力接入/`
  - 要明确：插件注册中心里的“启用 / 禁用”只是基础能力，这次由 `004.2.2` 继续把“最终启用状态”和“禁用后停用收口”讲完整。
- `../004.4-用户前端插件管理与插件市场/`
  - 要明确：前端插件管理页的启停交互以后要覆盖内置插件，且页面状态必须以后端返回的最终 `enabled` 为准，不能前端自己猜。

`004.3` 这轮先不改主规则，因为这次重点不是开发者怎么写插件，而是系统怎么正确停用插件。

## 主要文档

- `requirements.md`：说清楚插件禁用这次到底覆盖哪些行为，什么算禁成功
- `design.md`：说清楚最终 `enabled` 状态怎么合并、接口怎么改、哪些运行链路要收口
- `tasks.md`：说清楚先改数据模型、再改接口、再改前端、再补测试和旧 Spec 更新
