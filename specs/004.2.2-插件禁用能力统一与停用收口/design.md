# 设计文档 - 插件禁用能力统一与停用收口

状态：Draft

## 1. 概述

### 1.1 目标

- 把插件是否启用收成一个统一结果
- 让内置插件支持家庭级禁用
- 让禁用后的停用效果覆盖前端展示和后端运行链路

### 1.2 覆盖需求

- `requirements.md` 需求 1
- `requirements.md` 需求 2
- `requirements.md` 需求 3
- `requirements.md` 需求 4
- `requirements.md` 需求 5

### 1.3 技术约束

- 后端：继续复用现有插件注册、挂载、执行和任务模型，不推翻整套插件系统
- 前端：继续复用当前插件管理页和详情抽屉，不重做整页信息架构
- 数据存储：家庭级状态必须进正式数据存储，不继续拿本地文件冒充家庭配置
- 数据迁移：涉及表结构变更时必须走 Alembic migration
- 兼容边界：不能把当前已安装插件、已有配置和既有调用路径直接搞坏

## 2. 架构

### 2.1 系统结构

这次改造的核心不是“多一个按钮”，而是多一层统一状态合并。

整体结构分四块：

1. **基础插件状态层**
   - 内置插件继续保留现有注册中心基础状态
   - 挂载插件继续保留挂载记录基础状态
2. **家庭覆盖状态层**
   - 记录某个家庭对某个插件的显式启用或禁用选择
3. **最终状态解析层**
   - 合并基础状态和家庭覆盖状态，产出最终 `enabled`
4. **停用收口层**
   - 所有前端展示和后端运行入口都只认最终状态

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 插件状态覆盖存储 | 保存家庭级启停选择 | `household_id`、`plugin_id`、`enabled` | 覆盖记录 |
| 最终状态解析器 | 合并基础状态和家庭覆盖状态 | 注册表项、挂载项、覆盖项 | 最终 `enabled` |
| 插件管理接口 | 返回列表和更新启停状态 | 家庭上下文、插件 id、开关动作 | 最新插件状态 |
| 前端插件管理页 | 展示和修改启停状态 | 插件列表接口、状态更新接口 | 统一的列表和详情展示 |
| 运行时禁用拦截 | 在执行前检查插件是否可用 | 最终插件状态、执行请求 | 允许执行或结构化拒绝 |

### 2.3 关键流程

#### 2.3.1 当前家庭读取插件列表

1. 后端读取插件注册表基础数据。
2. 后端读取当前家庭的挂载数据和家庭覆盖状态。
3. 后端按统一规则合并，生成最终插件列表。
4. 前端直接使用接口返回的 `enabled` 渲染列表和详情。

#### 2.3.2 当前家庭禁用一个插件

1. 家庭管理员在插件管理页点击禁用。
2. 前端调用统一状态更新接口，而不是区分 builtin 和 mount 走不同页面逻辑。
3. 后端写入家庭覆盖状态，并重新计算最终 `enabled`。
4. 前端收到最新状态后刷新列表、详情和相关提示。

#### 2.3.3 已禁用插件收到执行请求

1. 任务、Agent、通道、地区 provider 或其他入口准备使用插件。
2. 入口先调用统一状态解析，拿到该家庭下插件的最终 `enabled`。
3. 如果插件已禁用，则拒绝执行、返回结构化结果，或者在需要聚合的场景下跳过该插件。
4. 如果插件启用，才进入原有执行流程。

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3、4、5

- `PluginStateOverride`：家庭级插件启停覆盖记录
- `PluginEffectiveStateResolver`：统一计算最终 `enabled`
- `PluginStateController`：统一启停更新接口
- `PluginAvailabilityGuard`：运行前统一拦截禁用插件

### 3.2 数据结构

覆盖需求：1、2、3

#### 3.2.1 `plugin_state_overrides`

这张表只做一件事：记录家庭对插件的显式启停选择。

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `id` | uuid | 是 | 主键 | 系统生成 |
| `household_id` | uuid | 是 | 家庭 id | 外键 |
| `plugin_id` | string | 是 | 插件稳定 id | 必须能在注册表中识别 |
| `enabled` | boolean | 是 | 当前家庭是否启用 | 只表示家庭覆盖结果 |
| `source_type` | string | 是 | 插件来源 | `builtin` / `official` / `third_party` |
| `updated_by` | uuid | 否 | 最近操作人 | 便于审计 |
| `created_at` | datetime | 是 | 创建时间 | 系统生成 |
| `updated_at` | datetime | 是 | 更新时间 | 系统生成 |

约束直接定清楚：

- 同一个 `household_id + plugin_id` 只能有一条生效记录
- 没有覆盖记录时，不代表禁用，代表“沿用基础状态”
- 这张表不保存插件目录、入口点、python 路径这些挂载细节

#### 3.2.2 `EffectivePluginState`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `plugin_id` | string | 是 | 插件 id | 稳定值 |
| `base_enabled` | boolean | 是 | 基础启用状态 | 来自注册表或挂载记录 |
| `household_enabled` | boolean | 否 | 家庭覆盖状态 | 无覆盖时为空 |
| `enabled` | boolean | 是 | 最终启用状态 | 前后端都只认它 |
| `disabled_reason` | string | 否 | 禁用原因摘要 | 用于提示和排查 |

最终状态规则定死为：

`enabled = base_enabled AND coalesce(household_enabled, true)`

这样做有两个现实好处：

- 不会打破当前已有基础启停能力
- 家庭禁用可以覆盖所有来源插件，但不能强行把基础上已禁用的插件点亮

### 3.3 接口契约

覆盖需求：1、2、4、5

#### 3.3.1 当前家庭插件列表接口

- 类型：HTTP
- 路径或标识：沿用当前家庭插件列表接口
- 输入：`household_id`
- 输出：插件列表，包含 `base_enabled`、`household_enabled`、最终 `enabled`
- 校验：只返回当前家庭可见插件
- 错误：列表构建失败时返回明确错误信息

说明：

- 这次不允许前端再靠挂载列表自己拼内置插件状态
- 如果兼容成本太高，`base_enabled` 和 `household_enabled` 可以先只在调试或扩展字段里返回，但最终 `enabled` 必须稳定返回

#### 3.3.2 当前家庭插件状态更新接口

- 类型：HTTP
- 路径或标识：新增统一插件状态更新接口
- 输入：`household_id`、`plugin_id`、目标 `enabled`
- 输出：更新后的最终插件状态
- 校验：只有管理员可以修改；插件必须存在且当前家庭可见
- 错误：插件不存在、权限不足、状态冲突时返回明确错误

说明：

- 这个接口语义上覆盖 `builtin`、`official`、`third_party`
- 不再让前端把 builtin 和 mount 拆成两套启停入口

#### 3.3.3 运行前可用性检查函数

- 类型：Function
- 路径或标识：插件模块内部统一能力
- 输入：`household_id`、`plugin_id`
- 输出：最终插件状态或结构化不可用原因
- 校验：必须复用统一解析器
- 错误：状态无法解析时返回结构化错误，不允许默认放行

## 4. 数据与状态模型

### 4.1 数据关系

- 插件注册表负责提供插件基础元数据
- 挂载表继续负责第三方或官方插件的安装与路径信息
- 家庭覆盖表只负责“这个家庭想不想启用它”
- 最终插件状态由解析器统一合并后再给前端和运行时使用

说成人话：

1. 插件“是什么”还是看注册表和挂载。
2. 家庭“要不要用它”单独看覆盖表。
3. 页面和运行时都不应该自己把这两件事重新拼一遍。

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `base_enabled` | 基础启用 | 插件注册或挂载后默认可用 | 基础状态被关闭 |
| `household_disabled` | 当前家庭显式禁用 | 家庭管理员关闭插件 | 家庭重新启用 |
| `effective_disabled` | 最终不可用 | 基础状态关闭或家庭显式禁用 | 基础和家庭状态都允许 |
| `effective_enabled` | 最终可用 | 基础状态允许且家庭未禁用 | 任一条件不满足 |

## 5. 错误处理

### 5.1 错误类型

- `plugin_state_override_invalid`：状态更新请求非法
- `plugin_not_visible_in_household`：当前家庭看不到该插件
- `plugin_disabled`：插件当前已禁用，不能执行
- `plugin_effective_state_unresolved`：最终状态解析失败

### 5.2 错误响应格式

```json
{
  "detail": "插件 health_basic 已在当前家庭停用，不能继续执行。",
  "error_code": "plugin_disabled",
  "field": "plugin_id",
  "timestamp": "2026-03-15T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：直接拒绝，不做模糊兼容。
2. 业务规则错误：返回“当前家庭已停用”这类人能看懂的提示。
3. 运行时调用错误：统一转成结构化不可用结果，不允许默默绕过。
4. 历史配置保留：保留配置，但前端必须明确标记“插件已停用”。

## 6. 正确性属性

### 6.1 属性 1：前后端看到的是同一个插件状态

*对于任何* 当前家庭插件列表和插件执行请求，系统都应该满足：前端展示的 `enabled` 和后端运行判断的 `enabled` 来源一致。

**验证需求：** 需求 1、需求 4

### 6.2 属性 2：内置插件不能再是假禁用

*对于任何* 当前家庭下的内置插件，系统都应该满足：如果页面允许禁用，那么禁用后运行链路真的停下来。

**验证需求：** 需求 2、需求 3

### 6.3 属性 3：禁用判断不能被某条运行链路绕过

*对于任何* 任务、Agent、通道、地区 provider、语言包等插件使用入口，系统都应该满足：在真正执行前先经过统一可用性判断。

**验证需求：** 需求 3

### 6.4 属性 4：旧 Spec 不再误导后续开发

*对于任何* 继续描述插件启停的旧文档，系统都应该满足：文档里明确指向 `004.2.2`，不再保留和当前方案冲突的说法。

**验证需求：** 需求 5

## 8. 风险与待确认项

### 8.1 风险

- 如果继续让前端本地推导插件状态，这次改造会只改表面，不改实质。
- 如果某些运行链路不接统一解析器，禁用能力还是会漏。
- 如果把家庭级禁用和平台级全局禁用混在一起，后面一定会把产品语义做乱。

### 8.2 待确认项

- 当前家庭覆盖状态是否需要记录操作备注。
- 前端是否要展示 `base_enabled` 和 `household_enabled` 的区别，还是只展示最终状态和简单原因。
- 如果历史上已经有内置插件全局状态文件，迁移时是继续保留为基础状态，还是后续另开 Spec 收掉。
