# Spec 002.1 - 家庭问答提醒与场景编排

状态：Draft

## 目标摘要

`Spec 002` 把“谁在家、谁在哪、当前家庭状态是什么”这层上下文底座立起来了，但如果没有后续服务层，这些上下文就只是漂亮数据，不会真正服务家庭成员。

本 Spec 002.1 的目标，是把上下文真正用起来，先落三类最关键的上层能力：

1. **家庭问答 v1**：让系统能回答成员状态、设备状态、提醒状态和基础家庭事务问题。
2. **提醒与广播 v1**：让系统能按成员、房间、时机和优先级触达提醒，并支持未响应升级。
3. **模板化场景编排 v1**：先把智能回家、儿童睡前、老人关怀这类高价值场景跑通，不做一上来就失控的通用规则引擎。
4. **AI 供应商抽象层 v1**：为问答、提醒文案生成、场景解释与未来多模态能力建立统一的模型供应商配置、能力路由与降级骨架。

## 与其他 Spec 的关系

- **依赖 `Spec 001`**：家庭成员、权限、偏好、房间、设备等主数据。
- **依赖 `Spec 002`**：在家状态、房间占用、上下文总览、基础设备动作执行。
- **协同 `Spec 003`**：问答与提醒会读取结构化记忆摘要，并把关键执行结果写回事件/记忆入口；但本 Spec 不重写完整记忆中心。
- **承接预留 `Spec 004` 与 `Spec 006`**：当前阶段先用 `002.1` 把问答、提醒、场景编排收敛成一套可执行子 Spec，后续若范围继续扩大，再拆回独立大 Spec。

## 当前文件

- `requirements.md`：业务范围、需求、验收标准
- `design.md`：架构、接口、数据模型、错误处理、测试策略
- `tasks.md`：阶段任务拆分、依赖、上下文入口、验收方式
- `docs/`：预留给联调说明、验收记录、风险说明

## 当前阶段策略

本 Spec 采用一个很明确的策略：

- **问答先做只读**，不在问答链路里直接偷偷执行设备动作
- **提醒先做可审计调度**，不做花哨但不可追踪的“智能提醒”
- **场景先做模板化**，不做全量可视化规则积木引擎
- **AI 供应商先做抽象层**，不把任何一家模型 SDK 直接焊死在业务代码里

这不是保守，这是实用主义。先把真正高频、真正能演示、真正能维护的闭环做出来。
