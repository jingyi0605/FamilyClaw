# 家庭服务中心与 AI 网关联调说明 v0.1

日期：2026-03-10
适用范围：`Spec 002.1` 第 2、3 阶段

## 1. 这份文档是干什么的

这份文档是给前后端联调时直接照着走的，不是写给人“感觉已经做完了”的。

你可以用它确认：

- 家庭问答怎么调
- 提醒怎么建、怎么触发、怎么确认
- 场景模板怎么注册、预览、手动触发
- AI 网关当前怎么回退

## 2. 当前已经能做的事

### 2.1 AI 网关

- 可以管理 AI 供应商档案
- 可以按 capability 配路由
- 可以做主供应商失败后切备供应商
- 可以在所有供应商失败后走模板降级
- 可以记录模型调用日志

### 2.2 家庭问答

- 可以问成员状态
- 可以问提醒状态
- 可以问场景状态
- 可以问部分设备状态
- 可以返回建议问题

当前问答是只读的，不会偷偷触发设备动作。

### 2.3 提醒服务

- 可以创建、修改、软删除提醒任务
- 可以看提醒总览
- 可以手动触发提醒
- 可以跑一次调度分发
- 可以对提醒运行做确认
- 可以在未确认情况下做升级尝试

### 2.4 场景服务

- 可以拿到三类内置模板：
  - `smart_homecoming`
  - `child_bedtime`
  - `elder_care`
- 可以注册模板
- 可以预览模板
- 可以手动触发模板
- 可以查看执行记录和步骤记录

### 2.5 管理台

- 已新增 `/service-center`
- 可执行问答
- 可查看提醒摘要和手动触发
- 可查看场景模板、预览和手动触发
- 可查看 AI 路由与最近调用摘要
- `/context-center` 已补轻量摘要卡和入口说明

## 3. 当前还没做的事

- 没有接真实云模型 SDK
- 没有做完整 AI 供应商治理台
- 没有做自然语言创建提醒
- 没有做任意拖拽式场景编排器
- 场景广播现在是受控占位，不是真实多渠道广播系统

## 4. 联调前准备

## 4.1 后端

建议执行：

```bash
cd apps/api-server
python -m compileall app migrations
alembic upgrade head
```

如果当前机器没有完整依赖，至少保证 `compileall` 通过。

## 4.2 前端

建议执行：

```bash
cd apps/admin-web
npm.cmd run build
```

## 4.3 数据准备

如果要演示，先准备：

- 至少 1 个家庭
- 至少 2 个成员
- 至少 1 个房间
- 至少 1 个可控设备
- 至少 1 条提醒任务

如需演示数据，可先运行项目现有种子脚本。

## 5. 建议联调顺序

### 5.1 先看 AI 网关

接口：

- `GET /api/v1/ai/runtime-defaults`
- `GET /api/v1/ai/providers`
- `GET /api/v1/ai/routes`
- `GET /api/v1/ai/call-logs`

验证点：

- 路由是不是按 capability 配
- 最近调用有没有日志
- 主备切换后 `fallback_used` 有没有打标

### 5.2 再看家庭问答

接口：

- `GET /api/v1/family-qa/suggestions?household_id=<id>`
- `POST /api/v1/family-qa/query`

最小请求：

```json
{
  "household_id": "<household_id>",
  "question": "爷爷今天吃药了吗？",
  "channel": "admin_web"
}
```

验证点：

- 能不能返回答案
- 能不能返回事实引用
- AI 失败时会不会安全降级

### 5.3 然后看提醒

接口：

- `POST /api/v1/reminders`
- `GET /api/v1/reminders?household_id=<id>`
- `GET /api/v1/reminders/overview?household_id=<id>`
- `POST /api/v1/reminders/{reminder_id}/trigger`
- `POST /api/v1/reminders/scheduler/dispatch?household_id=<id>`
- `POST /api/v1/reminder-runs/{run_id}/ack`

验证点：

- 创建后能不能出现在总览里
- 手动触发后能不能生成运行
- 确认后状态会不会变化
- 调度重复执行时会不会重复创建同一槽位运行

### 5.4 最后看场景

接口：

- `GET /api/v1/scenes/template-presets?household_id=<id>`
- `PUT /api/v1/scenes/templates/{template_code}`
- `POST /api/v1/scenes/templates/{template_code}/preview`
- `POST /api/v1/scenes/templates/{template_code}/trigger`
- `GET /api/v1/scenes/executions?household_id=<id>`
- `GET /api/v1/scenes/executions/{execution_id}`

验证点：

- 模板能不能注册
- 预览能不能显示步骤和阻断原因
- 手动触发后会不会生成执行记录
- 步骤失败时状态是不是 `partial` 或 `blocked`

## 6. 管理台联调建议

页面：

- `/context-center`
- `/service-center`

重点看：

- `/service-center` 能不能拉到问答建议、提醒总览、场景模板和 AI 摘要
- `/service-center` 的问答工作台能不能返回答案和证据
- `/service-center` 的场景面板能不能注册模板、预览、触发
- `/context-center` 是否只显示摘要，不承载全部服务管理

## 7. 已知限制和降级策略

### 7.1 AI 降级

- 主供应商失败：切备供应商
- 全部失败：走模板回答
- 禁止远端：直接阻断远端候选，只允许本地或模板

### 7.2 提醒降级

- 没有真实外部渠道时，用内部送达尝试记录占位
- 确认链路仍然保留，避免把“没确认”误当成“失败”

### 7.3 场景降级

- 高风险动作没确认：进入 `blocked`
- 某一步失败：可进入 `partial`
- 广播当前是占位执行，不是完整播报系统

## 8. 一句话结论

这一版已经够联调、够演示、够继续往下迭代，但还不是“完整商业化服务平台”。  
别自欺欺人，当前重点是：链路打通、状态可追、失败能解释。
