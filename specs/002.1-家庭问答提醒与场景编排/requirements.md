# 需求文档 - 家庭问答提醒与场景编排

状态：Draft

## 简介

`Spec 002` 已经把家庭上下文底座搭起来了：成员在家状态、房间占用、家庭模式、设备健康、基础设备动作都已经有了。

但上下文不是目的，上下文只是原料。

真正让家庭 AI 中枢开始像样工作的，是把这些原料送进三类上层能力：

- **问答**：系统能回答“谁在家、设备状态如何、今天有什么安排、提醒是否完成”
- **提醒**：系统能在正确时机把正确信息送给正确的人
- **场景编排**：系统能在明确触发条件下稳定执行一组低风险、高价值服务动作

结合 `docs/` 下的整体规划，本 Spec 聚焦的是首期**可执行、可解释、可审计**的服务层闭环，而不是追求一个什么都懂、什么都能自动做的幻想系统。

---

## 范围边界

本 Spec 包含：

- 家庭问答 API v1 与首批高频问题支持
- 问答链路的权限裁剪、证据返回与解释结构
- AI 供应商配置、能力路由、降级与隐私治理抽象层
- 提醒任务模型、调度、送达、确认、升级
- 家庭广播与按成员/房间差异化触达
- 模板化场景编排（智能回家、儿童睡前、老人关怀）
- 场景执行日志、冲突处理、安全守卫与审计
- 管理台服务中心页面与 `context-center` 摘要联动

本 Spec 不包含：

- 任意自然语言直接转任意设备动作的通用 Agent
- 全量长期记忆中心实现
- 完整照片检索问答
- 通用可视化规则引擎或 DAG 编排器
- 完整 TTS/ASR 基础设施实现
- 高风险动作的自动化放行策略

---

## 术语表

- **家庭问答（Family QA）**：围绕家庭状态、家庭事务、设备状态和基础记忆摘要的只读问答能力。
- **证据事实（Evidence Fact）**：用于支撑回答的结构化事实片段，包含来源、时间、可见范围和是否为推断。
- **提醒任务（Reminder Task）**：描述“提醒什么、提醒谁、什么时候提醒、通过什么方式提醒”的长期配置对象。
- **提醒运行（Reminder Run）**：某个提醒任务在某次具体触发时生成的一次执行实例。
- **送达尝试（Delivery Attempt）**：某次提醒运行在某个渠道、某个房间、某个目标成员上的一次具体触达行为。
- **确认事件（Ack Event）**：成员或管理员对提醒的确认、忽略、完成、转交等反馈记录。
- **场景模板（Scene Template）**：预定义的家庭场景方案，包含触发条件、守卫规则、执行动作和解释文案。
- **守卫条件（Guard）**：用于阻止不安全执行的保护条件，例如静默时段、儿童保护、敏感房间限制、高风险动作确认。
- **场景执行（Scene Execution）**：某个场景模板在某次触发下的完整执行记录。
- **服务中心（Service Center）**：管理台中用于查看问答、提醒、场景运行状态与最近执行记录的页面或功能分区。
- **AI 供应商档案（AI Provider Profile）**：描述某个模型供应商、接入方式、支持能力、时延预算、隐私级别和密钥引用的配置对象。
- **能力路由（Capability Route）**：把某种 AI 能力请求，例如问答生成、提醒文案生成、场景解释、Embedding、Rerank，路由到一组主备供应商的策略配置。
- **模型网关（Model Gateway）**：业务模块调用 AI 能力时统一经过的抽象层，负责路由、脱敏、超时、重试、降级和审计。
- **密钥引用（Secret Ref）**：用于指向环境变量、密钥管理器或部署侧密钥名的标识，不在数据库中明文存储供应商密钥。

## 需求

### 需求 1：家庭问答 v1

**用户故事：** 作为家庭成员或管理员，我希望系统能基于当前家庭上下文回答常见家庭问题，以便我不用自己在多个页面里翻找状态。

#### 验收标准

1. WHEN 用户提交一个家庭问答请求 THEN System SHALL 基于成员、房间、设备、上下文、提醒和基础记忆摘要组装回答。
2. WHEN 问题命中首期支持类型 THEN System SHALL 返回结构化响应，至少包含 `answer`、`answer_type`、`confidence` 与 `facts`。
3. WHEN 当前事实不足以支持明确回答 THEN System SHALL 明确返回“无法确认/信息不足”，不得编造结果。
4. WHEN 问题只涉及查询 THEN System SHALL 保持只读，不得在问答链路中直接执行设备动作。

### 需求 2：问答权限与解释

**用户故事：** 作为家庭管理员，我希望问答结果遵守权限边界，并能解释为什么这么回答，以便系统可信且可治理。

#### 验收标准

1. WHEN 问答涉及敏感成员、健康信息、私密提醒或敏感房间 THEN System SHALL 先做权限裁剪，再返回可见内容。
2. WHEN 回答引用了结构化事实 THEN System SHALL 返回可机读的证据事实列表，包括来源、时间和可见范围。
3. WHEN 回答包含推断成分 THEN System SHALL 显式标记该结论为推断而非原始事实。
4. WHEN 用户通过问答提出高风险执行意图 THEN System SHALL 拒绝直接执行，并指向受控的动作或管理路径。

### 需求 3：提醒任务管理

**用户故事：** 作为家庭管理员，我希望配置个人提醒、家庭提醒、课程提醒和服药提醒，以便系统能稳定承担家庭事务。

#### 验收标准

1. WHEN 管理员创建或编辑提醒任务 THEN System SHALL 支持配置目标成员、提醒类型、调度规则、优先级、房间/渠道偏好与升级策略。
2. WHEN 提醒任务引用成员、房间、设备或家庭配置 THEN System SHALL 校验这些对象属于同一家庭。
3. WHEN 提醒任务被禁用、恢复或删除 THEN System SHALL 保持状态一致，并保留必要的历史运行记录。
4. WHEN 管理员查询提醒列表或总览 THEN System SHALL 返回当前启用状态、下次触发时间和最近执行结果。

### 需求 4：提醒调度、送达与确认

**用户故事：** 作为家庭成员，我希望提醒在合适时机送达到我，并在我未响应时按规则升级，以便真正被感知和完成。

#### 验收标准

1. WHEN 提醒到达触发时间或命中上下文条件 THEN System SHALL 生成一条提醒运行记录，并尝试按既定渠道送达。
2. WHEN 提醒配置了按成员、房间或场景差异化触达 THEN System SHALL 根据当前上下文选择合适的房间、成员和渠道。
3. WHEN 提醒在规定时间内未被确认 THEN System SHALL 按升级策略继续触达、改用其他渠道或通知管理员。
4. WHEN 成员确认、忽略、完成或转交提醒 THEN System SHALL 写入确认事件，并停止或调整后续升级流程。

### 需求 5：模板化场景编排

**用户故事：** 作为家庭管理员，我希望系统支持少量高价值、可解释的家庭场景，以便把上下文和设备联动真正用起来。

#### 验收标准

1. WHEN 管理员启用首批模板场景 THEN System SHALL 支持至少“智能回家”“儿童睡前”“老人关怀”三类模板。
2. WHEN 模板场景被配置 THEN System SHALL 允许有限参数覆盖，例如目标房间、提醒文案、静默策略、优先级和冷却时间。
3. WHEN 管理员手动预览或手动触发场景 THEN System SHALL 返回计划执行动作与被命中的守卫条件。
4. WHEN 场景未命中首期模板能力范围 THEN System SHALL 拒绝配置为通用自由编排，避免形成不可维护规则泥团。

### 需求 6：场景执行、安全守卫与冲突处理

**用户故事：** 作为系统，我希望场景执行在安全边界内进行，并能处理冲突与失败，以便不会把家庭环境搞乱。

#### 验收标准

1. WHEN 场景触发且通过条件判断 THEN System SHALL 按步骤执行提醒、广播、上下文更新和设备动作。
2. WHEN 场景触发命中静默时段、儿童保护、敏感房间或高风险动作限制 THEN System SHALL 按守卫规则跳过、降级或要求确认。
3. WHEN 多个场景同时竞争同一设备、房间或播报通道 THEN System SHALL 按优先级、冷却期和手动覆盖规则解决冲突。
4. WHEN 场景部分步骤失败 THEN System SHALL 记录逐步执行结果，并按回滚/继续策略输出最终状态。

### 需求 7：管理台服务中心与上下文联动

**用户故事：** 作为家庭管理员，我希望在管理台里看到问答、提醒和场景的统一入口与运行状态，以便能理解、验证和纠偏系统服务。

#### 验收标准

1. WHEN 管理员进入服务中心 THEN System SHALL 展示问答入口、提醒总览、场景状态、最近执行记录和关键风险提示。
2. WHEN 管理员仍停留在 `/context-center` THEN System SHALL 至少展示问答、提醒、场景的摘要卡片与跳转入口。
3. WHEN 管理员执行手动提醒、手动触发场景或发起示例问答 THEN System SHALL 在页面中返回明确结果与状态反馈。
4. WHEN 服务中心中的配置变更已生效 THEN System SHALL 可在后续总览、执行日志和审计中追踪其影响。

### 需求 8：审计、降级与可恢复性

**用户故事：** 作为家庭管理员，我希望问答、提醒和场景能力都可审计、可降级、可恢复，以便系统出了问题时还能查和控。

#### 验收标准

1. WHEN 问答被执行、提醒被触发、场景被执行 THEN System SHALL 记录可追踪的审计或事件日志。
2. WHEN 外部依赖、记忆摘要、设备状态或播报渠道不可用 THEN System SHALL 进入降级路径，并给出明确的原因说明。
3. WHEN 调度器或服务进程重启 THEN System SHALL 恢复未完成但仍有效的提醒与场景评估，不得无控制重复触发。
4. WHEN 管理员需要事后追查一次服务行为 THEN System SHALL 能关联查询请求、提醒运行、场景执行与设备动作日志。

### 需求 9：AI 供应商配置与能力路由

**用户故事：** 作为平台架构师或家庭管理员，我希望系统能抽象多种 AI 供应商并按能力做主备路由，以便未来可以横向接入不同模型而不重写业务代码。

#### 验收标准

1. WHEN 系统为问答、提醒文案生成、场景解释或未来多模态能力发起 AI 请求 THEN System SHALL 先通过统一模型网关按能力路由到配置好的主供应商，而不是由业务模块直接依赖某家 SDK。
2. WHEN 某项能力配置了多家供应商 THEN System SHALL 支持主备顺序、超时预算、重试策略和降级模式配置。
3. WHEN 管理员或部署环境修改 AI 供应商档案或能力路由 THEN System SHALL 不需要改业务代码即可切换默认模型或新增同能力供应商。
4. WHEN 某项能力被显式配置为“本地优先”或“禁止远端” THEN System SHALL 强制遵守该路由策略，不得把受限请求发送到不允许的外部供应商。

### 需求 10：AI 调用治理与隐私边界

**用户故事：** 作为家庭管理员，我希望 AI 问答和未来 AI 能力调用遵守隐私、审计和成本边界，以便系统不会把家庭数据随便送出或失控烧钱。

#### 验收标准

1. WHEN 系统向 AI 供应商发送请求 THEN System SHALL 记录供应商、模型、能力类型、耗时、token 用量、降级路径和请求追踪 ID。
2. WHEN 请求中包含成员身份、健康信息、私密提醒、敏感房间或其他受限字段 THEN System SHALL 在模型网关层先做脱敏、裁剪或阻断，而不是把原始数据直接交给供应商。
3. WHEN 供应商密钥被配置到系统中 THEN System SHALL 通过环境变量或密钥引用管理，不得把明文密钥持久化到业务数据库。
4. WHEN 某供应商超时、失败、限流或成本超出策略 THEN System SHALL 按能力路由策略执行回退、模板回答或直接拒绝，并在结果中明确标识为降级路径。
