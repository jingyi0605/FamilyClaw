# Spec 010 - 通讯通道插件与多平台机器人接入

状态：Draft

## 这份 Spec 解决什么问题？
当前 `FamilyClaw` 的 AI 对话主链已经能在网页里工作，但它还被困在网页里。

这会直接带来三个问题：

- 家庭成员在 `Telegram`、`Discord`、`飞书`、`钉钉`、`企业微信` 里找不到同一个 AI 管家
- 管理员没法把“家庭成员”和“外部平台账号”稳定绑定，系统不知道该替谁回复
- 就算硬接一个机器人，也很容易绕开现有会话、记忆、提案、动作策略，最后变成两套脑子

这份 Spec 的目标很直接：在现有插件系统和对话系统上，再补一层正式的“通讯通道插件”能力，让多平台机器人接入成为标准能力，而不是每个平台单独写一套旁路逻辑。

## 这次要做到什么？

- 定义正式的通讯通道插件模型，不再把聊天平台硬塞进现有 `connector`
- 支持机器人账号配置、成员平台账号绑定、外部会话与内部会话映射
- 让外部平台对话复用当前 AI 对话主链，保证人格、记忆、提案、动作策略一致
- 在管理员视图里提供平台配置、绑定管理、状态观测和失败排查入口
- 按两批平台落地：
  - 第一批：`Telegram`、`Discord`、`飞书`
  - 第二批：`钉钉`、`企业微信`
- `企业微信` 优先支持自建应用模式，再补机器人模式

## 这次明确不做什么？

- 不做全新的聊天内核，不重写现有 `conversation` 主链
- 不做“所有平台功能完全一致”的第一版，先保证文本会话、绑定、原路回复、状态可观测
- 不做一上来就覆盖所有富媒体、卡片、投票、线程细节
- 不做任意第三方仓库远程一键安装
- 不做群机器人无限开放接入，第一版必须受账号、绑定、权限和回调校验约束

## 为什么现在做？

因为这不是锦上添花，而是产品能不能真正进入家庭日常沟通场景的分水岭。

如果现在不把“通道插件协议”立起来，后面每接一个平台都会重复出现同一类烂问题：

- webhook、轮询、回复逻辑散落在不同模块里
- 成员绑定规则各写各的
- 外部平台消息绕开现有 AI 会话主链
- 平台能力差异直接污染核心业务代码

这种代码最后一定会烂，而且是越接平台越烂。

## 这份 Spec 的核心判断

- 这是真问题，不是臆想需求
- 更简单的方法不是“先接一个平台试试”，而是先把通道协议立住
- 最大风险不是平台 API，而是破坏现有会话主链的一致性

## 主要文档

- `requirements.md`：说明这次到底要交付哪些能力，哪些结果算做完
- `design.md`：说明通道插件协议、数据模型、回调入口、会话映射，以及 `user-web` 设置页和成员页入口怎么设计
- `tasks.md`：说明按什么顺序落地，先做底层协议，再接平台，再补 `user-web` 接入入口、成员绑定和观测
- `docs/README.md`：给后续补平台接入说明、回调样例、联调清单留位置

## 当前阶段边界

到 `3.4` 为止，五个平台的落地边界已经明确：

- `Telegram`
  - 已完成：webhook 文本入站、成员绑定、复用 `conversation` 主链、文本原路回发、基础 probe
  - 明确没做：卡片、富媒体、复杂按钮、多消息排版器
- `Discord`
  - 已完成：interaction webhook 文本命令闭环、defer + followup 回发、签名校验、基础 probe
  - 明确没做：普通频道消息通过 Gateway/WebSocket 直接入站
- `飞书`
  - 已完成：明文 challenge、`encrypt` 加密回调解包、文本入站、文本回发、基础 probe
  - 明确没做：复杂卡片、文档类能力、群组高级事件
- `钉钉`
  - 已完成：统一 HTTP gateway 可消费的文本事件、`sessionWebhook` 原路回发、配置级 probe
  - 明确没做：常驻 `stream/websocket` 运行时
- `企业微信`
  - `wecom-app`
    - 已完成：回调握手、加密 XML 解包、文本入站、文本回发、凭证 probe
    - 明确没做：复杂群聊、菜单、审批事件
  - `wecom-bot`
    - 已完成：出站推送兼容边界、配置级 probe
    - 明确没做：双向用户消息入站
