# 五个平台落地边界与延期项

这份文档只回答一件事：到 `3.4` 为止，五个平台到底已经做到了哪，哪些是故意没做，后面该怎么补。

## 现在已经落地的边界

### `Telegram`

- 已落地：
  - webhook 文本消息标准化
  - 成员绑定解析
  - 复用 `conversation` 主链
  - 按 `chat_id + thread` 做文本原路回发
  - 基础账号 probe
- 暂不做：
  - 富媒体、投票、复杂按钮、多消息排版

### `Discord`

- 已落地：
  - interaction webhook 文本命令闭环
  - 签名校验
  - 先 `defer`，再用 followup 回发文本
  - 基础账号 probe
- 暂不做：
  - 普通频道消息通过 Gateway/WebSocket 直接入站
- 为什么没做：
  - 这不是插件里再补几行解析代码，而是要补一层长期运行的 Discord Gateway/WebSocket 入站运行时

### `飞书`

- 已落地：
  - 明文 challenge
  - `encrypt` 加密回调解包
  - 文本消息标准化
  - 文本原路回发
  - 基础账号 probe
- 暂不做：
  - 复杂卡片、文档能力、审批流、群组高级事件

### `钉钉`

- 已落地：
  - 统一 HTTP gateway 可消费的文本事件
  - 基于 `sessionWebhook` 的文本原路回发
  - 配置级 probe
- 暂不做：
  - 常驻 `stream/websocket` 运行时
- 为什么没做：
  - 当前先把文本闭环跑通，避免为了一个平台把整套通道运行时提前炸开

### `企业微信`

#### `wecom-app`

- 已落地：
  - callback 握手
  - 加密 XML 解包
  - 文本消息入站
  - 文本回发
  - 凭证 probe
- 暂不做：
  - 复杂群聊、菜单、审批事件

#### `wecom-bot`

- 已落地：
  - 出站推送兼容边界
  - 配置级 probe
- 暂不做：
  - 双向用户消息入站
- 为什么没做：
  - 这个模式当前更适合作为出站通知通道，不适合假装成完整双向聊天平台

## 统一延期项

- `Discord` 普通频道消息直连主链
- `钉钉` 常驻流式接入
- `wecom-bot` 双向接入
- 五个平台的富媒体、卡片、审批和复杂群事件

## 下一步怎么补

- 如果继续做管理端：
  - 直接进入 `4.1`，把平台账号、探测状态、最近失败接到管理页面
- 如果继续做平台深挖：
  - 优先级最高的是 `Discord 普通频道消息直连主链`
  - 但这应该单独当一个运行时任务来做，不要混在普通插件适配任务里
