# 20260318-地区provider坐标接入与联调说明

## 这份文档给谁看

- 要补内置地区数据坐标的人
- 要做外挂 `region-provider` 的人
- 要消费 `household_region_context` 的天气类或生活类插件作者
- 要排查“为什么这个家庭拿不到天气查询点”的后端同学

这份文档只讲已经落地的边界，不发明第二套协议。

## 1. `RegionNode` 代表坐标现在怎么填

标准地区节点现在可选补 5 个字段：

- `latitude`
- `longitude`
- `coordinate_precision`
- `coordinate_source`
- `coordinate_updated_at`

字段语义很直接：

- `latitude` / `longitude`：这个地区节点的代表查询点
- `coordinate_precision`：这组坐标代表到什么精度，当前正式值包括 `country`、`province`、`city`、`district`、`point`
- `coordinate_source`：坐标从哪来，当前正式值包括 `provider_builtin`、`provider_external`
- `coordinate_updated_at`：provider 侧最后更新时间，UTC 字符串

别把规则写烂：

1. 这 5 个字段是可选的，不强迫旧 provider 一夜之间全补齐。
2. 但一旦给了经纬度，就必须同时给 `coordinate_precision` 和 `coordinate_source`。
3. 不能只给纬度不给经度，也不能只给经纬度不给精度来源。

一个合法节点长这样：

```json
{
  "provider_code": "builtin.cn-mainland",
  "country_code": "CN",
  "region_code": "110105",
  "parent_region_code": "110100",
  "admin_level": "district",
  "name": "朝阳区",
  "full_name": "北京市朝阳区",
  "path_codes": ["110000", "110100", "110105"],
  "path_names": ["北京市", "北京市", "朝阳区"],
  "timezone": "Asia/Shanghai",
  "latitude": 39.9219,
  "longitude": 116.4436,
  "coordinate_precision": "district",
  "coordinate_source": "provider_builtin",
  "coordinate_updated_at": "2026-03-18T00:00:00Z"
}
```

## 2. 内置和外挂 provider 的兼容规则

这次没有另起炉灶。

- 内置 provider 和外挂 `region-provider` 走同一套 `RegionNode` 坐标字段契约。
- 旧 provider 没坐标时，不报致命错，不影响家庭地区绑定主流程。
- 只是上层统一结果里的 `coordinate.source_type` 会变成 `unavailable`。

还有一个关键兼容点：

- provider 在构建绑定快照时，如果节点有代表坐标，应该把它写进 `representative_coordinate`
- 这样 provider 临时不可用时，旧绑定仍然可以继续用快照里的代表坐标

这不是锦上添花，这是在避免“provider 一抖，天气全死”这种垃圾体验。

## 3. 插件现在该怎么读家庭地区上下文

声明了 `capabilities.context_reads.household_region_context=true` 的插件，会在：

- `_system_context.region.household_context`

拿到统一结果。

现在要看的正式入口不是 `city`，而是：

- `household_context.coordinate`

结构大致如下：

```json
{
  "status": "configured",
  "provider_code": "builtin.cn-mainland",
  "country_code": "CN",
  "region_code": "110105",
  "display_name": "北京市 朝阳区",
  "coordinate": {
    "available": true,
    "latitude": 39.9219,
    "longitude": 116.4436,
    "source_type": "region_representative",
    "precision": "district",
    "provider_code": "builtin.cn-mainland",
    "region_code": "110105",
    "region_path": ["北京市", "北京市", "朝阳区"],
    "updated_at": "2026-03-18T00:00:00Z"
  }
}
```

`coordinate.source_type` 只有 3 个正式值：

- `household_exact`
- `region_representative`
- `unavailable`

优先级固定，不允许调用方自己改：

1. 家庭精确坐标
2. 地区代表坐标
3. `unavailable`

上层插件必须遵守一条硬规则：

- 不要再从 `city`、`display_name` 或别的文本字段猜地理点

要么直接用 `coordinate`，要么显式处理 `unavailable`。

## 4. 家庭精确坐标怎么保存

正式保存入口：

- `PATCH /households/{id}/coordinate`

当前请求体最少要带：

- `latitude`
- `longitude`
- `coordinate_source`
- `confirmed`

一个有效例子：

```json
{
  "latitude": 39.9219,
  "longitude": 116.4436,
  "coordinate_source": "manual_browser",
  "confirmed": true
}
```

当前允许的家庭坐标来源值：

- `manual_browser`
- `manual_app`
- `manual_admin`

边界别写歪：

1. 浏览器或 App 定位结果只是候选输入。
2. 没有 `confirmed=true`，后端会直接拒绝，错误码是 `coordinate_unconfirmed`。
3. 这次保存只覆盖家庭精确坐标，不会改家庭现有 `provider_code`、`region_code`。

## 5. 解析链路现在怎么工作

统一解析器会按这个顺序产出最终坐标：

1. 先看家庭有没有已经确认的精确坐标
2. 没有的话，看绑定快照里有没有 `representative_coordinate`
3. 快照也没有，再尝试实时找 provider 节点坐标
4. 还没有，就返回 `unavailable`

这里有两个容易误判的点：

- provider 临时不可用，但快照里有代表坐标：
  - 结果仍然可能是 `region_representative`
  - 这是兼容行为，不是脏数据
- 旧 provider 没补坐标：
  - 家庭地区绑定照常可用
  - 只是 `coordinate.available=false`

## 6. 联调时怎么判断是哪一层出了问题

### 情况 1：`source_type=household_exact`

说明：

- 当前生效的是家庭精确坐标

优先检查：

- `PATCH /households/{id}/coordinate` 是否真的成功保存
- `coordinate_source` 和 `coordinate_updated_at` 是否符合预期
- 用户是不是保存了错误点位

### 情况 2：`source_type=region_representative`

说明：

- 当前生效的是地区代表坐标

优先检查：

- provider 节点有没有补 `latitude`、`longitude`
- 有没有同时补 `coordinate_precision`、`coordinate_source`
- 绑定快照里的 `representative_coordinate` 是否正确

### 情况 3：`source_type=unavailable`

先分两种：

- `status=provider_unavailable`
  - 说明 provider 当前不可用，而且快照里也没有可复用代表坐标
- `status=configured`
  - 说明家庭绑定还在，但节点和快照都没有可用坐标

这时候别犯蠢：

- 不要偷偷回退到 `city` 文本做 geocoding 猜测

那只会把问题从“显式 unavailable”变成“悄悄查错地方”。

## 7. 当前明确的风险和延期项

### 风险

- 一部分外挂 provider 短期内可能仍然补不齐区县代表坐标，这类家庭会继续返回 `unavailable`
- 如果第三方插件作者无视统一 `coordinate`，继续按 `city` 猜点，天气类结果仍然会出错

### 延期项

- 这次不做自动后台持续定位
- 这次不做地图手动选点或拖点微调
- 这次不做家庭坐标历史轨迹

这些不是忘了做，是这版故意不做，避免范围继续膨胀。
