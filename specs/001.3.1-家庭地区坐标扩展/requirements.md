# 需求文档 - 家庭地区坐标扩展

状态：Draft

## 简介

家庭地区体系已经能表达“这个家庭绑定到了哪个地区节点”，但还不能稳定回答“这个家庭应该用哪个经纬度去查天气、地图或本地生活能力”。

这不是小毛病，而是数据结构缺口。没有坐标，后面的天气插件只能退回到城市名猜测。猜测一多，特殊情况就开始爆炸。

这次需求只做一件事：给现有地区体系补上可复用的坐标能力，并且把“地区代表点”和“家庭真实精确位置”分开。

## 术语表

- **地区代表坐标**：挂在地区节点上的默认经纬度，用于表达“这个区县默认拿哪个点查询天气”
- **家庭精确坐标**：家庭用户主动确认后保存的精确经纬度，用于覆盖地区代表坐标
- **坐标解析结果**：系统对外输出的统一结果，包含最终经纬度、来源、精度和地区上下文
- **Region Provider**：地区提供方，既包括内置地区目录，也包括外挂地区插件包

## 范围说明

### In Scope

- 给地区节点增加可选坐标输出能力
- 给家庭增加可选精确坐标覆盖能力
- 提供统一坐标解析链路，优先给天气插件消费
- 兼容当前家庭地区绑定逻辑和外挂地区 provider 逻辑

### Out of Scope

- 自动、持续、后台追踪家庭真实位置
- 地图 UI、轨迹记录、围栏能力
- 所有第三方地理编码服务的统一接入

## 需求

### 需求 1：地区节点要能提供代表坐标

**用户故事：** 作为系统能力提供方，我希望每个可用于天气查询的地区节点都能附带代表坐标，这样插件就不需要再拿地区名称去猜经纬度。

#### 验收标准

1. WHEN 一个 Region Provider 提供地区节点数据 THEN System SHALL 允许节点携带 `latitude`、`longitude`、`coordinate_precision` 等坐标信息。
2. WHEN 家庭绑定到带坐标的地区节点 THEN System SHALL 能直接从该节点解析出默认查询坐标。
3. WHEN Region Provider 没有提供坐标 THEN System SHALL 保持原有地区绑定功能可用，只把“无坐标”显式暴露给上层能力处理。

### 需求 2：家庭要能保存精确坐标，但不能破坏地区绑定

**用户故事：** 作为家庭管理员，我希望在需要更准天气时手动确认家庭坐标，但我不希望这一步把原来的地区绑定关系改坏。

#### 验收标准

1. WHEN 用户主动确认并保存家庭坐标 THEN System SHALL 在家庭数据中保存独立的精确坐标字段和来源信息。
2. WHEN 家庭同时存在地区绑定和精确坐标 THEN System SHALL 保留原地区绑定不变，只在坐标解析时优先使用精确坐标。
3. WHEN 浏览器或应用拿到当前位置但用户没有确认保存 THEN System SHALL 不得自动覆盖家庭已存在的坐标或地区绑定。

### 需求 3：系统要输出统一的坐标解析结果

**用户故事：** 作为天气插件开发者，我希望只拿到一份统一的坐标解析结果，而不是自己判断家庭字段、地区字段和 provider 差异。

#### 验收标准

1. WHEN 插件请求家庭地区上下文 THEN System SHALL 返回统一的坐标解析结果，至少包含最终经纬度、坐标来源、坐标精度、地区 provider 和地区节点信息。
2. WHEN 家庭存在精确坐标 THEN System SHALL 按“家庭精确坐标优先，地区代表坐标兜底”的规则返回结果。
3. WHEN 最终仍然没有坐标 THEN System SHALL 返回明确的无坐标状态，而不是伪造城市名或隐式走文本猜测。

### 需求 4：外挂地区包必须能无痛接入

**用户故事：** 作为外挂地区插件作者，我希望只按统一字段补坐标就能接入，而不是被迫重写家庭绑定机制。

#### 验收标准

1. WHEN 外挂 Region Provider 按新契约提供坐标字段 THEN System SHALL 能把它纳入统一解析链路。
2. WHEN 外挂 Region Provider 仍停留在旧版契约 THEN System SHALL 保持旧逻辑可用，并把“无坐标能力”当作兼容降级而不是报错。
3. WHEN 系统读取不同 provider 的地区节点 THEN System SHALL 使用统一的数据结构暴露坐标，而不是按 provider 分叉接口。

## 非功能需求

### 非功能需求 1：向后兼容

1. WHEN 这次功能上线 THEN System SHALL 不破坏现有家庭地区绑定、地区显示和现有 provider 装载逻辑。
2. WHEN 旧数据没有坐标 THEN System SHALL 允许系统继续运行，并对需要坐标的能力给出明确提示。

### 非功能需求 2：可维护性

1. WHEN 后续新增国家或外挂地区包 THEN System SHALL 只要求补 provider 数据，不要求重写核心地区绑定模型。
2. WHEN 后续新增依赖坐标的插件 THEN System SHALL 允许它们复用统一解析接口，而不是复制坐标优先级逻辑。

### 非功能需求 3：数据语义清晰

1. WHEN 任意模块读取坐标 THEN System SHALL 能区分“这是地区代表点”还是“这是家庭精确点”。
2. WHEN 坐标被覆盖或更新 THEN System SHALL 记录来源、精度和更新时间，避免后续排障时只看到一对裸经纬度。

## 成功定义

- 家庭地区体系可以稳定输出经纬度，不再依赖 `city` 文本猜测
- 天气插件可以只通过统一坐标解析结果完成查询
- 内置地区和外挂地区 provider 都能在不改家庭绑定主流程的前提下接入坐标能力
