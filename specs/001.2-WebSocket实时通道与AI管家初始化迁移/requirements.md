# 需求文档 - WebSocket 实时通道与 AI 管家初始化迁移

状态：Draft

## 简介

项目后面会包含大量实时更新内容，继续把实时交互压在 HTTP + SSE 上，问题只会越来越多。当前最明显的爆点就是 AI 管家初始化对话：它既要逐字输出，又要在过程中更新草稿状态，还要支持刷新恢复和多轮继续聊。现在这条链路已经反复出现挂起、恢复不稳定、状态归属混乱的问题。

这份需求文档的目标不是“换个协议试试”，而是把实时通信这层正式收口到 WebSocket，并先用 AI 管家初始化对话完成第一条可验收的迁移主链路。

## 术语表

- **System**：FamilyClaw
- **实时通道**：浏览器和后端之间长期保持的一条 WebSocket 连接，用来传输会话事件
- **初始化会话**：AI 管家初始化过程中对应的一条服务端会话记录
- **请求轮次**：用户每发一次消息，就生成一次独立的实时请求，用来区分这一轮的 chunk、状态更新和结束事件
- **快照**：服务端当前保存的会话完整状态，包括消息记录、草稿和当前阶段

## 范围说明

### In Scope

- 建立统一的 WebSocket 实时通道和事件协议
- AI 管家初始化对话迁移到 WebSocket
- 初始化会话的消息持久化、快照读取、断线恢复
- HTTP 快照接口和确认创建接口与 WebSocket 协同工作
- 为后续更多实时能力预留统一接入方式

### Out of Scope

- 不在这次把所有 AI 场景全部迁到 WebSocket
- 不重做 AI 供应商适配器本身的业务能力
- 不重做整个前端路由或初始化向导页面结构

## 需求

### 需求 1：建立统一实时通道

**用户故事：** 作为前端开发者，我希望项目有一条统一的实时通道，以便后续实时能力不用每个功能都自己发明一套流式协议。

#### 验收标准

1. WHEN 前端连接实时通道 THEN System SHALL 通过 WebSocket 建立经过身份校验的长连接。
2. WHEN 前端通过实时通道发起业务事件 THEN System SHALL 使用统一事件结构返回结果，而不是返回临时拼接的松散 JSON。
3. WHEN 连接断开或重连 THEN System SHALL 能区分连接级错误和业务级错误，并返回明确事件。

### 需求 2：迁移 AI 管家初始化对话主链路

**用户故事：** 作为正在配置家庭的用户，我希望 AI 管家初始化对话能稳定逐步返回内容，以便我可以顺畅地完成初始化，不再被挂起打断。

#### 验收标准

1. WHEN 用户在初始化页面发送消息 THEN System SHALL 通过 WebSocket 返回本轮对话的 `accepted`、`chunk`、`state_patch`、`done/error` 事件。
2. WHEN 一轮对话结束 THEN System SHALL 明确发送结束事件，并让前端能继续发送下一轮消息。
3. WHEN 同一会话连续进行多轮对话 THEN System SHALL 保证轮次不串线，不会把上一轮的事件混到下一轮。

### 需求 2.1：彻底去掉标签协议

**用户故事：** 作为维护实时链路的开发者，我希望展示文本和结构化状态彻底分开，以便后面不再被 `<config>` 这种半吊子协议反复坑。

#### 验收标准

1. WHEN 系统向前端输出可见回复文本 THEN System SHALL 只发送纯文本内容，不允许在文本里混入 `<config>...</config>`、`<json>...</json>`、`---` 分隔块或等价标签协议。
2. WHEN 系统需要提取名字、性格、说话风格等结构化状态 THEN System SHALL 通过独立结构化输出通道返回，并由服务端 reducer 合并，而不是让前端从展示文本里反解析。
3. WHEN 当前供应商支持 JSON schema、tool calling 或等价结构化输出能力 THEN System SHALL 优先直接使用该能力。
4. WHEN 当前供应商不支持结构化输出能力 THEN System SHALL 使用单独二次提取调用生成结构化状态，而不是回退到标签协议。

### 需求 3：服务端持久化消息和会话状态

**用户故事：** 作为用户，我希望刷新页面后还能回到刚才那轮初始化对话，以便不用重复输入已经聊过的内容。

#### 验收标准

1. WHEN 用户发送消息或系统返回回复 THEN System SHALL 把消息记录持久化到服务端，而不是只保存在前端内存里。
2. WHEN 页面刷新并重新进入初始化页面 THEN System SHALL 能按当前会话返回完整历史消息快照。
3. WHEN 用户明确点击重新开始 THEN System SHALL 创建新会话，并且旧会话不再继续追加新消息。

### 需求 4：断线恢复和快照恢复

**用户故事：** 作为网络不稳定的用户，我希望页面刷新、临时断线或重新打开页面后还能继续当前初始化，以便不会因为一次连接问题就重头开始。

#### 验收标准

1. WHEN WebSocket 连接断开后重新建立 THEN System SHALL 返回当前会话快照，让前端恢复消息和状态。
2. WHEN 后端在断线期间完成了一轮回复 THEN System SHALL 在重连后仍能通过快照看到完整结果。
3. WHEN 当前没有可恢复的会话 THEN System SHALL 明确返回“没有可恢复会话”，而不是让前端自己猜。

### 需求 5：保留 HTTP 兜底与兼容边界

**用户故事：** 作为系统维护者，我希望迁移到 WebSocket 后仍保留必要的 HTTP 快照和确认接口，以便调试、恢复和非实时操作不被硬绑在一条连接上。

#### 验收标准

1. WHEN 前端需要读取最近会话或指定会话快照 THEN System SHALL 仍提供 HTTP 读取接口。
2. WHEN 用户确认创建 AI 管家 THEN System SHALL 仍通过明确的 HTTP 或等价确认接口完成最终提交。
3. WHEN WebSocket 不可用或握手失败 THEN System SHALL 返回明确错误，并保留后续兜底扩展空间。

## 非功能需求

### 非功能需求 1：可靠性

1. WHEN AI 管家初始化连续进行 20 轮对话 THEN System SHALL 不出现未结束连接导致的后续请求挂起。
2. WHEN 任一轮实时输出中途中断 THEN System SHALL 保证会话和消息记录不丢失，并可通过快照恢复。

### 非功能需求 2：可维护性

1. WHEN 后续新增实时能力 THEN System SHALL 复用统一事件协议、连接管理和会话恢复机制，而不是再单独实现一套传输逻辑。
2. WHEN 开发者排查实时问题 THEN System SHALL 能通过日志定位连接建立、事件发送、轮次结束和错误原因。
3. WHEN 开发者阅读实时对话链路 THEN System SHALL 能清楚区分展示文本流、结构化状态输出和服务端合并逻辑，不再存在隐藏在文本里的控制协议。

### 非功能需求 3：编码与数据一致性

1. WHEN 文本从上游模型进入系统再写入数据库 THEN System SHALL 保持 UTF-8 可读文本，不允许出现落库乱码。
2. WHEN 前端恢复消息历史 THEN System SHALL 使用服务端持久化消息作为权威来源，而不是只依赖浏览器本地缓存。

## 成功定义

- AI 管家初始化对话主链路不再依赖 SSE
- 刷新页面后能恢复当前会话历史消息
- 连续多轮初始化对话不再出现“上一轮没结束、下一轮挂起”的问题
- 实时协议、会话持久化和前端恢复逻辑能直接复用到后续其他实时功能
