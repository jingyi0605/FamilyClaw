# Spec 001.2 - WebSocket 实时通道与 AI 管家初始化迁移

状态：Draft

## 这份 Spec 到底要解决什么

现在项目里的 AI 管家初始化对话，已经暴露出几个不能再拖的问题：

- 流式输出会间歇性挂起
- 开发代理、浏览器流读取、后端连接收尾会互相影响
- 对话消息恢复不稳定，刷新后体验不可靠
- 事件、状态、展示文本和结构化提取搅在一起，后面越扩越乱

这说明问题不是“某一个接口偶尔不稳定”，而是**实时通信这一层本身该重写了**。

这份 Spec 要做的事情很明确：

1. 给项目建立一条统一的 WebSocket 实时通道
2. 先把 AI 管家初始化对话从 HTTP/SSE 迁过去
3. 把会话状态、消息持久化、事件协议、重连恢复一次理顺
4. 保留必要的 HTTP 快照和兜底接口，但不再把实时链路压在 SSE 上

## 规划结论

### 【核心判断】

✅ 值得做，而且要尽快做。  
继续在 SSE 链路上打补丁，已经是在给技术债续命。

### 【关键洞察】

- 真正需要统一的对象不是“流式文本”，而是**会话、请求、事件、快照**
- WebSocket 不是为了炫技，而是为了把多轮实时交互放到一条稳定通道里
- AI 管家初始化对话是第一条迁移链路，但设计必须能给后面的实时能力复用

### 【Linus 式方案】

1. 先定死实时事件协议
2. 再把状态归属收回服务端
3. 再迁移 AI 管家初始化主链路
4. 最后补齐快照恢复、断线重连、兜底回退

## 这份 Spec 包含什么

- `requirements.md`：定义实时通道、会话持久化、初始化对话迁移和恢复要求
- `design.md`：定义 WebSocket 协议、后端连接管理、会话模型、前端状态机和迁移方案
- `tasks.md`：定义分阶段落地顺序、涉及文件和验证方法
- `docs/README.md`：预留联调和验收记录入口

## 这份 Spec 明确不做什么

- 不在这份 Spec 里一次性把全站所有实时功能都迁完
- 不顺手重做整个 AI 网关或所有 Agent 能力
- 不把非实时接口也强行改成 WebSocket
- 不保留 SSE 作为长期主链路，SSE 只允许在迁移期间临时存在

## 依赖与协同

- **依赖 `Spec 001.1`**：复用账号与成员绑定后的身份体系和鉴权上下文
- **依赖 `Spec 003.2`**：AI 管家初始化对话目前挂在新家庭强制向导里，迁移时不能破坏这条主链路
- **依赖 `Spec 003.3`**：初始化步骤里的 AI 配置中心页面和接口要一起接入新的实时通道
