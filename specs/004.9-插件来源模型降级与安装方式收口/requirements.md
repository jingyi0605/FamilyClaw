# 需求文档 - 插件来源模型降级与安装方式收口

状态：In Review

## 简介

当前插件系统把三件不同的事情混在了一起：

1. 插件是不是内置能力
2. 插件代码是谁维护的
3. 插件是怎么安装进运行时的

这导致系统里同时存在 `builtin`、`official`、`third_party` 三套来源语义，目录结构、数据库字段、启动同步和市场安装流程都在围着这三个词打补丁。最直接的坏结果，就是官方插件开发目录和插件市场安装目录发生职责冲突，开发态源码和运行时产物互相踩。

这次 Spec 要把模型降级到两条真正有意义的轴线：

- 插件类型只有两种：`builtin`、`third_party`
- 第三方插件安装方式只有两种：`marketplace`、`local`

做完以后，系统不再区分“官方插件”和“第三方插件”，只区分“内置还是第三方”，以及“第三方是市场安装还是本地安装”。

## 术语表

- **System**：FamilyClaw 插件平台、插件运行时、插件市场接入层和插件落盘目录。
- **内置插件（builtin）**：跟宿主一起交付、由宿主注册表直接加载的插件，不走第三方安装流程。
- **第三方插件（third_party）**：所有非内置插件。无论它以前被叫“官方插件”还是“第三方插件”，这次统一收口到这里。
- **市场安装（marketplace）**：从插件市场源解析条目、下载包并安装到运行时目录的第三方插件。
- **本地安装（local）**：用户或开发者从本地包、本地目录或手工导入流程安装到运行时目录的第三方插件。
- **开发源码目录**：仓库内用于开发、调试第三方插件的源码目录，不属于运行时安装目录，也不应被插件市场安装流程写入。
- **运行时安装目录**：`apps/api-server/data/plugins/` 下真正给宿主加载和执行的插件产物目录。

## 范围说明

### In Scope

- 删除插件系统中的 `official` 来源类型和 `trusted_level` 语义。
- 把插件类型统一收口为 `builtin` 和 `third_party`。
- 给第三方插件补齐安装方式语义：`marketplace` 和 `local`。
- 重构运行时目录结构，保证开发源码目录和运行时安装目录彻底分离。
- 调整数据库字段、启动同步、插件安装、卸载、清理和回填逻辑。
- 调整插件市场源模型，去掉“官方市场 vs 第三方市场”的插件信任分叉。
- 补齐迁移策略，保证旧数据、旧目录、旧挂载记录可以平滑收口。
- 同步更新受影响的正式文档。

### Out of Scope

- 新增插件签名、付费、商业审核或新的安全沙箱机制。
- 重做插件市场 UI 信息架构。
- 引入远程插件执行、容器隔离或跨机器部署能力。
- 顺手扩展新的插件能力类型或新的业务插件。

## 需求

### 需求 1：插件类型必须只保留 builtin 和 third_party

**用户故事：** 作为平台维护者，我希望插件来源模型只表达真正影响系统行为的区别，这样我在看代码、改数据库和排查问题时，不需要再处理“official 到底算来源、信任级别还是安装来源”这种假问题。

#### 验收标准

1. WHEN System 读取、创建或更新插件挂载记录 THEN System SHALL 只接受 `builtin` 或 `third_party` 作为插件类型。
2. WHEN 现有代码路径仍然传入 `official` THEN System SHALL 在迁移窗口内把它兼容映射为 `third_party`，而不是继续保留新的 `official` 分支。
3. WHEN 宿主启动并同步插件状态 THEN System SHALL 不再扫描或维护“官方插件专属来源类型”分支。

### 需求 2：第三方插件必须明确区分市场安装和本地安装

**用户故事：** 作为插件开发者或运维人员，我希望系统明确知道一个第三方插件是从市场装的还是从本地装的，这样安装路径、升级路径、卸载逻辑和排障方式都能说清楚，而不是继续混在“official / third_party”这种伪分类里。

#### 验收标准

1. WHEN System 挂载第三方插件 THEN System SHALL 记录安装方式为 `marketplace` 或 `local`。
2. WHEN System 同步第三方插件落盘目录 THEN System SHALL 按安装方式分别扫描运行时目录，而不是按 `official` 和 `third_party` 两层目录扫描。
3. WHEN 用户卸载或清理第三方插件 THEN System SHALL 只删除该安装方式对应的运行时产物，不得误删开发源码目录或其他安装方式的数据。

### 需求 3：开发源码目录和运行时安装目录必须彻底分离

**用户故事：** 作为第三方插件开发者，我希望仓库里的开发源码目录只是开发源码，不会被插件市场安装流程覆盖，也不会被宿主误当成某个家庭的运行时安装目录。

#### 验收标准

1. WHEN 开发者在仓库内调试第三方插件 THEN System SHALL 使用独立的开发源码目录，不得与 `apps/api-server/data/plugins/` 运行时安装目录复用。
2. WHEN 插件市场安装第三方插件 THEN System SHALL 只写入运行时安装目录中的 `third_party/marketplace` 分支，不得写入开发源码目录。
3. WHEN 宿主启动、迁移或执行安装清理 THEN System SHALL 不得依赖开发源码目录存在，也不得因为开发源码目录缺失而启动失败。

### 需求 4：旧的 official 语义必须平滑迁移到新模型

**用户故事：** 作为已有系统的维护者，我希望这次收口不会把现有插件实例、市场源、挂载记录和运行时目录直接打爆，而是能把旧数据平滑收口到新模型。

#### 验收标准

1. WHEN 数据库里存在 `official` 插件挂载记录或市场源记录 THEN System SHALL 提供明确的迁移与回填策略，把旧值收口为新模型。
2. WHEN 磁盘上仍然存在旧的 `official/marketplace` 安装目录 THEN System SHALL 将其迁移或兼容映射到 `third_party/marketplace`，并确保后续只写新目录。
3. WHEN 迁移完成后 THEN System SHALL 不再新增任何 `official` 字段值、目录分支或专属逻辑判断。

### 需求 5：插件市场源必须去掉 trusted_level 分叉，但保留系统内置市场源能力

**用户故事：** 作为平台维护者，我希望“默认市场源是系统自带的”这件事能用简单字段表达，而不是继续把它绑到 `official` 这种会污染插件模型的概念上。

#### 验收标准

1. WHEN System 创建默认市场源 THEN System SHALL 使用“系统内置市场源”语义标记该源，而不是继续依赖 `trusted_level=official`。
2. WHEN 用户手动添加市场源 THEN System SHALL 不再要求市场源声明 `trusted_level=third_party`。
3. WHEN System 安装来自任意市场源的第三方插件 THEN System SHALL 统一按 `third_party + marketplace` 处理，不得因为市场源是否系统内置而分叉插件类型。

## 非功能需求

### 非功能需求 1：可维护性

1. WHEN 开发者搜索插件来源模型相关代码 THEN System SHALL 能把核心判断集中到“插件类型”和“安装方式”两个字段，而不是分散在 `official`、`trusted_level`、目录名和硬编码常量里。
2. WHEN 后续再新增第三方安装入口 THEN System SHALL 可以复用现有 `third_party + install_method` 模型，而不是再引入新的来源类型。

### 非功能需求 2：兼容性

1. WHEN 旧版本数据库、旧目录或旧市场源记录仍然存在 THEN System SHALL 提供兼容读取或一次性迁移，避免现有家庭插件实例直接失效。
2. WHEN 新模型落地后 THEN System SHALL NOT 再破坏现有内置插件加载、第三方插件启停、市场安装、手动安装和插件清理流程。

### 非功能需求 3：可观测性

1. WHEN 启动同步、安装、卸载或迁移失败 THEN System SHALL 在日志里明确写出插件类型、安装方式、家庭 ID、插件 ID 和磁盘路径。
2. WHEN 运维人员排查插件来源问题 THEN System SHALL 能通过数据库记录和运行时目录直接判断“这是内置插件还是第三方插件、这是市场安装还是本地安装”。

## 成功定义

- 插件来源模型里不再出现 `official` 作为正式运行时语义。
- 第三方插件的运行时目录只保留 `local` 和 `marketplace` 两类安装分支。
- 仓库内第三方插件开发源码目录不再和运行时安装目录冲突。
- 插件市场默认源仍然可用，但不再依赖 `trusted_level=official` 这类伪分类。
- 新代码、数据库和文档能让后来者一眼看明白插件到底是“内置还是第三方、市场装还是本地装”。
