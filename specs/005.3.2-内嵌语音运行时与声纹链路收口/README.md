# Spec 005.3.2 - 内嵌语音运行时与声纹链路收口

状态：Draft

## 这份 Spec 要解决什么问题

`005.3` 已经把小爱声纹采集、建档、识别和对话前身份解析这条链路立住了，但当前落地方式还有一个明显的问题：

- 本地开发和部署时还要额外启动一个 `voice-runtime` 进程
- `api-server` 自己已经有声纹算法、路由和降级逻辑，却还要通过 HTTP 再绕一圈
- 音频缓存、音频落盘和声纹识别边界分在两个进程里，调试和回归都更重
- 如果只是为了把 `pcm` 片段拼成 `.wav`，单独再起一个服务，成本有点蠢

这份 Spec 的目标很明确：

1. 把当前 `voice-runtime` 的音频缓存、落盘和 commit 兜底逻辑内嵌回 `api-server`
2. 不再要求本地默认场景额外启动独立 `voice-runtime` 进程
3. 保留现有语音主链、声纹主链和降级语义，不破坏现有用户空间
4. 严格遵守 `001.6` 的后端事件循环规范，避免把同步音频处理和声纹计算直接塞进主事件循环

一句话说清楚：

> `005.3.2` 不是重做声纹能力，而是把现在分裂在两个进程里的那点运行时逻辑收回来，做成一个不会卡主事件循环的内嵌实现。

## 【核心判断】

✅ 值得做，而且现在就该做。

理由很简单：

- 声纹算法本来就在 `api-server`
- 当前独立 `voice-runtime` 主要承担的是短生命周期音频缓存和 `.wav` 落盘
- 这类能力并不需要单独进程，只需要明确线程池边界和内存生命周期
- 只要阻塞任务下沉，单进程完全能扛住这条链路

## 和已有 Spec 的关系

- `005.3` 负责“小爱声纹采集、建档、识别、对话前身份解析”的业务能力本身
- `005.3.1` 负责“声纹管理前端实现”
- `005.3.2` 负责“把当前独立 `voice-runtime` 收回 `api-server`，并把运行时边界整理干净”

所以这份 Spec 的角色不是另起炉灶，而是给 `005.3` 补一个运行时收口层。

## 这次范围

这次要做的事：

- 新增 `api-server` 内嵌 runtime 模式
- 在 `api-server` 内部保存短生命周期音频分片，并在 commit 时落成音频产物
- 把音频落盘、hash、声纹 embedding、建档和识别调用统一下沉到 blocking helper
- 删除独立 `voice-runtime` 进程、启动脚本和相关配置入口，不再保留远程回退
- 迁移完成后，回写更新 `005.3` 中涉及运行时拓扑和启动方式的描述

这次明确不做的事：

- 不重做声纹模型，不讨论更换 `weSpeaker/ResNet34`
- 不新增多说话人分离、降噪或在线流式 ASR
- 不把普通语音网关协议推倒重来
- 不顺手重做前端声纹管理流程

## 主要文档

- `requirements.md`：把迁移要守住的兼容边界、事件循环边界和验收标准写清楚
- `design.md`：把内嵌 runtime 的模块边界、线程池策略和数据流写清楚
- `tasks.md`：把迁移拆成可以连续执行的阶段，最后明确回写 `005.3`
- `docs/README.md`：预留联调、验收和迁移记录入口
