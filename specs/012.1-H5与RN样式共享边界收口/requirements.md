# 需求文档 - H5与RN样式共享边界收口

状态：Done

## 简介

`user-app` 现在已经同时支持 `H5` 和 `RN`。问题不在于有没有跨端，而在于跨端之后样式职责没有切干净。

现在最常见的坏味道有三类：

- 明明是同一个视觉语义，两边各写一套，改一次得改两遍。
- 明明是平台专属能力，却总有人想硬共用，最后把组件写成一堆平台判断。
- 页面层拿不到稳定规则，只能继续在业务代码里补临时样式，补着补着又回到老路。

这份 Spec 要做的，就是把这三件事一次说清楚：

1. 什么该共享。
2. 什么不该共享。
3. 后续页面应该按什么分层去实现。

## 术语表

- **System**：`apps/user-app`、`packages/user-ui` 以及两端主题运行时。
- **基础 token**：颜色、字号、间距、圆角、阴影这类原始设计值。
- **语义 token**：例如“页面背景”“卡片边框”“主按钮背景”这种带用途的变量。
- **组件语义**：按钮、卡片、表单项、页头、空状态这类可以跨端保持一致含义的 UI 规则。
- **平台适配器**：把同一份共享契约分别转成 `H5` 的 CSS 变量和 `RN` 的 JS token/样式对象的实现层。
- **页面布局模式**：例如单列详情页、双栏桌面页、移动端单列页这种结构级规则。

## 范围说明

### In Scope

- 定义 `H5` 与 `RN` 共享的主题 token、语义 token 和组件语义边界。
- 定义 `H5` 与 `RN` 各自负责的样式实现边界。
- 定义页面布局模式在 `H5` 窄屏、`H5` 桌面、`RN` 三种场景下的统一语义。
- 选取高频页面做第一批迁移试点，验证分层可落地。
- 补充文档和任务清单，让后续页面迁移有统一入口。

### Out of Scope

- 强行合并 `H5` 的 `scss/css` 和 `RN` 的 `StyleSheet` 成一份源码。
- 全量重做所有页面视觉。
- 在这份 Spec 里重新定义品牌风格或重新做设计稿。
- 处理所有历史页面遗留问题。

## 需求

### 需求 1：系统必须先建立唯一的跨端共享契约

**用户故事：** 作为前端开发者，我希望颜色、间距、字号、卡片和按钮这些跨端共有语义只有一套正式定义，这样我改规则时不用在 `H5` 和 `RN` 两边自己猜。

#### 验收标准

1. WHEN 团队需要查找跨端共有的视觉规则 THEN System SHALL 提供唯一的共享契约入口，而不是要求开发者分别去翻 `H5` 和 `RN` 私有实现。
2. WHEN `H5` 与 `RN` 需要读取主题和组件语义 THEN System SHALL 从同一份共享契约派生各自的运行时结果。
3. WHEN 团队发现共享语义缺失 THEN System SHALL 先补共享契约，再补平台适配器，而不是直接在页面里写新的临时样式。

### 需求 2：系统必须明确禁止共享具体样式实现

**用户故事：** 作为维护者，我希望团队不要再尝试让 `H5` 和 `RN` 共用一份具体样式文件，这样代码不会因为平台差异被一堆条件分支搞烂。

#### 验收标准

1. WHEN 组件涉及 `media query`、`hover`、滚动条、`position: fixed` 等浏览器能力 THEN System SHALL 明确要求这些实现只留在 `H5` 适配层。
2. WHEN 组件涉及 `SafeArea`、触摸反馈、原生滚动、`elevation` 等 `RN` 能力 THEN System SHALL 明确要求这些实现只留在 `RN` 适配层。
3. WHEN 团队评审跨端组件方案 THEN System SHALL 能清楚区分“共享语义”与“平台实现”，而不是把两者混成一个对象。

### 需求 3：系统必须定义跨端一致的页面布局模式

**用户故事：** 作为产品和开发共同维护的页面负责人，我希望同一页面在 `H5` 桌面、`H5` 手机和 `RN` 上至少信息层级一致、操作顺序一致，这样跨端不会各说各话。

#### 验收标准

1. WHEN 同一页面在 `H5` 手机和 `RN` 上展示 THEN System SHALL 保持核心信息顺序、操作优先级和空状态语义一致。
2. WHEN `H5` 页面进入窄屏或触屏场景 THEN System SHALL 自动降级为触屏友好的布局，而不是继续暴露鼠标专属交互。
3. WHEN `H5` 页面处于桌面宽屏场景 THEN System SHALL 允许保留更丰富的桌面布局和交互，但不得破坏共享语义。

### 需求 4：系统必须给页面迁移提供明确顺序和边界

**用户故事：** 作为接手迁移的人，我希望知道先改哪些页面、哪些页面先不动、做完一批后怎么判断可以继续推进，这样不会一上来就冲进最脏的历史代码里送死。

#### 验收标准

1. WHEN 团队开始迁移 THEN System SHALL 明确第一批试点页面、第二批高频页面和暂缓迁移页面清单。
2. WHEN 某个页面还没完全迁移 THEN System SHALL 明确记录它当前的兼容状态和剩余债务，而不是模糊写成“后续再优化”。
3. WHEN 团队完成一批迁移 THEN System SHALL 有明确的回归检查口径，确认共享边界没有把现有功能搞坏。

### 需求 5：系统必须保证主题运行时和现有页面兼容

**用户故事：** 作为用户和维护者，我希望这次收口不会把现有主题切换、页面展示和平台运行时直接改坏，这样改造才能持续推进。

#### 验收标准

1. WHEN 共享契约接管跨端主题语义 THEN System SHALL 保持现有主题切换能力继续可用。
2. WHEN 部分页面仍处于兼容阶段 THEN System SHALL 允许旧页面继续运行，但不得再把旧做法扩散成新标准。
3. WHEN 第一批迁移完成 THEN System SHALL 能通过类型检查、样式守卫或人工验收确认主要页面无明显回归。

## 非功能需求

### 非功能需求 1：可维护性

1. WHEN 团队调整跨端视觉语义 THEN System SHALL 优先改共享契约，而不是要求同时修改多份平台私有定义。
2. WHEN 新增页面或组件 THEN System SHALL 让开发者先复用共享语义和基础组件，而不是重新发明一套样式。

### 非功能需求 2：兼容性

1. WHEN 历史页面尚未完全迁移 THEN System SHALL 提供兼容过渡层，避免一次性硬切导致大面积回归。
2. WHEN `H5` 与 `RN` 读取同一主题 THEN System SHALL 保持核心语义一致，即使两边渲染细节不同。

### 非功能需求 3：可验证性

1. WHEN 完成每一批迁移 THEN System SHALL 提供可执行的检查方式，包括类型检查、样式守卫和人工验收。
2. WHEN 团队审查某个跨端组件 THEN System SHALL 能快速判断它是否正确地落在共享层或平台层。

## 成功定义

- 团队能明确说出哪些东西共享、哪些东西不共享，而不是继续靠口头默契。
- `H5` 和 `RN` 都从同一份共享契约派生主题和组件语义。
- 第一批高频页面已经证明这套分层不是纸上谈兵。
- 后续新页面默认站在新规则上，而不是继续复制旧页面的散装样式。
