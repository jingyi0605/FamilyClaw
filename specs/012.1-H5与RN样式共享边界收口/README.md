# Spec 012.1 - H5与RN样式共享边界收口

状态：Done

## 这份 Spec 要解决什么问题

`apps/user-app` 现在已经不是“样式有点乱”这么简单，而是 `H5` 和 `RN` 到底该共享什么、不该共享什么，没有明确边界。

现状里同时存在几件事：

- 主题运行时已经有共享基础，但页面层和组件层没有完全收口。
- `H5` 继续写 `scss/css`，`RN` 继续写 `StyleSheet`，这是正常的；不正常的是两边经常重复定义同一套视觉语义。
- 有些人会自然地想“能不能共享一套样式”，但这条路走下去只会得到更多平台分支和更难维护的垃圾代码。

这份子 Spec 的目标不是重做一套设计系统，也不是要求 `H5` 和 `RN` 共用一份样式文件。目标很直接：

1. 把真正应该共享的东西收成唯一标准源。
2. 把必须按平台分开的实现边界写死。
3. 给后续页面迁移一条清楚路线，别再一边改页面一边重新争论规则。

## 这次明确要做什么

- 明确 `H5` 与 `RN` 共享的对象：主题 token、语义 token、组件语义、页面布局模式。
- 明确 `H5` 与 `RN` 不共享的对象：CSS/SCSS、`StyleSheet`、媒体查询、hover、Safe Area、平台专属交互细节。
- 定义一套“共享契约 + 平台适配器”的分层结构，让 `H5` 和 `RN` 都从同一份契约取值，但各自生成自己的样式实现。
- 先拿高频页面做第一批试点，验证这个边界不是纸上谈兵。

## 这次明确不做什么

- 不把 `H5` 和 `RN` 强行合并成一套通用样式文件。
- 不在这份子 Spec 里重做整套品牌视觉。
- 不要求一次性重写 `user-app` 所有页面。
- 不把桌面 `H5` 交互原样搬到手机 `H5` 或 `RN` 上。

## 和父 Spec 012 的关系

- 父 Spec [012-user-app界面标准化与样式收口](/C:/Code/FamilyClaw/specs/012-user-app界面标准化与样式收口/README.md) 解决的是 `user-app` 整体样式标准化。
- 这个子 Spec 解决的是其中最容易继续失控的一块：`H5` 和 `RN` 的共享边界、共享契约和平台适配分工。
