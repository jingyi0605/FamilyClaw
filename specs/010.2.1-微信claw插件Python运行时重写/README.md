# Spec 010.2.1 - 微信claw插件Python运行时重写

状态：Draft

## 这份 Spec 解决什么问题？

`010.2` 把微信 claw 通道正式收口成了第三方插件，但当时为了复用已经验证过的 transport，保留了一个 `Python -> Node bridge -> Node transport` 的双层运行时。

这条路现在已经暴露出两个现实问题：

- 宿主后端明明是 Python，却为了一个插件额外依赖 Node.js 运行时
- Docker 运行镜像默认不带 Node，插件一启用就要么改基础镜像，要么改部署说明，边界越来越脏

这次不是补一个安装说明，也不是继续给镜像塞依赖，而是把微信插件的后端实现彻底收回到 Python。

这份 Spec 只解决一件事：

把微信插件现有的 Node 子层删掉，用 Python 完整接管扫码登录、状态刷新、消息轮询、文本发送、媒体处理和运行时状态管理，同时保持宿主看到的插件契约不变。

## 为什么现在做？

因为问题已经不是“风格偏好”，而是运行边界失控。

- 今天为微信插件加 Node
- 明天别的插件再带一套 JS runtime
- 后天再来一个 Go sidecar

最后宿主不是在跑插件系统，是在养动物园。

这不叫灵活，这叫失控。

现在停下来重写，代价是一次插件内部迁移；继续拖，代价会变成部署复杂度、漏洞面、文档维护和排障成本一起往上滚。

## 核心判断

✅ 值得做：Node 方案已经证明“能跑”，但也已经证明“边界不好看”。真正该保留的是微信插件能力，不是 Node 运行时本身。

## 关键洞察

- 数据结构：真正值钱的是登录态、轮询游标、`context_token`、媒体缓存和标准化消息，不是 Node bridge 这层胶水代码。
- 复杂度：当前最大的无谓复杂度不是扫码或轮询，而是跨语言桥接。删掉 Node 子层，本身就是一次大简化。
- 风险点：Python 重写不能只把二维码生成迁走，必须把 `start_login / get_login_status / poll / send` 一条链一起收口，不然会留下双栈半残状态。

## 这次要交付什么？

- 一份明确禁止插件后端再引入第二套运行时的需求文档
- 一份把 Python transport 结构、接口、状态模型和迁移步骤写清楚的设计文档
- 一份可执行的任务拆分，覆盖实现、回归、文档和发布
- 一份更新后的插件手册约束，明确插件后端逻辑必须用 Python 完成

## 这次明确不做什么？

- 不改宿主通用 `channel`/`action` 契约
- 不顺手重做微信产品范围，比如群聊、联系人同步、朋友圈等
- 不为了追求“更优雅”重写插件全部业务文案和前端表现
- 不继续兼容 `Python + Node` 双栈运行模式；迁移完成后 Node bridge 必须退出历史舞台

## 和上层 Spec 的关系

这份 Spec 是 [`010.2-微信claw第三方通道插件`](/C:/Code/FamilyClaw/specs/010.2-微信claw第三方通道插件/README.md) 的子 Spec。

- `010.2` 解决“微信 claw 怎么以第三方插件身份接进来”
- `010.2.1` 解决“微信插件后端怎么从 Node 子层迁回 Python，并把运行时边界重新收紧”

这份 Spec 生效后，`010.2` 中所有涉及 “Python/Node 双层适配” 和 “Node bridge” 的设计，都应该视为待迁移旧方案，而不是长期目标。
