# 20260407-Node子层能力盘点

## 这份文档解决什么问题？

这份文档只做一件事：把旧 Node 子层到底替微信插件做了什么说清楚，避免 Python 重写时漏能力、漏状态、漏协议。

它不是最终设计文档，而是迁移对照表。

## 盘点范围

本次盘点覆盖下面这些旧实现文件：

- `apps/api-server/plugins-dev/weixin_claw_channel/plugin/bridge.py`
- `apps/api-server/plugins-dev/weixin_claw_channel/vendor/weixin_transport/bridge.mjs`
- `apps/api-server/plugins-dev/weixin_claw_channel/vendor/weixin_transport/src/dispatch.mjs`
- `apps/api-server/plugins-dev/weixin_claw_channel/vendor/weixin_transport/src/login-transport.mjs`
- `apps/api-server/plugins-dev/weixin_claw_channel/vendor/weixin_transport/src/channel-transport.mjs`
- `apps/api-server/plugins-dev/weixin_claw_channel/vendor/weixin_transport/src/media-transport.mjs`
- `apps/api-server/plugins-dev/weixin_claw_channel/vendor/weixin_transport/src/weixin-api.mjs`

## 旧 Node 子层的职责清单

### 1. 请求分发

入口在 `dispatch.mjs`。

它做的事不复杂：

- 判断当前请求是 `action` 还是 `channel`
- 把 `start_login / get_login_status / logout / purge_runtime_state` 分给登录逻辑
- 把 `poll / send / probe` 分给通道逻辑
- 测试环境下支持 stub 响应

Python 重写后，这一层可以直接消失，因为 Python 本身已经有 `action.py` 和 `channel.py` 两个正式入口。

### 2. 扫码登录

入口在 `login-transport.mjs`。

它当前负责：

- 调 `ilink/bot/get_bot_qrcode`
- 取回 `qrcode` 和 `qrcode_img_content`
- 把二维码目标 URL 转成 SVG data URL
- 生成 `login_session_key`
- 返回 `waiting_scan` 状态、`expires_at`、`api_base_url`

这里最关键的不是 Node，而是下面这几件具体事：

- 登录二维码接口地址
- 返回字段含义
- 二维码图片预览生成方式
- 登录会话的 TTL 和状态枚举

Python 重写时，这些都要保留，Node 不需要保留。

### 3. 登录状态刷新

也是 `login-transport.mjs` 负责。

它当前负责：

- 调 `ilink/bot/get_qrcode_status`
- 用 `qrcode` 查询当前扫码状态
- 把上游状态映射成：
  - `waiting_scan`
  - `scan_confirmed`
  - `active`
  - `expired`
- 返回 `bot_token`、`ilink_bot_id`、`ilink_user_id`、`baseurl`

Python 重写时必须保证这套状态映射继续成立，不然后面的 `action.py` 和宿主状态区都会被搞坏。

### 4. 消息轮询

入口在 `channel-transport.mjs` 的 `pollMessages(...)`。

它当前负责：

- 校验 token
- 调 `weixin-api.mjs` 的 `fetchUpdates(...)`
- 传入旧游标 `get_updates_buf`
- 接收 `msgs`
- 发现 `errcode == -14` 时转成 `login_expired`
- 返回：
  - `messages`
  - `transport_cursor`
  - `longpolling_timeout_ms`

Python 重写时必须保住下面这几个点：

- 游标继续可用
- `errcode == -14` 仍然能打成登录失效
- 原始消息仍然能被宿主后续标准化链路接住

### 5. 文本发送

入口在 `channel-transport.mjs` 的 `sendDelivery(...)` 和 `sendTextOnlyMessage(...)`。

它当前负责：

- 校验 `external_user_id`
- 校验 `delivery.text / delivery.attachments`
- 校验 `context_token`
- 用上游接口发送文本
- 返回 `provider_message_ref`

Python 重写时，这部分最容易出现的烂问题是：

- 忘了校验 `context_token`
- 成功和失败的错误码变了
- `provider_message_ref` 生成逻辑不一致

这些都不能乱。

### 6. 媒体上传下载

主要在 `channel-transport.mjs` 和 `media-transport.mjs`。

旧实现已经干了不少脏活：

- 根据附件类型区分 `image / video / file`
- 调上游拿上传 URL
- 计算文件 MD5
- 生成 AES key
- 做 AES-ECB 加解密
- 上传加密后的 buffer 到 CDN
- 下载入站媒体并解密
- 推断 MIME、扩展名、文件名
- 把媒体缓存到插件私有目录

这部分是 Python 重写里最硬的一块，也是最不适合“先简单做一版”的地方。因为它不是页面效果问题，而是协议细节问题。

### 7. 上游 HTTP 协议

在 `weixin-api.mjs` 里。

它当前负责：

- `getupdates`
- `getuploadurl`
- `sendmessage`
- 构建通用请求头
- 生成 `X-WECHAT-UIN`
- 设置超时

Python 重写时必须把这些请求入口、请求体和请求头逐项对齐。否则看起来是“已经改成 Python 了”，实际上只是换了语言继续调错协议。

## 迁移时必须保住的正式能力

### 必保能力

- 扫码登录
- 登录状态刷新
- 轮询收消息
- 文本发送
- 第一版基础媒体收发
- `context_token` 配合现有状态持久化继续可用

### 可以直接删掉的旧层

- `Python -> Node` 桥接协议本身
- `dispatch.mjs` 这种中间分发层
- Node 侧为了测试准备的 stub 包装壳

### 不能想当然跳过的细节

- 二维码 SVG 生成
- `errcode == -14` 的登录失效处理
- CDN 下载上传加密逻辑
- 旧游标字段 `get_updates_buf`
- 旧 token / bot id / user id 的返回字段映射

## 对 Python 重写的直接结论

1. 登录链路不是纯接口转发，还包括二维码图片生成
2. 发送链路不是纯文本发送，还包括附件上传和加密
3. 轮询链路不是简单拉消息，还包括登录失效识别和入站媒体补全
4. 真正可以删掉的是跨语言胶水，不是这些协议细节

一句话总结：

Python 重写不是“把 Node 代码翻译一下”，而是“在 Python 里把现有正式能力重新落一遍，并保证宿主契约不变”。
