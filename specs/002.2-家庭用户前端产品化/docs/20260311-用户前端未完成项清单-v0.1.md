## 2. 高优先级

这部分不补，阶段 2 和阶段 3 很难说已经形成产品闭环。

### 2.1 把家庭页补成真正可维护，不只是能看和部分新增

- 现在已有：房间、成员、关系的真实读接口已经接上；新增房间、成员、关系也已经能走通。
- 现在没做完：
  - 家庭概览还是只读，没有家庭基础信息编辑入口
  - 房间卡片上的“编辑”按钮还是禁用状态
  - 任务里提到“家庭、房间、成员、关系都可维护”，当前更接近“成员和关系能改，房间和家庭主体还不完整”
- 主要看哪里：
  - `apps/user-web/src/pages/FamilyPage.tsx`
- 做完后应该看到什么：
  - 用户能改家庭基础信息
  - 用户能编辑已有房间，而不是只能新增
  - 家庭页可以完整承担主数据维护职责

### 2.2 把首页快捷动作补成真入口

- 现在已有：首页聚合数据已经基本接真，降级也有。
- 现在没做完：快捷操作区只是按钮，没有页面跳转或动作绑定。
- 主要看哪里：
  - `apps/user-web/src/pages/HomePage.tsx`
- 做完后应该看到什么：
  - 点“对话 / 记忆 / AI配置 / 家庭”能直接进入对应页面
  - 首页上的异常处理和后续动作链路更顺

### 2.3 先把 `AI配置` 的后端和 `admin-web` 承载做实，再回头收口用户端占位配置

- 现在已有：
  - 设备与 HA 同步能工作
  - AI 页已经接了部分真实字段
  - 通知页已经接了免打扰开关
- 现在没做完：
  - `AI配置` 还没有多 Agent 的稳定后端承载，当前用户端只能看到零散字段
  - AI 页的“助手称呼、回复语气、回复长度、输出语言、是否使用记忆、是否建议提醒、是否建议场景”这套旧表述已经不准，后面应该改成 Agent 级配置
  - 通知页的“通知方式、通知范围”还是占位，其中一部分还是禁用状态
  - 任务里提到的 `ai/runtime-defaults`、`ai/routes` 以及后续多 Agent 配置接口还没形成稳定产品语义
- 主要看哪里：
  - `apps/api-server/app/modules/`
  - `apps/admin-web/src/`
  - `apps/user-web/src/pages/SettingsPage.tsx`
- 做完后应该看到什么：
  - 后端已经能承载多 Agent 的基础配置
  - `admin-web` 能先完成 Agent 级配置和联调验证
  - 用户端后续再接 `AI配置` 时，不需要继续摆假表单

### 2.4 先把“对话”需要的后端会话与多 Agent 路由补齐

- 现在已有：
  - 推荐问题、提问、连续追问、降级提示、事实引用都已经能工作
  - “转为提醒”“写入记忆”已经打通
- 现在没做完：
  - 会话持久化还没有稳定服务端承载
  - 当前还是单助手思路，不是多 Agent 运行时
  - 会话搜索和会话管理也还没有后端语义
- 主要看哪里：
  - `apps/api-server/app/modules/family_qa/`
  - `apps/api-server/app/modules/agent/`
- 做完后应该看到什么：
  - 对话不再只是前端本地壳子
  - 后端已经能按 Agent 维度承接会话和运行时上下文
  - 后续用户端改“对话”页时有稳定接口可接

### 2.5 先在管理侧把 Agent 到记忆 / 提醒等动作链路验证清楚

- 现在已有：“转为提醒”“写入记忆”这类能力已经有部分接口基础。
- 现在没做完：
  - 多 Agent 下到底哪些动作归主管家，哪些动作允许专业 Agent 触发，还没有稳定边界
  - 还没有一套先在 `admin-web` 验证的联调口径
- 主要看哪里：
  - `apps/admin-web/src/`
  - `apps/api-server/app/modules/agent/`
  - `specs/003.1-AI助手人格化设计/`
- 做完后应该看到什么：
  - 团队先知道多 Agent 的动作边界怎么分
  - 后续用户端做“对话”快捷动作时，不会又补一堆假入口

### 2.6 补手机端家庭切换入口

- 现在已有：移动端底部导航已经有了。
- 现在没做完：家庭切换下拉框放在左侧导航里，而移动端把这块直接隐藏了，手机上实际上没地方切家庭。
- 主要看哪里：
  - `apps/user-web/src/components/ShellNav.tsx`
  - `apps/user-web/src/styles.css`
- 做完后应该看到什么：
  - 手机上也能切当前家庭
  - 家庭上下文不会只在桌面可用

## 5. 建议的实际推进顺序

如果只按“先把产品做成能演示、能继续用”来排，建议顺序是：

1. 家庭页补全编辑能力
2. 设置页去掉假表单，能接真的全接真
3. 首页快捷动作接页面跳转
4. 先补后端与 `admin-web` 的 `AI配置 + 对话` 前置承载
5. 再回到 `user-web` 做对话页和 AI配置页产品化
6. 手机端家庭切换
7. 统一空态 / 错误态 / 降级提示
8. 联调说明、验收记录、任务状态回写

原因很简单：

- 先把主链路做实
- 再把 `AI配置 + 对话` 的后端和 `admin-web` 地基打稳
- 然后再回到 `user-web` 收产品体验
- 最后把文档和验收收口

---

## 6. 一句话结论

现在的 `user-web` 已经不是空壳了，但 `AI配置 + 对话` 这条线不能再靠前端壳子硬顶。下一步应该先补后端和 `admin-web` 的多 Agent 承载，再回到 `user-web` 做真正的产品化收口。
