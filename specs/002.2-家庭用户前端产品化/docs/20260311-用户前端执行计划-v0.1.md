# 20260311-用户前端执行计划-v0.1

## 1. 这份文档是干什么的

这不是讨论稿，是给开发直接照着排活的。

目标只有一个：

把 `user-web` 现在还没收口的事情，拆成一批能直接开干、能单独验收、做完就能回写状态的任务。

---

## 2. 执行原则

这轮按下面的顺序推进，不要并行乱铺：

1. 先补主链路，优先把“看状态 + 改配置”做完整
2. 再补 `AI配置 + 对话 + 记忆`，但这轮先做后端和 `admin-web` 的承载，不先继续改 `user-web`
3. 最后再回到用户前端，把体验统一和文档验收收口

控制范围：

- 不为了前端补齐去临时扩后端大接口
- 没有稳定后端承载的字段，不做假保存
- `AI配置` 和 `对话` 这条线，当前先不继续扩 `user-web` 页面实现，先把后端和 `admin-web` 做扎实
- 每做完一步，都要能单独演示和单独验证

## 8. 开发时的注意事项

- 每做完一个任务，就马上回写状态，不要等最后一起补
- 新增接口调用前，先确认后端已有稳定语义，再决定是“接真”还是“明确降级”
- 不要把用户端又做回管理台，能说人话就别露底层术语
- 移动端适配时，优先保关键操作可达，不要先卷动画

## 9. 一句话结论

接下来最正确的做法，不是继续散着补 `user-web` 页面，而是按“阶段 2 收口 -> 先补后端和 `admin-web` 的 `AI配置 + 对话` 前置承载 -> 再回到用户前端 -> 阶段 4 收口”的顺序，把每一步都拆成能独立提交、独立验证、独立回写状态的小任务。
