# 需求文档 - 家庭用户前端产品化

状态：Draft

## 简介

当前项目已经有了 `admin-web`，但它解决的是“开发者怎么快速调试和验证能力”，不是“家庭成员怎么真正使用产品”。

这两者不是一个东西。

真正的用户前端，应该围绕家庭成员的日常任务组织：

- 先看家里现在怎么样
- 再管理家庭、房间、成员和设备
- 再和 AI 助手持续对话
- 再维护家庭记忆
- 最后在设置里完成主题、语言、HA、AI 等完整配置

本 Spec 聚焦的是一套**可长期演进、视觉上明显区别于管理台、对普通用户足够易用**的家庭产品前端。

---

## 范围边界

本 Spec 包含：

- 新建独立的用户前端应用 `apps/user-web`
- 用户前端的全局布局、导航、响应式和家庭上下文切换
- 首页 / 仪表盘页面
- 家庭、房间、成员、关系管理页面
- 设备、HA 对接、AI 配置等用户可理解的配置页面
- AI 助手页面：多会话管理、连续对话管理、快捷动作
- 家庭记忆管理页面：搜索、查看、纠错、失效标记、可见范围
- 主题配置、国际化、长辈友好模式、空态和错误态规范
- 用户前端与 `admin-web` 的职责边界和信息暴露边界

本 Spec 不包含：

- 继续把 `admin-web` 改成双用途应用
- AI 供应商底层路由、密钥、调试日志直接暴露给普通用户
- 完整后端实现细节
- 自由编排器、可视化规则引擎、复杂 BI 大屏
- 真实移动端 App 原生实现

---

## 术语表

- **用户前端（User Web）**：给家庭成员日常使用的产品前端，不承担开发调试职责。
- **家庭工作台（Household Dashboard）**：首页聚合视图，用来快速看当前家庭状态和最近事件。
- **易用配置（User-friendly Configuration）**：保留完整配置能力，但使用普通用户能理解的语言、默认值和引导，不直接暴露底层技术对象。
- **助手会话（Assistant Session）**：围绕一个家庭话题持续进行的 AI 对话容器。
- **家庭记忆（Family Memory）**：事实、事件、偏好、关系等家庭长期信息及其来源和可见范围。
- **主题 token（Theme Token）**：控制颜色、字号、间距、圆角、阴影等视觉变量的统一设计令牌。

## 需求

### 需求 1：用户前端独立定位与入口边界

**用户故事：** 作为产品用户，我希望有一个真正面向家庭使用的前端，而不是开发调试台，这样我进入系统后看到的就是生活任务，而不是技术模块。

#### 验收标准

1. WHEN 用户进入产品前端 THEN System SHALL 提供独立于 `admin-web` 的应用入口、路由和页面壳子。
2. WHEN 用户在产品前端浏览页面 THEN System SHALL 以“首页、家庭、助手、记忆、设置”这类用户任务组织信息，而不是以开发模块名组织信息。
3. WHEN 某项信息只对开发联调有意义 THEN System SHALL 保留在 `admin-web`，不得默认暴露到用户前端。
4. WHEN 用户需要完成家庭配置 THEN System SHALL 在用户前端提供完整配置入口，但默认表达必须是用户语言而不是底层技术术语。

### 需求 2：首页 / 仪表盘

**用户故事：** 作为家庭成员，我希望打开系统就能快速看到当前家庭、房间、成员、设备和最近事件状态，这样我不用到处点来点去。

#### 验收标准

1. WHEN 用户进入首页 THEN System SHALL 展示家庭摘要、房间状态、成员状态、设备状态和最近事件流。
2. WHEN 用户查看首页卡片 THEN System SHALL 优先展示对当前决策最有价值的信息，例如谁在家、哪些房间活跃、是否存在异常提醒或场景执行。
3. WHEN 某类数据暂时不可用 THEN System SHALL 提供保守降级、空态解释或重试入口，而不是整页崩掉。
4. WHEN 用户希望快速执行高频操作 THEN System SHALL 提供明显的快捷入口，例如进入助手、查看记忆、触发家庭服务或进入配置页面。

### 需求 3：家庭、房间、成员与关系管理

**用户故事：** 作为家庭管理员，我希望能在用户前端直接管理家庭结构，这样我不需要为了日常维护再进入开发调试台。

#### 验收标准

1. WHEN 用户进入家庭模块 THEN System SHALL 提供家庭概览、房间管理、成员管理和关系管理入口。
2. WHEN 用户编辑家庭、房间、成员或关系信息 THEN System SHALL 提供清晰表单、默认值、校验说明和保存反馈。
3. WHEN 页面涉及家庭关系 THEN System SHALL 优先使用卡片、图谱、分组等更直观的表达，而不是默认堆技术表格。
4. WHEN 某项操作可能影响其他成员权限或可见范围 THEN System SHALL 在提交前明确提示影响范围。

### 需求 4：设置中的设备、HA 对接与家庭服务配置

**用户故事：** 作为家庭管理员，我希望在设置里完成设备归属、HA 连接状态和家庭服务相关配置，这样我能在一个地方把家庭系统用起来。

#### 验收标准

1. WHEN 用户进入设置下的设备与集成页面 THEN System SHALL 展示设备在线状态、房间归属、同步状态和 HA 集成健康情况。
2. WHEN 用户在设置里配置 HA 对接 THEN System SHALL 提供用户可理解的连接状态、同步范围和问题提示，而不是只暴露原始技术字段。
3. WHEN 用户在设置里配置家庭服务相关能力 THEN System SHALL 允许在用户前端完成基础服务配置，例如默认提醒方式、广播范围、场景开关和家庭模式偏好。
4. WHEN 某项底层供应商细节只用于开发排障 THEN System SHALL 收口到 `admin-web`，用户前端只展示必要的健康状态与可操作配置。

### 需求 5：设置里的 AI 配置必须完整但易用

**用户故事：** 作为家庭管理员，我希望用户前端在设置里也有完整 AI 配置页面，但我不想看到开发者才看得懂的一堆模型术语。

#### 验收标准

1. WHEN 用户进入设置下的 AI 配置页面 THEN System SHALL 支持配置默认回复风格、输出语言、记忆使用策略、隐私级别、家庭助手称呼等产品级设置。
2. WHEN 用户需要调整 AI 能力范围 THEN System SHALL 提供明确开关和解释，例如是否允许生成提醒建议、是否允许引用家庭记忆、是否允许推荐场景。
3. WHEN 某项 AI 设置存在风险或影响隐私 THEN System SHALL 在页面中给出明确解释和保守默认值。
4. WHEN 用户需要更底层的模型供应商配置或路由调试 THEN System SHALL 指向 `admin-web`，而不是把原始 provider 配置直接混进用户页面。

### 需求 6：AI 助手多会话与连续对话

**用户故事：** 作为家庭成员，我希望像用一个真正的助手一样和系统连续对话，并能管理多个家庭话题，这样助手才有用，不是一次性问答机。

#### 验收标准

1. WHEN 用户进入助手页面 THEN System SHALL 支持多会话列表、会话搜索、会话标题和最近活跃排序。
2. WHEN 用户在同一会话中连续提问 THEN System SHALL 保留上下文，并明确显示当前会话使用的家庭范围、记忆范围和降级状态。
3. WHEN 助手回答后存在后续动作 THEN System SHALL 提供快捷操作，例如继续追问、转为提醒、转为记忆、查看相关家庭状态。
4. WHEN AI 主链路失败 THEN System SHALL 返回保守可解释结果或模板降级结果，而不是给用户 500。

### 需求 7：家庭记忆管理

**用户故事：** 作为家庭管理员或家庭成员，我希望能查看和维护家庭记忆，这样系统长期记住的事情是可控的，不是黑盒。

#### 验收标准

1. WHEN 用户进入家庭记忆页面 THEN System SHALL 提供按事实、事件、偏好、关系等维度浏览和搜索的能力。
2. WHEN 用户查看一条记忆 THEN System SHALL 展示记忆内容、来源、更新时间、可见范围和当前状态。
3. WHEN 用户发现记忆错误、过期或重复 THEN System SHALL 允许纠错、标记失效、合并或删除，并保留追踪信息。
4. WHEN 某条记忆涉及敏感成员或隐私边界 THEN System SHALL 按可见范围显示或隐藏，不得越权暴露。

### 需求 8：主题、国际化与长辈友好模式

**用户故事：** 作为不同年龄和语言背景的家庭成员，我希望前端可以切换主题和语言，并具备更易读的展示方式，这样产品才真能给全家人用。

#### 验收标准

1. WHEN 用户首次进入产品前端 THEN System SHALL 支持默认主题和默认语言配置，并允许在设置中切换。
2. WHEN 用户切换主题 THEN System SHALL 保持页面结构稳定，只改变视觉 token，而不是导致布局错乱。
3. WHEN 用户切换语言 THEN System SHALL 同步处理导航、空态、表单、时间日期和系统提示文案。
4. WHEN 用户启用长辈友好模式 THEN System SHALL 提供更大的字号、更高对比和更低信息密度。

### 需求 9：空态、错误态、引导与恢复

**用户故事：** 作为用户，我希望产品在数据缺失、接口失败或第一次使用时也能给我清楚引导，而不是让我面对一堆空白。

#### 验收标准

1. WHEN 用户第一次进入某页面且还没有数据 THEN System SHALL 提供明确空态、下一步引导和示例内容。
2. WHEN 某个接口失败但页面主框架仍可展示 THEN System SHALL 局部降级并保留其他已成功数据。
3. WHEN 用户执行保存、删除、切换等关键操作 THEN System SHALL 提供即时反馈、回滚提示或重试入口。
4. WHEN 用户因权限、网络或外部依赖问题无法继续 THEN System SHALL 给出人话说明，而不是仅显示技术错误码。

### 需求 10：可扩展的前端架构

**用户故事：** 作为开发团队，我希望这套用户前端可以长期扩展，而不是做成第二个越来越难维护的管理台。

#### 验收标准

1. WHEN 新增页面或主题能力 THEN System SHALL 能在不重写整个应用壳子的前提下扩展。
2. WHEN 需要复用基础组件、接口类型或国际化能力 THEN System SHALL 通过共享层复用，而不是复制粘贴到多个应用。
3. WHEN 页面需要聚合多个后端能力 THEN System SHALL 优先通过稳定视图模型或聚合接口收敛，而不是让页面自己拼一堆散接口。
4. WHEN 开发团队排查问题 THEN System SHALL 能通过明确的模块边界、日志和状态切片快速定位问题。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 用户首次进入首页或助手页面 THEN System SHALL 在首屏可见区域优先渲染关键骨架与核心信息。
2. WHEN 页面包含多个聚合卡片 THEN System SHALL 支持并发加载、分区刷新和局部骨架，而不是整页等待最慢请求。

### 非功能需求 2：可靠性

1. WHEN 某个聚合接口或外部集成失败 THEN System SHALL 允许局部降级，不得导致整个用户前端主流程不可用。
2. WHEN 用户刷新页面或重新登录 THEN System SHALL 恢复当前家庭上下文、主题偏好、语言偏好和最近会话入口。

### 非功能需求 3：可维护性

1. WHEN 新增用户前端页面 THEN System SHALL 遵循统一路由、状态、组件和文案组织方式。
2. WHEN 需要和 `admin-web` 共享基础能力 THEN System SHALL 优先共享设计 token、API client 和类型，不共享产品壳子。

### 非功能需求 4：易用性与可访问性

1. WHEN 用户使用触屏、平板或桌面设备 THEN System SHALL 提供一致的核心任务路径和可点击区域。
2. WHEN 用户需要更高可读性或辅助功能 THEN System SHALL 支持键盘导航、对比度要求和可读性优化。

## 成功定义

- `admin-web` 与 `user-web` 的职责边界明确，团队不再把用户需求继续堆到调试台里。
- 首期用户前端能稳定覆盖首页、家庭、助手、记忆、设置五大入口。
- 用户前端具备完整配置页面，但普通用户不需要理解底层 provider、日志和调试概念。
- 主题和国际化从第一阶段就进入实现路径，而不是延期成技术债。
