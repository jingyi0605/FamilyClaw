# 设计文档 - 家庭用户前端产品化

状态：Draft

## 1. 概述

### 1.1 Spec 定位

`admin-web` 现在承担的是“把能力调通、把接口验掉、把异常查清”的职责。它是控制台，不是产品。

`Spec 002.2` 要解决的，是把项目现有的家庭、房间、成员、设备、问答、提醒、场景和记忆能力，收敛成一套真正可被家庭成员长期使用的前端产品。

这个 Spec 关注的是：

- 用户前端的信息架构
- 页面与视觉系统
- 主题与国际化骨架
- 助手与记忆的产品交互
- 配置页面的“完整能力 + 易用表达”

它明确不关注：

- 替代 `admin-web`
- 供应商级 AI 调试界面
- 原始审计日志页面
- 复杂运维配置入口

### 1.2 范围边界

本 Spec 包含四块核心交付：

1. **产品壳子**：`apps/user-web`、导航、路由、主题、国际化、用户偏好持久化。
2. **业务主页面**：首页、家庭、助手、记忆、设置。
3. **完整配置页面**：家庭入口保留家庭、房间、成员、关系；设置入口保留设备、HA、AI 配置、主题和语言，完整能力都在用户前端保留。
4. **产品化收口**：空态、错误态、骨架屏、响应式、长辈友好模式。

本 Spec 不包含：

- 原生移动端
- 可视化编排器
- 复杂报表系统
- 供应商级 AI 配置调试页面

### 1.3 分阶段交付策略

#### 第一阶段：先把产品骨架搭对

- 新建 `apps/user-web`
- 建立路由、布局、主题 token、i18n 和家庭上下文切换
- 明确和 `admin-web` 的边界

#### 第二阶段：把首页和家庭配置做成真入口

- 首页 / 仪表盘
- 家庭、房间、成员、关系
- 设置中的设备、HA、AI 易用配置

#### 第三阶段：把助手和记忆做成产品中心

- AI 助手多会话与连续对话
- 家庭记忆管理
- 助手到提醒、场景、记忆的快捷动作

#### 第四阶段：做产品化收口

- 空态、错误态、动效和骨架屏
- 响应式、无障碍、长辈友好模式
- 联调说明和验收记录

## 2. 架构设计

### 2.1 产品信息架构

用户前端不按后台模块组织，而按用户任务组织。

顶级导航固定为五个入口：

1. **首页**
2. **家庭**
3. **助手**
4. **记忆**
5. **设置**

这五个入口下再承载详细页面：

- `首页`
  - 家庭摘要
  - 房间状态
  - 成员状态
  - 最近事件
  - 快捷动作
- `家庭`
  - 家庭概览
  - 房间
  - 成员
  - 关系
- `助手`
  - 会话列表
  - 会话详情
  - 快捷建议
- `记忆`
  - 记忆列表
  - 记忆详情
  - 纠错与状态管理
- `设置`
  - 外观主题
  - AI 配置
  - 语言与地区
  - 通知偏好
  - 长辈友好模式
  - 设备与集成

### 2.2 应用边界与目录建议

建议新增：

- `apps/user-web/`

建议同步抽共享层：

- `packages/ui/`：基础组件、theme token、图标、布局原语
- `packages/api-client/`：接口封装、请求器、类型定义
- `packages/i18n/`：语言包、格式化函数、语言切换工具

边界原则：

- `admin-web` 保留开发与调试视角
- `user-web` 只保留用户视角
- 共享基础能力，不共享产品壳子

### 2.3 核心数据流

#### §2.3.1 首页聚合流

1. 用户进入首页。
2. 页面读取当前家庭上下文、主题、语言和最近访问状态。
3. 页面并发拉取家庭摘要、房间状态、成员状态、设备摘要和最近事件。
4. 页面将多接口结果收敛为 `DashboardOverview` 视图模型。
5. 某个区块失败时，仅该区块降级，不影响其他区块渲染。

#### §2.3.2 家庭配置编辑流

1. 用户进入家庭模块具体页面。
2. 页面加载主数据和字典数据。
3. 表单先在前端做即时校验，再提交保存。
4. 保存成功后刷新局部视图模型，并提示影响范围。
5. 保存失败时保留草稿和错误提示，不清空用户输入。

#### §2.3.3 助手会话流

1. 用户进入助手页面，先看到会话列表和推荐问题。
2. 用户新建或切换会话。
3. 会话消息提交后，带上当前家庭上下文、会话上下文和必要的记忆引用策略。
4. AI 响应返回后，页面渲染回答、解释、降级提示和快捷动作。
5. 用户可继续追问、改写、转为提醒、写入记忆或跳转相关页面。

#### §2.3.4 记忆检索与纠错流

1. 用户进入记忆页面。
2. 页面按筛选条件拉取记忆列表，并支持搜索和分类切换。
3. 用户查看某条记忆详情，读取来源、状态和可见范围。
4. 用户可执行纠错、合并、失效标记或删除。
5. 成功后刷新当前条目和列表摘要，保留审计追踪入口。

#### §2.3.5 主题与语言切换流

1. 用户在设置中切换主题或语言。
2. 变更先更新本地偏好状态，再同步服务端偏好存储。
3. 页面通过 token 和 i18n 重新渲染，不重建整个应用状态。
4. 刷新后仍恢复用户选择。

## 3. 页面与组件设计

### 3.1 用户前端壳子

路由建议：

- `/`
- `/family`
- `/family/rooms`
- `/family/members`
- `/family/relationships`
- `/assistant`
- `/assistant/:sessionId`
- `/memories`
- `/memories/:memoryId`
- `/settings/appearance`
- `/settings/ai`
- `/settings/integrations`
- `/settings/language`
- `/settings/notifications`
- `/settings/accessibility`

壳子设计要求：

- 左侧或底部主导航，随屏宽自适应
- 顶部保留当前家庭、全局搜索、快捷入口、用户菜单
- 页面标题区负责说明“我现在在哪、能做什么”
- 全局通知区统一承载保存成功、失败、降级和同步状态

### 3.2 首页 / 仪表盘

页面结构：

1. **欢迎区**：家庭名称、时间、模式、今日摘要
2. **关键指标卡**：在家人数、活跃房间、设备在线率、异常数
3. **房间状态带**：每个房间的占用、温度、灯光、异常
4. **成员状态区**：当前在家、休息、外出、需关注成员
5. **最近事件流**：提醒、场景执行、问答结果、设备变化
6. **快捷动作区**：进入助手、查看记忆、家庭服务入口

设计要求：

- 主视觉偏生活服务，而不是监控大屏
- 卡片信息层级明确，避免一屏塞满表格
- 关键异常必须有可点击的下一步

### 3.3 家庭概览、房间、成员、关系

#### 家庭概览

- 家庭名称、时区、默认语言、家庭模式、隐私模式
- 家庭服务开关摘要
- 当前风险和建议动作

#### 房间管理

- 房间卡片视图为主，列表视图为辅
- 显示房间类型、设备数、活跃状态、敏感级别
- 支持编辑房间策略和默认行为

#### 成员管理

- 成员卡片为主，展示头像、角色、状态、偏好摘要
- 编辑时按“基本信息 / 偏好 / 权限边界”分组

#### 关系管理

- 图谱表达为主，表单编辑为辅
- 明确照护、监护、可见范围、代办边界

### 3.4 设置中的设备、HA 与 AI 易用配置

#### 设备与集成页

- 设备列表按房间分组
- 显示在线状态、最近同步时间、房间归属、异常
- HA 页面显示连接状态、同步范围、上次同步结果、常见问题提示

#### AI 配置页

用户前端保留完整能力，但必须是产品语言：

- 助手名称
- 默认回复语气
- 输出语言
- 是否允许引用家庭记忆
- 是否主动给提醒建议
- 是否推荐场景动作
- 隐私级别
- 长回答 / 短回答偏好

下面这些内容不放到用户端主页面：

- provider 档案
- 主备路由细节
- 原始模型名称
- preview 调试调用
- token 与成本细节

这类内容继续放在 `admin-web`。

这两个页面都归到 `设置` 入口下，不归到 `家庭`。原因很简单：

- `家庭` 负责家庭结构和成员关系
- `设置` 负责系统表现、集成和智能行为

### 3.5 助手页面

页面分三栏或双栏：

1. **左侧会话列表**：搜索、新建、置顶、最近活跃
2. **中间对话区**：消息流、建议问题、引用卡片、降级提示
3. **右侧上下文侧栏**：当前家庭、可用记忆、相关事件、快捷动作

消息级动作：

- 继续追问
- 重新生成
- 转为提醒
- 写入记忆
- 查看相关页面

状态要求：

- 明确标记当前回答是否使用了记忆
- 明确标记当前回答是否进入降级
- 明确展示会话范围属于哪个家庭

### 3.6 记忆页面

页面结构：

1. 顶部搜索与筛选
2. 左侧分类导航：事实、事件、偏好、关系
3. 中部列表 / 时间轴
4. 右侧详情抽屉

单条记忆必须展示：

- 内容
- 类型
- 来源
- 可见范围
- 更新时间
- 当前状态

可执行动作：

- 编辑
- 纠错
- 标记失效
- 合并重复
- 删除

### 3.7 设置页面

设置不应该是垃圾桶，但也不能缺。

建议保留：

- 外观主题
- AI 配置
- 语言与地区
- 通知偏好
- 长辈友好模式
- 设备与集成
- 家庭默认显示偏好

设计要求：

- 每项设置都要说明“改了会发生什么”
- 高风险设置必须有二次确认或保守默认值
- 不允许把开发配置和用户配置混在一起

## 5. 正确性属性与业务不变量

### 5.0 前后台边界不变量

1. 用户前端不默认暴露只对开发者有意义的底层配置和调试信息。
2. `admin-web` 与 `user-web` 可以共享底层能力，但产品任务路径必须分离。

### 5.1 家庭上下文不变量

1. 所有页面都必须明确知道当前属于哪个家庭。
2. 跨家庭切换后，页面数据、会话和记忆列表必须同步刷新或明确提示。

### 5.2 易用配置不变量

1. 用户配置页面必须有默认值、说明文案和错误反馈。
2. 高风险设置不得默认开启。
3. 技术术语如果保留，第一次出现必须给出人话解释。

### 5.3 会话连续性不变量

1. 同一会话中的上下文必须稳定。
2. 回答是否降级、是否引用记忆必须可见。

### 5.4 记忆可追溯不变量

1. 记忆内容必须能追到来源或最少追到创建方式。
2. 记忆的可见范围必须始终受控。

### 5.5 主题与国际化不变量

1. 主题切换不改变信息结构。
2. 国际化不允许只翻一半，日期、时间、数字和空态文案都要统一处理。

## 6. 错误处理与降级

### 6.0 首页错误处理

- 某个卡片失败时显示局部错误卡和重试按钮
- 首页主框架始终可进入
- 最近事件和关键摘要优先保底

### 6.1 配置页面错误处理

- 保存失败保留草稿
- 字段错误就地提示
- 高风险操作提供撤销或确认提示

### 6.2 助手错误处理

- 主模型失败时展示保守回答或模板降级
- 明确显示 `degraded` 或等价降级标记
- 不把原始 500 直接扔给用户

### 6.3 记忆页面错误处理

- 列表失败与详情失败分开处理
- 纠错失败时保留用户编辑内容
- 删除或失效标记失败时允许重试

### 6.4 弱网与慢接口策略

- 骨架屏优先于整页空白
- 支持分区加载和延迟加载
- 会话列表、当前会话和侧栏上下文独立刷新

## 8. 风险与延期项

### 8.1 当前明确风险

1. 如果继续复用 `admin-web` 的页面思维，用户端会长成另一套管理台。
2. 如果主题和国际化延期，后面重构成本会非常高。
3. 助手与记忆需要更稳定的聚合接口，否则页面层会变脏。
4. HA 和 AI 配置如果表达不好，用户会被技术细节劝退。

### 8.2 当前明确延期

1. 原生移动端适配延后。
2. 高级视觉动效和复杂插画系统延后。
3. 完整多租户品牌皮肤系统延后。
4. 大规模家庭时间轴分析和 BI 报表延后。
