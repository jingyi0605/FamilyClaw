# Spec 004.4 - 用户前端插件管理与插件市场

状态：Draft

## 这份 Spec 解决什么问题

后端插件系统已经有了，但用户现在还看不到插件，也无法自己管理插件状态。

这会带来两个直接问题：

- 用户不知道系统里到底有哪些插件、哪些正在启用、哪些有风险
- 就算后面有第三方插件注册表，用户也没有地方去浏览和区分来源

这份 Spec 就是补用户看得见、用得上的前端层：把插件管理和插件市场界面做出来。

## 这次要做到什么

- 做一个插件列表页，能看到已安装插件和状态
- 做启用/禁用、详情查看这些基础管理能力
- 在插件管理里加入插件市场入口
- 支持展示内置、官方、第三方插件来源
- 支持读取官方注册表，也支持配置第三方注册表
- 把插件后台任务状态、失败提示、待确认动作带到前端，而不是继续假设“点击就同步完成”

## 这次明确不做什么

- 不做一键远程安装任意仓库代码
- 不做商业计费
- 不做复杂推荐系统

## 为什么现在做

因为插件系统如果只有后端底座，没有前端入口，用户根本感知不到这套能力。

先把插件管理和插件市场做出来，至少能解决三件事：

- 用户能看见系统里已经有什么插件
- 用户能管理插件启停和查看风险说明
- 用户能浏览市场来源，为后续官方和第三方扩展留出入口

现在还要再加一句：

- 插件执行已经是后台任务模型，所以前端也必须让用户看到“任务正在跑、失败了、等确认”这些状态，而不是只做静态列表页

## 主要文档

- `requirements.md`：说清楚插件管理页和插件市场到底做什么
- `design.md`：说清楚页面结构、数据来源、状态和交互边界
- `tasks.md`：说清楚按什么顺序做页面、接口对接和验收
- `docs/README.md`：说明后续补充页面稿、接口联调和验收截图放哪里
- `../004.2.2-插件禁用能力统一与停用收口/`：补清楚插件管理页里的启停语义，尤其是内置插件的家庭级禁用和最终 `enabled` 状态来源

## 关于前端插件启停，这里补一句别再写错

前端插件管理页后续必须遵守两条死规矩：

1. 内置插件不能再被硬编码成“不支持禁用”。
2. 页面展示的 `enabled` 必须以后端返回的最终状态为准，不能自己拿挂载记录推断。

这些规则的正式说明统一看 `../004.2.2-插件禁用能力统一与停用收口/`。
