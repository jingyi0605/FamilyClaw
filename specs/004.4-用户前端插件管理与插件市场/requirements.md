# 需求文档 - 用户前端插件管理与插件市场

状态：Draft

## 简介

用户前端现在还没有插件管理入口，这意味着插件系统虽然能运行，但用户不知道：

- 家里当前启用了哪些插件
- 哪些插件来自官方，哪些来自第三方
- 某个插件会读什么数据、申请什么权限、风险高不高
- 市场上还有哪些可接入的插件

这份 Spec 面向两个主要角色：

- **家庭管理员或普通成员**：查看和管理插件
- **产品和前端开发者**：把后端插件能力和注册表能力变成用户可操作的界面

成功后的核心收益是：用户开始真正拥有插件系统，而不是只有后台代码。

## 术语表

- **System**：FamilyClaw 用户前端中的插件管理和插件市场能力
- **已安装插件**：已经被系统识别并注册到当前家庭环境中的插件
- **插件市场**：展示官方和第三方插件元数据的浏览界面，不直接执行第三方代码
- **注册表源**：市场读取插件元数据的来源地址，可以是官方注册表，也可以是第三方注册表
- **来源标识**：区分插件是内置、官方还是第三方

## 范围说明

### In Scope

- 插件列表页
- 启用 / 禁用
- 插件详情页
- 插件任务状态展示
- 插件任务失败提示和待响应提示
- 来源标识展示
- 注册表拉取与展示
- 多注册表切换或配置

### Out of Scope

- 一键远程安装任意仓库代码
- 商业计费
- 复杂推荐系统

## 需求

### 需求 1：用户可以查看当前已安装插件

**用户故事：** 作为用户，我希望能看到当前系统里有哪些插件，以便我知道系统到底接了哪些能力。

#### 验收标准

1. WHEN 用户打开插件管理页 THEN System SHALL 展示已安装插件列表和启用状态。
2. WHEN 插件有风险等级和权限声明 THEN System SHALL 在列表或详情中清楚展示。
3. WHEN 当前没有已安装插件 THEN System SHALL 提供明确空态，而不是空白页面。

### 需求 2：用户可以管理插件启用和禁用

**用户故事：** 作为家庭管理员，我希望能启用或禁用插件，以便控制哪些能力当前生效。

#### 验收标准

1. WHEN 用户在管理页执行启用或禁用 THEN System SHALL 调用后端对应接口并刷新状态。
2. WHEN 操作成功 THEN System SHALL 给出明确反馈。
3. WHEN 操作失败 THEN System SHALL 给出可读错误信息，而不是静默失败。

### 需求 2A：用户可以看到插件后台任务状态

**用户故事：** 作为用户，我希望知道某次插件执行是在排队、执行中、失败还是等我确认，这样我才知道系统现在到底卡在哪。

#### 验收标准

1. WHEN 插件执行已经改成后台任务 THEN System SHALL 在前端展示最近任务状态，而不是假装操作已经同步完成。
2. WHEN 任务失败 THEN System SHALL 展示可读失败信息或失败摘要。
3. WHEN 任务进入等待响应状态 THEN System SHALL 明确提示用户需要确认、取消或补充输入。

### 需求 3：用户可以查看插件详情和来源

**用户故事：** 作为用户，我希望能查看插件详情和来源，以便判断这个插件是不是可信、能做什么。

#### 验收标准

1. WHEN 用户进入插件详情页 THEN System SHALL 展示插件名称、描述、类型、权限、风险等级和来源标识。
2. WHEN 插件来源是第三方 THEN System SHALL 清楚标记，不把它伪装成官方插件。
3. WHEN 插件有外部文档或仓库地址 THEN System SHALL 提供可访问入口。

### 需求 4：用户可以浏览插件市场

**用户故事：** 作为用户，我希望能在前端看到插件市场，以便了解还有哪些官方或第三方插件可用。

#### 验收标准

1. WHEN 用户打开插件市场 THEN System SHALL 展示从注册表读取到的插件条目。
2. WHEN 插件来自不同来源 THEN System SHALL 展示来源标识和注册表来源。
3. WHEN 注册表拉取失败 THEN System SHALL 给出降级提示，而不是让整个页面失效。

### 需求 5：系统支持多个注册表来源

**用户故事：** 作为高级用户或管理员，我希望能切换或配置多个注册表来源，以便浏览官方和第三方插件目录。

#### 验收标准

1. WHEN 系统配置多个注册表源 THEN System SHALL 能按来源展示或筛选市场数据。
2. WHEN 某个第三方注册表字段不完整或不可访问 THEN System SHALL 标记风险、忽略该源或给出错误提示。
3. WHEN 用户查看市场插件 THEN System SHALL 能知道插件来自哪个注册表源。

## 非功能需求

### 非功能需求 1：可理解性

1. WHEN 用户第一次进入插件管理页 THEN System SHALL 让用户一眼看懂“已安装”和“市场可浏览”是两件事。
2. WHEN 页面展示权限和风险等级 THEN System SHALL 用直白文案，而不是只给内部字段名。
3. WHEN 页面展示任务状态 THEN System SHALL 让用户看懂“排队中 / 执行中 / 失败 / 等待确认”这些状态，不要求用户理解后台实现细节。

### 非功能需求 2：稳定性

1. WHEN 市场注册表拉取失败 THEN System SHALL 保证已安装插件管理页仍可使用。
2. WHEN 某个第三方注册表异常 THEN System SHALL 允许局部降级，而不是拖垮全部市场页面。

### 非功能需求 3：可扩展性

1. WHEN 后续增加更多插件分类或来源 THEN System SHALL 在现有页面结构上扩展，而不是重做信息架构。
2. WHEN 后续增加安装流程 THEN System SHALL 仍能复用当前市场列表和详情结构。

## 成功定义

- 用户能在前端清楚看到已安装插件、插件详情和来源
- 用户能管理插件启停，并看到明确结果反馈
- 用户能看到插件后台任务状态、失败和待确认提示
- 插件市场能读取官方和第三方注册表，并在界面上清楚区分来源
