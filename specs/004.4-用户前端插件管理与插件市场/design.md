# 设计文档 - 用户前端插件管理与插件市场

状态：Draft

## 1. 设计目标

这份设计文档现在明确收三件事：

1. 已安装插件怎么展示和管理
2. 插件详情页怎么承接统一配置入口
3. 插件市场怎么展示来源，而不是把所有插件混成一锅

## 2. 页面结构

### 2.1 插件列表页

负责展示：

- 插件名
- 类型
- 风险等级
- 来源
- 最终启用状态
- 最近任务状态摘要

注意：

- `enabled` 一律以后端返回的最终状态为准
- 不允许前端再按 builtin / mount 自己推导

### 2.2 插件详情抽屉

`PluginDetailDrawer` 现在不只是看说明文案，它还要承担统一配置入口。

抽屉里的行为分两类：

- 基础信息：来源、权限、风险、说明
- 配置入口：读取可配置作用域、打开对应表单、保存配置

如果插件没有配置协议：

- 抽屉仍然照常可打开
- 只是不会出现正式配置表单

### 2.3 通用动态表单

前端不再按插件 id 写字段常量。

统一做法是：

1. 后端返回正式配置作用域
2. 后端返回某个作用域的表单协议
3. 前端通用 renderer 按协议渲染
4. 前端按统一保存接口提交

配置协议正文不在这里重复定义，统一引用：

- `../004.2.3-插件配置协议与动态表单/`

## 3. `channel` 首批迁移约束

`channel` 是第一批迁移对象，所以这里必须写死。

前端不能再把这些东西留在 `SettingsChannelAccessPage` 里当事实来源：

- 平台字段常量
- 平台表单布局常量
- secret 字段处理特例

现在应该改成：

- 通道插件 manifest 提供 `channel_account` 作用域配置协议
- 页面通过统一插件配置 API 读取和保存
- 动态表单 renderer 负责渲染

## 4. 市场页边界

市场页只负责：

- 看有哪些插件
- 看插件来自哪里
- 看风险和摘要

它不负责：

- 复制一份插件配置字段定义
- 替代插件详情页去维护配置逻辑

## 5. 与 004.2.3 的关系

这份 Spec 以后只保留前端消费视角，不再重复配置协议正文。

如果内容涉及这些关键词，统一引用 `004.2.3`：

- `config_specs`
- `config_schema`
- `ui_schema`
- `clear_secret_fields`
- secret 不回显
- 动态表单字段类型和 widget 类型

## 6. 兼容性要求

前端必须同时满足两件事：

1. 有正式配置协议的新插件，能走统一配置入口
2. 没有配置协议的旧插件，行为不回退

这条规则同样适用于：

- 插件详情抽屉
- 插件列表页
- 通道账号配置页
