# 设计文档 - 用户前端插件管理与插件市场

状态：Draft

## 1. 概述

### 1.1 目标

- 把后端插件系统变成用户可见、可管理的前端能力
- 把注册表数据变成可浏览的插件市场页面
- 把来源、权限、风险等级这些关键信息清楚展示出来

### 1.2 覆盖需求

- `requirements.md` 需求 1
- `requirements.md` 需求 2
- `requirements.md` 需求 3
- `requirements.md` 需求 4
- `requirements.md` 需求 5

### 1.3 技术约束

- 前端：需要复用项目现有前端技术栈和设计体系
- 后端：优先复用现有插件注册、启停和市场注册表相关接口
- 数据来源：已安装插件来自系统后端；市场插件来自注册表接口
- 安全边界：第一版不做远程代码安装，不直接执行外部仓库内容
- 交互边界：插件市场以浏览和来源识别为主，不承诺第一版支持一键安装

## 2. 架构

### 2.1 系统结构

前端这一块分成两个并列区域：

1. **插件管理**
   - 面向当前家庭已安装插件
   - 解决“看当前有什么、能不能启停、风险是什么”
2. **插件市场**
   - 面向注册表中的可浏览插件
   - 解决“外面还有什么、来自哪里、值不值得看”

两者共用一套基础信息：

- 来源标识
- 权限与风险展示
- 插件详情信息结构

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 插件列表页 | 展示已安装插件和状态 | 已安装插件接口 | 列表、空态、状态标签 |
| 插件详情页 | 展示单个插件详细信息 | 插件详情接口 / 注册表条目 | 详情信息、来源信息 |
| 启停操作层 | 发起启用或禁用 | 用户操作、后端接口 | 成功或失败反馈 |
| 插件市场页 | 展示注册表条目 | 官方/第三方注册表接口 | 市场列表、来源筛选 |
| 注册表配置层 | 管理市场来源 | 系统配置、用户选择 | 当前生效来源列表 |

### 2.3 关键流程

#### 2.3.1 已安装插件管理流程

1. 用户进入插件管理页。
2. 页面请求已安装插件列表。
3. 页面展示状态、来源、风险、权限摘要。
4. 用户可以进入详情或执行启用 / 禁用。

#### 2.3.2 插件市场浏览流程

1. 用户进入插件市场页。
2. 页面请求当前启用的注册表源。
3. 页面拉取市场条目并按来源展示。
4. 用户可以切换来源、查看详情、跳转文档或仓库。

#### 2.3.3 多注册表切换流程

1. 管理员查看当前注册表源配置。
2. 管理员启用或停用某个注册表源。
3. 市场页刷新后按新的来源列表展示数据。

### 3.3 接口契约

覆盖需求：1、2、3、4、5

#### 3.3.1 已安装插件列表接口

- 类型：HTTP
- 路径或标识：`GET /plugins`
- 输入：家庭上下文、分页参数（如需要）
- 输出：已安装插件列表和状态
- 校验：只返回当前家庭可见插件
- 错误：接口失败时页面显示错误提示和重试入口

#### 3.3.2 插件启停接口

- 类型：HTTP
- 路径或标识：`POST /plugins/{plugin_id}/enable`、`POST /plugins/{plugin_id}/disable`
- 输入：插件 id、家庭上下文
- 输出：最新插件状态
- 校验：只能操作当前家庭已注册插件
- 错误：失败时返回明确错误消息

#### 3.3.3 插件市场接口

- 类型：HTTP
- 路径或标识：`GET /plugin-market/plugins`
- 输入：注册表来源筛选、搜索词、分类筛选
- 输出：市场插件列表
- 校验：来源必须是已配置注册表
- 错误：某个来源失败时允许局部降级

#### 3.3.4 注册表来源接口

- 类型：HTTP
- 路径或标识：`GET /plugin-market/registries`、`PATCH /plugin-market/registries`
- 输入：当前来源列表、启用状态、优先级
- 输出：当前生效来源配置
- 校验：来源地址和类型必须合法
- 错误：非法来源配置应被拒绝

## 4. 数据与状态模型

### 4.1 数据关系

- 已安装插件数据来自系统后端插件注册表
- 市场插件数据来自官方或第三方注册表源
- 市场条目和已安装插件通过 `plugin_id` 关联
- 同一个插件在页面上可能同时有两种视角：
  - 已安装状态
  - 市场来源信息

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `loading` | 页面加载中 | 首次进入或刷新 | 成功或失败 |
| `ready` | 数据可展示 | 接口成功 | 再次刷新或切换来源 |
| `empty` | 没有数据 | 返回空列表 | 数据出现 |
| `error` | 当前视图加载失败 | 请求失败 | 用户重试成功 |
| `updating` | 启停或来源配置变更中 | 用户发起操作 | 成功或失败 |

## 5. 错误处理

### 5.1 错误类型

- `插件列表加载失败`：已安装插件接口异常
- `插件启停失败`：状态更新失败
- `市场来源拉取失败`：注册表接口异常
- `第三方来源无效`：返回字段不完整或不可访问
- `详情信息缺失`：市场条目或安装信息不完整

### 5.2 错误响应格式

```json
{
  "detail": "第三方注册表返回字段不完整，已忽略该来源",
  "error_code": "registry_source_invalid",
  "field": "plugin_id",
  "timestamp": "2026-03-13T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：表单或配置页直接阻止提交。
2. 业务规则错误：给出用户可读提示，不静默失败。
3. 外部依赖错误：市场页允许局部降级，不影响已安装插件页。
4. 重试、降级或补偿：支持手动刷新和重新拉取来源。

## 6. 正确性属性

### 6.1 属性 1：用户必须能分清“已安装”和“市场可浏览”

*对于任何* 插件管理和市场页面，系统都应该满足：用户能清楚区分当前已安装插件和只是市场可浏览的插件。

**验证需求：** 需求 1、需求 4

### 6.2 属性 2：第三方来源不能伪装成官方来源

*对于任何* 来自第三方注册表的插件，系统都应该满足：页面明确标记其来源，不把它展示成官方插件。

**验证需求：** 需求 3、需求 5

### 6.3 属性 3：单个来源故障不能拖垮全部页面

*对于任何* 注册表拉取失败场景，系统都应该满足：故障来源可以局部失败，但插件管理主页面仍然可用。

**验证需求：** 需求 4、需求 5

## 8. 风险与待确认项

### 8.1 风险

- 如果市场和已安装插件混在同一层级，用户会看不懂系统当前状态。
- 如果来源标识不够明显，第三方插件很容易被误认成官方插件。
- 如果第一版就做一键远程安装，安全和审核边界会立刻失控。

### 8.2 待确认项

- 前端插件管理入口放在系统设置、能力中心，还是单独一级导航。
- 多注册表来源是全局配置，还是允许家庭级配置。
- 第一版是否需要在市场里显示“已安装 / 未安装”联动状态。
