# 设计文档 - HA 彻底插件化收尾

状态：Draft

## 1. 概述

### 1.1 目标

- 把 HA 从“功能上大体已经插件化”收口成“边界上真正已经插件化”
- 给这次收尾建立一套不会被模糊表述绕过去的硬验收门槛
- 让后续维护者不用再重新判断“哪些是旧逻辑、哪些是正式路径”

### 1.2 覆盖需求

- `requirements.md` 需求 1：统一主口径
- `requirements.md` 需求 2：核心层停止认识 HA 平台细节
- `requirements.md` 需求 3：HA 插件通过正式宿主协议工作
- `requirements.md` 需求 4：前端停止写死 HA 特例
- `requirements.md` 需求 5：迁移期单轨执行
- `requirements.md` 需求 6：证据化验收
- `requirements.md` 需求 7：spec 与实现对齐

### 1.3 技术约束

- 后端：FastAPI、SQLAlchemy、Alembic、插件系统
- 前端：`apps/user-app`、统一 i18n 字典、统一插件与资源模型
- 数据存储：PostgreSQL
- 外部依赖：Home Assistant 插件、宿主插件运行时、已有设备/集成模块
- 关键约束：
  - 不破坏现有用户可见功能
  - 不新增 HA 业务能力
  - 不允许靠“先兼容着”长期保留双轨

## 2. 架构

### 2.1 这次收尾的主从关系

这次不是单独发明一套新系统，而是给已有几份 spec 建立明确主从关系。

| 文档 | 角色 | 这次怎么用 |
| --- | --- | --- |
| `005.1` | 原则 spec | 定义“核心只认协议，平台实现在插件” |
| `005.4` | 主战场 spec | 承担后端、前端、迁移和旧路径清理 |
| `005.6` | 前端局部 spec | 只说明前端迁移，不作为“彻底插件化完成”的证据 |
| `004.8.4` | 收尾总控 spec | 定义本轮完成条件、禁区、证据和任务编排 |

### 2.2 收尾流程

#### 2.2.1 基线冻结

1. 先盘出现有 HA 残留，不再凭感觉说“应该差不多了”
2. 把允许保留的白名单单独列出来
3. 把不允许出现的目录、关键词和依赖路径写死

#### 2.2.2 边界清理

1. 先收核心
2. 再收插件
3. 最后收前端和文档

顺序不能反过来。前端先换皮没有意义，底下还是脏的。

#### 2.2.3 证据验收

1. 跑固定 grep
2. 核对白名单
3. 跑回归和迁移校验
4. 回写 spec 任务状态
5. 输出剩余残留清单

没有这五样，任何“已完成”都不算数。

### 2.3 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| `004.8.4` 收尾 spec | 定义完成口径、禁区、任务和证据 | 现有代码、`005.1`、`005.4`、`005.6` | 收尾任务、验收规则、白名单规范 |
| 核心层清理任务 | 清理 HA 专名逻辑和平台特判 | 核心目录现存残留 | 只认统一协议的核心 |
| 插件层清理任务 | 清理对核心内部实现的偷读路径 | HA 插件目录现存残留 | 只通过宿主 contract 工作的插件 |
| 前端层清理任务 | 清理产品顶层 HA 特例 | `user-app` 残留文案、类型和流程 | 统一插件与资源语义 |
| 验收任务 | 生成并保管证据 | grep、测试、迁移结果 | 可复核的交付证据包 |

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3、4、5、6、7

- `边界残留审计`：发现核心越界、插件越界、前端越界和旧路径遗留
- `白名单管理`：显式登记允许暂留的残留点及其删除条件
- `任务关卡`：控制哪些任务能进 `DONE`
- `证据归档`：保存 grep、回归、迁移校验和剩余残留说明

### 3.2 数据结构

覆盖需求：1、5、6、7

#### 3.2.1 `BoundaryFinding`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `id` | string | 是 | 残留项唯一标识 | 在本次收尾中稳定不变 |
| `category` | enum | 是 | `core` / `plugin` / `frontend` / `migration` / `spec` | 只能取固定枚举 |
| `path` | string | 是 | 残留所在文件 | 必须可定位到具体文件 |
| `evidence` | string | 是 | 发现依据，如 grep 命中、代码位置 | 不能写成空话 |
| `owner_task` | string | 是 | 负责清理它的任务编号 | 必须映射到 `tasks.md` |
| `status` | enum | 是 | `open` / `whitelisted` / `closed` | 只能取固定枚举 |

#### 3.2.2 `WhitelistEntry`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `scope` | string | 是 | 白名单适用目录或文件 | 必须精确到目录、文件或调用点 |
| `reason` | string | 是 | 为什么暂时保留 | 不能写“兼容需要”这种空话 |
| `expiry_condition` | string | 是 | 何时必须删除 | 必须可执行、可验证 |
| `review_owner` | string | 是 | 复核责任归属 | 不能为空 |

#### 3.2.3 `VerificationEvidence`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `task_id` | string | 是 | 对应任务编号 | 必须存在于 `tasks.md` |
| `grep_checks` | string[] | 是 | 本任务使用的 grep 或静态检查 | 至少一项 |
| `tests` | string[] | 否 | 本任务关联测试 | 可为空，但必须写原因 |
| `manual_checks` | string[] | 否 | 人工走查项 | 可为空，但必须写原因 |
| `residuals` | string[] | 是 | 本任务完成后仍保留的残留 | 可为空数组，但不能省略 |

### 3.3 接口契约

覆盖需求：2、3、4、5、6

#### 3.3.1 边界 grep 契约

- 类型：静态审计规则
- 路径或标识：
  - 核心层：`apps/api-server/app/modules/**`、`apps/api-server/app/api/**`
  - 前端层：`apps/user-app/src/**`
  - 插件层：`apps/api-server/app/plugins/builtin/homeassistant_*/**`
- 输入：
  - 关键词：`homeassistant`、`home_assistant`、`Home Assistant`
  - 禁止依赖：`app.modules.*.repository`、`config_crypto`、`database_url`
- 输出：
  - 命中项清单
  - 白名单过滤后剩余项
  - 对应责任任务
- 校验：
  - 命中项必须被解释为 `closed`、`whitelisted` 或 `open`
  - 不能出现“命中了但暂时不管”的灰色状态
- 错误：
  - 未授权命中即视为验收失败

#### 3.3.2 任务关闭契约

- 类型：流程规则
- 路径或标识：`tasks.md` 单任务关闭门槛
- 输入：
  - 任务结果
  - grep 结果
  - 测试结果
  - 剩余残留清单
- 输出：
  - `DONE` / `IN_PROGRESS` / `BLOCKED`
- 校验：
  - 缺一项证据就不能进 `DONE`
  - 若 spec 与代码不一致，必须先回写 spec
- 错误：
  - 口头宣布完成但证据缺失，视为流程错误

#### 3.3.3 白名单契约

- 类型：文档化约束
- 路径或标识：`docs/20260319-HA彻底插件化边界验收口径.md`
- 输入：
  - 暂不能立即删除的残留点
  - 保留原因
  - 删除条件
- 输出：
  - 受控白名单清单
- 校验：
  - 白名单必须可定位、可解释、可删除
  - 白名单不能扩大到整个模块或整个功能域
- 错误：
  - 未登记的残留视为违规残留

## 4. 数据与状态模型

### 4.1 数据关系

这次的核心关系不是业务表关系，而是“代码现状、spec 任务、验收证据”三者的对应关系。

1. 每个边界残留都必须能映射到一个任务
2. 每个任务都必须能映射到一组证据
3. 每个仍保留的残留都必须能映射到白名单
4. 任何一条映射断了，收尾状态就不可信

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `TODO` | 任务尚未开始 | 任务已定义但无人处理 | 进入 `IN_PROGRESS` |
| `IN_PROGRESS` | 正在处理 | 已有明确责任和正在推进的实现 | 证据齐备后进 `IN_REVIEW`，或因阻塞转 `BLOCKED` |
| `BLOCKED` | 被外部问题卡住 | 发现必须先解决的依赖、迁移或风险 | 阻塞解除后回 `IN_PROGRESS` |
| `IN_REVIEW` | 代码和证据已提交复核 | 静态检查、测试和回写已完成 | 复核通过后进 `DONE` |
| `DONE` | 任务真正完成 | 代码收口、证据齐全、文档回写完成 | 不再退出；若发现误判，必须回滚状态并补原因 |

`DONE` 的额外不变量：

- 不能还有未解释的 grep 命中
- 不能没有剩余残留说明
- 不能缺失对应验证结果

## 5. 错误处理

### 5.1 错误类型

- `SpecCodeDrift`：spec 任务状态与代码现状不一致
- `UnauthorizedResidual`：命中未授权 HA 残留
- `PluginBoundaryViolation`：插件直接依赖核心内部实现
- `RegressionFailed`：清理后回归失败
- `WhitelistAbuse`：试图用模糊白名单掩盖未完成项

### 5.2 错误响应格式

```json
{
  "detail": "检测到未授权的 HA 残留，当前任务不能标记为 DONE。",
  "error_code": "UnauthorizedResidual",
  "field": "tasks.md",
  "timestamp": "2026-03-19T00:00:00Z"
}
```

### 5.3 处理策略

1. spec 与代码漂移：先回写 spec，再继续开发或验收。
2. 核心层残留：优先判定是否应删除；只有临时桥接才允许进入白名单。
3. 插件层越界：优先补宿主 contract，不允许继续保留偷读路径。
4. 回归失败：任务维持 `IN_PROGRESS` 或 `BLOCKED`，不得口头跳过。

## 6. 正确性属性

### 6.1 属性 1：核心只认协议，不认平台

*对于任何* 配置、同步、控制和状态读取主链，系统都应该满足：核心层只依赖统一协议、统一资源模型和正式插件入口，不直接依赖 HA 平台实现细节。

**验证需求：** 需求 2、需求 5

### 6.2 属性 2：插件只通过宿主 contract 工作

*对于任何* HA 插件执行路径，系统都应该满足：插件不直接偷读核心仓储、密钥或数据库内部实现，而是通过正式宿主 contract 获取所需信息。

**验证需求：** 需求 3、需求 5

### 6.3 属性 3：没有证据就没有完成

*对于任何* 准备关闭的任务，系统都应该满足：grep、白名单、测试和剩余残留说明必须齐全，否则任务不得进入 `DONE`。

**验证需求：** 需求 1、需求 6、需求 7

## 7. 测试策略

### 7.1 单元测试

- 与 HA 配置读取、实例配置、插件 contract 相关的单元测试
- 与统一资源、统一配置、统一错误语义相关的单元测试

### 7.2 集成测试

- HA 配置完成后同步链路仍可正常工作
- 设备同步、房间同步、设备控制和状态加载走正式插件路径
- 插件禁用、配置缺失、实例缺失时返回统一错误

### 7.3 端到端测试

- 前端设置页不再暴露 HA 顶层特例
- 统一实例配置、同步和资源查看流程可用
- 家庭页和设置页不会再回退到旧 HA 专用路径

## 8. 风险与待确认项

### 8.1 风险

- 核心里部分 HA 逻辑可能已经被别的模块间接依赖，直接删除会带来回归风险
- HA 插件当前可能缺少足够宿主 contract，贸然禁止直连会先暴露接口缺口
- `005.4` 里已有“当前执行说明”与真实代码不一致，回写成本会高于预期

### 8.2 待确认项

- 哪些现存 HA 残留应该删除，哪些只能短期白名单保留
- 宿主需要补哪些正式 contract，才能完全替代插件内偷读路径
- 最终验收时采用哪些固定命令和测试集合作为最小门槛
