# Spec 004.8.4 - HA 彻底插件化收尾

状态：Draft

## 这份 Spec 负责什么

这份 Spec 不是再定义一遍“HA 插件化愿景”，而是把最后收尾这件事钉死。

前面几轮的问题，不是没人动手，而是总把“链路能跑”当成“边界已经干净”。结果就是：

- 配置表单、实例链路、页面入口看起来已经插件化
- 但核心里还残留 `Home Assistant` 专名逻辑
- 插件里还残留偷读核心仓储、密钥和数据库的特权路径
- `005.4` 里部分任务还是 `IN_PROGRESS`，可交付口径却容易被说成“差不多完成”

这份 Spec 的任务只有一个：

- 把 HA 从“看起来像插件化”收口成“代码边界、任务状态、验收证据都是真的插件化”

## 这份 Spec 和旧 Spec 的关系

这次不能再把几份 spec 混着用。

- `005.1`
  - 它定义原则
  - 核心只保留统一协议和通用能力
  - 平台实现必须留在插件
- `005.4`
  - 它是设备与集成彻底插件化的主战场
  - 后端、前端、迁移、旧接口清理都要看它
- `005.6`
  - 它只是前端迁移与同步收口
  - 它自己明确写了“不改后端设备同步执行逻辑，不改设备控制执行逻辑”
  - 所以它不能证明 HA 已经彻底插件化

`004.8.4` 的角色不是替代 `005.1` 和 `005.4`，而是做“收尾总控 spec”：

- 统一这次收尾的完成定义
- 固化黑名单、白名单、证据要求
- 把“没有清理干净就不准说 DONE”变成硬规则

## 这次明确不允许什么

- 不允许再把“功能可用”当成“插件化完成”
- 不允许 `005.4` 关键任务还是 `IN_PROGRESS`，口头上却说“已收口”
- 不允许核心目录继续出现 HA 专名业务分支
- 不允许 HA 插件继续直接 import 核心 repository、`config_crypto` 或用 `database_url` 自建数据库会话
- 不允许只给结果，不给 grep、白名单、测试和剩余残留清单
