# Spec 004.6 - 基于 GitHub 仓库的插件市场、一键安装与手动启用

状态：Draft

## 这份 Spec 现在到底要解决什么

这次不是做一个“项目自己托管服务端目录”的插件市场。

你要的是另一条路：

- 插件市场本身不依赖专门的市场服务器
- 市场目录直接来自 GitHub 仓库
- 官方内置一个官方插件市场仓库地址
- 用户还能自己添加第三方插件市场仓库
- 第三方作者不是往本项目后端上传插件，而是给某个注册表仓库提 PR
- 插件代码放在独立插件仓库里，注册表条目只保存市场识别、审核和安装需要的元数据

一句话说清楚：

这次做的是“基于 GitHub 仓库的无服务器插件市场”，不是“平台中心化托管插件包的服务端市场”。

## 这份 Spec 要把哪几件事写死

### 1. 市场来源机制

- 官方市场仓库地址内置在项目里
- 支持用户追加第三方市场仓库
- 一个市场仓库就是一个 GitHub 仓库源
- 本地把多个仓库聚合成用户看到的市场列表

### 2. 发布机制

- 第三方作者通过 PR 往注册表仓库提交插件条目
- 插件代码放在独立插件仓库
- 注册表仓库只维护目录、元数据、说明和审核入口

### 3. 安装机制

- 用户在市场里点一次“安装”
- 系统根据注册表条目去插件仓库拉取指定版本产物
- 安装成功后默认禁用
- 用户自己手动启用后，插件才允许进入运行链路

### 4. 市场评价信息

- 市场需要从 GitHub 读取可公开获取的仓库指标
- 至少包括 `star`、`fork` 等公开指标
- 这些指标用于辅助用户判断插件活跃度和可信度
- 仓库“浏览量 / views”只在仓库所有者已授权或官方仓库可读时作为可选指标，不作为所有插件的必备字段

### 5. 交互边界

- 市场里可以展示“去 GitHub 查看 / Star”跳转入口
- 但不做市场内直接帮用户给仓库点 `star`

原因很简单：

- GitHub 的 star 写接口需要用户认证
- 你已经明确说了，如果需要登录，那这条就算了
- 那就别为了一个小按钮，把整套 GitHub 登录授权硬塞进来

### 6. 仓库规则

这次必须补清楚两类仓库的文件规则：

1. 插件市场注册表仓库该怎么组织
2. 插件源码仓库至少要提供哪些文件和元数据

这部分不写清楚，后面所谓“GitHub 市场”一定会变成每个人自己发明格式，最后根本没法兼容。

## 这份 Spec 明确不做什么

- 不做中心化市场后端
- 不做上传到本项目服务器的插件托管
- 不做安装后自动启用
- 不做静默自动更新
- 不做绕过 GitHub 仓库结构校验的任意 URL 安装
- 不做“合 PR 就自动上线并自动跑起来”的危险捷径
- 不为“在市场内直接点 star”引入 GitHub 登录授权

## 一句话结论

这次 `004.6` 的核心不是“市场页面多漂亮”，而是把 GitHub 仓库式插件市场的发布、发现、安装、启用边界一次写对：

- 市场来自 GitHub 仓库
- 插件通过 PR 进入市场
- 插件代码放独立仓库
- 可以一键安装
- 但绝不自动启用
