# 需求文档 - 基于 GitHub 仓库的插件市场、一键安装与手动启用

状态：Draft

## 简介

当前项目里的“插件市场”还不是你要的市场机制。它更像是插件管理页外加一个市场占位入口，没有真正定义：

- 市场目录从哪里来
- 第三方作者怎么发布插件
- 官方和第三方来源怎么共存
- 客户端怎么在没有中心化市场服务端的前提下发现和安装插件

本次需求要解决的真问题很简单：

项目要支持一个基于 GitHub 仓库的无服务器插件市场。

市场不是平台数据库里的几条记录，而是一个或多个 GitHub 仓库中的注册表文件。第三方作者通过提交 PR 的方式把自己的插件条目加入注册表仓库。插件代码则放在独立的插件仓库里。客户端读取这些仓库，聚合成市场列表，并支持一键安装。

同时必须继续守住一条底线：

- 安装成功 != 自动启用
- 启用必须由用户手动触发

## 术语表

- **插件市场仓库**：用于维护插件注册表的 GitHub 仓库，保存插件条目、市场说明和仓库级元数据。
- **官方市场仓库**：项目内置的默认 GitHub 插件市场仓库地址。
- **第三方市场仓库**：用户手动添加的其他 GitHub 插件市场仓库地址。
- **注册表条目**：某个插件在市场仓库里的元数据文件，描述插件是谁、来自哪里、怎么安装、风险如何。
- **插件源码仓库**：保存插件真实代码、manifest、依赖、README 和发布产物的独立仓库。
- **注册 PR**：第三方作者向某个市场仓库提交的 Pull Request，用于申请收录插件或更新条目。
- **市场源**：一个可被本项目读取的插件市场仓库配置，至少包含 GitHub 仓库地址、分支、路径和信任状态。
- **安装目标**：注册表条目里声明的可安装版本及其下载来源，比如 release asset、源码 archive 或指定 tag。
- **安装状态（install_status）**：插件是否已安装、安装中、安装失败等状态，不代表插件是否启用。
- **启用状态（enabled）**：插件是否允许进入运行链路的开关，只能由用户显式启用或禁用。
- **配置状态（config_status）**：插件配置是否完整、是否通过校验，避免用户装完就误以为能直接运行。

## 范围说明

### In Scope

- 定义基于 GitHub 仓库的插件市场机制
- 内置官方市场仓库地址
- 支持用户添加第三方市场仓库
- 定义市场仓库文件结构和插件源码仓库最小规则
- 定义通过 PR 提交插件注册条目的流程边界
- 从多个市场仓库聚合插件目录
- 读取 GitHub 可公开获取的仓库评价指标
- 支持按注册表条目一键安装插件
- 安装后默认禁用，用户手动启用
- 市场来源、风险、版本、安装失败原因可见

### Out of Scope

- 自建中心化插件市场服务端
- 自建插件包托管平台
- 合 PR 后自动启用插件
- 自动更新和自动升级策略
- 市场付费、结算、授权售卖
- 非 GitHub 仓库来源的通用任意链接安装
- 为了市场内点 `star` 而接入 GitHub 登录授权

## 需求

### 需求 1：市场必须基于 GitHub 仓库，而不是中心化服务端

**用户故事：** 作为项目维护者，我希望插件市场本身就是一个 GitHub 仓库，这样不需要额外搭建市场后端，也能靠仓库内容维护插件目录。

#### 验收标准

1. WHEN 系统初始化插件市场配置 THEN System SHALL 内置一个官方 GitHub 市场仓库地址，而不是依赖环境里先配好市场服务地址。
2. WHEN 系统读取市场目录 THEN System SHALL 直接从配置的 GitHub 仓库读取注册表文件，而不是要求专门的市场 API 服务先返回目录。
3. WHEN 某个市场源不是合法 GitHub 仓库配置 THEN System SHALL 拒绝接入，并明确告诉用户是仓库地址、分支还是目录结构不合法。

### 需求 2：必须支持官方市场和第三方市场仓库并存

**用户故事：** 作为普通用户，我希望既能看到官方插件市场，也能按需添加第三方市场仓库，而不是只能用项目内置那一份目录。

#### 验收标准

1. WHEN 用户首次打开市场 THEN System SHALL 默认展示官方市场仓库内容。
2. WHEN 用户添加一个新的第三方 GitHub 市场仓库 THEN System SHALL 校验其结构后接入，并把它作为独立来源展示。
3. WHEN 多个市场仓库中存在同名插件 THEN System SHALL 保留来源信息并按来源区分展示，而不是静默合并成一条匿名记录。

### 需求 3：第三方作者必须通过 PR 向市场仓库提交插件条目

**用户故事：** 作为第三方开发者，我希望通过 fork 仓库、提 PR 的方式申请收录插件，而不是登录某个后台上传 zip 包。

#### 验收标准

1. WHEN 第三方开发者申请收录插件 THEN System SHALL 以市场仓库中的注册表条目为收录入口，而不是要求向本项目后端上传插件文件。
2. WHEN 市场仓库审核一个插件 THEN System SHALL 以注册表条目、插件源码仓库地址、README 和元数据作为审核材料。
3. WHEN 注册 PR 尚未合并 THEN System SHALL 不把该插件视为市场正式可安装条目。

### 需求 4：插件代码必须放在独立插件仓库，注册表只存元数据

**用户故事：** 作为项目维护者，我希望插件代码和市场目录分开，这样注册表仓库只负责收录和审核，插件真实代码仍归插件作者自己的仓库维护。

#### 验收标准

1. WHEN 某个插件被市场收录 THEN System SHALL 能从注册表条目里拿到插件源码仓库地址，而不是把完整代码直接塞进市场仓库。
2. WHEN 系统展示插件详情 THEN System SHALL 同时展示注册表来源和插件源码仓库来源。
3. WHEN 注册表条目和插件仓库 manifest 不一致 THEN System SHALL 把该条目标记为不可安装或同步失败，而不是继续当正常插件展示。

### 需求 5：必须定义 GitHub 仓库式市场的文件规则

**用户故事：** 作为维护者和开发者，我希望市场仓库和插件仓库都有明确文件规则，这样别人提 PR 时不需要猜格式，客户端读取时也不用写一堆兼容补丁。

#### 验收标准

1. WHEN 维护者创建一个市场仓库 THEN System SHALL 有明确的仓库根目录规则、注册表索引规则和单插件条目规则。
2. WHEN 开发者准备一个插件源码仓库 THEN System SHALL 有明确的最小文件要求，比如 `manifest.json`、README、依赖清单和发布版本定位规则。
3. WHEN 任一仓库不满足文件规则 THEN System SHALL 在同步或安装前给出明确结构错误，而不是等到安装阶段才报模糊失败。

### 需求 6：用户必须可以基于注册表条目一键安装插件

**用户故事：** 作为普通用户，我希望在市场里点一次安装，系统就能根据注册表条目去插件仓库拉取指定版本，而不是让我自己去 GitHub 下载文件。

#### 验收标准

1. WHEN 用户点击“安装” THEN System SHALL 根据注册表条目里的仓库、版本和安装来源创建安装任务。
2. WHEN 安装任务执行 THEN System SHALL 从插件源码仓库的指定发布目标拉取插件包，并进行 manifest 与元数据校验。
3. WHEN 注册表条目缺少可安装版本信息或下载来源无效 THEN System SHALL 阻止安装，并明确提示条目不完整。

### 需求 7：安装成功后默认禁用，启用必须手动触发

**用户故事：** 作为普通用户，我希望先把插件装下来，再自己决定是否启用，而不是装完就被系统直接跑起来。

#### 验收标准

1. WHEN 插件安装成功 THEN System SHALL 将 `install_status` 标记为 `installed`，并将 `enabled` 保持为 `false`。
2. WHEN 用户没有执行启用操作 THEN System SHALL 不得因为安装成功、市场刷新、仓库同步或版本更新检查而自动启用插件。
3. WHEN 用户点击“启用插件” THEN System SHALL 单独调用启用链路，并继续校验配置状态是否允许启用。

### 需求 8：未配置完成的插件不能被误运行

**用户故事：** 作为普通用户，我希望系统能明确告诉我插件虽然装好了，但还没配完，别让我以为它已经能工作。

#### 验收标准

1. WHEN 插件声明需要配置但当前配置缺失 THEN System SHALL 将 `config_status` 标记为 `unconfigured` 或 `invalid`。
2. WHEN 用户尝试启用未配置完成的插件 THEN System SHALL 阻止启用，并告诉用户缺什么配置。
3. WHEN 任意执行链路检查插件可用性 THEN System SHALL 拒绝运行 `enabled=false` 或 `config_status!=configured` 的插件。

### 需求 9：市场同步和安装失败必须可见、可追踪

**用户故事：** 作为维护者和用户，我希望知道是市场仓库同步失败、注册表条目坏了，还是插件安装失败，而不是看到一句“加载失败”。

#### 验收标准

1. WHEN 市场仓库拉取失败、分支不存在或文件结构错误 THEN System SHALL 记录市场源同步错误，并保留来源级错误信息。
2. WHEN 插件安装失败 THEN System SHALL 记录失败阶段、错误码、来源仓库、插件仓库和目标版本。
3. WHEN 用户查看插件来源或任务详情 THEN System SHALL 能追溯到市场仓库地址、插件仓库地址和安装目标版本。

### 需求 10：市场要展示基于 GitHub 的仓库评价信息

**用户故事：** 作为普通用户，我希望在市场里看到插件对应 GitHub 仓库的 star、fork 等信息，帮助我判断这个插件是不是有人维护、是不是值得装。

#### 验收标准

1. WHEN 系统同步市场条目 THEN System SHALL 同步插件源码仓库的公开可读指标，至少包括 `stargazers_count` 和 `forks_count`。
2. WHEN 某个指标对第三方仓库不可公开读取 THEN System SHALL 把该指标标记为不可用，而不是伪造为 `0`。
3. WHEN 市场展示插件卡片或详情 THEN System SHALL 明确区分“仓库公开指标”和“平台内主观评分”，且本轮不引入平台内打分体系。

### 需求 11：仓库浏览量只能作为可选增强信息

**用户故事：** 作为维护者，我希望如果仓库所有者愿意授权，市场可以显示浏览量；但如果 GitHub 本身不公开这个数据，系统也不要为了这件事把设计搞复杂。

#### 验收标准

1. WHEN 系统读取 GitHub 仓库浏览量 THEN System SHALL 只在仓库拥有者提供足够权限时读取该指标。
2. WHEN 第三方仓库没有提供浏览量读取权限 THEN System SHALL 不把“浏览量缺失”视为条目错误，也不阻止市场展示和安装。
3. WHEN 市场展示浏览量 THEN System SHALL 明确标记其为“可选仓库流量指标”，不能把它写成所有仓库都保证存在的统一评价字段。

### 需求 12：市场不做内嵌 GitHub Star 写回

**用户故事：** 作为普通用户，我希望在市场里能快速跳到 GitHub 仓库，但如果要为了点 star 先登录 GitHub，那这轮就不要做市场内直接 star。

#### 验收标准

1. WHEN 用户在市场里看到插件详情 THEN System SHALL 提供跳转到 GitHub 仓库页的入口。
2. WHEN 系统未接入 GitHub 用户登录授权 THEN System SHALL 不提供“在市场内直接点 star 并写回 GitHub”的功能按钮。
3. WHEN 后续要支持市场内直接 star THEN System SHALL 单独走新的认证设计，而不是在本次 spec 中偷偷塞入 GitHub 登录能力。

## 非功能需求

### 非功能需求 1：安全性

1. WHEN 系统接入第三方市场仓库 THEN System SHALL 明确标记其为第三方来源，不得伪装成官方来源。
2. WHEN 系统安装插件 THEN System SHALL 校验注册表条目与插件仓库 manifest 的一致性，并校验发布目标的完整性摘要或等价校验信息。

### 非功能需求 2：可靠性

1. WHEN GitHub 仓库暂时不可访问或触发速率限制 THEN System SHALL 保留上次成功同步的市场快照，并明确标记当前同步失败。
2. WHEN 市场同步失败 THEN System SHALL 不得破坏已经安装的本地插件状态。
3. WHEN GitHub 的评价指标接口限流或部分不可用 THEN System SHALL 允许先展示条目主体信息，评价指标可降级为空。

### 非功能需求 3：可维护性

1. WHEN 后续新增新的官方市场仓库分支规则或文件字段 THEN System SHALL 基于统一仓库 schema 扩展，而不是在每个来源写特例。
2. WHEN 审核或排查第三方插件问题 THEN System SHALL 能通过注册表条目快速定位对应的插件源码仓库、版本和 PR 提交流程入口。

## 成功定义

- 项目内置一个官方 GitHub 市场仓库，并支持添加第三方市场仓库。
- 市场目录完全基于 GitHub 仓库注册表文件，不依赖中心化市场服务端。
- 第三方开发者通过提交注册 PR 的方式收录插件，插件代码保存在独立仓库。
- 市场仓库和插件仓库的文件规则清楚、稳定、可校验。
- 用户可以一键安装插件，但安装后默认禁用，仍需手动启用。
- 市场能够展示 GitHub 公开可读的仓库评价指标，但不会为了市场内点 star 引入 GitHub 登录授权。
