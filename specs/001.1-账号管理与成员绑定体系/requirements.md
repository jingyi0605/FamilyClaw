# 需求文档 - 账号管理与成员绑定体系

状态：Draft

## 简介

当前系统没有真正的账号体系，前端只要塞请求头就能假装成管理员或成员。这会直接导致三件事：

- 后端不知道请求到底是谁发的
- `admin-web` 和 `user-web` 没法真正隔离
- 后续初始化向导、成员权限、审计日志都没有可信基础

这份 Spec 要解决的是：给系统补一套完整账号体系，并让账号可以绑定到某个成员上。绑定之后，成员就是这个账号在家庭里的业务角色来源。

这里要特别说清楚：

- **账号** 负责登录、密码、会话、禁用状态
- **成员** 负责家庭里的身份，比如管理员、成人、儿童、长辈、访客
- **账号绑定成员** 以后，很多权限判断都要从成员角色和成员权限继续往下走

## 术语表

- **System**：FamilyClaw
- **账号**：能登录系统的身份对象，包含用户名、密码哈希、状态等信息
- **成员绑定**：账号和某个家庭成员建立一对一关联，表示该账号以这个成员身份参与家庭业务
- **系统账号**：不绑定家庭成员、只用于系统管理的账号，比如 `admin`
- **家庭账号**：绑定到某个家庭成员的账号，用于 `user-web` 或家庭内业务操作
- **Bootstrap 账号**：只在新家庭初始化前临时存在的默认账号，比如 `user/user`

## 范围说明

### In Scope

- 定义统一账号模型、会话模型、账号与成员绑定模型
- 支持系统账号与家庭账号两类登录方式
- 支持账号绑定到某个成员，并以成员角色作为后续业务权限来源
- 支持新家庭初始化阶段的 Bootstrap 账号与正式密码切换
- 为后端 API 提供统一认证上下文，替代当前伪造请求头方案

### Out of Scope

- 第三方登录（微信、Google、Apple 等）
- 多因素认证、短信验证码、邮箱验证码
- 一个账号同时绑定多个成员
- 复杂组织架构、多家庭共享同一个家庭账号

## 需求

### 需求 1：系统必须有真正的账号对象

**用户故事：** 作为系统维护者，我希望后端能识别真实账号，而不是信任前端传来的角色，以便系统具备最基本的安全基础。

#### 验收标准

1. WHEN 系统创建账号 THEN System SHALL 持久化用户名、密码哈希、账号类型、状态和必要审计字段。
2. WHEN 用户提交用户名和密码登录 THEN System SHALL 只根据服务端保存的账号数据完成认证，不再依赖前端传入的 `X-Actor-Role` 或 `X-Actor-Id`。
3. WHEN 账号被停用、锁定或删除 THEN System SHALL 拒绝该账号继续建立新会话。

### 需求 2：账号必须支持绑定到某个成员

**用户故事：** 作为家庭系统设计者，我希望账号能绑定到一个成员，以便成员成为这个账号在家庭里的业务身份来源。

#### 验收标准

1. WHEN 创建家庭账号 THEN System SHALL 支持将该账号绑定到一个成员。
2. WHEN 账号已绑定成员 THEN System SHALL 在认证上下文中返回该成员 ID、家庭 ID 和成员角色。
3. WHEN 账号未绑定成员但试图访问需要成员身份的家庭接口 THEN System SHALL 拒绝访问并给出明确错误。

### 需求 3：成员角色必须成为权限判断的主要来源

**用户故事：** 作为后端开发者，我希望账号登录后能继续沿用成员角色和成员权限，以便权限模型不再裂成两套。

#### 验收标准

1. WHEN 家庭账号访问业务接口 THEN System SHALL 优先使用绑定成员的角色和成员权限做授权判断。
2. WHEN 系统账号访问仅管理员可操作的接口 THEN System SHALL 允许访问，即使它没有绑定成员。
3. WHEN 家庭账号绑定的成员角色变化 THEN System SHALL 在新请求中按最新成员角色生效，而不是继续使用旧缓存结果。

### 需求 4：系统必须支持安全的登录与会话管理

**用户故事：** 作为前端用户，我希望登录后能保持会话，同时在登出、过期或禁用后被正确拦截，以便系统行为稳定可预期。

#### 验收标准

1. WHEN 账号登录成功 THEN System SHALL 创建服务端可校验的会话，并支持前端后续请求复用该会话。
2. WHEN 用户主动登出 THEN System SHALL 使当前会话失效。
3. WHEN 会话过期、被撤销或账号被禁用 THEN System SHALL 在后续请求中返回未认证状态。

### 需求 5：新家庭初始化必须支持 Bootstrap 账号过渡

**用户故事：** 作为新家庭用户，我希望在系统还没完成首成员和正式密码设置前有一个临时入口，但在初始化完成后这个默认入口会被关闭，以便既能启动系统，又不会长期留下默认弱口令。

#### 验收标准

1. WHEN 新家庭尚未完成初始化 THEN System SHALL 允许使用 Bootstrap 账号进入初始化流程。
2. WHEN 用户在初始化向导中完成首成员创建并设置正式密码 THEN System SHALL 创建正式家庭账号并绑定到该成员。
3. WHEN 初始化完成 THEN System SHALL 禁用该家庭的 Bootstrap 账号，默认 `user/user` 不再可用。

### 需求 6：后端接口必须基于认证上下文而不是前端自报身份

**用户故事：** 作为系统维护者，我希望所有 API 都从服务端认证上下文拿身份，以便前端无法通过改请求头越权。

#### 验收标准

1. WHEN 后端处理受保护接口 THEN System SHALL 从服务端认证上下文中读取账号、成员、家庭和角色信息。
2. WHEN 前端继续传入旧的 `X-Actor-Role` 或 `X-Actor-Id` THEN System SHALL 不把它们当作正式权限依据。
3. WHEN 用户访问不属于自己家庭的数据 THEN System SHALL 拒绝访问。

## 非功能需求

### 非功能需求 1：安全性

1. WHEN 保存密码 THEN System SHALL 只保存密码哈希，不保存明文密码。
2. WHEN 默认账号仍处于启用期 THEN System SHALL 明确标记其用途和失效条件，避免变成长期后门。

### 非功能需求 2：兼容性

1. WHEN 老数据还没有账号记录 THEN System SHALL 提供迁移或补录路径，不要求手工直接改库。
2. WHEN 现有接口逐步迁移到新认证模型 THEN System SHALL 允许按阶段收口，避免一次性把所有链路打断。
3. WHEN 项目发布新版本 THEN System SHALL 支持通过标准数据库迁移命令把旧版本数据库升级到新版本，而不是要求用户手工补字段。

### 非功能需求 3：可维护性

1. WHEN 后续增加更多登录方式 THEN System SHALL 复用同一套账号和会话主模型，而不是再造一套用户表。
2. WHEN 排查越权、登录失败、账号禁用问题 THEN System SHALL 能在审计日志中定位到具体账号和成员绑定关系。
3. WHEN 新增数据库字段或表 THEN System SHALL 为该变更提供唯一、可追溯、可连续执行的迁移版本，不允许出现重复 revision ID 或长期分叉不收口。

## 成功定义

- 后端不再把前端自报的角色头当正式身份依据
- `admin-web` 和 `user-web` 都能走统一认证入口
- 家庭账号能稳定绑定到成员，成员角色成为业务权限来源
- 新家庭初始化完成后，默认 `user/user` 会被自动禁用
