# 设计文档 - 账号管理与成员绑定体系

状态：Draft

## 1. 概述

### 1.1 目标

- 给系统补上真正可落地的账号、密码、会话基础能力
- 支持账号绑定到某个成员，并让成员成为家庭业务权限的主要来源
- 为 `admin-web`、`user-web` 和后端 API 提供统一认证入口

### 1.2 覆盖需求

- `requirements.md` 需求 1：系统必须有真正的账号对象
- `requirements.md` 需求 2：账号必须支持绑定到某个成员
- `requirements.md` 需求 3：成员角色必须成为权限判断的主要来源
- `requirements.md` 需求 4：系统必须支持安全的登录与会话管理
- `requirements.md` 需求 5：新家庭初始化必须支持 Bootstrap 账号过渡
- `requirements.md` 需求 6：后端接口必须基于认证上下文而不是前端自报身份

### 1.3 技术约束

- 后端：FastAPI + SQLAlchemy + SQLite
- 前端：`admin-web`、`user-web` 统一改成真实登录态
- 数据存储：首期不引入 Redis，先用数据库会话表
- 认证方式：服务端会话 + HttpOnly Cookie
- 外部依赖：不依赖第三方登录平台

## 2. 架构

### 2.1 系统结构

这套设计引入 4 个核心对象：

1. **账号**：保存用户名、密码哈希、账号类型、启停状态
2. **会话**：保存登录结果、过期时间、撤销状态
3. **成员绑定**：说明某个账号绑定到哪个成员
4. **认证上下文**：后端每次请求里统一可读的真实身份结果

整体链路：

1. 用户提交用户名和密码
2. 后端校验账号和密码哈希
3. 登录成功后写入会话表，并下发 Cookie
4. 后续请求通过 Cookie 找到会话和账号
5. 如果账号绑定了成员，再补齐成员 ID、家庭 ID、成员角色
6. 业务接口只读取认证上下文，不再信任前端自报身份

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 账号管理 | 创建、更新、禁用账号 | 用户名、密码、账号类型 | 账号记录 |
| 会话管理 | 创建、校验、撤销会话 | 账号 ID、登录请求 | 会话记录、登录态 |
| 成员绑定 | 绑定账号到成员 | 账号 ID、成员 ID | 绑定关系、家庭上下文 |
| 认证上下文 | 聚合账号和成员信息 | Cookie、会话、绑定关系 | 当前请求身份 |
| 授权判断 | 判断能否访问接口 | 认证上下文、业务参数 | 放行或拒绝 |

### 2.3 关键流程

#### 2.3.1 `admin-web` 登录

1. 管理员输入 `admin` 用户名和密码
2. 后端校验系统账号
3. 登录成功后创建会话并写 Cookie
4. 后续管理接口通过系统账号身份放行

#### 2.3.2 家庭账号登录

1. 用户输入家庭账号用户名和密码
2. 后端校验账号
3. 读取该账号绑定的成员
4. 认证上下文补齐成员 ID、家庭 ID、成员角色
5. 业务接口基于成员角色和成员权限继续判断

#### 2.3.3 新家庭初始化过渡

1. 新家庭创建后，系统为该家庭生成一个 Bootstrap 账号
2. 用户用默认账号进入初始化向导
3. 向导创建首成员并设置正式密码
4. 系统创建正式家庭账号并绑定到首成员
5. Bootstrap 账号被禁用

## 3. 组件和接口

### 3.1 核心组件

- `account` 模块：账号创建、密码校验、启停控制
- `session` 模块：会话签发、会话校验、登出撤销
- `account_member_binding` 模块：账号和成员的一对一绑定
- `auth_context` 依赖：把账号、成员、家庭、角色聚合进请求上下文
- `authorization` 依赖：提供 `require_admin_account`、`require_bound_member` 之类的收口入口

### 3.2 数据结构

#### 3.2.1 `accounts`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `id` | string | 是 | 账号 ID | UUID |
| `username` | string | 是 | 登录用户名 | 全局唯一 |
| `password_hash` | string | 是 | 密码哈希 | 不存明文 |
| `account_type` | enum | 是 | `system / household / bootstrap` | 固定枚举 |
| `status` | enum | 是 | `active / disabled / locked` | 固定枚举 |
| `household_id` | string \| null | 否 | 所属家庭 | 系统账号为空 |
| `must_change_password` | bool | 是 | 是否必须尽快改密 | Bootstrap 初期为 true |
| `created_at` | string | 是 | 创建时间 | ISO 时间 |
| `updated_at` | string | 是 | 更新时间 | ISO 时间 |

#### 3.2.2 `account_member_bindings`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `account_id` | string | 是 | 账号 ID | 唯一 |
| `member_id` | string | 是 | 绑定成员 ID | 唯一 |
| `household_id` | string | 是 | 所属家庭 ID | 必须与成员一致 |
| `binding_status` | enum | 是 | `active / revoked` | 固定枚举 |
| `created_at` | string | 是 | 创建时间 | ISO 时间 |
| `updated_at` | string | 是 | 更新时间 | ISO 时间 |

说明：第一版只允许一对一绑定。一个账号只能绑一个成员，一个成员也只能绑一个账号。

#### 3.2.3 `account_sessions`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `id` | string | 是 | 会话 ID | UUID |
| `account_id` | string | 是 | 账号 ID | 外键 |
| `session_token_hash` | string | 是 | 会话令牌哈希 | 不存原始 token |
| `status` | enum | 是 | `active / revoked / expired` | 固定枚举 |
| `expires_at` | string | 是 | 过期时间 | ISO 时间 |
| `last_seen_at` | string | 否 | 最近访问时间 | ISO 时间 |
| `created_at` | string | 是 | 创建时间 | ISO 时间 |

#### 3.2.4 `AuthContext`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `account_id` | string | 是 | 当前账号 ID | 不为空 |
| `account_type` | string | 是 | 账号类型 | 固定枚举 |
| `account_status` | string | 是 | 账号状态 | 固定枚举 |
| `username` | string | 是 | 当前用户名 | 不为空 |
| `member_id` | string \| null | 否 | 绑定成员 ID | 家庭账号通常有值 |
| `household_id` | string \| null | 否 | 当前家庭 ID | 家庭账号通常有值 |
| `member_role` | string \| null | 否 | 绑定成员角色 | 从成员表读取 |
| `is_authenticated` | bool | 是 | 是否已认证 | 固定布尔 |

### 3.3 接口契约

#### 3.3.1 登录

- 类型：HTTP
- 路径：`POST /api/v1/auth/login`
- 输入：用户名、密码
- 输出：当前账号摘要、当前成员摘要、是否需要改密
- 校验：账号必须启用；密码必须正确
- 错误：`401` 用户名或密码错误；`403` 账号被禁用；`423` 账号被锁定

#### 3.3.2 获取当前登录态

- 类型：HTTP
- 路径：`GET /api/v1/auth/me`
- 输入：Cookie 会话
- 输出：认证上下文摘要
- 校验：会话必须有效
- 错误：`401` 未登录或会话失效

#### 3.3.3 登出

- 类型：HTTP
- 路径：`POST /api/v1/auth/logout`
- 输入：Cookie 会话
- 输出：成功结果
- 校验：当前会话存在
- 错误：`401` 未登录

#### 3.3.4 创建家庭账号并绑定成员

- 类型：HTTP
- 路径：`POST /api/v1/accounts/household`
- 输入：用户名、密码、成员 ID
- 输出：账号信息、绑定信息
- 校验：成员必须存在；成员未被其他账号绑定；成员属于指定家庭
- 错误：`400` 参数非法；`409` 用户名已存在或成员已被绑定

#### 3.3.5 禁用 Bootstrap 账号

- 类型：HTTP
- 路径：`POST /api/v1/accounts/bootstrap/{household_id}/disable`
- 输入：家庭 ID
- 输出：处理结果
- 校验：该家庭已存在正式家庭账号
- 错误：`404` 家庭或 Bootstrap 账号不存在；`409` 还未创建正式账号

## 4. 数据与状态模型

### 4.1 数据关系

- `accounts` 是登录主表
- `members` 继续是家庭业务主表
- `account_member_bindings` 负责把两者接起来
- `account_sessions` 负责把“登录成功”变成“后续请求可识别”

关系约束：

- 系统账号可以不绑定成员
- 家庭账号必须绑定成员后才能访问成员态接口
- Bootstrap 账号属于某个家庭，但不直接代表正式成员身份

### 4.2 状态流转

#### 4.2.1 账号状态

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `active` | 可登录可建会话 | 新建成功或重新启用 | 被禁用或锁定 |
| `disabled` | 不允许登录 | 管理员禁用、Bootstrap 被停用 | 重新启用 |
| `locked` | 临时锁住 | 连续失败或人工锁定 | 解锁后恢复 |

#### 4.2.2 绑定状态

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `active` | 当前绑定有效 | 账号成功绑定成员 | 被撤销 |
| `revoked` | 绑定已撤销 | 解绑或换绑 | 一般不恢复，重新建新绑定 |

## 5. 关键设计决策

### 5.1 账号和成员分开建，不混成一张表

原因很简单：

- 成员是家庭业务对象
- 账号是认证对象

把两者硬塞在一起，后面一定会长出一堆特殊情况：

- 系统管理员不属于任何成员怎么办
- Bootstrap 账号还没绑定成员怎么办
- 一个成员被停用但账号还要保留审计怎么办

所以必须拆开，再用绑定表连接。

### 5.2 第一版只做一对一绑定

这是故意收窄范围，不是偷懒。

如果第一版就支持：

- 一个账号绑定多个成员
- 一个成员被多个账号共用

后面的权限、审计、当前角色选择都会立刻变脏。

第一版先坚持：

- 一个账号最多一个成员
- 一个成员最多一个账号

把主链路做干净。

### 5.3 会话放服务端，不让前端自己保管权限真相

当前系统的问题就是前端说自己是谁，后端就信。

所以这次反着来：

- 前端只拿一个会话 Cookie
- 后端自己查会话、查账号、查成员绑定
- 真正身份只在服务端决定

## 6. 风险与边界

### 6.1 风险

- 风险 1：`user-web` 当前很多页面其实在跑管理员能力，接入新账号体系后会暴露产品边界问题
- 风险 2：老数据没有账号记录，需要补迁移和初始化逻辑
- 风险 3：如果接口收口顺序不对，容易出现部分页面突然 401/403

### 6.2 这份设计先不做什么

- 不做社交登录
- 不做 MFA
- 不做复杂设备级风控
- 不做一个账号切多个成员视角

## 7. 验证策略

- 账号创建：能建系统账号、家庭账号、Bootstrap 账号
- 绑定关系：家庭账号能绑定成员，重复绑定会失败
- 登录会话：登录成功后能拿到 `auth/me`，登出后立即失效
- 权限上下文：家庭账号访问接口时，后端能拿到成员 ID、家庭 ID、成员角色
- Bootstrap 收口：初始化完成后默认 `user/user` 无法继续登录
