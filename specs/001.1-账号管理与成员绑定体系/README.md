# Spec 001.1 - 账号管理与成员绑定体系

状态：Draft

## 这份 Spec 到底要解决什么

现在项目里最大的问题不是“还没有登录页”，而是后端根本没有可信账号。

当前前后端是这样工作的：

- 前端自己塞一个 `X-Actor-Role`
- 后端就信了
- `admin-web` 和 `user-web` 都能假装自己是谁

这不叫权限，这叫裸奔。

这份 Spec 要补的是一套真正能落地的账号体系，并把账号和成员关系讲清楚。

核心目标只有三个：

1. 系统里要有真正的账号，而不是靠请求头假扮身份
2. 账号要能绑定到某个家庭成员
3. 绑定后的成员，才是这个账号在家庭里的“角色来源”

换句话说，后面很多权限判断都不该再问“前端说自己是谁”，而应该问：

- 这个账号是谁
- 它绑定了哪个成员
- 这个成员在家庭里是什么角色

## 规划结论

### 【核心判断】

✅ 值得做，而且必须先做。  
没有账号体系，后面所有权限、初始化向导、用户端正式化，都会建立在假地基上。

### 【关键洞察】

- 真正该新增的主对象不是“登录页”，而是**账号**
- 成员不是登录凭证，成员是家庭里的业务身份
- 账号和成员必须解耦，但又要允许绑定
- 权限判断要分两层：先看账号能不能进系统，再看成员在家庭里能做什么

### 【Linus式方案】

1. 先把账号、会话、绑定关系建出来
2. 再把“成员就是角色来源”这件事固化到后端
3. 再让 `admin-web` 和 `user-web` 走同一套认证入口
4. 最后再去收口现有 API 的权限漏洞

## 这份 Spec 包含什么

- `requirements.md`：定义账号、成员绑定、登录、会话、权限边界
- `design.md`：定义数据模型、接口、绑定规则、迁移和兼容策略
- `tasks.md`：定义落地顺序、涉及文件和验证方式

## 这份 Spec 明确不做什么

- 不在这里细写完整 UI 视觉稿
- 不在这里把所有页面都改完，这份 Spec 先把底座和边界定清楚
- 不做第三方 OAuth、短信验证码、多因素认证
- 不在第一版支持一个账号同时绑定多个成员

## 依赖与协同

- **依赖 `Spec 001`**：复用家庭、成员、权限基础模型
- **协同 `Spec 003.2`**：初始化向导里的首账号创建和默认口令禁用要走这套体系
- **影响 `admin-web` / `user-web` / `api-server`**：三端都要改，不能只改一边
