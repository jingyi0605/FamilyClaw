# 需求文档 - 事件循环与后台任务阻塞治理

状态：Draft

## 简介

当前后端把一部分同步数据库访问、外部通道调用、插件执行和定时任务逻辑放进了 `async` 路径或常驻 worker。只要这些调用里出现慢查询、慢网络、长时间重试或第三方阻塞，主事件循环就可能被拖住。

这个问题的坏处不是“后台任务慢一点”这么简单，而是会直接反映到用户侧：前端连不上后端、API 响应很慢、WebSocket 不稳定。

这份需求文档要解决的核心问题只有一句话：

让后台慢任务和外部通道故障只影响它们自己，不要再把整个 API / WebSocket 主链路一起拖死。

## 术语表

- **主事件循环**：承载 FastAPI `async` endpoint、WebSocket 处理、应用生命周期协程的核心执行线程。
- **阻塞操作**：会长时间占住线程而不把控制权还给事件循环的操作，例如同步 ORM、同步网络调用、长时间 CPU 计算、`time.sleep()`。
- **后台任务 worker**：应用启动后常驻运行的轮询或消费循环，例如通道轮询、插件任务、调度器 worker。
- **执行边界**：一段逻辑到底是允许直接运行在事件循环里，还是必须下沉到线程池、进程池或独立 worker。
- **通道轮询**：对 Telegram 等外部消息通道做周期性拉取、收发和状态刷新。

## 范围说明

### In Scope

- 盘点后端 `async` 路径、WebSocket、生命周期任务中的阻塞操作。
- 盘点 Telegram 轮询、插件任务、调度 worker 等常驻后台任务的执行方式。
- 统一定义“什么逻辑可以留在主事件循环、什么必须下沉”。
- 为后台任务补统一的超时、取消、重试、降级、日志和健康检查规范。
- 为高风险路径补测试、文档和回归检查点。

### Out of Scope

- 全量替换项目里所有同步数据库和同步网络库。
- 设计一套新的通用消息总线或复杂事件驱动平台。
- 重做前端的断线重连和超时提示策略。
- 在本轮里处理与主循环阻塞无关的普通性能问题。

## 需求

### 需求 1：主事件循环不能再被后台慢任务拖住

**用户故事：** 作为正在使用前端页面的用户，我希望后端 API 和 WebSocket 在后台轮询或插件任务变慢时仍然保持基本可用，以便我不会频繁遇到连不上后端或响应极慢的问题。

#### 验收标准

1. WHEN 后端执行 `async` endpoint、WebSocket handler 或生命周期协程 THEN System SHALL 明确区分哪些逻辑允许直接运行、哪些逻辑必须下沉执行。
2. WHEN 后台 worker 出现慢网络、慢数据库或长时间插件执行 THEN System SHALL 不让这类操作直接阻塞主事件循环。
3. WHEN 某个后台任务卡住或超时 THEN System SHALL 让 API / WebSocket 主链路继续提供降级可用能力，而不是整体失去响应。

### 需求 2：后台任务必须遵守统一执行规范

**用户故事：** 作为维护后端的开发者，我希望通道轮询、插件任务和调度任务都遵守同一套执行规则，以便以后新增后台任务时不用每次重新猜“这里到底能不能直接 await”。

#### 验收标准

1. WHEN 新增或修改后台 worker THEN System SHALL 提供统一的执行模板，明确循环节拍、阻塞调用包装、超时和取消处理方式。
2. WHEN worker 需要访问同步 ORM、同步 HTTP 客户端或插件代码 THEN System SHALL 要求使用统一的下沉方式，而不是各写各的线程切换逻辑。
3. WHEN 代码评审检查后台任务实现 THEN System SHALL 能根据明确规则判断当前实现是否可能阻塞主循环。

### 需求 3：外部通道和插件失败要被隔离

**用户故事：** 作为系统维护者，我希望 Telegram 等外部通道故障、插件超时或第三方接口抖动只影响对应任务本身，以便局部故障不会扩散成全站不可用。

#### 验收标准

1. WHEN Telegram 轮询、插件执行或调度任务依赖的第三方服务不可用 THEN System SHALL 对失败做超时、重试或降级处理，并限制影响范围。
2. WHEN 某个 worker 连续失败 THEN System SHALL 暴露足够的日志和健康状态，便于快速判断故障点。
3. WHEN 单个外部依赖长期不稳定 THEN System SHALL 允许暂停、跳过或降级该任务，而不是持续挤占主链路资源。

### 需求 4：治理规则必须可测试、可回归、可交接

**用户故事：** 作为后续接手这个项目的人，我希望这次治理不仅有代码改动，还有明确的文档、测试和检查清单，以便后面的人不会再把阻塞逻辑偷偷塞回主循环。

#### 验收标准

1. WHEN 完成一条高风险治理链路 THEN System SHALL 补对应的单元测试或集成测试，证明阻塞调用已被正确隔离。
2. WHEN 新人阅读这份 Spec 或补充文档 THEN System SHALL 让他快速看懂哪些路径被治理、哪些路径仍有风险、怎么验证。
3. WHEN 以后新增后台任务或修改 async 入口 THEN System SHALL 有明确的检查项判断是否违反执行边界规范。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 后台通道轮询或插件任务出现单次慢调用 THEN System SHALL 让主事件循环继续及时处理新的 HTTP / WebSocket 请求，而不是被同一慢调用长时间占住。
2. WHEN 系统存在多个后台 worker 并发运行 THEN System SHALL 控制下沉执行的并发度和节拍，避免把线程池或数据库连接池打爆。

### 非功能需求 2：可靠性

1. WHEN 外部依赖超时、网络抖动或插件执行失败 THEN System SHALL 采用明确的超时、重试、跳过或熔断策略，而不是无限等待。
2. WHEN worker 异常退出或进入连续失败状态 THEN System SHALL 记录足够的上下文，让维护者能够恢复或临时停用问题任务。

### 非功能需求 3：可维护性

1. WHEN 新增后台任务或修改 async 链路 THEN System SHALL 复用统一 helper、基类或检查规则，而不是复制一套新的调度逻辑。
2. WHEN 需要排查“前端连不上后端”的问题 THEN System SHALL 提供清楚的日志、指标或排障文档，帮助快速定位是主循环阻塞、线程池拥塞，还是外部通道故障。

## 成功定义

- Telegram 轮询、插件任务、调度 worker 和高风险 async 入口都被盘点并分级。
- 主事件循环与阻塞操作的边界被写成明确规则，并落实到公共实现或检查点。
- 至少覆盖高风险链路的回归测试和排障文档已经补齐。
- 以后再有人把同步慢调用直接塞进主事件循环时，评审和测试都能把它拦下来。
