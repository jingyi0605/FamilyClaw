# 阻塞风险盘点 - 001.6 事件循环与后台任务阻塞治理

状态：Draft

## 这份盘点回答什么

这份文档只回答三件事：

1. 现在到底有哪些 async 入口和后台 worker 在主事件循环里干了不该干的事。
2. 哪些链路是真风险，哪些只是看着吓人。
3. 第一批先治哪里，避免范围失控。

## 风险分级标准

- `P0`：已经直接威胁主事件循环。常见形态是常驻 worker 或 WebSocket 循环里直接跑同步网络、同步 ORM、插件执行，或者把 `time.sleep()` 这类硬阻塞带进来。会直接拖慢 API / WebSocket。
- `P1`：明显会在 async 链路里阻塞，但触发频率或影响面比 `P0` 小一些，或者当前链路只在特定功能下触发。
- `P2`：已经做了部分隔离，但执行边界还不干净，比如没有统一超时、没有失败降级、把请求作用域 `Session` 带进线程池。
- `P3`：当前主要是观测缺失或潜在问题，这一轮先记账，不抢第一批资源。

## 盘点结论

### 1. 应用生命周期启动链路

| 项目 | 结论 |
| --- | --- |
| 链路 | `main.lifespan` -> `SessionLocal()` -> `ensure_bootstrap_admin_account` / `ensure_pending_household_bootstrap_accounts` -> 启动三个 worker |
| 证据 | `apps/api-server/app/main.py:27-40` |
| 为什么会阻塞 | `lifespan` 是 async 入口，但启动时直接跑同步数据库初始化，没有下沉。数据库慢时，应用 ready 会被直接拖住。 |
| 影响面 | API / WebSocket 冷启动、重启恢复 |
| 风险等级 | `P1` |
| 处理建议 | 阶段 2 一并纳入统一阻塞 helper；启动阶段至少要有受控超时和日志。 |

### 2. 通道轮询 worker

| 项目 | 结论 |
| --- | --- |
| 链路 | `ChannelPollingWorker.run_forever` -> `run_channel_polling_worker_cycle` -> `_load_polling_accounts` -> `poll_channel_account` -> `execute_household_plugin` -> 通道插件同步 HTTP / 重试 / `time.sleep()` |
| 证据 | `apps/api-server/app/modules/channel/polling_worker.py:33-78`、`apps/api-server/app/modules/channel/polling_service.py:42-139`、`apps/api-server/app/modules/plugin/service.py:806-858`、`apps/api-server/app/plugins/builtin/channel_telegram/channel.py:182-204` |
| 为什么会阻塞 | 这是最典型的烂味道：`async` worker 外壳包着整条同步链路。轮询账户列表、插件执行、入站事件处理都在主事件循环线程里串行跑。Telegram 插件里还是同步 `httpx.get/post` 加 `time.sleep()` 重试。 |
| 影响面 | 后台轮询一慢，主事件循环一起慢，API / WebSocket 跟着抖。 |
| 风险等级 | `P0` |
| 处理建议 | 第一批必须治理。先把整轮 tick 下沉到统一 worker 模板，再把插件执行、同步 DB 和失败回写收口到统一 helper。 |

### 3. 通道 webhook async 入口

| 项目 | 结论 |
| --- | --- |
| 链路 | `channel_gateway_webhook_endpoint` -> `handle_channel_webhook` -> `execute_household_plugin` -> `record_channel_inbound_event` -> 视情况同步 `process_channel_inbound_event` |
| 证据 | `apps/api-server/app/api/v1/endpoints/channel_gateways.py:24-62`、`apps/api-server/app/modules/channel/gateway_service.py:42-198` |
| 为什么会阻塞 | 这是 async HTTP 入口直接跑同步插件和同步 DB。即使开启了 `defer_processing`，前半段 webhook 插件执行和事件入库也还是同步堵在请求线程上。 |
| 影响面 | API 响应时间、通道 webhook ACK、同进程其他请求 |
| 风险等级 | `P0` |
| 处理建议 | 第一批必须治理。先把 webhook 入口里的同步插件执行和同步入站处理显式下沉；保留现有 ACK 语义，不改对外协议。 |

### 4. 实时对话 WebSocket 链路

| 项目 | 结论 |
| --- | --- |
| 链路 | `realtime_agent_bootstrap_websocket` / `realtime_conversation_websocket` -> `SessionLocal()` -> `while True` -> 同步仓储取 session / seq -> `run_butler_bootstrap_realtime_turn` / `run_conversation_realtime_turn` |
| 证据 | `apps/api-server/app/api/v1/endpoints/realtime.py:38-145`、`apps/api-server/app/api/v1/endpoints/realtime.py:148-257`、`apps/api-server/app/modules/agent/bootstrap_service.py:127-250`、`apps/api-server/app/modules/conversation/service.py:2289-2395` |
| 为什么会阻塞 | WebSocket 是长连接，最怕循环体里塞同步 DB。现在连接建立、事件序号申请、session 快照、turn 准备和部分落库都直接用同步 `Session`。一旦 turn 内部再触发 LLM、编排、记忆写入，事件循环会被长时间占住。 |
| 影响面 | 前端实时聊天、WebSocket 保活、同进程其他实时连接 |
| 风险等级 | `P0` |
| 处理建议 | 第一批必须治理。优先把 WebSocket 循环中的同步 DB 操作和 turn 执行边界切干净，避免“长连接 + 同步仓储 + 长耗时业务”三连击。 |

### 5. 插件异步包装层

| 项目 | 结论 |
| --- | --- |
| 链路 | `aexecute_household_plugin` / `arun_plugin_sync_pipeline` -> `run_in_executor` / `to_thread` -> 在线程池里继续使用调用方传入的 `db: Session` |
| 证据 | `apps/api-server/app/modules/plugin/service.py:961-985`、`apps/api-server/app/modules/plugin/service.py:1454-1554` |
| 为什么会阻塞 | 这条链路比直接阻塞主循环好一点，因为插件执行已经下沉到线程池了；但它把请求作用域 `Session` 直接带进线程池，执行边界还是脏的。线程安全、事务边界、取消和超时都没有统一规则。 |
| 影响面 | 所有依赖 `aexecute_household_plugin` / `arun_plugin_sync_pipeline` 的 async endpoint，例如设备同步、插件调用、部分 AI 配置入口 |
| 风险等级 | `P1` |
| 处理建议 | 第一批纳入公共 helper 设计，但不在阶段 1 直接改业务。阶段 2 要先定清楚“线程池任务谁创建 DB Session，谁提交事务”。 |

### 6. 插件 job worker

| 项目 | 结论 |
| --- | --- |
| 链路 | `PluginJobWorker.run_forever` -> `run_plugin_job_worker_cycle` -> 同步 DB claim / recover -> `publish_plugin_job_updates` -> `asyncio.to_thread(_execute_plugin_job_sync)` |
| 证据 | `apps/api-server/app/modules/plugin/job_worker.py:49-116`、`apps/api-server/app/modules/plugin/job_worker.py:178-245`、`apps/api-server/app/modules/plugin/job_notifier.py:16-35` |
| 为什么会阻塞 | 插件真正执行已经放到线程里，说明这条链路没有前面几条那么烂；但 worker 自己还是裸写 `while not stop_event + wait_for`，没有统一 tick 超时、失败阈值、降级和健康快照。通知广播也在 worker 协程里串行等待。 |
| 影响面 | 插件后台任务吞吐、重试恢复、实时通知稳定性 |
| 风险等级 | `P2` |
| 处理建议 | 第一批只收口运行模板和健康状态，不重写业务流水。它是阶段 2 的重点，不是阶段 1 先动刀的第一现场。 |

### 7. scheduler worker

| 项目 | 结论 |
| --- | --- |
| 链路 | `ScheduledTaskWorker.run_forever` -> `run_scheduled_task_worker_cycle` -> `process_due_schedule_tick` / `process_due_heartbeat_tick` -> `evaluate_heartbeat_rule` -> `build_context_overview` |
| 证据 | `apps/api-server/app/modules/scheduler/worker.py:32-57`、`apps/api-server/app/modules/scheduler/service.py:323-426`、`apps/api-server/app/modules/scheduler/rules.py:29-187` |
| 为什么会阻塞 | scheduler worker 也是 async 外壳包同步 DB / 上下文聚合。虽然它当前主要是“扫到期任务并创建 queued run”，不是直接执行外部插件，但 tick 内部照样在主事件循环里做同步查询和规则评估。 |
| 影响面 | 常驻后台 tick、主循环空闲时间、后端整体抖动 |
| 风险等级 | `P1` |
| 处理建议 | 第一批纳入统一 worker 模板和阻塞 helper。注意一个事实：当前代码里的 `dispatch_task_run` 还没接到常驻 worker 上，所以这轮先治 tick 阻塞，不扩成“重做调度系统”。 |

### 8. 语音 WebSocket 链路

| 项目 | 结论 |
| --- | --- |
| 链路 | `handle_gateway_websocket` -> `SessionLocal()` -> `while True` -> `voice_pipeline_service.handle_inbound_event` -> 同步 DB / 上下文 / 快动作服务 |
| 证据 | `apps/api-server/app/modules/voice/realtime_service.py:46-121`、`apps/api-server/app/modules/voice/pipeline.py:34-118`、`apps/api-server/app/modules/voice/pipeline.py:230-379` |
| 为什么会阻塞 | 这也是长连接循环里混同步 DB 的路子。好的一面是语音运行时 HTTP 已经用了 `httpx.AsyncClient`；坏的一面是身份解析、上下文路由、快动作服务仍然拿同步 `Session` 在 async 链路里跑。 |
| 影响面 | 语音网关 WebSocket、同进程实时能力 |
| 风险等级 | `P1` |
| 处理建议 | 记入首轮盘点，但不放进第一批最先落地的治理范围。先把更直接影响前端连后端的通道和文本实时链路收掉。 |

## 首批治理范围

第一批只做真正会拖死主循环、而且已经能和前端症状对上的链路：

1. 通道轮询 worker
   - 原因：常驻运行、直接同步插件调用、外部网络抖动会持续拖慢整个循环。
2. 通道 webhook async 入口
   - 原因：用户一发消息就会命中，ACK 之前已经在主循环里跑同步插件和同步入站处理。
3. 文本实时 WebSocket 链路
   - 范围：`/realtime/agent-bootstrap`、`/realtime/conversation`
   - 原因：长连接循环里直接混同步 DB 和长耗时 turn 执行，和“前端连不上后端/很慢”最接近。
4. 公共插件异步包装边界
   - 范围：`aexecute_household_plugin`、`arun_plugin_sync_pipeline`
   - 原因：阶段 2 的 helper 不先把这层边界定清楚，后面设备同步、插件调用和 worker 改造都会继续踩同一个坑。
5. scheduler worker 运行模板
   - 原因：当前 tick 仍然在主循环里做同步 DB 和规则评估；虽然不是最危险的外部网络链路，但它是常驻噪音源，必须和统一 worker 模板一起收口。

## 明确暂不纳入第一批的内容

这些不是没问题，而是先不抢第一批资源：

- 插件 job worker 的业务语义重写
  - 原因：插件执行本体已经下沉线程池，主要问题是模板、超时、健康状态，不是最急的主循环硬阻塞。
- 语音 WebSocket 全链路
  - 原因：确实有 async + 同步 DB 混用，但当前用户给出的核心症状更贴近文本实时链路和通道链路。
- 所有零散 async endpoint 的逐个清洗
  - 原因：先抓公共边界和最坏链路，别把范围做炸。

## 盘点后得到的统一坏味道

现在代码里的问题不是一个点，是同一类坏味道在到处复制：

1. `async` 外壳包同步正文
   - 典型例子：channel worker、scheduler worker。
2. 长连接循环里拿同步 `Session`
   - 典型例子：文本实时 WebSocket、语音 WebSocket。
3. 线程池只是临时补丁，没有统一边界
   - 典型例子：`aexecute_household_plugin`、`arun_plugin_sync_pipeline` 把请求作用域 `Session` 带进线程池。
4. worker 自己手写循环，没有统一超时、失败计数和降级
   - 典型例子：三个常驻 worker 都是自己写 `while ... wait_for`。
5. 外部通道故障没有真正隔离
   - 典型例子：Telegram 插件同步 HTTP + `time.sleep()` 重试直接暴露给主事件循环。

## 1.3 是否已经足够支撑后续设计

结论：够了，可以进入阶段 2。

理由很简单：

1. 已经知道“谁在阻塞”
   - 不是抽象概念，是具体到文件和调用链的真实入口。
2. 已经知道“怎么阻塞”
   - 主要就三类：同步 DB、同步 HTTP / SDK、插件执行。
3. 已经知道“先治谁”
   - 首批范围已经锁定，不需要继续无限盘点。
4. 已经知道“先别碰什么”
   - 不需要现在就重写整个调度系统，也不需要把所有 async endpoint 一口气扫完。

阶段 2 设计必须直接围绕这些已确认事实展开，不能再发明大而全的新平台。
