# 20260316-主循环阻塞排障手册

状态：Draft

## 先判断是不是这个问题

先别上来就怪前端、网关或者数据库。按这个顺序看：

1. 前端报“后端连不上”或 WebSocket 明显卡顿时，先看同时段是不是有通道轮询、scheduler、插件任务或实时 turn 在变慢。
2. 如果一个慢任务开始后，别的 HTTP / WebSocket 也一起变慢，那优先怀疑主事件循环被同步阻塞拖住。
3. 如果只有单个 worker 或单个会话变慢，但其他请求还能正常返回，那说明故障已经被压在局部范围内。

## 最该看的信号

### 1. 阻塞调用日志

关键实现：

- `app/core/blocking.py`

重点看这些信号：

- `阻塞调用超时 label=... kind=... timeout_seconds=...`
- `BlockingCallTimeoutError`

看到这类日志，先记三件事：

1. `label` 指向哪条链路。
2. `kind` 是 `sync_db`、`sync_network`、`plugin_code` 还是 `cpu_bound`。
3. 是单点超时，还是连续一段时间都在超时。

### 2. worker 健康快照

当前不是管理接口，先看进程内快照：

- `ChannelPollingWorker.get_health_snapshot()`
- `PluginJobWorker.get_health_snapshot()`
- `ScheduledTaskWorker.get_health_snapshot()`

对应代码：

- `app/modules/channel/polling_worker.py`
- `app/modules/plugin/job_worker.py`
- `app/modules/scheduler/worker.py`

至少要看这几个字段：

- `state`
- `consecutive_failures`
- `last_succeeded_at`
- `last_failed_at`
- `last_duration_ms`
- `last_error_summary`

判断标准很简单：

- `running` 但 `last_succeeded_at` 很久不动：看是不是 tick 被慢调用拖住。
- `degraded` 且 `consecutive_failures` 持续增加：看具体外部依赖是不是已经坏了。
- 只有单个 worker 在抖，API / WebSocket 正常：这是局部故障，不是全局阻塞。

### 3. WebSocket 和 webhook 链路表现

重点文件：

- `app/api/v1/endpoints/realtime.py`
- `app/api/v1/endpoints/channel_gateways.py`

先问两个问题：

1. 慢的是不是单个会话、单个 webhook 请求？
2. 同时段其他 HTTP 请求能不能正常返回？

如果其他请求也一起卡住，说明执行边界可能被人改坏了。

## 快速排查顺序

### 场景 1：WebSocket 变慢或断断续续

1. 看是不是 `/realtime/agent-bootstrap` 或 `/realtime/conversation`。
2. 查同时段有没有 `realtime.*` 相关 `label` 的阻塞调用超时日志。
3. 看是否只有某个 `session_id` 受影响，还是所有请求一起变慢。
4. 如果是文本实时链路，重点复核：
   - 鉴权、快照、事件序号是不是还走 `run_blocking_db(...)`
   - turn 的 prepare / finish / fail 是否还在独立线程内 `Session` 执行

### 场景 2：channel webhook 很慢

1. 看 webhook 路由是否仍通过 `run_blocking_db(...)` 执行 `handle_channel_webhook(...)`。
2. 看慢点是在同步插件执行、同步入站处理，还是第三方通道 HTTP。
3. 同时发一个普通 HTTP 请求；如果普通请求也一起卡，说明边界可能回退了。

### 场景 3：后台任务一抖，全站跟着抖

1. 看 `ChannelPollingWorker`、`ScheduledTaskWorker`、`PluginJobWorker` 的健康快照。
2. 看 `consecutive_failures` 和 `state` 有没有进入 `degraded`。
3. 如果 worker 失败了但 API / WebSocket 还正常，说明隔离还在。
4. 如果 worker 一慢所有请求都跟着慢，优先排查是不是有人又把同步 ORM、同步 HTTP 或插件执行塞回 async 主链路。

不要先启动开发服务器。先跑定向测试。


这组测试验证：

- `run_blocking(...)` / `run_blocking_db(...)` 的超时和线程内 `Session` 边界
- worker 的成功、失败、降级和健康状态
- channel polling / scheduler / plugin job 的边界没有退回裸跑

### 2. 验证 webhook 和文本实时链路

```powershell
C:\Code\FamilyClaw\apps\api-server\.venv\Scripts\python.exe -m unittest `
```

这组测试验证：

## 怎么判断“已经修好”

满足下面 4 条，才算真修好：

1. 慢调用还在时，其他 HTTP / WebSocket 请求能继续响应。
2. 对应 worker 进入 `degraded` 或记录失败，而不是主链路一起拖死。
3. 日志里能从 `label` 直接看出是哪条链路超时。
4. 定向测试能稳定复现并通过，不靠“我本地看着像好了”。

## 当前首批已知残余风险

这轮已经把最危险的入口收住了，但还有两类残余风险要记住：

1. 文本实时 turn 更深层的流式编排内部，仍可能有局部同步 DB 路径。
2. `bootstrap_service.py` 和 `conversation/service.py` 里还留着历史乱码抢救时的死代码占位，行为没坏，但阅读成本高。

这两件事都是真的，但它们现在已经不再等价于“全局事件循环会被直接拖死”。
