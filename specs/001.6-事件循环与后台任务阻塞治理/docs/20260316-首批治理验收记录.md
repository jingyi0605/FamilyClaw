# 20260316-首批治理验收记录

状态：Draft

## 这一步到底验什么

只验一件事：阶段 3 的首批治理，是否真的把风险从“某条慢链路拖死整个事件循环”改成了“局部失败、局部降级、主链路还能继续跑”。

这一步不补新范围，不重做设计，也不假装一次把所有同步逻辑全清干净。

## 核心判断

✅ 值得做，而且已经做到首批目标。

原因很直接：这轮已经把最危险的三类入口收口了。

1. 通道轮询 worker 不再把同步 DB 和插件执行直接挂在 async 外壳里裸跑。
2. 通道 webhook 不再在请求线程里直接跑同步插件和同步入站处理。
3. 文本实时 WebSocket 主循环不再长时间持有同步 `Session`，鉴权、快照、事件序号和 turn 前后写库都改成了显式下沉。

这不是“代码搬家”。因为现在慢调用卡住时，先卡住的是对应线程池任务或对应 worker tick，不是整个主事件循环。

## 这是个真问题还是假问题

是真问题，不是臆想。

盘点阶段已经确认过：

- Telegram 轮询 worker 里混着同步 DB、同步插件执行、同步 HTTP 和重试睡眠。
- webhook async 入口前半段直接跑同步插件和同步入站处理。
- 文本实时 WebSocket 的 `while True` 主循环里直接用了同步 `Session` 和长 turn 链路。

这些东西放在主事件循环里，本来就会拖垮 API 和 WebSocket。现在的治理目标也不是“更优雅”，而是先把最容易拖死全局的点切开。

## 有没有更简单的方法

有，而且这轮已经用了最小做法。

不是重写成全 async，也不是造一个新平台，而是只做这几个收口：

1. 用 `run_blocking(...)` / `run_blocking_db(...)` 明确下沉同步阻塞调用。
2. 在线程池内新建和管理 `Session`，不再把请求作用域 `Session` 生搬进去。
3. 用统一 `WorkerRuntime` 兜住轮询、超时、连续失败和健康状态。
4. 保持 webhook ACK、WebSocket 事件顺序和现有业务语义不变。

再简单就会退回老问题：每条链路继续自己写 `to_thread`、自己带 `Session`、自己赌不会卡事件循环。

## 会破坏什么吗

这轮验收结论是：没有发现对外行为破坏。

明确守住的边界：

- webhook 的响应语义没变，`defer_processing` 还是原来的 ACK 行为。
- 文本实时 WebSocket 仍然按原顺序发送 `session.ready`、`session.snapshot`、`pong`、`user.message.accepted`、`agent.*`。
- worker 语义没有被重写，只是把执行边界、超时和失败隔离收口。

当前还存在的残余风险也要说清楚：

- 文本实时 turn 更深层的 `stream_orchestrated_turn(...)` / `stream_llm(...)` 内部，仍可能存在未完全清洗的同步 DB 路径。这一轮已经把 endpoint 主循环和 turn 前后显式同步 DB 操作收掉，所以风险已经从“拖死整个事件循环”降成“单个 turn 自己变慢”。
- `bootstrap_service.py` 和 `conversation/service.py` 为了绕过历史乱码，暂时保留了死代码占位。它不影响当前执行边界，但接手成本偏高，后面应该单独清掉。

## 验收范围和证据

### 1. 通道 worker 已从全局阻塞改成局部可控

关键代码：

- `apps/api-server/app/modules/channel/polling_worker.py`

已确认的事实：

- worker 已切到 `WorkerRuntimeConfig` + `WorkerRuntime`。
- 轮询账户加载、单账户轮询、结果落库都走 `run_blocking_db(...)`。
- worker 暴露 `get_health_snapshot()`，连续失败和降级状态可观察。

验证证据：

- `tests.test_channel_polling_boundary`
- `tests.test_worker_runtime`

2026-03-16 定向执行结果：

- `C:\Code\FamilyClaw\apps\api-server\.venv\Scripts\python.exe -m unittest tests.test_worker_runtime tests.test_blocking_boundary tests.test_channel_polling_boundary tests.test_scheduler_worker_boundary tests.test_plugin_jobs`
- 结果：`Ran 21 tests ... OK`

这说明通道轮询失败时，影响被收在 worker tick 和健康状态里，不再需要主事件循环替它背锅。

### 2. 调度 / 插件 worker 已有统一失败隔离骨架

关键代码：

- `apps/api-server/app/modules/scheduler/worker.py`
- `apps/api-server/app/modules/plugin/job_worker.py`

已确认的事实：

- scheduler worker 已切到统一 runtime。
- scheduler tick 内部同步 DB 改成了 `run_blocking_db(...)`。
- plugin job worker 已纳入统一运行模板，失败和健康状态不再散落在业务代码里。

验证证据：

- `tests.test_scheduler_worker_boundary`
- `tests.test_plugin_jobs`
- `tests.test_worker_runtime`

这说明外部依赖抖动时，至少 worker 这一层已经能把故障压在本任务范围内，不再默认扩散到 API / WebSocket 主链路。

### 3. 高风险 async 主链路已从主循环裸跑改成显式下沉

关键代码：

- `apps/api-server/app/api/v1/endpoints/channel_gateways.py`
- `apps/api-server/app/api/v1/endpoints/realtime.py`
- `apps/api-server/app/modules/agent/bootstrap_service.py`
- `apps/api-server/app/modules/conversation/service.py`

已确认的事实：

- webhook 入口通过 `run_blocking_db(...)` 执行 `handle_channel_webhook(...)`，请求作用域 `Session` 不再直接承担同步插件和同步 DB 链路。
- WebSocket 鉴权、快照读取、事件序号 claim 都显式下沉。
- 文本实时每次 `user.message` 单独创建一个 turn `Session`，用完就关，不再整条连接长期持有同步 `Session`。
- bootstrap / conversation realtime turn 的 prepare、finish、fail 已拆到线程内 `Session` 执行。

验证证据：

- `tests.test_channel_gateway_api`
- `tests.test_channel_gateway_builtin_deferred`
- `tests.test_realtime_ws`

2026-03-16 定向执行结果：

它验证了一个慢 WebSocket 流式 turn 进行中时，另一个 HTTP 请求仍然能先返回 `200`。这就是“从会拖死全局降到局部可控”的直接证据。

## 结论

首批治理已经满足 `3.4` 的完成条件：

1. 至少一条通道 worker、一条调度或插件 worker、一条 async 主链路都已完成治理。
2. 测试和代码复核都能证明 API / WebSocket 不再被这些高风险链路一起拖慢。

当前更准确的状态不是“完全没有阻塞风险”，而是：

- 全局拖死风险：首批已压住。
- 局部慢链路风险：仍有残余，但范围已经收窄到具体 worker 或具体 turn。

这就够支撑进入阶段 4，开始补更完整的回归测试和排障文档。
