# 20260316-主事件循环边界与Session规则

状态：Draft

## 这是不是个真问题

是，还是已经打到用户脸上的真问题。

现在的坏味道不是“某个函数写得不优雅”，而是 async 入口、WebSocket 长连接和常驻 worker 里直接混了同步 ORM、同步插件执行、同步 HTTP，甚至还有 `time.sleep()`。这会把主事件循环一起拖慢，前端看到的就是 API 卡、WebSocket 抖、偶发连不上后端。

所以这一步不是为了架构好看，是为了先把主链路从慢任务里救出来。

## 有没有更简单的方法

有，而且就该这么做。

不要造新平台，不要重写全站 async，不要先搞一层大而全的“任务中台”。这一步只做两个最小 helper：

1. `run_blocking(...)`
   - 负责把普通同步阻塞调用下沉到线程池。
2. `run_blocking_db(...)`
   - 负责把同步数据库逻辑下沉到线程池，并在线程里自己创建和管理 `Session`。

再简单就会退回老路子：每条链路继续自己写 `run_in_executor`、自己决定带不带 `Session`、自己碰运气处理超时。那不是简单，是放任代码继续烂下去。

## 会破坏什么吗

真正要防的是两个破坏点：

1. 线程池里看不到调用方还没提交的数据。
   - 处理方式：调用方先把需要的数据整理成普通参数，或者先提交事务，再下沉执行。
2. 插件执行超时后线程还在后台跑。
   - 处理方式：先接受“等待方超时返回，但线程不会被强杀”这个现实；这一步的目标是别堵主事件循环，不是一步到位解决所有线程取消问题。

只要守住“不把请求作用域 `Session` 直接带进线程池”这条线，就不会比现在更脏。

## 主事件循环允许做什么

- 参数解析、鉴权、权限判断。
- WebSocket 收发、连接注册、事件类型分发。
- 轻量状态判断和响应拼装。
- 调用已经确认不会阻塞的 async API。

一句话：能很快把控制权还给事件循环的逻辑，才允许留在主线程。

## 哪些逻辑必须下沉

- 同步 ORM 查询、写入、事务提交。
- 同步 HTTP / SDK 调用。
- 同步插件执行。
- 盘点里已经确认会走长链路的规则评估、上下文聚合、turn 正文。
- 长时间 CPU 计算。

一句话：只要会卡线程，就下沉，别争论。

## 最小 helper 边界

### `run_blocking(...)`

- 适用场景：同步网络、同步插件、普通阻塞函数。
- 输入：同步 callable、超时策略、日志标签、可选 executor。
- 输出：原始返回值或受控超时异常。
- 明确不做：不负责数据库事务，不接受调用方 `Session`。

### `run_blocking_db(...)`

- 适用场景：需要同步 `Session` 的数据库读写。
- 输入：一个接收线程内 `Session` 的同步函数、超时策略、是否自动提交。
- 输出：同步函数返回值或受控异常。
- 明确不做：不允许把主线程 `Session` 传进去，不负责跨线程共享 ORM 对象。

## Session 边界规则

- 请求作用域 `Session` 只能留在当前线程使用，不能塞进线程池。
- 线程池里的数据库逻辑必须自己拿 `SessionLocal()` 新建 `Session`。
- 调用方如果需要数据库上下文，只能传普通值，比如 `household_id`、`plugin_id`、配置字典、序列化后的 payload。
- 线程池任务里的提交、回滚、关闭都交给公共 helper，不再让业务代码散着写。

## 这一步明确不做什么

- 不在这一步重写 webhook、WebSocket、scheduler 全链路。
- 不把所有同步库一次性替换成 async 版本。
- 不先碰 plugin job worker 的业务语义。
- 不假装线程池超时就等于线程被取消。
