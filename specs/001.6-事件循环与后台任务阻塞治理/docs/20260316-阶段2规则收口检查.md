# 20260316-阶段2规则收口检查

状态：Draft

## 这一步到底检查什么

只检查一件事：现在的执行边界和 worker 模板，够不够支撑第一批真实阻塞链路的后续改造。

不检查新需求，不扩大范围，不假装要一步做完整个平台。

## 核心判断

✅ 够了，可以继续往 3.1、3.2、3.3 推。

原因很简单：

1. 主事件循环边界已经写死
   - 哪些逻辑允许留在主线程
   - 哪些逻辑必须下沉
   - 线程池里的 `Session` 谁创建、谁提交、谁回滚
2. 最小 helper 已经存在
   - `run_blocking(...)`
   - `run_blocking_db(...)`
3. 三个常驻 worker 的骨架已经统一
   - `ChannelPollingWorker`
   - `PluginJobWorker`
   - `ScheduledTaskWorker`

## 这是不是个真问题

是。

如果没有这一步，后面的 worker 改造就会继续各写各的：有人自己 `wait_for`，有人自己 `sleep`，有人继续把 `Session` 带进线程池。那不是在治理，是在复制问题。

## 有没有更简单的方法

有，已经做了。

不是再造一套任务平台，而是只保留最小公共面：

- 阻塞调用统一下沉
- DB 阻塞统一在线程内自建 `Session`
- worker 循环统一交给 runtime

再往上加层，就是没必要的复杂度。

## 会破坏什么吗

当前明确守住了三条线：

1. 不把请求作用域 `Session` 继续带进线程池
2. 不改变“busy tick 继续跑、idle tick 再等待”的现有语义
3. 不重写 plugin job worker 的业务语义，只收口运行骨架

所以这一步不会破坏现有 API、WebSocket、webhook 的对外行为。

## 还没做什么

这一步还没有开始改下面这些具体高风险链路：

- 通道轮询 worker 的同步插件和同步 HTTP 正文
- 通道 webhook async 入口
- 文本实时 WebSocket 链路
- scheduler tick 内部的同步 DB / 规则评估正文

这是故意的。阶段 2 只负责把工具和边界备齐，不抢阶段 3 的活。
