# Spec 001.6 - 事件循环与后台任务阻塞治理

状态：Draft

## 这份 Spec 要解决什么问题

后端现在有一类很危险的坏味道：把同步数据库访问、同步网络调用、插件执行、外部通道轮询这些慢操作，直接塞进了 `async` 路径或者常驻 worker 里。

这类代码平时可能看起来“能跑”，但只要外部网络一抖、Telegram 轮询变慢、插件执行卡住，主事件循环就会被拖慢。最终前端看到的症状很直接：

1. API 响应明显变慢。
2. WebSocket 建连或保活不稳定。
3. 前端偶发连不上后端，或者连上以后卡很久。

这不是一个“也许以后会遇到”的假问题，而是已经表现到前端连接层面的真实问题。继续在每个地方临时打补丁，只会让问题越来越散。

## 核心判断

✅ 值得做：因为问题已经影响用户可见链路，而且根因不是某一个接口慢，而是执行边界混乱。先把“什么能跑在主事件循环里、什么必须下沉到后台”定清楚，后面才不会反复踩坑。

## 这次 Spec 要产出什么

- 明确主事件循环、后台 worker、线程池下沉三者的职责边界。
- 盘点现有 async 路径、Telegram 轮询、插件任务、调度任务里的阻塞点。
- 设计统一的阻塞调用包装、超时、取消、重试、降级和日志规则。
- 给高风险链路补上可回归的测试和排障文档。

## 这次明确不做什么

- 不在这一轮里重写整个调度系统，也不引入新的“大而全”任务框架。
- 不为了追求“纯异步”把所有同步库一次性全部替换。
- 不修改前端重试策略来掩盖后端主循环阻塞问题。
- 不在没有盘点清楚现状前，先拍脑袋上一个复杂事件总线。

## 当前判断的关键洞察

- 数据结构：真正需要治理的不是“某个 Telegram 函数”，而是“请求入口、事件循环、后台任务、外部依赖”之间的执行边界。
- 复杂度：如果每个 worker 都自己决定是否 `await`、是否开线程、是否重试，那就是烂设计，必须收口成统一规则。
- 风险点：最大的破坏性风险不是任务失败，而是后台慢任务把 API 和 WebSocket 一起拖死。

## 主要文档

- `requirements.md`：写清楚为什么做、做什么、什么算完成。
- `design.md`：写清楚执行边界、统一 worker 结构、错误处理和验证方案。
- `tasks.md`：把后续工作拆成可直接执行的阶段任务。
- `docs/README.md`：约定后续排查记录、验证记录和补充文档放哪里。
