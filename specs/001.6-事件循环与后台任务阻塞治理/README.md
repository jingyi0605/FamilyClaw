# Spec 001.6 - 事件循环与后台任务阻塞治理

状态：IN_REVIEW（首轮验收已完成，残余风险已记录）

## 这份 Spec 现在解决到了哪

这轮首批治理已经收住了最容易拖死主链路的几个点：

1. 通道轮询 worker
2. 通道 webhook async 入口
3. 文本实时 WebSocket 链路
4. 公共插件异步包装边界
5. scheduler worker 运行模板

现在的结论不是“项目里再也没有阻塞风险”，而是：

- 最危险的全局拖死风险，首批已经压住。
- 还剩下的风险，已经收窄到具体 worker、具体 turn 或后续单独任务。

## 核心判断

✅ 值得做，而且首轮目标已经达成。

原因很直接：

- 以前是 async 主链路和后台 worker 直接混着跑同步 ORM、同步插件、同步 HTTP。
- 现在已经明确分出主事件循环边界，并把高风险同步调用收口到统一 helper 和统一 worker 模板里。
- 验证证据已经能说明“慢任务影响自己，不再默认拖死 API / WebSocket”。

## 这轮明确不做什么

- 不重做整个调度系统
- 不重写 plugin job worker 的业务语义
- 不把所有同步库一次性改成 async
- 不把所有零散 async endpoint 全部顺手重构

## 首轮验收结论

需求、设计、任务、测试和补充文档已经能一一对应。

已治理并有明确证据的路径：

- 通道轮询 worker
- scheduler / plugin worker 运行模板
- channel webhook async 入口
- 文本实时 WebSocket 链路

已明确记录的残余风险：

- 文本实时 turn 更深层的流式编排内部，仍可能有局部同步 DB 路径
- `bootstrap_service.py` 和 `conversation/service.py` 里还有历史乱码抢救留下的死代码占位，行为没坏，但后续应该清理
