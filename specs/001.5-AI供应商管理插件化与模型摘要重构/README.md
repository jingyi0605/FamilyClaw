# Spec 001.5 - AI供应商管理插件化与模型摘要重构

状态：Blocked

## 这份 Spec 要解决什么问题

现在的 `user-app` AI 供应商管理页有两个根本问题：

1. 页面把所有配置平铺出来，信息很多，但真正常用的只有“看当前有哪些模型”和“知道它们怎么路由”。
2. 供应商配置虽然看起来像动态表单，实际上还是把能力写死在一堆前后端逻辑里。每多一个供应商，就要多补一轮字段、文案和流程。

这不是界面不好看这么简单，而是数据结构没立住。页面、供应商元数据、模型表单、LLM 调用流程说明，本来就该围着同一份“供应商插件描述”转。

## 这次改造做了什么

- 把 `user-app` AI 供应商管理页收口成“供应商列表 + 配置摘要”，不再在主页面平铺大段表单。
- 把新增/编辑模型改成弹窗流程。新增时先选供应商插件，再根据插件提供的字段渲染动态表单。
- 给供应商补上插件元数据：`plugin_id`、`plugin_name`、`supported_model_types`、`llm_workflow`、`field_schema`。
- 后端把 AI provider adapter 改成“内置注册表 + 外部 JSON 插件文件”的模式，允许继续扩展新供应商。
- 页面上明确展示供应商或模型支持类型：`LLM`、`嵌入`、`视觉`、`语音`、`生图`，并展示对应 `llm_workflow`。

## 这次明确不做什么

- 不把 AI 供应商接到通用插件市场或插件安装流里，这一轮只做 AI provider 专用插件化。
- 不重做现有 AI route 的存储模型，这次只补供应商元数据展示和新增流程。
- 不为了这次改造去推翻现有 `AiProviderProfile` 数据结构，旧配置要能继续读。
- 不在这次里强行完成线上联调。没有有效登录和有效 OpenAI key，联调就是假把式。

## 当前结论

- 核心开发已经完成，代码层面的结构改造已经落地。
- 还不能说“整件事百分之百闭环”，因为当前还卡着三件事：
  - `user-app` 全量类型检查被工作区里其他页面的编译错误拦住了，不是这次 AI 供应商页面自身的直接报错。
  - 页面联调需要有效登录账号，当前测试凭据不可用。
  - 一条依赖真实 OpenAI 调用的后端流程测试会因为无效 key 失败。

## 主要文档

- `requirements.md`：说明这次到底要交付什么能力，什么算完成。
- `design.md`：说明插件元数据、前后端结构、动态表单流程和兼容策略。
- `tasks.md`：说明哪些任务已经做完，哪些验证还卡着，卡点是什么。
- `docs/README.md`：记录实际验证命令、现存阻塞和建议验收步骤。
