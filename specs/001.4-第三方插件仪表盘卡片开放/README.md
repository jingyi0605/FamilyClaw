# Spec 001.4 - 第三方插件仪表盘卡片开放

状态：IN_REVIEW

## 这份 Spec 解决什么问题

现在 `user-app` 首页已经有一套仪表盘卡片，但它本质上还是前端写死的页面结构，不是真正可扩展的开放机制。

- 第三方插件现在能接入后端能力、配置协议和插件管理页，但还不能把自己的结果稳定展示到首页
- 首页卡片类型和渲染逻辑写死在前端，后面每加一种插件卡片都得改主应用代码
- 如果直接让第三方塞前端组件、HTML 或脚本进首页，安全、性能、兼容和审核都会失控

这份 Spec 的目标很明确：给插件系统补一条正式的“首页卡片开放链路”，让第三方插件可以声明卡片、产出卡片快照、接入统一布局和统一渲染，同时不破坏现有首页和插件体系。

## 这次要做到什么

- 给插件 manifest 增加正式的仪表盘卡片声明
- 给后端增加卡片快照和成员布局的正式数据模型
- 给 `user-app` 首页增加统一的卡片聚合接口和统一渲染入口
- 让第三方插件只能走受控模板，不允许把任意前端代码直接塞进首页
- 保证插件卡片失败、超时、禁用时，只影响自己，不拖垮整个首页

## 这次明确不做什么

- 不开放第三方直接注入 React 组件、HTML、CSS、JavaScript
- 不做插件自定义前端运行沙箱
- 不顺手重做整个首页视觉和所有内置卡片
- 不在第一版支持所有页面位置，只先支持 `user-app` 首页
- 不把插件市场、插件配置、插件详情抽屉的全部历史问题一次性打包解决

## 阶段 1 已经定死的边界

- manifest 里的首页卡片正式入口叫 `dashboard_cards`，不是首页代码里的手写注册表
- 第一版卡片位置只允许 `home`
- 第一版模板只允许 `metric`、`status_list`、`timeline`、`insight`、`action_group`
- 第一版动作白名单只允许 `navigate`、`open_plugin_detail`、`trigger_plugin_action`
- `max_items` 只允许列表类卡片声明
- `dashboard_cards` 里的未知字段一律拒绝，不接受“先吞掉以后再补”
- 第三方插件依然不允许注入 React 组件、HTML、CSS、JavaScript

## 为什么现在做

因为插件系统已经有了 manifest、配置协议、配置实例和插件管理页基础，如果首页还是硬编码卡片，插件生态永远只能停在“后台能跑，前台看不见”的半成品状态。

继续拖着不做，后面只会出现三种烂问题：

- 每种插件展示都临时加一个首页特例，越写越散
- 第三方插件无法形成完整产品闭环，接入价值很弱
- 一旦有人提议“直接让插件带 UI”，系统会被推向一条很难收回来的坏路

## 主要文档

- `requirements.md`：说清楚为什么开放、开放到什么边界、什么算完成
- `design.md`：说清楚卡片声明、卡片快照、布局、接口和渲染边界怎么落
- `tasks.md`：说清楚先收口协议，再建后端，再接前端，再验收的顺序
- `docs/README.md`：给后续接口样例、迁移说明、验收记录留位置
