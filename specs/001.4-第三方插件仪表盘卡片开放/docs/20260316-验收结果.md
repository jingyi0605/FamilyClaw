## 这次已经形成的闭环

### 1. manifest 协议已经落地

- 插件 manifest 已支持正式字段 `dashboard_cards`
- 第一版卡片位置、模板、尺寸、刷新策略、动作白名单已经在后端 schema 收口
- 未知字段、非法模板、非法动作、越界 payload 会被拒绝

### 2. 后端主链路已经打通

- 已新增插件卡片快照表和成员首页布局表
- 已实现插件卡片快照写入、错误记录、首页聚合读取、布局保存
- 首页聚合会把内置卡片和插件卡片统一收口成 `HomeDashboardRead`
- 单张插件卡片坏掉时只影响自己，不会把整个首页接口打崩

### 3. 前端统一卡片模型已经接上

- `user-app` 首页已经改为消费统一卡片模型
- H5 首页已经按受控模板渲染 `metric`、`status_list`、`timeline`、`insight`、`action_group`
- RN 首页已经移除对旧首页共享数据结构的依赖，改为消费统一卡片模型
- 首页布局顺序和显隐不再只靠本地缓存，而是走后端布局接口

### 4. 示例插件链路已经跑通

- `health-basic-reader` manifest 已声明首页卡片 `daily-steps-summary`
- 插件 connector 现在会返回受控 `dashboard_snapshots`
- 插件同步流水线跑完后会把示例卡片快照写入正式表
- 首页聚合接口能把这张插件卡片读出来并返回给前端

## 现在还剩什么风险

### 1. 还没有做开发服务器级手工回归

这次遵守约束，没有主动启动开发服务器。

这意味着：

- 我能确认类型、接口和测试主链路是通的
- 但还没有做浏览器里的真实拖拽、点击、页面联调截图

### 2. `trigger_plugin_action` 目前还是保守降级

现在前端动作分发里：

- `navigate` 已接站内跳转
- `open_plugin_detail` 已接受控入口
- `trigger_plugin_action` 还是提示性占位，不执行任意插件前端逻辑

这是刻意保守，不是漏做。原因很简单：这次要守住“声明式卡片 + 后端快照 + 统一渲染器”的边界，不能为了图快把任意执行口子重新打开。

### 3. 示例插件只有一个

当前已经证明链路不是纸面设计，但仍然只用 `health-basic-reader` 做了样板。后续如果再接第二个、第三个插件，主要是复用现在这套协议，不该再扩平行体系。

## 结论

这份 Spec 现在已经不是“只写了协议和表结构”的半成品了。

它已经具备下面这条完整链路：

1. 插件在 manifest 里声明首页卡片
2. 后端按受控协议校验声明和快照
3. 插件执行后把卡片快照写入正式表
4. 首页统一接口聚合内置卡片、插件卡片和成员布局
5. `user-app` 用统一渲染器展示结果
6. 单卡失败时只影响自己

当前状态可以进入评审和手工联调，不需要再回头改协议方向。
