# 需求文档 - 第三方插件仪表盘卡片开放

状态：Draft

## 简介

`FamilyClaw` 现在已经有正式插件系统，也已经有用户端首页仪表盘，但这两件事还没有真正连起来。

今天的问题不是“首页有没有卡片”，而是“插件能不能以不破坏主应用的方式，把自己的结果展示成首页卡片”。如果继续靠主应用硬编码，每新增一个插件卡片都要改首页；如果反过来让第三方直接往首页塞前端代码，那就是把安全和维护成本一起送走。

这份需求文档只解决一件事：让第三方插件能在 `user-app` 首页开发和接入仪表盘卡片，但整个链路必须是受控、可审核、可降级、可维护的。

## 术语表

- **System**：`FamilyClaw` 系统本身，包含 `api-server`、`user-app`、插件运行时和数据库
- **插件卡片定义（Card Definition）**：插件在 manifest 中声明的卡片元数据，说明这张卡片叫什么、放哪里、用什么模板渲染
- **插件卡片快照（Card Snapshot）**：插件在某个时间点产出的展示数据，供首页直接读取，不在首屏请求时临时执行插件
- **成员布局（Member Layout）**：某个成员在首页里看到的卡片顺序、显隐和尺寸偏好
- **受控模板（Controlled Template）**：系统内置的有限卡片模板，插件只能提供数据，不能提供任意前端代码

## 范围说明

### In Scope

- 插件 manifest 新增首页卡片声明协议
- 后端存储和读取插件卡片快照
- 首页统一聚合内置卡片和插件卡片
- 成员级首页布局持久化
- 插件卡片的权限、禁用、失败和降级规则

### Out of Scope

- 第三方自定义 React 组件或任意脚本执行
- 首页以外页面位置的卡片开放
- 插件市场 UI 重做
- 通用前端沙箱平台
- 卡片级富文本编辑器、拖拽设计器

## 需求

### 需求 1：插件可以正式声明首页卡片

**用户故事：** 作为插件开发者，我希望在插件 manifest 里正式声明可用的首页卡片，这样系统能稳定识别、审核和展示我的卡片，而不是靠首页写死特例。

#### 验收标准

1. WHEN 插件需要在首页展示内容 THEN System SHALL 允许它在 manifest 中声明卡片定义，而不是要求主应用手写注册表。
2. WHEN 系统加载插件 manifest THEN System SHALL 校验卡片定义是否合法，包括位置、模板类型、尺寸、标题和动作声明。
3. WHEN 插件没有声明首页卡片 THEN System SHALL 继续按普通插件处理，不因为缺少卡片声明而判定插件非法。
4. WHEN 第三方插件试图声明不支持的模板、位置或任意前端代码 THEN System SHALL 拒绝该声明并给出明确错误。

### 需求 2：首页通过统一接口读取插件卡片

**用户故事：** 作为家庭成员，我希望首页像正常产品页面一样直接看到插件卡片，而不是等插件临时执行后才慢慢拼出来，这样首页才能稳定、快和可预期。

#### 验收标准

1. WHEN 用户进入 `user-app` 首页 THEN System SHALL 通过统一接口返回首页要展示的内置卡片、插件卡片和当前成员布局。
2. WHEN 某张插件卡片需要展示数据 THEN System SHALL 优先读取已生成的卡片快照，而不是在首页请求里同步执行插件。
3. WHEN 某个插件卡片快照过期、缺失或生成失败 THEN System SHALL 只让该卡片进入降级态，不得拖垮整个首页。
4. WHEN 插件被停用、对当前家庭不可见或权限不足 THEN System SHALL 自动从首页聚合结果中移除对应卡片。

### 需求 3：插件卡片必须走受控模板和受控动作

**用户故事：** 作为产品和平台维护者，我希望第三方插件只能用系统认可的模板和动作，这样首页的安全、兼容和体验不会被插件各搞一套。

#### 验收标准

1. WHEN 插件卡片被渲染 THEN System SHALL 只允许使用系统内置模板，不允许插件直接提供 React 组件、HTML、CSS 或 JavaScript。
2. WHEN 插件卡片声明交互动作 THEN System SHALL 只允许受控动作类型，例如站内跳转、打开插件详情或触发受控插件 action。
3. WHEN 插件卡片输出超出模板允许范围的数据 THEN System SHALL 拒绝该快照或按降级态展示，而不是原样透传到前端。
4. WHEN 首页视觉或前端框架后续演进 THEN System SHALL 允许系统只改统一渲染器，而不要求第三方插件逐个改 UI 代码。

### 需求 4：成员可以保存自己的首页布局

**用户故事：** 作为家庭成员，我希望决定首页卡片的顺序和显隐，这样首页更贴近我自己的使用习惯，而不是所有人一模一样。

#### 验收标准

1. WHEN 成员调整首页卡片顺序或显隐 THEN System SHALL 将结果保存为成员级布局，而不是只写浏览器本地缓存。
2. WHEN 首页同时存在内置卡片和插件卡片 THEN System SHALL 用统一布局模型管理它们，而不是两套规则并存。
3. WHEN 布局里包含已删除、已停用或当前不可见的插件卡片 THEN System SHALL 在读取时自动清理无效项。
4. WHEN 成员从未保存过布局 THEN System SHALL 返回一份稳定默认布局，不得因为插件动态接入而导致首页顺序随机漂移。

### 需求 5：平台必须保证插件卡片不会拖垮首页

**用户故事：** 作为系统维护者，我希望插件卡片就算出问题也只影响自己，这样第三方插件不会把首页主链路一起带崩。

#### 验收标准

1. WHEN 插件卡片生成失败、超时或返回非法数据 THEN System SHALL 记录明确状态和错误原因，并让该卡片进入可见降级态或被安全忽略。
2. WHEN 首页聚合插件卡片 THEN System SHALL 对单卡大小、字段数量和动作数量有硬限制，避免异常插件撑爆首屏数据。
3. WHEN 插件卡片刷新频率过高 THEN System SHALL 通过快照过期策略、后台任务或缓存限制刷新成本，不得让首页请求直接承担刷新压力。
4. WHEN 平台排查首页问题 THEN System SHALL 能明确区分是内置卡片问题、插件卡片问题还是布局数据问题。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 用户进入首页 THEN System SHALL 以聚合读取为主，不在首屏同步执行插件主逻辑。
2. WHEN 首页返回卡片列表 THEN System SHALL 支持按卡片级降级，避免单个插件卡片导致整体接口明显变慢。

### 非功能需求 2：可靠性

1. WHEN 插件卡片快照缺失或损坏 THEN System SHALL 返回明确的降级状态，而不是让首页接口报 500。
2. WHEN 插件被禁用、卸载或升级 manifest THEN System SHALL 在后续聚合中自动处理失效卡片，不要求手工清理数据库垃圾。

### 非功能需求 3：可维护性

1. WHEN 后续新增卡片样式 THEN System SHALL 通过扩展受控模板实现，而不是为某个插件加首页特例。
2. WHEN 后续接入更多第三方插件 THEN System SHALL 继续复用现有插件 manifest、配置协议和插件管理链路，不另起一套首页专用插件系统。

## 成功定义

- 第三方插件可以在不改首页业务代码的前提下，声明并接入自己的首页卡片
- 首页能稳定聚合内置卡片和插件卡片，且插件卡片故障不会拖垮首屏
- 成员布局从本地临时缓存升级为正式后端数据，并能兼容插件卡片动态增减
- 平台明确守住“声明式数据开放，不开放任意前端执行”这条边界
