# 需求文档 - 小爱音箱全插件化迁移
状态：Draft

## 这次要解决什么问题

`005.4` 已经把 Home Assistant 设备同步和设备控制主链路推进到“实例 + 插件”驱动，设备页和控制链路也已经围绕正式 `plugin_id` 和 `integration_instance_id` 运行。

但小爱音箱还停在旧世界里：

- 发现走的是专用 `voice-terminals/discoveries` 接口
- 绑定走的是专用 `claim` 逻辑
- 设备绑定没有完整进入正式的实例驱动模型
- 后续开发很容易分不清“这是正式插件链路”还是“这是小爱特例”

这会直接带来三个问题：

1. 平台已经有一套正式的插件、实例、设备绑定、设备控制链路，但小爱没有走进去
2. 集成页和设备管理页会同时存在“正式入口”和“小爱特例入口”，后续改动很容易越改越乱
3. 只要旧代码继续留着，后面再接别的音箱、别的终端、别的设备插件时，就会再次复制特例

这次的目标很明确：把小爱音箱的发现、绑定、控制全部迁到标准插件链路里，平台只保留它本来就该保留的 `speaker` 控制协议和声纹管理能力，不再允许例外入口长期存在。

## 术语说明

- **集成实例**：用户在平台里添加的一条插件实例配置，例如“客厅小爱网关”“家里 HA”
- **设备插件**：通过正式 `manifest + connector + action` 进入平台的插件实现
- **发现**：平台或插件识别到一个还没被正式添加的外部设备
- **绑定**：把外部设备和本地 `Device + DeviceBinding` 建立正式关系
- **标准插件控制链路**：设备动作从平台统一入口进入，经设备路由找到正式 `plugin_id`，再由插件执行
- **平台能力**：仍然保留在平台核心而不是下沉到插件的通用逻辑，例如 `speaker` 动作协议、设备页、声纹管理
- **旧小爱专用链路**：当前 `voice discovery / claim / open_xiaoai` 这些没有完整走实例化插件主链的历史代码

## 范围说明

### In Scope

- 把小爱音箱的发现流程迁到正式插件链路
- 把小爱音箱的绑定和设备添加流程迁到正式插件链路
- 把小爱音箱的设备控制迁到正式插件控制链路
- 保留并复用平台现有的 `speaker` 控制协议和声纹管理逻辑
- 改造实例创建、设备发现、设备添加的插件选择逻辑，不能继续硬编码只认某一个插件
- 在迁移完成后删除旧的小爱专用实现和旧入口
- 补齐能证明迁移完成的回归测试和清理验证

### Out of Scope

- 这次不重做整个语音对话主链、ASR、TTS 或声纹算法
- 这次不新增超出当前范围的小爱设备新能力，例如全新的媒体控制协议或小爱技能系统
- 这次不做新的插件市场或全新集成页视觉改版
- 这次不要求第一天就支持所有未来插件的全部能力，但架构上必须允许它们进入同一条实例与设备添加主链

## 需求

### 需求 1：小爱音箱必须通过正式插件实例进入平台

**用户故事**
作为维护平台的人，我希望小爱音箱和 HA 一样先对应一个正式插件实例，这样后面的发现、添加、控制都能挂在统一模型上，而不是继续靠单独接口偷跑。

#### 验收标准

1. WHEN 用户在集成页查看可添加插件 THEN System SHALL 能把小爱音箱插件作为正式插件展示出来
2. WHEN 用户创建小爱音箱插件实例 THEN System SHALL 生成正式的 `integration_instance`
3. WHEN 小爱音箱后续进入发现、添加、控制流程 THEN System SHALL 基于该实例继续，而不是绕开实例模型

### 需求 2：小爱音箱发现流程必须走正式插件发现链路

**用户故事**
作为平台开发者，我希望小爱音箱的“待添加设备”来自正式插件发现链路，而不是留一套只有小爱自己能用的专用 discovery 接口。

#### 验收标准

1. WHEN 插件实例发起发现或读取候选设备 THEN System SHALL 通过正式插件 connector 获取或读取小爱音箱候选项
2. WHEN 平台展示待添加的小爱音箱 THEN System SHALL 能把这些候选项放入统一的集成发现或候选设备模型里
3. WHEN 其他插件后续也实现同样的发现能力 THEN System SHALL 不需要新增硬编码分支才能进入同一入口
4. WHEN 服务重启或插件重新加载 THEN System SHALL 不依赖易失内存作为唯一发现来源

### 需求 3：小爱音箱绑定与设备添加必须走正式绑定链路

**用户故事**
作为平台开发者，我希望用户添加小爱音箱时，得到的是一个标准 `Device + DeviceBinding`，并且绑定里有正式 `plugin_id` 和 `integration_instance_id`，这样设备页、控制页、审计链路才能一致。

#### 验收标准

1. WHEN 用户确认添加某台小爱音箱 THEN System SHALL 通过正式插件同步或绑定逻辑创建设备
2. WHEN 绑定创建完成 THEN System SHALL 写入正式 `plugin_id`
3. WHEN 绑定创建完成 THEN System SHALL 写入正式 `integration_instance_id`
4. WHEN 后续设备资源列表展示这台音箱 THEN System SHALL 能在统一集成资源列表里看到它
5. WHEN 设备已经绑定 THEN System SHALL 不再要求调用旧的小爱专用 claim 入口

### 需求 4：小爱音箱控制必须走标准插件控制链路

**用户故事**
作为平台开发者，我希望小爱音箱的开关、暂停、音量等动作和其他设备一样都从统一设备控制入口进去，而不是继续保留例外路径。

#### 验收标准

1. WHEN 平台对小爱音箱执行设备动作 THEN System SHALL 走统一设备动作入口和设备插件路由
2. WHEN 系统根据绑定路由插件 THEN System SHALL 通过正式 `plugin_id` 找到小爱插件，而不是靠 `platform == open_xiaoai` 这种特例判断
3. WHEN 小爱插件执行动作 THEN System SHALL 只负责把平台定义好的 `speaker` 动作翻译到实际终端或网关
4. WHEN 动作成功或失败 THEN System SHALL 继续复用平台现有的统一操作记录与结果回传方式

### 需求 5：平台必须继续保留 speaker 与声纹的核心能力

**用户故事**
作为平台开发者，我不希望这次迁移把平台已经具备的 `speaker` 通用动作协议和声纹管理逻辑拆碎到插件里，导致相同能力被每个插件各写一套。

#### 验收标准

1. WHEN 小爱音箱完成插件化迁移 THEN System SHALL 继续复用平台现有的 `speaker` 动作协议定义
2. WHEN 小爱音箱进入设备页和控制页 THEN System SHALL 继续复用平台现有的设备页、设备日志、动作协议结构
3. WHEN 用户使用声纹管理 THEN System SHALL 继续通过平台现有声纹管理链路完成，而不是迁到插件内部
4. WHEN 插件需要支持声纹相关设备上下文 THEN System SHALL 通过正式绑定和平台接口协作，而不是复制一套声纹业务实现

### 需求 6：插件选择与实例添加逻辑不能硬编码

**用户故事**
作为平台架构维护者，我希望集成页、实例创建、设备发现、设备添加这些入口面对的是“所有已注册并声明支持的插件”，而不是继续写死“这里只有 HA”“这里只有小爱”。

#### 验收标准

1. WHEN 平台列出可添加的集成插件 THEN System SHALL 基于已注册插件和 manifest 能力决定，而不是硬编码插件 id 白名单
2. WHEN 平台执行某个实例的发现或同步动作 THEN System SHALL 按插件能力分发，而不是在服务层写死只支持单一插件
3. WHEN 新插件以后声明同类能力 THEN System SHALL 不需要复制小爱专用或 HA 专用流程
4. WHEN 某个插件不支持某项能力 THEN System SHALL 明确禁用该动作，而不是通过硬编码排除其他插件

### 需求 7：旧小爱专用代码必须在迁移完成后删除

**用户故事**
作为后续接手的人，我希望仓库里只剩一条清晰主链，而不是看见新旧两套入口同时存在，分不清哪套还能用。

#### 验收标准

1. WHEN 小爱插件化主链完成 THEN System SHALL 删除旧的小爱专用发现、绑定、控制入口与无效前端调用
2. WHEN 删除旧代码后 THEN System SHALL 不影响新的正式插件链路
3. WHEN 开发者在仓库里搜索小爱添加主链 THEN System SHALL 只看到正式插件链路，而不是同时看到历史特例和新实现
4. WHEN 保留极少量兼容层确有必要 THEN System SHALL 在 Spec 和代码里明确写清保留原因、退出条件和移除时间点

## 非功能需求

### 非功能需求 1：架构一致性

1. WHEN 设备进入平台 THEN System SHALL 尽量复用现有实例、绑定、设备控制、设备页模型，而不是再造一套平行结构
2. WHEN 新插件后续接入同类能力 THEN System SHALL 能沿用这次抽出来的通用入口

### 非功能需求 2：可维护性

1. WHEN 小爱迁移完成 THEN System SHALL 不留下“名字像插件，实则是特例服务”的代码
2. WHEN 后续开发者排查问题 THEN System SHALL 能从实例、绑定、插件、设备动作四个维度顺着一条主链追下去

### 非功能需求 3：可验证性

1. WHEN 迁移完成 THEN System SHALL 有测试覆盖实例创建、发现候选、正式绑定、设备控制、旧代码清理这几条关键路径
2. WHEN 涉及数据库结构调整 THEN System SHALL 严格通过 Alembic migration 落地并完成 PostgreSQL 升级验证

## 成功定义

- 小爱音箱和 HA 一样成为正式插件实例驱动的集成
- 小爱音箱的发现、添加、绑定、控制不再依赖专用旧接口
- 平台继续保留 `speaker` 通用动作协议和声纹管理能力，不把业务碎片化到插件里
- 集成页和后端服务不再硬编码只支持某一个设备插件
- 旧的小爱专用链路在迁移完成后被明确删除，不再误导后续开发
