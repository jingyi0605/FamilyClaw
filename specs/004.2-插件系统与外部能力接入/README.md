# Spec 004.2 - 插件系统与外部能力接入

状态：Draft

## 这份 Spec 解决什么问题

项目下一步要接智能家居、健康、陪伴、教育这些能力。问题不在于先做哪一个，而在于如果每接一个能力都直接改主系统，代码很快就会乱，数据格式也会越来越散。

这份 Spec 先解决一个更基础的问题：给项目加一层清楚、可控的插件机制，让外部能力可以接进来，并且把有用信息写进家庭记忆，给 Agent 使用。

## 这次要做到什么

- 系统知道“什么是插件”，并能识别插件提供的能力
- 插件执行对外先走任务创建，不再把同步跑完插件当默认接口语义
- 外部数据接进来后，先保留原始记录，再写成统一的家庭记忆
- Agent 可以调用插件提供的读数据、写动作、领域工具能力
- 高风险动作默认不能自动执行，必须走权限和确认

## 这次明确不做什么

- 不做插件市场
- 不做随便执行任意脚本的开放平台
- 不做高风险全自动控制
- 不要求第一版就接完所有第三方平台

## 为什么现在做

因为后面无论是智能家居、健康分析、陪伴提醒还是教育规划，最终都绕不开两件事：

- 接更多外部信息
- 把信息沉淀进家庭记忆，或者触发受控动作

这层底子不先收好，后面每条业务线都会自己造一套接入逻辑。

## 现在的主链路怎么理解

这里要把一句大实话写清楚：

- `execute_plugin()`、`run_plugin_sync_pipeline()` 这些函数还存在，但它们现在主要是内部执行层和测试层工具
- 对外接口、Agent 调用入口、动作确认入口，主路径已经改成“先创建插件任务，再由后台 worker 执行”

也就是说，现在系统对外更像这样：

1. 调用方提交插件请求
2. 系统创建 `plugin_job`
3. worker 领取任务并执行插件
4. 调用方通过任务查询、通知和响应接口继续跟进结果

如果后续文档还把“调接口 = 同步拿执行结果”当默认模型，那就是错的。
