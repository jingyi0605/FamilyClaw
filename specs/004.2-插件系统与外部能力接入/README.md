# Spec 004.2 - 插件系统与外部能力接入

状态：Draft

## 这份 Spec 解决什么问题

项目后面会持续接入智能家居、健康、教育、陪伴、通讯平台这些外部能力。真正的问题不是“先接哪一个”，而是如果每接一个能力都直接改主系统，最后一定会烂成一锅粥。

这份 Spec 只做一件基础事：

- 把“外部能力怎么接进来、怎么被系统调用、怎么把结果写回系统”收口成正式插件机制

## 这次要做到什么

- 系统知道什么是插件，并能识别正式插件类型
- 对外执行先走 `plugin_job`，不再把“同步跑完插件”当默认接口语义
- 外部数据先保留原始记录，再转成统一家庭记忆
- Agent 和动作链路复用统一插件能力模型
- 后续扩展类型有正式落点，而不是继续搞页面特例

## 现在已经发生的事实变化

这份 Spec 不能再假装系统里只有最早那几类插件了。

当前正式体系里已经存在或已经落地到设计里的类型包括：

- `connector`
- `memory-ingestor`
- `action`
- `agent-skill`
- `locale-pack`
- `channel`
- `region-provider`

其中：

- 前四类是主执行链路里的核心能力
- `locale-pack` 是资源型插件
- `channel` 是通讯通道扩展
- `region-provider` 是地区目录扩展

## 这次明确不做什么

- 不做插件市场
- 不做远程任意代码安装
- 不做开放脚本执行平台
- 不把所有插件配置细节重新写一遍

插件配置协议已经由：

- `../004.2.3-插件配置协议与动态表单/`

单独承接。这里不再发明第二套。

## 现在的主链路怎么理解

有一句话必须写死：

- `execute_plugin()`、`run_plugin_sync_pipeline()` 还在，但它们主要是内部执行层和测试工具
- 对外主路径已经是“创建插件任务 -> worker 执行 -> 查询状态 / 等通知 / 走确认”

也就是说，现在系统对外更像这样：

1. 调用方提交插件请求
2. 系统创建 `plugin_job`
3. worker 领取任务并执行
4. 调用方通过任务状态继续跟结果

如果后续文档还把“调接口 = 同步拿到插件最终结果”当默认模型，那就是旧文档没跟上实现。
