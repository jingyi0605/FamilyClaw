# 需求文档 - 插件系统与外部能力接入

状态：Draft

## 简介

这次不是直接做一个个新功能，而是先把扩展底子搭起来。

目标很直接：以后不管接智能家居、健康平台、教育信息源，还是给 Agent 加新工具，都走同一套插件入口；外部数据进来以后，也走同一套家庭记忆写入方式。

这样做有两个好处：

- 后面加新能力时，不用每次都改主链路
- Agent 看到的数据结构更统一，不会一会儿一个格式

## 术语表

- **插件**：按统一规则接进系统的一块扩展能力。
- **家庭记忆**：系统长期保存的家庭成员、设备、事件、观察记录、提醒和偏好等信息。
- **原始记录**：插件从外部拿到的原始数据，先保留，方便排错和重算。
- **标准记忆**：把原始记录转成系统统一结构后的数据，给 Agent 和业务逻辑使用。
- **主动控制**：系统不只是给建议，还会去执行动作，比如开灯、发提醒、创建设备指令。

## 范围说明

### In Scope

- 定义插件的最小类型和最小边界
- 定义外部数据进入家庭记忆的统一入口
- 定义 Agent 如何使用插件能力
- 定义插件权限和主动控制边界
- 规划两个样板插件，验证整条链路可用

### Out of Scope

- 插件市场和插件分发平台
- 复杂流程编排平台
- 任意第三方脚本执行
- 门锁、摄像头、支付这类高风险能力的自动执行

## 需求

### 需求 1：系统必须用统一方式识别插件能力

**用户故事：** 作为项目维护者，我希望每个插件都按统一规则声明自己能做什么，这样系统才能稳定加载、校验和调用。

#### 验收标准

1. WHEN 一个插件接入系统 THEN System SHALL 能识别它的基本信息、版本、能力类型和所需权限。
2. WHEN 一个插件缺少必要声明 THEN System SHALL 拒绝启用它，并给出明确原因。
3. WHEN 系统查询当前可用能力 THEN System SHALL 能区分读数据、执行动作、记忆写入和 Agent 工具这几类能力。

### 需求 2：外部数据必须先保留原始记录，再写成统一记忆

**用户故事：** 作为项目维护者，我希望不同平台来的数据最终都能进同一套家庭记忆，这样后面的分析和问答才能说同一种话。

#### 验收标准

1. WHEN 插件从外部系统拿到数据 THEN System SHALL 先保存原始记录，并带上来源插件、时间和同步任务信息。
2. WHEN 原始记录需要进入家庭记忆 THEN System SHALL 按统一结构写成标准记忆，而不是直接把原始 JSON 塞给 Agent。
3. WHEN 某条标准记忆需要追查来源 THEN System SHALL 能追到对应的原始记录和插件。

### 需求 3：Agent 必须通过统一入口使用插件能力

**用户故事：** 作为产品维护者，我希望 Agent 能调用插件能力，但不要被插件直接接管主流程。

#### 验收标准

1. WHEN Agent 需要领域能力或实时数据 THEN System SHALL 通过统一调用入口使用插件，而不是让插件直接改 Agent 内部状态。
2. WHEN 多个插件提供相近能力 THEN System SHALL 能明确知道当前调用了哪个插件。
3. WHEN 插件调用失败 THEN System SHALL 返回可判断的失败结果，不把整条主链路一起拖死。

### 需求 4：主动控制必须有清楚的权限边界

**用户故事：** 作为家庭管理员，我希望系统能做受控动作，但不希望插件一装上就能自动控制家里的东西。

#### 验收标准

1. WHEN 一个插件只申请读权限 THEN System SHALL 不允许它执行外部动作。
2. WHEN 一个插件申请执行能力 THEN System SHALL 先校验插件权限、家庭授权和动作风险等级。
3. WHEN 动作属于高风险范围 THEN System SHALL 默认要求人工确认，不能自动执行。

### 需求 5：第一版必须先用样板插件跑通整条链路

**用户故事：** 作为项目维护者，我希望第一版先拿少量真实场景跑通，不要一开始就铺太大。

#### 验收标准

1. WHEN 第一版落地 THEN System SHALL 至少支持一个智能家居读数据插件和一个健康数据插件。
2. WHEN 这两个样板插件运行成功 THEN System SHALL 能把设备数据和成员健康数据写入家庭记忆。
3. WHEN Agent 读取这些记忆 THEN System SHALL 能基于它们生成摘要、提醒或建议。

## 非功能需求

### 非功能需求 1：可维护性

1. WHEN 后续新增一个同类型插件 THEN System SHALL 不需要重写主链路，只需要补插件实现和配置。
2. WHEN 开发排查问题 THEN System SHALL 能看到插件加载、同步、转换、动作执行这几步的关键日志。

### 非功能需求 2：可靠性

1. WHEN 单个插件运行失败 THEN System SHALL 不影响其他插件和核心主流程继续运行。
2. WHEN 外部平台短时异常 THEN System SHALL 能返回明确失败状态，并支持后续重试。

### 非功能需求 3：安全性

1. WHEN 插件访问数据或执行动作 THEN System SHALL 记录审计信息，至少包含谁触发、用了哪个插件、做了什么。
2. WHEN 插件超出授权范围 THEN System SHALL 拒绝执行，并留下可追查记录。

## 成功定义

- 插件系统第一版能稳定识别和加载样板插件
- 样板插件数据能进入统一家庭记忆
- Agent 能基于这些记忆给出有用结果
- 高风险动作没有绕过权限直接自动执行
