# 20260312-家庭记忆标准记录示例

这份文档说明两件事：

1. 原始记录长什么样
2. 标准记忆长什么样

重点不是字段越多越好，而是让大家知道为什么要分两层。

## 一、为什么要分原始记录和标准记忆

原因很简单：

- 原始记录用来保留外部平台真实返回的数据，方便排错和重算
- 标准记忆用来给 Agent 和业务逻辑统一使用，避免每个来源一个格式

如果不分层，后面插件一多，系统会变成一堆互不兼容的 JSON 垃圾堆。

## 二、原始记录示例

假设健康插件从外部平台拿到了这样一条睡眠数据：

```json
{
  "source": "health-basic-reader",
  "external_member_id": "mom-001",
  "record_type": "sleep",
  "date": "2026-03-12",
  "sleep_hours": 6.2,
  "raw_payload": {
    "provider": "demo-health",
    "sleep_score": 78,
    "deep_sleep_minutes": 95
  }
}
```

这类数据的特点是：

- 它保留了外部来源的原样信息
- 有些字段只对原插件有意义
- 不适合直接给 Agent 当统一上下文

## 三、转成标准记忆后的示例

上面这条原始记录，进入系统后，可以先转成一条 `Observation`：

```json
{
  "type": "Observation",
  "subject_type": "Person",
  "subject_id": "mom",
  "category": "sleep_duration",
  "value": 6.2,
  "unit": "hour",
  "observed_at": "2026-03-12",
  "source_plugin_id": "health-basic-reader",
  "source_record_ref": "raw-record-001"
}
```

这条记录表达得很直接：

- 谁：`mom`
- 观察到了什么：`sleep_duration`
- 值是多少：`6.2`
- 单位是什么：`hour`
- 来源是谁：`health-basic-reader`

这样 Agent 不需要知道原平台叫不叫 `sleep_score`，它只要看统一结构就行。

## 四、再看一个智能家居例子

原始记录：

```json
{
  "source": "smart-home-reader",
  "external_device_id": "sensor-living-room-01",
  "record_type": "temperature",
  "captured_at": "2026-03-12T08:30:00Z",
  "value": 23.5,
  "unit": "celsius"
}
```

标准记忆：

```json
{
  "type": "Observation",
  "subject_type": "Device",
  "subject_id": "living-room-sensor",
  "category": "room_temperature",
  "value": 23.5,
  "unit": "celsius",
  "observed_at": "2026-03-12T08:30:00Z",
  "source_plugin_id": "smart-home-reader",
  "source_record_ref": "raw-record-002"
}
```

你会看到，健康和家居虽然是两条业务线，但都能先落成 `Observation`。这就是统一模型的价值。

## 五、第一版建议统一保留的关键信息

无论最后落到哪种标准记忆，第一版都建议至少带上这些信息：

- 记录类型
- 关联对象是谁
- 记录值是什么
- 记录时间是什么时候
- 来源插件是谁
- 对应原始记录是哪一条

这样后面做问答、摘要、提醒、审计时，最基本的链路就不会断。

## 六、第一版先不要做复杂推断入库

比如：

- “睡眠不足，所以一定需要医疗干预”
- “学习时长下降，所以孩子状态异常”

这种结论第一版先不要直接写成事实记忆。

更稳的做法是：

- 先把事实记录写进去
- 推断结果单独作为提醒或建议处理

这样不容易把系统记忆写脏。
