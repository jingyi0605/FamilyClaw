# 20260312-实现模块与开发顺序

这份文档不是详细设计书，只解决一个很实际的问题：

- 第一版到底先做哪些模块
- 每个模块负责什么
- 开发顺序怎么排才不会把项目做乱

一句话先说结论：

先把插件注册、记忆写入、样板插件打通，再让 Agent 来用，最后才碰动作执行。

## 一、第一版模块清单

第一版只建议拆下面 7 个模块。再多，容易虚胖。

### 1. 插件注册模块

这个模块负责回答两个问题：

- 这个插件能不能装
- 装进来以后系统怎么认出它

至少要做的事：

- 读取插件 manifest
- 校验必要字段
- 保存插件基本信息
- 支持启用和禁用

做完后能看到什么：

- 系统能列出当前有哪些插件
- 系统知道每个插件属于哪一类能力

### 2. 插件运行模块

这个模块负责真正去调用插件。

至少要做的事：

- 按统一入口调用插件方法
- 处理超时和失败
- 记录每次运行结果

做完后能看到什么：

- 插件不是“看起来注册了”，而是真的能跑一次

### 3. 原始记录保存模块

这个模块负责把外部拿回来的原始数据先存下来。

至少要做的事：

- 保存插件返回的原始数据
- 记录来源插件、时间、触发方式
- 关联到一次具体运行记录

做完后能看到什么：

- 外部数据没有直接丢进业务逻辑里
- 出问题时能回头查原始内容

### 4. 标准记忆写入模块

这个模块负责把原始记录转成系统统一能看懂的数据。

至少要做的事：

- 调用 `memory-ingestor`
- 生成标准记忆
- 建立标准记忆和原始记录的关联

做完后能看到什么：

- 健康数据和家居数据都能进入统一家庭记忆

### 5. Agent 桥接模块

这个模块负责让 Agent 使用插件能力，但不让插件接管 Agent。

至少要做的事：

- 给 Agent 暴露统一工具入口
- 标明当前调用了哪个插件
- 返回统一成功或失败结果

做完后能看到什么：

- Agent 能读插件能力
- 但插件不能反过来直接改 Agent 主流程

### 6. 权限和策略模块

这个模块负责卡住风险。

至少要做的事：

- 区分读权限和执行权限
- 判断动作是否高风险
- 判断是否需要人工确认

做完后能看到什么：

- 新插件默认只能读数据
- 动作能力不会一接进来就自动开跑

### 7. 审计和运行记录模块

这个模块负责让后续排错有抓手。

至少要做的事：

- 记录插件何时运行
- 记录成功还是失败
- 记录写了什么记忆
- 记录有没有触发动作

做完后能看到什么：

- 出问题时不是只能靠猜

## 二、第一版开发顺序

下面这个顺序不要乱。

### 第 1 步：先做插件注册模块

原因：

- 不先定义插件怎么进系统，后面所有实现都没有统一入口

这一步完成标志：

- 至少能识别一个 demo 插件
- 缺字段的插件会被拒绝

### 第 2 步：再做插件运行模块

原因：

- 只会注册不会运行，没有任何价值

这一步完成标志：

- 能手动跑一次 demo 插件
- 能看到成功和失败结果

### 第 3 步：接上原始记录保存模块

原因：

- 外部数据一旦直接跳过原始层，后面排错会很痛苦

这一步完成标志：

- 插件跑完后，能看到原始记录落下来
- 原始记录能关联到插件和运行批次

### 第 4 步：接上标准记忆写入模块

原因：

- 到这一步，系统才真正从“接数据”变成“能沉淀家庭记忆”

这一步完成标志：

- 原始记录能转成标准记忆
- 至少能生成 `Observation`

### 第 5 步：做两个样板插件

建议固定为：

- 智能家居读数据插件
- 健康基础数据插件

原因：

- 一个代表设备类数据
- 一个代表成员类数据
- 这两个跑通后，很多后续能力都能顺着接

这一步完成标志：

- 两类数据都能进家庭记忆
- 不需要为每种数据单独造主链路

### 第 6 步：接上 Agent 桥接模块

原因：

- 这时记忆里已经有真实数据了，Agent 才有东西可用

这一步完成标志：

- Agent 能读到插件写入的记忆
- Agent 能按统一入口调用插件工具

### 第 7 步：最后再做权限和动作执行

原因：

- 动作执行是最容易出事故的部分，必须放到后面

这一步完成标志：

- 只读插件不能执行动作
- 执行动作前会做权限和风险判断

## 三、每一步主要改什么

这里先不写死具体代码文件，只写到模块层，方便后面按真实代码结构落。

### 阶段 1：基础框架

- 插件 manifest 读取和校验
- 插件注册表
- 插件运行入口
- 运行结果模型

### 阶段 2：记忆主链路

- 原始记录保存
- 标准记忆转换
- `Observation` 写入
- 原始记录和标准记忆关联

### 阶段 3：样板插件

- 智能家居读数据插件
- 健康基础数据插件
- 两类插件的转换逻辑

### 阶段 4：Agent 接入

- Agent 统一工具入口
- 调用结果包装
- 失败降级处理

### 阶段 5：受控动作

- 动作权限校验
- 风险等级判断
- 人工确认入口
- 审计记录补齐

## 四、第一版先不碰的东西

这些东西不是永远不做，而是第一版碰了只会把项目拖慢：

- 插件市场
- 复杂工作流编排
- 任意脚本沙箱
- 全量教育平台接入
- 自动医疗结论
- 高风险设备自动控制

先把主链路做通，比写一个看起来很大的平台名字重要得多。

## 五、给 AI IDE 的直接执行提示

如果后面让 AI IDE 继续往下拆任务，可以直接按下面顺序理解：

1. 先实现插件注册和 manifest 校验
2. 再实现插件运行和运行记录
3. 再实现原始记录落库
4. 再实现标准记忆转换和写入
5. 再补两个样板插件
6. 再接 Agent 工具桥接
7. 最后补权限、审计和动作执行

这个顺序的核心目的只有一个：

不要在还没把数据链路站稳的时候，先去碰高风险控制。
