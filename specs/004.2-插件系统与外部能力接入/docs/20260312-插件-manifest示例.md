# 20260312-插件 manifest 示例

这份文档只做一件事：给出一个第一版可理解的插件声明例子。

重点不是字段多漂亮，而是让人一眼看懂：

- 这个插件是谁
- 它能做什么
- 它要什么权限
- 它从哪里进来

## 一、最小示例

```json
{
  "id": "health-basic-reader",
  "name": "健康基础数据接入",
  "version": "0.1.0",
  "types": ["connector", "memory-ingestor"],
  "permissions": [
    "health.read",
    "memory.write.observation"
  ],
  "risk_level": "low",
  "triggers": ["manual", "schedule"],
  "entrypoints": {
    "connector": "plugins.health_basic.connector.sync",
    "memory_ingestor": "plugins.health_basic.ingestor.transform"
  }
}
```

## 二、每个字段是什么意思

- `id`：插件唯一标识。系统靠它区分插件，不能重复。
- `name`：给人看的名字。
- `version`：插件版本。后面升级和排错会用到。
- `types`：这个插件提供哪几类能力。第一版只允许四类：`connector`、`memory-ingestor`、`action`、`agent-skill`。
- `permissions`：这个插件想要哪些权限。没有声明，就默认没有。
- `risk_level`：风险等级。第一版只需要先区分 `low`、`medium`、`high`。
- `triggers`：这个插件允许怎么触发。比如手动触发，或者定时触发。
- `entrypoints`：系统真正调用插件代码时要找的入口。

## 三、再看一个智能家居示例

```json
{
  "id": "smart-home-reader",
  "name": "智能家居读数据插件",
  "version": "0.1.0",
  "types": ["connector", "memory-ingestor"],
  "permissions": [
    "device.read",
    "memory.write.device",
    "memory.write.observation"
  ],
  "risk_level": "low",
  "triggers": ["manual", "schedule"],
  "entrypoints": {
    "connector": "plugins.smart_home.connector.sync",
    "memory_ingestor": "plugins.smart_home.ingestor.transform"
  }
}
```

这个例子表达的是：

- 它只能读设备数据
- 它可以把设备和环境信息写进家庭记忆
- 它没有申请 `action`，所以不能控制设备

## 四、动作型插件再看一个例子

```json
{
  "id": "smart-home-action",
  "name": "智能家居动作插件",
  "version": "0.1.0",
  "types": ["action"],
  "permissions": [
    "device.control"
  ],
  "risk_level": "medium",
  "triggers": ["manual"],
  "entrypoints": {
    "action": "plugins.smart_home_action.executor.run"
  }
}
```

这个例子表达的是：

- 它有执行能力
- 但它只是“有资格申请执行”
- 真正执行前，系统还要继续看家庭授权和动作风险

## 五、第一版先别加的东西

下面这些字段第一版可以先不急着放进去，避免文档看起来很全，实际没人能落：

- 插件商店地址
- 远程签名验证细节
- 复杂依赖图
- 多环境动态配置模板
- 可视化表单元数据

先把最小声明做好，比堆一堆花活有用得多。
