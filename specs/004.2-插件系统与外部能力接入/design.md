# 设计文档 - 插件系统与外部能力接入

状态：Draft

## 设计目标

第一版设计只做一件事：把“外部能力怎么接进来”这件事收口。

不要把它做成万能平台。先把最小主链路做对：

1. 插件声明能力
2. 系统校验权限
3. 插件读取外部数据或执行动作
4. 数据写入家庭记忆
5. Agent 通过统一入口使用这些能力

## 一、插件类型

第一版只保留四类插件能力：

- `connector`：从外部系统读数据
- `memory-ingestor`：把原始数据转成标准记忆
- `action`：对外执行动作
- `agent-skill`：给 Agent 提供额外工具或规则

这样分已经够用，不要继续拆细。

## 二、核心模块

### 1. 插件注册中心

负责：

- 保存插件基本信息
- 校验插件声明是否完整
- 管理启用、禁用、版本状态
- 对外提供“当前有哪些能力可用”

### 2. 插件运行层

负责：

- 按统一入口调用插件
- 处理超时、失败、重试
- 隔离单个插件故障

### 3. 家庭记忆写入层

负责：

- 先保存原始记录
- 再转成标准记忆
- 建立原始记录和标准记忆的关联

### 4. 权限与策略层

负责：

- 判断插件有没有读、写、执行权限
- 判断动作是不是高风险
- 判断是否需要人工确认

### 5. Agent 工具桥接层

负责：

- 把插件能力暴露给 Agent
- 统一 Agent 调用方式
- 保证插件只能提供能力，不能直接改 Agent 主流程

## 三、家庭记忆最小数据约定

第一版不要追求全，先统一几类最常用对象：

- `Person`：家庭成员
- `Device`：设备
- `Event`：事件
- `Observation`：观察记录，比如睡眠、心率、步数、温度、学习时长
- `Task`：提醒、待办、建议行动
- `Alert`：需要关注的异常或风险

这里最关键的是 `Observation`。很多不同来源的数据，最后都可以先落到这一类里。

例如：

- 睡眠时长
- 心率
- 房间温度
- 步数
- 学习时长

这些看起来是不同业务，实际上都可以先作为“某个对象在某个时间的观察记录”。

## 四、数据进入系统的主链路

统一走下面这条路：

1. `connector` 从外部拿到原始数据
2. 系统把原始数据保存成原始记录
3. `memory-ingestor` 把原始记录转成标准记忆
4. 系统把标准记忆写入家庭记忆
5. Agent 或业务模块读取标准记忆

这样做的目的很简单：

- 原始数据不丢，方便排错
- 标准数据统一，方便分析和问答

## 五、主动控制边界

第一版把主动能力分成三层：

- 只读：只能拿数据
- 建议：可以生成提醒、建议、待确认动作
- 执行：可以真正调用外部动作

默认新插件只给“只读”。

下面这些动作第一版默认按高风险处理：

- 门锁
- 摄像头
- 支付或下单
- 医疗类自动结论
- 儿童敏感信息外发

这些动作就算插件支持，也不能默认自动执行。

## 六、第一版样板插件

只做两个，够了：

### 1. 智能家居读数据插件

目标：验证设备状态和环境数据可以进家庭记忆。

先做：

- 设备状态读取
- 房间温湿度读取

先不做：

- 设备自动控制

### 2. 健康数据插件

目标：验证成员相关数据可以进家庭记忆。

先做：

- 步数
- 睡眠
- 心率

先不做：

- 医疗判断
- 自动干预

## 七、落地顺序

按这个顺序做最稳：

1. 先做插件声明、注册、加载
2. 再做原始记录和标准记忆写入
3. 再做样板插件
4. 再让 Agent 调用这些能力
5. 最后才碰受控动作

顺序不能反。先做自动控制，只会把风险提前放大。
