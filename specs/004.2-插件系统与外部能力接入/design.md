# 设计文档 - 插件系统与外部能力接入

状态：Draft

## 1. 设计目标

这份设计文档只收一件事：

- 把“外部能力怎么进入系统”讲成一条清楚、可维护、不靠特例的主链路

不要把它做成万能开放平台。第一版先把主链路做对。

## 2. 正式插件家族

`004.2` 不再把插件理解成只有最早的 4 类。

当前正式插件家族分两层看：

### 2.1 核心执行类型

- `connector`
- `memory-ingestor`
- `action`
- `agent-skill`

这四类决定主执行链路。

### 2.2 正式扩展类型

- `locale-pack`
- `channel`
- `region-provider`

这些不是页面特例，也不是临时挂靠字段，而是正式类型：

- `locale-pack` 负责语言资源
- `channel` 负责通讯平台接入
- `region-provider` 负责地区目录能力

## 3. 核心模块

### 3.1 插件注册中心

负责：

- 保存插件基本信息
- 校验 `manifest`
- 对外提供插件能力元数据

### 3.2 插件运行层

负责：

- 把外部调用统一收口成 `plugin_job`
- 由 worker 执行插件
- 处理超时、失败、重试和人工确认

### 3.3 原始记录与家庭记忆写入层

负责：

- 先保存原始记录
- 再转成统一记忆
- 建立原始记录和记忆之间的追踪关系

### 3.4 权限与策略层

负责：

- 校验权限
- 校验风险等级
- 决定是否进入人工确认

### 3.5 Agent 桥接层

负责：

- 把受控插件能力暴露给 Agent
- 保持 Agent 只能调正式开放的插件能力

## 4. 数据主链路

统一走这条路：

1. 调用方提交插件请求
2. 系统创建 `plugin_job`
3. worker 按插件类型执行
4. 如果有原始数据，先写原始记录
5. 如果有记忆转换，再写标准记忆
6. 系统更新任务状态和通知
7. 调用方或 Agent 再继续读取结果

这样做的目的很朴素：

- 原始数据不丢
- 标准数据统一
- 长任务不拖垮接口
- 失败和人工确认都有正式落点

## 5. 扩展类型怎么挂进来

### 5.1 `locale-pack`

- 不走执行链路
- 负责资源注册

### 5.2 `channel`

- 走正式通道接入链路
- 配置协议和动态表单统一看 `004.2.3`

### 5.3 `region-provider`

- 走正式地区 provider 协议
- 不再靠临时 capability 占坑说明

## 6. 配置协议边界

这部分必须明确切出去。

`004.2` 不再维护这些内容：

- `config_specs` 结构
- `config_schema`
- `ui_schema`
- 配置作用域
- 配置实例持久化
- secret 字段语义

这些统一由：

- `../004.2.3-插件配置协议与动态表单/`

承接。

原因很简单：

- 插件系统主链路是一层问题
- 配置协议和动态表单是另一层问题

混在一起写，只会让同一件事在两份 Spec 里漂移。

## 7. 主动控制边界

第一版主动能力继续分三层：

- 只读
- 建议
- 执行

默认新插件先站在保守一边，不要先放大自动控制范围。

高风险动作继续要求：

- 权限校验
- 风险判定
- 必要时人工确认

## 8. 落地顺序

按这个顺序最稳：

1. 插件声明、注册、加载
2. 统一任务链路
3. 原始记录和记忆写入
4. 扩展类型接入
5. Agent 桥接
6. 受控动作完善

配置协议这一条已经拆到 `004.2.3`，不要在这里再插回主链里重新讲。
