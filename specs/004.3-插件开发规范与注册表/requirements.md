# 需求文档 - 插件开发规范与注册表

状态：Draft

## 简介

插件系统后端基础能力已经有了，但现在还不适合对外开放开发，因为规则还不够清楚。

如果没有统一规范，第三方开发者会马上遇到这些问题：

- 不知道插件目录该怎么放
- 不知道 `manifest.json` 哪些字段必填，哪些能选填
- 不知道什么能力可以做，什么能力第一版不要做
- 不知道测试做到什么程度才算能提 PR
- 不知道插件市场到底识别什么字段

这份 Spec 面向两个角色：

- **第三方开发者**：按规范开发和提交插件
- **官方维护者**：按统一格式审核插件、维护注册表、给前端市场提供稳定数据源

成功后的核心收益不是“插件数量变多”，而是“插件接入方式开始稳定下来”。

## 术语表

- **System**：FamilyClaw 插件系统与插件市场相关能力
- **插件开发规范**：指导第三方开发者开发插件的一套文档和示例
- **插件注册表**：按统一 schema 保存插件元数据的索引源，不直接保存插件运行代码
- **官方注册表**：由项目官方维护的注册表源
- **第三方注册表**：由社区或其他组织维护，但仍遵守同一 schema 的注册表源

## 范围说明

### In Scope

- 插件开发规范文档结构
- `manifest` 字段规范
- 插件目录结构规范
- 测试与验收规范
- GitHub 注册表格式
- 插件注册 PR 提交流程
- 官方注册表 / 第三方注册表机制

### Out of Scope

- 自动下载安装第三方代码
- 远程执行外部仓库插件
- 完整插件签名体系
- 运行时代码沙箱

## 需求

### 需求 1：第三方开发者可以按统一规范开发插件

**用户故事：** 作为第三方开发者，我希望有一套清楚的插件开发规范，以便我能按项目规则快速做出可提交的插件。

#### 验收标准

1. WHEN 开发者阅读插件开发规范 THEN System SHALL 说明插件支持的类型、入口约定、目录结构和最小实现要求。
2. WHEN 开发者创建新插件 THEN System SHALL 提供统一的 `manifest.json` 字段说明和示例。
3. WHEN 开发者尝试实现超出第一版范围的能力 THEN System SHALL 明确说明哪些能力当前先不支持。

### 需求 2：官方和市场可以按统一 schema 识别插件

**用户故事：** 作为官方维护者或前端市场开发者，我希望插件元数据有统一格式，以便市场和管理界面能稳定识别插件。

#### 验收标准

1. WHEN 维护者查看注册表规范 THEN System SHALL 定义插件注册项的字段、必填项、校验规则和来源说明。
2. WHEN 前端市场读取注册表 THEN System SHALL 能区分内置插件、官方插件和第三方插件。
3. WHEN 注册表中缺少关键字段或字段格式错误 THEN System SHALL 明确要求拒绝该注册项。

### 需求 3：第三方可以按明确流程提交插件注册

**用户故事：** 作为第三方开发者，我希望知道怎么通过 GitHub 提交插件注册，以便我的插件能进入市场可见范围。

#### 验收标准

1. WHEN 第三方要提交插件注册 THEN System SHALL 说明 PR 需要提交哪些元数据和链接。
2. WHEN 官方审核插件注册 THEN System SHALL 提供最小审核清单，包括字段完整性、仓库可访问性、文档可读性和风险声明。
3. WHEN 插件未满足注册要求 THEN System SHALL 说明可以拒绝或要求补充，而不是模糊通过。

### 需求 4：系统支持官方注册表和第三方注册表并存

**用户故事：** 作为系统维护者，我希望插件市场既能使用官方注册表，也能接入第三方注册表，以便后续扩展来源而不用重写底层。

#### 验收标准

1. WHEN 系统接入注册表源 THEN System SHALL 使用统一 schema 处理官方和第三方注册表。
2. WHEN 同一个插件来自不同注册表 THEN System SHALL 要求用稳定的 `plugin_id` 和来源字段区分。
3. WHEN 第三方注册表不可信或字段不完整 THEN System SHALL 允许前端标记来源风险或直接忽略该源。

## 非功能需求

### 非功能需求 1：可读性

1. WHEN 新开发者第一次阅读规范 THEN System SHALL 用人话说明规则，而不是只堆术语。
2. WHEN 维护者定位某个规则 THEN System SHALL 让规则、示例和验收项能快速互相跳转。

### 非功能需求 2：可扩展性

1. WHEN 后续新增插件类型或注册字段 THEN System SHALL 能在不推翻现有 schema 的前提下扩展。
2. WHEN 前端市场接入多个注册表 THEN System SHALL 继续使用统一的插件注册项结构。

### 非功能需求 3：可审计性

1. WHEN 官方审核第三方插件注册 THEN System SHALL 能明确看到插件来源、仓库地址、维护者信息和风险等级。
2. WHEN 后续排查某个市场插件来源 THEN System SHALL 能追踪它来自哪个注册表、哪个仓库。

## 成功定义

- 第三方开发者只靠文档和样板，就能做出一个符合规则的插件
- 前端和后端可以基于统一注册表 schema 识别插件来源和元数据
- 官方可以按统一 PR 流程审核插件注册，而不是靠口头约定
