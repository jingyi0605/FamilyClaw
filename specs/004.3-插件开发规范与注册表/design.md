# 设计文档 - 插件开发规范与注册表

状态：Draft

## 1. 设计目标

这份设计文档只做三件事：

1. 定开发者写插件时该遵守的稳定规则
2. 定注册表条目该长什么样
3. 定文档以后怎么维护，避免每次系统迭代都全盘返工

## 2. 文档分层策略

这是这轮最重要的新设计之一。

插件开发文档以后分两层：

### 2.1 稳定规范层

放这些内容：

- 边界
- 硬规则
- 不要做什么
- 接手时该先看什么

这层尽量少改。

### 2.2 高频变化引用层

放这些内容：

- 当前接口样例
- 当前字段矩阵
- 当前支持的 widget / 条件 / schema 细节
- 当前代码里的真实结构

这层不在多份文档里复制，统一保留少量事实来源。

当前插件配置这条线的事实来源包括：

## 3. 开发规范层设计

### 3.1 manifest 规则

manifest 规则统一看开发者手册，不在这里复制一整份字段大表。

这份 Spec 只保留稳定要求：

- 插件必须有稳定 `id`
- 插件类型必须使用系统正式支持的类型
- 声明了可执行类型，就必须有对应入口
- 需要正式配置的插件，必须走 `manifest.config_specs`

配置协议细节统一引用：

- `../004.2.3-插件配置协议与动态表单/`

### 3.2 目录结构规则

开发者必须能一眼看出：

- manifest 在哪
- 入口代码在哪
- 测试在哪

第三方插件最小目录继续建议：

```text
<plugin_repo>/
  manifest.json
  requirements.txt
  README.md
  plugin/
  tests/
```

### 3.3 运行时方向

第三方插件后续优先走：

- 主服务 + 同容器子进程 runner

这里先定边界，不在这份 Spec 里落具体实现。

### 3.4 地区坐标上下文边界

这一节只定稳定规则，不重复复制接口细节。

#### `region-provider` 节点坐标契约

- 标准地区节点现在允许可选返回：
  - `latitude`
  - `longitude`
  - `coordinate_precision`
  - `coordinate_source`
  - `coordinate_updated_at`
- 这些字段没提供时，旧 provider 继续兼容，只是上层会拿到 `unavailable`。
- 这些字段一旦提供，就必须和经纬度一起自洽，不能出现“有纬度没经度”或者“有经纬度没精度/来源”这种垃圾状态。
- provider 构建地区绑定快照时，应该把可用代表坐标写进 `representative_coordinate`，这样 provider 临时不可用时，旧绑定也还能复用快照代表点。

#### 家庭地区上下文统一结果

- 读取 `household_region_context` 的插件，不应该自己再拼坐标优先级。
- 运行时注入的 `_system_context.region.household_context` 里，正式坐标入口是 `coordinate`。
- `coordinate.source_type` 只允许：
  - `household_exact`
  - `region_representative`
  - `unavailable`
- 这条规则的目的很简单：
  - 上层插件直接消费统一结果
  - 不再从 `city` 文本反推天气查询点
  - provider 缺坐标时显式报 `unavailable`，而不是偷偷猜

#### 家庭精确坐标写入边界

- 浏览器或 App 定位只能提供候选值。
- 只有用户确认后，系统才允许通过 `PATCH /households/{id}/coordinate` 保存家庭精确坐标。
- 这个写入只覆盖坐标解析结果，不改家庭现有地区绑定。

## 4. 注册表层设计

### 4.1 条目结构

注册表条目只保留市场识别和审核需要的元数据：

- `plugin_id`
- 展示信息
- 来源信息
- 维护者信息
- 运行时摘要

这里故意不复制整份 manifest，也不把 `entrypoints` 抄进来。

原因很简单：

- 运行时真实字段应该回到插件仓库里的 manifest
- 注册表只负责识别，不负责替代源码事实来源

### 4.2 来源并存

官方注册表和第三方注册表可以并存，但必须守两条：

1. 条目结构统一
2. 来源标识必须保留

不能为了“看起来方便”把不同来源硬合并成一条匿名记录。

## 5. 与 004.2.3 的关系

这份 Spec 明确不再承接插件配置协议正文。

如果文档内容涉及下面这些词，就统一跳到 `004.2.3`：

- `config_specs`
- `config_schema`
- `ui_schema`
- `plugin` / `channel_account`
- secret 语义
- 动态表单渲染

这里最多只保留一句稳定规则：

- 插件需要正式配置时，必须接入正式协议，而不是继续让宿主页硬编码字段

## 6. 验收原则

如果后续有人改插件系统，文档更新顺序应该是：

1. 先确认稳定规则有没有变
2. 如果稳定规则没变，只更新事实来源
3. 如果稳定规则真的变了，再改开发者手册
4. 不允许把相同接口样例复制到多份手册里一起维护

这条规则看起来无聊，但能直接减少一大半文档漂移。
