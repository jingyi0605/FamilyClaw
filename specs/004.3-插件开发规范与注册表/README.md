# Spec 004.3 - 插件开发规范与注册表

状态：Draft

## 2026-03-17 最新边界说明

这份 Spec 现在继续负责“插件开发规范”和“注册表条目长什么样”，但插件体系的最新边界已经往前走了一步。

从 `004.5` 开始，下面这些事实要以新规则为准：

- 正式插件类型已经扩展为 9 类：
  - `connector`
  - `memory-ingestor`
  - `action`
  - `agent-skill`
  - `channel`
  - `locale-pack`
  - `region-provider`
  - `theme-pack`
  - `ai-provider`
- 主题包和 AI 供应商不再算“像插件的特例”，而是通用插件系统里的正式类型。
- 插件启用/禁用规则统一看：
  - `docs/开发设计规范/20260317-插件启用禁用统一规则.md`
- 最小版本治理边界统一看：
  - `specs/004.5-插件能力统一接入与版本治理/docs/20260317-插件版本治理现状与最小能力说明.md`

如果你现在是在补插件开发规范，不要再单独发明“主题规范”或“AI 供应商规范”的平行版本，直接按通用插件规范补。

## 这份 Spec 解决什么问题

`004.2` 解决了插件系统怎么跑，但还没解决另外两个现实问题：

- 开发者到底该按什么规则写插件
- 插件市场和注册表到底该按什么格式识别插件

这份 Spec 负责把这两件事讲清楚。

## 这次要做到什么

- 给开发者一套能照着做的插件开发规则
- 给注册表一套稳定、可审核的条目结构
- 给后续插件市场一套可复用的元数据来源
- 把“开发文档怎么维护”也收口，不再每次插件系统一更新就全盘改文档

## 开发者文档的维护策略

这轮必须明确一条新规则：

- 稳定规范尽量少改
- 高频变化内容集中引用

也就是：

- 开发者手册负责写边界、规则、不要做什么
- 当前支持的字段细节、API 样例、运行时结构，统一去看事实来源

当前插件开发文档以这里为准：

- `docs/开发者文档/插件开发/`

不是再回 `specs/004.3/docs/` 里堆第二套正文。

## 这次明确不做什么

- 不做自动下载安装平台
- 不做远程执行平台
- 不做签名体系全量落地
- 不在这份 Spec 里复制一整套插件配置协议

插件配置协议统一看：

- `../004.2.3-插件配置协议与动态表单/`

## 这份 Spec 只负责哪两层

### 1. 开发规范层

讲清楚：

- 插件目录怎么放
- manifest 怎么写
- 入口函数怎么约定
- 测试最少做到什么程度

### 2. 注册表层

讲清楚：

- 注册表条目长什么样
- PR 怎么提
- 官方和第三方注册表怎么并存

换句话说：

- 怎么写插件，这里管
- 怎么收插件，这里管
- 配置协议长什么样，不在这里重新发明
