# Spec 004.3 - 插件开发规范与注册表

状态：Draft

## 这份 Spec 解决什么问题

`004.2` 已经把插件系统的后端底座做出来了，但现在还缺一件很现实的东西：

- 第三方开发者不知道该怎么按项目规则开发插件
- 官方也没有一套稳定的注册表格式，无法让插件市场识别插件
- 后面就算做了插件市场，没有统一规范，也只会变成一堆互不兼容的插件介绍页

这份 Spec 就是把这件事收好：先把“怎么开发插件、怎么提交插件、市场怎么识别插件”讲清楚。

## 这次要做到什么

- 给第三方开发者一套能照着做的插件开发规范
- 把 `manifest` 字段、插件目录结构、测试要求写成统一规则
- 定义 GitHub 注册表格式和 PR 提交流程
- 支持官方注册表和第三方注册表并存

## 这次明确不做什么

- 不做自动下载安装代码
- 不做远程执行插件代码
- 不做签名体系全量落地
- 不做脚本沙箱

## 为什么现在做

因为后面不管是官方继续扩插件，还是第三方开发者来接能力，都需要先解决三个基本问题：

- 插件到底按什么结构开发
- 市场到底按什么格式识别插件
- 官方到底按什么规则审核和接收插件注册

这些事情不先定下来，后面的插件市场和前端管理页都会反复返工。

## 主要文档

- `requirements.md`：说清楚开发规范和注册表到底要覆盖什么
- `design.md`：说清楚规范结构、注册表格式和提交流程怎么落
- `tasks.md`：说清楚先写什么文档、先定什么格式、先补什么样板
- `docs/README.md`：说明后续补充示例、模板、审核说明该放哪里

## 开发者先看什么

如果你是第一次给项目写插件，先按这个顺序看：

1. `docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`：先看支持什么类型、第一版边界和最小开发流程。
2. `docs/开发者文档/插件开发/zh-CN/02-插件对接方式说明.md`：再看插件到底怎么被系统调用、返回什么、怎么同步数据。
3. `design.md`：再看 `manifest`、注册表和提交流程的正式规则。
4. `tasks.md`：如果你在继续完善这份 Spec，按这里的顺序推进，不要自己跳步骤。

插件开发相关的开发者文档，后续统一放在 `docs/开发者文档/插件开发/`，并同时维护 `zh-CN` 和 `en` 两套版本。

这份 Spec 当前的重点还是两件事：

- 让第三方开发者知道插件该怎么做
- 让后续插件市场知道插件元数据该怎么认

跟这两件事无关的内容，这一版先别塞进来。
