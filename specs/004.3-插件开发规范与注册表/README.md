# Spec 004.3 - 插件开发规范与注册表

状态：Draft

## 这份 Spec 解决什么问题

`004.2` 解决了插件系统怎么跑，但还没解决另外两个现实问题：

- 开发者到底该按什么规则写插件
- 插件市场和注册表到底该按什么格式识别插件

这份 Spec 负责把这两件事讲清楚。

## 这次要做到什么

- 给开发者一套能照着做的插件开发规则
- 给注册表一套稳定、可审核的条目结构
- 给后续插件市场一套可复用的元数据来源
- 把“开发文档怎么维护”也收口，不再每次插件系统一更新就全盘改文档

## 开发者文档的维护策略

这轮必须明确一条新规则：

- 稳定规范尽量少改
- 高频变化内容集中引用

也就是：

- 开发者手册负责写边界、规则、不要做什么
- 当前支持的字段细节、API 样例、运行时结构，统一去看事实来源

当前插件开发文档以这里为准：

- `docs/开发者文档/插件开发/`

不是再回 `specs/004.3/docs/` 里堆第二套正文。

## 先看什么

如果你第一次接手插件开发，建议按这个顺序看：

1. `docs/开发者文档/插件开发/zh-CN/00-文档使用与维护原则.md`
2. `docs/开发者文档/插件开发/zh-CN/01-插件开发总览.md`
3. `docs/开发者文档/插件开发/zh-CN/03-manifest字段规范.md`
4. `docs/开发者文档/插件开发/zh-CN/05-插件对接方式说明.md`
5. `docs/开发者文档/插件开发/zh-CN/11-插件配置接入说明.md`

7. 再回来看这份 Spec 的 `design.md`

## 这次明确不做什么

- 不做自动下载安装平台
- 不做远程执行平台
- 不做签名体系全量落地
- 不在这份 Spec 里复制一整套插件配置协议

插件配置协议统一看：

- `../004.2.3-插件配置协议与动态表单/`

## 这份 Spec 只负责哪两层

### 1. 开发规范层

讲清楚：

- 插件目录怎么放
- manifest 怎么写
- 入口函数怎么约定
- 测试最少做到什么程度

### 2. 注册表层

讲清楚：

- 注册表条目长什么样
- PR 怎么提
- 官方和第三方注册表怎么并存

换句话说：

- 怎么写插件，这里管
- 怎么收插件，这里管
- 配置协议长什么样，不在这里重新发明
