# 需求文档 - 设备页面实体操控增强
状态：Draft

## 这次要解决什么问题

`005.4` 已经把“设备与集成”主链路往实例驱动方向推进了一大步，设备同步和基础控制链路也已经打通，至少可以真实控制已同步的设备。

问题是，用户真正打开设备页面时，还是不够好用：

- 一个设备下面实体很多时，很难快速找到自己常用的实体
- 设备虽然已经能被控制，但页面上还没有稳定、统一的操作入口
- 设备缺少明确的删除和停用入口，管理动作还不完整
- 用户看不到设备操作记录，出了问题也很难追查

这次不是再做一层测试工具，而是把这些能力真正集成进设备页面，让用户能在页面里完成常用管理和控制。

## 术语说明

- **设备**：用户在页面里看到的可管理对象，例如“书房灯”
- **实体**：设备下可读取或可控制的能力点，例如开关、亮度、色温、电量
- **收藏实体**：用户手动标记为常用的实体，用来减少实体过多时的查找成本
- **操作记录**：设备被执行控制动作后的记录，包括谁在什么时候执行了什么动作，以及结果如何

## 范围说明

### In Scope

- 在设备页面增加两个标签页：
  - `收藏的实体`
  - `全部实体`
- 默认进入 `收藏的实体` 标签页
- 支持把实体加入收藏或取消收藏
- 设备页面集成支持类型设备的实体控制入口
- 支持停用设备
- 支持删除设备
- 支持查看设备操作记录
- 明确删除、停用、收藏、查看记录的页面反馈和错误提示

### Out of Scope

- 这次不重做整个设备信息架构，仍然基于现有设备页继续增强
- 这次不做复杂的多维筛选系统，例如按房间、品牌、能力类型做高级筛选
- 这次不做批量删除、批量停用
- 这次不做“实体级删除”，删除动作只针对设备
- 这次不做完整审计平台，只做设备页面可查看的操作记录
- 这次不要求所有插件在第一天就支持完整控制面板，但支持类型必须能接入统一入口

## 需求

### 需求 1：设备页默认显示收藏实体

**用户故事：**
作为设备较多、实体也很多的用户，我希望设备页默认先显示我收藏的实体，这样我打开页面后先看到最常用的控制项，而不是被全量实体淹没。

#### 验收标准

1. WHEN 用户打开某个设备页面 THEN System SHALL 默认选中 `收藏的实体` 标签页
2. WHEN 当前设备存在已收藏实体 THEN System SHALL 在默认标签页中只展示这些实体
3. WHEN 当前设备还没有任何收藏实体 THEN System SHALL 在默认标签页中显示空状态提示，并提供前往 `全部实体` 的明确入口
4. WHEN 用户切换到 `全部实体` 标签页 THEN System SHALL 展示该设备下全部可展示实体

### 需求 2：支持在设备页管理实体收藏

**用户故事：**
作为用户，我希望在查看设备实体时，能直接把常用实体加入收藏或取消收藏，而不是靠记忆去找。

#### 验收标准

1. WHEN 用户在 `全部实体` 标签页查看某个实体 THEN System SHALL 提供收藏或取消收藏入口
2. WHEN 用户收藏某个实体 THEN System SHALL 立即更新该实体在 `收藏的实体` 标签页中的可见性
3. WHEN 用户取消收藏某个实体 THEN System SHALL 立即将其从 `收藏的实体` 标签页移除
4. WHEN 收藏状态更新失败 THEN System SHALL 保留原始状态并给出清楚错误提示

### 需求 3：支持在设备页直接操控支持类型的实体

**用户故事：**
作为用户，我希望在设备页面里直接操作设备，而不是同步完以后还要靠脚本或临时接口验证控制能力。

#### 验收标准

1. WHEN 设备下存在可控制实体 THEN System SHALL 在实体列表中展示统一的控制入口
2. WHEN 用户执行开关、模式切换、数值调整等已支持动作 THEN System SHALL 调用统一设备控制链路执行
3. WHEN 控制成功 THEN System SHALL 刷新实体状态，并在页面上反馈成功结果
4. WHEN 控制失败 THEN System SHALL 给出错误反馈，并允许用户继续查看该实体当前状态
5. WHEN 某实体暂不支持页面内控制 THEN System SHALL 明确显示为只读状态，而不是让用户点了没反应

### 需求 4：支持停用设备

**用户故事：**
作为用户，我希望对暂时不想使用的设备执行停用，而不是只能彻底删除，这样能减少误删风险。

#### 验收标准

1. WHEN 用户在设备页执行停用设备 THEN System SHALL 要求用户确认该操作
2. WHEN 停用成功 THEN System SHALL 将设备状态更新为停用，并在设备页明确展示
3. WHEN 设备已停用 THEN System SHALL 阻止继续执行控制动作，或明确提示该设备当前不可用
4. WHEN 用户返回设备列表 THEN System SHALL 能看到该设备的停用状态

### 需求 5：支持删除设备

**用户故事：**
作为用户，我希望删除已经不再需要的设备，并且页面要把影响说清楚，避免误删后不知道发生了什么。

#### 验收标准

1. WHEN 用户在设备页执行删除设备 THEN System SHALL 先展示确认信息，明确说明这是删除设备，不是删除单个实体
2. WHEN 删除成功 THEN System SHALL 将用户带离当前设备详情，并从设备列表中移除该设备
3. WHEN 删除设备后仍保留历史记录 THEN System SHALL 在产品和接口层明确说明保留策略
4. WHEN 删除失败 THEN System SHALL 保持当前页面状态，并给出清楚错误提示

### 需求 6：支持查看设备操作记录

**用户故事：**
作为用户，我希望在设备页直接查看设备操作记录，这样我能知道最近是谁操作过、操作是否成功，也方便排查异常。

#### 验收标准

1. WHEN 用户进入设备页 THEN System SHALL 提供查看操作记录的入口
2. WHEN 用户查看操作记录 THEN System SHALL 展示至少包括时间、动作、目标实体、结果状态
3. WHEN 能识别操作者来源 THEN System SHALL 展示操作者信息，例如用户、系统或自动化任务
4. WHEN 当前设备还没有操作记录 THEN System SHALL 显示明确的空状态，而不是空白区域

## 非功能需求

### 非功能需求 1：页面可读性

1. WHEN 设备下存在大量实体 THEN System SHALL 仍然保证“默认先看收藏实体”的阅读效率
2. WHEN 用户在移动端或窄屏查看设备页 THEN System SHALL 仍能清楚切换标签页并完成常用操作

### 非功能需求 2：一致性

1. WHEN 不同插件提供可控制实体 THEN System SHALL 复用统一的设备页控制入口和反馈模式
2. WHEN 用户执行删除、停用、收藏、控制等动作 THEN System SHALL 采用一致的确认、加载和错误反馈方式

### 非功能需求 3：可追踪性

1. WHEN 控制动作执行成功或失败 THEN System SHALL 生成可查询的操作记录
2. WHEN 后续排查问题 THEN System SHALL 能按设备维度读取历史操作结果，而不是只能翻系统日志

## 成功定义

- 用户打开设备页后，默认先看到 `收藏的实体`，而不是全量实体
- 用户可以在 `全部实体` 里收藏常用实体，并在 `收藏的实体` 中快速找到它们
- 支持类型的设备可以直接在设备页面执行控制，不再依赖临时测试方式
- 设备页具备停用和删除入口，且行为和反馈清楚
- 用户可以在设备页查看该设备的操作记录
- 整个页面从“能同步设备”提升到“能真实管理和使用设备”
