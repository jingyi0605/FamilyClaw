# 需求文档 - 聊天主链路车道化与异步提案重构

状态：Draft

## 简介

当前聊天主链路的问题，不是某一个 `memory_extraction` 提示词写坏了，而是顶层模型把太多职责塞进了一次前置意图判定里。

现在这条链路默认假设：用户每发一条消息，系统都要先判定这轮到底属于哪一个业务意图，然后再进入对应分支。这个假设在能力很少的时候还能勉强工作，但在家庭场景里会越来越差，因为家庭对话天然会混合：

- 普通闲聊
- 事实查询
- 设备控制
- 提醒草稿
- 长期记忆线索
- Agent 配置修改

这些东西并不总是互斥的。如果还坚持“先判一个主意图，再让每个意图自己补规则”，系统只会越来越依赖特殊情况。

本次重构的目标，是把主链路从“业务意图树”改成“执行约束车道”：

- 哪些请求必须同步判断并立即执行
- 哪些请求需要同步取数后回复
- 哪些请求应该先完成聊天，再异步产出提案

## 术语表

- **快执行车道**：必须快速判定、快速执行、快速回执的链路，典型场景是设备控制、场景触发、停止中的动作。
- **实时取数车道**：需要读取家庭实时状态或结构化数据再回答的链路，例如设备状态、成员状态、提醒状态查询。
- **异步提案车道**：不阻塞主回复，回合结束后统一分析并产出提案的链路，例如记忆、配置建议、提醒草稿。
- **提案分析器（Proposal Analyzer）**：对一轮对话做后处理分析的组件，只负责判断“有没有值得形成提案的东西”。
- **提案批次（Proposal Batch）**：一轮对话的统一提案容器。一个回合最多产出一个批次，里面可以包含 0 到多条提案项。
- **提案项（Proposal Item）**：批次中的具体建议，例如一条记忆提案、一条提醒草稿、一条配置修改建议。
- **策略层（Policy Layer）**：根据提案类型、风险、家庭策略决定是 `ask`、`auto` 还是 `ignore`。
- **证据边界**：某一类提案允许使用哪些消息或事件作为依据。比如记忆提案只能用用户、系统、事件证据，不能直接把助手生成文本当事实。

## 需求

### 需求 1：聊天主链路必须按执行约束分车道，而不是继续扩张业务意图树

**用户故事：** 作为系统维护者，我希望聊天主链路只关心稳定的执行约束，而不是随着业务能力增长不停新增顶层意图分支，这样后续扩展才不会越改越乱。

#### 验收标准

1. WHEN 系统接收一轮聊天请求 THEN System SHALL 先判断它属于快执行、实时取数还是异步提案车道。
2. WHEN 新增一种非强实时业务能力 THEN System SHALL 尽量复用现有车道，而不是要求新增一个顶层主意图分支。
3. WHEN 用户消息同时包含闲聊和后续建议线索 THEN System SHALL 允许先完成聊天，再由异步提案车道统一分析，而不是强行二选一。

### 需求 2：只有强实时且有副作用的动作才能走前置快判

**用户故事：** 作为家庭用户，我希望像开灯、关灯、执行场景这类需要立刻生效的操作能快速执行，但不希望普通聊天也被强行套进动作判定链路。

#### 验收标准

1. WHEN 用户请求设备控制、场景触发或停止中的动作 THEN System SHALL 允许前置快判并优先进入快执行车道。
2. WHEN 用户请求不需要立即执行副作用动作 THEN System SHALL 不因为存在模糊业务意图就抢占快执行车道。
3. WHEN 快执行车道无法稳定识别目标设备、动作或权限 THEN System SHALL 回退到普通聊天解释或澄清，而不是误执行。

### 需求 3：记忆、配置、提醒等非强实时能力必须统一改成异步提案

**用户故事：** 作为产品维护者，我希望长期记忆、配置修改建议、提醒草稿这些能力都走统一的提案机制，而不是每个能力都自己接一条旁路。

#### 验收标准

1. WHEN 一轮对话结束 THEN System SHALL 在不阻塞主回复的前提下统一运行异步提案分析。
2. WHEN 提案分析完成 THEN System SHALL 以提案批次和提案项的形式输出，而不是让每种能力各自产生一套不兼容结果。
3. WHEN 某类提案未命中 THEN System SHALL 返回空结果，而不是触发额外的补丁式回退分支。

### 需求 4：提案必须有严格的证据边界，不能把助手生成内容当事实

**用户故事：** 作为家庭用户，我希望系统只根据我真实表达过的内容、系统事件或家庭事实形成提案，而不是把助手自己编的笑话、建议或总结写进记忆中心。

#### 验收标准

1. WHEN 系统生成记忆提案 THEN System SHALL 只使用用户消息、系统事件或可信外部事件作为事实证据。
2. WHEN 系统生成配置提案 THEN System SHALL 只使用用户明确表达的新设定作为证据，而不是直接采信助手回复。
3. WHEN 某个提案需要引用证据 THEN System SHALL 能追溯到来源消息或事件，而不是只保存一段脱离来源的总结文本。

### 需求 5：提案策略必须统一收口，避免每类能力各自决定执行方式

**用户故事：** 作为系统维护者，我希望提案是否确认、是否自动执行、是否忽略由统一策略层决定，而不是每种能力各写一套 ask/auto 逻辑。

#### 验收标准

1. WHEN 系统产出提案项 THEN System SHALL 统一交给策略层决定 `ask`、`auto` 或 `ignore`。
2. WHEN 家庭策略或风险等级变化 THEN System SHALL 可以在不改主链路编排的前提下调整执行模式。
3. WHEN 提案需要用户确认 THEN System SHALL 产出统一的待确认记录和可追踪状态。

### 需求 6：主链路必须支持未来继续增量扩展，而不是每加能力都改路由器

**用户故事：** 作为后续开发者，我希望未来增加新的异步提案能力时，只需要新增一个分析器并注册，而不是去修改顶层编排核心。

#### 验收标准

1. WHEN 新增一种异步提案能力 THEN System SHALL 支持通过注册分析器接入，而不是修改顶层主路由枚举。
2. WHEN 某个分析器暂时不可用或失败 THEN System SHALL 不影响主聊天回复完成。
3. WHEN 多个分析器同时命中 THEN System SHALL 能在同一个提案批次中统一收口，而不是互相覆盖。

### 需求 7：轻语义路由必须采用统一的 Embedding 检索方案

**用户故事：** 作为系统维护者，我希望第二层语义路由现在就确定成一套稳定方案，而不是后续实现时再在关键词、小模型分类器和大模型之间摇摆。

#### 验收标准

1. WHEN 系统执行第二层轻语义路由 THEN System SHALL 使用 `Embedding + Capability Descriptor Retrieval`，而不是把关键词匹配当主路由。
2. WHEN 新增一个能力或提案类型 THEN System SHALL 通过新增 descriptor 和样例完成接入，而不是要求重新训练分类器。
3. WHEN 语义相似度不足或结果不稳定 THEN System SHALL 保守回落到 `free_chat` 或澄清，而不是误入快执行车道。

### 需求 8：Embedding 供应方式必须同时支持内置轻量模型和用户自定义模型

**用户故事：** 作为系统维护者，我希望系统默认自带可用的轻量 Embedding 能力，开箱即可跑；作为高级用户，我也希望能替换成自己的 Embedding 模型，而不是被系统写死。

#### 验收标准

1. WHEN 系统首次部署且用户没有额外配置 THEN System SHALL 提供一个内置轻量 Embedding 模型作为默认语义路由能力。
2. WHEN 用户提供自定义 Embedding 配置 THEN System SHALL 允许切换到用户指定模型，而不需要修改主链路代码。
3. WHEN 用户自定义 Embedding 不可用 THEN System SHALL 可以回退到内置 Embedding，或者明确拒绝启用新语义路由，而不是静默失效。

### 需求 9：新链路必须能兼容旧聊天能力，并提供可落地迁移路径

**用户故事：** 作为项目负责人，我希望在不打断当前聊天可用性的前提下完成架构切换，而不是一次性推翻重写导致现有能力全坏掉。

#### 验收标准

1. WHEN 新链路落地 THEN System SHALL 保留对现有聊天会话、消息、调试日志和动作记录的兼容读取能力。
2. WHEN 旧的记忆候选或动作确认入口仍在使用 THEN System SHALL 提供兼容映射，不要求前后端同一天全部重写完成。
3. WHEN 新旧链路并存一段时间 THEN System SHALL 能明确区分哪些请求走旧逻辑、哪些请求走新逻辑，并保留可观测性。
