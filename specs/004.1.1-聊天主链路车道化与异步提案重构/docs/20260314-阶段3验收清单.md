# 阶段 3 验收清单

这份清单只回答一个问题：**现在直接切到新提案模型和新动作模型之后，哪些地方必须逐项确认。**

## 1. 主链路

- 普通闲聊还能正常回复
- 实时查询还能并回主聊天回复
- 设备快控还能执行、澄清、拒绝
- 流式返回没有被提案后处理阻塞

通过标准：

- 用户发出一轮消息后，主回复稳定返回
- 没有出现“为了生成提案把主回复拖挂”的情况

## 2. 新提案模型

- 一轮聊天最多只有一个 `proposal batch`
- 一个 `proposal batch` 下面可以有多条 `proposal item`
- `proposal item` 有证据消息 ID、证据角色、去重键、payload
- 记忆和配置提案不会只靠 assistant 文本生成

通过标准：

- 会话详情里能直接看到 `proposal_batches`
- 不再依赖旧 `memory_candidates` / `action_records` 作为主读模型

## 3. 新确认入口

- 能确认 `memory_write`
- 能确认 `config_apply`
- 能确认 `reminder_create`
- 能忽略待确认提案

通过标准：

- 确认后 proposal item 状态正确变化
- 对应目标对象真的创建或更新成功
- 批次状态会跟着 item 状态变化

## 4. 调试日志

一轮请求至少能追到这些信息：

- lane 结果
- proposal pipeline 是否执行
- proposal batch id
- proposal 数量
- proposal item 确认或忽略动作
- 失败时的错误信息

通过标准：

- 拿一轮请求 ID，可以从日志里一路看到主回复、提案、确认动作

## 5. 旧入口下线

- 会话详情不再把旧候选和旧动作记录当主模型返回
- 正式 API 不再要求走旧记忆候选确认入口
- 正式 API 不再要求走旧动作确认入口

通过标准：

- 新前后端调用只依赖 proposal batch / proposal item
- 旧入口即使还留在代码里，也不是正式主路径

## 6. 建议回放样例

建议至少人工回放下面这些输入：

- “讲个笑话”
- “现在家里有人吗”
- “把客厅灯关掉”
- “把灯关掉”
- “以后你就叫阿福”
- “记住，我不吃香菜”
- “明天早上八点提醒我开会”

这些样例分别对应：

- 普通聊天
- 实时取数
- 快执行成功
- 快执行澄清
- 配置提案
- 记忆提案
- 提醒提案
