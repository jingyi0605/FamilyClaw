# 聊天主链路架构取舍说明

日期：2026-03-14

## 这份说明回答什么问题

这份说明只回答一个问题：

当前聊天主链路，到底应该继续走“先判业务意图，再分支执行”，还是改成“先完成聊天，再统一做后处理提案”？

结论先说：

- 不能全部都先聊后判
- 也不能继续把所有能力都塞进前置意图树
- 更合适的是三车道

## 方案一：继续扩张前置业务意图树

典型做法：

- 先判 `free_chat`
- 再判 `structured_qa`
- 再判 `memory_write`
- 再判 `config_change`
- 再判 `reminder_create`
- 以后还会继续加别的

### 优点

- 对少量、边界很硬的能力来说，写起来直接
- 每个分支局部调试简单

### 致命问题

- 新能力越来越多时，顶层路由会越来越重
- 各分支会发展出自己的 fallback、guardrail 和补丁
- 很多能力天然不是互斥分支，而是聊天后的建议或草稿
- 系统会越来越依赖“特殊情况处理”

一句话：这条路短期省事，长期一定烂。

## 方案二：所有事情都先自由聊天，再统一后判

典型做法：

- 每一轮都先回复用户
- 然后再异步分析有没有动作、记忆、提醒、配置建议

### 优点

- 顶层主链路很干净
- 记忆、提醒、配置这类非强实时能力很适合这样做

### 致命问题

- 设备控制、场景触发这类动作需要快判和快回执
- 门锁、灯光、停止中的动作不能先聊一段再慢慢判断
- 一些实时查询也需要先拿到真实数据再答，不能纯靠闲聊兜底

一句话：这条路对提案型能力很好，但不能覆盖全部家庭场景。

## 方案三：三车道

这是当前最合适的方案。

### 车道一：快执行

只给这类请求：

- 开灯、关灯
- 执行场景
- 停止中的动作
- 其他明确的强实时副作用动作

特点：

- 前置快判
- 权限先过
- 目标不明确先澄清
- 执行后给回执

### 车道二：实时取数

这类请求本质上还是聊天，只是回答前必须同步查真实数据：

- 现在谁在家
- 客厅灯开着没
- 今天还有没有提醒

特点：

- 不必占顶层业务意图树分支
- 更适合作为聊天主回复中的实时能力

### 车道三：异步提案

这类请求不要求立刻落动作：

- 记忆提案
- 配置建议
- 提醒草稿
- 以后扩展的偏好、关系、成长类提案

特点：

- 主回复先完成
- 回合后统一分析
- 统一策略层决定 ask / auto / ignore

## 为什么最终选三车道

因为它把顶层判断从“业务名词”换成了“执行约束”。

业务名词会越来越多，执行约束不会。

顶层真正稳定的问题只有这几个：

1. 这轮是不是必须立刻做事
2. 这轮是不是必须立刻拿到真实数据
3. 这轮会不会带来副作用

只要顶层围绕这几个问题组织，后面再加能力也不容易失控。

## 对当前项目的直接影响

### 该保留的

- 设备控制的快路径思路
- 主聊天回复能力
- 现有会话和消息持久化
- 动作策略层的收口思路

### 该收掉的

- 让 `memory_write`、`config_change`、`reminder_create` 继续占据顶层主路由
- `free_chat` 结束后再偷偷补一条暗路提取
- 把助手文本直接当长期记忆事实

## 最后的判断

如果后面能力会越来越多，那么继续扩张前置意图树，几乎肯定会变成维护灾难。

更合理的方向是：

- 顶层只有三车道
- 非强实时能力统一做提案分析器
- 新能力优先通过 analyzer 注册，而不是继续改主路由

## 004.1 旧主链路真实入口和暗路

这一段专门回答一个很实际的问题：**今天线上这条链路到底是怎么跑的。**

别再靠感觉说“应该是这样的”。当前真实走法可以收口成下面这条顺序：

1. `conversation/service.py` 接收一轮消息，先写入 user message，再创建 assistant message 占位。
2. service 调 `run_orchestrated_turn` / `stream_orchestrated_turn` 进入编排层。
3. `conversation/orchestrator.py` 先做一次顶层意图识别，当前返回的还是业务意图名词：
   - `structured_qa`
   - `free_chat`
   - `config_change`
   - `memory_write`
   - `reminder_create`
4. 编排层直接按这些业务意图名词分支：
   - `structured_qa` 走家庭问答
   - `config_change` 走配置提取
   - `memory_write` 走记忆提取
   - `reminder_create` 走提醒提取
   - 其他落回 `free_chat`
5. 编排层返回结果后，service 再进入动作策略层，把配置建议、记忆候选、提醒草稿写成旧候选或旧动作记录。
6. 旧前端继续读取旧候选入口和旧确认入口完成展示、确认、执行、撤销。

把它画成一行，就是：

`service 收消息 -> orchestrator 判业务意图 -> 各提取分支/聊天分支 -> service 动作策略层 -> 旧候选表/旧动作记录 -> 前端确认入口`

### 旧链路里的暗路在哪里

旧链路最脏的地方不是“提取效果偶尔不准”，而是顶层模型本身就把不该放在一起的东西塞在一起了。

当前至少有这几条暗路：

- `free_chat` 和 `memory_write` / `config_change` / `reminder_create` 是同一层竞争关系，谁命中先谁抢主链路。
- 配置、记忆、提醒本来应该是回合后分析的提案型能力，现在却直接挤进主路由。
- service 里的动作策略层继续消费 orchestrator 输出，把旧候选表和旧确认入口挂在主回复后面，导致“主回复”和“提案执行”逻辑缠死。

### 哪些旧分支以后要下线

后面真正要回收的，不是聊天主回复本身，而是这些顶层脏分支：

- 顶层直接把 `config_change` 当主路由
- 顶层直接把 `memory_write` 当主路由
- 顶层直接把 `reminder_create` 当主路由
- `free_chat` 后面再偷偷补提取的暗路思维

要保留的地板很明确：

- 会话和消息持久化
- 当前主聊天回复
- 当前流式返回
- 当前旧确认入口

一句话总结：

**旧链路真正的问题，不是某个分支提示词不够准，而是顶层把“聊天、提案、动作确认”揉成了一锅。**
