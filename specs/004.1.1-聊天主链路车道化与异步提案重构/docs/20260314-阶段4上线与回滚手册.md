# 阶段 4 上线与回滚手册

这份手册只回答三件事：

1. 影子模式怎么开
2. 真正切换怎么做
3. 出问题时怎么回滚

## 1. 影子模式

影子模式的目标不是接管，而是先看新链路判断得稳不稳。

当前开关：

- `conversation_lane_shadow_enabled`
- `conversation_proposal_shadow_enabled`

影子模式下应该满足：

- lane 结果会记录到日志
- proposal pipeline 会记录到日志
- 主回复不因为影子链路改变
- 新提案不会落正式写入

如果影子模式下主回复被影响，说明不是影子模式，是实现有 bug。

## 2. 切换顺序

建议顺序：

1. 先开 `conversation_lane_shadow_enabled`
2. 再开 `conversation_proposal_shadow_enabled`
3. 看日志确认 lane 和 proposal 都稳定
4. 再开 `conversation_lane_takeover_enabled`
5. 最后确认 `conversation_proposal_write_enabled`

这里不要反过来。

如果连影子日志都还没看明白，就不要直接切换。

## 3. 切换前要看什么

至少看这几类输入：

- 普通闲聊
- 实时查询
- 快执行设备控制
- 配置修改
- 记忆提案
- 提醒提案

重点不是“模型看起来聪明不聪明”，重点是：

- 会不会误入快执行
- 会不会误产出脏提案
- 会不会拖慢主回复

## 4. 立即暂停条件

下面任何一条出现，都该立刻暂停：

- 主回复失败率明显升高
- 设备快控误执行
- 普通闲聊持续产出脏提案
- 新确认入口不可用
- 日志看不出 lane / proposal / confirm 动作

## 5. 回滚动作

直接回滚顺序：

1. 关 `conversation_lane_takeover_enabled`
2. 关 `conversation_proposal_write_enabled`
3. 必要时连 `conversation_proposal_shadow_enabled` 也关掉
4. 如果还不稳，回退到上一版稳定发布

不要在生产故障时一边猜一边改代码。

先关开关，先恢复服务，再排查。

## 6. 回滚后核对

回滚后至少确认：

- 普通聊天恢复
- 实时查询恢复
- 设备快控恢复
- 会话和消息没有丢
- 新旧日志能分辨出回滚前后差异

## 7. 最后的原则

上线不是比谁胆子大。

真正靠谱的做法是：

- 先看影子
- 再小范围切换
- 出问题先回滚
- 回来再修
