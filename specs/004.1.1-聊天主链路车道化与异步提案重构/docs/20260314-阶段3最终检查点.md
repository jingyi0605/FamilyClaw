# 阶段 3 最终检查点

这一步不是再加功能，而是确认我们有没有把旧复杂度真的砍掉。

## 现在应该成立的事实

1. 顶层主链路只看三车道：
   - `fast_action`
   - `realtime_query`
   - `free_chat`
2. `memory_write`、`config_change`、`reminder_create` 不再作为顶层主路由继续膨胀
3. 非强实时能力统一走提案机制
4. 会话详情默认读新提案模型
5. 正式确认入口默认走新 proposal item

## 现在不该再成立的事实

- 不该再把旧 `memory_candidates` 当主展示模型
- 不该再把旧 `action_records` 当主确认模型
- 不该再把“保守记忆补提取”当正式主路径
- 不该再把“先判业务意图名词”当顶层设计前提

## 如果后面有人继续开发

后面再加能力时，应该优先问这三个问题：

1. 这是快执行、实时取数，还是异步提案？
2. 这东西能不能复用现有车道，而不是再加一个顶层意图？
3. 这东西是不是应该新增 analyzer，而不是改顶层路由？

如果回答偏离这三条，说明又开始往旧脏路上滑了。
