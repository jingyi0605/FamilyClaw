# 20260314-语义路由回退说明

## 这份说明解决什么问题

这份文档专门记录一件事：`004.1.1` 里原本尝试引入的语义路由方案，已经在 2026-03-14 回退，不能再把它当成当前默认实现。

如果不把这件事写清楚，后面接手的人会犯两个低级错误：

1. 以为 `design.md` 里写过的语义路由还是现行基线。
2. 以为代码里只是暂时关掉开关，随时可以再打开。

这两个判断现在都不对。

## 发生了什么

2026-03-14 的评估报告已经给出明确结果：

- 旧方案准确率：`24/24 = 100%`
- 新方案准确率：`22/24 = 91.67%`
- 旧方案热路径平均耗时：`0.01 ms`
- 新方案热路径平均耗时：`8.647 ms`
- 新方案冷启动首轮平均耗时：`2592.514 ms`

结论很直接：这次语义路由不是“还有点问题”，而是当前版本下既更慢，也更不准。

## 当前已经做掉的变更

- 车道选择回到旧方案：`hard_signal + intent_mapping`
- 删除 `semantic_router`、descriptor、embedding provider 相关代码
- 删除相关依赖和测试
- 保留评估报告，作为回退依据

## 当前基线到底是什么

`004.1.1` 当前还成立的东西只有这些：

- 主链路继续保持三车道思路：`fast_action`、`realtime_query`、`free_chat`
- `memory_write`、`config_change`、`reminder_create` 继续留在 `free_chat` 之后的统一提案链路
- 顶层路由不再继续膨胀成业务意图树

当前不成立、也不应该再被当成默认实现的东西：

- `Embedding + Capability Descriptor Retrieval`
- `semantic_router.py`
- embedding provider 切换与 descriptor cache
- `conversation_embedding_provider_enabled` 这一套配置和开关

## 这次明确不做什么

- 不继续在当前方案上微调阈值、分差、descriptor 文案
- 不保留“先删代码，回头再开开关”的伪兼容
- 不把已经回退的方案继续写成默认路线

## 如果以后还想再试语义路由

可以，但前提必须写死：

1. 先补新的评估数据，不要靠感觉。
2. 至少同时证明准确率不低于旧方案、速度成本可接受。
3. 重新开 Spec 修订任务，再决定要不要进主链路。

在这三个条件满足前，当前基线就是旧方案，不要自欺欺人。
