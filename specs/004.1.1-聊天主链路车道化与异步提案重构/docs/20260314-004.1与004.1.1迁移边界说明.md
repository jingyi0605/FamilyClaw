# 004.1 与 004.1.1 迁移边界说明

日期：2026-03-14

## 这份说明为什么重要

这份说明只解决一件事：

在引入 `004.1.1-聊天主链路车道化与异步提案重构` 时，怎么保证不把当前本来还能正常工作的聊天链路一起打坏。

这里必须先把一句话写死：

**`004.1.1` 不是替换 `004.1` 的立即重写命令，而是建立在 `004.1` 现有可用基础上的渐进式收口方案。**

也就是说：

- `004.1` 先继续承担当前生产可用聊天链路的稳定基线
- `004.1.1` 只负责把顶层编排方式慢慢收口
- 没有完成迁移之前，不能为了追新架构去破坏当前正常聊天能力

## 两个 Spec 分别负责什么

### `004.1` 的职责

`004.1-聊天模块全链路重构` 负责的是当前聊天系统“先能用起来”的基础设施和主流程，包括：

- 服务端真实会话
- 服务端真实消息
- 多轮上下文拼装
- 聊天回复生成
- 流式返回
- 失败回执
- 调试日志
- 当前版本的记忆候选、动作记录、确认入口

一句人话：

`004.1` 负责把聊天系统从“看起来像聊天”变成“至少是一个可用的聊天系统”。

### `004.1.1` 的职责

`004.1.1-聊天主链路车道化与异步提案重构` 负责的是：

- 不再继续扩张顶层业务意图树
- 把顶层主链路收成三车道
- 把记忆、配置、提醒等非强实时能力从主路由里剥出去
- 建立统一的异步提案模型、策略层和迁移路径

一句人话：

`004.1.1` 负责把现在这条已经能跑的链路，改造成以后还能继续扩展而不发臭的链路。

## 这两个 Spec 的边界

### 边界一：`004.1` 是当前生产基线，`004.1.1` 不是立即替换

在 `004.1.1` 真正完成之前，默认规则必须是：

- 当前线上聊天主链路继续按 `004.1` 的可用实现运行
- `004.1.1` 的新能力先以开关、影子模式、兼容写入的方式接入
- 不能因为新链路还没完全补齐，就把旧链路硬拆掉

这条边界很重要。否则就会出现最蠢的情况：

- 新链路只做了一半
- 旧链路先被砍了
- 最后用户看到的是“架构更先进了，但聊天反而坏了”

这种事不能发生。

### 边界二：`004.1` 继续保底聊天可用性，`004.1.1` 只收口顶层编排

在迁移期内，以下能力继续优先归 `004.1` 保底：

- 会话创建与查询
- 消息写入与恢复
- 流式返回
- 失败处理
- 当前聊天页能正常收发消息
- 当前结构化问答和普通聊天能正常返回结果

`004.1.1` 在迁移期优先改的是：

- 顶层如何决定走哪条车道
- 提案后处理怎么统一收口
- 新提案模型怎么兼容旧候选入口

换句话说：

`004.1.1` 先改“脑子”，不先改“手脚”。

### 边界三：不允许一上来就全面替换前端读模型

迁移期内：

- 旧前端如果还在读取记忆候选或动作确认入口，允许继续读旧接口或兼容视图
- 新提案模型可以先在后端落地，再通过兼容层映射给旧 UI
- 只有当新提案批次和提案项已经稳定后，才允许前端逐步切到新读模型

这能避免前后端同步改动过大，把当前聊天页一起拖死。

## 迁移原则

### 原则 1：先加新路，不先拆旧路

落地顺序必须是：

1. 先引入新车道选择器或提案模型
2. 先保留旧链路继续可用
3. 先做兼容映射
4. 等观测稳定后，再逐步下线旧分支

不能倒过来。

### 原则 2：任何时候都要保住当前正常聊天

“当前正常聊天”包括最基本的几件事：

- 用户发消息后能收到回复
- 会话和消息不会丢
- 流式体验不被新后处理阻塞
- 现有页面不因为提案模型升级而白屏或失去确认入口

如果某个 `004.1.1` 的改动威胁到这些基础能力，默认应该：

- 关开关
- 回退到 `004.1` 基线
- 先修新链路，再考虑再次放量

### 原则 3：新链路先做影子运行，再做真实接管

建议按下面三段走：

#### 第一段：影子模式

新链路运行，但只做记录，不改变主回复结果，不改变现有前端行为。

重点观察：

- 车道判定是否稳定
- 异步提案是否产生脏数据
- 与旧候选结果相比偏差多大

#### 第二段：兼容模式

新链路开始真实写入新提案模型，但旧前端仍然走兼容读取。

重点观察：

- 旧页面能否稳定展示兼容结果
- 新旧状态是否能对应
- 是否出现重复卡片、重复确认、重复写入

#### 第三段：接管模式

当新链路足够稳定后，才逐步切换：

- 顶层车道选择器接管主路由
- 提案模型接管旧候选模型
- 旧暗路和补提取分支逐步下线

## 哪些东西现在绝对不能动

在 `004.1.1` 早期阶段，下面这些不能因为“架构更优雅”就先拆：

- `004.1` 已经可用的会话和消息持久化
- 当前聊天主回复的基础链路
- 当前流式返回机制
- 当前错误回执和失败展示
- 当前前端能正常工作的确认入口

原因很简单：

这些东西是用户现在正在用的地板，不是以后再补的天花板。

## 哪些东西应该优先迁移

优先迁移那些“坏味道重，但对主回复不致命”的部分：

1. 顶层业务意图树的收口
2. `free_chat` 后面的保守补提取暗路
3. 记忆、配置、提醒的统一提案化
4. 提案证据边界
5. 旧候选表到新提案模型的兼容映射

这些地方先改，收益最大，也最不容易把主聊天回复直接打断。

## 推荐切换顺序

### 第一步：只加观测，不改主行为

先给新车道选择器和提案分析器加影子运行：

- 不改当前回复
- 不改当前 UI
- 先记录日志和比较结果

### 第二步：先接管提案，不先接管主回复

因为提案型能力本来就适合回合后处理，所以可以先迁：

- 记忆提案
- 配置提案
- 提醒草稿提案

这样即使有问题，也不至于让聊天主回复直接挂掉。

### 第三步：最后再收顶层路由

等提案层稳定后，再逐步把顶层从业务意图树切成三车道。

原因是顶层路由一旦切错，影响的是整轮聊天；提案层切错，影响的通常只是候选或建议。

所以顶层改动必须放后面，不放前面。

## 风险控制要求

为了保证不影响当前正常链路，`004.1.1` 的实施至少要满足这些要求：

1. 新链路必须有功能开关。
2. 新链路必须支持影子运行。
3. 新旧结果必须能在日志里对比。
4. 提案新模型上线前必须先有兼容读取方案。
5. 任何时候都必须能快速回退到 `004.1` 的生产基线。

如果这些前提没满足，就不应该让 `004.1.1` 去接管主链路。

## 最后的落地判断

真正安全的迁移方式不是：

- 先把旧东西推翻
- 再指望新架构一次成功

真正安全的迁移方式是：

- 先承认 `004.1` 是当前可用基线
- 再用 `004.1.1` 去逐步收口顶层脏结构
- 每一步都保证聊天主回复、会话持久化、流式体验和现有确认入口不被打断

一句话总结：

**`004.1` 负责今天还能稳，`004.1.1` 负责明天不会烂。迁移时先保今天，再换明天。**
