# Spec 004.1.1 - 聊天主链路车道化、提案统一化与直接切换

状态：已修订（2026-03-14，语义路由方案已回退）

## 2026-03-14 当前基线修订

这份 Spec 现在必须补一条修订说明。原因很直接：文档里原先把 `Embedding + Capability Descriptor Retrieval` 写成了既定落地方案，但 2026-03-14 的评估已经证明，这套实现当前版本同时输了速度和准确率。

- 当前已经确认的事实：
  - 旧方案准确率 `24/24 = 100%`
  - 新语义路由方案准确率 `22/24 = 91.67%`
  - 旧方案热路径平均 `0.01 ms`
  - 新方案热路径平均 `8.647 ms`
  - 新方案冷启动首轮平均 `2592.514 ms`
- 当前已经落地的调整：
  - 运行时代码已移除 `semantic_router`、descriptor、embedding provider 和相关依赖
  - 车道选择已回到 `hard_signal + intent_mapping`
  - 评估报告保留，作为这次回退的直接证据
- 这份 Spec 从现在开始怎么读：
  - 三车道主链路、统一异步提案，仍然是当前有效方向
  - 语义路由和 embedding 章节，改为“尝试过但已回退”的设计记录，不再代表当前实现基线
  - 如果以后还想重新引入语义路由，必须先有新的评估结果，再开新的修订任务，不能把死方案继续当默认方案

## 这份 Spec 解决什么问题

这份 Spec 不再继续给 `004.1-聊天模块全链路重构` 打补丁，而是直接重写聊天主链路的顶层架构，并切到新的提案与动作模型。

当前链路最大的坏味道不是某一个记忆提取阈值不对，而是主链路把“聊天回复”“意图判断”“提案生成”“动作执行”混在了一棵不断长分支的意图树里。结果就是：

- 普通闲聊也会被拉进记忆、配置、提醒等提取链路
- 为了修误判，只能不停加边界规则和例外分支
- 新能力一多，主路由会越来越像垃圾场

这份 Spec 的目标很直接：

- 把聊天主链路改成稳定的三车道模型
- 只有强实时动作走前置快判
- 其他能力先完成正常聊天，再走统一的异步提案分析
- 不再让记忆、配置、提醒这些能力抢占主聊天链路

## 为什么要单开 004.1.1

因为这里改的不是某个局部实现，而是顶层编排模型。

如果继续在 `004.1` 上修，只会变成：

- 再补一个 guardrail
- 再多一个 fallback
- 再多一个“只对某类意图生效”的例外

这种改法没有品味，也扛不住后续能力扩张。正确做法是单开一个子 Spec，把主链路模型重说一遍，然后按新模型回收旧实现。

## 本次范围

本次只处理聊天主链路的编排方式，包括：

- 顶层路由从“业务意图树”改成“执行约束车道”
- 区分强实时动作、实时取数、异步提案三类链路
- 设计统一的提案分析器注册表和策略层
- 明确记忆、配置、提醒等提案的证据边界
- 设计新的提案批次与提案项数据模型
- 规划直接切到新提案模型和新动作模型后的落地顺序

## 本次明确不做

- 不在这里展开具体设备控制协议细节
- 不在这里重写插件市场或插件注册表
- 不在这里直接替换所有前端展示细节
- 不在这里把每一种未来能力都提前设计完

## 一页看懂阶段 0

阶段 0 只做两件事：

1. 把旧主链路的真实入口、暗路、动作策略层、旧候选入口画清楚。
2. 把顶层判断先收口成三车道：
   - `fast_action`
   - `realtime_query`
   - `free_chat`

这里最重要的一条边界是：

- `memory_write`
- `config_change`
- `reminder_create`

这三个东西以后都**不再继续扩成顶层主路由名词**，它们属于 `free_chat` 车道后的提案能力，不该继续抢主回复入口。

当前方案已经改成直接切换，不再长期维护新旧双轨。阶段 0 的目的仍然是先把旧问题说明白，避免在错误模型上继续补丁。
