# Spec 004.1.1 - 聊天主链路车道化与异步提案重构

状态：Draft

## 这份 Spec 解决什么问题

这份 Spec 不再继续给 `004.1-聊天模块全链路重构` 打补丁，而是单独处理聊天主链路的顶层架构问题。

当前链路最大的坏味道不是某一个记忆提取阈值不对，而是主链路把“聊天回复”“意图判断”“提案生成”“动作执行”混在了一棵不断长分支的意图树里。结果就是：

- 普通闲聊也会被拉进记忆、配置、提醒等提取链路
- 为了修误判，只能不停加边界规则和例外分支
- 新能力一多，主路由会越来越像垃圾场

这份 Spec 的目标很直接：

- 把聊天主链路改成稳定的三车道模型
- 只有强实时动作走前置快判
- 其他能力先完成正常聊天，再走统一的异步提案分析
- 不再让记忆、配置、提醒这些能力抢占主聊天链路

## 为什么要单开 004.1.1

因为这里改的不是某个局部实现，而是顶层编排模型。

如果继续在 `004.1` 上修，只会变成：

- 再补一个 guardrail
- 再多一个 fallback
- 再多一个“只对某类意图生效”的例外

这种改法没有品味，也扛不住后续能力扩张。正确做法是单开一个子 Spec，把主链路模型重说一遍，然后按新模型回收旧实现。

## 本次范围

本次只处理聊天主链路的编排方式，包括：

- 顶层路由从“业务意图树”改成“执行约束车道”
- 区分强实时动作、实时取数、异步提案三类链路
- 设计统一的提案分析器注册表和策略层
- 明确记忆、配置、提醒等提案的证据边界
- 设计新的提案批次与提案项数据模型
- 规划与 `004.1` 现有聊天实现的兼容迁移路径

## 本次明确不做

- 不在这里展开具体设备控制协议细节
- 不在这里重写插件市场或插件注册表
- 不在这里直接替换所有前端展示细节
- 不在这里把每一种未来能力都提前设计完

## 主要文档

- `requirements.md`：为什么要改、要改成什么样、什么算完成
- `design.md`：主链路怎么拆、数据怎么收口、如何迁移
- `tasks.md`：按什么顺序落地，先改哪里，怎么验收
- `docs/20260314-聊天主链路架构取舍说明.md`：为什么不继续堆意图树
- `docs/20260314-004.1与004.1.1迁移边界说明.md`：说明 `004.1` 和 `004.1.1` 的职责边界、切换顺序和保底策略

## 一页看懂阶段 0

阶段 0 只做两件事：

1. 把 `004.1` 旧主链路的真实入口、暗路、动作策略层、旧候选入口画清楚。
2. 把顶层判断先收口成三车道：
   - `fast_action`
   - `realtime_query`
   - `free_chat`

这里最重要的一条边界是：

- `memory_write`
- `config_change`
- `reminder_create`

这三个东西以后都**不再继续扩成顶层主路由名词**，它们属于 `free_chat` 车道后的提案能力，不该继续抢主回复入口。

阶段 0 不追求一步接管生产链路，只先把旧问题说清楚、把 lane 骨架立起来，避免后面继续在错误模型上打补丁。
