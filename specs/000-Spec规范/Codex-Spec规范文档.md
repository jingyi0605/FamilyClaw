# Codex Spec 规范文档

## 1. 文档目的

本规范文档用于指导 AI IDE 或 Codex 在 `specs/` 目录下创建一套完整、可追踪、可执行、可维护的 Spec。

目标不是只生成三份形式上的 Markdown，而是生成一套真正可用于后续实现、评审、交接、验收和归档的 Spec 资产。

后续 AI IDE 创建新 Spec 时，应同时参考：

1. 本规范文档
2. 模板示例 `/Users/jackson/Code/FamilyClaw/specs/000-Spec规范/Spec模板`

---

## 2. Spec 的最小交付标准

每个新 Spec 至少必须包含以下文件：

```text
specs/<spec-id>/
├── requirements.md
├── design.md
├── tasks.md
└── docs/               # 可选
```

说明：

- `requirements.md`：定义业务目标、范围和验收标准
- `design.md`：定义技术方案、接口、数据结构、错误处理、测试策略
- `tasks.md`：定义执行计划、任务依赖、状态、上下文入口和任务验收
- `docs/`：放接口文档、联调说明、验收结果、补充说明等

如果一个 Spec 没有这三份主文档，则视为不完整。

---

## 3. 三份主文档的职责分工

### 3.1 `requirements.md`

作用：回答“为什么做”“做什么”“什么算完成”

必须包含：

- 简介或背景
- 术语表
- 需求列表
- 每条需求的用户故事
- 每条需求的验收标准

### 3.2 `design.md`

作用：回答“怎么做”“由哪些模块组成”“边界和错误如何处理”“如何验证方案正确”

必须包含：

- 概述
- 架构或信息结构
- 组件与接口
- 数据模型、状态模型或外部对象结构
- 错误处理
- 测试策略

建议包含：

- 正确性属性 / 不变量
- 风险与回滚策略
- 性能约束

### 3.3 `tasks.md`

作用：回答“先做什么、后做什么、每个任务依赖什么、完成标准是什么、打开单个任务后如何立即开始”

必须包含：

- 任务分组
- 每个任务的唯一编号
- 任务状态
- 任务依赖
- 需求关联
- 设计关联
- 上下文入口
- 验收标准
- 验证方式

---

## 4. `requirements.md` 编写规范

### 4.1 标准结构

推荐结构如下：

```markdown
# 需求文档 - <功能名称>

状态：Draft

## 简介

## 术语表

## 需求

### 需求 1：<需求名称>

**用户故事：** 作为 <角色>，我希望 <能力>，以便 <价值>

#### 验收标准

1. WHEN <触发条件> THEN System SHALL <可验证结果>
2. WHEN <触发条件> THEN System SHALL <可验证结果>
```

### 4.2 编写规则

1. 每条需求必须有稳定编号，例如 `需求 1`、`需求 2`
2. 每条需求必须有用户故事，明确角色、能力、价值
3. 每条需求必须有验收标准
4. 验收标准必须使用 `WHEN / THEN / SHALL`
5. 验收标准必须能映射到设计和任务，避免空泛表述

### 4.3 常见问题

不要这样写：

- 支持站点管理
- 优化用户体验
- 增强系统稳定性

应改成可验证形式，例如：

- WHEN 用户访问站点列表 THEN System SHALL 在 2 秒内返回结果
- WHEN 用户输入非法 CIDR THEN System SHALL 阻止提交并显示明确错误消息

### 4.4 AI 生成要求

AI IDE 在生成 `requirements.md` 时，必须先做以下动作：

1. 识别业务背景和目标
2. 拆出核心对象和术语
3. 列出功能需求
4. 列出非功能需求（如性能、安全、可靠性）
5. 为每条需求补全可验证的验收标准

---

## 5. `design.md` 编写规范

### 5.1 标准结构

推荐结构如下：

```markdown
# 设计文档 - <功能名称>

状态：Draft

## 1. 概述

## 2. 架构

## 3. 组件和接口

## 4. 数据模型

## 5. 错误处理

## 6. 测试策略
```

### 5.2 编写规则

1. 设计文档必须使用稳定章节编号，例如 `§2.1`、`§4.3`
2. 每个关键章节都必须能被任务精确引用
3. 每个接口描述至少包含：
   - 输入
   - 输出
   - 校验约束
   - 边界条件
   - 错误返回
4. 每个关键数据结构必须说明字段、约束和语义
5. 每个关键业务规则必须尽量写成“正确性属性”或“不变量”

### 5.3 正确性属性建议

如果功能包含 CRUD、流程状态、数据同步、权限控制等场景，建议补充正确性属性，例如：

- 创建后可查询
- 更新后关键字段已改变
- 删除后不可查询
- 非法输入必须拒绝
- 有依赖对象时禁止删除

### 5.4 AI 生成要求

AI IDE 在生成 `design.md` 时，必须确保：

1. 设计章节能覆盖所有需求
2. 每个关键模块都能映射到后续任务
3. 接口、模型和错误处理足够细，不能只写抽象描述
4. 测试策略能直接支持 `tasks.md` 的任务拆分

---

## 6. `tasks.md` 编写规范

这是最关键的部分。AI IDE 生成 Spec 时，`tasks.md` 不能只是一个待办清单，而必须是可被 Codex 独立执行的任务编排文档。

### 6.1 任务分组规则

任务必须先分组，再列任务。推荐分组方式：

- 按阶段：基础搭建、后端、前端、联调、验收、归档
- 按工作流：建模、接口、页面、测试、发布
- 按里程碑：M1、M2、M3

禁止把所有任务混成一个长列表。

### 6.2 每个任务必须具备的字段

每个任务必须包含以下字段：

- 编号
- 标题
- 状态
- 目标
- 依赖
- 需求关联
- 设计关联
- 上下文入口
- 涉及产物
- 执行说明
- 验收标准
- 验证方式

推荐模板：

```markdown
- [ ] 1.1 <任务标题>
  - 状态：TODO
  - 目标：<一句话说明完成后交付什么>
  - 依赖：无 / 1.1, 1.2
  - 需求关联：`requirements.md` 需求 4 / 验收 4.2、4.4
  - 设计关联：`design.md` §5.1、§5.4、§5.6
  - 上下文入口：
    - `requirements.md` 需求 4
    - `design.md` §5.4「任务上下文包」
    - `docs/api.md` §2（如适用）
  - 涉及产物：`tasks.md`、`design.md`、`docs/api.md`
  - 执行说明：<本任务改什么，不改什么>
  - 验收标准：
    1. <可检查结果>
    2. <可检查结果>
  - 验证方式：
    - <命令、人工检查或测试方式>
```

### 6.3 任务状态规则

任务状态统一使用：

- `TODO`
- `IN_PROGRESS`
- `BLOCKED`
- `IN_REVIEW`
- `DONE`
- `CANCELLED`

补充规则：

- 只有 `状态：DONE` 的任务才能勾选 `[x]`
- `BLOCKED` 必须写阻塞原因和恢复条件
- `CANCELLED` 必须写取消原因和替代方案（如有）

### 6.4 任务依赖规则

每个任务都必须有 `依赖` 字段：

- 无依赖时写 `无`
- 有依赖时写任务编号，例如 `2.1、2.2`

依赖不是推荐顺序，而是执行前提。

如果一个任务需要接口先完成、数据模型先存在、上游方案先明确，就必须明确写入依赖。

### 6.5 任务上下文包规则

为了保证“任意打开一个任务，Codex 都能快速执行”，每个任务必须提供最小上下文包。

最小上下文包至少包括：

1. 当前任务目标
2. 对应的需求编号和验收标准
3. 对应的设计章节
4. 开始前必须阅读的文档入口
5. 需要修改的文件、接口、模块或命令
6. 不在本任务范围内的内容
7. 完成后的验证方法

这条规则是本规范最重要的要求之一。

### 6.6 任务验收规则

任务验收和需求验收不是一回事：

- 需求验收：判断“产品能力是否成立”
- 任务验收：判断“当前工作单元是否完成”

因此，每个任务都必须有自己的任务级验收标准。

例如：

- 需求可能要求“支持站点 CRUD”
- 某个任务只负责“创建 Site API 路由”
- 那么该任务的验收必须是“相关路由已实现、认证已接入、错误处理已补齐、可通过指定方式验证”

## 7. Spec 生命周期规范

Spec 级状态建议统一使用：

- `Draft`
- `In Review`
- `Approved`
- `In Progress`
- `Blocked`
- `Done`
- `Archived`

建议流转为：

`Draft` → `In Review` → `Approved` → `In Progress` → `Done` → `Archived`

`Blocked` 为中途异常状态。

进入各状态的建议条件：

- `Draft`：文档开始编写
- `In Review`：三份主文档已齐备
- `Approved`：需求、设计、任务已通过评审
- `In Progress`：存在可执行任务
- `Done`：关键任务和检查点已完成
- `Archived`：文档和验证证据已归档

## 8. AI IDE 创建新 Spec 的标准流程

当 AI IDE 被要求“创建一个新 Spec”时，应严格按以下顺序执行：

### 第一步：分析需求

输出：

- 背景
- 目标
- 范围
- 关键对象
- 风险和边界

### 第二步：生成 `requirements.md`

要求：

- 至少列出主要功能需求
- 每条需求都有用户故事
- 每条需求都有 `WHEN / THEN / SHALL` 验收标准

### 第三步：生成 `design.md`

要求：

- 根据需求设计模块、接口、模型和错误处理
- 提供稳定章节编号
- 给出测试策略和关键正确性属性

### 第四步：生成 `tasks.md`

要求：

- 按阶段或工作流分组
- 每个任务都带状态、依赖、需求关联、设计关联、上下文入口
- 每个任务都能让新的 Codex 上下文独立启动

### 第五步：自检

AI IDE 必须做以下检查：

1. 是否有缺失的主文档
2. 是否有需求没有设计支撑
3. 是否有设计没有对应任务
4. 是否有任务没有需求关联
5. 是否有任务没有设计关联
6. 是否有任务缺少依赖或验收方式

---

## 9. AI IDE 生成结果的验收清单

创建新 Spec 后，应至少满足以下检查项：

### 9.1 结构检查

- 存在 `requirements.md`
- 存在 `design.md`
- 存在 `tasks.md`
- 如有补充资料，存在 `docs/`

### 9.2 需求检查

- 每条需求有编号
- 每条需求有用户故事
- 每条需求有可验证验收标准

### 9.3 设计检查

- 设计章节有编号
- 设计覆盖全部需求
- 接口、模型、错误处理足够细
- 有测试策略

### 9.4 任务检查

- 任务已分组
- 每个任务有状态
- 每个任务有依赖
- 每个任务有关联需求
- 每个任务有关联设计章节
- 每个任务有上下文入口
- 每个任务有验收标准
- 每个任务有验证方式

### 9.5 Codex 可执行性检查

随机打开一个任务，应该能回答：

1. 为什么做这个任务
2. 这个任务满足哪个需求
3. 这个任务依据哪个设计章节
4. 需要修改哪些文件或模块
5. 执行前还依赖什么
6. 做完后如何验证

如果以上任一问题无法回答，则说明任务信息不完整。

## 10. 建议 AI IDE 使用的创建指令模板

后续可以直接把下面这段话交给 AI IDE：

```text

创建一个新的 Spec。

要求：
1. 在 `specs/<新Spec目录>/` 下生成 `requirements.md`、`design.md`、`tasks.md`
2. `requirements.md` 必须包含需求编号、用户故事、WHEN/THEN/SHALL 验收标准
3. `design.md` 必须包含稳定章节编号、架构、接口、模型、错误处理、测试策略
4. `tasks.md` 必须按阶段分组，每个任务必须包含：
   - 状态
   - 依赖
   - 需求关联
   - 设计关联
   - 上下文入口
   - 涉及产物
   - 执行说明
   - 验收标准
   - 验证方式
5. 任意打开一个任务，都要能让 Codex 快速理解所需需求、技术设计、接口规范和验证方式
6. 生成后请自检需求、设计、任务三者之间的追踪关系是否完整
```

## 12. 最终原则

AI IDE 创建 Spec 时，必须遵守以下原则：

1. **不是只生成文档，而是生成可执行规范**
2. **不是只写需求，而是建立需求 → 设计 → 任务 → 验证的链路**
3. **不是只写任务标题，而是让单个任务可被独立执行**
4. **不是只考虑首次创建，而是考虑后续评审、变更、交接、验收、归档**

如果一个 Spec 不能支持新的 Codex 上下文快速接手任意任务，那么这个 Spec 就还不够好。
