# 20260323-POC验证与风险结论

## POC 已验证结论

截至 2026-03-23，下面这些链路已经通过本地 POC 验证：

1. 扫码登录可以成功建立微信登录态
2. 微信消息可以通过 `getUpdates` 拉到本地服务
3. 基于 `context_token` 的文本回复可以成功发出
4. 本地网页 demo 可以显示二维码、完成绑定，并手动发文本进行联调

这说明一个关键事实：微信 transport 是可用的，当前阻碍正式接入的不是“能不能跑”，而是“怎么插件化、怎么持久化、怎么不污染宿主”。

## 已确认的技术判断

### 1. 真正值钱的是 transport，不是 OpenClaw 路由层

当前最有复用价值的是：

- 扫码登录
- `getUpdates`
- `sendMessage`
- 媒体上传下载
- `context_token`

当前明确不该继续带入宿主的是：

- `src/channel.ts`
- `src/runtime.ts`
- `src/messaging/process-message.ts`
- `src/backends/openclaw/*`

这些东西本质上是 OpenClaw runtime 的耦合层，不是微信 transport 本身。

### 2. `context_token` 是当前最大技术风险

已确认上游实现里，`context_token` 默认可以只存在进程内缓存。

这会直接导致下面几种问题：

- 进程重启后无法继续原路回复
- 轮询 worker 切换后上下文丢失
- 延迟回复或失败重试时找不到可用 token

所以正式实现里，`context_token` 必须变成插件私有持久化状态，而不是继续靠内存 `Map` 硬撑。

### 3. Python 宿主和 Node transport 的桥接是实现复杂度，不是架构借口

宿主插件入口当前是 Python，这不等于要把 transport 重写成 Python。

正确做法是：

- 插件入口仍然走 Python
- 微信 transport 继续保留在插件内部的 Node 子层
- Python 和 Node 之间用插件私有桥接协议通信

错误做法是：

- 把 Node transport 直接灌进宿主核心
- 为了省事重写半套 transport
- 把 bridge 逻辑扩散到宿主公共模块

## 已确认的法律和合规风险

### 1. 许可证风险

已确认 `@tencent-weixin/openclaw-weixin` 在 npm 上公开发布，许可证为 MIT。

这个结论说明：

- 代码复用本身的开源许可证风险相对低
- 但这不等于平台使用风险也低

### 2. 平台条款风险

当前仍然存在中风险，原因很简单：

- 微信 transport 的可用，不等于平台官方允许该接入方式
- 自动化登录、消息拉取、账号使用边界，可能受平台条款限制
- 如果后续涉及公开分发、商业化或多账号托管，风险会放大

所以发布前必须单独确认：

- 平台使用条款
- 自动化登录边界
- 商用或分发边界

## 正式开发前必须坚持的边界

1. 微信逻辑只能在 `apps/api-server/plugins-dev` 里实现
2. 宿主核心代码库里禁止出现微信专属逻辑分支
3. 宿主如果要改，只能改通用插件能力
4. `context_token` 必须持久化
5. 运行态数据必须落在插件私有运行目录，而不是塞回宿主微信专用表

## 结论

这件事已经不是“理论可行”，而是“工程上可做”。

真正需要盯死的，不是再做一轮 POC，而是下面三件事：

1. 插件边界不能破
2. `context_token` 不能再是内存状态
3. 发布前的法律边界不能装作不存在
