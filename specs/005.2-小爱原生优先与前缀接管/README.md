# Spec 005.2 - 小爱原生优先与前缀接管

状态：Draft

## 这份 Spec 解决什么问题

`Spec 005` 已经把小爱音箱接成了 FamilyClaw 的正式语音终端，主链也已经能跑起来。

但现在有一个新的、而且是已经被 `mi-gpt` 路线验证过的产品诉求：

- 普通对话继续走小爱音箱自己的原生逻辑
- 只有命中特定前缀时，才由 FamilyClaw 接管

一句话说清楚，这次要补的不是“能不能接小爱”，而是：

> 怎么把当前“默认进 FamilyClaw 主链”的模式，改成“默认原生、命中前缀才接管”的模式，而且不把边界搞脏。

## 这次变更的核心判断

### 【核心判断】

✅ 值得做：这不是花哨需求，而是明确的交互模型切换。`mi-gpt` 这套“默认原生、命中再接管”已经被真实用户路径验证过，值得作为 `open-xiaoai` 正式接入的可选模式。

### 【关键洞察】

- 数据结构：这次新增的不是新业务对象，而是一道“是否接管这次语音”的入口判定
- 复杂度：接管闸门必须前移到 gateway，不能等文本进了 `api-server` 再说“这句不该我处理”
- 风险点：如果把快路径、慢路径、权限判断一起塞进 gateway，网关会瞬间变垃圾；所以 gateway 只能多做一件事，就是“前缀命中后才把文本交给 FamilyClaw”

### 【Linus式方案】

1. 默认仍由小爱原生处理普通对话
2. gateway 只新增“是否接管”判定，不新增业务决策
3. 命中前缀后，仍然复用现有 `voice_pipeline / router / fast_action / conversation`
4. 未命中前缀时，FamilyClaw 主链保持静默，不假装处理
5. 先做最小可用版本，不一口气扩成配置平台

## 和 `Spec 005` 的关系

- `Spec 005` 解决“正式语音终端接入和主链打通”
- `Spec 005.2` 解决“接入后是否默认接管，以及接管闸门放在哪”

这两者不是互斥关系。

更准确地说：

- `005` 定了正式链路
- `005.2` 改的是**进入这条正式链路的条件**

## 本次范围

这次只做这些事：

- 定义 native-first 模式的交互规则
- 定义 gateway 侧前缀接管闸门
- 定义命中前缀后的 takeover 时机和最小事件流
- 定义未命中前缀时 FamilyClaw 应该保持什么程度的静默
- 定义 `api-server` 在接管后继续复用现有主链的边界
- 拆出实现任务和测试任务

这次明确不做：

- 不引入 `mi-gpt` 代码
- 不在 gateway 做快路径、慢路径或权限判断
- 不做前端配置页
- 不做家庭级动态前缀配置后台
- 不把当前 `open-xiaoai` 正式主链推翻重写
