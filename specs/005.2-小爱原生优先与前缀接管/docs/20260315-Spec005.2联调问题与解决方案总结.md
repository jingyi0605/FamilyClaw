# Spec005.2 联调问题与解决方案总结

日期：2026-03-15

## 1. 这份文档记录什么

这份文档记录 `Spec 005.2 - 小爱原生优先与前缀接管` 在实际联调过程中遇到的主要问题、根因、修复方案和最终结果。

重点不是复述设计稿，而是把真正踩过的坑写清楚，避免后面的人再走一遍弯路。

## 2. 总体结论

这次联调里，最核心的结论有四个：

1. 接管闸门必须放在 `gateway`，不能放到 `api-server` 再决定。
2. 小爱音响上的 TTS 必须走单通道、前台阻塞播放，不能回退到后台并发播放。
3. “停止/闭嘴/不要再说了”这类打断，不能等最终识别文本回来才处理，必须在用户一开口时就抢断。
4. 原生优先相关配置不能只靠 `.env`，最终要落到数据库配置并支持热刷新。

## 3. 问题清单

### 问题 1：`websockets` 升级后，连接一建立就报 `ServerConnection has no attribute client`

现象：

- 网关启动后，小爱音响一连上来就报错。
- 报错位置在 `bridge.py`，代码直接读取 `websocket.client.host`。

根因：

- 新版 `websockets` 的服务端连接对象不再保证有 `client` 属性。
- 继续写死 `websocket.client.host`，就是拿旧 API 赌运气。

解决方案：

- 改为优先读取 `remote_address`。
- 如果 `remote_address` 不可用，再兼容旧的 `client.host`。
- 抽成 `_extract_remote_addr(...)`，统一处理不同版本连接对象。

结果：

- 连接建立恢复正常。
- 不再因为三方库版本差异在建连阶段崩掉。

### 问题 2：终端发现阶段 `get_version` 超时

现象：

- 网关建立 WebSocket 后，`_discover_terminal()` 调 `get_version` 超时。
- 日志反复出现 `terminal rpc timeout command=get_version`。

根因：

- 终端 RPC 应答和终端事件读取需要并发处理。
- 如果读循环和处理循环职责没分清，RPC 响应容易被卡住或者没及时 resolve。

解决方案：

- 明确拆成终端消息转发任务和终端消息处理任务。
- 字符串帧里如果是 `Response`，优先交给 `TerminalRpcClient.resolve(...)`。
- 只把真正的事件帧放进后续处理队列。

结果：

- 发现流程恢复。
- 终端版本、型号、序列号都能正常拿到。

### 问题 3：设备已选房间，但绑定接口仍返回 404

现象：

- 后端日志里，`/binding` 持续返回 `404`。
- 页面上明明已经选过房间，但绑定一直失败。

根因：

- 绑定状态依赖发现记录和绑定条件同步刷新。
- 网关侧和服务端侧有静态判断过严、刷新不及时的问题，导致前端看起来“选好了”，网关拿到的却还是“未可绑定”状态。

解决方案：

- 增加绑定状态轮询与热刷新日志，先把状态链路看清。
- 校正绑定可用条件，避免被过期状态卡死。
- 对已激活终端支持热刷新绑定信息，而不是必须重连才生效。

结果：

- 绑定成功。
- 已选房间的设备不再反复卡在“请先选择所在房间”。

### 问题 4：`native_first` 配置改了，但网关行为不跟着变

现象：

- `.env` 里已经改成：
  - `FAMILYCLAW_OPEN_XIAOAI_GATEWAY_INVOCATION_MODE=native_first`
  - `FAMILYCLAW_OPEN_XIAOAI_GATEWAY_TAKEOVER_PREFIXES=请`
- 但设备行为还是像“默认全接管”。

根因：

- 只靠进程启动时读取环境变量，配置是死的。
- 设备已经绑定并激活后，网关如果不热刷新，运行中的终端上下文不会更新。

解决方案：

- 把“是否默认接管”和“接管前缀”从环境变量迁到数据库绑定配置。
- 在设备与集成页面的音响卡片增加设置入口。
- 网关轮询绑定信息并支持热刷新运行中的 `invocation_mode` 和 `takeover_prefixes`。

结果：

- 配置从“改环境变量+重启”升级成“页面可改+在线热刷新”。
- 同一个已激活音响可以动态切换接管策略。

### 问题 5：前缀能匹配，但小爱原生回答抢在前面

现象：

- 说“请问天气怎么样”时，FamilyClaw 能命中前缀。
- 但小爱原生还是先回答一段，然后 FamilyClaw 再回答。

根因：

- 只做了“文本命中前缀后转发到 FamilyClaw”，没真正打断小爱当前原生会话。
- 这不是接管，只是“别人先说完，我再补一句”。

解决方案：

- 参考 `mi-gpt` 和 `open-xiaoai` 的思路，接管时在网关本地先打断小爱原生链路。
- 采用 `/etc/init.d/mico_aivs_lab restart >/dev/null 2>&1` 作为原生链路中断命令。
- 为避免刚中断完立刻恢复冲撞，补一个短暂恢复窗口。

结果：

- 命中前缀后，小爱原生不再先说一轮。
- `native_first` 的“前缀接管”语义真正成立。

### 问题 6：FamilyClaw 已经生成答案，但音响不发声

现象：

- 后端已经发出 `play.start`。
- 网关也收到 `play.start`，但小爱音响没有任何播报。

根因：

- 早期尝试过 `ubus call mibrain text_to_speech` 等路径，不稳定。
- 后来切到 `tts_play.sh` 后，又踩到系统环境差异，比如 `nohup: not found`。

解决方案：

- 放弃依赖不稳定的原生 TTS 调用，统一走 `/usr/sbin/tts_play.sh`。
- 不再依赖 `nohup` 这类设备环境不一定有的命令。
- 补终端 TTS 日志探针，能直接看到设备上实际报错。

结果：

- 音响播报恢复。
- 设备环境问题能通过日志直接定位，不再靠猜。

### 问题 7：整段回答生成完才播，等待时间太长

现象：

- 用户问一句话后，要等整段内容都生成完，音响才开始播第一句。

根因：

- `api-server` 原先是整段会话完成后，才统一下发一次 `play.start`。
- 这导致首句延迟完全取决于整段生成时长。

解决方案：

- 在 `conversation_bridge.py` 改成句子级流式播报。
- 复用实时会话流，把 `agent.chunk` 按句末符号累积成句。
- 每凑够一句，立刻下发一次 `voice_playback_service.start_text_playback(...)`。
- `pipeline.py` 增加 `streaming_playback` 标记，避免末尾再补发整段。

结果：

- 音响能边生成边播。
- 首句等待明显缩短。

### 问题 8：句子级流式已经有了，但句间空档仍然很大

现象：

- 看起来已经是一句一句播。
- 但每句之间还有明显停顿，体验很割裂。

根因：

- 网关最初是“边读 API WebSocket，边同步播当前句子”。
- 当前句没播完，后续 `play.start` 根本读不到。
- 这等于把“队列”做出来了，但没真正提前蓄水。

解决方案：

- 重构网关播放模型，拆成：
  - API 读循环：持续收 `play.start`
  - playback worker：单独串行消费本地队列
- 后续句子在前一句播放时就能提前入队。

结果：

- 句间空档明显下降。
- 用户实测反馈已经“可以接受”。

### 问题 9：停止词最初会直接落到小爱原生逻辑

现象：

- 播放期间说“停止”，日志显示还是 `native_passthrough`。
- 小爱自己去处理了“停止”，FamilyClaw 这边没停播。

根因：

- 最初停止词只是普通文本，没有单独高优先级本地分支。
- `native_first` 下未命中前缀，文本自然被让回给小爱原生。

解决方案：

- 在网关终端消息处理层增加本地停止词强匹配：
  - `停止`
  - `停下`
  - `停一下`
  - `不要再说了`
  - `不要说了`
  - `别说了`
  - `别讲了`
  - `闭嘴`
- 仅当当前确实存在“正在播”或“待播队列”时，才本地截获。
- 截获后：
  - 清空待播队列
  - 回传 `playback.interrupted`
  - 不再把这句文本继续交给 FamilyClaw 正常链路

结果：

- 停止词不再默认漏给小爱原生。

### 问题 10：停止了当前句，但后面排队的句子还会继续播

现象：

- 用户说“停止”后，当前一句停了。
- 但过一会儿队列里的下一句又开始播。

根因：

- 只停止了当前播放，没有同步处理 playback worker 和待播队列。
- worker 仍然会继续消费后续 `play.start`。

解决方案：

- 停止路径补全为完整中断：
  - 取消当前 playback worker
  - 清空待播队列
  - 标记中断过的会话 `session_id`
  - 丢弃同一会话后续到达的 `play.start`

结果：

- “停止”不再只停当前一句，而是终止整次播报。

### 问题 11：停止词虽然本地拦住了，但小爱原生仍会处理同一句“停止”

现象：

- 日志里已经出现 `intercepted local playback stop phrase`。
- 但小爱仍然会自己响应“停止”。

根因：

- 最终识别到“停止”时再打断，时机太晚。
- 对小爱原生来说，用户这句语音已经进入它自己的识别链路。

解决方案：

- 把抢断时机前移到用户一开口：
  - 收到 `kws`
  - 或收到 `instruction` 且 `is_vad_begin=true`
- 如果此时 FamilyClaw 正在播或队列未清空，立即同时做两件事：
  - 打断小爱原生链路
  - 中断 FamilyClaw 本地播放
- 最终文本如果又回来了，而且是“停止”，再本地吞掉，避免二次外溢。

结果：

- 停止指令不再落到小爱原生处理逻辑。

### 问题 12：一度回退到后台并发 TTS，导致双播和停不干净

现象：

- 日志里重新出现：
  - `dispatch async terminal notify`
  - `tts_play.sh ... &`
- 设备上出现“我们自己的 TTS”和“小爱原生”同时说话的双播。

根因：

- 播放路径曾回退到了后台 `tts_play.sh &` 模式。
- 这会让多句播报在设备上并发存在，停播只能碰运气。

解决方案：

- 删除后台 TTS 分支。
- `translator.py` 恢复只生成前台阻塞 `tts_play.sh` 命令，不再带 `&`。
- `bridge.py` 删除后台完成等待逻辑，强制单通道播放。

结果：

- 网关重新回到“一次只播一句”的可控模型。
- 双播问题消失。

### 问题 13：设备停播命令返回非零，导致“停止”不够硬

现象：

- 日志中出现：
  - `local playback stop command returned non-zero`

根因：

- 设备环境不完全统一。
- 单一 `mphelper pause` 不足以保证停干净 `tts_play.sh` 及底层播放器进程。

解决方案：

- 把停播命令做成多级兜底组合：
  - `mphelper pause`
  - `mphelper stop`
  - `killall`
  - `pkill -f`
  - `pidof + kill -9`
- 覆盖 `tts_play.sh`、`aplay`、`tinyplay`、`madplay` 等可能进程。

结果：

- 停播成功率提升。
- 即便某条命令不存在，也不会影响整条停止链路继续执行。

## 4. 最终落地的设计取舍

这次真正证明有效的方案，不是“最花哨”的方案，而是下面这套最笨但最稳的组合：

1. `gateway` 负责接管闸门，不让未命中前缀的普通对话进入 FamilyClaw。
2. 设备绑定配置落数据库，由后端下发，网关热刷新。
3. 后端按句子流式推送 `play.start`，缩短首句等待。
4. 网关内部读写分离，持续收句子，单独 worker 串行播。
5. 设备端 TTS 一律前台阻塞，不走后台并发。
6. 用户一开口就 barge-in，中断本地播放和小爱原生链路。
7. 最终“停止”文本只作为补充兜底，不再作为唯一打断入口。

## 6. 验证结果

联调完成后，最终确认以下行为成立：

1. `native_first` 模式下，普通请求继续由小爱原生处理。
2. 命中前缀时，FamilyClaw 能成功接管，不再被小爱抢先回答。
3. 设备端支持句子级流式播报，首句等待缩短。
4. 句间停顿已经压缩到可接受范围。
5. 播放期间说“停止”，不会再落到小爱原生处理逻辑。
6. 播放期间说“停止”，会中断当前播放并终止后续排队播报。

## 7. 后续维护建议

后面如果继续改这条链路，优先守住下面几条，不要再回退：

1. 不要再把 `tts_play.sh` 改回后台 `&` 模式。
2. 不要再把停止逻辑只放在最终识别文本上。
3. 不要让 `api-server` 承担第一道“是否接管”的责任。
4. 任何修改都要保留 `native_first` 和“默认全接管”两种模式的自动化测试。
5. 设备侧日志一定要保留，不然这类问题只能靠猜。
