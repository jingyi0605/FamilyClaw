# 20260315-H5平替评估与3.4-3.5检查清单

## 这份文档现在解决什么问题

这份文档只做一件事：把 `3.4` 和 `3.5` 的进入条件、阻塞项、非阻塞项和验收证据说清楚。

这一轮不做真实灰度切流，不改发布入口，不改网关，不接真实流量。  
先把“能不能切、差什么、要拿什么证据说服自己”写明白。

## 当前判断

### 【核心判断】

✅ 值得继续推进 `3.4`，而且功能对齐侧已经进入“可以执行真实灰度切流的候选态”；但 ❌ 现在还不具备直接宣布完成灰度切流和整体平替的条件。

原因很直接：

1. H5 的高频主链和剩余三块真实设置页都已经落地，说明新主线不是摆设
2. 但真实切流、回滚演练、发布入口接流这些证据还没做，不能把“功能补齐”和“已完成切流”混成一坨
3. H5 入口包体积 warning 仍是明确性能风险，灰度前必须带着证据复核

## H5 入口包体积 warning 算不算 `3.4` 阻塞

现在不把它算成 `3.4` 的单独阻塞项，理由只有一个：  
它目前是 warning，不是构建失败，也不是已证实的功能不可用。

但这东西绝对不能轻描淡写：

1. 如果灰度前没有把 warning 的具体体积、影响入口和增长趋势记账，后面出首屏慢、弱网卡死，就会变成真实事故
2. `3.5` 阶段检查必须把它当成明确风险项复核，而不是一句“构建过了”就糊弄过去

结论：

- `3.4` 准备阶段：不单独阻塞
- `3.4` 真实切流前：必须有 warning 记录、影响说明和接受结论
- `3.5` 退出判断时：如果 warning 已经演变成不可接受的首屏性能问题，就要升级为阻塞

## `3.4` 进入前要准备什么

这一步不改真实入口，只准备切流所需的纸面和检查材料。

### 必须先准备好的清单

1. 功能对齐清单必须把 `settings/ai`、`settings/integrations`、`settings/channel-access` 明确记成已落地入口，不能再沿用旧的 `BLOCKED` 结论。
2. parity registry 必须和对齐清单一致，把这三项同步改成 `ready`。
3. 必须写清楚真实灰度切流前的前提：
   - 哪些入口已经可以交给 H5
   - 哪些入口还必须保留 `user-web`
   - 回滚时要退到哪里
4. 必须写清楚 `3.5` 看什么证据，不然后面又会变成拍脑袋验收。

## `3.5` 阶段检查要看哪些证据和判定标准

### 证据 1：功能对齐证据

判定标准：

- 核心平替入口全部有明确归宿
- 所有保留在 `user-web` 的入口都写清楚为什么还留着
- parity registry 里不能存在“文档说 READY，registry 实际漏记”的情况

要看的证据：

- `docs/20260315-user-web功能对齐清单初稿.md`
- `packages/user-testing/src/parity/user-web-pages.ts`

### 证据 2：基础质量证据

判定标准：

- `user-app` 类型检查通过
- H5 构建通过
- H5 构建 warning 已记录，不允许“我知道有 warning，但没记在哪里”

要看的证据：

- `npm.cmd --prefix ./apps/user-app run typecheck`
- `npm.cmd run build:user-app:h5`
- 构建日志里的 warning 摘要

### 证据 3：主链回归证据

判定标准：

- 登录、初始化向导、首页、家庭、助手、记忆、设置、插件这些主链全部有可回归结果
- 家庭切换后的刷新一致性仍然成立
- 助手实时链路断开回退、待确认动作和同步反馈仍然成立

要看的证据：

- 手工回归记录
- 对应页面或链路的验收备注

### 证据 4：未迁项处置证据

判定标准：

- `settings/ai`
- `settings/integrations`
- `settings/channel-access`

这三项现在已经完成迁移，所以这里要检查的不是“有没有页面”，而是：

- 对齐清单和 parity registry 是否已经同步改口
- 新入口是否都挂进设置导航和页面注册
- 家庭切换后的刷新一致性有没有被回归打断

要看的证据：

- 路由与入口对照
- 灰度方案说明
- 回滚说明

### 证据 5：切流与回滚证据

判定标准：

- 已明确切到哪里
- 已明确怎么回
- 已明确谁来判断回滚

要看的证据：

- 灰度入口方案
- 回滚路径说明
- 发布前 smoke checklist

## 对 `3.4` 是否可进入的判断

### 可以进入什么

可以进入：

- 对齐状态修正
- 检查清单补齐
- 灰度/回滚前置文档准备
- 构建与 warning 记录

### 现在还不能进入什么

现在还不能进入：

- 真实灰度切流
- 发布入口切换
- 网关或部署配置接流
- 宣布 `user-app` H5 已可完整平替 `user-web`

## 一句话结论

`3.4` 现在已经从“准备态”推进到“可以执行真实切流的候选态”，但还不能进“已完成执行态”。  
功能缺口已经补齐，剩下卡住的是切流、回滚和发布入口证据；H5 包体积 warning 目前不是功能阻塞，但仍是必须带着证据复核的性能风险。
