# 20260315-user-web功能对齐清单初稿

## 这份清单现在解决什么问题

这不是验收成品，而是迁移启动后的第一版对齐底稿。

它先回答三件事：

1. `user-web` 现在到底有哪些主入口
2. 哪些入口已经在 `user-app` 里有壳了
3. 哪些东西现在不能硬搬，必须先继续抽共享层

## 冻结规则

从 `spec011` 开始，`apps/user-web` 进入“只修不增”阶段。

允许做的事：

- 修线上阻断级 bug
- 为新旧入口切换补临时兼容桥接
- 为共享层抽离做必要的小改动

不允许做的事：

- 承接正式新功能
- 继续堆浏览器特有能力当长期方案
- 为了赶进度把整页直接复制到 `apps/user-app`

## 3.4 前置判断

这轮补完 `assistant`、`memories`、`plugins`、`setup` 以及 `settings/ai`、`settings/integrations`、`settings/channel-access` 之后，功能缺口已经清掉，H5 已进入“可以执行真实切流的候选态”；但还不够宣布“已经完成整体平替 `user-web`”。

### 现在可以算 READY 的东西

- 登录、初始化向导、首页、家庭、助手、记忆、设置基础页、插件管理
- `settings/ai`
- `settings/integrations`
- `settings/channel-access`
- 这些链路依赖的共享 API、共享状态、主题、语言和实时适配层

### 现在还不能装作“已经完成 3.4 / 3.5”的东西

- 真实灰度切流
- 回滚演练
- 发布入口 / 网关 / 部署配置接流
- H5 入口包体积 warning 的风险接受结论

现在缺的已经不是这三块页面功能，而是真实切流和回滚证据。  
另外通知独立页还没补，但旧页本身主要是占位说明，不构成这轮 H5 平替的硬阻塞。

详细判断和 `3.5` 证据标准见：`docs/20260315-H5平替评估与3.4-3.5检查清单.md`

## 这轮已经抽出的共享层

- `packages/user-core`
  - 启动链路、首页、家庭、设置会用到的核心类型和领域模型
  - 共享 API client：认证、家庭、房间、成员、设备、上下文、提醒、locale
  - 助手会话共享模型和 API：agent 列表、会话、消息、动作、建议、会话发送
  - 插件共享模型和 API：注册表、挂载、启停状态和最近任务
  - AI 设置共享模型和 API：provider 管理、能力路由、Agent 创建 / 详情 / 人格 / 运行策略 / 成员认知
  - 集成页共享模型和 API：Home Assistant 配置、设备 / 房间候选同步、语音终端发现 / 认领
  - 通讯接入共享模型和 API：平台账号、状态探测、失败记录、成员绑定
  - 启动摘要加载服务
  - 认证清理、家庭摘要、setup、主题偏好、语言偏好 helper
- `packages/user-platform`
  - 浏览器与 Taro 存储适配
  - 平台目标探测
  - 实时协议、URL 构造和浏览器 websocket client
- `packages/user-ui`
  - 页面分区组件
  - 状态卡片组件
  - 基础视觉 token
- `packages/user-testing`
  - 对齐清单 registry
  - 当前页面迁移状态初稿

## 暂时不要硬搬的东西

下面这些不是不能做，而是现在硬搬会把旧问题一起带过来：

- 助手页里复杂的上下文侧栏、历史抽屉和富操作布局
- 记忆页里的复杂列表和纠错交互
- 设置页里的通知独立子页和更细的展示打磨
- 家庭页的大表单和复杂关系编辑 UI

先继续抽共享 API、状态和平台接口，再迁页面壳。顺序不能反。
