# 20260314-Taro4多端迁移阶段与切换原则

## 这份补充文档解决什么问题？

主文档已经把路线讲清楚了，但迁移这种事最容易死在两个地方：

1. 旧系统还在继续长新功能
2. 大家嘴上都说“以后删掉 `user-web`”，结果一年后它还在那里

所以这里单独把迁移波次、切流条件和下线原则写死。

## 迁移总原则

### 1. 只有一条长期主线

长期主线只有一条：`apps/user-app`

`apps/user-web` 只是过渡产物，不是备用长期方案。

### 2. 先抽共享层，再迁页面

如果直接把旧页面复制进 Taro，新应用只会继承旧问题，平台差异还会更多。

正确顺序是：

1. 抽类型、API、状态和主题
2. 建平台适配层
3. 迁页面壳
4. 做平台差异

### 3. 先让 H5 站稳，再谈移动端全面发布

H5 是旧系统的直接接班人。  
如果 H5 都不能平替 `user-web`，移动端上线就是自欺欺人。

### 4. 不允许双主线长期并存

迁移期间允许并行，迁移完成后必须收口。  
任何“先都留着以后再看”的说法，最后都会变成技术债。

## 迁移波次

### 波次 1：建新主线

目标：

- 建 `apps/user-app`
- 建 `packages/user-core`
- 建 `packages/user-platform`
- 建 `packages/user-ui`

这一步做完后，仓库里就要出现新的正式主线骨架。

### 波次 2：抽共享逻辑

目标：

- 把 `user-web` 的类型、API、认证、家庭上下文、主题、语言、实时逻辑抽出来
- 建立功能对齐清单

这一步做完后，后面迁页面才不会全靠复制。

### 波次 3：H5 平替

目标：

- 让 `user-app` H5 覆盖核心用户链路
- 逐步接真实流量
- 把 `user-web` 降到保底角色

这是最关键的一波。  
H5 不稳，后面的移动端就是空谈。

### 波次 4：移动端能力补齐

目标：

- 补权限
- 补通知
- 补文件、拍照、上传
- 补分享、深链、实时连接
- 补移动端导航、安全区、键盘行为

这一步不是“做个手机样式”，而是把平台能力真接起来。

### 波次 5：正式切流和旧系统退场

目标：

- 完成灰度
- 完成回滚演练
- 完成真机验收
- 删除 `user-web`

如果这一步不做，前面四波都白干。

## user-web 的并行边界

### 可以做的事

- 修线上阻断 bug
- 补迁移必需的兼容修复
- 配合新旧入口切换做临时桥接

### 不可以做的事

- 承接正式新功能
- 重做页面大交互
- 引入新的跨端方案
- 继续堆浏览器特有能力

一句话：  
`user-web` 可以保命，不能继续生长。

## H5 切流条件

H5 可以开始灰度，但不能一次性全切。  
开始灰度前至少满足下面条件：

1. 登录、初始化向导、首页、家庭、助手、记忆、设置这些核心链路已经可用
2. API、状态、主题、语言、实时逻辑已经走共享层
3. 对齐清单里没有阻断级缺口
4. 入口切换和回滚方式已经明确

## user-web 下线条件

删除 `user-web` 前，至少同时满足下面条件：

1. `user-app` H5 已稳定承接核心用户流量
2. `iOS / Android / 鸿蒙` 目标已经有可用构建和基础验收结果
3. 权限、通知、文件、分享、深链、实时这些关键平台能力都已落到统一适配层
4. 灰度切流和回滚演练都跑通过
5. 功能对齐清单里没有阻断级未迁项
6. 项目文档、发布脚本、入口说明都已经指向新主线

少一条都别删。

## 哪些页面优先迁

优先级从高到低建议这样排：

1. 登录与初始化向导
2. 首页
3. 家庭
4. 设置
5. 助手
6. 记忆
7. 插件管理、通讯接入等低频但重要页面

理由很简单：

- 先迁高频入口，能最快暴露结构问题
- 先迁基础配置，后面移动端能力才有落点
- 聊天和记忆必须进核心波次，但可以排在主壳稳定之后

## 哪些东西必须先抽适配层

下面这些能力不先抽，后面一定烂：

- 本地存储
- 权限
- 通知
- 文件选择
- 拍照
- 分享
- 深链
- 实时连接

别再让页面去碰 `window`、`document`、`localStorage`、浏览器通知或者平台原生对象。

## 最后一句话

这次迁移真正要防的，不是“工作量大”，而是“边迁边乱改，最后两套都半死不活”。

所以规矩很简单：

- 新功能进 `user-app`
- 旧前端只保底
- H5 先接班
- 平台能力统一抽象
- 达标后删掉 `user-web`
