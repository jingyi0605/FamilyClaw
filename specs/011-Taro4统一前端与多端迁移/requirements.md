# 需求文档 - Taro 4 统一前端与多端迁移

状态：Draft

## 简介

`FamilyClaw` 现在已经有可运行的 `user-web`，但它解决的是“网页端能不能工作”，不是“项目未来有没有一条稳定的多端主线”。

这次不是简单重写 UI，也不是另起一套移动端壳，而是要建立一条长期能活下去的前端架构：

- 一套正式用户前端代码主线
- 同时覆盖 `PC 浏览器 H5`、`iOS`、`Android`、`鸿蒙`
- 平台差异收口到适配层，而不是散在页面里
- 用清晰的迁移节奏逐步废掉 `user-web`

如果这次不把主线定死，后面每加一个能力，都会变成网页写一遍、移动端补一遍、平台权限再修一遍，最后没人愿意碰。

## 术语表

- **System**：当前 `FamilyClaw` 系统
- **Legacy Web**：现有 `apps/user-web`
- **User App**：新的 `apps/user-app`，基于 `Taro 4 + React`
- **Taro H5**：`user-app` 的 PC 浏览器目标
- **Taro RN**：`user-app` 的 iOS / Android 目标
- **Taro Harmony**：`user-app` 的鸿蒙目标
- **共享层**：跨端复用的类型、API、状态模型、业务逻辑和基础组件
- **平台适配层**：封装权限、通知、文件、分享、深链、存储、实时连接等平台能力的统一接口
- **功能对齐清单**：把 `user-web` 现有能力逐项映射到 `user-app` 的迁移清单

## 范围说明

### In Scope

- 新建 `apps/user-app`，并明确它是唯一长期用户前端主应用
- 规划并落地跨端共享包
- 覆盖 `PC H5`、`iOS`、`Android`、`鸿蒙`
- 定义统一平台适配层
- 规划从 `user-web` 到 `user-app` 的迁移路线、灰度节奏和退出条件
- 把用户端核心页面迁入新应用
- 建立多端构建、环境配置、测试和发布方案

### Out of Scope

- Windows / macOS 原生桌面客户端
- 保留双主线长期并存
- 第一阶段就追求所有边角页面一次性全部迁完
- 继续把新功能优先写进 `user-web`

## 需求

### 需求 1：系统必须只有一条正式用户前端主线

**用户故事：** 作为项目负责人，我希望用户前端未来只有一条正式技术路线，这样团队不会长期维护两套同类应用。

#### 验收标准

1. WHEN 团队规划后续用户端能力 THEN System SHALL 以 `apps/user-app` 作为唯一长期主应用，而不是继续把 `apps/user-web` 视为长期主线。
2. WHEN 新前端主线建立完成 THEN System SHALL 明确 `PC 浏览器 H5`、`iOS`、`Android`、`鸿蒙` 都由 `Taro 4 + React` 覆盖。
3. WHEN 迁移进入实施阶段 THEN System SHALL 把 `user-web` 标记为过渡应用，只允许兼容修复和迁移辅助改动。

### 需求 2：新前端必须同时覆盖 PC 和移动多端

**用户故事：** 作为产品用户，我希望同一套用户前端能力可以在 PC 和手机上使用，而不是每个平台一套割裂体验。

#### 验收标准

1. WHEN 新前端交付 THEN System SHALL 通过 `Taro H5` 覆盖 PC 浏览器场景。
2. WHEN 新前端交付 THEN System SHALL 通过 `Taro RN` 覆盖 iOS 和 Android 场景。
3. WHEN 新前端交付 THEN System SHALL 通过 `Taro Harmony` 覆盖鸿蒙场景。
4. WHEN 页面能力属于跨端共性能力 THEN System SHALL 优先复用共享层，而不是为每个平台重复实现一套页面逻辑。

### 需求 3：平台差异必须收口到统一适配层

**用户故事：** 作为开发者，我希望权限、通知、文件、分享、深链、存储这些平台差异都收口到统一接口里，这样页面代码不会充满平台特判。

#### 验收标准

1. WHEN 页面需要访问权限、通知、文件、分享、深链、存储或实时连接 THEN System SHALL 通过平台适配层调用，而不是直接在页面里写浏览器 API 或原生平台 API。
2. WHEN 某个平台不支持某项能力 THEN System SHALL 由平台适配层返回明确的降级结果或禁用信息，而不是让页面自己猜。
3. WHEN 后续新增平台能力 THEN System SHALL 优先扩展适配层接口，而不是在业务页面里新增散乱分支。

### 需求 4：新前端必须先覆盖 user-web 的核心用户链路

**用户故事：** 作为家庭用户，我希望新前端先把最常用的链路迁过去，而不是先重做样式、最后发现关键功能没法用。

#### 验收标准

1. WHEN `user-app` 的 H5 版本进入可用阶段 THEN System SHALL 至少覆盖登录、初始化向导、首页、家庭、助手、记忆、设置这几条核心链路。
2. WHEN 某项 `user-web` 现有能力准备迁移 THEN System SHALL 先在功能对齐清单里标明现状、差异和迁移状态。
3. WHEN 某项核心用户链路尚未在 `user-app` 达到可用标准 THEN System SHALL 不得宣称 `user-web` 可以下线。

### 需求 5：通知与权限体系必须可以按平台灵活配置

**用户故事：** 作为家庭管理员，我希望不同平台上的通知、权限和提醒方式可以灵活配置，这样产品既能统一体验，也不会被某个平台限制拖死。

#### 验收标准

1. WHEN 用户在任一平台管理通知偏好 THEN System SHALL 提供统一的通知配置模型，并允许平台侧做能力映射。
2. WHEN 平台需要申请通知、相册、相机、位置、文件等权限 THEN System SHALL 提供统一的请求、查询和解释流程。
3. WHEN 某个平台的通知或权限能力受限 THEN System SHALL 给出明确的禁用、降级或替代说明，而不是展示假开关。
4. WHEN 通知或权限策略发生调整 THEN System SHALL 尽量通过配置和适配层变更完成，而不是重写页面业务逻辑。

### 需求 6：迁移必须可灰度、可回滚、可分阶段切流

**用户故事：** 作为项目负责人，我希望迁移过程中可以分阶段切流和回滚，这样不会因为新前端一次性替换失败把用户全部带崩。

#### 验收标准

1. WHEN `user-app` 开始承接真实流量 THEN System SHALL 支持按平台、按功能或按用户范围灰度切流。
2. WHEN `user-app` 某个阶段出现阻断问题 THEN System SHALL 允许回退到 `user-web` 或上一阶段稳定版本。
3. WHEN 准备删除 `user-web` THEN System SHALL 先满足明确的退出条件，而不是凭感觉下线。

### 需求 7：迁移期间必须控制 user-web 的范围膨胀

**用户故事：** 作为维护者，我希望迁移期间旧前端不会继续长新功能，否则迁移永远追不上。

#### 验收标准

1. WHEN `user-app` 立项后 THEN System SHALL 把 `user-web` 纳入“只修不增”的迁移策略。
2. WHEN 出现新的用户端需求 THEN System SHALL 默认落到 `user-app`，除非该需求是迁移完成前必须修复的线上缺陷。
3. WHEN 确实需要在 `user-web` 补改动 THEN System SHALL 记录原因，并同步更新功能对齐清单，防止迁移漏项。

### 需求 8：新前端必须具备可持续维护的工程结构

**用户故事：** 作为开发团队，我希望未来新增页面、接入新平台或调整平台能力时，改动边界清楚，不会再把前端做成一坨。

#### 验收标准

1. WHEN 新增页面或新平台 THEN System SHALL 通过清晰的目录边界、共享包和平台文件约定扩展，而不是在单个巨型文件里继续堆代码。
2. WHEN 团队复用类型、API、状态逻辑或基础组件 THEN System SHALL 优先放入共享层，而不是复制到多个端目录。
3. WHEN 排查线上问题 THEN System SHALL 能从页面、共享层、平台适配层和构建目标四个层次快速定位问题。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 用户首次进入 `user-app` 的 H5 或移动端首页 THEN System SHALL 优先渲染首屏关键内容，不得把所有次级数据阻塞在首屏之前。
2. WHEN 页面在移动端运行 THEN System SHALL 控制首屏包体和分包策略，避免单个入口过重导致冷启动明显变慢。

### 非功能需求 2：可靠性

1. WHEN 某个平台构建失败或某项平台能力异常 THEN System SHALL 尽量限制影响范围，不得拖垮其他平台构建和核心用户链路。
2. WHEN 发生权限拒绝、通知注册失败、实时连接失败或深链解析失败 THEN System SHALL 有明确的降级和恢复路径。

### 非功能需求 3：可维护性

1. WHEN 新增共享能力 THEN System SHALL 保持共享层与平台适配层边界稳定，不得把平台细节重新泄漏到页面层。
2. WHEN 团队继续迁移 `user-web` 页面 THEN System SHALL 先复用已有共享层，再写平台壳和展示层，不允许页面整坨复制。

### 非功能需求 4：迁移安全性

1. WHEN 迁移阶段尚未达到退出条件 THEN System SHALL 保持 `user-web` 可用，并确保用户入口切换是可控的。
2. WHEN `user-web` 准备删除 THEN System SHALL 完成功能对齐、灰度验证、回滚验证和文档回写。

## 成功定义

- `apps/user-app` 成为唯一长期用户前端主应用，技术路线不再摇摆。
- `PC H5`、`iOS`、`Android`、`鸿蒙` 都有明确目标和落地路径。
- 核心用户链路从 `user-web` 迁入 `user-app`，并能逐步接管真实流量。
- 权限、通知、文件、分享、深链、存储、实时连接都通过统一平台适配层收口。
- `user-web` 在满足退出条件后被正式移除，而不是无限期挂着。
