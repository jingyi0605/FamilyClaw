# Spec 011 - Taro 4 统一前端与多端迁移
状态：Draft

## 这份 Spec 解决什么问题？

现在仓库里的 `apps/user-web` 本质上是一套标准 `React + Vite` 网页应用。它已经能承载不少真实业务，但它天生是浏览器程序，不是多端程序。继续在这套代码上硬套 iOS、Android、鸿蒙，最后只会得到一堆平台特判、权限分支和通知补丁，代码会越补越烂。

这份 Spec 的目标很直接：

- 明确项目未来只保留一套正式用户前端技术路线：`Taro 4 + React`
- 让这套前端同时覆盖 `PC 浏览器 H5`、`iOS`、`Android`、`鸿蒙`
- 把权限、通知、文件、分享、深链、实时连接这些平台能力做成统一适配层
- 在迁移期间允许 `user-web` 短期共存，但它只能作为过渡产物，最终会被废掉

这不是“再起一个新前端试试看”，而是前端主线切换。

## 这次拍板的结论

- 正式路线只保留 `Taro 4 + React`
- `PC 端` 走 `Taro H5`
- `iOS / Android` 走 `Taro React Native`
- `鸿蒙` 走 `Taro Harmony`
- 不再规划 `Capacitor`、`Flutter`、`uni-app x`、`React Native 直写` 这些平行路线
- `user-web` 进入“只修不增、逐步下线”的过渡状态

## 为什么现在做？

因为再拖下去，前端会同时出现三种坏味道：

1. 新需求继续堆到 `user-web`，后面越迁越难。
2. 移动端能力迟迟没有统一抽象，权限、通知、文件、推送会各做各的。
3. 团队会默认接受“两套前端长期并存”，这不是方案，这是维护灾难。

## 这次明确要做到什么？

- 建立新的 `apps/user-app` 作为唯一长期用户前端主应用
- 规划共享包，把业务类型、API、状态模型、平台服务从页面里拆出来
- 先让 `Taro H5` 覆盖 `user-web` 的核心用户链路，再推进移动端原生能力
- 建立从 `user-web` 到 `user-app` 的迁移节奏、灰度策略和下线条件
- 补齐平台能力边界：权限、通知、深链、分享、文件、上传、存储、实时链路

## 这次明确不做什么？

- 不保留双主线前端架构，不允许未来长期同时维护 `user-web` 和 `user-app`
- 不做 Windows / macOS 原生桌面客户端
- 不在第一阶段追求所有页面一次性迁完
- 不为了“看起来零改动复用”继续绑死浏览器 API
- 不在没有迁移完成前直接删除 `user-web`

## 和现有 Spec 的关系

- 承接 `002.2-家庭用户前端产品化`
  - 那份 Spec 定义了 `user-web` 的产品形态，这份 Spec 定义它的继任者和退场路线
- 承接 `004.4-用户前端插件管理与插件市场`
  - 插件管理、插件市场等用户前端能力后续都要迁入 `user-app`
- 承接 `010-通讯通道插件与多平台机器人接入`
  - 通讯接入、消息能力、通知偏好后续都要在 Taro 前端里继续消费

## 这份 Spec 的核心判断

- 这是个真问题，不是臆想需求。
- 最简单的长期方案不是“继续修 `user-web`”，而是尽快确立单一主线：`Taro 4 + React`
- 最大风险不是“迁移工作量大”，而是迁移期间边做边改、让 `user-web` 继续失控膨胀

## 主要文档

- `requirements.md`
  - 说明统一前端到底要交付哪些能力，什么叫迁移完成
- `design.md`
  - 说明新目录结构、共享层、平台适配层、迁移路线和发布策略
- `tasks.md`
  - 说明按什么顺序迁、每一步改什么、验证什么、什么阶段可以切流
- `docs/20260314-Taro4多端迁移阶段与切换原则.md`
  - 用人话单独说明迁移波次、并行边界和 `user-web` 下线条件

## 当前阶段策略

- 先建新应用，不在 `user-web` 里继续试图“改造成多端”
- 先抽共享层，再迁页面，不允许把旧页面整坨复制到新应用里
- 先让 `H5` 成为新主线，再让移动端吃共享层
- 迁移期间新功能默认进 `user-app`，`user-web` 只允许修缺陷和补迁移所需兼容
- 达到退出条件后，明确删除 `user-web`
