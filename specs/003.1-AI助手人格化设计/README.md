# Spec 003.1 - AI多Agent与管家角色设计

状态：Draft

## 这份 Spec 到底要解决什么

现在再把“管家”当成唯一角色，已经不够用了。

更准确的产品结构应该是：

- `AI配置` 是一层系统配置入口
- 下面承载多个 `Agent`
- `管家` 只是其中一个默认角色
- 其他角色可以是 `营养师`、`健身教练`、`学习教练` 等

每个 Agent 都应该有自己独立的：

- 身份认知
- 性格
- 长期记忆视角
- 成员认知
- 外观

同时还要守住几个硬边界：

- 这些 Agent 都不是家庭成员
- 这些 Agent 都不出现在家庭成员列表里
- 这些 Agent 都不出现在家庭图谱里
- 长期记忆仍然必须复用 `Spec 003`，不能给每个 Agent 再造一套记忆系统

前端产品层也要同步校准：

- 原来的 `AI设置` 不能改成“管家设置”，而是要保留成 **AI配置**
- 原来的“助手页面”也不该叫“管家”，应该直接改成 **对话**

## 规划结论

### 【核心判断】

✅ 值得做：这不是加几个角色名，而是把单助手思路升级成多 Agent 架构。  
如果现在把“管家”做死成唯一角色，后面再扩营养师、健身教练时，代码和产品结构都会变成补丁堆。

### 【关键洞察】

- 数据结构：真正要建的是 `Agent 基座`，不是单个“管家对象”。
- 复杂度：每个 Agent 都有独立人格和运行时上下文，但长期记忆底层必须共用一个家庭记忆中心。
- 风险点：最大的坑还是两个——把 Agent 混进家庭成员体系，或者为每个 Agent 复制一套记忆存储。

### 【Linus式方案】

1. 先把 `AI助手` 抽象成 **家庭 AI 多Agent体系**
2. `管家` 是默认主 Agent，不是唯一 Agent
3. `AI配置` 保留为统一配置入口，里面管理多个 Agent
4. 原“助手页面”直接改成 `对话`，对话页负责和 Agent 交互
5. 长期记忆继续复用 `Spec 003`，通过 `agent_id` 和作用域做隔离，不搞双份真相

## 本 Spec 包含什么

- `requirements.md`：定义多 Agent、管家角色、AI配置入口和对话页的业务边界
- `design.md`：定义数据模型、运行时组装、记忆复用、外观系统和兼容方案
- `tasks.md`：定义分阶段落地顺序、主要改动文件和验证方式

## 本 Spec 明确不做什么

- 不把任何 Agent 显示为家庭成员
- 不把任何 Agent 显示在家庭图谱中
- 不做实时 3D 形象、口型驱动、动态虚拟人
- 不给每个 Agent 新建独立长期记忆数据库
- 不做无限制的自动角色繁殖和自动角色漂移

## 依赖与协同

- **依赖 `Spec 001`**：复用家庭成员、关系、权限基础能力，但不改默认语义
- **依赖 `Spec 002.2`**：用户端需要保留 `AI配置`，并把原“助手页面”改成“对话”
- **强依赖 `Spec 003`**：长期记忆必须复用家庭记忆中心
- **协同 `ai_gateway`**：不同 Agent 的模型调用与外观生成统一走 AI 网关
