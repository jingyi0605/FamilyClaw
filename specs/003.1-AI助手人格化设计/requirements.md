# 需求文档 - AI多Agent与管家角色设计

状态：Draft

## 简介

当前产品里只有一个模糊的 `AI助手` 概念，这太扁平了。

真正合理的产品结构不是“一个助手”，而是：

- 一个统一的 `AI配置`
- 多个可管理的 `Agent`
- 一个面向用户的 `对话` 入口

其中：

- `管家` 是默认主角色，负责家庭陪伴和综合服务
- `营养师` 可以负责饮食建议和家庭饮食记忆
- `健身教练` 可以负责运动计划和训练反馈
- 后续还可以扩展其他角色

每个 Agent 都应该有独立的：

- `soul`（身份认知）
- 性格与表达风格
- 运行时记忆视角
- 家庭成员认知
- 外观档案

但这些 Agent 都不属于家庭成员，不应该显示在成员列表或图谱中。

这个 Spec 要解决的根问题有五个：

1. 把单一 `AI助手` 升级成多 Agent 体系
2. 明确 `管家` 只是其中一个默认角色
3. 保留 `AI配置` 作为统一管理入口
4. 把原“助手页面”改成“对话”
5. 让每个 Agent 复用家庭记忆中心，而不是新建独立记忆库

## 术语表

- **System**：家庭版 OpenClaw / FamilyClaw
- **Agent**：家庭 AI 体系中的一个独立角色实例，拥有独立人格、成员认知、记忆视角和外观
- **主管家（Butler Agent）**：默认主 Agent，负责家庭级陪伴、解释和综合服务
- **AI配置（AI Config）**：管理多个 Agent 的统一设置入口
- **对话（Conversation）**：面向用户的统一交互页面，用于与一个或多个 Agent 对话
- **Soul**：某个 Agent 对“我是谁、我的职责是什么、我的边界是什么”的稳定自我认知
- **成员认知（Member Cognition）**：某个 Agent 对家庭成员的称呼、沟通方式、服务重点和注意事项
- **外观档案（Appearance Profile）**：某个 Agent 的视觉设定、生成提示词、候选素材和已发布形象
- **记忆中心复用**：所有 Agent 的长期记忆都复用 `Spec 003` 的事件、记忆卡、热摘要、权限与修订能力

## 范围说明

### In Scope

- 建立多 Agent 基础模型和配置体系
- 将 `管家` 定义为默认主 Agent，而不是唯一角色
- 为每个 Agent 建立独立的 `soul`、成员认知和外观档案
- 让每个 Agent 的长期记忆复用现有家庭记忆中心
- 保留 `AI配置` 作为统一设置入口
- 将原“助手页面”改为“对话”页面

### Out of Scope

- 将任意 Agent 作为家庭成员展示
- 将任意 Agent 放入家庭图谱节点或关系边
- 实时虚拟人、3D 形象、口型驱动
- 为不同 Agent 建立不同的长期记忆数据库
- 自动无限扩张 Agent 数量和自动生成新角色

## 需求

### 需求 1：系统支持多 Agent 机制

**用户故事：** 作为家庭管理员，我希望系统支持多个 Agent，而不是只能有一个助手，这样不同角色才能承担不同职责。

#### 验收标准

1. WHEN 管理员进入 AI配置 THEN System SHALL 返回该家庭下的多个 Agent 列表及其角色类型、状态和摘要信息。
2. WHEN 管理员新增或启用一个 Agent THEN System SHALL 支持将其配置为某个明确角色，如管家、营养师、健身教练。
3. WHEN 系统运行时加载 Agent THEN System SHALL 允许每个 Agent 拥有独立人格、成员认知、记忆视角和外观配置。

### 需求 2：管家是默认主 Agent，不是唯一 Agent

**用户故事：** 作为产品设计者，我希望“管家”是默认主角色，但不是唯一角色，这样家庭综合服务和专业角色能同时成立。

#### 验收标准

1. WHEN 一个家庭初始化 AI 角色 THEN System SHALL 默认创建或保留一个主管家 Agent。
2. WHEN 用户没有手动切换其他 Agent THEN System SHALL 默认由主管家承接综合型对话和服务。
3. WHEN 家庭启用了营养师、健身教练等其他 Agent THEN System SHALL 保持这些 Agent 与管家并存，而不是互相覆盖。

### 需求 3：每个 Agent 都有独立的 Soul、性格和成员认知

**用户故事：** 作为家庭管理员，我希望不同 Agent 有各自稳定的身份和性格，而不是共用一套模板换个名字。

#### 验收标准

1. WHEN 管理员编辑某个 Agent 的人格 THEN System SHALL 支持独立配置名称、角色定位、自我介绍、说话风格、服务重点和服务边界。
2. WHEN 管理员为某个 Agent 配置成员认知 THEN System SHALL 支持独立设置称呼、沟通方式、服务优先级和注意事项。
3. WHEN 不同 Agent 响应同一家庭成员 THEN System SHALL 能体现各自不同的人格、职责和表达方式。

### 需求 4：所有 Agent 的长期记忆都必须复用家庭记忆中心

**用户故事：** 作为系统维护者，我希望所有 Agent 都直接复用现有家庭记忆中心，这样系统只有一套真实长期记忆来源。

#### 验收标准

1. WHEN 任意 Agent 需要读取长期记忆 THEN System SHALL 复用 `Spec 003` 已有的 `event_records`、`memory_cards`、`hot_summary`、权限过滤和修订链路。
2. WHEN 任意 Agent 沉淀新的长期记忆线索 THEN System SHALL 写入现有记忆中心的统一入口，并标记 `agent_id` 或角色作用域。
3. WHEN 记忆中心中的相关记忆被纠错、失效或删除 THEN System SHALL 让相关 Agent 后续的记忆检索和上下文拼装立即受到影响。

### 需求 5：每个 Agent 具备可配置、可生成、可发布的外观系统

**用户故事：** 作为家庭管理员，我希望每个 Agent 都可以拥有具象外观，并能通过多模态模型生成候选形象。

#### 验收标准

1. WHEN 管理员配置某个 Agent 的外观 THEN System SHALL 支持设定视觉风格、人物描述、色彩偏好、禁用元素和参考图来源。
2. WHEN 管理员发起外观生成 THEN System SHALL 通过统一 AI 网关调用多模态模型生成该 Agent 的候选形象，并保留生成记录、参数和结果状态。
3. WHEN 候选形象被人工选中并发布 THEN System SHALL 让对话页、AI配置页和相关卡片统一使用该外观；若生成失败，则保留默认占位形象，不影响核心能力。

### 需求 6：AI配置保留为统一设置入口

**用户故事：** 作为产品使用者，我希望系统继续保留“AI配置”这个入口，因为这里管理的是整个 AI 多 Agent 体系，而不是只有管家。

#### 验收标准

1. WHEN 用户进入设置中的 AI 相关入口 THEN System SHALL 继续以“AI配置”作为页面名称和导航名称。
2. WHEN 用户查看 AI配置 THEN System SHALL 能看到多个 Agent 的列表、摘要状态以及进入单个 Agent 配置的入口。
3. WHEN 后续新增新的 Agent 角色 THEN System SHALL 能在 AI配置中扩展，而不用改掉整个产品信息架构。

### 需求 7：原“助手页面”改为“对话”

**用户故事：** 作为产品使用者，我希望面向聊天和交互的入口直接叫“对话”，而不是叫“助手”，这样它表达的是交互场景，不是底层角色实现。

#### 验收标准

1. WHEN 用户进入原“助手页面” THEN System SHALL 将该入口、页面标题和核心文案改为“对话”。
2. WHEN 用户进入对话页 THEN System SHALL 支持与默认管家或其他已启用 Agent 进行对话。
3. WHEN 当前家庭只启用了主管家 THEN System SHALL 继续能用，不因为多 Agent 设计而强制要求用户理解复杂角色切换。

### 需求 8：多 Agent 能接入对话和家庭服务主链路

**用户故事：** 作为家庭成员，我希望多 Agent 不是停留在配置页，而是真的影响对话和服务结果。

#### 验收标准

1. WHEN 任意 Agent 进入问答或服务流程 THEN System SHALL 组合该 Agent 的 `soul`、成员认知和记忆中心结果生成运行时上下文。
2. WHEN 当前请求更适合某个专业 Agent 处理 THEN System SHALL 支持显式切换或后续扩展自动路由，而不破坏主管家默认链路。
3. WHEN 外观系统或某个 Agent 配置服务异常 THEN System SHALL 允许核心对话与家庭服务降级运行，而不是整个 AI 能力不可用。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 任意 Agent 组装运行时上下文 THEN System SHALL 优先复用热摘要和结构化成员认知数据，使额外上下文拼装时间控制在 500 毫秒级别。
2. WHEN 发起外观生成 THEN System SHALL 采用异步任务或状态轮询，不阻塞对话页和主链路。

### 非功能需求 2：可靠性与安全

1. WHEN 外观生成结果不合规、失败或超时 THEN System SHALL 保留默认形象并记录失败原因，不破坏对应 Agent 主能力。
2. WHEN 管理员追查人格或记忆问题 THEN System SHALL 能从当前表现追到具体 `agent_id`、`soul` 版本、成员认知配置和记忆来源。

### 非功能需求 3：可维护性

1. WHEN 后续继续扩 Agent 角色 THEN System SHALL 允许在 `soul`、成员认知、外观和记忆复用四块独立扩展，而不是继续堆成一个大配置对象。
2. WHEN 团队排查兼容问题 THEN System SHALL 保持现有 `members`、`member_relationships`、`memory` 模块的默认语义不变。

## 成功定义

- 系统已经具备正式的多 Agent 基座，而不是单一助手模式
- `管家` 已明确成为默认主 Agent，但不是唯一 Agent
- `AI配置` 保持为统一设置入口
- 原“助手页面”已改成“对话”
- 所有 Agent 都不出现在家庭成员和家庭图谱里
- 所有 Agent 的长期记忆都复用现有记忆中心，而不是第二套记忆系统
