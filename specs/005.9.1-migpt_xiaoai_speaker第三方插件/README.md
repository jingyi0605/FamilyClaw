# Spec 005.9.1 - migpt_xiaoai_speaker 第三方插件

状态：Draft

## 这份 Spec 要解决什么问题

现有内置 `open-xiaoai` 方案更偏向刷机后的网关模式，覆盖范围有限。为了兼容更新的小爱音箱，需要一条不依赖刷机、能覆盖更多机型的接入路径。

参考 `mi-gpt` 源码后，可以把它的核心接入方式概括成一句话：

> 它不是拿到原始麦克风流去做真音频实时对话，而是通过小米账号登录后调用 MiNA / MIoT 能力，轮询会话记录拿文本请求，再用 TTS 指令或音频播放把回复播回音箱。

也就是说，`mi-gpt` 本质上是一种“文本轮询型实时对话 + 统一 speaker 控制”的接入方案。

这正好对应 `005.9` 要补的宿主契约。`005.9.1` 的目标不是把 `mi-gpt` 逻辑塞回核心，而是把它完整装进一个第三方插件里。

## 核心判断

✅ 值得做。

原因很简单：

1. 它能覆盖更多不刷机的新小爱音箱。
2. 它走的是公开云端接口和设备控制接口，问题是稳定性和风控，不是宿主架构不可能支持。
3. 只要宿主先把 `005.9` 的通用契约立住，这类插件完全可以独立存在，不需要污染核心。

## 对 `mi-gpt` 的关键分析结论

### 结论 1：输入侧是文本轮询，不是原始音频流

`mi-gpt` 会周期性拉取 MiNA 会话记录，比较时间戳，拿到最新用户 query，再把这条 query 当作一次新消息处理。

这意味着：

- 它拿到的是“用户说了什么”的文本结果。
- 它没有正式的原始音频采集入口。
- 所以首版插件必须接入 `005.9` 里的 `text_turn` 车道，而不是 `audio_session` 车道。

### 结论 2：输出侧是 MIoT 指令或 MiNA 播放

`mi-gpt` 的回复播报主要有两种方式：

- 直接调用 MIoT TTS 指令，让小爱音箱说出文本。
- 先用第三方 TTS 生成音频，再通过 MiNA 播放音频 URL。

这意味着插件应该向宿主请求“标准回复结果”，再自行决定怎么播回去，而不是让宿主理解小米播放细节。

### 结论 3：设备控制依赖机型指令 profile

`mi-gpt` 里不同机型会有不同的：

- `ttsCommand`
- `wakeUpCommand`
- `playingCommand`

这说明“机型指令差异”必须做成插件内部 profile，不允许写死在宿主。

### 结论 4：连续对话是模拟出来的，不是完美实时

`mi-gpt` 为了维持连续对话，会：

- 轮询会话记录
- 轮询播放状态
- 在一些情况下用静音音频或空白 TTS 让设备尽快闭嘴

这套东西能用，但不是完美实时，也会受网络延迟、云端状态同步和风控影响。

所以这个插件必须明确标注为“文本轮询型实时对话”，不能冒充“原始音频实时对话”。

## 本 Spec 的边界

### In Scope

- 定义 `migpt_xiaoai_speaker` 第三方插件的实例配置、设备发现、文本轮询对话、统一控制和 runtime 边界。
- 明确插件如何只通过 `005.9` 的正式宿主能力接入。
- 明确所有小米专有逻辑都留在插件内。

### Out of Scope

- 不修改宿主核心去理解小米 cookie、MiNA 协议、MIoT 指令表。
- 不承诺首版支持原始音频流接入。
- 不在宿主内新增小米专用页面、专用 API 或专用状态模型。

## 成功标准

- `migpt_xiaoai_speaker` 能作为第三方插件独立接入。
- 它只调用宿主正式的 `speaker/voice` 能力，不读宿主私有数据库、不走私有旁路。
- 所有小米账号、设备 profile、轮询逻辑、风控恢复都留在插件内。
