# Spec 009 - 计划任务系统

状态：Draft

## 这份 Spec 解决什么问题

现在项目里已经有几块相关能力，但它们还是散的：

- 插件可以被手动触发，也已经有后台任务模型，但还没有正式的“到点自动跑”入口
- Agent 可以发起插件任务和对话能力，但还没有“系统自己决定什么时候提醒一下”的正式机制
- 家庭状态已经能汇总成员在家情况、房间占用、设备离线、老人关怀、儿童保护这些信息，但还没有周期巡检和提醒链路
- 用户端已经有账号、成员、家庭三层对象，但计划任务还没有正式区分“这是全家的任务”还是“这是某个成员自己的任务”

结果就是：只要一碰到“每天 8 点提醒吃药”“晚上回家后检查儿童保护没开”“每天同步一次健康插件”“Agent 主动提醒家庭异常”“给我自己建一个睡前提醒”这类需求，大家就只能各写各的临时逻辑。这种写法很快会烂。

这份 Spec 的目标很直接：补一套正式的计划任务系统，让系统能按时间或心跳规则主动创建任务、触发插件、调用 Agent、生成提醒，而且能和现有插件后台任务、家庭状态、实时通知链路接上。同时把前端管理入口、对话创建入口、账号与成员归属边界一起收好。

## 这次要做到什么

- 建一套统一的计划任务定义、触发记录、执行记录和心跳巡检机制
- 支持两类触发：固定时间调度、按家庭状态周期检查后触发
- 让计划任务可以正式对接现有插件后台任务体系，而不是再造一套执行器
- 让计划任务可以驱动 Agent 主动提醒，且保留审计、冷却和人工兜底
- 把家庭状态接入规则判断，支持“满足条件就提醒”这类主动能力
- 允许插件声明“可被计划任务触发”，但真正启用仍由家庭级配置决定
- 把计划任务分成家庭公共任务和成员私有任务，让每个家庭成员都能创建属于自己的任务
- 给用户前端补正式管理入口，不只让后台接口可用
- 支持通过对话把一句人话请求收口成计划任务草稿，再由用户确认创建

## 这次明确不做什么

- 不做通用工作流平台，不上 DAG，不做拖拽编排
- 不做跨服务分布式调度集群，第一版按现有单服务模型设计
- 不让 Agent 自己无限决定要不要跑任务，第一版只允许命中的规则主动触发
- 不做短信、电话这类高成本通知渠道
- 不做复杂的前端可视化编排器，先保证配置、执行、提醒三条主链清楚
- 不做“用户说一句模糊的话就直接落正式任务”，对话创建第一版必须经过结构化确认

## 为什么现在做

因为这个需求不是“以后可能用到”，而是项目往下走一定会撞上的基础能力。

如果现在不收口，后面会出现几种典型垃圾结果：

- 插件各自偷偷带 cron 逻辑，排查时根本不知道是谁在什么时候触发了什么
- 家庭状态提醒逻辑散落在页面、场景、对话、插件里，规则互相打架
- Agent 想主动做事时没有正式边界，要么完全不能做，要么直接失控
- 账号、成员、家庭三层归属不清，最后谁都不知道这条任务到底是谁建的、又归谁管

## 主要文档

- `requirements.md`：说清楚计划任务系统到底要覆盖什么场景，什么算完成
- `design.md`：说清楚调度器、心跳扫描、执行分发、插件接入、家庭状态规则、前端入口、对话创建怎么落地
- `tasks.md`：说清楚分几批实现，先搭底座，后接插件，再接 Agent、家庭状态、对话入口和前端管理
- `docs/README.md`：给后续补接口样例、规则样例、联调说明留位置
