# 需求文档 - 计划任务系统

状态：Draft

## 简介

项目已经有 Agent、插件后台任务、家庭状态汇总、实时通知这些基础件，但还缺一个“主动触发”的中枢。现在系统更像只能等人来点按钮，缺少按时间、按状态、按规则主动发起任务或提醒的正式入口。

这份需求文档要把这件事收口成一套可落地能力：

- 系统能按时间表主动创建任务
- 系统能按心跳周期检查家庭状态并命中规则
- 计划任务能接到现有插件任务体系、Agent 提醒体系和家庭状态体系
- 用户可以在前端正式管理任务，也可以通过对话更自然地创建任务
- 计划任务既支持家庭公共任务，也支持成员自己的私有任务
- 所有主动行为都有边界、有记录、能限流、能排查

主要服务的角色有五类：

- 家庭成员：希望收到稳定、不过度打扰的主动提醒，也希望能创建自己的任务
- 家庭管理员：希望配置家庭公共任务，也能代成员创建或调整任务
- Agent：希望通过受控规则主动提醒或发起低风险任务，而不是永远被动等用户输入
- 前端页面：希望有正式管理入口，不要只靠后台接口和数据库排查
- 后端维护者：希望主动链路统一，不要再出现每个模块各写一套定时逻辑

## 术语表

- **System**：FamilyClaw API Server
- **计划任务定义**：一条长期存在的规则，说明什么时候触发、触发后做什么、属于哪个家庭
- **任务所有者**：这条计划任务归谁管理，第一版分成家庭公共任务和成员私有任务两类
- **账号创建者**：真正发起创建或修改操作的登录账号
- **归属成员**：计划任务实际服务的家庭成员；一个账号可以为自己创建，也可能由管理员代某个成员创建
- **计划任务运行**：某条计划任务定义在某次命中后产生的一次执行记录
- **调度触发**：按固定时间、固定间隔或 cron 表达式命中任务
- **心跳巡检**：系统周期检查家庭状态、插件状态或其他上下文后，决定是否触发任务
- **执行适配器**：把计划任务运行转成具体动作的模块，比如创建插件后台任务、生成 Agent 提醒
- **冷却时间**：同一条规则在一段时间内不重复提醒、不重复创建等价任务的保护窗口
- **任务草稿**：对话入口先解析出来、但还没正式落库的结构化计划任务候选

## 范围说明

### In Scope

- 为家庭级主动能力建立统一的计划任务定义、运行记录和触发记录模型
- 支持家庭公共任务和成员私有任务两种归属模型
- 支持固定时间调度和心跳巡检两类触发方式
- 支持把计划任务运行分发到插件后台任务、Agent 提醒、系统通知这三类执行目标
- 接入家庭状态概览，支持基于成员在家、设备离线、安静时段、关怀开关等状态规则触发
- 支持插件声明可被计划任务触发，并由家庭级配置启用具体任务
- 提供用户前端的计划任务查看、创建、启停和运行记录入口
- 支持通过对话创建计划任务草稿，并在确认后正式落库
- 提供必要的查询、启停、审计、限流、幂等和失败收口能力

### Out of Scope

- 不做跨集群分布式调度和多活抢占
- 不做图形化工作流编排器
- 不做高风险自动动作默认放开执行
- 不做自由脚本执行和任意代码注入式规则
- 不做复杂机器学习决策或 Agent 自主意图生成

## 需求

### 需求 1：系统必须支持正式的计划任务定义和运行记录

**用户故事：** 作为后端维护者，我希望系统有统一的计划任务模型，以便所有主动触发能力都走正式入口，而不是散落在各个模块里。

#### 验收标准

1. WHEN 家庭或家庭成员为某个主动能力创建计划任务 THEN System SHALL 保存任务名称、触发方式、执行目标、参数、启用状态和冷却配置。
2. WHEN 计划任务被命中 THEN System SHALL 生成独立的运行记录，并保存触发时间、触发原因、执行状态和关联结果。
3. WHEN 运维或开发排查主动行为来源 THEN System SHALL 能根据家庭、任务定义、时间范围查询到计划任务和运行记录。

### 需求 1A：计划任务必须区分家庭公共任务和成员私有任务

**用户故事：** 作为家庭成员，我希望能创建属于我自己的计划任务，而不是所有任务都只能挂在整个家庭名下，以便区分公共提醒和我自己的私人安排。

#### 验收标准

1. WHEN 成员创建自己的计划任务 THEN System SHALL 保存 `owner_scope=member`、归属成员和创建账号信息。
2. WHEN 家庭管理员创建公共计划任务 THEN System SHALL 保存 `owner_scope=household`，并让该任务默认面向家庭公共范围生效。
3. WHEN 查询计划任务列表 THEN System SHALL 能区分家庭公共任务、当前成员私有任务，以及当前账号无权查看的其他成员私有任务。
4. WHEN 非管理员账号尝试修改其他成员的私有任务 THEN System SHALL 拒绝该操作。

### 需求 1B：计划任务必须同时记录账号操作人和归属成员

**用户故事：** 作为系统维护者，我希望知道到底是哪个账号创建或修改了任务，以及这个任务实际属于哪个家庭成员，以便权限校验和审计都说得清。

#### 验收标准

1. WHEN 任意计划任务被创建或修改 THEN System SHALL 记录创建账号或最后修改账号。
2. WHEN 计划任务归属于某个成员 THEN System SHALL 同时记录归属成员 id，而不是只留账号 id。
3. WHEN 一个账号与成员存在绑定关系 THEN System SHALL 允许系统根据当前登录态自动推导默认归属成员，但允许管理员显式改为其他成员或家庭公共任务。
4. WHEN 账号和成员绑定缺失且又要创建成员私有任务 THEN System SHALL 拒绝直接落库，并要求先明确归属成员。

### 需求 2：系统必须支持固定时间调度和心跳巡检两类触发方式

**用户故事：** 作为家庭管理员，我希望既能配置每天固定时间执行的任务，也能配置“每隔一段时间检查状态，符合条件再提醒”的任务，以便覆盖同步和提醒两类场景。

#### 验收标准

1. WHEN 计划任务定义为固定时间调度 THEN System SHALL 按配置的时区、cron 或固定间隔计算下一次触发时间。
2. WHEN 计划任务定义为心跳巡检 THEN System SHALL 按巡检频率执行规则检查，并只在条件命中时创建运行记录。
3. WHEN 同一时间窗口内已经存在等价运行记录 THEN System SHALL 基于幂等键或冷却窗口避免重复触发。
4. WHEN 服务重启后恢复调度器 THEN System SHALL 能继续从持久化状态恢复下一次触发计算，而不是丢失计划任务。

### 需求 2A：用户必须能从前端正式管理自己的计划任务

**用户故事：** 作为家庭成员，我希望在用户前端看到我自己的计划任务和家庭公共任务，并能直接创建、启停和查看运行结果，以便不需要只靠后台接口或数据库操作。

#### 验收标准

1. WHEN 成员进入计划任务管理页 THEN System SHALL 默认展示“我的任务”和“家庭任务”两个视角。
2. WHEN 成员创建或编辑自己的计划任务 THEN System SHALL 提供最小可用表单，能配置触发方式、执行目标、提醒对象和冷却策略。
3. WHEN 成员查看运行记录 THEN System SHALL 展示最近执行时间、最近结果、失败原因和下一次执行时间。
4. WHEN 当前账号没有权限管理某条任务 THEN System SHALL 在前端隐藏编辑入口或明确提示无权限。

### 需求 2B：用户必须能通过对话创建计划任务草稿

**用户故事：** 作为家庭成员，我希望直接用自然语言说“每天晚上九点提醒我吃药”这类话，让系统帮我生成计划任务草稿并确认，以便减少复杂表单操作。

#### 验收标准

1. WHEN 用户在对话里表达明确的计划任务意图 THEN System SHALL 把自然语言解析成结构化草稿，而不是直接落正式任务。
2. WHEN 草稿关键信息缺失 THEN System SHALL 在对话中继续追问缺失字段，比如时间、归属对象、提醒范围或执行目标。
3. WHEN 用户确认草稿 THEN System SHALL 通过正式计划任务创建入口落库，并返回创建结果。
4. WHEN 草稿涉及高风险执行目标或跨成员创建 THEN System SHALL 要求更明确的确认或权限校验，不能静默创建。

### 需求 3：计划任务必须能对接现有插件后台任务体系

**用户故事：** 作为插件能力维护者，我希望计划任务触发插件时复用现有插件后台任务链路，以便统一任务状态、重试、通知和审计。

#### 验收标准

1. WHEN 计划任务的执行目标是插件 THEN System SHALL 创建正式的 `plugin_job`，而不是绕过后台任务模型直接同步执行插件。
2. WHEN 插件声明不支持计划触发或家庭未启用该配置 THEN System SHALL 拒绝创建该计划任务或拒绝执行对应运行。
3. WHEN 插件由计划任务触发 THEN System SHALL 在插件任务记录里保留来源任务定义、来源运行和触发类型。
4. WHEN 插件任务失败或等待人工响应 THEN System SHALL 把状态回传到计划任务运行记录，便于统一排查。

### 需求 4：计划任务必须能驱动 Agent 主动提醒，但要有边界

**用户故事：** 作为产品设计者，我希望 Agent 能在规则命中时主动提醒家庭成员，以便形成真正有用的主动服务，而不是无限制地自己乱发消息。

#### 验收标准

1. WHEN 计划任务的执行目标是 Agent 提醒 THEN System SHALL 指定目标家庭、目标 Agent、提醒文案模板或提醒生成参数。
2. WHEN Agent 主动提醒被触发 THEN System SHALL 记录使用的 Agent、触发来源、提醒摘要和投递结果。
3. WHEN 同一条提醒规则在冷却窗口内重复命中 THEN System SHALL 抑制重复提醒，避免刷屏。
4. WHEN 计划任务尝试触发高风险动作型 Agent 行为 THEN System SHALL 拒绝自动执行，或要求显式确认配置，不允许默认放开。

### 需求 5：计划任务必须能读取家庭状态并支持规则判断

**用户故事：** 作为家庭管理员，我希望系统能根据家庭成员在家情况、设备状态和家庭模式主动提醒，以便真正基于家庭状态工作，而不是只会按死时间表广播。

#### 验收标准

1. WHEN 心跳巡检执行家庭状态规则 THEN System SHALL 通过正式家庭状态入口读取成员状态、房间占用、设备摘要和关键 insight，而不是绕过服务层直接拼数据。
2. WHEN 家庭状态满足规则条件 THEN System SHALL 创建对应运行记录，并把命中的状态摘要保存下来。
3. WHEN 家庭状态不满足规则条件 THEN System SHALL 只记录巡检结果或更新时间，不创建多余提醒。
4. WHEN 家庭处于安静时段、访客模式或其他抑制条件 THEN System SHALL 按规则决定延迟、降级或跳过提醒。

### 需求 6：主动链路必须具备可控性和可维护性

**用户故事：** 作为系统维护者，我希望主动任务体系可观测、可审计、可停用，以便不会因为一条坏规则把全家轰炸一遍。

#### 验收标准

1. WHEN 管理员停用计划任务 THEN System SHALL 立即停止后续触发，但保留历史定义和运行记录。
2. WHEN 某条计划任务连续失败超过阈值 THEN System SHALL 标记该任务进入异常状态，并产出明确告警或待处理信息。
3. WHEN 排查一条主动触发记录 THEN System SHALL 能看到触发器、规则输入、执行结果和关联对象。
4. WHEN 后续新增新的执行目标或新的规则类型 THEN System SHALL 复用统一任务定义和分发接口，而不是重写整套调度框架。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 调度器或心跳扫描执行 THEN System SHALL 以批量方式读取到期任务，避免每条规则都单独扫全库。
2. WHEN 家庭数量增长 THEN System SHALL 支持按家庭、按到期时间窗口分批扫描，而不是把全部任务一次性拉进内存。

### 非功能需求 2：可靠性

1. WHEN 服务重启、中断或短时异常 THEN System SHALL 保留下一次执行时间、最近一次运行结果和必要的恢复信息。
2. WHEN 心跳扫描或调度分发失败 THEN System SHALL 让失败收口到明确运行状态，而不是静默吞掉。

### 非功能需求 3：可维护性

1. WHEN 新增插件型计划任务、Agent 提醒型计划任务或家庭状态规则 THEN System SHALL 只新增适配器或规则处理器，不改调度主循环。
2. WHEN 需要排查重复触发或漏触发 THEN System SHALL 能通过日志、运行记录和幂等键快速定位问题。
3. WHEN 需要扩展新的前端入口或新的自然语言创建入口 THEN System SHALL 复用统一的计划任务 service，而不是再造第二套创建逻辑。

## 成功定义

- 项目内不再需要为定时触发和周期巡检各自偷写独立循环
- 插件、Agent、家庭状态三条链路都能通过统一计划任务系统接入
- 用户前端能正式管理“我的任务”和“家庭任务”
- 用户能通过对话创建一条明确的计划任务并完成确认
- 至少一条固定时间任务和一条家庭状态心跳规则能完整跑通
- 失败、重复触发、冷却抑制和停用场景都有正式记录和排查入口
