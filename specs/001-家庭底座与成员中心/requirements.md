# Spec 001 - 家庭底座与成员中心

## 背景

当前项目已有设计方案与开发计划，但尚未进入可执行的工程阶段。为了降低返工风险，首个实现切片需要先构建所有后续能力都依赖的底座：

- 家庭主数据
- 成员主数据
- 关系与权限模型
- 房间与设备基础模型
- Home Assistant 同步入口

该切片完成后，项目将从“文档阶段”进入“工程阶段”。

---

## 目标

本 Spec 的目标是交付一个最小但正确的家庭底座，使系统能够：

1. 创建一个家庭
2. 管理家庭成员
3. 管理成员关系、偏好和权限
4. 管理房间与设备
5. 从 Home Assistant 同步设备实体
6. 为后续提醒、问答、场景编排提供稳定主数据

数据库选型约束：

- 首期实现统一使用 `SQLite`
- 不引入 `PostgreSQL` 等额外数据库依赖
- 设计时保留后续迁移到更强数据库的空间，但本 Spec 不以迁移为目标

---

## 非目标

本 Spec 不包括：

- 完整语音链路
- 声纹识别与人脸识别细节
- 家庭问答能力
- 记忆卡生成
- 复杂场景编排
- 小米设备的深入接入

---

## 用户角色

### 家庭管理员

需要：

- 创建家庭
- 录入家庭成员
- 配置成员关系
- 配置权限和偏好
- 管理房间和设备

### 系统服务

需要：

- 有稳定的家庭主键与成员主键
- 能查询成员与角色
- 能查询房间与设备映射
- 能通过外部实体 ID 找到本地设备

---

## 功能需求

## FR-1 家庭管理

系统必须支持管理员创建家庭并查看家庭信息。

### 验收标准

- 可以创建一个家庭
- 家庭包含名称、时区、语言区域
- 家庭创建后生成唯一 `household_id`

## FR-2 成员管理

系统必须支持管理员对成员进行增删改查。

### 验收标准

- 可新增成员
- 可编辑成员基础资料
- 可停用成员
- 可按家庭查询成员列表

## FR-3 角色与关系管理

系统必须支持配置成员角色与家庭关系。

### 验收标准

- 支持管理员、成人、儿童、老人、访客角色
- 支持配偶、父母子女、监护关系
- 关系数据可查询

## FR-4 偏好管理

系统必须支持配置成员偏好。

### 验收标准

- 支持称呼偏好
- 支持灯光偏好
- 支持空调偏好
- 支持提醒方式偏好

## FR-5 权限管理

系统必须支持最小可用权限模型。

### 验收标准

- 支持公共、个人、敏感、高风险资源分级
- 支持读、写、执行、管理四类动作
- 支持 allow/deny 模型

## FR-6 房间管理

系统必须支持创建与查询房间。

### 验收标准

- 支持客厅、卧室、书房、玄关等房间
- 支持房间隐私等级
- 支持按家庭查询房间列表

## FR-7 设备管理

系统必须支持设备主数据管理。

### 验收标准

- 支持设备列表查询
- 支持房间与设备绑定
- 支持设备类型和控制能力字段

## FR-8 Home Assistant 同步

系统必须支持同步 Home Assistant 设备实体到本地。

### 验收标准

- 能触发一次同步
- 能把实体写入本地 `devices` 与 `device_bindings`
- 同步过程可审计

## FR-9 审计日志

系统必须对关键管理动作记审计日志。

### 验收标准

- 家庭创建写日志
- 成员编辑写日志
- 设备同步写日志

---

## 非功能需求

## NFR-1 可维护性

- 模块边界清晰
- 代码结构支持后续拆分
- API 命名一致

## NFR-2 安全性

- 只有管理员可执行管理动作
- 敏感字段不能无权限暴露

## NFR-3 可扩展性

- 所有主业务表必须带 `household_id`
- 成员、设备、房间必须使用 `uuid`

## NFR-4 可观测性

- 关键 API 有日志
- 同步任务有成功/失败记录

---

## 成功定义

当以下条件满足时，该 Spec 视为完成：

1. 后端 API 可跑通家庭、成员、关系、房间、设备 CRUD
2. 管理台可完成家庭与成员管理基本操作
3. 可从 Home Assistant 同步一批设备实体
4. 所有关键动作写入审计日志
5. 数据模型可被下一阶段的提醒、问答与场景模块直接复用
