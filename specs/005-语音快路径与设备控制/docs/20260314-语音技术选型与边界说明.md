# 20260314-语音技术选型与边界说明

## 这份文档是干什么的

这份文档只做一件事：把 `Spec 005` 里的技术选型和边界判断写死，免得后面实现时又变成“先随便接一个 demo 试试”。

## 先说结论

这次的推荐架构不是“一切都进 `conversation`”，也不是“把业务搬进小爱音箱”。

这次的正式架构是：

1. **小爱音箱 + `open-xiaoai` Client**
   - 只负责音频输入、播放输出和中断

2. **`open-xiaoai-gateway`**
   - 独立进程，监听 `4399`
   - 只做协议翻译、播放控制中转和危险能力裁剪

3. **`voice-runtime`**
   - 负责流式 ASR、声纹注册、声纹验证
   - 第一版优先选 `FunASR + 3D-Speaker`

4. **`api-server` 内的 `voice_pipeline`**
   - 负责会话、身份融合、快慢路径分流
   - 快路径复用 `device_action / scene`
   - 慢路径复用 `conversation`

## 为什么不是别的方案

### 1. 为什么不把 `open-xiaoai` 协议直接塞进 `api-server`

因为这样做会直接把核心业务搞脏：

- 终端私有协议和业务逻辑混在一起
- 播放控制、中断回执和语音路由搅成一锅
- 后续换终端时，每次都得改核心后端

这不是简单，这是把坑挖到主链里。

### 2. 为什么不把官方 demo server 直接当正式服务

因为 demo server 的职责是“演示能跑”，不是“长期维护的生产边界”。

项目真正需要的是：

- 一个可控的协议翻译器
- 一个能明确裁剪危险能力的接入层
- 一个后续能扩展成插件体系的终端适配层

demo 不是这个东西。

### 3. 为什么不把业务放到小爱上

因为业务真正要依赖的是：

- `context`
- `conversation`
- `device_action`
- `scene`
- 审计和权限边界

这些都在 FamilyClaw 内部。把业务下沉到音箱侧，只会制造第二套主链。

### 4. 为什么初版不接 `mi-gpt`

因为现在接它不划算：

- 当前仓库已停止维护
- 官方 README 也更推荐 `open-xiaoai`
- 初版核心目标是把主链收干净，不是同时兼容两套小米终端路线

它可以留作后续插件位，但不该干扰当前设计。

## 具体选型

### A. 终端接入

推荐：

- 初版唯一终端适配器：`open-xiaoai`
- 初版唯一接入层：`open-xiaoai-gateway`

原因：

- 用户已完成实机验证
- 终端链路已经跑通
- 当前最该做的是“产品化接入”，不是“重新发明终端”

### B. 流式 ASR

推荐：

- `FunASR`

原因：

- 中文家庭场景优先
- 可本地部署
- 适合快路径和短句控制

### C. 声纹识别

推荐：

- `3D-Speaker`

原因：

- 可本地部署
- 更适合做成员候选增强
- 不需要把身份判断押给云端

### D. 设备控制

结论：

- 继续复用现有 `ha_integration`、`device_action`、`scene`

原因：

- 这部分系统已经有正式边界
- 高风险确认和场景守卫已经存在
- 再写一套语音专用控制器只会制造分叉

## 初版明确不做什么

- 不把 `open-xiaoai` demo server 当正式服务
- 不把业务逻辑搬到小爱终端
- 不开放任意脚本执行、系统升级、重启控制
- 不初版接 `mi-gpt`
- 不做完整 TTS 平台化
- 不做多人混说和远场分离
- 不做原始音频默认长期留存

## 这套判断和当前项目怎么对上

当前代码里已经有：

- `conversation` 三车道
- `realtime` WebSocket
- `device_action`
- `scene`
- `context`
- `scheduler`

所以第一版最稳的落法就是：

- 新增 `open-xiaoai-gateway`
- 新增 `voice` 模块
- 新增 `voice-runtime`
- 不改现有设备、场景、聊天主链的正式边界

这才叫不破坏现有用户空间。

## 最后一句

这次真正要守住的，不是某个具体模型名字，而是这条边界：

> 小爱只做话筒和音箱，网关只做协议翻译，业务核心继续留在 FamilyClaw。
