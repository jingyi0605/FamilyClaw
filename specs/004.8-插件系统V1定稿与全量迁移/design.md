### 1.1 目标

- 把插件系统从“已经有很多部件”收口成“边界明确的正式平台”
- 把插件文档重构成可长期维护的官方开发者手册
- 把宿主插件核心和现有内置插件全部迁到同一套 V1 规则

### 1.2 覆盖需求

- `requirements.md` 需求 1
- `requirements.md` 需求 2
- `requirements.md` 需求 3

### 1.3 技术约束

- 后端：FastAPI、SQLAlchemy、Alembic、异步 worker

- 外部依赖：现有内置插件接入的第三方 API 与设备平台

## 2. 架构

### 2.1 系统结构

这次改造后的插件平台结构固定为四层：

1. **宿主内核层**
   负责家庭、成员、权限、审计、插件运行时、设备/实体标准模型、卡片规范、记忆标准、配置保存、调度和错误治理。
2. **插件契约层**
   负责 manifest、正式类型、配置作用域、标准 DTO、槽位契约和生命周期。
3. **插件实现层**
   负责第三方 API 或设备对接、字段映射、资源发现、状态刷新、动作执行和卡片快照生产。
4. **开发者手册层**
   负责给官方和第三方开发者提供稳定规则、接口表、调用方式、样板和验证方式。

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 宿主插件核心 | 注册、启停、配置、实例、运行时、DTO 校验、槽位调度 | manifest、配置、插件输出 | 标准设备/实体/动作/卡片/记忆调用 |
| 开发者手册 | 说明边界、类型、字段、调用方式、样例、验证规则 | 最新规范与代码事实 | 官方开发文档 |
| 普通插件 | 对接外部能力并产出标准模型 | 第三方平台数据、宿主上下文 | 标准 DTO、动作、卡片快照 |
| 槽位型插件 | 记忆提取、记忆后端、上下文组装 | 宿主记忆事件与查询请求 | 受宿主管控的槽位结果 |

### 2.3 关键流程

#### 2.3.1 插件文档收口流程

1. 提炼少改动的固定规则，放入总规范和开发者手册首页。
2. 提炼高频变化的接口表、字段表、调用样例，放入事实来源章节或附录。
3. 让 Spec、开发者手册和核心代码统一引用相同命名和相同契约。

#### 2.3.2 普通插件运行流程

1. 宿主加载 manifest，校验正式插件类型和能力声明。
2. 宿主按 `plugin`、`integration_instance`、`device`、`channel_account` 作用域注入配置。
3. `integration` 插件按统一 `refresh` 契约返回实例、设备、实体、卡片或动作输出。
4. 宿主统一做校验、去重、落库、状态更新和错误收口。
5. 运行态标准实体统一写入宿主公共实体承载层，后续设备读取、控制路由、快捷指令和卡片展示都只读这层标准实体，不再读领域专用快照。

#### 2.3.3 宿主与插件私有状态边界

1. 宿主公共层只定义插件无关的标准模型，包括插件实例、标准设备、标准实体、标准卡片快照、任务状态与错误状态。
2. 任何领域专用绑定关系、provider 鉴权信息、刷新游标、缓存原始响应和领域聚合结果，都属于插件私有状态，不进入宿主核心模型注册入口，也不进入宿主核心 Alembic migration 历史。
3. 插件刷新链路必须先在插件内部完成 provider 拉取、领域映射、标准设备构造和标准实体归一化，再把结果提交给宿主。
4. 宿主只做 schema 校验、事务落库、关联更新、错误收口和事件广播，不根据插件类型推断天气、电费、健康等领域字段。
5. 卡片 payload 里的用户可见文案也属于插件标准输出的一部分；宿主只认标准可见字段，不再按插件 id、领域类型或插件词典 key 做页面专用翻译分支。

#### 2.3.4 槽位型插件运行流程

1. 宿主激活独占槽位插件。
2. 宿主以统一契约调用 `memory_engine`、`memory_provider`、`context_engine`。
3. 宿主保留权限、可见性、修订、人工修正和 fallback。
4. 插件失败时宿主回退到内置默认实现。

#### 2.3.5 插件私有持久化与迁移收口

1. 宿主只提供插件无关的公共持久化模型和运行时入口，不再把天气、电费、健康等领域私有表写进宿主核心 Alembic。
2. 插件如果需要私有表，必须通过插件私有 migration 机制或插件私有存储边界管理，而不是反向把领域表塞回宿主。
3. 当前正式口径是：`builtin` / `official` 插件可以在插件目录下自带 `migrations/env.py` 与版本脚本，由宿主在插件执行准备阶段按插件隔离执行私有 Alembic migration。
4. 插件私有 ORM 模型可以复用宿主共享 `Base`，但不得再注册进 `app.db.models` 这类宿主核心聚合入口；宿主核心 import 链也不得静态依赖这些模型。
5. 如果插件不用宿主共享 `Base`，那跨宿主公共表的外键约束、metadata 组织和 flush 顺序都必须由插件自己兜住；宿主不为此增加领域特判。
6. `official` 与 `third_party` 插件目录缺失时，宿主核心导入、宿主核心 Alembic 和应用初始化都必须继续成立。

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3

- **插件规范文档层**：开发设计规范与开发者手册，明确稳定规则与事实来源。
- **插件类型与生命周期层**：统一 `integration`、`action`、`channel`、`agent-skill`、`ai-provider`、`region-provider`、`locale-pack`、`theme-pack` 与三类记忆槽位。
- **标准数据契约层**：统一实例、设备、实体、动作、卡片和记忆接口 DTO。
- **插件迁移适配层**：将现有内置插件逐个迁到新入口、新 manifest 和新输出模型。
- **部署与加载边界层**：明确 `builtin`、`official`、`third_party` 的目录、镜像和导入边界，确保宿主缺失官方插件时仍可启动。
- **插件私有持久化边界层**：明确插件私有表、插件私有迁移和宿主核心 Alembic 的隔离规则。

### 3.3 接口契约

覆盖需求：1、2、3

#### 3.3.1 插件文档索引契约

- 类型：Documentation
- 标识：官方开发者手册目录
- 输入：正式规范、代码事实来源、接口表、示例
- 输出：稳定规则页、接口表页、接入样板页、迁移说明页
- 校验：稳定规则不得重复复制高频接口细节
- 错误：文档中出现漂移命名、过时类型或重复事实来源时必须回收

#### 3.3.2 `integration.refresh`

- 类型：Function / Worker 调用
- 路径或标识：`integration.refresh`
- 输入：实例上下文、宿主上下文、配置、上次刷新状态
- 输出：`IntegrationRefreshResult`
- 校验：设备、实体、卡片、动作必须符合 V1 标准 DTO
- 校验补充：卡片 payload 内的用户可见字段必须符合标准可见字段规则，宿主不得通过插件特判补翻译
- 错误：刷新失败时必须返回统一错误状态与降级信息

#### 3.3.3 记忆槽位调用

- 类型：Function / Slot Contract
- 路径或标识：
  - `memory_engine.*`
  - `memory_provider.*`
  - `context_engine.*`
- 输入：宿主标准记忆事件、查询请求或上下文构建请求
- 输出：受槽位契约约束的结构化结果
- 校验：插件不能越过宿主权限、可见性、修订和审计边界
- 错误：调用失败时宿主必须回退内置实现

## 4. 数据与状态模型

### 4.1 数据关系

本次改造后的正式关系如下：

- **普通插件**
  - 插件 -> 实例 -> 设备 -> 实体 -> 卡片 / 动作
- **通道插件**
  - 插件 -> 通道账号 -> 会话与消息
- **记忆槽位插件**
  - 插件 -> 独占槽位 -> 宿主统一记忆接口

宿主不再为天气、健康、家居等单一领域保留专用主链路模型。
宿主也不再导入或注册插件私有领域表，只通过标准 DTO 与插件结果交互。
运行态标准实体现在由宿主公共实体承载层统一保存；`DeviceBinding.capabilities` 只保留绑定摘要、能力标签和调试可读的轻量元数据。

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `registered` | 插件已注册 | manifest 合法并入库 | 启用或禁用 |
| `enabled` | 插件可执行 | 宿主启用插件 | 禁用、报错或卸载 |
| `refreshing` | 正在刷新 | 触发调度或手动刷新 | 成功、失败、降级 |
| `degraded` | 降级运行 | 外部依赖失败但宿主仍可保底 | 刷新恢复或停用 |
| `disabled` | 已停用 | 用户禁用或宿主停用 | 重新启用 |

## 5. 错误处理

### 5.1 错误类型

- **规范漂移错误**：文档、manifest、代码命名不一致
- **能力声明错误**：插件类型、入口、能力声明互相冲突
- **运行时契约错误**：插件返回非标准 DTO 或缺字段
- **外部依赖错误**：第三方 API、设备平台或外部存储失败
- **槽位越权错误**：记忆插件试图绕过宿主主权边界

### 5.2 错误响应格式

```json
{
  "detail": "Plugin contract validation failed",
  "error_code": "plugin_contract_invalid",
  "field": "capabilities.integration.entity_types",
  "timestamp": "2026-03-18T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：拒绝加载或拒绝执行，并给出明确字段路径。
2. 业务规则错误：阻断不合法类型组合或越权调用。
3. 外部依赖错误：标记降级状态，保留最近一次可用结果或 fallback。
4. 重试、降级或补偿：严格遵守后台任务和启停统一规则，不在请求线程里偷偷长阻塞。
5. 插件缺失、插件未安装或插件加载失败：按插件级错误处理，不升级为宿主启动级错误。

## 6. 正确性属性

### 6.1 属性 1：宿主主权不外包

*对于任何* 插件接入场景，系统都应该满足：宿主仍然掌握权限、审计、设备/实体标准、卡片规范和记忆主权。

**验证需求：** `requirements.md` 需求 2、需求 3

### 6.2 属性 2：同一能力只有一套正式语义

*对于任何* 插件类型、槽位名、配置作用域和 DTO 字段，系统都应该满足：规范文档、开发者手册和核心代码使用同一命名和同一契约。

**验证需求：** `requirements.md` 需求 1、需求 2

## 8. 风险与待确认项

### 8.1 风险

- 当前宿主内已有 `weather`、`ha_integration`、`device_integration` 等历史模块，迁移时容易边改边留旧特判。
- 内置插件现在实现风格不统一，迁移过程中可能暴露更多隐藏耦合。
- 开发者文档现有中英双份内容较多，重构时如果没有事实来源总表，很容易再次漂移。

### 8.2 待确认项

- 开发者手册最终是否保留双语并行，还是先以中文正式版为主、英文延后同步。
- `homeassistant` 相关能力最终合并为单插件还是继续保留多插件拆分但统一类型语义。
- `third_party` 插件未来是否也开放宿主私有 migration runner，还是统一要求第三方走外部安装器 / 独立初始化流程。

### 8.3 2026-03-19 收尾结果

- 宿主公共标准实体承载层已经落地，运行态标准实体由 `device_entities` 正式承载。
- `DeviceBinding.capabilities` 只保留绑定摘要、主实体标识、实体 ID 列表和轻量元数据，不再承载完整运行态实体快照。
- 设备读取、设备控制、快捷指令和天气插件都已经切到“公共实体层优先，旧 JSON 仅兼容兜底”的统一口径。
