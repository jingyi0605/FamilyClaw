# Spec 004.8 - 插件系统 V1 定稿与全量迁移

状态：Done

## 2026-03-18 子 Spec 说明

`004.8` 继续作为插件系统 V1 的父级总 Spec。

从今天开始，AI 供应商这条子线由下面这份新主 Spec 继续展开：

- `specs/004.8.1-AI供应商彻底插件化迁移/`

如果你现在要处理：

- AI 供应商彻底插件化迁移
- 宿主与 `ai-provider` 插件边界
- `provider driver contract`
- AI Gateway 核心去硬编码

请不要只停留在 `004.8`，而是直接进入 `004.8.1`。

## 这次要解决什么问题

插件系统现在最大的问题不是“能不能继续加功能”，而是边界还没彻底收口：

- 宿主里还残留领域模块和插件特判
- 插件类型语义还不够统一，`connector`、`memory-ingestor` 这类旧说法还在污染实现
- 开发者文档已经开始分散，稳定规则和高频变动接口混在一起，后面只会越改越乱
- 内置插件的实现风格也不一致，有的像集成，有的像动作插件，有的还兼带旧记忆写入语义

这次 Spec 不是补丁式修修补补，而是直接把插件系统收口到最新的 V1 规范：

1. 让宿主只保留平台内核
2. 让插件按正式类型和正式接口接入
3. 把开发者文档重构成可长期维护的官方手册
4. 把现有内置插件、官方插件统一迁到新架构

## 这次要钉死的架构边界

- 宿主核心只负责插件运行时、插件实例、虚拟设备、标准实体、标准卡片、统一任务与错误语义，不再保留天气、电费、健康等领域专用主链路。
- 插件私有表、插件私有缓存、provider 绑定关系、领域刷新游标和领域快照都属于插件私有实现，不得注册进宿主核心 ORM 总模型入口，也不得由宿主核心 Alembic migration 直接创建、升级或删除。
- 宿主读取设备和实体时，只消费标准实体，不按 `weather`、`power`、`health` 这类领域做专用归一化或乱码修补。
- `official` 与 `third_party` 插件不是宿主镜像的一部分，也不得成为宿主导入期依赖；缺失这些目录时，宿主迁移和启动仍必须成立。
- 插件必须在写入宿主前自行产出完整、可展示、可持久化的标准设备和标准实体；这次验收不是“天气能用”，而是“宿主不再知道天气是什么”。

## 这轮最终收口结果

- 宿主对 `official` 天气插件的导入期依赖、天气专用 API 和设备读取链路里的天气专用修补已经移除。
- 插件私有持久化 / 插件私有迁移机制已经收口成正式宿主能力，天气私有绑定表也已经迁出宿主核心 Alembic 历史。
- 宿主公共标准实体承载层已经补齐，运行态标准实体不再以 `DeviceBinding.capabilities.entities` 作为正式事实源。

## 为什么现在做

现在项目还没正式发布，这是最适合清债的时候。

如果继续拖着旧语义和半插件化结构往前走，后面每加一个天气、电费、设备、健康能力，宿主都会再膨胀一圈。那不是“灵活”，那是架构失控。

## 这次 Spec 的交付物

- 一套按 V1 规范重构后的插件开发者手册
- 一套按 V1 规范重构后的宿主插件核心代码
