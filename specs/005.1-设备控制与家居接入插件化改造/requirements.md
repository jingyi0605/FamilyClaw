# 需求文档 - 设备控制与家居接入插件化改造

状态：Draft

## 简介

项目现在已经有两套互相打架的现实：

- 一边已经有插件系统、内置插件目录、插件执行链路和插件任务体系
- 另一边 Home Assistant 的设备接入和控制主链仍然写在核心模块里，插件目录里的 HA 插件大多只是占位壳

这会带来三个直接问题：

1. 核心代码里已经开始长出平台细节，后面再接米家、涂鸦之类平台时，复杂度会爆炸。
2. 现在所谓“HA 内置插件”名义上是插件，实际上没有接管真正主链，架构信息是假的。
3. 语音快路径、场景、手动控制、未来自动化流程都需要设备控制，但这些调用方不该知道具体平台怎么发命令。

这次 Spec 的目标很明确：

- 本地只保留统一控制协议、参数校验、权限、审计、风险控制、编排这些通用能力
- 真实设备接入和真实设备控制都下放给插件实现
- 先把 Home Assistant 从“内置壳插件”改成真正承载主链的内置插件
- 让后续米家、涂鸦之类平台可以按同一套路接进来，而不是继续在核心里堆 `if vendor == xxx`

## 术语表

- **System**：FamilyClaw API Server
- **统一控制协议**：系统内部统一使用的设备动作定义，例如 `turn_on`、`turn_off`、`set_temperature`、`lock`
- **动作定义**：一个动作的名字、参数 schema、默认校验规则、风险等级和返回格式
- **控制请求**：调用方发起的一次设备控制，包含家庭、设备、动作、参数、原因和幂等信息
- **控制执行器**：核心层的统一接口，只负责把控制请求路由给合适插件，不负责具体平台协议
- **集成插件**：负责某个平台设备同步、状态采集、能力声明或控制执行的插件
- **平台适配层**：核心里保留的一层薄封装，用来做插件发现、标准 payload 组装、错误归一；不允许在这里写平台设备实现
- **设备实现层**：`light.turn_on`、`climate.set_temperature`、米家云接口调用之类真实平台逻辑。这个层必须留在插件里
- **绑定信息**：本地设备和外部平台对象之间的映射，例如外部平台、插件 id、外部设备 id、主实体 id、能力快照
- **接入同步**：把外部平台的房间、设备、实体、能力同步到本地模型的过程

## 范围说明

### In Scope

- 定义统一设备控制协议、标准动作、标准参数和校验规则
- 定义统一设备接入协议，让插件负责候选发现、设备同步、房间同步和绑定写入
- 把现有 HA 设备控制主链改成通过插件执行
- 把现有 HA 设备接入主链改成通过插件执行
- 把语音快路径、对话快动作、场景执行继续收口到统一控制协议，不再直接耦合 HA
- 为插件执行补上权限控制、高风险确认、审计日志、幂等、超时、错误归一
- 把现有 HA 内置插件从占位壳改成真正执行插件
- 保留最小平台适配层和执行器接口，不在核心继续写平台设备实现
- 为后续米家、涂鸦等平台留出接入位，但本 Spec 只要求把 HA 跑通

### Out of Scope

- 本次不实现米家、涂鸦等新平台的正式接入和控制
- 本次不重做整个设备数据模型，只在必要范围内补字段和约束
- 本次不把所有设备页面一起重做成新 UI
- 本次不承诺补完所有厂商设备的高级能力，只先覆盖现有已支持动作的插件化迁移
- 本次不做实时状态订阅总线重构，仍允许先以同步、回写和按需刷新为主
- 本次不做插件市场安装能力，只处理系统内置插件和现有挂载能力

## 需求

### 需求 1：系统必须把设备控制收口到统一控制协议

**用户故事：** 作为后端维护者，我希望所有调用方都只面对统一动作和标准参数，这样语音、场景、页面和未来自动化都不用知道底层到底是 HA 还是别的平台。

#### 验收标准

1. WHEN 任一调用方发起设备控制 THEN System SHALL 使用统一动作名和标准参数，而不是直接传平台 service 名称。
2. WHEN 核心层处理控制请求 THEN System SHALL 先做标准参数校验、设备归属校验和动作支持校验，再进入插件执行。
3. WHEN 未来新增一个平台 THEN System SHALL 不要求在核心控制主链新增 `if vendor == xxx` 分支才能接入。
4. WHEN 某个动作需要平台特定参数映射 THEN System SHALL 由插件完成平台参数翻译，而不是让调用方或核心服务直接拼平台 payload。

### 需求 2：系统必须把真实设备控制执行下放给插件

**用户故事：** 作为架构维护者，我希望核心层只负责通用规矩，真实平台怎么控制设备由插件自己处理，这样平台差异不会把主系统拖烂。

#### 验收标准

1. WHEN 系统执行 `turn_on`、`set_temperature`、`lock` 等动作 THEN System SHALL 通过统一执行器路由到对应插件，而不是直接调用 HA 客户端或平台 SDK。
2. WHEN 执行目标是 Home Assistant 设备 THEN System SHALL 由 HA 动作插件把统一动作映射成真实 HA service call。
3. WHEN 插件执行超时、失败或返回非法结果 THEN System SHALL 收口成统一错误结构，而不是把平台内部异常直接漏给上层调用方。
4. WHEN 高风险动作需要确认 THEN System SHALL 在进入插件前完成确认拦截，而不是把风险判断下放给插件自行决定。

### 需求 3：系统必须把设备接入和同步主链下放给插件

**用户故事：** 作为平台接入维护者，我希望设备候选发现、设备同步、房间同步都由平台插件负责，这样接别的平台时不需要继续复制一套 `ha_integration` 风格的核心模块。

#### 验收标准

1. WHEN 用户查看平台设备候选列表 THEN System SHALL 通过接入插件获取候选项，而不是由核心直连 HA registry。
2. WHEN 用户执行设备同步或房间同步 THEN System SHALL 通过接入插件获取平台数据，并由核心统一落本地设备、房间和绑定信息。
3. WHEN 同步过程出现部分失败 THEN System SHALL 返回统一同步摘要，包括创建数、更新数、跳过数和失败原因。
4. WHEN 某个平台没有实现接入插件 THEN System SHALL 拒绝对应同步请求，并返回明确错误，而不是退回平台硬编码逻辑。

### 需求 4：系统必须保留通用控制能力在核心层

**用户故事：** 作为产品维护者，我希望权限、高风险确认、场景编排、审计、幂等和超时这些规矩保持统一，这样换平台不会把安全和行为搞乱。

#### 验收标准

1. WHEN 任一控制请求进入系统 THEN System SHALL 在插件执行前完成权限、家庭归属和设备可控性校验。
2. WHEN 任一高风险动作进入系统 THEN System SHALL 在插件执行前完成确认判断，并把确认结果写进审计上下文。
3. WHEN 场景、语音快路径或手动操作触发控制 THEN System SHALL 复用同一条通用控制主链，而不是各自调用平台实现。
4. WHEN 控制请求成功或失败 THEN System SHALL 统一写审计日志、结构化执行结果和错误上下文。
5. WHEN 同一控制请求被重复提交 THEN System SHALL 通过幂等键、请求指纹或执行上下文避免重复落执行。

### 需求 5：系统必须把插件选择和设备绑定做成正式数据

**用户故事：** 作为后端维护者，我希望本地设备能明确知道自己归哪个插件负责执行和同步，这样后续一个家庭里同时存在多个平台时不会乱。

#### 验收标准

1. WHEN 设备由某个平台同步进入系统 THEN System SHALL 在正式绑定信息里记录负责它的 `plugin_id`、平台类型、外部对象标识和能力快照。
2. WHEN 系统执行某个设备动作 THEN System SHALL 根据设备绑定信息定位执行插件，而不是靠 `vendor` 字段临时猜。
3. WHEN 同一个家庭存在多个平台设备 THEN System SHALL 能按设备级绑定路由到正确插件，而不是假设“全家只有 HA”。
4. WHEN 绑定信息缺失、损坏或指向已停用插件 THEN System SHALL 拒绝执行并给出明确错误。

### 需求 6：Home Assistant 必须从占位内置插件改成真正内置插件

**用户故事：** 作为项目负责人，我希望 HA 这条主链真的走插件执行，而不是文档里叫插件、代码里还是核心直连，不然以后没人敢相信插件系统到底是不是正式能力。

#### 验收标准

1. WHEN 系统执行 HA 设备控制 THEN System SHALL 走 `homeassistant-device-action` 或等价正式 HA 控制插件，而不是核心直接调用 `ha_integration` 客户端。
2. WHEN 系统执行 HA 设备同步或房间同步 THEN System SHALL 走 `homeassistant-device-sync` 或等价正式 HA 接入插件，而不是核心直接拉 HA registry。
3. WHEN 查看 HA 内置插件代码 THEN System SHALL 能看到真实平台实现，而不是只返回 demo 数据或固定假结果。
4. WHEN HA 插件被当前家庭停用 THEN System SHALL 阻止继续同步或控制 HA 设备，并给出可理解错误。

### 需求 7：迁移过程不能破坏现有上层调用方

**用户故事：** 作为现有功能维护者，我希望语音、场景、对话和后台接口继续工作，只是底层换成插件化执行，而不是为了改架构先把用户功能全弄坏。

#### 验收标准

1. WHEN 语音快路径、对话快动作或场景执行触发设备控制 THEN System SHALL 继续使用现有入口，但底层改为统一控制协议 + 插件执行。
2. WHEN 前端调用现有设备同步、HA 配置或设备控制接口 THEN System SHALL 保持核心接口语义稳定，除非本 Spec 明确声明变更。
3. WHEN 迁移期间旧代码路径尚未完全删掉 THEN System SHALL 保证单次请求只走一条正式执行链，不允许新旧逻辑双写双执行。
4. WHEN HA 插件化完成后 THEN System SHALL 删除或冻结核心里的 HA 设备实现层，避免长期双轨。

## 非功能需求

### 非功能需求 1：可靠性

1. WHEN 插件执行进程异常、平台接口异常或网络超时 THEN System SHALL 返回统一错误码和可追踪错误信息。
2. WHEN 同步请求发生部分失败 THEN System SHALL 保持成功部分可提交、失败部分可追踪，不要求一次全成或全回滚。
3. WHEN 插件结果格式不符合协议 THEN System SHALL 拒绝接收并记录结构化错误，避免脏数据写入本地模型。

### 非功能需求 2：可维护性

1. WHEN 新增一个设备平台 THEN System SHALL 主要通过新增插件和少量配置完成，不要求复制一套核心控制模块。
2. WHEN 维护者排查一次控制失败或同步失败 THEN System SHALL 能从请求、执行器、插件、平台错误和审计日志一路串起来。
3. WHEN 维护者阅读核心代码 THEN System SHALL 能明显看出“统一协议在核心，平台实现在插件”，而不是边界混乱。

### 非功能需求 3：性能与执行约束

1. WHEN 设备控制请求命中本地已配置插件 THEN System SHALL 在可接受的同步超时阈值内完成，并支持统一超时配置。
2. WHEN 批量同步设备或房间 THEN System SHALL 允许插件批量拉取外部数据，避免逐设备往返拖慢整体过程。
3. WHEN 同一家庭短时间内重复发起相同控制请求 THEN System SHALL 通过幂等控制避免重复执行真实平台动作。

## 成功定义

- `device_action` 主链不再直接耦合 HA 实现，而是改为统一控制协议 + 插件执行
- HA 设备接入和控制两条主链都已经真正走内置插件
- 现有语音、场景、对话和手动接口不需要理解平台细节
- 核心代码里不再出现继续扩张的平台硬编码控制分支
- 后续接米家、涂鸦时，新增工作主要落在插件，不再重写核心主链
