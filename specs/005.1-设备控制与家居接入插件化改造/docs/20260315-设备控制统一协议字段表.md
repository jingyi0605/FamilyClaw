# 设备控制统一协议字段表

## 这份文档是干什么的

这份文档专门把 `005.1` 里提到的“统一设备控制协议”拆成可以直接落代码的字段表。

它主要回答这些问题：

1. 一次标准设备控制请求到底长什么样
2. 插件拿到的 payload 到底有哪些字段
3. 插件返回结果必须长什么样
4. 统一动作有哪些、每个动作允许带什么参数
5. 风险等级、错误码和幂等规则怎么定

这份文档不是大方向说明，而是后续写 schema、service、插件 entrypoint 时直接对照用的。

---

## 1. 协议设计原则

先把死规矩写清楚，不然后面字段会越长越乱。

### 1.1 核心原则

- 上层调用方只发统一动作，不发平台 service 名
- 插件只做平台翻译和真实执行，不做权限和高风险判断
- 核心统一负责审计、幂等、超时、错误归一
- payload 要能支持多平台，但不能抽象到没人看得懂

### 1.2 当前版本号

- 控制请求发给插件的 payload 版本：`device-control.v1`
- 插件返回结果的 payload 版本：`device-control-result.v1`

后面如果要加字段，优先新增可选字段；只有出现不兼容变化时才升大版本。

## 2. 统一动作定义表

这一层是系统协议，不是平台协议。

## 2.1 动作总表

| 动作名 | 说明 | 允许设备类型 | 风险等级 | 备注 |
| --- | --- | --- | --- | --- |
| `turn_on` | 打开设备或启动工作模式 | `light`、`climate`、`media_player` | `low` | 最常用基础动作 |
| `turn_off` | 关闭设备或停止工作模式 | `light`、`climate`、`media_player` | `low` | 最常用基础动作 |
| `set_brightness` | 调整亮度 | `light` | `low` | 亮度统一按百分比 |
| `set_temperature` | 调整目标温度 | `climate` | `medium` | 温度单位先统一摄氏度 |
| `set_hvac_mode` | 调整空调模式 | `climate` | `medium` | 先限定固定枚举 |
| `open` | 打开类动作 | `cover` | `low` | 用于窗帘等 |
| `close` | 关闭类动作 | `cover` | `low` | 用于窗帘等 |
| `stop` | 停止当前开合动作 | `cover` | `low` | 用于窗帘等 |
| `play_pause` | 播放或暂停切换 | `media_player` | `low` | 统一做成切换动作 |
| `set_volume` | 调整音量 | `media_player` | `low` | 音量统一按百分比 |
| `lock` | 上锁 | `lock` | `medium` | 是否高风险可后续再细调 |
| `unlock` | 解锁 | `lock` | `high` | 必须走高风险确认 |

### 2.2 当前明确不进协议的动作

这些动作不是永远不做，而是当前版本先别放进统一协议，避免一开始就发散：

- `set_color`
- `set_fan_mode`
- `set_cover_position`
- `mute`
- `unmute`
- `next_track`
- `previous_track`
- `start_clean`
- `stop_clean`

原因很简单：

- 这些动作的平台差异更大
- 现有主链还没稳定
- 先把当前项目已经真实用到的动作收口更重要

## 3. 动作参数字段表

## 3.1 通用规则

- 所有动作都允许 `params = {}`
- 没有参数的动作必须传空对象，不要传 `null`
- 参数校验先在核心完成，插件默认收到的是已通过校验的标准参数
- 插件不能要求上层再补一份平台专用参数

## 3.2 各动作参数定义

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| 无 | - | - | 该动作不要求标准参数 | `params` 必须是空对象或被忽略 |

### `turn_off`

| --- | --- | --- | --- | --- |
| 无 | - | - | 该动作不要求标准参数 | `params` 必须是空对象或被忽略 |

### `set_brightness`

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `brightness_pct` | integer | 是 | 目标亮度百分比 | `1~100` |

说明：

- 核心协议只认百分比，不认 HA 的 `0~255`
- 插件负责把百分比换成平台真实取值

说明：

说明：

## 4. 核心控制请求字段表

这部分是统一控制服务收到的标准请求。

## 4.1 `DeviceControlRequest`

| 字段 | 类型 | 必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| `household_id` | string | 是 | 家庭 id | `household_demo` |
| `device_id` | string | 是 | 本地设备 id | `device_light_001` |
| `action` | string | 是 | 统一动作名 | `turn_on` |
| `params` | object | 是 | 标准参数对象 | `{}` |
| `reason` | string | 是 | 触发原因 | `voice_fast_path` |
| `confirm_high_risk` | boolean | 是 | 是否已经确认高风险动作 | `false` |
| `idempotency_key` | string | 否 | 幂等键 | `voice:session123:step1` |
| `requested_by` | object | 否 | 操作人上下文摘要 | `{ "actor_type": "admin", "actor_id": "member_1" }` |

## 4.2 核心层校验规则

进入插件前，核心必须至少做这些校验：

1. `household_id` 和设备归属一致
2. 设备存在且允许控制
3. `action` 存在于统一协议注册表
4. 设备类型支持该动作
5. `params` 符合该动作参数 schema
6. 高风险动作已确认
7. 绑定信息存在，且能定位到启用中的插件

如果上面任何一条不成立，必须在核心直接失败，不能把脏请求扔给插件。

## 5. 发给插件的标准 payload 字段表

## 5.1 `DeviceControlPluginPayload`

| 字段 | 类型 | 必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| `schema_version` | string | 是 | payload 版本 | `device-control.v1` |
| `request_id` | string | 是 | 一次控制请求唯一 id | `req_123` |
| `household_id` | string | 是 | 家庭 id | `household_demo` |
| `plugin_id` | string | 是 | 目标插件 id | `homeassistant-device-action` |
| `binding` | object | 是 | 本次执行使用的绑定信息 | 见下表 |
| `device_snapshot` | object | 是 | 本地设备快照 | 见下表 |
| `action` | string | 是 | 统一动作名 | `set_brightness` |
| `params` | object | 是 | 已标准化参数 | `{ "brightness_pct": 80 }` |
| `timeout_seconds` | number | 是 | 插件执行超时 | `8` |
| `reason` | string | 是 | 调用原因 | `scene.sleep_mode` |
| `risk_level` | string | 是 | 本次动作风险级别 | `low` |
| `idempotency_key` | string | 否 | 幂等键 | `scene:exec_1:step_2` |

### 5.2 `binding` 子对象字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `binding_id` | string | 是 | 本地绑定记录 id |
| `platform` | string | 是 | 平台标识，例如 `home_assistant` |
| `plugin_id` | string | 是 | 负责插件 id |
| `external_device_id` | string | 否 | 平台设备级 id |
| `external_entity_id` | string | 否 | 平台主实体 id |
| `capabilities` | object | 否 | 最近一次同步得到的能力快照 |

### 5.3 `device_snapshot` 子对象字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 本地设备 id |
| `name` | string | 是 | 本地设备名 |
| `device_type` | string | 是 | 本地设备类型 |
| `status` | string | 是 | 当前本地状态 |
| `controllable` | boolean | 是 | 当前是否允许控制 |
| `room_id` | string | 否 | 所属房间 id |

### 5.4 插件看到但不应该改的字段

这些字段插件可以读，但不应该拿来重写系统规则：

- `risk_level`
- `reason`
- `idempotency_key`

插件最多可以把它们带到平台日志里做追踪，不能反过来改变核心策略。

## 6. 插件返回结果字段表

## 6.1 `DeviceControlPluginResult`

| 字段 | 类型 | 必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| `schema_version` | string | 是 | 结果版本 | `device-control-result.v1` |
| `success` | boolean | 是 | 是否执行成功 | `true` |
| `platform` | string | 是 | 平台标识 | `home_assistant` |
| `plugin_id` | string | 是 | 执行插件 id | `homeassistant-device-action` |
| `executed_action` | string | 是 | 实际执行的统一动作 | `turn_on` |
| `external_request` | object | 否 | 发给平台的摘要 | `{ "domain": "light", "service": "turn_on" }` |
| `external_response` | object | 否 | 平台返回摘要 | `{ "status": "ok" }` |
| `normalized_state_patch` | object | 否 | 对本地状态的标准回写建议 | `{ "status": "active" }` |
| `error_code` | string | 否 | 错误码 | `platform_request_failed` |
| `error_message` | string | 否 | 错误说明 | `HA service call failed` |

## 6.2 `normalized_state_patch` 允许字段

当前版本只允许插件建议回写这些标准字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `status` | string | 本地设备状态建议，如 `active`、`offline` |
| `last_seen_at` | string | 平台最近成功响应时间 |
| `last_action_at` | string | 最近控制执行时间 |

先只开放少量字段，防止插件直接把本地设备快照写乱。

## 6.3 插件结果的硬性规则

1. `success=true` 时，不允许同时带 `error_code`
2. `success=false` 时，必须带 `error_code` 和 `error_message`
3. 插件返回的 `executed_action` 必须和请求动作一致，除非协议后续明确支持动作降级
4. `external_request` 和 `external_response` 只能放摘要，不能泄漏 token、cookie 或完整敏感 payload

## 7. 统一错误码表

## 7.1 核心层错误码

| 错误码 | 谁产生 | 含义 |
| --- | --- | --- |
| `device_not_found` | 核心 | 设备不存在 |
| `device_not_controllable` | 核心 | 设备存在，但不允许控制 |
| `device_binding_missing` | 核心 | 设备没有正式绑定 |
| `plugin_not_available` | 核心 | 插件不存在、停用或未注册 |
| `action_not_supported` | 核心 | 当前设备类型不支持该统一动作 |
| `invalid_action_params` | 核心 | 参数不符合标准 schema |
| `high_risk_confirmation_required` | 核心 | 高风险动作未确认 |
| `permission_denied` | 核心 | 当前 actor 不允许执行 |
| `idempotency_conflict` | 核心 | 幂等判断命中重复请求 |
| `plugin_execution_timeout` | 核心 | 插件执行超时 |
| `plugin_result_invalid` | 核心 | 插件结果结构非法 |

## 7.2 插件层错误码

插件允许返回这些归一错误码：

| 错误码 | 含义 |
| --- | --- |
| `platform_request_failed` | 平台请求失败 |
| `platform_auth_failed` | 平台认证失败 |
| `platform_target_not_found` | 平台对象不存在 |
| `action_not_supported_by_platform` | 平台不支持该动作或参数 |
| `platform_rate_limited` | 平台限流 |
| `platform_response_invalid` | 平台回包异常 |
| `plugin_internal_error` | 插件自身异常 |

说明：

- 插件可以保留更细的内部错误，但对核心主链必须先翻译成统一错误码
- 核心再决定这些错误如何映射到 API 响应

## 8. 幂等规则表

设备控制最怕重复执行，尤其是门锁和场景链路。

## 8.1 总体规则

- 核心负责幂等判断
- 插件可以利用 `idempotency_key` 做平台侧辅助追踪，但不能替代核心幂等
- 默认不做自动重试，避免真实设备重复执行

## 8.2 推荐幂等策略

| 场景 | 幂等键建议 |
| --- | --- |
| 页面手动控制 | `ui:<actor_id>:<device_id>:<action>:<timestamp_bucket>` |
| 语音快路径 | `voice:<session_id>:<step_index>` |
| 场景执行 | `scene:<execution_id>:<step_index>` |
| 对话快动作 | `conversation:<message_id>:<device_id>:<action>` |

## 8.3 行为规则

1. 同一幂等键的请求在未过期窗口内重复提交，核心应返回已有结果或明确拒绝重复执行。
2. `unlock` 这类高风险动作必须启用幂等控制，不能因为网络抖动重复开门。
3. `turn_on` 这类低风险动作虽然容错更高，也仍然建议走统一幂等链，保持系统行为一致。

## 9. 插件动作映射示例

这里只举例说明“谁负责翻译”。

### 9.1 `set_brightness`

核心请求：

```json
{
  "action": "set_brightness",
  "params": {
    "brightness_pct": 80
  }
}
```

HA 插件内部可以翻成：

```json
{
  "domain": "light",
  "service": "turn_on",
  "service_data": {
    "entity_id": "light.living_room_main",
    "brightness_pct": 80
  }
}
```

注意：

- 核心不应该知道这是 `light.turn_on`
- 这件事只能在 HA 插件里发生

### 9.2 `set_temperature`

核心请求：

```json
{
  "action": "set_temperature",
  "params": {
    "temperature_c": 24
  }
}
```

HA 插件内部可以翻成：

```json
{
  "domain": "climate",
  "service": "set_temperature",
  "service_data": {
    "entity_id": "climate.bedroom_ac",
    "temperature": 24
  }
}
```

## 10. 当前版本先不支持的协议能力

为了避免一版协议就把自己搞死，下面这些先别进：

- 一次请求控制多个设备
- 一次请求下发多个动作
- 平台专属参数透传
- 插件直接写本地设备表
- 插件直接决定权限和风险策略
- 插件自由扩展核心错误码

这些以后不是不能做，但当前阶段会直接把边界搞烂。

---

## 11. 推荐落地顺序

这份字段表如果要落到代码，建议按这个顺序来：

1. 先把动作定义表和参数 schema 落进代码注册表
2. 再定义 `DeviceControlRequest` / `DeviceControlPluginPayload` / `DeviceControlPluginResult`
3. 再把核心校验和错误码收进统一控制服务
4. 最后改 HA 插件，让它按这份协议收发

---

## 12. 最终判断

统一设备控制协议这件事，重点不是“字段尽量多”，而是：

- 上层只说业务动作
- 核心只管统一规矩
- 插件才做平台翻译

只要这三层边界守住，字段是活的；
边界守不住，字段表写再漂亮也没用。
