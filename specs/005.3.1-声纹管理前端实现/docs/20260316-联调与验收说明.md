# 20260316-联调与验收说明

状态：Draft

## 这份文档是干什么的

这份文档只回答一件事：声纹管理前端现在怎么联、怎么验、出了问题先看哪里。

不要再靠口头传达“你先点这个，再等那个接口回来”。后面谁接手，直接照着这里走。

## 当前实现范围

本轮已经落地这些能力：

- `设置 -> 设备与集成 -> 小爱音箱设备详情 -> 声纹管理`
- 设备级 `开启声纹识别` 开关
- 公开对话 / 成员路由说明
- 家庭成员声纹状态列表
- 首次录入向导
- 更新声纹入口
- 非管理员只读
- 声纹区域失败局部降级

本轮不做这些：

- 不重做 `005.3` 后端识别主链
- 不新增 CAM++ 或模型切换
- 不新增独立设备管理页面
- 不做删除声纹或停用声纹入口

## 联调前提

联调前先确认下面几件事：

1. `api-server` 已经包含 `voiceprint_identity_enabled` 字段和对应 Alembic migration。
2. `api-server` 已经提供 `GET /api/v1/voiceprints/households/{household_id}/summary?terminal_id=...`。
3. 当前家庭下至少有一台小爱音箱设备，且设备详情入口已经能正常打开。
4. 当前家庭下至少有一个成员，不然录入向导不会给你可选列表。
5. 联调账号如果要改配置，必须是家庭管理员。

## 入口和页面结构

正式入口只有一条：

- `设置 -> 设备与集成 -> 设备列表 -> 小米小爱音箱 -> 设备详情 -> 声纹管理`

设备详情里现在有两个标签页：

- `语音接管`
- `声纹管理`

不要跑去家庭页找，不在那里。

## 设备级开关联调

### 要看什么

- 设备当前是 `公开对话` 还是 `成员路由`
- 开关切换后文案有没有跟着变
- 保存失败时 UI 有没有回滚

### 联调步骤

1. 打开一台小爱音箱的设备详情，切到 `声纹管理`。
2. 看顶部策略卡，确认当前文案是否和设备真实状态一致。
3. 打开 `开启声纹识别`：
   - 页面应显示“系统会优先按声纹识别成员”
   - 页面应显示“识别成功后进入对应成员对话”
   - 页面应显示“识别失败时按后端既有降级规则继续处理”
4. 关闭 `开启声纹识别`：
   - 页面应显示“当前按公开对话处理”
   - 页面应显示“所有家庭成员都可以看到这台设备的对话内容”
5. 人为制造保存失败时，确认：
   - 开关会回滚到旧值
   - 只在声纹区域显示错误
   - 设备详情其他区域还能继续用

### 对应接口

- `PATCH /api/v1/devices/{device_id}`
- 关键字段：`voiceprint_identity_enabled`

## 成员状态列表联调

### 目标

确认管理员能一眼看懂谁还没建档、谁在建档、谁可用、谁失败、谁已经停用。

### 当前状态映射

- `not_enrolled`：未建档
- `pending`：建档中
- `active`：可用
- `failed`：失败
- `disabled`：已停用

### 每种状态现在应该看到什么

- `未建档`
  - 文案：还没有可用声纹档案
  - 操作：`开始录入`
- `建档中`
  - 文案：当前正在处理样本，显示已采集轮数
  - 操作：`查看进度`
- `可用`
  - 文案：当前档案已可用，显示累计样本轮数
  - 操作：`更新声纹`
- `失败`
  - 文案：最近一次建档没有成功，展示错误信息
  - 操作：`重新录入`
- `已停用`
  - 文案：已有历史档案，但当前不可用
  - 操作：`重新录入`

### 联调步骤

1. 打开声纹管理页。
2. 看成员列表是否覆盖所有家庭成员。
3. 随机抽一个有声纹、一个没声纹、一个处理中或失败的成员，核对状态和按钮是否匹配。
4. 如果设备存在进行中的建档任务，顶部应出现横幅，并提供 `继续查看进度` 入口。

### 对应接口

- `GET /api/v1/voiceprints/households/{household_id}/summary?terminal_id=...`

## 首次录入联调

### 正常流程

1. 在成员列表里点击 `开始录入`。
2. 第一步应看到成员选择。
3. 第二步应看到：
   - 目标成员
   - 目标设备
   - “首版默认采 3 轮样本”的说明
   - “识别失败不会打断现有语音主链”的说明
4. 点击开始后，应创建建档任务并进入进度页。
5. 进度页应显示：
   - 当前轮数
   - 任务状态
   - 最近更新时间
6. 完成后应进入成功页，并刷新成员状态列表。

### 对应接口

- `POST /api/v1/voiceprints/enrollments`
- `GET /api/v1/voiceprints/enrollments/{enrollment_id}`

## 更新声纹联调

### 要看什么

- 已有可用声纹的成员显示 `更新声纹`
- 进入的是同一套向导，不是第二套页面
- 只改文案，不改流程

### 联调步骤

1. 找一个 `可用` 状态的成员。
2. 点击 `更新声纹`。
3. 检查向导标题和按钮文案已经变成更新语义。
4. 后续步骤仍然是：
   - 确认成员和设备
   - 创建建档任务
   - 看进度
   - 看结果
5. 成功后回到列表，确认该成员状态仍是 `可用`，更新时间已刷新。

## 权限联调

### 非管理员视角

非管理员进入同一页面时应该看到：

- 可以看设备策略和成员状态
- 设备级开关不可修改
- `开始录入`、`更新声纹`、`继续查看进度` 都不能操作
- 页面说明明确提示当前只有查看权限

### 这一步重点不是“完全隐藏”

这里做的是只读，不是把整块页面藏掉。原因很简单：成员也需要知道这台设备现在按什么策略处理对话。

## 失败排查

### 现象 1：声纹区域打不开，但设备详情别的内容正常

先看：

- `GET /api/v1/voiceprints/households/{household_id}/summary?terminal_id=...` 是否失败
- `terminal_id` 是否传成了别的设备 ID
- 当前设备是否真的属于当前家庭

这是预期中的局部降级，不该把整个设备页拖死。

### 现象 2：开关切了又弹回去

先看：

- `PATCH /api/v1/devices/{device_id}` 是否报错
- 返回体里是否包含 `voiceprint_identity_enabled`
- 设备 schema 是否已经补齐这个字段

这也是预期行为：保存失败时会回滚，不允许前端假装保存成功。

### 现象 3：成员显示建档中，但管理员看不到进度

先看：

- summary 里的 `pending_enrollment.enrollment_id`
- 成员项里的 `pending_enrollment_id`
- `GET /api/v1/voiceprints/enrollments/{enrollment_id}` 是否可读

现在前端已经支持从顶部横幅或成员按钮继续查看进度。

### 现象 4：录入失败后用户以为整条语音能力坏了

先看页面文案是不是被改坏了。

正确文案应该始终表达：

- 识别失败只影响声纹能力
- 后端会按既有降级规则继续处理
- 不会把整条语音主链打断
