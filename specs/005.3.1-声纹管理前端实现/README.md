# Spec 005.3.1 - 声纹管理前端实现

状态：Draft

## 这份 Spec 要解决什么

`005.3` 已经把后端主链路立住了：

- 小爱普通对话已经能在进 LLM 前先做声纹识别
- 建档、档案、识别、降级都已经有后端能力
- 首版方案已经钉死：本地 ONNX + `sherpa-onnx` + `weSpeaker/ResNet34`

现在真正缺的不是算法，而是用户能不能把这套能力用起来。

如果 `user-app` 没有正式入口，结果就是：

- 家庭管理员不知道哪台音箱开了声纹识别
- 不知道关闭声纹后，系统会按什么规则处理对话
- 不知道怎么给成员录入声纹、更新声纹
- 这套能力虽然在后端存在，但对真实用户来说等于没交付

所以 `005.3.1` 的目标很明确：

1. 在音响设备管理界面补一个正式的“声纹管理”标签页
2. 让管理员可以按设备开启或关闭声纹识别
3. 让管理员可以通过向导给家庭成员录入并绑定声纹
4. 让已有声纹支持更新，而不是只能删库重来

## 核心判断

✅ 值得做，而且现在就该做。

后端已经把“先声纹，后上下文兜底”的普通对话主链做出来了。前端如果继续没有入口，这套能力就只能停在联调层，根本不算产品能力。

## 和 `005.3` 的关系

- `005.3` 负责后端能力和普通对话识别主链
- `005.3.1` 负责把这些能力接到 `user-app` 的设备管理界面
- `005.3.1` 不重做算法，不改首版模型，不讨论 CAM++

一句话说清：

> `005.3` 解决“系统怎么识别人”，`005.3.1` 解决“用户怎么配置和管理这件事”。

## 当前前端现实情况

基于 `apps/user-app` 当前代码，现状已经很清楚：

- 音箱发现、设备列表和音箱语音接管设置，已经在 `设置 -> 设备与集成` 页面
- 当前设备页已经有音箱专属“语音设置”弹窗
- 家庭页会拉设备数据，但不是单设备详情页，不适合承接完整的声纹管理流程

所以本 Spec 的首版入口不另起炉灶，直接承接现有设备管理结构：

- 主入口：`设置 -> 设备与集成`
- 设备级入口：小爱音箱设备详情
- 标签页：`语音接管` / `声纹管理`

## 这份 Spec 包含什么

- `requirements.md`：把设备级开关、公开对话、成员绑定、更新声纹这些产品要求写清
- `design.md`：把前端入口、状态模型、页面结构、向导流程和接口依赖写清
- `tasks.md`：把后续实现顺序拆开，避免一上来把 UI、接口、联调搅成一锅粥
- `docs/README.md`：给后续联调文档、验收文档预留入口

## 这份 Spec 明确不做什么

- 不重做 `005.3` 的后端识别主链
- 不在前端讨论模型切换和算法路线
- 不把网关或 `voice-runtime` 的职责搬到前端
- 不把整个设备管理重写成另一套新页面
- 不在这份 Spec 里扩展“多说话人分离”之类超范围能力

## 计划产物

- 设备管理里的音箱详情补“声纹管理”标签页
- 设备级“开启声纹识别”开关及公开对话提示
- 成员声纹状态列表
- 新建声纹向导
- 更新声纹流程
- 前端测试、联调文档和验收口径
