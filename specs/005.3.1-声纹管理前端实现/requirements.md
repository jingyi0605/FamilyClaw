# 需求文档 - 声纹管理前端实现

状态：Draft

## 简介

`005.3` 已经把后端声纹能力接进普通对话主链，但 `user-app` 还没有正式的管理入口。现在管理员只能看见音箱设备和语音接管设置，看不见这台设备是否启用声纹识别，也没法给家庭成员录入、更新声纹。

这会直接带来两个问题：

- 功能已经存在，但用户根本不会用
- 对话到底是公开可见，还是按成员身份路由，前端没有明确表达

本 Spec 只处理前端产品化，把设备管理里的声纹管理入口、设备级开关、成员建档和更新流程补齐。

## 术语表

- **System**：`user-app` 前端及其依赖的 `api-server` 接口
- **音箱设备管理界面**：`设置 -> 设备与集成` 中的小爱音箱设备管理区域
- **声纹管理标签页**：音箱设备详情中的一个独立标签页，用来管理该设备的声纹识别配置和成员声纹状态
- **公开对话**：设备未启用声纹识别时，这台音箱进入 FamilyClaw 的对话按公开会话处理，所有家庭成员都可以看到
- **成员路由**：设备启用声纹识别后，系统优先按声纹识别出的成员身份路由对话；识别失败时仍按 `005.3` 的降级规则处理
- **声纹建档向导**：前端引导管理员为某个家庭成员发起建档、采集多轮样本并完成绑定的流程
- **声纹更新**：针对已有声纹档案追加新样本并更新档案，不要求用户先手动删旧档案

## 范围说明

### In Scope

- 在现有音箱设备管理界面补“声纹管理”入口
- 设备级开启或关闭声纹识别
- 明确显示“公开对话”与“成员路由”的区别
- 展示家庭成员的声纹状态
- 通过向导给成员录入并绑定声纹
- 发起成员声纹更新
- 补前端测试和联调文档

### Out of Scope

- 重做 `005.3` 后端算法和主链
- 新增模型选择、模型切换或模型评测页面
- 多说话人分离
- 语音网关或 `voice-runtime` 侧的身份决策
- 新开一套独立设备管理页面替代现有“设备与集成”

## 需求

### 需求 1：系统必须在音箱设备管理界面提供正式的声纹管理入口

**用户故事：** 作为家庭管理员，我希望在管理某台小爱音箱时，能直接看到和声纹相关的设置与状态，而不是去猜这项能力藏在哪个页面。

#### 验收标准

1. WHEN 管理员进入 `设置 -> 设备与集成` 并打开一台小爱音箱详情 THEN System SHALL 提供 `语音接管` 和 `声纹管理` 两个清晰分开的标签页。
2. WHEN 当前设备不是支持声纹识别的小爱音箱 THEN System SHALL 不展示 `声纹管理` 标签页，也不显示误导性的空入口。
3. WHEN 管理员第一次进入 `声纹管理` 标签页 THEN System SHALL 先展示这台设备当前是否启用声纹识别、启用后的行为和当前成员声纹概况。

### 需求 2：系统必须支持按设备开启或关闭声纹识别，并明确说明公开对话行为

**用户故事：** 作为家庭管理员，我希望能控制某台音箱是否启用声纹识别，并清楚知道关闭后系统会按公开对话处理，而不是留下一堆模糊状态。

#### 验收标准

1. WHEN 管理员在某台音箱上关闭声纹识别 THEN System SHALL 在界面上明确展示“该设备当前按公开对话处理，所有家庭成员都可以看到对话内容”。
2. WHEN 管理员在某台音箱上开启声纹识别 THEN System SHALL 在界面上明确展示“系统会优先按声纹识别成员，再进入对应成员对话”。
3. WHEN 管理员切换设备级开关 THEN System SHALL 调用正式设备配置接口保存状态，而不是只改前端本地状态。
4. WHEN 保存设备级开关失败 THEN System SHALL 回滚开关展示并给出明确错误提示。

### 需求 3：系统必须展示当前家庭成员的声纹状态，避免管理员盲配

**用户故事：** 作为家庭管理员，我希望在设备的声纹管理页里一眼看出哪些成员还没录过声纹、哪些成员已经建档、哪些成员最近在处理中或失败了。

#### 验收标准

1. WHEN 管理员打开 `声纹管理` 标签页 THEN System SHALL 展示当前家庭成员列表及每个成员的声纹状态摘要。
2. WHEN 某成员还没有声纹档案 THEN System SHALL 显示明确的“未建档”状态和“开始录入”操作入口。
3. WHEN 某成员已有生效中的声纹档案 THEN System SHALL 显示最近更新时间、样本数量或等价状态摘要，并提供“更新声纹”入口。
4. WHEN 某成员存在进行中的建档任务或最近失败记录 THEN System SHALL 展示处理中或失败状态，而不是只显示成“有档案”或“没档案”。

### 需求 4：系统必须支持通过向导为家庭成员录入并绑定声纹

**用户故事：** 作为家庭管理员，我希望用一个明确的向导流程完成“选成员、确认设备、开始录音、看进度、得到结果”这件事，而不是靠猜接口是否成功。

#### 验收标准

1. WHEN 管理员点击“开始录入” THEN System SHALL 进入一个明确分步的声纹建档向导。
2. WHEN 向导启动 THEN System SHALL 要求管理员先确认目标成员、目标音箱和录入前提示，不允许在成员和设备信息不清楚时直接开录。
3. WHEN 管理员确认开始 THEN System SHALL 发起正式建档任务，并展示这次录入默认按多轮样本处理的进度。
4. WHEN 建档进行中 THEN System SHALL 展示当前轮次、当前状态和用户下一步该做什么，而不是只显示一个一直转圈的加载状态。
5. WHEN 建档成功 THEN System SHALL 明确显示这次录入已经绑定到目标成员，并刷新成员声纹状态列表。
6. WHEN 建档失败、取消或超时 THEN System SHALL 给出明确结果，并允许管理员重新发起，而不是把界面卡死在处理中。

### 需求 5：系统必须支持更新已有成员的声纹

**用户故事：** 作为家庭管理员，我希望已有声纹的成员可以重新录入或追加样本来更新声纹，而不是先删掉旧数据再手动重建。

#### 验收标准

1. WHEN 某成员已有生效中的声纹档案 THEN System SHALL 提供“更新声纹”入口。
2. WHEN 管理员发起更新声纹 THEN System SHALL 复用声纹建档向导，但文案要明确这是更新，不是首次录入。
3. WHEN 更新成功 THEN System SHALL 刷新该成员的声纹更新时间和状态摘要。
4. WHEN 更新过程中存在进行中的任务 THEN System SHALL 阻止重复发起并说明原因。

### 需求 6：系统必须保证权限、降级和现有行为不被前端误导

**用户故事：** 作为产品负责人，我希望这套前端不会把 `005 / 005.2 / 005.3` 的边界搞乱。管理员能改配置，普通成员不会误操作，识别失败也不能被前端说成“系统坏了”。

#### 验收标准

1. WHEN 当前用户不是家庭管理员 THEN System SHALL 不允许修改设备级声纹开关，也不允许发起成员建档或更新。
2. WHEN 声纹识别开启但后端识别失败 THEN System SHALL 在帮助文案里明确这是“先声纹，失败后系统会降级处理”，而不是暗示对话一定失败。
3. WHEN 设备关闭声纹识别 THEN System SHALL 保持现有设备管理和语音接管行为不被破坏，只把对话身份策略显示为公开对话。
4. WHEN 前端拿不到声纹汇总数据 THEN System SHALL 保持设备页其他功能可用，并把声纹区域降级成可理解的错误或重试提示。

### 需求 7：系统必须用现有后端能力优先落地，并明确接口缺口

**用户故事：** 作为开发者，我希望这份前端 Spec 不会假装后端什么都有，也不会为了一页 UI 硬造一堆新体系。现有接口能复用就复用，缺口要明说。

#### 验收标准

1. WHEN 前端需要读取成员声纹状态 THEN System SHALL 优先复用 `005.3` 已有的 `voiceprints` 能力，不能重新定义一套声纹档案体系。
2. WHEN 现有设备接口还没有声纹开关字段 THEN Spec SHALL 明确写出需要补齐的最小字段和用途。
3. WHEN 现有接口不足以支撑可用前端 THEN Spec SHALL 明确写出推荐的聚合接口或兜底方案，而不是把实现责任丢给开发时再猜。

## 非功能需求

### 非功能需求 1：清晰性

1. WHEN 管理员进入声纹管理页 THEN System SHALL 用用户能看懂的话解释“公开对话”和“成员路由”，不能直接丢底层术语。
2. WHEN 向导展示进度 THEN System SHALL 明确说明当前在第几步、接下来做什么、失败后怎么办。

### 非功能需求 2：可维护性

1. WHEN 后续继续扩展音箱设备详情 THEN System SHALL 复用现有“设备与集成”页面和音箱详情弹层结构，而不是再造一套平行页面。
2. WHEN 后续后端补齐聚合接口 THEN 前端状态模型 SHALL 能从“多接口拼装”平滑切到“单接口汇总”，不要求整页推倒重写。

### 非功能需求 3：可靠性

1. WHEN 某个声纹接口失败 THEN System SHALL 只影响声纹管理区域，不影响设备页其他操作。
2. WHEN 页面刷新或重新进入 THEN System SHALL 能重新读取最新建档状态，不把用户困在前端缓存里。

## 成功定义

- 管理员可以在现有音箱设备管理界面直接找到声纹管理入口
- 管理员可以按设备开启或关闭声纹识别，并明白关闭后是公开对话
- 管理员可以为成员发起首次录入并完成绑定
- 管理员可以为已有成员更新声纹
- 页面能明确展示成员声纹状态、进行中任务和失败结果
- 前端设计没有破坏现有设备管理结构，也没有和 `005.3` 的后端边界打架
