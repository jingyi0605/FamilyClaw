# Spec 001.5.1 - AI供应商官方对齐与 Coding Plan 官方插件扩展

状态：Historical

## 2026-03-18 历史说明

这份 Spec 保留的是一个过渡阶段：

- 对齐部分内置供应商的官方文档
- 补部分流式能力
- 以阶段性方式扩展 Coding Plan 供应商

它记录的是“旧实现如何往前推一步”，不是 AI 供应商彻底插件化迁移的最终规则。

从现在开始，下面这些问题统一由 `004.8.1` 定义：

- 宿主和 `ai-provider` 插件的最终边界
- 供应商 driver contract
- 供应商声明、协议适配、流式输出迁出核心的路径
- 核心兼容层的退场标准

如果你现在是在推进 AI 供应商彻底插件化，请优先看：

- `specs/004.8.1-AI供应商彻底插件化迁移/`

## 这份 Spec 解决什么问题

这次不是搞新花样，是把已经暴露出来的问题收干净。

现在 AI 供应商这块有两类真问题：

1. 现有 14 家内置供应商虽然非流式主链路都不是空实现，但有一部分默认配置已经和官方文档不对齐，继续放着迟早出错。
2. `claude` 和 `gemini` 的流式能力没补齐，表面上像“支持供应商”，实际上能力不完整。

除此之外，业务还要新增 3 家专门做 Coding Plan 的供应商：

- 百炼 Coding Plan
- Kimi Coding Plan
- GLM Coding Plan

这 3 家不能再按旧路子塞进内置 provider 注册表里，必须按正式插件方式接入，而且是 `ai-provider` 官方插件，不是 builtin provider 特判。

## 这次明确要交付什么

- 把已有 14 家内置供应商按官方文档和 OpenClaw 逐项对齐，先处理报告里的 P0 / P1。
- 补齐 `claude` 和 `gemini` 的流式通讯能力。
- 新增 3 家 Coding Plan 供应商，并按官方插件方式接入到现有插件系统。
- 把官方插件和第三方插件的新挂载、新安装目录统一收口到 `apps/api-server/data/plugins`，为后续 Docker 把 `data/` 挂到外部做准备。
- 服务启动时自动扫描 `data/plugins`，恢复官方插件挂载、第三方手工挂载和插件市场已安装实例，但不把“恢复”做成“重新安装”。
- 保证新插件服从统一插件启停规则，不绕开 `plugin_mounts`、家庭可见性和执行前校验。
- 把这次核查证据、官方资料和 OpenClaw 对照都收进 `docs/`，后面改代码时不再来回翻资料。

## 这次明确不做什么

- 不顺手重写整套 AI Gateway 架构。
- 不把所有内置供应商一次性全搬成插件，只处理本次新增的 3 家 Coding Plan。
- 不把这 3 家再做成 builtin provider 或虚拟 manifest。
- 不扩成“任意新供应商市场”，这次只做报告里明确点名的问题和 3 个新增供应商。

## 当前判断

这事值得做，而且要尽快做。

原因很直接：

- 真问题已经被报告点出来了，不是臆想需求。
- 现有插件系统已经支持 `ai-provider`，再把 3 家 Coding Plan 写成 builtin provider 只是继续堆垃圾。
- 现在收口，能避免后面每加一家供应商都再补一层特判。
