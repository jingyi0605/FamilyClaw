## 2. 先给结论

### 【核心判断】

✅ 值得立刻修一部分。

原因很简单：

- **非流式 14 家都已经接上真实调用逻辑，不存在内置供应商直接落到空实现的情况。**
- **但这不等于“14 家都已经完整正确”。**
- 当前真正的问题有两类：
  - **协议不完整**：`Claude`、`Gemini` 没有流式实现。
  - **默认配置和官方文档存在高风险偏差**：`MiniMax`、`Doubao Coding`、`BytePlus ModelArk`、`BytePlus Coding`。

### 【关键洞察】

- 数据结构：当前系统把 14 家供应商压成 3 条通讯链路。
  - OpenAI 兼容：12 家
  - Anthropic 原生：1 家
  - Gemini 原生：1 家
- 复杂度：非流式链路并不乱，乱的是“供应商默认 base_url”和“流式支持范围”。
- 风险点：最危险的不是 `_invoke_simulated()`，而是**配置看起来像接上了，实际上和官方最新文档不一致**。

## 3. 当前实现概览

### 3.1 当前代码如何分流

`provider_runtime.py` 的实际分流逻辑是：

- OpenAI 兼容链路：
  - `chatgpt`
  - `deepseek`
  - `qwen`
  - `glm`
  - `siliconflow`
  - `kimi`
  - `minimax`
  - `openrouter`
  - `doubao`
  - `doubao-coding`
  - `byteplus`
  - `byteplus-coding`
- Anthropic 原生链路：
  - `claude`
- Gemini 原生链路：
  - `gemini`

对应实现位置：

- OpenAI 兼容调用：`provider_runtime.py` 中 `_invoke_openai_compatible()`，通过 `_resolve_chat_endpoint()` 拼出 `/chat/completions`
- Claude：`provider_runtime.py` 中 `_invoke_anthropic_messages()`，走 `POST /messages`
- Gemini：`provider_runtime.py` 中 `_invoke_gemini_generate_content()`，走 `POST /models/{model}:generateContent?key=...`

### 3.2 是否存在空实现

结论先说死：

- **14 个内置供应商的非流式调用，不会落到 `_invoke_simulated()`。**
- `_invoke_simulated()` 仍然存在，但它不是这 14 家内置供应商当前的实际主链路。
- 真正缺的是 **流式实现**，不是“空实现”。

### 3.3 流式支持范围

`stream_provider_invoke()` 现在只允许：

- `api_family == "openai_chat_completions"`
- 或 `transport_type in {"openai_compatible", "local_gateway"}`

这意味着：

- `Claude` 流式：**没有实现**
- `Gemini` 流式：**没有实现**

影响范围：

- `family_qa` 的流式问答有异常兜底，会降级成非流式调用。
- `llm_task.stream_llm()` 没有同级别兜底，如果主供应商切到 `Claude` 或 `Gemini`，会直接报 `stream_not_supported`。

## 5. 空实现与缺实现，别混为一谈

### 5.1 没有空实现的部分

当前 14 个内置供应商在 **非流式** 情况下都能进入真实调用分支：

- OpenAI 兼容供应商进入 `_invoke_openai_compatible()`
- `Claude` 进入 `_invoke_anthropic_messages()`
- `Gemini` 进入 `_invoke_gemini_generate_content()`

所以这件事可以定性为：

- **没有“14 家里某几家根本没接”的空实现问题**

### 5.2 真正缺实现的部分

当前缺的是 **流式**：

- `Claude`：无流式实现
- `Gemini`：无流式实现

这不是“伪装成已完成但实际上空函数”，而是**明确没做**。

这件事在产品层面的实际影响比“空实现”更大，因为：

- 配置页里你能选到它
- 非流式你也能调通
- 但一旦进实时流式对话链路，就会炸

这比直接空实现更坑人，因为它会让人误以为“已经全量支持”。

## 6. 和 openclaw 的源码对照

openclaw 已经把几个关键供应商的路由分清楚了，说明这不是我在瞎猜，别人已经踩过坑了。

### 6.1 MiniMax

openclaw 的文档明确写：

- 优先 `https://api.minimax.io/anthropic`
- 协议优先 `anthropic-messages`
- `https://api.minimax.io/v1` 只是 OpenAI 兼容可选项

对应位置：

- `openclaw-main/docs/providers/minimax.md`

这和当前项目默认 `https://api.minimax.chat/v1` 差得比较远。

### 6.2 Moonshot / Kimi

openclaw 明确区分：

- 国际：`https://api.moonshot.ai/v1`
- 中国区：`https://api.moonshot.cn/v1`

对应位置：

- `openclaw-main/docs/providers/moonshot.md`

所以当前项目固定写 `.cn` 不是错，但它只是一个区域默认值，不是全局默认值。

### 6.3 Doubao Coding / BytePlus Coding

openclaw 直接把 coding 独立成另一条基址：

- Doubao 普通：`https://ark.cn-beijing.volces.com/api/v3`
- Doubao Coding：`https://ark.cn-beijing.volces.com/api/coding/v3`
- BytePlus 普通：`https://ark.ap-southeast.bytepluses.com/api/v3`
- BytePlus Coding：`https://ark.ap-southeast.bytepluses.com/api/coding/v3`

对应位置：

- `openclaw-main/src/agents/doubao-models.ts`
- `openclaw-main/src/agents/byteplus-models.ts`

这和当前项目“普通版 / coding 版共用同一个通用 `/api/v3` 基址”的做法不一致。

## 7. 修正优先级

### P0：应该先改

1. `BytePlus Coding`
   - 当前默认值高风险错误。
   - 既不是官方当前主域名，也不是 coding 专用路径。

2. `MiniMax`
   - 默认基址 `api.minimax.chat` 没有看到当前官方主文档继续推荐。
   - 如果继续走 OpenAI 兼容，至少也该对齐 `api.minimax.io/v1`。
   - 更稳妥的是直接补齐 Anthropic 兼容实现。

3. `Claude` / `Gemini` 流式
   - 如果产品里存在实时流式对话场景，这两家现在就不算“完整对接”。

### P1：紧随其后

1. `Doubao Coding`
   - 官方公开文档对独立 coding endpoint 的公开证据不够集中，但 openclaw 已经分成 `/api/coding/v3`。
   - 当前继续共用 `/api/v3` 风险不低。

2. `BytePlus ModelArk`
   - 协议没问题，但默认域名落后官方主文档，应该尽快对齐。

### P2：顺手一起修

1. `SiliconFlow`
   - 需要确认 `.cn` 是否仍然是 LLM chat 的长期正式入口。
   - 同时要把供应商特判从域名判断改成供应商代码判断，不要把兼容逻辑绑死在 `.cn` 字符串上。

2. `Kimi`
   - 建议像 openclaw 一样显式区分 `.ai` 国际入口和 `.cn` 中国区入口。

## 8. 最终结论

### 8.1 如果问题是“14 家是不是都接上了真实 LLM 通讯逻辑？”

回答是：

- **非流式：是，14 家都接上了真实逻辑。**
- **流式：不是，Claude 和 Gemini 没接完整。**

### 8.2 如果问题是“有没有空实现？”

回答是：

- **没有发现内置 14 家供应商在非流式主链路上落入空实现。**
- 但存在 **缺实现**，就是 `Claude` 和 `Gemini` 的流式能力没做。

### 8.3 如果问题是“当前调用方式是不是都和官方文档一致？”

回答是：

- **不是。**
- 目前最明显的高风险项是：
  - `MiniMax`
  - `Doubao Coding`
  - `BytePlus ModelArk`
  - `BytePlus Coding`

一句话总结：

**这套系统不是没接上，而是“接上了大部分非流式调用”，但其中几家的默认配置已经落后官方文档，另外两家的流式根本没补齐。**
