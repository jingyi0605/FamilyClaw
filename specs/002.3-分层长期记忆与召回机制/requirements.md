# 需求文档 - 分层长期记忆与召回机制

状态：Draft

## 简介

FamilyClaw 现在已经有记忆卡片、事件入库、记忆候选和基础权限控制，但这些能力还只是“能存”，不是“能长期记、能按需找、能稳定注入对话”。用户今天说过的话、家庭最近发生的事、反复被观察到的偏好、插件里已有的知识，仍然分散在不同链路里。

这次要解决的不是做一个更花哨的记忆页，而是把长期记忆变成主链路的一部分：

- 让系统知道哪些内容只属于当前轮，哪些应该留在会话层，哪些应该沉淀成稳定事实。
- 让 PostgreSQL 负责结构化真相源，别再靠字符串拼接和临时摘要撑着。
- 让 `free_chat` 真正读到命中的记忆，而不是只读一段“热摘要”。
- 让后续新能力可以继续接进来，而不是每加一个来源就再补一层 if/else。

## 术语表

- **L0 当前轮上下文**：本轮对话需要的最近消息、设备上下文和实时状态，不要求长期持久化。
- **L1 会话记忆**：一个 `session` 当前阶段的滚动摘要、未完成话题、最近确认事项。
- **L2 情节记忆**：带明确时间锚点的事件摘要或会话摘要，强调“什么时候发生过什么”。
- **L3 语义记忆**：稳定事实、长期偏好、成员关系等可长期复用的信息。
- **L4 外部知识**：插件原始记录、说明文档、规则文档等不直接属于用户对话，但对回答有帮助的知识。
- **真相源**：业务上真正负责保存事实的数据表，不为了检索方便去篡改语义。
- **召回投影**：为了关键词检索、向量检索和混合排序而生成的检索数据，不直接代表业务真相。
- **Hybrid Recall**：把关键词分数、向量相似度和业务权重合并后的召回结果。
- **Promotion**：把多次重复出现的观察或事件提升成稳定语义记忆的过程。

## 范围说明

### In Scope

- 建立 `L0` 到 `L4` 的分层长期记忆模型和对应边界。
- 使用 PostgreSQL 落地关键词索引、向量索引和混合排序。
- 区分“用户明确要求记住”“事件流入库”“会话沉淀”“外部知识导入”四类写入路径。
- 修复 `free_chat` 的记忆注入逻辑，改为分组注入真正命中的结果。
- 持久化记录每轮注入了哪些记忆，便于 debug 和回放。
- 分三期推进，先做最小闭环，再扩到会话记忆和统一召回。

### Out of Scope

- 第一版不做通用知识图谱，不引入新的大而全知识治理平台。
- 第一版不做多向量、多模型、多租户跨库检索平台。
- 第一版不重做前端记忆中心全部交互，只补主链路必须的接口和调试信息。
- 第一版不把所有历史文件、所有插件数据一次性全量导入；先支持可控来源。

## 需求

### 需求 1：系统必须提供清晰的分层记忆边界

**用户故事：** 作为开发者，我希望系统能区分当前轮上下文、会话记忆、情节记忆、语义记忆和外部知识，这样后续接入新来源时不会把不同语义的数据混成一锅粥。

#### 验收标准

1. WHEN 设计或实现记忆写入链路 THEN System SHALL 明确标注写入目标属于 `L1`、`L2`、`L3` 或 `L4`，而不是用一个通用表兜底所有语义。
2. WHEN 对话主链路组装上下文 THEN System SHALL 只把最近消息和设备上下文当作 `L0`，不得把 `L0` 误当长期记忆落库。
3. WHEN 新增记忆来源 THEN System SHALL 能先判断它属于哪一层，再决定真相源和召回投影，避免继续追加临时分支。

### 需求 2：系统必须用 PostgreSQL 提供关键词和语义混合召回

**用户故事：** 作为家庭助手的后端维护者，我希望长期记忆的检索建立在 PostgreSQL 原生能力上，这样我们能先用现有技术栈把召回做对，而不是引入一套额外的复杂基础设施。

#### 验收标准

1. WHEN 记忆进入可召回状态 THEN System SHALL 为其建立 `tsvector + GIN` 关键词检索能力。
2. WHEN 记忆进入可召回状态 THEN System SHALL 为其建立 `pgvector` 语义检索能力，或在扩展缺失时明确标记为降级模式。
3. WHEN 用户发起记忆相关问题 THEN System SHALL 使用关键词分数、向量分数以及 `importance/confidence/recency/member match/visibility` 共同参与排序。
4. WHEN 向量召回暂时不可用 THEN System SHALL 保留关键词召回，不得让整条问答链路直接失效。

### 需求 3：系统必须区分不同的记忆写入路径

**用户故事：** 作为用户，我希望“请记住这个”“最近发生了一件事”“我们聊了很久的一段计划”这几类输入走不同写入路径，这样系统既不会漏记，也不会把短期噪音都塞进长期记忆。

#### 验收标准

1. WHEN 用户明确表达“记住”“以后都按这个来”这类指令 THEN System SHALL 走显式记忆候选或自动确认链路，而不是等后台事件碰运气。
2. WHEN 插件或系统事件进入记忆链路 THEN System SHALL 先记录到事件或情节层，再决定是否提升成语义记忆。
3. WHEN 同一观察在一段时间内被重复命中 THEN System SHALL 支持把它从情节层提升为语义层，而不是无止境堆重复事件。
4. WHEN 输入只适合当前轮或短期会话 THEN System SHALL 不得直接写成长期稳定事实。

### 需求 4：系统必须生成并维护会话记忆

**用户故事：** 作为用户，我希望系统记得“这次会话刚聊到哪”“还有什么没说完”“刚刚确认了什么”，这样对话不会因为轮数变多就瞬间失忆。

#### 验收标准

1. WHEN 一个会话达到摘要阈值或进入结束态 THEN System SHALL 生成或刷新该会话的滚动摘要。
2. WHEN 会话中存在未完成话题或最近确认事项 THEN System SHALL 把它们保存在可查询的会话记忆结构中。
3. WHEN 后续同一会话继续聊天 THEN System SHALL 优先读取最新有效的会话记忆，而不是只回看最近几条消息。

### 需求 5：系统必须把真正命中的记忆分组注入 free_chat

**用户故事：** 作为用户，我希望 AI 助手回答时读到真正相关的长期记忆，而不是只读一小段通用摘要。

#### 验收标准

1. WHEN `free_chat` 组装提示词 THEN System SHALL 按“稳定事实”“最近事件”“当前会话摘要”至少三组注入命中的记忆。
2. WHEN 某一组没有命中结果 THEN System SHALL 明确留空或省略该组，不得把其他组内容混充进去。
3. WHEN 记忆召回结果参与回答 THEN System SHALL 记录本轮注入了哪些 `memory_id` 或来源 ID，便于 debug 和回放。
4. WHEN 权限不允许查看某条记忆 THEN System SHALL 在注入前完成过滤，不得把不可见记忆送进 prompt。

### 需求 6：系统必须支持统一纳入外部知识

**用户故事：** 作为系统维护者，我希望插件原始记录、说明文档和规则文档以后也能进入统一召回，而不是各模块自己读自己的一份。

#### 验收标准

1. WHEN 插件原始记录被声明为可召回来源 THEN System SHALL 能把它转换成 `L4` 外部知识来源并进入统一召回。
2. WHEN 项目说明文档或规则文档被导入 THEN System SHALL 能保留来源信息、更新时间和可见范围。
3. WHEN 外部知识进入统一召回 THEN System SHALL 与 `L1/L2/L3` 使用同一套排序和注入边界，而不是独立绕过主链路。

### 需求 7：系统必须保证旧能力不被破坏

**用户故事：** 作为现有用户和维护者，我希望新记忆系统上线时不破坏当前问答、记忆管理、提醒和事件链路。

#### 验收标准

1. WHEN 新增分层记忆和召回投影 THEN System SHALL 保持现有 `memory_cards`、事件写入和记忆纠错接口的兼容性。
2. WHEN 新链路出现异常 THEN System SHALL 允许问答主链路降级继续运行，而不是整轮失败。
3. WHEN 召回投影需要重建或刷新 THEN System SHALL 不要求删除真相源数据后重来。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 家庭规模处于常规范围且召回文档量在可控区间内 THEN System SHALL 让单次记忆召回在可接受的接口延迟内完成，目标为后端 `p95 < 800ms`。
2. WHEN 需要刷新向量或关键词投影 THEN System SHALL 优先走后台异步任务，不得阻塞用户当前请求。

### 非功能需求 2：可靠性

1. WHEN `pgvector` 扩展缺失、embedding 失败或投影刷新异常 THEN System SHALL 进入可观测的降级模式，并保留关键词召回或旧投影。
2. WHEN 真相源和召回投影暂时不一致 THEN System SHALL 以真相源为准，投影可以重建，真相源不得被覆盖破坏。

### 非功能需求 3：安全与权限

1. WHEN 系统查询任意层级的记忆 THEN System SHALL 始终执行 `household/member/visibility` 三类权限过滤。
2. WHEN 调试链路记录注入结果 THEN System SHALL 记录来源 ID、分组和分数，不得在日志里额外泄露超出当前 actor 权限范围的明文内容。

### 非功能需求 4：可维护性

1. WHEN 后续新增新的记忆来源或新的召回分组 THEN System SHALL 优先通过扩展来源适配器或投影器完成，而不是继续在 orchestrator 里叠条件分支。
2. WHEN 需要排查“为什么这轮答错了” THEN System SHALL 能从请求 trace 或 debug 记录中看出当时注入了哪些记忆、来自哪一层、为什么命中。

## 成功定义

- `free_chat` 不再只注入热摘要，而是能注入按组组织的真实命中记忆。
- `memory_cards` 至少具备 PostgreSQL 关键词召回和向量召回闭环。
- 会话达到阈值后能产出 `L1` 会话记忆。
- 事件和重复观察可以沉淀成 `L2/L3`，而不是一直停留在散乱记录里。
- 任何一轮对话都能追踪“本轮读了哪些记忆”。
