# 20260318-性能评估与准确率验收方案

状态：Draft

## 1. 这份文档解决什么问题

`002.3` 这套长期记忆方案如果只写“支持 PostgreSQL 混合召回”“支持 free_chat 分组注入”，那还不够。真正要上线，必须先回答下面几个硬问题：

- 1 年记忆、上千条记录时，一次 recall 到底要多久。
- `L1/L2/L3/L4` 各层命中的质量怎么量化，不再靠主观感觉。
- `pgvector` 缺失、embedding 失败、权限过滤生效时，系统是不是还能正常退化。
- `free_chat` 最终回答到底有没有真的用到召回结果，而不是查到了也白查。

这份文档就是把“性能评估”和“准确率验收”收成统一口径，后面开发、压测、回归和上线都按这一套来，不准各说各话。

## 2. 评估目标

本次评估只看和长期记忆主链路直接相关的四件事：

- recall 延迟是不是在可接受范围内。
- recall 结果是不是能把正确记忆捞出来。
- 注入到 `free_chat` 的分组是不是对、来源是不是可追踪。
- 降级路径是不是可用，不会因为向量链路炸掉就把整轮问答拖死。

不在本次评估范围内的内容：

- 大模型最终文风是否更自然。
- 前端记忆中心页面是否好看。
- 跨家庭、跨租户的全局知识治理。

## 3. 基准数据集设计

### 3.1 数据集分层

每份基准数据集都必须明确分成下面四层，不能只拿一锅 `memory_cards` 糊弄：

- `L1 session summaries`：滚动摘要、未完成话题、最近确认事项。
- `L2 episodic memories`：带时间锚点的事件摘要、会话片段摘要。
- `L3 semantic memories`：稳定事实、偏好、关系、长期约定。
- `L4 external knowledge`：插件原始记录、规则文档、说明文档。

每条数据至少要带这些字段：

- `household_id`
- `subject_member_id`
- `visibility`
- `importance`
- `confidence`
- `occurred_at` 或 `updated_at`
- `source_kind`
- `gold_group`

### 3.2 数据集规模

验收时至少准备三档数据，不准只拿玩具数据跑：

| 档位 | 单户总记录数 | 典型场景 | 用途 |
| --- | --- | --- | --- |
| S | 300 | 刚开始使用 1 到 3 个月 | 开发联调 |
| M | 1,500 | 约 1 年记忆 | 主验收基线 |
| L | 8,000 | 多成员长期使用 2 到 3 年 | 压力边界 |

建议默认按下面比例造数：

| 层级 | S | M | L |
| --- | --- | --- | --- |
| `L1` | 20 | 120 | 600 |
| `L2` | 80 | 500 | 3,000 |
| `L3` | 150 | 600 | 2,500 |
| `L4` | 50 | 280 | 1,900 |

### 3.3 查询集设计

每档数据集都要配一份标注好的 query 集。每个 query 至少有一个主答案来源，必要时允许多个可接受来源。

查询集至少包含这 8 类：

| 类别 | 例子 | 主要命中层 |
| --- | --- | --- |
| 稳定事实 | “我女儿乳糖不耐受吗” | `L3` |
| 长期偏好 | “以后提醒我都用中文” | `L3` |
| 最近事件 | “上周五我们最后确认了什么” | `L2` |
| 会话续聊 | “刚才说到哪了，继续” | `L1` |
| 多成员区分 | “妈妈的药几点吃，不是孩子的” | `L2/L3` |
| 规则知识 | “这个插件失败后系统会怎么处理” | `L4` |
| 时间约束 | “春节前谁说过要买空气净化器” | `L2` |
| 权限隔离 | “孩子的敏感记忆能不能被普通成员看到” | 权限过滤验证 |

每类 query 在 M 档至少 `40` 条，总数不低于 `320` 条。

### 3.4 数据来源

基准数据分两套，不能只做其中一套：

- 合成数据集：用脚本稳定生成，便于回归对比。
- 真实脱敏回放集：从现有 `memory_cards`、事件记录和文档中抽样脱敏，避免只在人造样本上表现好。

### 3.5 标注要求

每条 query 至少要标这些标签：

- `expected_layer`
- `expected_group`
- `acceptable_source_ids`
- `must_not_hit_source_ids`
- `requires_member_match`
- `requires_recency_bias`
- `requires_visibility_filter`

没有标注的数据，不算验收数据。

## 4. 压测口径

### 4.1 需要单独测的阶段

记忆 recall 不能只测“总时间”，必须拆开看：

| 阶段 | 说明 |
| --- | --- |
| `query_embedding` | 本轮 query 生成 embedding 的时间 |
| `fts_recall` | `tsvector + GIN` 关键词召回 |
| `vector_recall` | `pgvector` top-k 召回 |
| `hybrid_rerank` | 合并打分、排序、分组 |
| `trace_write` | 召回 trace 持久化 |
| `bundle_build_total` | recall bundle 总耗时 |
| `free_chat_prompt_build` | 组装 prompt 的总耗时 |

### 4.2 测试模式

每次验收至少跑这 4 种模式：

| 模式 | 说明 |
| --- | --- |
| FTS only | 禁用向量召回，只测关键词链路 |
| Hybrid local | embedding 使用本地或内网低延迟服务 |
| Hybrid remote | embedding 走真实外部模型服务 |
| Degraded | `pgvector` 不可用或 embedding 失败时的降级链路 |

### 4.3 并发口径

每种模式至少跑下面三档并发：

| 并发 | 说明 |
| --- | --- |
| 1 | 单请求基线 |
| 10 | 常规家庭活跃期 |
| 30 | 短时峰值压力 |

### 4.4 冷热启动口径

每种模式都要区分：

- 冷启动：数据库缓存未预热，应用缓存为空。
- 热启动：常见索引页和 household 级缓存已预热。

只报热启动成绩没有意义。

## 5. 延迟验收标准

### 5.1 单次 recall 验收门槛

下面是主验收基线，数据档位默认使用 M 档，即单户约 `1,500` 条可召回记录。

| 指标 | 目标 | 失败线 |
| --- | --- | --- |
| `fts_recall p95` | `<= 80ms` | `> 150ms` |
| `vector_recall p95` | `<= 120ms` | `> 220ms` |
| `hybrid_rerank p95` | `<= 30ms` | `> 60ms` |
| `bundle_build_total p95` 不含 query embedding | `<= 250ms` | `> 400ms` |
| `bundle_build_total p95` 含本地 embedding | `<= 350ms` | `> 500ms` |
| `bundle_build_total p95` 含远程 embedding | `<= 800ms` | `> 1,200ms` |

### 5.2 1 年记忆的预期范围

以“1 年记忆、上千条记录”为主参考时，预期是：

- 只算数据库召回本身，常规应落在 `30ms ~ 150ms`。
- 算上 query embedding，如果是远程模型，常规应落在 `180ms ~ 600ms`。
- 如果总 recall 时间长期超过 `800ms`，优先怀疑 embedding 调用、SQL 没走索引，或者召回候选数设得太大。

### 5.3 `free_chat` 主链路附加门槛

| 指标 | 目标 | 失败线 |
| --- | --- | --- |
| `free_chat_prompt_build p95` | `<= 900ms` | `> 1,300ms` |
| 召回失败后的主链路可降级继续回复比例 | `100%` | `< 99%` |
| trace 落库成功率 | `>= 99.9%` | `< 99%` |

## 6. 准确率验收指标

### 6.1 Recall@K

Recall@K 是主指标，至少统计 `K=1/3/5`。

计算规则：

- 只要 `acceptable_source_ids` 中任意一条出现在前 K 个结果里，就算命中。
- 对 `expected_group` 不匹配的结果，即使语义有点像，也不算正确命中。
- 越权结果一律算错，不参与“勉强可用”的解释。

### 6.2 辅助指标

除了 Recall@K，还必须统计：

- `Precision@5`
- `MRR`
- `Wrong-layer hit rate`
- `Wrong-member hit rate`
- `Visibility violation rate`
- `Injection grounding rate`

这里最重要的两个补充指标是：

- `Wrong-layer hit rate`：明明该命中 `L1`，却老把 `L3/L4` 顶上来的比例。
- `Injection grounding rate`：最终回答里真正使用了被注入记忆的比例。

### 6.3 分层验收门槛

M 档数据下，最低验收线如下：

| 场景 | Recall@1 | Recall@3 | Recall@5 | 备注 |
| --- | --- | --- | --- | --- |
| `L1 session summary` | `>= 0.75` | `>= 0.90` | `>= 0.95` | 同会话续聊必须高 |
| `L2 recent events` | `>= 0.55` | `>= 0.72` | `>= 0.82` | 时间偏置要生效 |
| `L3 stable facts` | `>= 0.70` | `>= 0.85` | `>= 0.92` | 明确记住的事实不能太差 |
| `L4 external knowledge` | `>= 0.50` | `>= 0.68` | `>= 0.78` | 文档切块差会拖后腿 |

### 6.4 负向指标门槛

| 指标 | 目标 | 失败线 |
| --- | --- | --- |
| `Wrong-layer hit rate` | `<= 12%` | `> 20%` |
| `Wrong-member hit rate` | `<= 5%` | `> 10%` |
| `Visibility violation rate` | `0` | `> 0` |
| `Injection grounding rate` | `>= 80%` | `< 65%` |

### 7.1 第一阶段验收

| P2-E2E-02 | 未完成话题 | 摘要里存在 open topic | “还有什么没说完” | 命中 `L1` open topic | 回答列出未完成事项 |

### 7.3 第三阶段验收

| 编号 | 场景 | 前置数据 | 用户问题 | 期望 recall | 期望回答 |
| --- | --- | --- | --- | --- | --- |
| P3-E2E-01 | 情节转语义 | 同一观察在 30 天内重复出现 3 次 | “孩子最近固定几点睡” | `L3 stable_facts` 优先命中提升后的语义记忆 | 回答使用稳定事实，而不是罗列 3 条事件 |
| P3-E2E-02 | 规则文档召回 | 已导入插件规则文档 | “插件失败后会怎么退化” | `external_knowledge` 命中对应规则文档 | 回答符合规则文档 |
| P3-E2E-03 | 权限隔离 | 存在仅 owner 可见的记忆 | 普通成员发起查询 | 不得召回越权记录 | 回答不泄露敏感信息 |

## 8. 分期上线门槛

### 8.1 第一期上线门槛

- `memory_cards` 已具备 `FTS + pgvector + hybrid rerank`。
- `stable_facts Recall@5 >= 0.85`。
- `recent_events Recall@5 >= 0.75`。
- `bundle_build_total p95` 在远程 embedding 模式下 `<= 800ms`。
- 降级模式下主链路成功回复率 `= 100%`。

### 8.2 第二期上线门槛

- `session_summary Recall@3 >= 0.90`。
- 同会话续聊端到端测试全部通过。
- `Wrong-layer hit rate <= 15%`。
- `trace` 能明确区分 `L1` 和 `L3` 来源。

### 8.3 第三期上线门槛

- `L1/L2/L3/L4` 全部进入统一 recall。
- `Visibility violation rate = 0`。
- `Injection grounding rate >= 80%`。
- 统一 recall 端到端回归全部通过。

## 9. 失败判定与排查顺序

只要出现下面任意一种情况，本期就不算通过：

- 远程 embedding 模式下 `bundle_build_total p95 > 1,200ms`。
- `stable_facts Recall@5 < 0.85`。
- `session_summary Recall@3 < 0.90`。
- 出现任何越权注入。
- trace 记录的来源集合和实际注入内容对不上。

排查顺序固定如下，不要一上来就怪模型：

1. 先看 query embedding 是否稳定，是否超时。
2. 再看 SQL 是否命中 `GIN` 和向量索引。
3. 再看 rerank 权重是否把 `importance/confidence/recency/member match/visibility` 配歪了。
4. 再看分组注入是否把正确结果丢了。
5. 最后才怀疑大模型没用好注入内容。

## 10. 建议的实现形态

为了后续回归不痛苦，建议把评估实现为固定目录下的一组脚本和测试：

- `apps/api-server/tests/benchmarks/memory_recall/`
- `apps/api-server/tests/integration/memory_recall/`
- `apps/api-server/tests/e2e/free_chat_memory/`

至少产出这 4 类结果文件：

- 数据集构造说明
- 原始压测结果 JSON
- Recall@K 统计结果
- 端到端验收报告

## 11. 最后一句话

这套长期记忆要验收，不是看 demo 时感觉“好像更聪明了”，而是看下面这三件事有没有过线：

- 查得够快
- 找得够准
- 答得可追踪

过不了这三条，就别吹长期记忆，只是把旧问题换了个说法。
