# Spec 004.5.1 - 官方天气插件

状态：Draft

## 这份 Spec 现在为什么要重写

这份 Spec 原来默认沿着“宿主里做天气主链路，插件补一个天气源”的方向写。现在这条路已经被明确判定为不对。

新的方向是：

- 宿主只提供插件运行时、设备、实体、动作、卡片这些平台内核
- 天气作为标准 `integration` 插件接入
- 默认家庭天气和附加地区天气都落到同一套实例、设备、实体模型

所以这份 Spec 现在不再是“做一个天气功能”，而是“做一个符合新插件架构的官方天气集成样板”。

## 这份 Spec 最终要建出什么

- 一个官方天气 `integration` 插件
- 默认天气源为 `MET Norway`
- 可选接入 `OpenWeather`、`WeatherAPI`
- 每个家庭启用后自动拥有一个默认天气实例
- 一个实例下可管理多个地区天气设备
- 所有天气数据通过统一实体暴露
- 仪表盘卡片按设备选择展示

## 这份 Spec 依赖什么前提

这份 Spec 不会自己偷偷补平台缺口。

它依赖下面三份规范先定稿并开始落地：

- [20260318-插件能力与接口规范-v1](C:/Code/FamilyClaw/docs/开发设计规范/20260318-插件能力与接口规范-v1.md)
- [20260318-现有插件系统迁移路线图](C:/Code/FamilyClaw/docs/开发设计规范/20260318-现有插件系统迁移路线图.md)
- [20260318-记忆中心内核与插件边界规范](C:/Code/FamilyClaw/docs/开发设计规范/20260318-记忆中心内核与插件边界规范.md)

同时还依赖这些既有能力：

- `001.3.1` 家庭地区坐标扩展
- `004.5` 插件能力统一接入与版本治理
- 插件启用禁用统一规则
- 后端事件循环与周期任务规范

## 这份 Spec 这次明确不做什么

- 不再把天气主链路建立在宿主领域模块上
- 不再把卡片绑定 city 文本
- 不再把天气源写死在宿主内部
- 第一版不扩 AQI、预警、日出日落

## 主要文档

- `requirements.md`：说明第一版天气集成到底交付什么
- `design.md`：说明实例、设备、实体、卡片和刷新策略怎么落
- `tasks.md`：按新平台边界重排实施顺序
