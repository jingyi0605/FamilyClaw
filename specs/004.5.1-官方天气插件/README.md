# Spec 004.5.1 - 官方天气插件

状态：Draft

## 这份 Spec 解决什么问题

用户要的是一个官方可装、默认能用、支持多地区、能上仪表盘的天气插件。真正困难的地方不在“查天气”三个字，而在边界：

- 默认天气源不能把官方插件绑死在不稳妥的商业条款上
- 查询不能继续依赖城市名猜测，必须走经纬度
- 家庭默认地区和手动添加的更多地区，都要落到统一的设备模型里
- 仪表盘卡片不能自造协议，必须复用现有插件、设备、实体体系

这份 Spec 的核心判断很简单：做可插拔天气源，默认 `MET Norway`，允许用户切换到 `OpenWeather` 或 `WeatherAPI` 并自行填写 key。

## 这次要做什么

- 做一个官方天气插件，默认使用 `MET Norway`
- 插件通过家庭地区解析出的经纬度查询天气
- 支持手动添加更多地区，每个地区作为一个天气设备
- 每个天气设备暴露一组固定天气实体
- 提供仪表盘卡片，支持选择指定天气设备展示

## 这次明确不做什么

- 不做集中式 SaaS 天气转发服务
- 不在第一版承诺 AQI、预警、日出日落全都有
- 不做自由文本城市名猜测查询主链路
- 不把天气供应商写死成单一实现

## 依赖关系

- 依赖 `../001.3.1-家庭地区坐标扩展/` 提供统一经纬度输入
- 复用 `../004.5-插件能力统一接入与版本治理/` 的插件、设备、实体和卡片接入规则
