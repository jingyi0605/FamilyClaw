# 20260318-官方天气插件接入说明

## 这份文档回答什么

这份文档只回答第一版真正已经落下来的边界：

- 默认天气设备怎么自动建出来
- 多地区天气设备怎么复用同一套设备模型
- 仪表盘怎么在不新增协议的前提下选择展示哪个天气设备
- 缓存、失败降级和错误状态到底怎么工作
- 哪些东西这次明确还没做

## 默认设备链路

插件 ID：`official-weather`

默认天气源：`MET Norway`

默认天气设备创建链路：

1. 家庭启用 `official-weather` 插件
2. 插件启用链路调用天气服务，检查当前家庭下是否已有默认天气设备
3. 如果没有，就自动创建一个标准 `Device` 和对应的 `DeviceBinding`
4. 默认天气设备的坐标不读 `city` 文本，而是调用 `001.3.1` 提供的家庭地区坐标解析能力
5. 能拿到经纬度时，按当前天气源刷新天气快照并写回统一实体
6. 拿不到经纬度时，设备进入 `pending_coordinate`，保留待补全状态，不乱报脏错误

这条链路里没有单独造一套“天气专用设备协议”。天气就是标准设备，只是 `platform = weather`，实体由 `capabilities.entities` 暴露。

## 多地区设备怎么建模

第一版多地区天气设备继续复用同一套天气设备模型：

- 默认家庭天气：`binding_type = default_household`
- 附加地区天气：`binding_type = region_node`

附加地区设备当前只支持从已有地区节点创建，要求这个节点本身已经有坐标。

这次明确没做：

- `custom_coordinate`
- 城市文本直查
- 任何绕开地区坐标体系的主链路

这样做的原因很简单：统一从经纬度出发，才能让默认家庭天气和附加地区天气走同一套刷新、缓存、实体和卡片链路。

## 天气源边界

当前支持 3 个天气源：

- `MET Norway`
  - 默认源
  - 免 key
  - 适合把默认主链路先跑通
- `OpenWeather`
  - 可选源
  - 用户自己填写 `OpenWeather API Key`
- `WeatherAPI`
  - 可选源
  - 用户自己填写 `WeatherAPI Key`

这次没有做中心化 SaaS 转发。请求直接从插件所在服务端发往天气供应商。

切源时有两个约束：

1. 设备模型和实体字段不变，只切换取数适配器
2. 切到需要 key 的来源时，如果 key 缺失，配置保存会直接报错，不让坏配置进主链路

## 实体字段

第一版统一暴露这些实体：

- `weather.condition`
- `weather.temperature`
- `weather.humidity`
- `weather.wind_speed`
- `weather.wind_direction`
- `weather.pressure`
- `weather.cloud_cover`
- `weather.precipitation_next_1h`
- `weather.forecast_6h`
- `weather.updated_at`

这些字段同时适用于默认家庭天气设备和附加地区天气设备。

## 仪表盘怎么选设备

这次没有给仪表盘布局项新增“卡片绑定 device_id”的专用协议。

实际做法是：

1. 首页聚合时，后端按天气设备数量动态产出多张天气卡
2. 默认家庭天气是一张卡，附加地区天气每个设备也是一张卡
3. 每张卡都有稳定的 `card_ref`
4. 用户在首页编辑模式里，通过现有的卡片显隐和排序能力，选择保留哪张天气卡、隐藏哪张天气卡

换句话说，“选择指定天气设备展示”在第一版里是通过“选择保留哪张天气卡”完成的，而不是靠新增卡片配置协议硬塞一个 `device_id` 字段。

这个实现边界的好处是：

- 复用现有仪表盘卡片体系
- 不改 `MemberDashboardLayoutItem` 结构
- 多设备场景天然支持一张或多张天气卡并存

## 卡片展示字段

天气卡当前展示的核心内容包括：

- 地区名
- 当前天气摘要
- 当前温度
- 湿度
- 风速
- 未来 1 小时降水
- 未来 6 小时摘要
- 最近更新时间

卡片异常态也直接展示：

- 无坐标
- 刷新失败
- 过期快照

## 缓存、刷新和降级

设备刷新策略：

- `ready` 且缓存未过期：复用最近一次成功快照
- 没有坐标：进入 `pending_coordinate`
- 请求失败但有旧快照：保留旧快照，标记 `stale`
- 请求失败且没有旧快照：进入 `error`

补充约束：

- 首页卡片读取当前设备状态，不会为了读卡片再偷偷触发一次刷新
- 这样可以避免“刚进入 stale，读卡片又把状态覆盖掉”的脏行为
- 真正的刷新仍然走插件启用、设备创建、手动刷新或后续调度链路

## 延期项

第一版明确延期：

- AQI
- 天气预警
- 日出日落
- `custom_coordinate`
- 城市文本主链路查询
- 复杂趋势图和地图化交互

这些东西不是做不了，而是现在加进来只会把适配器、实体和卡片一起搞脏。
