# 需求文档 - 官方天气插件

状态：Draft

## 简介

这不是做一个演示页面，而是做一个官方插件。官方插件的标准只有一个：默认装上去就能用，边界清楚，坏了知道怎么降级，后面还能扩展。

基于当前调研，这次天气插件采用以下路线：

- 默认天气源：`MET Norway`
- 可选天气源：`OpenWeather`、`WeatherAPI`
- 默认查询方式：经纬度
- 默认地区来源：当前家庭的解析坐标

原因也很现实：

- `MET Norway` 不需要 key，适合本地部署项目每户独立使用
- `OpenWeather` 和 `WeatherAPI` 适合作为用户自带 key 的增强选项
- 继续用城市名字查天气，会把当前地区体系重新搞回文本猜测老路

## 术语表

- **天气设备**：插件为某个地区生成的一个设备实例
- **默认天气设备**：跟随当前家庭地区的那一个天气设备
- **附加天气设备**：用户手动添加的额外地区天气设备
- **天气源**：具体天气供应商适配器，如 `MET Norway`
- **天气实体**：挂在天气设备上的结构化天气数据单元

## 范围说明

### In Scope

- 官方天气插件后端与插件配置
- 默认 `MET Norway` 适配器
- 可选 `OpenWeather`、`WeatherAPI` key 配置能力
- 当前家庭地区自动生成默认天气设备
- 用户手动添加更多地区天气设备
- 天气实体和仪表盘卡片

### Out of Scope

- 集中式代理服务和统一代管 API key
- 第一版的 AQI、天气预警、日出日落
- 自由输入城市名后自动猜测坐标的主流程
- 复杂地图交互和地理搜索产品化页面

## 需求

### 需求 1：官方插件默认装上就能查到当前家庭天气

**用户故事：** 作为家庭用户，我希望启用官方天气插件后，不填 key 也能看到当前家庭地区的天气。

#### 验收标准

1. WHEN 插件启用且家庭已有可用坐标 THEN System SHALL 自动创建或激活默认天气设备，并用默认天气源完成查询。
2. WHEN 插件启用但家庭没有可用坐标 THEN System SHALL 给出明确提示，告诉用户需要补地区坐标或确认家庭位置。
3. WHEN 默认天气源可用 THEN System SHALL 不要求用户先填写 API key 才能使用基础天气能力。

### 需求 2：天气源必须可插拔，默认免 key，可选 key 型供应商

**用户故事：** 作为系统管理员，我希望官方插件默认可用，同时也能在需要时换成别的天气源，而不是被单一供应商绑死。

#### 验收标准

1. WHEN 插件首次安装 THEN System SHALL 默认选择 `MET Norway` 作为天气源。
2. WHEN 用户切换到 `OpenWeather` 或 `WeatherAPI` THEN System SHALL 提供对应 key 配置入口，并在缺 key 时阻止启用该来源。
3. WHEN 后续新增天气源 THEN System SHALL 通过统一适配接口接入，而不是重写设备、实体和卡片逻辑。

### 需求 3：支持手动添加更多地区，每个地区都是一个设备

**用户故事：** 作为家庭用户，我希望除了当前家庭所在地区，还能单独添加别的城市或区县天气，并把它们当成独立设备管理。

#### 验收标准

1. WHEN 用户手动添加一个地区 THEN System SHALL 创建一个新的天气设备，并绑定该地区的坐标解析结果。
2. WHEN 用户添加多个地区 THEN System SHALL 为每个地区维护独立设备、独立刷新状态和独立天气实体。
3. WHEN 设备绑定的地区没有可用坐标 THEN System SHALL 阻止创建或将设备标记为待补全，而不是偷偷退回文本猜测。

### 需求 4：天气内容要以实体形式暴露

**用户故事：** 作为系统和前端开发者，我希望天气数据不是一坨 JSON，而是一组稳定实体，便于卡片、自动化和设备页复用。

#### 验收标准

1. WHEN 天气设备刷新成功 THEN System SHALL 产出一组固定的核心天气实体，至少包含天气状态、温度、湿度、风速、风向、气压、未来 1 小时降水和更新时间。
2. WHEN 默认天气源无法稳定提供某个字段 THEN System SHALL 不把该字段列为第一版必备实体。
3. WHEN 不同天气源返回字段差异存在 THEN System SHALL 通过统一映射输出同一套实体语义。

### 需求 5：插件要带仪表盘卡片，支持选择显示哪个地区

**用户故事：** 作为家庭用户，我希望在仪表盘上放一个天气卡片，并选择展示默认地区或我手动添加的某个地区。

#### 验收标准

1. WHEN 用户添加天气卡片 THEN System SHALL 允许选择一个天气设备作为卡片数据源。
2. WHEN 卡片绑定的天气设备刷新成功 THEN System SHALL 展示当前天气摘要、温度、天气图标、地区名和更新时间。
3. WHEN 卡片绑定的设备无坐标、刷新失败或天气源配置缺失 THEN System SHALL 在卡片上展示可理解的异常状态，而不是空白。

### 需求 6：刷新、缓存和失败降级要可控

**用户故事：** 作为本地部署项目维护者，我希望天气插件既不会把免费源刷爆，也不会因为一次失败把整个设备打死。

#### 验收标准

1. WHEN 同一个天气设备在缓存有效期内被重复读取 THEN System SHALL 复用最近一次成功结果，而不是每次都请求外部天气源。
2. WHEN 外部天气源短暂失败 THEN System SHALL 保留最近一次成功快照，并标记数据已过期或刷新失败。
3. WHEN 设备数量增加 THEN System SHALL 通过分设备刷新、限流和退避策略避免短时间内产生无意义的高频请求。

## 非功能需求

### 非功能需求 1：法务边界清楚

1. WHEN 使用默认天气源 THEN System SHALL 在插件文档中明确说明默认来源、使用边界和可替换方案。
2. WHEN 用户切换到 key 型天气源 THEN System SHALL 让 key 由用户自己配置和持有，不依赖中心化代管。

### 非功能需求 2：可维护性

1. WHEN 新增天气源 THEN System SHALL 只新增适配器和字段映射，不重写插件主流程。
2. WHEN 新增天气设备实体消费场景 THEN System SHALL 复用现有实体模型，而不是为卡片单独造一套天气协议。

### 非功能需求 3：性能与稳定性

1. WHEN 正常刷新天气 THEN System SHALL 在合理超时内返回结果或明确失败状态，避免后台任务长时间挂死。
2. WHEN 外部天气源返回不完整字段 THEN System SHALL 输出部分可用结果，并明确缺失字段，而不是整单失败。

## 成功定义

- 官方天气插件在家庭已有坐标时默认可用，不强制要求 key
- 当前家庭和手动添加地区都能以独立天气设备方式呈现
- 仪表盘卡片可以选择指定天气设备并稳定展示核心天气信息
- 第一版字段范围与延期项清楚，不承诺默认天气源给不了的能力
