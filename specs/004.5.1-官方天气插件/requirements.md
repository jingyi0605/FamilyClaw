# 需求文档 - 官方天气插件

状态：Draft

## 简介

这份需求文档现在只回答一件事：

在新的插件系统边界下，官方天气能力到底应该怎么存在。

结论已经很明确：

- 天气不是宿主内置业务模块
- 天气是官方 `integration` 插件
- 天气设备和附加地区设备必须走同一套设备模型
- 天气卡片只能消费标准设备和标准实体

## 前置依赖

天气插件的第一版依赖下面这些平台能力已经可用：

1. 宿主支持一等 `integration` 类型
2. 宿主支持 `integration_instance` 配置作用域
3. 宿主支持标准实体状态落库与读取
4. 仪表盘卡片支持按设备绑定数据源
5. `001.3.1` 的坐标解析能力可用

如果这些前提没满足，天气插件实现应视为阻塞，而不是继续把逻辑塞回宿主。

## 术语

- **天气实例**：某个家庭启用官方天气插件后创建的一次接入实例
- **默认天气设备**：自动跟随当前家庭坐标的天气设备
- **附加地区天气设备**：用户手动添加的额外地区天气设备
- **天气位置子项**：实例内部的一个天气位置配置单元，对外表现为一个天气设备
- **天气源**：具体天气供应商适配器，如 `MET Norway`

## 范围说明

### In Scope

- 官方天气插件作为 `integration` 接入
- 默认天气源 `MET Norway`
- 可选天气源 `OpenWeather`、`WeatherAPI`
- 默认天气实例自动创建
- 默认天气设备自动创建
- 手动添加附加地区天气设备
- 标准天气实体
- 仪表盘天气卡片
- 缓存、降级、错误状态

### Out of Scope

- AQI
- 天气预警
- 日出日落
- 中心化 SaaS 转发
- 基于 city 文本的主查询链路

## 需求 1：天气必须作为标准 integration 插件接入

**用户故事：** 作为平台维护者，我希望天气能力走统一插件接入链路，而不是继续在宿主里加专门天气模块。

### 验收标准

1. WHEN 官方天气能力上线 THEN System SHALL 以 `integration` 插件形式接入，而不是通过宿主专用服务暴露主链路。
2. WHEN 天气插件创建实例 THEN System SHALL 使用统一集成实例、设备、实体、卡片模型。
3. WHEN 后续新增别的天气源 THEN System SHALL 只扩展插件适配器，不重写宿主主链路。

## 需求 2：默认家庭天气必须启用即用

**用户故事：** 作为家庭用户，我希望启用天气插件后，不填 key 也能看到当前家庭地区的天气。

### 验收标准

1. WHEN 家庭启用官方天气插件且家庭坐标可用 THEN System SHALL 自动创建默认天气实例和默认天气设备。
2. WHEN 默认天气源为 `MET Norway` THEN System SHALL 不要求用户预先填写 API key。
3. WHEN 家庭坐标不可用 THEN System SHALL 明确提示坐标缺失，而不是退回 city 文本猜测。

## 需求 3：多地区天气必须和默认天气走同一套设备模型

**用户故事：** 作为家庭用户，我希望除了当前家庭地区，还能添加别的地区天气，而且这些地区天气都和默认天气一样可管理。

### 验收标准

1. WHEN 用户手动添加一个地区天气 THEN System SHALL 在同一天气实例下新增一个天气位置子项，并暴露为一个标准天气设备。
2. WHEN 用户添加多个地区天气 THEN System SHALL 为每个地区维护独立设备、独立刷新状态、独立卡片展示。
3. WHEN 设备使用的地区没有坐标 THEN System SHALL 阻止创建或将其标为待补全状态，不允许偷偷切回文本查询。

## 需求 4：天气源必须可插拔，默认源固定为 MET Norway

**用户故事：** 作为系统管理员，我希望默认源开箱即用，但后续可以换成用户自带 key 的天气源。

### 验收标准

1. WHEN 插件首次启用 THEN System SHALL 默认使用 `MET Norway`。
2. WHEN 用户切换到 `OpenWeather` 或 `WeatherAPI` THEN System SHALL 要求对应 key 配置完整。
3. WHEN 后续新增天气源 THEN System SHALL 通过统一适配接口接入，而不是改设备和实体模型。

## 需求 5：天气数据必须通过统一实体暴露

**用户故事：** 作为平台和前端开发者，我希望天气数据是稳定实体，而不是某个插件私有 JSON。

### 验收标准

1. WHEN 天气设备刷新成功 THEN System SHALL 至少暴露以下实体：
   - 天气状态
   - 温度
   - 湿度
   - 风速
   - 风向
   - 气压
   - 云量
   - 未来 1 小时降水
   - 未来 6 小时摘要
   - 更新时间
2. WHEN 不同天气源返回字段不同 THEN System SHALL 通过统一映射保持同一套实体语义。
3. WHEN 某个字段当前源稳定性不够 THEN System SHALL 允许该字段为空，但不能偷偷改语义。

## 需求 6：仪表盘卡片必须按设备选择，而不是按文本位置选择

**用户故事：** 作为家庭用户，我希望仪表盘卡片可以明确展示哪个天气设备，而不是依赖城市名猜测。

### 验收标准

1. WHEN 用户添加天气卡片 THEN System SHALL 允许按 `device_id` 选择天气设备。
2. WHEN 卡片绑定某个天气设备 THEN System SHALL 展示该设备当前摘要、温度、地区名和更新时间。
3. WHEN 卡片绑定设备缺数据、过期或出错 THEN System SHALL 展示可理解的异常状态。

## 需求 7：缓存、刷新、失败降级必须可控

**用户故事：** 作为本地部署维护者，我希望天气插件不会把上游免费源打爆，也不会因为一次失败把整张卡片打成空白。

### 验收标准

1. WHEN 同一设备在缓存有效期内被重复读取 THEN System SHALL 优先复用最近一次成功快照。
2. WHEN 上游天气源短暂失败 THEN System SHALL 保留最后一次成功结果，并将设备标记为 `stale` 或 `degraded`。
3. WHEN 上游持续失败或配置错误 THEN System SHALL 给出明确错误状态，不得伪装成正常数据。

## 成功定义

这份 Spec 成功，不是因为“能查天气”，而是因为它满足下面四条：

1. 天气作为标准插件接入，而不是宿主特例。
2. 默认天气和多地区天气走同一套模型。
3. 卡片、设备、实体、刷新链路全部统一。
4. 宿主不再为天气保留专门业务主链路。
