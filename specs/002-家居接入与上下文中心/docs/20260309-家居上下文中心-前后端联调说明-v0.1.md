## 1. 这份文档解决什么问题

别把联调写成废话。这里明确三件事：

1. 前端现在到底吃哪些后端接口。
2. 页面上哪些区域展示“已生效状态”，哪些区域只是“待保存草稿”。
3. 后端暂时不可用时，页面怎么退化，而不是直接瘫掉。

这份说明同时覆盖任务 `3.2` 的联调要求，也给任务 `3.3` 的收口检查提供边界说明与已知风险记录。

## 2. 当前联调范围

### 2.1 页面范围

- 管理台页面：`apps/admin-web/src/pages/ContextCenterPage.tsx`
- 路由入口：`/context-center`

### 2.2 后端接口范围

#### 家居上下文接口

- `GET /api/v1/context/overview?household_id={household_id}`
- `GET /api/v1/context/configs/{household_id}`
- `PUT /api/v1/context/configs/{household_id}`

#### 页面补充数据接口

- `GET /api/v1/members?household_id={household_id}`
- `GET /api/v1/rooms?household_id={household_id}`
- `GET /api/v1/devices?household_id={household_id}`
- `GET /api/v1/audit-logs?household_id={household_id}`

这些补充接口不是为了重复造轮子，而是为了：

- 给配置编辑区提供完整成员 / 房间选项
- 给房间卡片补设备明细
- 给“最近活动”区补审计线索

## 3. 前后端职责边界

### 3.1 `context overview` 的职责

`context overview` 负责提供**当前已生效的家庭上下文总览**，包括：

- 家庭模式、隐私模式、自动化等级
- 当前活跃成员
- 成员状态面板
- 房间占用与房间策略快照
- 设备汇总计数
- 风险 / 洞察列表
- 总览生成时间与是否降级

页面上半部分的仪表盘、成员状态面板、房间热区、关键洞察，**优先使用这个接口**。

### 3.2 `context configs` 的职责

`context configs` 负责提供和持久化**可编辑配置结构**，包括：

- 家庭级策略
- 成员状态草稿
- 房间策略草稿
- 配置版本、更新时间、更新人

页面下半部分配置区的编辑数据，**优先使用这个接口**。

### 3.3 `localStorage` 的职责

本地草稿现在只做两件事：

1. 后端 `context configs` 不可用时的兜底。
2. 保存成功后的浏览器侧备份快照，避免下次后端抖动时页面直接失忆。

它不是主存储，也不是长期真相来源。

## 4. 页面加载流程

当前页面加载时，会并发请求以下数据：

1. `context overview`
2. `context configs`
3. `members`
4. `rooms`
5. `devices`
6. `audit-logs`

### 4.1 展示层取数规则

- 有 `overview`：优先展示后端聚合结果
- 没有 `overview`：退回本地草稿 + 主数据推导

### 4.2 编辑层取数规则

- 有 `configs`：将后端配置结构标准化为前端 `draft`
- 没有 `configs`：从 `localStorage` 读取草稿，再按成员 / 房间主数据补齐默认值

### 4.3 主数据失败时的处理

- `members / rooms` 失败，但 `overview` 成功：前端会从 `overview` 合成简化成员 / 房间列表，保证页面还能编辑基础字段
- `devices` 失败：房间卡片仍能展示总览里的设备总数，但设备明细 chip 可能缺失
- `audit-logs` 失败：不影响上下文展示，只影响“最近活动”区

## 5. 配置保存流程

点击“保存到后端”后，前端执行以下顺序：

1. 将当前页面 `draft` 转成 `ContextConfigUpsertPayload`
2. 调用 `PUT /api/v1/context/configs/{household_id}`
3. 保存成功后：
   - 更新本地草稿备份
   - 重新拉取 `context overview`
   - 重新拉取 `audit-logs`
4. 如果保存失败：
   - 把当前草稿写入 `localStorage`
   - 页面提示“后端保存失败，已回退本地草稿”

这条链路很实用：

- 成功时，后端成为真相来源
- 失败时，用户至少不会丢配置

## 6. 当前页面上的“真实状态”与“编辑草稿”区别

这是联调时最容易搞混的地方，必须说清楚。

### 6.1 已生效状态

页面这些区域展示的是**后端当前状态**：

- 顶部 Hero 区
- 四张关键指标卡
- 关键洞察
- 成员状态面板
- 房间热区与设备状态
- 最近活动与同步线索

### 6.2 编辑草稿

页面这些区域展示的是**待保存配置**：

- 家庭级策略
- 成员状态草稿
- 房间策略草稿

所以当你在配置区改值时：

- 下半部分会立即变化
- 上半部分不会假装“已经生效”
- 只有保存成功并重新拉取 `overview` 后，上半部分才会更新

这不是缺陷，这是诚实。

## 7. 关键数据映射

### 7.1 后端配置 → 前端草稿

- 来源：`GET /context/configs/{household_id}`
- 转换函数：`createDraftFromContextConfig()`
- 目标结构：`ContextCenterDraft`

### 7.2 前端草稿 → 后端配置

- 来源：页面当前 `draft`
- 转换函数：`toContextConfigPayload()`
- 目标结构：`ContextConfigUpsertPayload`

### 7.3 后端总览 → 页面展示

- 成员面板：`overview.member_states`
- 房间热区：`overview.room_occupancy`
- 仪表盘设备计数：`overview.device_summary`
- 洞察卡片：`overview.insights`
- 活跃成员：`overview.active_member`

## 9. 已知限制与延期项

### 9.1 当前缓存还不是真 Redis

后端当前的 `context cache` 仍是进程内内存实现，不是真正的 Redis 热缓存。
这意味着：

- 当前阶段足够支撑联调
- 但还不适合多实例一致性场景

### 9.2 设备明细仍依赖 `devices` 主数据接口

`overview` 只提供房间级设备汇总计数，不提供每台设备明细。
所以：

- 房间卡片的总数可以只靠 `overview`
- 设备 chip 明细仍要靠 `GET /devices`

### 9.3 主数据接口失败时，前端只能做简化降级

如果 `members / rooms` 接口失败而只剩 `overview`：

- 页面仍可展示
- 也能维持基础编辑能力
- 但编辑区拿到的是简化成员 / 房间结构，不包含更丰富主数据字段

### 9.4 当前页面还没有直接打通设备动作按钮

本阶段已经完成了 `device-actions` 后端接口闭环，但 `/context-center` 页面目前仍以展示与配置为主，未在页面中直接放设备动作执行按钮。
这不是 bug，是当前阶段范围控制。

## 10. 收口判断

### 【核心判断】

✅ 值得做：前端已经从“纯本地原型”切到了“后端真接口 + 本地兜底”，这才是能继续支撑问答、提醒和场景编排的底座。

### 【关键洞察】

- 数据结构：`overview` 负责已生效状态，`configs` 负责编辑结构，职责清楚了，页面就不会一团糟。
- 复杂度：不重写 UI，只替换数据源，这是对的。重写页面只会制造垃圾。
- 风险点：缓存还不是 Redis、设备明细还依赖主数据接口，这两个点已经被明确记录，不再假装不存在。

### 【结论】

就当前 Spec 002 的范围来说，前后端边界、联调路径、降级策略和已知风险都已经落地，可进入最终检查点收口。
