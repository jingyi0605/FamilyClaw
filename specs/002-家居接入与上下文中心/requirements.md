# 需求文档 - 家居接入与上下文中心

状态：Draft

## 简介

`Spec 001` 解决的是家庭主数据问题，但那只是地基，不是房子。

真正让系统像“家庭 AI 中枢”而不是“另一个设备后台”的关键，在于把下面这条链路打通：

**设备事件 → 在家状态 → 活跃成员 → 房间占用 → 家庭上下文 → 设备动作 / 问答 / 提醒 / 广播**

结合 `docs/` 下的全局设计，本 Spec 聚焦第二阶段最该做的东西：

- 设备接入不再只是同步清单，还要能承接状态输入和动作执行
- 系统要能维护“谁在家、谁在哪个房间、当前谁最值得服务”
- 管理员要能在一个页面里看懂当前家庭状态，并能快速调整上下文策略

如果这一层还是空的，后面的问答、广播、场景编排全都会沦为拍脑袋。

---

## 范围边界

本 Spec 包含：

- `Home Assistant` 设备接入的可用化增强
- 在家事件接入、成员状态聚合、房间占用缓存
- 家庭上下文总览查询
- 基础设备动作执行入口
- 管理台仪表盘与配置界面

本 Spec 不包含：

- 完整人脸识别算法实现
- 完整声纹识别算法实现
- 深度小米生态适配细节
- 复杂场景编排规则引擎
- 记忆卡生成与家庭问答完整闭环

---

## 术语表

- **设备实体**：来自 `Home Assistant` 的设备或实体对象。
- **在家事件（presence event）**：来自门锁、摄像头、蓝牙、传感器、语音终端等来源的原始状态事件。
- **成员在家快照**：系统对某个成员当前 `home / away / unknown` 状态的最新判断结果。
- **房间占用**：某个房间当前是否有人、是否多人、是否应触发房间级联动。
- **活跃成员**：当前最应被系统优先服务的成员。
- **家庭上下文总览**：面向前端展示的聚合结果，包含家庭模式、成员状态、房间热区、设备健康与关键风险提示。
- **快路径执行**：不进入复杂推理链路，直接对高频控制命令执行设备动作的路径。

## 需求

### 需求 1：设备接入与实体归一化

**用户故事：** 作为系统，我希望稳定接入 `Home Assistant` 设备并归一化成本地设备模型，以便后续状态感知和动作执行都建立在统一对象之上。

#### 验收标准

1. WHEN 管理员触发设备同步 THEN System SHALL 从 `Home Assistant` 拉取实体并落到本地设备模型。
2. WHEN 同步到首期支持类型的实体 THEN System SHALL 写入或更新本地 `devices` 与绑定关系。
3. WHEN 遇到不支持或字段不完整的实体 THEN System SHALL 跳过该实体并在结果摘要中给出原因。
4. WHEN 同步结束 THEN System SHALL 记录审计日志并返回创建、更新、跳过、失败数量。

### 需求 2：在家事件接入

**用户故事：** 作为系统，我希望接收门锁、摄像头、蓝牙、传感器、语音等来源的在家事件，以便后续推断成员状态。

#### 验收标准

1. WHEN 外部适配器上报一条在家事件 THEN System SHALL 校验 `household_id`、来源、发生时间与原始摘要。
2. WHEN 事件命中某个成员 THEN System SHALL 记录成员 ID、房间 ID、来源类型和置信度。
3. WHEN 事件不包含可识别成员 THEN System SHALL 允许以匿名事件写入，但不得伪造成员身份。
4. WHEN 事件格式非法或缺少关键字段 THEN System SHALL 拒绝写入并返回明确错误信息。

### 需求 3：成员在家状态聚合

**用户故事：** 作为系统，我希望把原始在家事件聚合成成员级快照，以便快速回答“谁在家”“谁在哪”。

#### 验收标准

1. WHEN 新的在家事件进入系统 THEN System SHALL 更新对应成员的最新在家快照。
2. WHEN 多个来源对同一成员给出冲突状态 THEN System SHALL 基于置信度与时间新鲜度进行聚合。
3. WHEN 置信度不足以支持敏感结论 THEN System SHALL 将状态降级为 `unknown` 或匿名模式。
4. WHEN 成员状态更新 THEN System SHALL 输出可被查询和缓存的标准化快照结果。

### 需求 4：房间占用与活跃成员缓存

**用户故事：** 作为系统，我希望维护房间占用和当前活跃成员缓存，以便提醒、广播和设备联动能低延迟使用上下文。

#### 验收标准

1. WHEN 成员在家快照发生变化 THEN System SHALL 重新计算受影响房间的占用结果。
2. WHEN 多名成员同时在场 THEN System SHALL 区分公共空间占用与当前优先服务对象。
3. WHEN 上下文结果生成 THEN System SHALL 写入热缓存，供后续问答、广播与场景执行直接读取。
4. WHEN 缓存暂时不可用 THEN System SHALL 允许从数据库快照降级查询，而不是直接返回空结果。

### 需求 5：家庭上下文总览查询

**用户故事：** 作为管理员，我希望获得一个统一的家庭上下文总览接口，以便前端仪表盘可以一次性展示关键状态。

#### 验收标准

1. WHEN 前端请求家庭上下文总览 THEN System SHALL 返回家庭模式、活跃成员、成员快照、房间占用、设备健康和关键提示。
2. WHEN 某些子数据源不可用 THEN System SHALL 尽可能返回可用部分，并明确标记缺失字段或降级状态。
3. WHEN 请求家庭上下文总览 THEN System SHALL 保证返回的数据只属于当前家庭。
4. WHEN 家庭无成员、无房间或无设备 THEN System SHALL 返回空集合而不是报错。

### 需求 6：基础设备动作执行

**用户故事：** 作为系统，我希望通过统一入口调用 `Home Assistant` 执行基础设备动作，以便支撑快路径控制和场景联动。

#### 验收标准

1. WHEN 调用基础设备动作接口 THEN System SHALL 校验目标设备是否属于当前家庭且具备可控能力。
2. WHEN 请求为灯光、空调、窗帘、音箱等首期支持能力 THEN System SHALL 转换为对应的 `Home Assistant` 服务调用。
3. WHEN 请求涉及敏感设备或高风险动作 THEN System SHALL 执行权限与风险校验。
4. WHEN 动作执行成功或失败 THEN System SHALL 写审计日志并返回明确结果。

### 需求 7：管理台仪表盘与配置界面

**用户故事：** 作为家庭管理员，我希望在一个友好的页面里查看家庭状态、成员状态和关键指标，并配置上下文策略，以便我能看懂系统现在怎么想、必要时能马上纠偏。

#### 验收标准

1. WHEN 管理员进入家居上下文页面 THEN Frontend SHALL 展示家庭状态总览、成员状态卡片、房间热区、设备健康和关键风险提示。
2. WHEN 管理员切换家庭 THEN Frontend SHALL 刷新对应家庭的数据与配置视图。
3. WHEN 当前阶段后端配置接口尚未完成 THEN Frontend SHALL 允许使用本地持久化保存演示配置，不得阻塞页面交付。
4. WHEN 管理员修改家庭模式、隐私模式、成员在家状态或房间策略 THEN Frontend SHALL 立即反映到仪表盘预览并允许保存。

### 需求 8：安全、审计与降级

**用户故事：** 作为系统所有者，我希望上下文系统在不确定、异常或高风险情况下有明确边界，以便系统不会因为“猜错”就做错事。

#### 验收标准

1. WHEN 成员识别置信度低于阈值 THEN System SHALL 禁止敏感回答和高风险动作自动执行。
2. WHEN 访客模式开启或敏感房间被标记为隐私保护 THEN System SHALL 收紧可见信息和动作范围。
3. WHEN 上下文配置被修改 THEN System SHALL 记录可追踪的审计日志或配置变更记录。
4. WHEN 外部集成故障、缓存失效或状态不一致 THEN System SHALL 进入降级模式，并向前端暴露明确状态而不是静默失败。

### 需求 9：性能与可观察性

**用户故事：** 作为开发与运维人员，我希望上下文链路具有基本性能和可观察性，以便知道它到底有没有工作而不是靠猜。

#### 验收标准

1. WHEN 查询家庭上下文总览 THEN System SHALL 优先命中热缓存，并在可用条件下以低延迟返回聚合结果。
2. WHEN 在家事件被写入 THEN System SHALL 记录事件处理、快照更新与缓存刷新结果。
3. WHEN 设备动作执行 THEN System SHALL 记录目标设备、执行结果、失败原因与耗时。
4. WHEN 核心链路出现连续失败 THEN System SHALL 能通过日志或指标快速定位失败点。
