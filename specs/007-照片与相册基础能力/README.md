# Spec 007 - 照片与相册基础能力

状态：Draft

## 这份 Spec 解决什么问题

现在项目里已经有几块能复用的地基，但照片这条主链还是断的：

- 后端已经有 `conversation`、`memory`、`plugin_job`、`channel` 这些正式模块，但对话还只会收文本，不会正式接照片。
- 项目技术愿景里已经明确要做 `Photo/Immich Adapter`、人脸识别、照片记忆和相册回忆，但仓库里还没有照片域模型和接入主链。
- 记忆中心已经能沉淀长期记忆，但还没有把“这张照片里是谁、发生了什么、能生成什么故事”写成正式数据。
- AI 网关已经预留了 `vision` 能力，但没有和照片资产、会话附件、家庭成员绑定打通。

结果就是：如果现在直接往对话里塞图片识别，只会变成几段临时逻辑。识别结果没地方落，人物绑定不稳定，隐私边界也会烂掉。

这份 Spec 要做的事很直接：在当前模块化单体架构上，补一条正式的照片主链，让系统能接照片、识别人、把照片写进家庭记忆，并进一步支撑家庭朋友圈故事和照片剧情时间线。

这里先把架构边界定死：

- `Immich` 管原图、缩略图、基础相册和基础人物聚类。
- `FamilyClaw` 管照片业务索引、家庭成员绑定、对话附件、故事、时间线、权限和审计。
- 也就是说，这不是“把照片能力全塞进本项目”，也不是“本项目只透传 `Immich` 接口”。这是正式的混合架构。

## 当前架构判断

基于仓库现状，这个能力不该另起一个独立微服务，第一版就按现有项目结构落：

- 后端继续用 `FastAPI + SQLAlchemy + Alembic` 的模块化单体。
- 对话入口继续复用 `conversation` 主链，不再额外造一套“图片问答接口”。
- 异步导入、批量同步和重分析继续复用现有 `plugin_job`，但照片领域规则必须落在本项目自己的 `photo` 模块里。
- 长期结果继续写进现有 `memory_cards`，不要再造第二套故事库和时间线库。
- 外部照片底座优先集成 `Immich`，但本项目必须保留正式 `photo_assets` 等业务索引，不能只靠外部 ID 活着。
- 人脸识别能力走“可替换 provider”思路：默认优先对接 `Immich` 的人物/人脸能力，必要时补 `CompreFace` 作为增强识别后端。

## 这次要做到什么

- 补正式的照片资产模型、成员关联模型和分析结果模型。
- 支持 `user-web` 和通讯通道把照片送进对话主链。
- 支持把照片同步到 `Immich`，并把 `Immich` 的人物聚类结果收口成家庭成员绑定关系。
- 支持图片内容分析、人脸识别、照片记忆写回。
- 给家庭朋友圈故事和照片剧情时间线打基础，但先把数据主链做扎实。

## 这次明确不做什么

- 不自研照片云盘，不替代 `Immich` 的原始文件管理。
- 不做完整视频理解和视频时间轴，第一版只覆盖静态照片，视频只留后续扩展口。
- 不做公开社交分享平台，家庭朋友圈只做家庭内故事内容。
- 不做全自动高风险识别决策，低置信度的人脸结果必须允许人工确认。
- 不做复杂多租户媒体对象存储平台，第一版保持家庭级、项目内可维护的结构。
