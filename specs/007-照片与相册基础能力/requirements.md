# 需求文档 - 照片与相册基础能力

状态：Draft

## 简介

这份需求文档只解决一件事：把“照片怎么进入系统、怎么认人、怎么变成家庭记忆和故事”说清楚。

当前项目已经有对话、插件、记忆和通讯通道几条主链，但照片还是空档。用户现在要的也不是“做一个新相册 App”，而是把照片变成家庭 AI 能理解、能检索、能讲故事的一等数据。

这里的核心架构前提也要先说清楚：

- `Immich` 继续做照片文件底座。
- 本项目必须保存正式照片业务索引，不能只拿 `Immich` 的接口现查现拼。
- 家庭成员绑定、故事、时间线、权限和审计都归本项目负责。

这次能力主要服务下面几类人：

- 家庭管理员：希望把相册底座接进来，把照片和成员关系管清楚。
- 普通家庭成员：希望在对话里直接发照片提问，得到是谁、发生了什么、这张照片值不值得记住。
- 儿童和老人家属：希望照片能自动形成成长记录、家庭回忆和时间线。
- 后续功能模块：家庭记忆、家庭问答、家人圈、时间线都需要复用这批正式照片数据。

成功后的核心收益很直接：

- 照片不再只是附件，而是正式家庭资产。
- 对话里的照片能直接进入识别和记忆主链。
- 人脸识别结果能和家庭成员中心稳定关联。
- 后续故事生成和时间线不需要再重造底层数据。

## 术语表

- **System**：当前 FamilyClaw 项目。
- **照片资产（Photo Asset）**：系统里一张正式可追踪的照片业务记录，包含来源、时间、权限、外部引用和分析结果；它不是原图文件本身，而是本项目自己的业务索引。
- **照片接入（Photo Intake）**：照片从 `user-web`、通讯通道或 `Immich` 进入系统并生成正式资产的过程。
- **人脸组（Face Group）**：外部识别系统或本系统聚合出来的一组“看起来是同一个人”的人脸集合。
- **成员绑定（Member Binding）**：把人脸组或单张照片里的识别结果正式关联到家庭成员。
- **照片故事（Photo Story）**：基于一张或一组照片生成的家庭内可读故事内容。
- **照片时间线（Photo Timeline）**：按时间整理的照片事件流，用来回顾成长、旅行、纪念日等过程。

## 范围说明

### In Scope

- 照片资产接入、落库和来源追踪。
- 对话中发送照片并触发图片分析。
- `Immich` 照片同步和外部人物聚类接入。
- 人脸结果与家庭成员的关联、确认和纠错。
- 照片内容分析写回家庭记忆。
- 家人圈故事和照片时间线的基础数据与生成入口。
- 照片权限、隐私和审计边界。

### Out of Scope

- 完整视频分析、视频剪辑和视频剧情线。
- 自研对象存储、NAS 文件管理器或照片云盘替代品。
- 面向外部社交平台的一键分享和公开社交互动。
- 无确认的高风险自动人脸结论落库。

## 需求

### 需求 1：统一照片资产接入

**用户故事：** 作为家庭管理员或家庭成员，我希望照片无论来自网页上传、聊天通道还是 `Immich` 同步，都能进入本项目统一的正式资产模型，以便后续识别、检索和记忆沉淀都走同一条链。

#### 验收标准

1. WHEN 系统收到一张新照片 THEN System SHALL 创建一条正式的照片资产记录，并保留来源、外部资产引用、时间和所属家庭信息。
2. WHEN 同一张照片被重复导入 THEN System SHALL 通过稳定的去重键或内容摘要避免生成重复资产。
3. WHEN 外部照片源不可用或同步失败 THEN System SHALL 保留失败状态和错误原因，而不是静默丢失。

### 需求 2：对话中发送照片并触发分析

**用户故事：** 作为家庭成员，我希望在对话中直接发送照片提问，以便系统能结合图片内容和家庭上下文回答问题。

#### 验收标准

1. WHEN 用户在 `user-web` 或通讯通道发送照片 THEN System SHALL 把这张照片挂到正式会话消息，而不是只当临时文件处理。
2. WHEN 会话消息包含照片 THEN System SHALL 能触发图片分析，并把分析结果回写到会话回复和后续资产记录。
3. WHEN 图片分析超时、失败或能力不可用 THEN System SHALL 给出清楚的降级反馈，并保留会话主链可继续使用。

### 需求 3：人脸识别与家庭成员关联

**用户故事：** 作为家庭管理员，我希望系统能把照片里的人脸和家庭成员稳定关联，以便后续问答、回忆、故事和时间线都能按成员工作。

#### 验收标准

1. WHEN 照片同步或上传后触发人脸识别 THEN System SHALL 保存每个识别结果的来源、人脸组、置信度和当前绑定状态。
2. WHEN 某个人脸组被人工确认属于某个家庭成员 THEN System SHALL 更新后续照片中的成员关联结果，并支持回写纠错记录。
3. WHEN 识别置信度低或结果冲突 THEN System SHALL 不自动写成确定成员，而是标记为待确认或候选结果。

### 需求 4：照片内容分析写回家庭记忆

**用户故事：** 作为家庭成员，我希望系统不只知道照片里有什么，还能把有价值的照片沉淀进家庭记忆，以便后续问答和回顾能真正用上。

#### 验收标准

1. WHEN 图片分析识别出明确人物、事件、地点或成长节点 THEN System SHALL 能生成或更新对应的记忆事件与记忆卡。
2. WHEN 多张照片属于同一事件或同一时间段 THEN System SHALL 能把它们归到同一个事件上下文，而不是每张都写一张重复记忆。
3. WHEN 用户纠正成员或事件信息 THEN System SHALL 同步修正后续记忆结果，并保留修订痕迹。

### 需求 5：家人圈故事生成

**用户故事：** 作为家庭成员，我希望系统能基于一组照片生成带 AI 润色的家庭故事，以便家庭内部回顾、分享和保存纪念内容。

#### 验收标准

1. WHEN 用户选择一张或一组照片生成故事 THEN System SHALL 基于照片分析结果、成员关系和时间信息生成家庭内可读的故事草稿。
2. WHEN 生成故事时涉及儿童、健康或敏感照片 THEN System SHALL 先执行权限判断和脱敏规则。
3. WHEN 故事生成完成 THEN System SHALL 把结果保存为可追踪内容，并能关联回照片资产和家庭记忆。

### 需求 6：照片剧情时间线

**用户故事：** 作为家庭成员，我希望系统能把照片按时间、人物和事件整理成剧情时间线，以便查看某个成员成长过程或某次家庭活动的完整脉络。

#### 验收标准

1. WHEN 用户按成员、事件或时间范围查询 THEN System SHALL 返回结构化的照片时间线结果。
2. WHEN 同一事件跨多张照片或多天持续 THEN System SHALL 能把这些照片挂到同一条时间线上，而不是只按单张照片平铺。
3. WHEN 某些照片没有足够时间或人物信息 THEN System SHALL 允许它们以“待补充”状态存在，而不是直接丢弃。

### 需求 7：照片权限、隐私与审计

**用户故事：** 作为家庭管理员，我希望照片能力从第一天起就带权限和审计，不要等数据满天飞以后再补洞。

#### 验收标准

1. WHEN 用户访问照片、人脸绑定、故事或时间线 THEN System SHALL 按家庭成员权限和照片隐私级别做过滤。
2. WHEN 用户进行成员绑定、故事生成、敏感照片查看或纠错 THEN System SHALL 记录正式审计日志。
3. WHEN 外部视觉服务不满足当前隐私策略 THEN System SHALL 拒绝把敏感图片送去外部分析，或走本地优先降级路径。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 用户在对话里发送单张普通照片 THEN System SHALL 在 1 秒内返回“已收到，正在分析”的确认反馈。
2. WHEN 系统执行单张照片基础分析 THEN System SHALL 在可用依赖正常时于 3~8 秒内给出首个可见结果或明确降级提示。

### 非功能需求 2：可靠性

1. WHEN `Immich`、人脸识别服务或视觉模型异常 THEN System SHALL 保留照片资产主记录，并把外部能力状态标记为失败或降级。
2. WHEN 照片导入、分析或记忆写回任务重复执行 THEN System SHALL 通过幂等键和去重策略避免重复落库。

### 非功能需求 3：可维护性

1. WHEN 后续要替换人脸识别服务或补视频能力 THEN System SHALL 允许在 provider 层替换，不要求推翻照片主数据结构。
2. WHEN 排查某张照片为什么没有识别人或没有写成记忆 THEN System SHALL 能通过资产、任务、分析记录和审计日志串起完整链路。

## 成功定义

- 用户可以在对话里发照片并拿到正式分析结果，不再靠临时附件逻辑。
- 家庭管理员可以把 `Immich` 里的照片同步进系统，并把人物正式绑定到家庭成员。
- 至少一条“照片 -> 成员 -> 记忆 -> 故事/时间线”主链能完整跑通。
- 低置信度识别、外部依赖失败、敏感照片权限三类关键异常都有正式收口。
