# 需求文档 - 小爱音箱插件界面与声纹管理迁移

状态：Draft

## 简介

`005.7` 已经把小爱音箱的发现、实例、绑定、控制主链往正式插件体系里迁，但设备详情入口还没有同步跟上。

当前最糟糕的地方不是“没有页面”，而是边界错了：

- 小爱专属的“语音接管策略”还写在通用设备模型里
- 通用的“声纹管理”又和小爱专属设置混在同一个弹层里
- 宿主前端还在直接认识“小爱这个插件应该显示什么”，而不是消费正式插件能力

这会直接带来三个问题：

1. 插件化只完成了一半，前端设备详情仍然是特例堆
2. 后面如果再接别的语音终端，宿主还得继续复制“小爱专用”逻辑
3. `speaker` 类型和“支持语音终端/声纹”的能力被混为一谈，普通音箱也可能被误判

这次的目标很明确：

- 把“小爱专属配置”收口为插件提供的设备详情页签
- 把“声纹管理”收口为平台提供的通用设备详情页签
- 给当前插件体系补齐设备级扩展点和设备级配置的正式落点
- 用兼容迁移的方式完成收口，不破坏现有语音链路

## 术语表

- **System**：`apps/api-server`、`apps/user-app` 以及它们之间的正式接口
- **设备详情页签**：用户在设备详情弹层里看到的标签页，例如“语音接管”“声纹管理”
- **插件专属页签**：只对某个插件或某类设备生效，由插件声明、宿主渲染的设备详情页签
- **通用页签**：由平台直接提供，面向具备某种平台能力的设备详情页签
- **语音终端能力**：设备具备接入语音会话、维护终端身份、支持声纹与会话路由的能力，不等于所有 `speaker` 都有
- **设备级插件配置**：配置作用范围是某一台设备，不是整家庭，也不是整条通道账号
- **兼容字段**：现有仍被运行链路读取的旧 `Device` 字段，迁移期间可以保留，但不再作为长期扩展边界

## 范围说明

### In Scope

- 定义小爱音箱设备详情在插件化体系下的页面职责边界
- 为插件体系设计正式的设备详情页签扩展点
- 为插件体系设计正式的设备级配置落点
- 把小爱“语音接管”界面收口到插件专属页签
- 把“声纹管理”界面收口到平台通用页签
- 明确通用页签的显示条件，避免把所有 `speaker` 都误判为语音终端
- 给出现有 `Device` 旧字段的兼容迁移策略
- 给出前后端实施任务和验证要求

### Out of Scope

- 这次不做插件前端 bundle 动态加载或沙箱执行
- 这次不重做声纹建档、识别、降级算法
- 这次不扩展新的小爱媒体控制协议
- 这次不做设备详情大改版或新的设计系统
- 这次不把平台通用 `speaker` 协议和声纹业务整个迁到插件内部

## 需求

### 需求 1：系统必须把设备详情里的“小爱专属配置”和“平台通用能力”拆开

**用户故事：** 作为平台维护者，我希望设备详情里的页面职责是清楚的，这样插件负责插件该负责的东西，平台负责平台该负责的东西，而不是继续把两者混在一页里。

#### 验收标准

1. WHEN 用户打开一台小爱音箱设备详情 THEN System SHALL 把“小爱语音接管”显示为插件专属页签，而不是写死在通用设备详情主体里。
2. WHEN 用户打开一台支持语音终端能力的设备详情 THEN System SHALL 把“声纹管理”显示为平台通用页签，而不是依赖小爱专属组件才能出现。
3. WHEN 某项能力属于平台通用能力 THEN System SHALL 不要求每个插件各自实现一套重复 UI 和业务逻辑。

### 需求 2：系统必须提供正式的设备详情插件页签扩展点

**用户故事：** 作为插件体系维护者，我希望插件能正式声明“我要在设备详情挂一个页签”，而不是每次都改宿主前端硬编码判断。

#### 验收标准

1. WHEN 插件需要在设备详情展示专属页签 THEN System SHALL 允许插件通过 manifest 或等价正式协议声明该页签。
2. WHEN 宿主前端渲染设备详情 THEN System SHALL 基于正式声明决定显示哪些插件页签，而不是按插件 id 写死条件分支。
3. WHEN 设备不满足插件页签的适用条件 THEN System SHALL 不显示该页签，也不显示空壳入口。
4. WHEN 后续新增别的语音终端插件 THEN System SHALL 不需要复制“小爱专用分支”才能接入同一套设备详情扩展点。

### 需求 3：系统必须提供正式的设备级插件配置落点

**用户故事：** 作为后端开发者，我希望插件专属的设备配置有正式存储位置，而不是继续把字段塞进通用 `Device` 模型里。

#### 验收标准

1. WHEN 插件需要保存某台设备的专属配置 THEN System SHALL 提供正式的设备级配置作用域或等价正式模型承载该配置。
2. WHEN 宿主读取插件专属设备页签数据 THEN System SHALL 通过正式配置协议读取，而不是直接从通用 `Device` 字段拼装。
3. WHEN 插件保存设备级配置 THEN System SHALL 保持字段校验、错误语义和 secret 处理规则与正式插件配置体系一致。
4. WHEN 系统迁移现有小爱语音接管配置 THEN System SHALL 有兼容读取与回写策略，不要求一次性砍掉旧字段。

### 需求 4：小爱语音接管页签必须由插件体系提供，但首版实现不能过度设计

**用户故事：** 作为开发者，我希望先用最简单稳定的方式把小爱专属配置收回插件体系，而不是为了“插件 UI”一下子造出远程前端加载平台。

#### 验收标准

1. WHEN 首版实现小爱语音接管页签 THEN System SHALL 允许宿主用统一 renderer 渲染插件声明的配置表单，而不是要求插件自己下发前端 bundle。
2. WHEN 小爱页签展示“默认接管全部语音请求”和“响应前缀” THEN System SHALL 继续使用正式配置协议字段，而不是保留额外宿主硬编码字段定义。
3. WHEN 用户保存小爱语音接管配置 THEN System SHALL 通过正式插件设备配置入口保存，并在必要时同步兼容字段。
4. WHEN 插件被禁用 THEN System SHALL 按统一插件禁用规则保留查看和配置能力，但禁止任何执行型动作伪装成配置动作。

### 需求 5：声纹管理页签必须是平台通用能力，并且显示条件必须看能力而不是只看设备类型

**用户故事：** 作为平台架构维护者，我希望“声纹管理”只出现给真正支持语音终端和声纹的设备，而不是所有 `speaker` 都跟着显示一个错误入口。

#### 验收标准

1. WHEN 设备只满足 `device_type = speaker` 但不具备语音终端能力 THEN System SHALL 不显示“声纹管理”页签。
2. WHEN 设备具备正式的语音终端能力标记 THEN System SHALL 显示“声纹管理”页签，并继续复用平台现有声纹链路。
3. WHEN 小爱音箱完成插件化迁移 THEN System SHALL 仍然通过平台现有声纹 summary、建档和更新能力工作，而不是把声纹业务复制到插件内部。
4. WHEN 后续新增其他语音终端插件 THEN System SHALL 只要设备能力标记满足条件，就能复用同一个通用声纹页签。

### 需求 6：迁移必须保持现有语音链路可用，不允许一刀切砍断旧字段

**用户故事：** 作为维护线上稳定性的开发者，我希望在新配置协议落地时，现有语音接管、声纹汇总和终端绑定链路还能继续工作，而不是被一次性大拆。

#### 验收标准

1. WHEN 新的设备级插件配置尚未把所有运行链路切完 THEN System SHALL 允许“新配置优先，旧字段兜底”的兼容读取方式。
2. WHEN 语音终端绑定、声纹汇总或其他现有服务仍依赖旧 `Device` 字段 THEN System SHALL 明确这些兼容依赖和退出条件。
3. WHEN 迁移过程发生回滚或局部失败 THEN System SHALL 不导致已有小爱设备失去语音接管或声纹管理能力。
4. WHEN 兼容层最终可以移除 THEN System SHALL 在任务和文档里明确删除步骤，而不是让兼容层永久存活。

### 需求 7：系统必须删除宿主里与新边界冲突的硬编码

**用户故事：** 作为后续接手的人，我希望仓库里只剩一套明确边界，而不是既有“通用 speaker 弹层”又有“小爱专属弹层”继续打架。

#### 验收标准

1. WHEN 新的插件页签和通用页签主链落地 THEN System SHALL 删除宿主前端里专门为小爱语音接管写死的设备详情分支。
2. WHEN 系统判断是否显示声纹管理页签 THEN System SHALL 不再只靠 `device_type == speaker` 或 `vendor == xiaomi` 这类粗暴条件。
3. WHEN 开发者搜索小爱设备详情主链 THEN System SHALL 能看到正式插件扩展点和通用能力判断，而不是继续依赖历史特例组件。
4. WHEN 保留少量兼容代码确有必要 THEN System SHALL 明确写出保留原因和删除时机。

## 非功能需求

### 非功能需求 1：架构一致性

1. WHEN 宿主新增设备详情页签能力 THEN System SHALL 优先复用现有插件配置协议和宿主页渲染能力，而不是新造第二套插件 UI 协议。
2. WHEN 后续插件继续接入设备详情页签 THEN System SHALL 只通过正式声明和能力判断扩展，而不是继续新增插件 id 白名单。

### 非功能需求 2：可靠性

1. WHEN 新设备级插件配置保存失败 THEN System SHALL 只影响当前插件页签，不影响设备详情里的其他通用能力。
2. WHEN 兼容字段与新配置短暂并存 THEN System SHALL 保持读写顺序清楚可追踪，避免出现一半读新、一半读旧的随机行为。

### 非功能需求 3：可维护性

1. WHEN 后续开发者排查设备详情行为 THEN System SHALL 能清楚区分“插件专属页签”“平台通用页签”“兼容字段”三层职责。
2. WHEN 兼容迁移完成 THEN System SHALL 能明确删除旧字段和旧分支，不把过渡代码永久留在主链。

## 成功定义

- 小爱专属“语音接管”不再直接写死在宿主通用设备模型和组件里
- “声纹管理”成为平台通用页签，而不是小爱专属页签的一部分
- 插件体系具备正式的设备详情页签扩展点和设备级配置落点
- 通用页签显示条件基于设备能力而不是只看 `speaker` 类型
- 现有语音链路在迁移期间不中断，兼容层有明确退出路径
