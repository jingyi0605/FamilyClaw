# Spec 005.7.1 - 小爱音箱插件界面与声纹管理迁移

状态：Draft

## 这份 Spec 要解决什么

`005.7` 已经把“小爱音箱必须回到正式插件主链”这件事钉死了，但设备控制界面的边界还没收口。

现在仓库里的真实情况是：

- 小爱专属的“默认接管全部语音请求 / 只响应哪些前缀”还写在通用 `Device` 字段和宿主前端组件里
- 通用的“声纹管理 / 是否启用声纹识别”也和小爱专属设置混在同一个弹层里
- 插件配置协议只有 `plugin` 和 `channel_account` 两种正式作用域，还没有“设备详情里的插件页签”这个正式挂点
- 前端如果继续按 `speaker` 类型粗暴判断，很容易把普通音箱也误判成支持声纹或语音终端能力

一句话说清：

> 运行链路已经插件化了，但设备详情页还停在“宿主硬编码 + 小爱特例”阶段。

## 这次要做到什么

这次不重写整套页面，也不搞远程前端加载。只做下面四件事：

1. 把“小爱专属配置”正式定义为插件提供的设备详情页签
2. 把“声纹管理”正式定义为平台提供的通用 `speaker/voice-terminal` 页签
3. 给插件体系补齐设备详情扩展点和设备级配置落点
4. 提供一条不砸现有语音链路的迁移路线

## 核心判断

✅ 值得做，而且必须现在做。

原因很简单：

- 再继续把小爱专属设置塞进 `Device` 模型，后面每来一个语音终端都要再复制一遍
- 再继续把声纹入口和小爱专属入口绑死，插件化就只是后端换了名字，前端还是特例堆
- 当前正式协议缺少设备级扩展点，这不是“页面没写”，是模型还没立住

## 和已有 Spec 的关系

- `005.7-小爱音箱全插件化迁移`
  - 负责发现、实例、绑定、控制主链回到正式插件模型
  - `005.7.1` 负责设备详情页签和配置入口怎么跟着回到正式模型

- `005.3.1-声纹管理前端实现`
  - 负责把声纹管理做成用户可用的前端能力
  - `005.7.1` 不推翻这件事，只重新定义它在插件化设备体系里的落点

一句人话：

> `005.7` 解决“小爱怎么进入正式插件主链”，`005.7.1` 解决“进入以后，设备详情页该由谁负责什么”。

## 这份 Spec 包含什么

- `requirements.md`
  - 明确插件专属页签和通用页签各自负责什么
  - 明确哪些旧耦合必须拆掉

- `design.md`
  - 设计设备详情页插件扩展点
  - 设计设备级插件配置作用域或等价正式存储
  - 设计通用声纹页签的显示条件和兼容策略

- `tasks.md`
  - 拆出后端协议、宿主页渲染、旧字段迁移、清理验证的实施顺序

- `docs/README.md`
  - 给后续接口样例、迁移记录和联调说明留入口

## 这份 Spec 明确不做什么

- 不做插件前端 bundle 远程加载平台
- 不重写声纹算法、建档流程和识别主链
- 不把平台的 `speaker` 协议和声纹业务整个下沉到插件里
- 不把所有 `speaker` 设备都一刀切当成支持语音终端
- 不在这一轮做新的设备详情视觉大改版

## 计划产物

- 一套正式的“设备详情插件页签”扩展模型
- 一套正式的设备级插件配置落点
- 小爱插件专属“语音接管”页签
- 平台通用“声纹管理”页签
- 兼容旧字段但不再继续扩散的迁移方案
