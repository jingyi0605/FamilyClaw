# 需求文档 - AI助手设置生效与对话配置回写统一

状态：Draft

## 简介

现在“设置 > AI设置 > AI助手”这套配置，存在三类明显问题：

- 有些字段只是存进数据库，但运行时根本不看，属于假配置。
- 有些字段前端会用，后端不会用，导致设置页和真实对话行为不一致。
- 有些字段可以在对话里改，但改完以后设置页回填不完整，用户看见的是旧值。

这不是小毛病，这是产品边界没收口。用户看到的是一套“像是能配”的页面，系统跑的是另一套逻辑。  
这份 Spec 要解决的就是这件事：把 AI 助手设置变成真正的单一真值源，保证展示出来的字段都真实生效，对话和设置页走同一套配置契约。

## 术语表

- **AI助手设置**：用户在“设置 > AI设置 > AI助手”里能看到和编辑的配置。
- **真值源**：系统里唯一可信的配置来源。页面展示、对话运行、对话回写都必须读写同一份数据。
- **运行策略**：是否允许对话、是否默认进入、记忆/设置/后续操作的处置方式等会直接影响对话行为的配置。
- **助手资料**：Agent 名称、角色摘要、简介、说话风格、性格标签、服务重点等资料字段。
- **成员互动设置**：助手面对某个家庭成员时的称呼、沟通方式、补充注意事项等个性化设置。
- **配置回写**：用户在对话里修改 Agent 资料后，系统把变更落库，并让设置页读到同一份最新数据。

## 范围说明

### In Scope

- 统一 AI 助手设置页所有可见字段的真值源和运行时消费关系。
- 修正 `conversation_enabled`、`default_entry`、`ask/notify/auto` 的前后端不一致问题。
- 让对话中的配置修改和设置页中的保存走同一套 Agent 更新链路。
- 收口成员互动设置：保留并生效真正有业务意义的字段，下线没有稳定消费方的假字段。
- 让设置页、对话页在配置变更后能读到同一份最新数据。

### Out of Scope

- 不在本期重做 AI 设置页面的视觉样式。
- 不在本期新增新的 Agent 类型。
- 不在本期重做提醒、设备控制、长期记忆本身的业务能力，只收口它们与 AI 助手设置的连接方式。

## 需求

### 需求 1：设置页展示给用户的字段必须都是真字段

**用户故事：** 作为用户，我希望设置页里每个我能改的字段都真的有用，而不是我改了半天，系统根本不看。

#### 验收标准

1. WHEN 某个字段继续保留在 AI 助手设置页 THEN System SHALL 明确它的存储位置、运行时消费者和测试覆盖点。
2. WHEN 某个字段在本期找不到稳定消费者 THEN System SHALL 在同一变更里把它从用户可见表单中移除，而不是继续假装可配。
3. WHEN 设置页打开某个 Agent 详情 THEN System SHALL 只回填本期定义为真实可见的字段，不再混入无稳定语义的遗留字段。

### 需求 2：运行策略必须在前后端使用同一套语义

**用户故事：** 作为用户，我希望“允许和家人对话”“设为默认助手”“涉及记忆/设置/后续操作时如何处置”这些配置，设置页怎么写，系统就怎么执行。

#### 验收标准

1. WHEN 用户把某个 Agent 设为“不允许和家人对话” THEN System SHALL 同时在助手列表过滤、默认 Agent 选择、后端会话创建和运行时切换中阻止它成为可对话 Agent。
2. WHEN 用户把某个 Agent 设为“默认助手” THEN System SHALL 在前端默认选择和后端默认会话解析中都优先使用该 Agent，前提是它仍然处于可对话状态。
3. WHEN 运行策略中的 `memory/config/action` 分别配置为 `ask`、`notify`、`auto` THEN System SHALL 在动作记录和提案执行里使用一致语义，而不是把 `notify` 和 `auto` 当成同一种行为。

### 需求 3：助手资料必须收口成一套可维护的编辑契约

**用户故事：** 作为用户，我希望我在设置页里改的名称、简介、角色设定等资料，和对话里改的内容是同一套东西，不要一边能改，一边不认。

#### 验收标准

1. WHEN 用户在设置页编辑 Agent 名称、角色摘要、简介、说话风格、性格标签或服务重点 THEN System SHALL 把这些字段落到统一的 Agent 资料更新链路。
2. WHEN 用户在对话里明确要求修改 Agent 名称、简介、角色设定、说话风格或性格标签 THEN System SHALL 生成同一套资料更新 payload，并写回与设置页相同的真值源。
3. WHEN 某个资料字段不是稳定的用户配置字段，只是系统内部派生信息 THEN System SHALL 把它降为派生字段或隐藏字段，而不是继续和用户可编辑字段并列。

### 需求 4：成员互动设置必须要么真实生效，要么下线

**用户故事：** 作为用户，我希望“与家人的互动方式”里看到的设置，真的会影响助手跟我的对话，不想再看到存了也没用的项目。

#### 验收标准

1. WHEN 用户配置“怎么称呼 Ta”“沟通方式”“补充备注” THEN System SHALL 在 Prompt 组装和运行时对话中使用这些字段。
2. WHEN 当前表单字段在运行时没有稳定业务语义 THEN System SHALL 在本期把该字段从用户表单里移除，而不是继续保留。
3. WHEN 旧数据中仍然存在本期下线的字段 THEN System SHALL 保持兼容读取，不要求数据库清洗才能上线新逻辑。

### 需求 5：对话内配置修改完成后，设置页必须能读到最新结果

**用户故事：** 作为用户，我希望我在对话里改完助手名字或设定后，回到设置页看到的是新值，而不是旧表单。

#### 验收标准

1. WHEN 对话中的 `config.apply` 或 `config_apply` 成功修改当前 Agent THEN System SHALL 产出可用于前端失效缓存和重新拉取详情的变更信息。
2. WHEN 设置页重新打开该 Agent 详情弹窗 THEN System SHALL 读取到对话刚刚写入的最新资料。
3. WHEN 设置页和对话页在同一浏览器会话中同时存在 THEN System SHALL 提供最小可用的刷新机制，使已打开的详情弹窗能够在配置更新后重新拉取最新数据。

### 需求 6：整个链路必须可验证、可追踪，不能再靠人工猜

**用户故事：** 作为开发和排查问题的人，我希望能明确知道某个设置到底存在哪里、被谁消费、什么时候生效，不想再靠翻日志猜。

#### 验收标准

1. WHEN 后端接收 Agent 设置写入、对话配置回写或运行策略执行 THEN System SHALL 在日志和审计记录里带上 Agent ID、字段范围和来源。
2. WHEN 自动化测试运行 THEN System SHALL 至少覆盖设置保存、会话默认 Agent 解析、动作策略执行语义、对话配置回写和前端回填这几条关键链路。
3. WHEN 开发者排查“设置没生效”问题 THEN System SHALL 能通过代码、日志和测试快速判断是字段未消费、消费错误还是前端未刷新。

## 非功能需求

### 非功能需求 1：向后兼容

1. WHEN 旧家庭已经保存过现有 Agent 数据 THEN System SHALL 不要求人工改库或删数据才能继续使用。
2. WHEN 本期决定下线某些假字段 THEN System SHALL 优先通过隐藏和停止消费处理，而不是直接破坏历史记录。

### 非功能需求 2：可维护性

1. WHEN 以后新增新的 Agent 设置字段 THEN System SHALL 要求在同一处登记“字段定义、存储位置、消费者、回写支持、测试点”。
2. WHEN 以后继续扩展对话配置回写能力 THEN System SHALL 复用同一条 Agent 更新服务，而不是再写一套旁路更新逻辑。

### 非功能需求 3：可观测性

1. WHEN 某个设置被执行或被忽略 THEN System SHALL 能从日志或调试信息中看出所使用的最终策略和最终 Agent。
2. WHEN 配置被对话自动应用 THEN System SHALL 能记录变更来源是“用户设置页”还是“对话配置回写”。

## 成功定义

- 用户设置页里保留下来的字段，全部都能说明“存哪、谁用、怎么测”。
- `conversation_enabled`、`default_entry`、`ask/notify/auto` 在前后端都使用同一套规则。
- 对话里修改 Agent 资料后，重新打开设置页能看到同一份最新值。
- 成员互动设置不再包含明显的假字段。
- “这个设置到底生没生效”可以通过代码和测试直接回答，而不是靠猜。
