# 需求文档 - 设备管理前端迁移与同步收口

状态：Draft

## 简介

现在的设备管理入口放错了地方。

`设置-设备与集成` 页面本来应该干“接入什么插件、什么时候同步、同步了多少东西”这类插件级工作，但现在它还塞了设备详情、实体控制、停用、删除、操作记录，结果就是一页承担两套职责，页面越来越乱，代码也越来越容易继续长成特例堆。

更糟的是，真正和家庭成员日常使用有关的设备操作，却不在 `家庭页`。当前 `家庭页` 已经有“概览 / 房间 / 人员 / 关系”四个标签，设备明明是家庭结构的一部分，却被塞到设置页里，这就是信息架构错位。

这次要做的不是改后端控制逻辑，也不是重写 HA 接入，而是把前端职责摆正：

- `设置-设备与集成` 只做插件同步和只读查看
- `家庭页` 新增“设备”标签，承接设备筛选、查看和操作
- 同步交互补足防误触和候选筛选，尤其是全量同步确认与部分同步筛选

## 术语表

- **System**：FamilyClaw 用户端与 API Server
- **设备与集成页**：`apps/user-app/src/pages/settings/integrations/index.tsx`
- **家庭页**：`apps/user-app/src/pages/family/LegacyFamilyPage.tsx`
- **集成实例**：某个家庭下已经配置完成的插件实例，例如某个 Home Assistant 接入实例
- **已同步设备**：已经通过插件同步并落到本地 `devices` / `device_bindings` 的设备
- **候选设备**：HA 中能被同步，但还没最终执行同步动作的设备候选列表
- **HA 房间**：Home Assistant 的 Area（区域），当前项目里主要通过 HA area registry 和状态属性里的房间字段解析
- **集成分类**：HA 中设备来源所属的平台或集成标识，例如某个实体对应的 `platform` / config entry 所属集成

## 范围说明

### In Scope

- 调整 `设置-设备与集成` 页面，只保留插件实例管理、同步入口和已同步设备只读预览
- 在 `家庭页` 的“房间”和“人员”之间新增“设备”标签页
- 设备标签页支持按房间、设备类型、设备状态筛选，并承接现有设备操作能力
- 细化全量同步交互，增加多次确认，防止误触全量同步
- 细化部分同步交互，支持按名称搜索、按 HA 房间筛选、按集成分类筛选候选设备
- 复用现有后端设备详情、实体收藏、设备控制、停用、删除、操作记录接口
- 补充 HA 接口与本地字段映射分析，明确筛选能力依赖什么数据

### Out of Scope

- 这次不改后端设备同步执行逻辑，不改设备控制执行逻辑，不改 HA 服务映射逻辑
- 这次不新增批量停用、批量删除、批量控制
- 这次不重写 `家庭页` 其它标签的功能，只新增设备标签并做必要共享组件抽离
- 这次不新增新的后端设备控制接口；如果需要补字段，只允许补候选列表的只读展示字段

## 需求

### 需求 1：设备与集成页回到“插件同步中心”

**用户故事：** 作为家庭管理员，我希望设备与集成页只负责插件接入和同步，而不是混着做设备操作，这样我能清楚知道这个页面到底是管什么的。

#### 验收标准

1. WHEN 用户进入 `设置-设备与集成` 页面 THEN System SHALL 只展示集成实例、同步状态、同步入口和已同步设备统计，不展示设备控制、停用、删除、操作记录入口。
2. WHEN 用户点击某个插件实例 THEN System SHALL 通过模态框或等价弹层展示该实例下的已同步设备列表，并明确这是只读查看，不提供设备操作按钮。
3. WHEN 用户需要控制或管理某个设备 THEN System SHALL 引导用户前往 `家庭页-设备` 标签，而不是继续留在设置页处理。

### 需求 2：家庭页新增设备标签并承接设备管理

**用户故事：** 作为家庭成员或管理员，我希望在家庭页直接查看和操作设备，因为设备本来就是家庭结构的一部分，不应该藏在设置页里。

#### 验收标准

1. WHEN 用户进入家庭页 THEN System SHALL 在“房间”和“人员”之间新增“设备”标签。
2. WHEN 用户切换到“设备”标签 THEN System SHALL 展示当前家庭的设备列表，并支持按房间、设备类型、设备状态筛选。
3. WHEN 用户打开某个设备详情 THEN System SHALL 复用现有设备详情能力，支持查看收藏实体、全部实体、执行实体控制、停用设备、删除设备和查看操作记录。

### 需求 3：设备操作能力迁移但不改后端执行语义

**用户故事：** 作为项目维护者，我希望只是把入口搬对位置，而不是把已经打通的设备控制链路再拆一遍，这样迁移风险才可控。

#### 验收标准

1. WHEN 家庭页设备标签执行设备控制 THEN System SHALL 继续复用现有 `/devices/*` 与 `/device-actions/execute` 链路，不新增第二套控制逻辑。
2. WHEN 家庭页设备标签执行停用、删除、查看日志、实体收藏 THEN System SHALL 继续复用当前后端接口语义，不改变状态机和动作结果含义。
3. WHEN 本次迁移完成 THEN System SHALL 从设置页前端代码中移除对设备控制、停用、删除、日志入口的依赖，避免同一能力存在两个入口。

### 需求 4：全量同步必须多次确认

**用户故事：** 作为家庭管理员，我希望全量同步前被反复提醒，因为这个动作影响范围大，误点一次就可能把整个设备池重新刷一遍。

#### 验收标准

1. WHEN 用户点击“全量同步设备” THEN System SHALL 先展示第一次风险确认，明确说明这是针对当前插件实例的全量同步，不是查看候选列表。
2. WHEN 用户通过第一次确认 THEN System SHALL 展示第二次确认，明确说明可能更新已有设备、导入新增设备，并展示预计影响范围或无法预估的提示。
3. WHEN 用户未完成全部确认步骤 THEN System SHALL 不发起真正的同步请求。

### 需求 5：部分同步支持按名称、HA 房间、集成分类筛选

**用户故事：** 作为家庭管理员，我希望在部分同步时先筛出我要的设备，再决定同步哪些，否则候选设备一多就只能靠肉眼翻垃圾堆。

#### 验收标准

1. WHEN 用户打开“部分同步设备”候选弹层 THEN System SHALL 支持按设备名称搜索。
2. WHEN 用户打开候选弹层 THEN System SHALL 支持按 HA 房间筛选候选设备，并展示可选房间项。
3. WHEN 用户打开候选弹层 THEN System SHALL 支持按集成分类筛选候选设备；若当前后端候选数据未直接返回该分类，System SHALL 采用明确的降级策略并在设计中写清楚。
4. WHEN 用户在筛选条件下勾选部分设备 THEN System SHALL 保持勾选状态稳定，不因切换筛选条件而丢失已选结果。

### 需求 6：HA 筛选能力必须有明确数据来源

**用户故事：** 作为维护者，我希望这次筛选不是拍脑袋造字段，而是明确知道每个筛选条件来自 HA 哪个接口、当前项目哪段代码已经拿到了它。

#### 验收标准

1. WHEN Spec 完成 THEN System SHALL 提供一份 HA 接口分析，说明当前项目如何从 `/api/states`、device registry、entity registry、area registry 读取数据。
2. WHEN 设计部分同步筛选 THEN System SHALL 明确写出“名称、HA 房间、集成分类”分别依赖哪些 HA 字段，以及本地前后端如何承接这些字段。
3. WHEN 某个筛选字段在当前项目链路里拿不到 THEN System SHALL 明确写出最小改动方案，且说明这不属于修改设备同步执行逻辑。

## 非功能需求

### 非功能需求 1：兼容性

1. WHEN 迁移设备前端入口 THEN System SHALL 保持现有已同步设备、设备控制、停用、删除、日志链路可继续工作，不破坏用户当前数据。
2. WHEN 家庭页新增设备标签 THEN System SHALL 不影响原有“概览 / 房间 / 人员 / 关系”标签的可访问性和已有交互。

### 非功能需求 2：可读性

1. WHEN 用户在设置页看插件同步 THEN System SHALL 明确区分“插件级动作”和“设备级动作”，不再把两类动作混在一起。
2. WHEN 用户在家庭页看设备 THEN System SHALL 让筛选、设备状态、可操作项一眼能看懂，不要求用户理解插件内部实现。

### 非功能需求 3：可维护性

1. WHEN 后续继续扩展设备页面 THEN System SHALL 复用共享设备管理组件，而不是再把一整套设备详情逻辑塞回设置页。
2. WHEN 后续继续扩展 HA 候选筛选 THEN System SHALL 基于明确的候选字段模型扩展，而不是继续在页面里写散乱的临时判断。

## 成功定义

- `设置-设备与集成` 页面不再承担设备控制、停用、删除、日志职责，只保留插件同步与只读预览。
- `家庭页` 新增设备标签，并成为设备筛选、查看和操作的唯一主入口。
- 全量同步有多次确认，部分同步支持名称、HA 房间、集成分类筛选。
- 本次改造不改后端设备同步执行逻辑、不改设备控制执行逻辑，但前端信息架构明显变干净。
