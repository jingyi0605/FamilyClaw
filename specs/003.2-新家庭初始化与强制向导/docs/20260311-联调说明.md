# 003.2 联调说明

## 这份说明解决什么

这一步不是讲废话，是把 `003.2` 现在已经落地的链路说清楚，免得联调时每个人口径都不一样。

现在系统里有两类家庭：

- **新家庭**：创建时会写入 `setup_status = pending`，必须先走初始化向导
- **历史家庭**：没有这个标记，先按“主链路是不是已经能用”做兼容判断

## 当前联调范围

本轮已经落地：

1. 家庭初始化状态读模型
2. `user-web` 路由准入守卫
3. 向导四步最小闭环
   - 家庭资料
   - 首位成员
   - AI 供应商
   - 首个管家 Agent

本轮**没有**展开：

- 完整用户端 AI 配置中心
- 复杂供应商适配器 UI
- 对话式首个管家 bootstrap

## 当前状态判定规则

### 新家庭

新家庭会严格检查四步：

1. `family_profile`
2. `first_member`
3. `provider_setup`
4. `first_butler_agent`

只要没做完，就会被 guard 拦回 `/setup`。

### 历史家庭

历史家庭不能一刀切重跑。

当前兼容规则是：

- 缺 `city` 或缺首位成员，**不直接锁死**
- 缺 `qa_generation` 供应商路由，视为主链路不可用
- 缺激活的 `butler`，视为主链路不可用

也就是说，历史家庭只有在**AI 主链路真的废了**时，才会被要求补录。

## 关键接口

### 家庭初始化状态

- `GET /api/v1/households/{household_id}/setup-status`

看这个接口就够了。别让页面自己猜。

返回里最关键的是：

- `status`
- `current_step`
- `missing_requirements`
- `is_required`

### 家庭资料保存

- `PATCH /api/v1/households/{household_id}`

当前向导会写：

- `name`
- `city`
- `timezone`
- `locale`

### 首位成员创建

- `POST /api/v1/members`

### 供应商配置

当前为了先闭环，向导支持两种方式：

1. 直接复用已有供应商
2. 创建一个 `native_sdk` 模拟供应商，然后给当前家庭挂 `qa_generation` / `qa_structured_answer` 路由

相关接口：

- `GET /api/v1/ai/providers`
- `POST /api/v1/ai/providers`
- `GET /api/v1/ai/routes?household_id=...`
- `PUT /api/v1/ai/routes/{capability}`

### 首个管家创建

- `POST /api/v1/ai-config/{household_id}/agents`

当前创建的是最小可用的激活态 `butler`，同时会写入：

- agent 基础记录
- soul profile
- runtime policy

## 前端准入口径

### 强制拦截页面

当前 guard 已接到：

- `/`
- `/conversation`
- `/settings/ai`

### `/setup` 的访问规则

- 新家庭未完成：允许进入
- 历史家庭主链路不可用：允许进入补录
- 历史家庭主链路可用但有缺口：也允许手动进入补录
- 完全没缺口：访问 `/setup` 会自动跳回主页面

## 联调建议

### 造一个全新家庭

用它验证强制链路：

1. 创建家庭
2. 打开 `user-web`
3. 确认直接被带到 `/setup`
4. 按四步走完
5. 确认最终可进入主页、对话页、AI 配置页

### 造一个历史家庭

至少测两种：

1. **可用历史家庭**
   - 有供应商路由
   - 有激活的 `butler`
   - 缺 `city` 也没关系
   - 预期：正常放行，设置页可看到补录入口

2. **不可用历史家庭**
   - 没供应商路由或没 `butler`
   - 预期：进入受保护页面会被带到 `/setup`

## 当前已知取舍

- 供应商步骤现在优先保证“能用”，不是保证“配置中心完整”
- 首个管家步骤现在优先保证“有一个活的 butler”，不是保证“创建体验多高级”
- `pytest` 当前环境没装，联调主要靠 `python -m compileall app` 和 `npm.cmd run build`
