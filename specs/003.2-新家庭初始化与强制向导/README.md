# Spec 003.2 - 新家庭初始化与强制向导

状态：Draft

## 这份 Spec 到底要解决什么

现在产品的主链路有个硬伤：新建家庭以后，用户马上能进入主页面，但系统其实还没准备好。

具体问题很直接：

- 家庭基础资料可能还没填完
- 第一位家庭成员还没建立
- AI 供应商还没配置
- 默认管家 Agent 还没创建
- 用户进入对话页时，只会看到“没有可用 Agent”这种废话提示

这说明现在缺的不是“再补一个页面”，而是**新家庭从 0 到可用的强制初始化主链路**。

这份 Spec 要做的就是：

1. 给新家庭增加一条强制配置向导
2. 在向导里按顺序完成基础资料、首成员、AI 供应商、首个管家 Agent 创建
3. 在这些步骤完成前，不允许用户进入主页面
4. 对已有家庭做兼容，不把已经能用的用户突然锁死

## 规划结论

### 【核心判断】

✅ 值得做，而且优先级高。  
这不是体验优化，而是产品首次可用链路的修复。

### 【关键洞察】

- 真正要新增的核心对象不是“几个向导页面”，而是**家庭初始化状态**
- 主页面、对话页、AI 配置页都不该自己乱判断，而应该统一读一个“是否允许进入产品主链路”的结果
- `003.3` 负责把 AI 配置能力下沉到 `user-web`
- `003.2` 负责把这些能力串成一条强制、可恢复、可校验的初始化流程

### 【Linus式方案】

1. 先定义一个清晰的初始化状态模型
2. 再定义统一的路由准入规则
3. 再把向导步骤按顺序串起来
4. 不破坏已有家庭，旧数据必须有兼容路径

## 这份 Spec 包含什么

- `requirements.md`：定义新家庭强制向导、主页面准入、已有家庭兼容规则
- `design.md`：定义初始化状态机、向导步骤编排、路由守卫和恢复策略
- `tasks.md`：定义落地顺序、涉及文件和验证方式

## 这份 Spec 明确不做什么

- 不在这里详细展开 AI 供应商适配器和 Agent 配置字段本身，那些放进 `003.3`
- 不重做现有家庭、成员、记忆中心的底层数据模型
- 不强制所有已有家庭重新走完整向导
- 不把 `admin-web` 删除；它仍可保留为开发调试入口，但不再是正式产品唯一入口

## 依赖与协同

- **依赖 `Spec 001`**：复用家庭和成员基础模型
- **依赖 `Spec 002.2`**：复用 `user-web` 产品结构和路由框架
- **协同 `Spec 003.1`**：复用多 Agent 基础模型和对话主链路
- **强协同 `Spec 003.3`**：向导里的 AI 供应商配置和首个 Agent 创建能力由 `003.3` 提供
