# 需求文档 - 新家庭初始化与强制向导

状态：Draft

## 简介

产品现在缺一条真正可用的新家庭启动流程。

用户创建家庭以后，系统不能假装自己已经准备好了。  
如果家庭资料、首成员、AI 供应商和默认管家都还没准备好，就应该把用户带进初始化向导，而不是把他丢进一个到处报空态的主页面。

这份需求文档定义的是：

- 什么情况下必须进入初始化向导
- 向导里必须完成哪些步骤
- 什么条件下才允许进入主页面
- 如何兼容已有家庭，避免破坏现有用户

## 需求 1：新家庭必须先完成初始化，才能进入主页面

**用户故事：** 作为第一次创建家庭的用户，我希望系统明确告诉我还差什么配置，而不是让我先进主页面再到处撞墙。

### 验收标准

1. WHEN 用户创建了一个新的家庭 THEN System SHALL 将该家庭标记为“初始化未完成”。
2. WHEN 初始化未完成的家庭被选中 THEN System SHALL 将用户引导到初始化向导，而不是直接进入主页面。
3. WHEN 用户尝试直接访问主页、对话页、AI 配置页或其他主功能页面 THEN System SHALL 统一拦截并跳回向导。
4. WHEN 初始化完成 THEN System SHALL 放行进入主页面，不再重复强制跳转。

## 需求 2：向导必须先完成家庭基础资料

**用户故事：** 作为家庭创建者，我希望先把家庭名字和城市填好，这样后续时区、天气、提醒和上下文才有落点。

### 验收标准

1. WHEN 用户进入向导第一步 THEN System SHALL 要求填写家庭名称和城市。
2. WHEN 家庭名称或城市为空 THEN System SHALL 不允许进入下一步。
3. WHEN 用户提交家庭资料 THEN System SHALL 保存家庭名称、城市以及后续需要的地区信息。

## 需求 3：向导必须建立家庭的第一位成员

**用户故事：** 作为家庭创建者，我希望先把自己建成家庭的第一位成员，并定义系统该怎么称呼我。

### 验收标准

1. WHEN 用户进入首成员步骤 THEN System SHALL 要求填写“你的名字”和“系统如何称呼你”。
2. WHEN 当前家庭还没有任何成员 THEN System SHALL 强制完成这个步骤。
3. WHEN 首成员创建成功 THEN System SHALL 将其标记为该家庭初始化过程中的默认当前成员。

## 需求 4：向导必须完成至少一个可用 AI 供应商配置

**用户故事：** 作为新用户，我希望系统先完成一个能工作的 AI 供应商配置，否则后面的 Agent 创建和对话都只是空壳。

### 验收标准

1. WHEN 用户进入 AI 供应商步骤 THEN System SHALL 提供正式的用户端配置入口，而不是提示去 `admin-web`。
2. WHEN 当前家庭还没有任何可用 AI 供应商配置 THEN System SHALL 不允许跳过该步骤。
3. WHEN 用户完成供应商配置 THEN System SHALL 校验该配置至少可用于后续首个管家 Agent 创建。
4. WHEN 供应商配置校验失败 THEN System SHALL 明确指出失败原因，并停留在当前步骤。

## 需求 5：向导必须通过对话方式创建第一个管家 Agent

**用户故事：** 作为新用户，我希望系统用对话式引导帮我创建第一个管家，而不是逼我先面对一堆冷冰冰的配置表单。

### 验收标准

1. WHEN 用户进入首个 Agent 创建步骤 THEN System SHALL 使用对话式流程引导用户为第一个管家起名、设置性格和基本资料。
2. WHEN 首个管家 Agent 尚未创建成功 THEN System SHALL 不允许完成整个初始化向导。
3. WHEN 首个管家 Agent 创建完成 THEN System SHALL 将其设置为默认入口 Agent 或明确的默认候选。
4. WHEN 对话式创建中断 THEN System SHALL 允许用户恢复继续，而不是丢失全部进度。

## 需求 6：已有家庭必须有兼容路径

**用户故事：** 作为已经在用系统的用户，我不希望因为新向导上线，自己突然被锁在门外。

### 验收标准

1. WHEN 系统中存在历史家庭 THEN System SHALL 按兼容规则判断其是否已具备可用状态，而不是一刀切要求重跑新向导。
2. WHEN 历史家庭缺少部分新字段但主链路已可用 THEN System SHALL 允许正常进入主页面，并把补录入口放到设置中。
3. WHEN 历史家庭主链路确实不可用 THEN System SHALL 只引导缺失步骤，而不是无脑重头再来。

## 需求 7：向导必须支持中断恢复

**用户故事：** 作为用户，我可能在配置过程中关闭页面或切换设备，我希望回来时能接着走，而不是重来一遍。

### 验收标准

1. WHEN 用户在向导过程中刷新页面或重新进入 THEN System SHALL 恢复到上次未完成的步骤。
2. WHEN 某一步已完成 THEN System SHALL 不要求重复填写。
3. WHEN 后续步骤依赖前置步骤结果 THEN System SHALL 使用已保存结果继续，而不是让用户重新输入。

## 非功能需求

### 非功能需求 1：可理解性

1. WHEN 用户进入向导任一步 THEN System SHALL 明确告诉用户“这一步在做什么、做完能得到什么、现在还差什么”。

### 非功能需求 2：可靠性

1. WHEN 单一步骤失败 THEN System SHALL 只阻塞当前步骤，不影响已完成步骤的数据保存。
2. WHEN 向导恢复时读取失败 THEN System SHALL 至少能给出明确错误，并允许重试。

### 非功能需求 3：兼容性

1. WHEN 新向导上线 THEN System SHALL 不破坏已有家庭的默认访问能力。

## 成功定义

- 新家庭创建后，不再能跳过关键配置直接进入一个半残废的主页面
- 向导完成后，家庭、首成员、AI 供应商和默认管家 Agent 都已具备
- 主页面、对话页、AI 配置页都共享统一的准入判断
- 历史家庭有兼容策略，不被粗暴锁死
