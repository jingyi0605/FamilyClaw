# 设计文档 - 新家庭初始化与强制向导

状态：Draft

## 1. 概述

### 1.1 目标

- 给新家庭建立一条从“刚创建”到“真正可用”的强制初始化主链路
- 用统一的初始化状态控制产品准入，避免页面各自乱判断
- 与 `003.3` 提供的用户端 AI 配置能力协同，但不把两份设计混成一锅

### 1.2 覆盖需求

- `requirements.md` 需求 1：新家庭必须先完成初始化
- `requirements.md` 需求 2：向导必须完成家庭基础资料
- `requirements.md` 需求 3：向导必须建立首成员
- `requirements.md` 需求 4：向导必须完成 AI 供应商配置
- `requirements.md` 需求 5：向导必须创建首个管家 Agent
- `requirements.md` 需求 6、7：兼容与恢复

### 1.3 技术约束

- 前端主入口在 `apps/user-web`
- 不新增第二套家庭、成员、Agent 主数据模型
- `admin-web` 仍可保留为调试工具，但不再承担正式产品初始化职责
- 供应商配置和 Agent 创建能力复用 `003.3`

## 2. 架构

### 2.1 系统结构

这份设计引入一个核心概念：**家庭初始化状态读模型**。

所有准入判断都应该读这份结果，而不是散落在各个页面里自己推断。

整体流程：

1. 用户创建家庭
2. 系统写入“初始化未完成”状态
3. `user-web` 检查当前家庭初始化状态
4. 若未完成，则强制进入初始化向导
5. 向导按步骤保存进度
6. 所有必需步骤完成后，系统将家庭标记为“初始化完成”
7. 主页面和其他页面才放行

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 初始化状态聚合 | 判断家庭当前缺什么 | 家庭、成员、供应商、Agent 数据 | 当前步骤、完成状态、缺失项 |
| 向导路由守卫 | 控制是否允许进入主页面 | 初始化状态读模型 | 放行或重定向 |
| 向导步骤页 | 引导用户补齐当前步骤 | 用户输入 | 已保存的步骤结果 |
| 向导进度持久化 | 保存和恢复向导状态 | 当前步骤和结果 | 可恢复的进度 |
| AI 配置能力复用层 | 承接供应商和 Agent 创建能力 | 向导调用 | 标准化配置结果 |

### 2.3 关键流程

#### 2.3.1 新家庭首次进入

1. 用户创建家庭
2. 系统写入初始化状态为 `pending`
3. 用户进入 `user-web`
4. 路由守卫读取初始化状态
5. 若未完成，则跳转到向导第一页

#### 2.3.2 向导完成放行

1. 用户按步骤完成家庭资料、首成员、供应商配置、首个 Agent 创建
2. 初始化状态聚合确认必需条件已满足
3. 系统将家庭标记为 `completed`
4. 路由守卫放行进入主页面

#### 2.3.3 历史家庭兼容

1. 用户选择历史家庭
2. 初始化状态聚合根据兼容规则判断其是否已可用
3. 已可用则直接放行
4. 不可用则只引导缺失步骤

## 3. 组件和接口

### 3.1 核心组件

- `household_setup_status` 读模型：统一描述家庭初始化状态
- `SetupWizardLayout`：承载强制向导的外层布局
- `SetupGuard`：拦截主页面访问
- `SetupProgressStore`：保存和恢复用户的向导进度
- `SetupCompletionService`：统一判断“是否已完成”

### 3.2 数据结构

#### 3.2.1 家庭初始化状态读模型

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `household_id` | string | 是 | 家庭 ID | 唯一 |
| `status` | enum | 是 | `pending / in_progress / completed / blocked` | 固定枚举 |
| `current_step` | string | 是 | 当前应该执行的步骤 | 固定步骤码 |
| `completed_steps` | string[] | 是 | 已完成步骤列表 | 不重复 |
| `missing_requirements` | string[] | 是 | 当前缺失项 | 不重复 |
| `is_required` | boolean | 是 | 当前是否必须走向导 | 固定布尔 |
| `resume_token` | string \| null | 否 | 用于恢复当前进度 | 可为空 |
| `updated_at` | string | 是 | 状态更新时间 | ISO 时间 |

#### 3.2.2 向导步骤建议

| 步骤码 | 含义 | 说明 |
| --- | --- | --- |
| `family_profile` | 家庭基础资料 | 家庭名、城市 |
| `first_member` | 第一位成员 | 名字、称呼 |
| `provider_setup` | AI 供应商配置 | 至少一个可用供应商 |
| `first_butler_agent` | 首个管家创建 | 对话式创建第一个 Agent |
| `finish` | 完成页 | 展示完成结果并放行 |

### 3.3 接口契约

#### 3.3.1 获取家庭初始化状态

- 类型：HTTP
- 路径：`GET /api/v1/households/{household_id}/setup-status`
- 输入：`household_id`
- 输出：初始化状态读模型
- 校验：家庭必须存在
- 错误：`404` 家庭不存在；`403` 无访问权限

#### 3.3.2 更新向导步骤结果

- 类型：HTTP
- 路径：`PUT /api/v1/households/{household_id}/setup-steps/{step_code}`
- 输入：步骤数据
- 输出：最新初始化状态
- 校验：前置步骤必须满足
- 错误：`400` 步骤数据非法；`409` 前置步骤未完成

#### 3.3.3 完成初始化

- 类型：HTTP
- 路径：`POST /api/v1/households/{household_id}/setup-complete`
- 输入：家庭 ID
- 输出：完成结果和跳转建议
- 校验：所有必需步骤已完成
- 错误：`409` 仍有缺失项

## 4. 数据与状态模型

### 4.1 数据关系

- 初始化状态不是第二份主数据，只是聚合结果
- 家庭、成员、供应商、Agent 仍是各自原本的主表
- 初始化状态负责把这些结果变成一句人话：**这个家庭现在能不能进主页面**

### 4.2 状态流转

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `pending` | 刚创建，还没开始 | 新家庭创建完成 | 用户完成第一步或开始填写 |
| `in_progress` | 正在初始化 | 至少已完成一步 | 所有必需步骤完成 |
| `blocked` | 当前步骤失败或依赖缺失 | 供应商校验失败、创建 Agent 失败 | 用户修复并重新提交 |
| `completed` | 已可进入主页面 | 全部关键步骤完成 | 一般不回退；如后续人工重置则特殊处理 |

## 5. 关键设计决策

### 5.1 用统一状态读模型，而不是页面各自拼判断

坏设计是：

- 页面 A 查成员
- 页面 B 查 Agent
- 页面 C 查供应商
- 然后每页自己判断能不能进

这会迅速烂掉。

正确做法是：

- 后端或前端聚合层统一给出初始化状态
- 所有页面只读这一份结果

### 5.2 历史家庭兼容优先

新向导上线以后：

- 对新家庭：严格强制
- 对历史家庭：按可用性放行
- 只在确实缺关键能力时，才要求补录

### 5.3 向导步骤可恢复

第一版不需要搞复杂工作流引擎。  
只要做到：

- 每一步提交即保存
- 刷新后能恢复当前步骤
- 已完成步骤不重做

这就够了。

## 6. 风险与边界

### 6.1 风险

- 风险 1：把新家庭和历史家庭一刀切处理，导致旧用户被锁死
- 风险 2：把向导状态散落到多个页面判断，后续维护失控
- 风险 3：把供应商配置和 Agent 创建逻辑重复写两套，而不是复用 `003.3`

### 6.2 这份设计先不做什么

- 不做复杂可视化流程引擎
- 不做跨设备实时同步提示
- 不做初始化流程中复杂的自动推荐路由

## 7. 验证策略

- 路由守卫：新家庭访问主页、对话页、AI 配置页时会被拦回向导
- 兼容策略：历史家庭不会被粗暴锁死
- 中断恢复：刷新页面后仍能继续当前步骤
- 完成放行：完成向导后可正常进入主页面
