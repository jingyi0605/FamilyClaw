# 版本治理前端联调与验收说明

日期：2026-03-17

## 这份说明解决什么问题

`004.7` 到这一步，后端的版本治理、手动升级和手动回滚已经落地了，前端也接上了。

这份文档不讲大词，只讲三件事：

1. 现在页面到底展示了什么
2. 用户从哪里发起安装、升级、回滚
3. 这轮明确还没做什么，避免别人顺手脑补

## 当前页面行为

### 1. 市场页

- 市场卡片会展示：
  - 已安装版本
  - 市场最新版本
  - 最新可兼容版本
  - 更新状态
  - 兼容状态
  - 阻断原因（如果有）
- 新安装默认优先使用后端返回的 `latest_compatible_version`
- 如果当前没有可兼容目标版本，前端不会再继续拿 `latest_version` 硬装
- 已安装且存在更高可兼容版本时，市场卡片会给出直接升级入口

### 2. 插件详情抽屉

- 抽屉新增“版本治理”区块
- 会读取统一版本治理结果，而不是前端自己拼 `update_state`
- 市场管理插件会额外加载可选版本列表
- 用户可以在抽屉里选择目标版本，然后手动执行：
  - 升级
  - 回滚

## 本轮联调验证

已完成的最小必要验证：

- 后端：
  - `.\.venv\Scripts\python.exe -m unittest tests.test_plugin_versioning`
  - `.\.venv\Scripts\python.exe -m unittest tests.test_plugin_marketplace_service`
  - `.\.venv\Scripts\python.exe -m unittest tests.test_plugin_marketplace_api`
- 前端：
  - `apps/user-app` 下执行 `npm.cmd run typecheck`
- 人工走查：
  - 确认市场安装改为优先走 `latest_compatible_version`
  - 确认页面展示 `version_governance`
  - 确认详情抽屉存在指定版本升级 / 回滚入口

## 这轮明确没做什么

- 没做自动升级
- 没做后台定时检查新版本
- 没做完整 semver 平台
- 没做签名体系
- 没做任意 URL 安装
- 没做前端本地二次版本比较，目标版本是否合法仍以后端校验为准
