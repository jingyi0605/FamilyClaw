# Spec 004.7 - 插件版本治理与手动升级

状态：Draft

## 这份 Spec 解决什么问题

现在项目里的插件版本信息已经不算少：

- 插件 manifest 里有 `version`
- 插件市场条目里有 `latest_version`、`versions[]`
- 安装实例里有 `installed_version`
- 兼容性字段里还能塞一点说明

但这些东西现在还是散的。

系统最多只做到：

- 统一把字段展示出来
- 用最粗糙的字符串相等判断给一个 `update_state`

这不叫治理，这叫占位。

真正缺的东西有四块：

1. 谁才是“当前已安装版本”的唯一真相
2. 市场最新版本和当前宿主是否兼容，系统到底怎么判断
3. 手动升级时，哪些状态应该保留，哪些应该阻断
4. 前后端到底该怎么把“可升级 / 被兼容性阻断 / 可回滚”讲清楚

一句话说清楚：

这份 `004.7` 不是做自动升级平台，而是把插件版本治理从“字段展示”推进到“手动升级可用、兼容性可判断、回滚有边界”的最小可用能力。

## 这次要做到什么

- 统一插件版本真相来源和版本比较逻辑
- 基于插件市场版本列表实现“最新可兼容版本”判断
- 在安装链路里落实宿主版本兼容性校验
- 实现手动升级和手动回滚的后端边界
- 让升级默认不破坏现有启停和配置状态
- 让前端明确展示已安装版本、市场最新版本、最新可兼容版本和更新状态
- 把版本治理边界写进文档，避免继续把 `version` 当摆设

## 这次明确不做什么

- 不做自动升级
- 不做后台静默更新
- 不做完整 semver 规则平台化治理
- 不做插件签名体系
- 不做版本差异迁移脚本自动生成
- 不做“任意来源 URL 版本安装”

## 和已有 Spec 的关系

- `004.5-插件能力统一接入与版本治理`
  - 已经把“版本字段统一展示”这件事收口了，但明确还没有远程市场比对、自动升级和完整 semver 治理
- `004.6-插件市场一键安装与手动启用`
  - 已经把市场同步、一键安装、默认禁用、手动启用打通了，但没有把升级、回滚和兼容性判断做完整

`004.7` 要做的事情很明确：

- 不重写 `004.5`
- 不推翻 `004.6`
- 而是在两者之上，把“版本治理”和“手动升级”单独收成一份能执行的 Spec
