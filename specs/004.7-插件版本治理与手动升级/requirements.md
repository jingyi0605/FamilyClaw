# 需求文档 - 插件版本治理与手动升级

状态：Draft

## 简介

当前 FamilyClaw 的插件市场已经能发现插件、同步目录、按指定版本安装插件，也已经把安装状态、启用状态和配置状态拆开了。

问题不是没有版本字段，问题是这些字段现在还没有真正承担治理责任。

最典型的几个坏点：

- 市场页默认拿 `latest_version` 直接安装，但不判断这个版本是不是和当前宿主兼容
- 注册表里的 `update_state` 现在基本只是“字符串相等 / 不相等”，这太糙了
- 升级如果继续复用“首次安装”逻辑，很容易把已经启用的插件重新打回禁用
- 前端能看到版本号，但看不到“最新可兼容版本”和“为什么现在不能升”

这轮要解决的是真问题：

在不做自动升级平台的前提下，把插件版本治理推进到能支持手动升级、手动回滚、兼容性判断和统一状态展示。

这份 Spec 服务四类人：

- **家庭管理员**：希望知道插件能不能升、为什么不能升、升级后会不会把当前配置打坏
- **后端开发者**：希望版本比较和兼容性判断有统一服务，不要每条链路自己猜
- **前端开发者**：希望 UI 直接消费统一版本状态，不再手搓升级提示
- **插件开发者**：希望知道哪些字段现在会参与兼容性判断，哪些字段还只是说明

成功后的核心收益很简单：

系统至少能把“已安装版本”“市场最新版本”“最新可兼容版本”“当前升级状态”说清楚，并且真的能安全手动升级。

## 术语表

- **System**：FamilyClaw 当前的插件市场、插件注册和插件运行系统
- **声明版本**：插件 `manifest.version` 或兼容注册源里声明的版本
- **已安装版本**：当前家庭插件实例实际落盘并正在使用的版本
- **市场最新版本**：插件市场条目里 `latest_version` 指向的版本
- **最新可兼容版本**：在当前宿主版本条件下，可被安装的最高市场版本
- **宿主版本**：当前 FamilyClaw 应用版本，例如 `settings.app_version`
- **手动升级**：由用户明确触发的插件版本切换，不包含后台静默更新
- **手动回滚**：由用户明确触发的插件版本回退到历史已知版本

## 范围说明

### In Scope

- 统一插件版本真相和版本状态输出
- 基于市场版本列表的兼容版本筛选
- 宿主版本和插件市场 `min_app_version` 的兼容性校验
- 手动升级和手动回滚的后端能力
- 升级后的启停与配置状态保持规则
- 前端版本信息、更新状态和升级/回滚入口
- 版本治理文档和验证边界

### Out of Scope

- 自动升级
- 定时检查并自动安装新版本
- 完整 semver 语义治理平台
- 插件二进制签名校验体系
- 配置迁移脚本自动执行框架
- 多来源插件并行安装模型重构

## 需求

### 需求 1：系统必须提供统一插件版本结果

**用户故事：** 作为前后端开发者，我希望系统对每个插件都给出统一的版本结果，以便页面展示和后端校验都不用各写各的判断。

#### 验收标准

1. WHEN System 返回插件注册结果或市场目录结果 THEN System SHALL 提供统一版本字段，至少覆盖声明版本、已安装版本、市场最新版本、最新可兼容版本和更新状态。
2. WHEN 同一个插件来自市场安装链路 THEN System SHALL 以插件实例记录中的已安装版本作为当前家庭的版本真相，而不是让页面自己拼接推断。
3. WHEN 某类插件不受市场管理 THEN System SHALL 明确返回“不可比较”或等价状态，而不是伪造升级结果。

### 需求 2：系统必须在安装和升级前做宿主兼容性判断

**用户故事：** 作为家庭管理员，我希望系统在我点击安装或升级前就告诉我版本是否兼容，而不是装到一半才炸。

#### 验收标准

1. WHEN 用户安装或升级到某个市场版本 THEN System SHALL 在真正下载前校验该版本与当前宿主版本是否兼容。
2. WHEN 目标版本声明了最低宿主版本且当前宿主过低 THEN System SHALL 阻断安装或升级，并返回明确兼容性错误。
3. WHEN 市场版本没有提供足够兼容性信息 THEN System SHALL 按保守策略处理，并明确告诉用户当前判断依据不足。

### 需求 3：系统必须支持手动升级和手动回滚

**用户故事：** 作为家庭管理员，我希望能明确把插件升级到某个可兼容版本，或者回滚到旧版本，以便控制风险。

#### 验收标准

1. WHEN 用户明确发起升级 THEN System SHALL 支持把插件切换到指定可兼容版本，而不是只能重装最新版本。
2. WHEN 用户明确发起回滚 THEN System SHALL 支持把插件切换回指定旧版本，前提是该版本仍然满足当前最小安装约束。
3. WHEN 目标版本不存在、不可安装或不兼容 THEN System SHALL 阻断操作，并返回具体错误原因。

### 需求 4：升级和回滚不能破坏现有用户空间

**用户故事：** 作为现有用户，我希望手动升级后不要把已经启用、已经配置好的插件莫名其妙打回初始状态。

#### 验收标准

1. WHEN 已安装插件执行同源升级 THEN System SHALL 默认保留原有启用状态，而不是一律重置为禁用。
2. WHEN 已安装插件执行同源升级或回滚 THEN System SHALL 默认保留现有配置，除非系统明确判断配置已不兼容。
3. WHEN 升级或回滚导致配置 schema、运行能力或兼容性不再满足要求 THEN System SHALL 将插件状态明确降级为待配置或配置无效，而不是静默继续运行。

### 需求 5：前端必须清楚展示版本治理状态

**用户故事：** 作为家庭管理员，我希望在页面上直接看懂当前版本、可升级状态和阻断原因，而不是去猜。

#### 验收标准

1. WHEN 页面展示市场插件或已安装插件 THEN System SHALL 明确展示已安装版本、市场最新版本和最新可兼容版本。
2. WHEN 插件可升级、已是最新、被宿主版本阻断或不受市场管理 THEN System SHALL 给出清楚且稳定的状态文案。
3. WHEN 用户准备升级或回滚 THEN System SHALL 在操作入口附近明确展示目标版本和主要风险提示。

### 需求 6：版本治理规则和文档必须保持一致

**用户故事：** 作为后续接手项目的人，我希望文档说的版本治理边界和真实实现一致，不要再出现 README 说一套、代码做一套。

#### 验收标准

1. WHEN `004.7` 建立后 THEN System SHALL 明确写清楚版本字段、兼容性字段、升级状态字段分别负责什么。
2. WHEN 开发者查看相关 Spec 或开发文档 THEN System SHALL 能看到“这轮做了什么、没做什么、后续还能往哪里扩”的清楚说明。
3. WHEN 当前能力仍然不包含自动升级或完整 semver 治理 THEN System SHALL 在文档中明确写出边界，而不是暗示系统已经具备这些能力。

## 非功能需求

### 非功能需求 1：一致性

1. WHEN 任一后端模块需要判断插件是否可升级 THEN System SHALL 复用统一版本比较逻辑，而不是各自比较字符串。
2. WHEN 任一前端页面展示插件版本状态 THEN System SHALL 优先消费后端统一输出，不允许页面自己推导第二套状态。

### 非功能需求 2：可靠性

1. WHEN 市场最新版本当前不可兼容 THEN System SHALL 仍然保留并展示已安装版本，不得把现有插件实例判成损坏。
2. WHEN 升级或回滚失败 THEN System SHALL 保留失败前的已安装版本信息和错误定位信息，避免落成半装状态。

### 非功能需求 3：可维护性

1. WHEN 后续需要扩展更完整的 semver、自动升级或版本迁移规则 THEN System SHALL 能在本次统一版本模型上继续扩展，而不是推翻重来。
2. WHEN 排查版本问题 THEN System SHALL 能从接口、日志或任务记录中明确看到插件 ID、来源、已安装版本、目标版本和兼容性结果。

## 成功定义

- 后端已经有统一插件版本治理结果，不再靠零散字段拼装
- 安装、升级和回滚前已经接入宿主兼容性判断
- 用户可以手动升级和手动回滚插件
- 升级默认不破坏现有启停和配置状态
- 前端能够清楚展示版本状态、阻断原因和升级入口
- 文档明确写清当前边界：有手动治理，没有自动升级
