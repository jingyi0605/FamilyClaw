# 设计文档 - 用户端 AI 配置中心下沉

状态：Draft

## 1. 概述

### 1.1 目标

- 让 `user-web` 具备正式可用的 AI 配置中心
- 把 AI 供应商配置通过适配器层抽象出来
- 把 Agent 管理能力从只读展示升级为正式可编辑能力
- 提供一个既能在向导中使用，也能在 AI 配置中心复用的首个管家对话式创建流程

### 1.2 覆盖需求

- `requirements.md` 需求 1：用户端正式供应商配置入口
- `requirements.md` 需求 2：供应商适配器层
- `requirements.md` 需求 3：用户端正式 Agent 配置入口
- `requirements.md` 需求 4：首个管家 Agent 对话式创建
- `requirements.md` 需求 5、6：正式入口和对话主链路联动

### 1.3 技术约束

- `admin-web` 仍可保留调试页，但不再是正式产品依赖
- 供应商调用仍复用 `ai_gateway`
- Agent 主数据仍复用 `003.1` 已定义模型
- `003.2` 的强制向导会复用这里的供应商和 Agent 能力

## 2. 架构

### 2.1 系统结构

整体上分三层：

1. **供应商适配器层**：定义每家供应商需要哪些配置项、如何校验、如何映射到统一配置模型
2. **配置中心层**：`user-web` 中的供应商管理和 Agent 管理 UI
3. **对话式创建层**：把首个管家 Agent 的创建做成一段可复用流程

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| 供应商适配器注册表 | 维护可选供应商和字段定义 | 供应商代码 | 字段 schema、说明、校验规则 |
| 用户端供应商配置页 | 展示供应商列表和表单 | 用户输入 | 标准化供应商配置 |
| Agent 配置中心 | 展示和编辑 Agent | Agent 读写数据 | 保存后的 Agent 配置 |
| 首个管家对话式创建器 | 用对话采集首个管家资料 | 用户自然语言输入 | 结构化 Agent 创建 payload |
| 配置联动层 | 把配置结果同步到对话主链路 | 供应商 / Agent 配置变化 | 对话页可用状态变化 |

### 2.3 关键流程

#### 2.3.1 新增 AI 供应商

1. 用户进入 AI 配置中心
2. 选择供应商类型，如 `ChatGPT`、`GLM`、`硅基流动`、`KIMI`、`MINIMAX`
3. 页面根据适配器定义渲染字段
4. 用户提交配置
5. 系统做字段校验和可用性校验
6. 保存为统一供应商配置

#### 2.3.2 配置 Agent

1. 用户进入 Agent 列表
2. 选择已有 Agent 或新增 Agent
3. 编辑基础资料、人格摘要、运行时策略、成员认知
4. 保存后刷新聚合读模型
5. 对话页读取最新结果

#### 2.3.3 首个管家对话式创建

1. 系统发起引导式对话
2. 用户描述管家名字、性格、说话风格、服务重点
3. 系统提炼成结构化字段
4. 用户确认
5. 系统创建默认管家 Agent

## 3. 组件和接口

### 3.1 核心组件

- `provider_adapter_registry`：供应商适配器注册表
- `ProviderProfileForm`：基于适配器动态渲染表单
- `AiConfigCenterPage`：正式 AI 配置中心页面
- `AgentEditor`：Agent 详情配置编辑器
- `ButlerBootstrapConversation`：首个管家对话式创建器

### 3.2 数据结构

#### 3.2.1 供应商适配器定义

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `provider_code` | string | 是 | 供应商代码 | 唯一 |
| `display_name` | string | 是 | 显示名称 | 非空 |
| `field_schema` | object[] | 是 | 需要的字段定义 | 至少一个字段 |
| `validation_rules` | object | 是 | 校验规则 | 标准结构 |
| `capabilities` | string[] | 是 | 支持的能力，如 chat、image | 固定枚举集合 |

#### 3.2.2 用户端供应商配置

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `profile_id` | string | 是 | 配置 ID | 唯一 |
| `household_id` | string | 是 | 家庭 ID | 必须存在 |
| `provider_code` | string | 是 | 供应商代码 | 必须已注册 |
| `display_name` | string | 是 | 用户侧显示名称 | 非空 |
| `config_payload` | object | 是 | 适配器配置结果 | 结构依适配器 |
| `enabled` | boolean | 是 | 是否启用 | 固定布尔 |
| `last_check_status` | string | 否 | 最近校验结果 | 可空 |

#### 3.2.3 首个管家创建草稿

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `household_id` | string | 是 | 家庭 ID | 必须存在 |
| `display_name` | string | 是 | 管家名称 | 非空 |
| `role_summary` | string | 是 | 角色摘要 | 非空 |
| `speaking_style` | string | 否 | 说话风格 | 可空 |
| `personality_traits` | string[] | 是 | 人格特征 | 至少 1 项 |
| `service_focus` | string[] | 是 | 服务重点 | 至少 1 项 |

### 3.3 接口契约

#### 3.3.1 获取供应商适配器列表

- 类型：HTTP
- 路径：`GET /api/v1/ai-config/provider-adapters`
- 输入：无
- 输出：适配器定义列表
- 校验：仅返回允许给用户端使用的适配器
- 错误：`500` 适配器注册表异常

#### 3.3.2 用户端供应商配置读写

- 类型：HTTP
- 路径：
  - `GET /api/v1/ai-config/{household_id}/provider-profiles`
  - `POST /api/v1/ai-config/{household_id}/provider-profiles`
  - `PUT /api/v1/ai-config/{household_id}/provider-profiles/{profile_id}`
- 输入：供应商配置 payload
- 输出：标准化供应商配置
- 校验：依适配器字段定义执行
- 错误：`400` 参数非法；`409` 重复或不可用

#### 3.3.3 Agent 配置读写

- 类型：HTTP
- 路径：沿用并扩展 `003.1` 的 `ai-config` 接口
- 输入：Agent 基础资料、人格、策略、成员认知
- 输出：更新后的 Agent 读模型
- 校验：字段合法、同家庭约束成立
- 错误：`404` Agent 不存在；`400` 参数非法

#### 3.3.4 首个管家对话式创建

- 类型：HTTP
- 路径：
  - `POST /api/v1/ai-config/{household_id}/butler-bootstrap/sessions`
  - `POST /api/v1/ai-config/{household_id}/butler-bootstrap/sessions/{session_id}/messages`
  - `POST /api/v1/ai-config/{household_id}/butler-bootstrap/sessions/{session_id}/confirm`
- 输入：自然语言消息和确认操作
- 输出：对话回复、结构化草稿、最终创建结果
- 校验：家庭至少已有可用供应商
- 错误：`409` 前置条件未满足

## 4. 数据与状态模型

### 4.1 数据关系

- 供应商适配器是产品层定义，不是用户数据
- 用户真正保存的是“某家庭下的供应商配置”
- Agent 配置继续复用 `003.1` 主数据模型
- 首个管家对话式创建只是一个创建流程，不是第二套 Agent 模型

### 4.2 状态流转

#### 4.2.1 供应商配置状态

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `draft` | 已填写未校验 | 创建草稿 | 提交校验 |
| `valid` | 校验通过可用 | 校验成功 | 停用或配置变更 |
| `invalid` | 校验失败 | 校验失败 | 用户修正并重试 |
| `disabled` | 手动停用 | 用户停用 | 重新启用 |

#### 4.2.2 首个管家创建状态

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `collecting` | 采集资料中 | 会话开始 | 草稿生成 |
| `reviewing` | 等待用户确认 | 草稿已生成 | 用户确认或返回修改 |
| `completed` | 管家已创建 | 创建成功 | 一般不回退 |

## 5. 关键设计决策

### 5.1 必须有供应商适配器层

如果直接把不同供应商字段硬写在页面里：

- 今天支持 5 家，页面就长成怪物
- 明天加第 6 家，就得继续补丁

所以必须有一层统一适配器定义，页面只是按定义渲染和校验。

### 5.2 正式入口在 user-web，不再拿 admin-web 兜底

`admin-web` 可以继续保留，但定位必须收窄成：

- 开发调试
- 联调排错
- 后台治理

不能再把真实用户配置入口推给它。

### 5.3 首个管家创建要和普通 Agent 编辑分开

首个管家的创建本质上是 onboarding 式流程。  
它和日常编辑 Agent 不是一回事。

所以：

- 创建时用对话式流程
- 创建后再进入普通 Agent 编辑页维护

## 6. 风险与边界

### 6.1 风险

- 风险 1：把供应商差异硬编码到页面，后续维护爆炸
- 风险 2：把首个管家创建和普通 Agent 编辑揉成一个巨大表单
- 风险 3：页面看起来能配，结果保存后并不影响对话主链路

### 6.2 这份设计先不做什么

- 先不做无限多供应商高级参数暴露
- 先不做复杂多模型自动路由编排
- 先不做图片外观生成流程，那仍属于其他 spec

## 7. 验证策略

- 用户端能新增、编辑、启用、停用供应商配置
- 用户端能新增、编辑 Agent
- 首个管家能通过对话式流程创建
- 配置结果能进入对话页主链路
