# Spec 003.3 - 用户端 AI 配置中心下沉

状态：Draft

## 这份 Spec 到底要解决什么

现在 AI 相关能力有个很糟糕的问题：

- `user-web` 的 AI 配置页只是个样子货
- 真正的 AI 供应商配置和 Agent 配置都还在 `admin-web`
- 对真实用户来说，这等于正式产品链路根本没闭环

而我们已经明确了一个原则：

**`admin-web` 只是开发调试工具，不是正式产品入口。**

所以这份 Spec 要做的事情很明确：

1. 把 AI 供应商配置正式下沉到 `user-web`
2. 把 Agent 配置正式下沉到 `user-web`
3. 让首个管家 Agent 的创建既能在向导里使用，也能在正式 AI 配置中心里继续维护
4. 建一层清晰的供应商适配器模型，别把各家供应商字段硬糊在一起

## 规划结论

### 【核心判断】

✅ 值得做，而且必须做。  
如果 `user-web` 不能完整配置 AI，产品就只是个 demo。

### 【关键洞察】

- 真正要下沉的不是几个表单，而是**完整的 AI 配置能力**
- 核心不是“支持多少家模型”，而是先把**供应商适配器层**建对
- Agent 配置要区分：
  - 正式资料配置
  - 运行时策略配置
  - 首个管家的对话式创建

### 【Linus式方案】

1. 先做统一供应商适配器层
2. 再做用户端供应商配置入口
3. 再做用户端 Agent 列表和详情配置
4. 再把首个管家创建做成可复用的对话式流程

## 这份 Spec 包含什么

- `requirements.md`：定义 user-web 中完整 AI 配置能力的产品要求
- `design.md`：定义供应商适配器、配置读写模型、Agent 配置页和对话式创建流程
- `tasks.md`：定义落地顺序、文件和验证方法

## 这份 Spec 明确不做什么

- 不在这里定义新家庭强制向导的准入规则，那些由 `003.2` 承担
- 不重做 `ai_gateway` 已有的底层供应商调用框架，只在正式产品入口上补齐
- 不把 `admin-web` 当正式交付页面继续扩写

## 依赖与协同

- **依赖 `Spec 003.1`**：复用多 Agent 数据模型和对话主链路
- **协同 `Spec 003.2`**：作为新家庭初始化向导中的 AI 配置能力提供方
- **协同 `ai_gateway`**：继续走统一的供应商调用和模型路由
