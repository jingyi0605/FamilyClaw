# Spec 001.3 - 家庭地区模型与插件接口

状态：Draft

## 这份 Spec 解决什么问题

现在家庭资料里只有一个 `city` 字段，这个设计太粗，而且以后一定会坏。

- 只存城市，没法稳定落到区县，天气、地区问答、区域提醒都会变模糊
- 只存展示文本，后面接天气、地图、节假日、地域知识时，很难可靠匹配
- 如果以后要支持海外地区，再继续往 `households` 里硬塞字段，只会把模型搞烂

这份 Spec 的目标很直接：先把“家庭属于哪个地区”这件事做成正式模型，第一版只落中国大陆，最细到区县；同时把这套地区能力暴露给插件系统，后面天气插件、地区问答插件、海外地区插件都能复用。

## 这次要做到什么

- 给家庭增加结构化地区绑定，不再只靠 `city` 文本
- 第一版内置中国大陆省 / 市 / 区县目录和编码
- 新老家庭都能兼容，旧的 `city` 不会突然失效
- 插件能通过统一接口拿到家庭地区上下文
- 后续新增其他国家和地区时，走插件扩展，不改核心家庭绑定模型

## 这次明确不做什么

- 不做全球全量行政区目录
- 不做地图展示、坐标拾取、逆地理编码页面
- 不做复杂地理围栏、轨迹、定位历史
- 不做天气业务本身，只把天气等插件需要的地区接口先铺好

## 为什么现在做

因为这不是一个“以后也许会用到”的装饰字段，而是天气、地区问答、节假日、生活服务这类能力的基础输入。

如果继续只用 `city` 糊弄，后面会出现三种典型烂问题：

- 同名城市或区县匹配不准
- 插件各自维护一套地区解析逻辑，越写越散
- 一旦要支持海外地区，就得重改家庭模型和插件接口

## 主要文档

- `requirements.md`：说清楚地区能力要覆盖什么，什么算完成
- `design.md`：说清楚地区目录、家庭绑定、兼容策略、插件接口怎么落地
- `tasks.md`：说清楚先建数据模型、再接家庭接口、再接插件接口的顺序
- `docs/README.md`：给后续补数据导入说明、接口样例、联调说明留位置
