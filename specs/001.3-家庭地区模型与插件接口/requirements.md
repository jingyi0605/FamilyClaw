# 需求文档 - 家庭地区模型与插件接口

状态：Draft

## 简介

项目里已经开始依赖家庭所在地做天气、初始化向导和上下文问答，但当前家庭资料只有一个 `city` 文本字段，这个结构撑不住后面的真实需求。

这次不追求做全球地理平台，只先把最小但正确的事情做好：

- 家庭地区改成正式的结构化数据
- 第一版只支持中国大陆，并细到区县
- 现有接口和已有家庭继续兼容
- 插件系统能拿到统一地区上下文
- 未来其他国家和地区通过插件扩展，而不是重改家庭主模型

## 术语表

- **System**：FamilyClaw 的家庭主数据、地区服务和插件系统。
- **地区目录**：系统可识别的一套标准地区节点数据，至少包含国家、省、市、区县层级关系和稳定编码。
- **地区提供方**：负责提供地区目录和解析规则的实现。第一版内置中国大陆提供方，后续其他国家和地区通过插件接入。
- **家庭地区绑定**：家庭和某个标准地区节点之间的正式关联，不再只是一个展示文本。
- **地区上下文**：给天气、地区问答、生活服务插件使用的标准地区信息，至少包含国家、提供方、当前选中节点及其上级路径。

## 范围说明

### In Scope

- 设计家庭的结构化地区绑定模型
- 第一版内置中国大陆省 / 市 / 区县目录
- 新建家庭、编辑家庭、初始化向导接入地区选择
- 保留现有 `city` 兼容字段和迁移策略
- 为插件系统暴露统一地区查询和上下文接口
- 设计地区提供方插件扩展点，支持未来新增其他国家和地区

### Out of Scope

- 中国大陆以外地区的真实目录数据导入
- 地图选点、经纬度反查、POI 搜索
- 天气插件、节假日插件等具体业务插件实现
- 多地址家庭、常住地 / 工作地 / 学校地址等复杂地点模型

## 需求

### 需求 1：家庭必须绑定结构化地区，最细支持到区县

**用户故事：** 作为家庭管理员，我希望家庭资料能明确到区县，而不是只写一个城市名字，这样天气、地区问答和本地化服务才不会猜错地方。

#### 验收标准

1. WHEN 管理员创建或更新家庭资料 THEN System SHALL 支持保存结构化地区信息，至少包含 `provider_code`、`country_code`、`region_code`、`admin_level` 和上级路径信息。
2. WHEN 家庭已经绑定地区 THEN System SHALL 能稳定返回省、市、区县三级名称和编码，而不是只返回一个模糊文本。
3. WHEN 管理员选择的地区不是区县级节点 THEN System SHALL 明确说明当前只接受细化到区县的正式绑定。

### 需求 2：第一版只支持中国大陆地区目录，但要用稳定编码而不是自由文本

**用户故事：** 作为系统维护者，我希望第一版只做中国大陆，并且使用稳定地区编码，这样范围可控，也方便后续天气和知识服务可靠匹配。

#### 验收标准

1. WHEN 第一版系统提供地区选择 THEN System SHALL 只暴露中国大陆的省、市、区县目录。
2. WHEN 用户提交地区绑定 THEN System SHALL 校验该地区编码存在于中国大陆目录中，并且父子层级关系正确。
3. WHEN 请求了目录中不存在的地区编码或跨层级错误组合 THEN System SHALL 拒绝保存并返回明确错误。

### 需求 3：现有家庭和现有接口必须有兼容路径

**用户故事：** 作为已经在用系统的用户，我不希望因为地区模型升级，现有家庭列表、初始化流程和接口调用突然坏掉。

#### 验收标准

1. WHEN 旧家庭只有 `city` 字段、没有正式地区绑定 THEN System SHALL 继续允许读取和编辑家庭资料，不因为新字段缺失直接报错。
2. WHEN 新接口返回家庭资料 THEN System SHALL 同时返回结构化地区对象和兼容用的 `city` 展示值。
3. WHEN 旧客户端仍然只识别 `city` THEN System SHALL 继续提供可读的城市展示文本，直到客户端完成迁移。
4. WHEN 系统判断新家庭初始化是否完成 THEN System SHALL 以正式地区绑定是否完成为准，而不是只看 `city` 有没有值。

### 需求 4：地区能力必须通过统一接口提供给业务模块和插件

**用户故事：** 作为天气插件或地区问答插件开发者，我希望通过统一接口拿到家庭地区上下文，而不是自己再去猜家庭资料和地区编码。

#### 验收标准

1. WHEN 业务模块或插件请求家庭地区上下文 THEN System SHALL 返回标准结构，至少包含国家、提供方、当前区县节点和省 / 市路径信息。
2. WHEN 前端或后端需要做地区选择 THEN System SHALL 提供统一的目录查询能力，至少支持按父节点列子节点和按关键词搜索。
3. WHEN 家庭尚未绑定正式地区 THEN System SHALL 返回明确的未配置状态，而不是让调用方从空字符串里自己猜。
4. WHEN 天气、地区问答或其他地区相关能力读取家庭所在地 THEN System SHALL 通过统一地区上下文读取，不再把 `city` 当正式判断依据。

### 需求 5：其他国家和地区必须能通过插件扩展接入

**用户故事：** 作为系统维护者，我希望未来接入日本、美国或其他地区时，不需要重改家庭主数据表和插件调用方式。

#### 验收标准

1. WHEN 新增一个非中国大陆地区提供方 THEN System SHALL 允许它按统一地区提供方接口接入，而不是要求重写家庭绑定模型。
2. WHEN 不同地区提供方共存 THEN System SHALL 能用 `provider_code` 和 `country_code` 区分来源，不把所有地区节点混成一锅。
3. WHEN 某个地区提供方暂时不可用 THEN System SHALL 继续保留家庭已绑定的地区快照，并返回明确降级信息。

## 非功能需求

### 非功能需求 1：可维护性

1. WHEN 后续新增其他国家和地区 THEN System SHALL 主要新增地区提供方实现和数据导入逻辑，而不是修改家庭主数据核心字段设计。
2. WHEN 开发者排查地区问题 THEN System SHALL 能看到地区绑定、目录查询和插件上下文输出的关键日志。

### 非功能需求 2：兼容性

1. WHEN 地区模型上线 THEN System SHALL 不破坏现有家庭列表、家庭详情、初始化向导和依赖 `city` 的旧逻辑。
2. WHEN 新旧客户端并存 THEN System SHALL 在过渡期同时支持结构化地区对象和 `city` 兼容字段。
3. WHEN 旧家庭仍未补录正式地区 THEN System SHALL 允许非地区强依赖功能继续运行，并让天气、地区问答这类强依赖能力返回明确提示或补录引导。

### 非功能需求 3：性能

1. WHEN 前端按父节点查询中国大陆某一级地区目录 THEN System SHALL 在正常数据量下快速返回结果，不要求全量扫表。
2. WHEN 插件读取家庭地区上下文 THEN System SHALL 复用标准查询结果，不为每次调用重复拼装整棵树。

### 非功能需求 4：治理

1. WHEN 系统导入或更新中国大陆地区目录 THEN System SHALL 记录使用的数据源版本和导入时间，避免后面连自己用了哪套编码都说不清。
2. WHEN 地区目录编码发生调整 THEN System SHALL 优先保证已有家庭绑定可追溯，不允许一次更新把历史家庭全部打成脏数据。
3. WHEN 开放地区提供方插件 THEN System SHALL 在插件规范里明确声明地区提供方的 manifest 和接口要求，避免后面每个插件各写一套。

## 成功定义

- 新建家庭和编辑家庭可以稳定绑定到中国大陆区县
- 家庭资料接口同时返回结构化地区和兼容 `city` 值
- 插件可以通过统一接口拿到家庭地区上下文
- 第一版没有把全球地区一次性做烂，后续海外接入有清楚扩展点
