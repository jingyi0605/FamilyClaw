# Spec 001.8 - 项目版本治理与 GitHub Release 发布
状态：Draft

## 这份 Spec 解决什么问题？

现在仓库里的版本号是散的：

- `api-server` 有自己的 `0.1.0`
- `open-xiaoai-gateway` 有自己的 `0.1.0`
- `user-app` 和 `packages/*` 也各写了一份 `0.1.0`
- 后端运行时还单独维护了 `settings.app_version`

这套东西第一次发版还能凑合，第二次开始就会互相打架。更糟的是，插件兼容判断已经直接依赖宿主版本；如果宿主版本漂了，插件升级阻断和兼容性提示都会跟着变假。

这份 `001.8` 要做的事情很直接：

1. 把整个项目的应用版本收口成一个唯一事实源
2. 用 Git tag、GitHub Release 和 GHCR 镜像把发布链路收口
3. 让运行中的后端、前端、镜像和 Release 能对上同一个版本
4. 把“应用版本”和“数据库 schema 版本”的关系写清楚，避免以后升级靠猜

## 先说结论

- 应用版本应该统一使用 SemVer，例如 `0.3.0`、`0.3.1`、`0.4.0-rc.1`
- GitHub Release 应该作为发布事实源，不需要额外部署版本服务
- 数据库不应该再发明一套独立的 `051`、`052` 这种版本号来充当主事实源
- 正确做法是：继续以 Alembic revision / head 作为数据库 schema 真相，再由每个应用 Release 显式记录自己对应的 schema head、升级边界和是否需要人工迁移

## 为什么不建议再造一套数据库版本号？

用户提出的想法不是没道理，它想解决的是真问题：应用升级时，怎么知道数据库该升到哪里、能不能从当前库直接升。

但如果再手工维护一套“应用版本 `0.1.0` 对应数据库版本 `051`”的平行编号，通常会带来更多垃圾：

1. 数据库本来已经有 Alembic revision，这就是现成的、可执行的 schema 版本线
2. 很多应用补丁版根本不改 schema，两个应用版本可能对应同一个 Alembic head
3. Alembic 允许 merge revision，多 head 收口时，单个整数版本号表达能力反而更差
4. 你得同时维护两套数据库版本语义，迟早漂

所以这份 Spec 的判断是：

- 不采用“独立数据库版本号 051”作为主方案
- 采用“应用 Release 清单显式记录 schema 兼容信息”的方案

## 这次明确不做什么？

- 不做在线版本服务
- 不做必须联网的自动更新系统
- 不做自建发布后台
- 不把数据库 schema 版本重新发明成另一套手工编号体系
- 不在第一版里做复杂的差量升级平台
