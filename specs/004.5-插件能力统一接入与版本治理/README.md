# Spec 004.5 - 插件能力统一接入与版本治理

状态：Draft

## 这份 Spec 解决什么问题

当前项目里的“插件化”已经做了不少，但还不彻底。

现在至少有三块东西还没完全站到同一套规则上：

- 主题系统还是前端内置常量，没有进插件注册和插件状态体系
- AI 供应商虽然长得像插件，但还在走 `ai_gateway` 里的专用注册表，不受通用插件启停规则约束
- 设备与集成、通讯平台、计划任务、语言包、仪表盘卡片这些前后端入口，对插件启用 / 禁用状态的处理还没有完全统一

除此之外，插件还有一个老问题一直没彻底收口：

- `manifest.version` 这个字段有了
- 但“版本怎么管理、怎么比对、怎么发现兼容性问题、怎么给前后端展示”这套机制并不完整

这份 Spec 不是要再发明一套新插件系统，而是把已经存在但还散着的能力收回到一套干净规则里。

## 这次要做到什么

- 把主题能力接入插件系统，变成可注册、可识别、可扩展的插件能力
- 把 AI 供应商接入通用插件系统，不再继续维持一套平行的专用“伪插件”注册表
- 把插件启用 / 禁用规则真正落实到各个前后端入口，不再每个页面、每个模块各写各的判断
- 盘清当前插件版本机制到底有什么、缺什么，并补足最基本的版本治理能力
- 把插件规范文档、旧 Spec、开发者手册同步更新到同一套说法，避免后面继续看旧文档做错

## 这次明确不做什么

- 不做远程插件市场安装平台
- 不做插件签名和沙箱安全体系全量升级
- 不做主题视觉方案重设计
- 不做 AI 供应商业务能力重构，只处理它如何接入插件系统和插件状态体系
- 不做“所有旧插件一夜之间重写”这种不现实的动作

## 为什么现在做

因为继续拖下去，只会继续长出更多平行体系：

- 一套叫“插件”
- 一套叫“供应商适配器”
- 一套叫“主题选项”
- 一堆入口各自决定禁用到底是什么意思

这就是坏品味。数据结构分裂，规则也就跟着分裂，后面每接一条新业务线都得补一轮状态判断。

现在必须把这件事讲清楚：

1. 什么能力算插件
2. 插件状态到底谁说了算
3. 禁用后哪些东西必须停
4. 版本字段到底只是展示，还是要承担治理责任

## 和旧 Spec 的关系

这份 Spec 不是替代下面这些旧 Spec，而是把它们继续往前推一层：

- `../004.2-插件系统与外部能力接入/`
  - 继续承接插件底座，但本次把“哪些能力必须接入通用插件系统”讲完整
- `../004.2.2-插件禁用能力统一与停用收口/`
  - 继续承接启停统一规则，但这次把主题、AI 供应商和更多入口一起纳入范围
- `../004.3-插件开发规范与注册表/`
  - 本次要同步更新开发规范、manifest 约束、开发者手册引用
- `../004.4-用户前端插件管理与插件市场/`
  - 本次要同步更新前端消费规则，让主题、AI 供应商和已有插件入口都尊重同一套状态
- `../001.5-AI供应商管理插件化与模型摘要重构/`
  - 那一轮只做了 AI provider 专用插件化；这一轮要把它正式并入通用插件系统

如果你要实现这份 Spec，建议先按这个顺序看：

1. `../004.2.2-插件禁用能力统一与停用收口/`
2. `../004.3-插件开发规范与注册表/`
3. `../004.4-用户前端插件管理与插件市场/`
4. `../001.5-AI供应商管理插件化与模型摘要重构/`

## 主要文档

- `requirements.md`：说清楚这次到底要统一哪些能力、哪些入口、哪些版本规则
- `design.md`：说清楚通用插件模型怎么扩到主题和 AI 供应商、各入口怎么收口、版本治理怎么补
- `tasks.md`：说清楚先做模型和后端，再做前端入口，再做文档和版本治理校准
- `docs/README.md`：记录这份 Spec 后续补充文档应该放哪里、主要写什么
