## 1. 概述

### 1.1 目标

- 把主题和 AI 供应商能力并入通用插件系统，结束平行注册表
- 把插件启停规则落实到所有核心前后端入口
- 建立最小但真实可用的插件版本治理边界
- 把插件规范文档、旧 Spec 和开发者手册更新到同一套说法

### 1.2 覆盖需求

- `requirements.md` 需求 1：主题和 AI 供应商必须进入通用插件系统
- `requirements.md` 需求 2：所有前后端入口必须统一尊重插件启停状态
- `requirements.md` 需求 3：主题和 AI 供应商的前后端消费入口必须完成插件状态适配
- `requirements.md` 需求 4：插件版本管理机制必须被盘点并补齐基本治理能力
- `requirements.md` 需求 5：插件规范文档、旧 Spec 和开发者手册必须同步更新
- `requirements.md` 需求 6：改造必须保持现有用户空间兼容

### 1.3 技术约束

- 后端：FastAPI + SQLAlchemy + 现有 `app.modules.plugin`、`app.modules.ai_gateway`
- 前端：`apps/user-app` + `packages/user-core` + `packages/user-ui`
- 数据存储：当前插件状态、插件配置、AI provider profile、现有本地主题存储
- 认证授权：沿用现有家庭管理员和成员权限边界
- 外部依赖：现有插件 manifest、AI provider adapter 元数据、前端主题资源

### 1.4 当前代码真相（2026-03-17 盘点）

- 通用插件注册表现在只覆盖 `app/plugins/builtin` 里的 manifest 和 `plugin_mounts` 挂载记录，类型只认连接器、动作、语言包、地区 provider、通道这些现有类型。
- 主题还完全在前端本地体系里：`packages/user-core/src/state/theme.ts` 维护主题 id 和说明，`apps/user-app/src/runtime/h5-shell/theme/ThemeProvider.tsx` 直接读写浏览器 `localStorage`，后端没有主题注册出口。
- AI 供应商虽然已经带上了 `plugin_id`、`plugin_name` 这类字段，但这些字段来自 `app.modules.ai_gateway.provider_adapter_registry` 的专用字典，不来自 `app.modules.plugin` 的统一注册结果。
- 当前版本字段也还是散的：插件系统只有 `manifest.version`，挂载记录不单独存安装版本；AI 供应商没有插件版本字段，只有上游 API 的 `api_version`；主题更没有版本概念。
- 这次阶段 1 不做行为改造，先把这些真相写清楚，再用统一模型把后面的接入边界钉住。详细盘点见 `docs/20260317-阶段1现状盘点与统一模型草图.md`。

## 2. 架构

### 2.1 系统结构

这次不是重写插件底座，而是在现有插件底座上补两层能力：

1. **能力接入层**
   - 把主题和 AI 供应商接成通用插件类型或等价能力类型
   - 让注册、状态、版本、配置入口都回到同一模型

2. **状态执法层**
   - 让新增入口、存量对象、执行入口统一尊重 `enabled` / `disabled_reason`
   - 统一后端错误语义和前端禁用展示

3. **版本治理层**
   - 不再把 `version` 仅仅当展示文案
   - 补上兼容性声明、已安装版本、升级提示所需的数据结构和接口边界

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| `app.modules.plugin` | 统一插件注册、状态计算、配置和版本元数据出口 | manifest、挂载记录、家庭覆盖状态 | `PluginRegistryItem`、统一配置和状态结果 |
| `app.modules.ai_gateway` | 继续承接 AI provider 业务逻辑，但不再维护平行注册表语义 | AI 供应商插件元数据、provider profile | 供应商列表、配置表单、运行时配置 |
| `theme runtime / user-core` | 消费主题插件元数据和主题资源，处理兼容降级 | 主题插件列表、当前主题选择 | 可选主题列表、主题资源 |
| `integration / channel / scheduler / device / dashboard` | 消费统一插件状态，执行入口统一执法 | 插件注册快照、可用性检查 | 可新增列表、存量状态、执行结果 |
| 文档体系 | 更新规范、旧 Spec、开发者手册 | 新规则、新能力边界 | 一致的文档入口和引用关系 |

### 2.3 关键流程

#### 2.3.1 主题与 AI 供应商接入通用插件系统

1. 定义主题插件和 AI 供应商插件需要的 manifest 能力字段。
2. 扩展插件注册读取逻辑，让这两类能力进入统一 `PluginRegistryItem` 输出。
3. 为现有主题列表和 AI provider adapter 提供兼容适配层。
4. 前端和业务模块改为消费统一插件结果，而不是再读旧常量或平行注册表。

#### 2.3.2 插件禁用在新增入口和执行入口生效

1. 新增入口默认只展示可新增使用的插件。
2. 已有对象继续显示，但状态标记为“插件已禁用”。
3. 执行入口统一走 `require_available_household_plugin(...)`。
4. worker、计划任务、自动触发在真正执行前再次校验。

#### 2.3.3 插件版本治理补齐

1. 盘点现有 `manifest.version`、挂载版本展示和接口返回能力。
2. 定义统一版本元数据输出，包括插件版本、安装版本、兼容性声明和升级提示字段。
3. 前后端按统一结构展示和校验版本信息。
4. 开发者文档明确哪些字段必填、哪些只做展示、哪些参与兼容性判断。

## 3. 组件和接口

### 3.1 核心组件

覆盖需求：1、2、3、4、5、6

- **插件能力类型扩展**：让主题和 AI 供应商进入统一插件模型
- **统一插件状态出口**：所有业务入口只认 `PluginRegistryItem.enabled` 和相关禁用原因
- **版本元数据出口**：统一返回插件版本、兼容性和升级提示字段
- **入口适配层**：让集成页、主题页、AI 供应商页、计划任务、仪表盘等页面和后端模块都消费统一结果
- **文档更新矩阵**：把旧 Spec、开发手册和新规则关系写清楚

### 3.2 数据结构

覆盖需求：1、2、3、4、6

#### 3.2.1 插件类型扩展

建议在当前 `PluginManifestType` 基础上扩展两个能力类型：

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `theme-pack` | `PluginManifestType` | 是 | 主题资源插件 | 只承载主题元数据和资源，不承载任意可执行代码 |
| `ai-provider` | `PluginManifestType` | 是 | AI 供应商插件 | 提供 provider adapter 元数据、配置 schema、兼容性说明 |

说明：

- 这不要求主题和 AI 供应商一定都变成“可执行插件”
- 但它们必须进入统一注册、状态、版本、配置消费模型

#### 3.2.2 PluginRegistryItem 扩展

建议在统一插件读取结果中补齐下面这些字段：

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `enabled` | `bool` | 是 | 当前家庭最终可用状态 | 已存在，继续作为唯一真相 |
| `disabled_reason` | `str \| null` | 否 | 禁用原因 | 已存在，继续统一消费 |
| `version` | `str` | 是 | 插件声明版本 | 已存在 |
| `installed_version` | `str \| null` | 否 | 当前已安装版本 | 对挂载插件和迁移能力更有意义 |
| `compatibility` | `object \| null` | 否 | 兼容性声明 | 例如最低宿主版本、配置 schema 兼容边界 |
| `update_state` | `str \| null` | 否 | 是否可升级、是否版本不一致 | 用于前后端展示 |

#### 3.2.3 主题插件能力结构

主题插件至少需要下面这些能力字段：

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `theme_id` | `str` | 是 | 主题唯一标识 | 与旧 `ThemeId` 做兼容映射 |
| `display_name` | `str` | 是 | 主题展示名 | 前端可直接显示 |
| `description` | `str \| null` | 否 | 主题说明 | 人话，不写黑话 |
| `tokens_resource` | `str` | 是 | 主题 token 资源路径 | 可被前端加载 |
| `preview` | `object` | 否 | 预览信息 | 用于主题选择页展示 |
| `fallback_theme_id` | `str \| null` | 否 | 降级主题 | 当前主题被禁用时用于兼容回退 |

#### 3.2.4 AI 供应商插件能力结构

AI 供应商插件至少需要下面这些能力字段：

| 字段 | 类型 | 必填 | 说明 | 约束 |
| --- | --- | --- | --- | --- |
| `adapter_code` | `str` | 是 | 供应商适配器编码 | 与现有 provider profile 兼容 |
| `display_name` | `str` | 是 | 展示名称 | 沿用现有页面展示 |
| `field_schema` | `list` | 是 | 动态表单定义 | 与现有 provider adapter 模式兼容 |
| `supported_model_types` | `list[str]` | 是 | 支持模型类型 | 沿用现有摘要展示 |
| `llm_workflow` | `str` | 是 | 调用工作流标识 | 用于兼容现有逻辑 |
| `runtime_capability` | `object` | 否 | 运行时能力说明 | 例如 transport、api family |

### 3.3 接口契约

覆盖需求：1、2、3、4、6

#### 3.3.1 插件列表与插件详情接口

- 类型：HTTP
- 路径或标识：沿用当前插件列表接口，并扩展主题插件和 AI 供应商插件输出
- 输入：家庭上下文、筛选条件
- 输出：统一 `PluginRegistryItem` 列表
- 校验：不同类型插件必须返回可消费的能力字段
- 错误：禁用状态不在这里报错，而是通过状态字段表达

#### 3.3.2 新增入口目录接口

- 类型：HTTP / Function
- 路径或标识：设备与集成目录、AI 供应商目录、主题列表目录
- 输入：家庭上下文、搜索条件
- 输出：仅可新增使用的插件，或明确禁用且不可选的项
- 校验：默认不暴露 `enabled=false` 的新增候选
- 错误：若指定禁用插件创建实例，返回 `409/plugin_disabled`

#### 3.3.3 执行入口统一校验

- 类型：Function
- 路径或标识：`require_available_household_plugin(...)`
- 输入：家庭 ID、插件 ID、插件类型、触发方式
- 输出：可执行插件对象
- 校验：插件存在、插件类型匹配、触发方式匹配、插件当前启用
- 错误：统一返回 `409/plugin_disabled` 或现有结构化错误

#### 3.3.4 主题和 AI 供应商消费接口

- 类型：HTTP / Function
- 路径或标识：主题列表、AI provider adapter 列表、配置入口
- 输入：家庭上下文、筛选条件
- 输出：来自统一插件系统的能力结果
- 校验：不得继续直接消费旧常量或平行注册表作为最终真相
- 错误：当依赖插件被禁用时，返回禁用状态和禁用原因

## 4. 数据与状态模型

### 4.1 数据关系

本次的核心关系是：

- **通用插件注册表** 是主题插件、AI 供应商插件和现有业务插件的统一入口
- **插件状态** 由插件模块统一计算
- **业务模块** 只消费最终状态，不重算
- **版本元数据** 也由统一插件注册结果输出

旧体系和新体系的关系：

- 旧主题常量表变成主题插件的兼容源
- 旧 AI provider adapter 注册表变成 AI 供应商插件的兼容源或迁移输入
- 迁移期允许兼容读取，但不允许继续把旧平行结构当长期真相

### 4.2 状态流转

插件状态统一继续按下面的思路走：

| 状态 | 含义 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `enabled` | 当前家庭可新增、可执行、可消费 | 基础状态启用且家庭未禁用 | 基础状态关闭或家庭禁用 |
| `disabled` | 当前家庭不可新增、不可执行、不可自动触发 | 基础状态关闭或家庭禁用 | 状态被重新启用 |

对于存量对象：

- 对象本身不进入“删除”状态
- 但关联插件禁用后，其执行动作必须转入不可用

## 5. 错误处理

### 5.1 错误类型

- `plugin_disabled`：插件存在，但当前家庭不可用
- `plugin_type_not_supported`：入口使用了错误的插件类型
- `plugin_compatibility_invalid`：插件版本或兼容性声明不满足要求
- `plugin_manifest_invalid`：插件声明缺失新能力所需字段
- `plugin_migration_incomplete`：迁移尚未完成，当前入口无法安全使用

### 5.2 错误响应格式

```json
{
  "detail": "插件当前已禁用，不能继续执行。",
  "error_code": "plugin_disabled",
  "field": "plugin_id",
  "timestamp": "2026-01-01T00:00:00Z"
}
```

### 5.3 处理策略

1. 输入验证错误：按现有结构化校验返回
2. 插件禁用错误：统一返回 `409/plugin_disabled`
3. 兼容性错误：返回明确版本或兼容性原因，不允许只说“失败”
4. 迁移期兼容：允许查看和配置，禁止不安全执行

## 6. 正确性属性

### 6.1 属性 1：统一状态单一真相

*对于任何* 需要判断插件是否可新增、可执行、可消费的入口，系统都应该满足：最终是否可用只由统一插件状态决定，而不是页面或业务模块再次拼条件。

**验证需求：** 需求 2、需求 3

### 6.2 属性 2：平行体系最终收口

*对于任何* 主题或 AI 供应商元数据读取场景，系统都应该满足：最终输出能够回到通用插件系统表示，而不是长期依赖一套独立注册表。

**验证需求：** 需求 1、需求 3

### 6.3 属性 3：版本字段不再只是展示

*对于任何* 插件注册结果，系统都应该满足：版本信息至少能支持统一展示、兼容性校验和升级提示边界，而不是只有一个孤立字符串。

**验证需求：** 需求 4

## 8. 风险与待确认项

### 8.1 风险

- 主题插件化如果直接硬切，容易把现有前端主题存储和默认主题逻辑打坏
- AI 供应商插件化如果只改元数据出口，不改消费入口，最后还是会留下平行逻辑
- 版本治理如果一次设计过大，容易写成大而空文档，迟迟落不了地
- 文档只新建不回写旧 Spec，后续接手的人还是会被旧文档带偏

### 8.2 待确认项

- 主题插件是否完全复用现有插件 manifest，还是需要单独的资源声明块
- AI 供应商插件是否需要新的配置 scope，还是继续复用现有 provider profile 配置模型
- 版本兼容性字段最小要做到哪一层：只做展示和告警，还是要参与强校验
- 旧 `001.5` 的哪些内容应保留，哪些内容应明确标记为被 `004.5` 接管
