# Spec 004.1 - 聊天模块全链路重构

状态：Draft

## 这份 Spec 解决什么问题

这份 Spec 专门处理当前用户端聊天模块的整条链路问题，不再继续给现有页面打零碎补丁。

本次会话已经确认过的事实：

- `/conversation` 页面不是没发请求，而是发到了单轮 `family_qa` 接口。
- 后端已经能调用 LLM，但前端默认 8 秒超时，模型一次成功调用实际用了 10 秒左右，结果被前端先判死刑。
- 当前“会话列表”和“消息记录”主要存在浏览器本地，不是后端真正的会话。
- 后端已经有家庭上下文、Agent 上下文、记忆上下文的组合能力，但没有把“多轮会话历史”当成正式上下文输入。
- `llm-task` 已经在管家初始化里用于提取配置草稿，但聊天链路还没把 `memory_extraction` 真正接起来。

所以这个 Spec 的目标很直接：把聊天模块改成一个真正可用、可恢复、可追踪、可扩展的聊天系统。

## 本次范围

本次只解决聊天模块本身的问题，包含：

- 后端真实会话和消息持久化
- 前端从本地伪会话切到服务端会话
- 多轮上下文拼装
- LLM 调用与超时、流式返回、失败反馈
- 记忆提取候选接入 `llm-task`
- 与现有 Agent、家庭上下文、记忆中心的衔接

## 本次明确不做

- 不重做整个 AI 配置中心
- 不把普通聊天直接改成“随便一句话就自动改 Agent 配置”
- 不在业务代码里绕过 Alembic 直接建表
- 不强行把所有旧接口删掉；能兼容就兼容，先保证不破坏现有页面和已有数据

## 主要文档

- `requirements.md`：为什么做、做什么、什么算做完
- `design.md`：怎么做、改哪些模块、接口和数据怎么落
- `tasks.md`：按什么顺序做、每一步看什么文件、怎么验收
