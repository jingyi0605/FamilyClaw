# 需求文档 - 聊天模块全链路重构

状态：Draft

## 简介

当前聊天模块的问题不是一个接口超时这么简单，而是整条链路设计没收口：

- 前端把单轮问答包装成了多轮聊天的样子
- 会话记录主要靠 `localStorage`
- 后端只保存单次 QA 日志，不保存真正的消息流
- LLM 成功返回时，前端可能已经因为超时先报错
- 已有的上下文组合和 `llm-task` 能力没有完整接进聊天主链路

这次重构的目标，是把聊天模块变成“真正的聊天模块”，而不是“问答接口加点样式”。

## 术语表

- **聊天会话**：用户与某个家庭 Agent 的一段连续交流，包含多条消息和状态。
- **聊天消息**：会话中的一条用户消息、助手消息或系统事件。
- **会话模式**：这段会话拿来干什么。比如普通家庭聊天、管家初始化、Agent 配置引导。
- **上下文组合**：把家庭实时状态、Agent 设定、记忆摘要、权限范围、多轮历史拼成一次模型输入。
- **记忆候选**：模型从会话中识别出的“可能值得长期记住的信息”，先给出候选，再决定是否落成正式记忆卡。
- **配置提取**：从会话里提取 Agent 配置草稿，例如名称、说话风格、性格标签。

## 需求

### 需求 1：后端必须有真正的聊天会话和消息

**用户故事：** 作为家庭用户，我希望聊天记录保存在服务端，而不是只存在我的浏览器里，这样我刷新页面、换设备或重新登录后还能看到同一段对话。

#### 验收标准

1. WHEN 用户在聊天页发出第一条消息 THEN System SHALL 在服务端创建会话并保存首条用户消息。
2. WHEN 用户继续追问 THEN System SHALL 把新消息写入同一个会话，而不是只在浏览器本地拼接。
3. WHEN 用户刷新页面或重新登录 THEN System SHALL 能从服务端恢复最近会话和消息列表。
4. WHEN 用户登出后由另一个账号登录同一浏览器 THEN System SHALL 不把前一个账号的聊天记录直接显示给后一个账号。

### 需求 2：聊天必须是真正的多轮对话

**用户故事：** 作为家庭用户，我希望“继续追问”真的能带上前文，而不是只把上一条回答截一段文字再发一次。

#### 验收标准

1. WHEN 用户在同一会话中连续发送多条消息 THEN System SHALL 在生成新回复时带上该会话最近的有效历史消息。
2. WHEN 用户切换 Agent 且当前会话已经有消息 THEN System SHALL 明确分出新会话，而不是让两个 Agent 共用一段上下文。
3. WHEN 用户打开某条历史会话 THEN System SHALL 显示该会话当时绑定的 Agent 和对应消息顺序。

### 需求 3：聊天必须接入现有上下文组合能力

**用户故事：** 作为家庭用户，我希望助手回答时能利用当前家庭状态、相关记忆和 Agent 人设，而不是只看我眼前这句话。

#### 验收标准

1. WHEN 后端生成一轮聊天回复 THEN System SHALL 组合家庭上下文、Agent 运行时上下文、记忆上下文和多轮会话历史。
2. WHEN 请求者没有某类数据的查看权限 THEN System SHALL 在上下文组合阶段裁掉该部分内容。
3. WHEN 家庭实时上下文处于降级状态 THEN System SHALL 把降级状态传递到回复结果和前端提示中。

### 需求 4：LLM 调用必须可靠，前端不能先把成功请求判失败

**用户故事：** 作为家庭用户，我希望模型慢一点时页面是“等待中”或“流式返回中”，而不是明明后端成功了，前端却告诉我失败了。

#### 验收标准

1. WHEN 后端模型调用耗时超过前端旧的 8 秒阈值 THEN System SHALL 仍能正确等待、流式展示或拿到最终结果。
2. WHEN 模型供应商失败但后端有明确降级结果 THEN System SHALL 在前端展示真实降级原因，而不是伪造一条通用安慰文案。
3. WHEN 一轮请求失败 THEN System SHALL 保留用户消息、失败状态和可重试入口。

### 需求 5：聊天链路必须复用现有 `llm-task` 能力

**用户故事：** 作为产品维护者，我希望已经做好的 `llm-task` 框架别继续摆着吃灰，能在聊天链路里真的发挥作用。

#### 验收标准

1. WHEN 会话模式是管家初始化或显式配置引导 THEN System SHALL 复用现有配置提取任务，而不是新造一套重复逻辑。
2. WHEN 普通聊天中出现值得长期记住的信息 THEN System SHALL 使用 `memory_extraction` 生成记忆候选，而不是只能靠用户手动复制粘贴。
3. WHEN 生成了记忆候选 THEN System SHALL 允许用户确认写入或忽略，而不是无提示直接落库。
4. WHEN 会话模式不是配置引导 THEN System SHALL 不自动覆盖 Agent 配置。

### 需求 6：前端状态必须和服务端状态保持一致

**用户故事：** 作为家庭用户，我希望会话列表、当前上下文、发送状态和错误提示是可信的，不要页面说一套，后端实际是另一套。

#### 验收标准

1. WHEN 用户进入聊天页 THEN System SHALL 从服务端加载会话列表和当前会话，而不是以本地缓存作为唯一事实来源。
2. WHEN 用户发送中、收到分块回复、回复完成或失败 THEN System SHALL 准确更新消息状态。
3. WHEN 页面存在本地缓存 THEN System SHALL 只把它当成短期 UI 缓存，不能覆盖服务端真实消息。

### 需求 7：改造必须兼容现有能力，不要把旧功能打烂

**用户故事：** 作为项目维护者，我希望这次重构能把聊天链路做好，但不要顺手把已有的家庭问答、Agent 选择、记忆页和初始化向导搞坏。

#### 验收标准

1. WHEN 新聊天链路上线 THEN System SHALL 继续兼容现有 Agent 选择、家庭上下文组合和 `qa_query_logs` 审计记录。
2. WHEN 现有初始化向导继续使用 `ButlerBootstrapConversation` THEN System SHALL 不因本次重构丢失原有能力。
3. WHEN 旧前端缓存里还有历史本地会话 THEN System SHALL 提供可控迁移或清理策略，而不是悄悄忽略导致用户困惑。

### 需求 8：聊天链路必须可验证、可排错

**用户故事：** 作为开发和联调人员，我希望能看出一轮聊天到底卡在了哪里，而不是只能猜。

#### 验收标准

1. WHEN 一轮聊天完成 THEN System SHALL 记录会话 ID、消息 ID、请求 ID、供应商、trace_id、是否降级。
2. WHEN 一轮聊天失败 THEN System SHALL 能区分前端取消、请求超时、供应商失败、权限失败、上下文构建失败等不同原因。
3. WHEN 开发或测试要验收聊天链路 THEN System SHALL 提供明确的验证步骤和预期结果。
