# 需求文档 - 对话设备控制MCP化与快捷路径收口

状态：Draft

## 简介

现在对话里的设备控制主链已经能识别一部分“打开客厅灯”“关空调”这类命令，但它的决策方式太脆弱：

- 动作主要靠关键词硬匹配
- 设备主要靠名字打分猜测
- 实体常常在执行阶段才临时决定
- 成功过一次的说法，下次还要重新猜一遍

这套做法的最大问题不是“偶尔会错”，而是它的上限很低。设备一多、别名一多、一个设备下挂多个实体、用户说法不固定，命中率就会明显下降。

这次 Spec 要解决的不是再补几条关键词规则，而是把对话设备控制改成一个更像人干活的流程：

1. 先查有没有已经验证过的快捷路径
2. 没命中时，让模型通过工具查询设备和实体
3. 拿到明确目标后，继续走现有统一设备执行链
4. 执行成功后，把“这句话对应哪个实体、做了什么动作”沉淀成快捷路径

这样做的目标很明确：让对话设备控制从“规则猜测”变成“工具检索 + 明确执行 + 持续学习”。

## 术语表

- **System**：FamilyClaw API Server 与现有对话主链
- **设备控制工具集**：供模型通过 tool use 调用的一组内部工具，用来查设备、查实体、读状态、执行动作
- **MCP 风格工具层**：对外语义按 MCP（Model Context Protocol，模型上下文协议）风格组织，但第一阶段不强制拆成独立进程
- **设备解析计划**：模型在一次对话设备控制中最终产出的结果，至少包括 `device_id / entity_id / action / params`
- **快捷路径**：一次成功设备操作沉淀出的可复用映射，记录用户说法、目标实体和执行动作
- **候选实体**：某个设备下实际可控的实体，例如 `light.study_main`
- **统一执行链**：现有 `device_action -> device_control -> plugin` 的正式设备执行主链

## 范围说明

### In Scope

- 把对话设备控制从纯规则匹配改成“快捷路径优先 + 模型工具决策”的双层方案
- 为对话主链补一套设备控制工具集，支持搜索设备、读取实体、查看当前状态、执行动作
- 在工具决策完成后，继续复用现有统一设备执行链，不新造第二套控制器
- 为成功执行的对话设备操作保存快捷路径，并支持后续优先命中
- 为高风险动作、歧义设备、多实体设备和低置信结果补清晰的追问或拦截策略
- 为整个设备解析与执行过程补结构化日志、审计字段和回归测试

### Out of Scope

- 这次不重做设备页面、设备同步页面和前端设备管理信息架构
- 这次不替换现有 `/device-actions/execute`、`device_control`、插件执行主链
- 这次不把场景执行、计划任务、网页按钮控制全部改成模型决策
- 这次不引入一个通用外部 MCP 平台市场；第一阶段只做对话设备控制所需的内部工具层
- 这次不做开放式“任意工具调用”，只开放设备控制白名单工具

## 需求

### 需求 1：系统必须先命中快捷路径，再决定是否进入工具决策

**用户故事：** 作为家庭成员，我希望常用说法在验证成功后能直接命中，不要每次都重新让系统猜设备和实体。

#### 验收标准

1. WHEN 用户发送设备控制语句 THEN System SHALL 先检查当前家庭下是否存在可用快捷路径，再决定是否进入工具决策。
2. WHEN 快捷路径命中且目标设备、实体、动作仍然有效 THEN System SHALL 直接走统一执行链，而不是重复做整套检索。
3. WHEN 快捷路径命中但目标设备已失效、实体不可控或动作不再支持 THEN System SHALL 放弃该快捷路径并回退到工具决策，而不是继续盲执行。
4. WHEN 快捷路径命中被使用 THEN System SHALL 记录命中次数和最近使用时间，供后续排序和清理使用。
5. WHEN 用户再次使用语义等价但不完全相同的说法，且系统能低风险确认目标仍是同一设备、同一实体、同一动作 THEN System SHALL 复用已有快捷路径，而不是新增第二条重复记录。

### 需求 2：系统必须支持模型通过工具查询设备和实体

**用户故事：** 作为家庭成员，我希望系统在不确定目标时先查设备和实体，而不是靠名字碰运气。

#### 验收标准

1. WHEN 快捷路径未命中或命中失效 THEN System SHALL 允许模型调用设备控制工具集来搜索候选设备和实体。
2. WHEN 模型需要判断某个设备下到底能操作什么 THEN System SHALL 提供读取实体列表、实体状态和控制能力的工具结果。
3. WHEN 同名设备、同房间多个实体或房间指代不明确 THEN System SHALL 把候选结果返回给模型继续决策，而不是直接在规则层拍脑袋选一个。
4. WHEN 工具查询结果不足以唯一确定目标 THEN System SHALL 返回明确追问，而不是伪装成已找到目标。

### 需求 3：系统必须让模型决策只负责“选目标”，真正执行仍走统一执行链

**用户故事：** 作为后端维护者，我希望模型只负责决定操作哪个实体、做什么动作，真正执行仍然走现有统一控制主链，这样权限、审计和风险控制不会散掉。

#### 验收标准

1. WHEN 模型已经得到明确的 `device_id / entity_id / action / params` THEN System SHALL 继续调用现有统一设备执行入口完成真实控制。
2. WHEN 设备控制请求进入执行阶段 THEN System SHALL 继续复用现有权限校验、设备归属校验、高风险确认、插件启用校验和审计逻辑。
3. WHEN 执行成功或失败 THEN System SHALL 返回统一的设备执行结果和错误结构，不允许模型直接绕过后端执行真实平台动作。
4. WHEN 现有页面或语音链路继续使用统一执行链 THEN System SHALL 不因为本 Spec 破坏它们的接口语义。

### 需求 4：系统必须把成功设备操作沉淀成快捷路径

**用户故事：** 作为经常使用对话控制设备的成员，我希望系统能记住“我平时怎么说”，这样同一句话下次可以更快、更稳地命中。

#### 验收标准

1. WHEN 一次对话设备操作执行成功 THEN System SHALL 记录该轮原始说法、标准动作、目标设备和目标实体，形成快捷路径候选。
2. WHEN 相同家庭成员重复使用相近说法并稳定命中同一实体 THEN System SHALL 合并或更新已有快捷路径，而不是无限新增重复垃圾记录。
3. WHEN 快捷路径长期不再可用、设备被删除或实体失效 THEN System SHALL 支持失效标记或自动降级，而不是永远保留脏记录。
4. WHEN 用户没有明确说设备名，但系统通过上下文成功解析了目标 THEN System SHALL 仍可保存快捷路径，但必须保留置信和来源信息。
5. WHEN 多条快捷路径都可能匹配同一句相近说法且系统无法拉开足够差距 THEN System SHALL 返回未命中或继续澄清，而不是为了复用缓存强行挑一条。

### 需求 5：系统必须正确处理歧义、高风险和低置信场景

**用户故事：** 作为家庭成员，我希望系统在拿不准时追问，在高风险动作前确认，而不是为了显得聪明就乱动设备。

#### 验收标准

1. WHEN 模型给出多个同等候选设备或多个同等候选实体 THEN System SHALL 明确追问用户指定目标。
2. WHEN 动作属于高风险操作，例如 `unlock` THEN System SHALL 继续遵守现有高风险确认边界，不因为进入模型工具决策就放宽。
3. WHEN 模型输出的目标实体与设备能力不匹配 THEN System SHALL 拒绝执行并回到澄清路径，而不是强行映射。
4. WHEN 外部依赖不可达、工具查询失败或插件执行失败 THEN System SHALL 返回用户可理解的失败提示，并保留结构化排查信息。

### 需求 6：系统必须对设备控制决策过程可观测、可回放

**用户故事：** 作为维护者，我希望出了问题时能看清这次到底命中了快捷路径、调用了哪些工具、最后为什么选了这个实体，而不是只看到一句“控制失败”。

#### 验收标准

1. WHEN 一次对话设备控制开始执行 THEN System SHALL 记录是否命中快捷路径、是否进入工具决策、工具调用摘要和最终执行计划。
2. WHEN 设备控制失败 THEN System SHALL 能区分“快捷路径失效”“模型未能唯一定位”“统一执行链失败”“平台不可达”这些不同阶段的错误。
3. WHEN 维护者排查一次控制问题 THEN System SHALL 能从对话日志、设备控制日志和快捷路径记录中串起完整链路。

## 非功能需求

### 非功能需求 1：性能

1. WHEN 快捷路径命中 THEN System SHALL 在现有快控链路基础上尽量减少额外耗时，不引入明显多跳。
2. WHEN 进入工具决策 THEN System SHALL 控制工具调用轮数，避免为了查一个灯把整轮对话拖成慢查询。
3. WHEN 成功写入快捷路径 THEN System SHALL 采用轻量回写，不阻塞主回复超过可接受阈值。

### 非功能需求 2：可靠性

1. WHEN 模型工具决策失败 THEN System SHALL 回退为明确澄清或失败提示，不允许默默执行错误设备。
2. WHEN 快捷路径与当前设备状态不一致 THEN System SHALL 优先保证安全和正确性，宁可回退查询，也不允许直接误控。
3. WHEN 同一次设备控制重复提交 THEN System SHALL 继续复用现有幂等与统一执行保护，不重复落平台动作。

### 非功能需求 3：可维护性

1. WHEN 后续需要把语音设备控制接入同一能力 THEN System SHALL 复用这次抽出的工具层和快捷路径层，而不是再写一套语音专用猜测逻辑。
2. WHEN 后续需要把内部工具层外置成正式 MCP server THEN System SHALL 主要调整工具暴露方式，而不是重写设备解析和执行逻辑。
3. WHEN 维护者阅读代码 THEN System SHALL 能明显看出“决策层”和“执行层”边界，不再把关键词匹配、实体选择和平台控制混在一起。

## 成功定义

- 对话设备控制不再主要依赖 `_infer_fast_action + _match_fast_action_devices` 这类硬编码规则
- 常用说法能通过快捷路径稳定复用，不再每次重新猜设备和实体
- 未命中快捷路径时，系统能通过工具查询设备和实体后再执行，而不是直接拍脑袋
- 统一设备执行链保持不变，页面控制、场景控制和后续语音控制不会被这次改造打断
- 维护者能够从日志里看清一次控制到底是“怎么找到实体并执行”的
