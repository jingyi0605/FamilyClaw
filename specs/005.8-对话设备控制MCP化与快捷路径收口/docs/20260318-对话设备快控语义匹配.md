# 对话设备快控语义匹配

## 这页解决什么

这页说明对话里的设备快控 shortcut 现在怎么命中。

以前的规则很死，只认“字符串归一化后完全相同”。
现在改成了“精确优先，语义兜底”：

- 先走原来的精确命中
- 精确没命中，再做一次低风险语义匹配
- 如果结果有歧义，宁可不命中，也不误控设备

这不是开放式模糊搜索，也不是向量召回，更不是把整条推理链缓存下来重复放一遍。

## 现在缓存的到底是什么

系统保存的仍然是设备快控 shortcut 映射，不是完整的推理链。

每条 shortcut 记录的核心信息是：

- 用户当时说的原始短语 `source_text`
- 归一化后的短语 `normalized_text`
- 命中的设备 `device_id`
- 命中的实体 `entity_id`
- 动作 `action`
- 命中次数 `hit_count`
- 最近使用时间 `last_used_at`

所以它缓存的是“这句话最终控制了哪个设备、哪个实体、执行了什么动作”。
后续再来一句类似指令，系统先尝试复用这条 shortcut。

## 匹配顺序

### 第一步：精确命中

先把用户输入做归一化：

- 去空格
- 转小写
- 去常见标点
- 去掉“请、帮我、一下”之类不影响语义的填充词

如果归一化结果和已有 shortcut 的 `normalized_text` 一样，就按原逻辑继续走。

这一步命中后，还会检查目标设备和实体是否还有效。
如果设备已经没了，或者实体已经失效，这条 shortcut 会被标成 `stale`，不会继续复用。

### 第二步：语义兜底

精确没命中时，才会扫描当前家庭下可用的 active shortcuts，做一次保守评分。

评分思路很简单，没搞花活：

1. 先按设备类型推断动作语义  
   比如灯的“打开/开启/点亮”会归到 `turn_on`，窗帘的“打开/拉开”会归到 `open`。

2. 从用户输入里剥掉动作词，只保留目标短语  
   例如“开启书房主灯”会提取成“书房主灯”。

3. 用目标短语去和候选 shortcut 的目标信息比重合度  
   候选目标信息来自三类文本：
   - shortcut 自己历史上保存的短语
   - 设备名
   - 对应实体名

4. 只有分数够高，且第一名明显好于第二名，才算命中

当前实现里的安全阈值：

- 最低命中分：`24`
- 第一名和第二名的最小分差：`8`

这两个阈值的意义很直接：不够确定就别乱猜。

## 什么情况会命中

下面这些属于预期命中：

- 已有 shortcut 是“打开书房主灯”，再次说“开启书房主灯”
- 已有 shortcut 是“打开客厅灯”，再次说“开启客厅灯”
- 历史短语里没直接写实体名，但设备名或实体别名能把目标补全

这类场景的共同点是：动作语义一致，目标对象也足够明确。

## 什么情况不会命中

下面这些场景会故意返回 `None`：

- 只看得出“书房灯”，但书房里同时有主灯和壁灯
- 动作词和设备类型不匹配，推断不出合理动作
- 目标短语太短，和多个 shortcut 都接近
- shortcut 指向的设备或实体已经失效

这不是能力不够，这是故意收紧边界。
设备控制链路最怕的不是“没帮你省一步”，而是“替你点错了东西”。

## 为什么要按设备类型做动作归一化

“打开”这个词本身就很脏。

对灯来说，“打开”更像 `turn_on`。
对窗帘来说，“打开”更像 `open`。

如果不看设备类型，直接拿“打开”去做统一动作归一化，结果就是把窗帘、灯、门锁搅成一锅粥，误命中概率会直线上升。

所以现在的做法是：

- 灯、空调、音箱优先走 `turn_on` / `turn_off`
- 窗帘优先走 `open` / `close`
- `lock` / `unlock` / `stop` 单独处理

这才是正常的数据建模。先把动作语义和设备类型绑紧，再谈复用 shortcut。

## 对现有行为的影响

这次改动没有改数据库表结构，也没有把旧 shortcut 作废。

已有 shortcut 仍然会继续使用，只是匹配方式从：

- 以前：归一化后完全相等才命中

变成了：

- 现在：完全相等优先，失败后再尝试低风险语义匹配

所以这是兼容性增强，不是行为翻盘。

## 回归检查

本次已经补了三类测试：

- 同一目标的语义别名可以命中同一条 shortcut
- 多个候选目标接近时，语义匹配返回 `None`
- 端到端执行时，语义别名会复用旧 shortcut，而不是新建第二条

如果后面继续扩动作词或设备类型，优先补测试，不要先加分支再赌运气。
