# 任务清单 - 对话设备控制MCP化与快捷路径收口（人话版）

状态：Draft

## 这份任务清单是干什么的

这份任务清单只做一件事：把 005.8 这个 Spec 拆成真正能落地的步骤。

别人接手时，应该一眼看明白：

- 先做哪一层
- 哪一步只是铺路，哪一步会真正改变主链
- 每一步主要改哪些文件
- 做完以后怎么验收，不靠“我感觉差不多了”

## 状态说明

- `TODO`：还没开始
- `IN_PROGRESS`：正在做
- `BLOCKED`：被外部问题卡住
- `IN_REVIEW`：已经有结果，等复核
- `DONE`：已经完成，并且已经回写状态
- `CANCELLED`：取消，不做了，但要写原因

规则：

- 只有 `状态：DONE` 的任务才能勾成 `[x]`
- 每完成一个任务，必须立刻回写这里
- 如果范围变化，先改 Spec，再改任务，不要闷头改代码

## 阶段 2：把对话主链切到新决策方式

- [x] 2.1 在对话设备控制入口接入“快捷路径优先”
  - 状态：DONE
  - 这一步做什么：在 `conversation fast_action` 入口里先查快捷路径，再决定是否进入模型工具决策。
  - 做完以后能看到什么：常用说法会先走快捷路径，未命中时才走下一层。
  - 依赖什么：1.3
  - 主要改哪些文件：
    - `apps/api-server/app/modules/conversation/orchestrator.py`
    - `apps/api-server/app/modules/conversation/`
  - 这一步不做什么：不把旧规则兜底一次性删光，避免直接打断现有能力。
  - 完成标准：
    1. 命中快捷路径时能直接生成执行计划
    2. 快捷路径失效时会回退，而不是硬执行
    3. 精确短语未命中时，会继续尝试低风险语义匹配，而不是立刻放弃
    4. 语义候选有歧义时，必须返回未命中，不能盲猜
  - 怎么验证：
    - 对话设备控制定向测试
    - 快捷路径命中与失效回退测试
    - 语义别名命中测试
    - 语义歧义拒绝命中测试

- [x] 2.2 接入模型工具规划器，替换主要的设备/实体猜测逻辑
  - 状态：DONE
  - 这一步做什么：当快捷路径未命中时，让模型通过工具查询设备和实体，产出正式执行计划，再调用统一执行链。
  - 做完以后能看到什么：对话设备控制的主决策不再主要依赖关键词和名字打分。
  - 依赖什么：2.1
  - 主要改哪些文件：
    - `apps/api-server/app/modules/conversation/orchestrator.py`
    - `apps/api-server/app/modules/conversation/` 下 planner / tool registry 相关组件
    - `apps/api-server/app/modules/llm_task/`（如需要补 tool use 支持）
  - 这一步不做什么：不把语音链路一起切过来，避免范围失控。
  - 完成标准：
    1. 未命中快捷路径时，系统能通过工具拿到候选设备和实体
    2. 歧义场景会追问，不会乱控
    3. 高风险动作仍然遵守现有确认边界
  - 怎么验证：
    - 多实体设备测试
    - 歧义追问测试
    - 高风险动作测试

- [x] 2.3 主链检查点
  - 状态：DONE
  - 这一步做什么：确认新主链已经能覆盖“命中快捷路径 / 工具决策 / 统一执行 / 澄清失败”这几个核心场景。
  - 做完以后能看到什么：对话设备控制不再是几块零散逻辑，而是闭环。

## 阶段 3：把快捷路径闭环和排查能力补齐
