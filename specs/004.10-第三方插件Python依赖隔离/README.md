# Spec 004.10 - 第三方插件Python依赖隔离

状态：Draft

## 这份 Spec 解决什么问题？

当前第三方插件系统有个很实在的硬伤：

- 市场安装和本地安装都要求插件仓库带 `requirements.txt`
- 但系统实际上只检查这个文件在不在
- 执行插件时仍然直接复用宿主的 `sys.executable`
- 插件代码能不能 import 成功，取决于宿主环境里碰巧装了什么包

这不是“体验问题”，这是运行模型本身有洞。

一个第三方插件如果需要额外 Python 依赖，当前只有两条路：

1. 把插件依赖偷偷加进宿主依赖列表
2. 等运行时 `ModuleNotFoundError` 再炸

这两条都很烂。

这份 Spec 只解决一件事：

让第三方插件拥有自己的 Python 依赖环境。安装、升级、启动恢复和执行时，都不再复用宿主解释器。

## 为什么现在做？

因为问题已经被微信插件踩出来了，而且不是特例。

今天是 `qrcode`
明天可能是某个厂商 SDK
后天可能是数据库驱动、加密库或者 OCR 包

如果每次都把插件依赖加回宿主，那“第三方插件”这四个字就成了笑话。

现在停下来补这个能力，成本是一次插件系统重构；继续拖，代价会变成：

- 宿主依赖膨胀
- 第三方插件边界失真
- 部署结果不可预测
- 排障时永远分不清是插件缺依赖，还是宿主环境脏了

## 核心判断

✅ 值得做：这是插件系统的基础能力缺口，不补上，后面任何需要独立 Python 依赖的第三方插件都会重复踩坑。

## 关键洞察

- 数据结构：真正缺的不是“再多一个字段”，而是“第三方插件运行时环境”这个正式对象。
- 复杂度：现在的复杂度不是来自 venv，而是来自“插件名义独立、运行时却偷宿主环境”这种半吊子设计。
- 风险点：如果只给安装流程加 venv，不改 runner 和启动恢复，最后会变成“装的时候像是隔离了，跑的时候还是宿主解释器”。

## 这次要交付什么？

- 一份明确写清现状缺陷和目标边界的需求文档
- 一份说明第三方插件 venv 生命周期、依赖安装、runner 切换和恢复策略的设计文档
- 一份可执行的任务拆分，覆盖安装、升级、启动恢复、执行、文档和回归
- 一份现状缺陷确认文档，给后续评审和验收当事实依据

## 这次明确不做什么？

- 不把 builtin 插件也改成独立 venv
- 不引入 pipx、conda、poetry 之类新的包管理体系
- 不顺手重做整个插件市场
- 不要求一次性支持非 Python 后端语言隔离
