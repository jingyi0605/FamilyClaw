# 20260407-现状缺陷确认

## 这份文档解决什么问题？

这份文档只做一件事：把当前第三方插件系统为什么确实没有 Python 依赖隔离写成事实，避免后面再有人把它说成“只是某个插件的偶发问题”。

## 已确认的事实

### 1. `requirements.txt` 现在只是存在性检查

在 `app/modules/plugin_marketplace/service.py` 里：

- 系统会解析 `requirements_path`
- 系统会检查文件存在
- 如果不存在会报错

但到这里就结束了。

当前实现没有看到后续真正执行：

- `pip install -r requirements.txt`
- `python -m pip install -r requirements.txt`
- 或任何等价依赖安装逻辑

结论：

`requirements.txt` 现在不是可执行安装契约，只是仓库结构校验材料。

### 2. 第三方插件的 `python_path` 现在默认写成宿主解释器

在这些位置都能看到：

- `app/modules/plugin/service.py`
- `app/modules/plugin/startup_sync_service.py`
- `app/modules/plugin_marketplace/service.py`

当前第三方插件挂载和市场实例写回 `python_path` 时，直接使用的是：

- `sys.executable`

结论：

第三方插件没有自己的正式解释器路径模型，当前默认就是偷宿主解释器。

### 3. runner 执行时只是拼 `PYTHONPATH`

在 `app/modules/plugin/executors.py` 里，第三方插件走 `subprocess_runner` 时：

- 命令是 `[runner_config.python_path, "-m", RUNNER_MODULE]`
- 运行前会拼 `PYTHONPATH`
- 但不会创建插件 venv，也不会装依赖

结论：

当前 runner 的“隔离”只隔离了 import 路径，没有隔离 Python 依赖环境。

### 4. 开发版插件更直接，就是把路径插到宿主 `sys.path`

在 `app/modules/plugin/import_path.py` 里：

- `plugin_runtime_import_path(...)` 直接把插件路径插进宿主 `sys.path`

结论：

开发版插件同样没有依赖隔离，只是更直接地复用宿主环境。

## 这个缺陷会造成什么结果？

### 结果 1：插件依赖被迫污染宿主依赖

只要某个第三方插件需要额外 Python 包，当前系统里最简单但最烂的解法就是：

- 把插件依赖加回宿主依赖表

这会直接破坏第三方插件边界。

### 结果 2：运行成功取决于宿主环境偶然性

如果宿主环境里刚好装了这个包，插件就“看起来没问题”；
如果没装，运行时就会直接 `ModuleNotFoundError`。

这不是稳定运行模型，这是碰运气。

### 结果 3：`requirements.txt` 变成假承诺

插件作者明明写了 `requirements.txt`，系统也明明要求它存在，但最后并不会替插件安装。

这会让文档、仓库结构和真实行为完全脱节。

## 一句话结论

当前第三方插件系统确实存在核心缺陷：

它要求插件声明 Python 依赖，却没有提供插件级 Python 依赖隔离和安装能力。

所以这次要修的不是某个插件的特例，而是插件系统自己的正式运行模型。
